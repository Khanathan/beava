How to build a per-customer analytics dashboard

Part 1 · Meet beava

Same shape as pandas, live forever.

beava is one binary that ingests events and keeps per-entity aggregates up to date in memory. You describe the pipeline in Python, it serves features over HTTP. If you know df.groupby(...).agg(...), you already know the shape.

A stream is a dataframe that never stops appending. A table is a groupby(key).agg(...) over that stream, maintained incrementally on every event.

pandas (batch)
events.groupby("user_id").agg(
    views_total=("path", "count"),
    last_seen=("ts", "max"),
)
≈
beava (live)
events.groupby("user_id").agg(
    views_total=bv.count(),
    last_seen=bv.latest(e.ts),
)
Same shape, different time model. pandas runs once over a frame. beava maintains the groupby forever.
Part 2 · Your first pipeline

Three lines. One table. Live.

Declare a PageView event shape and a UserStats table keyed by user. Every event updates exactly one row, atomically. Register the pipeline, then send an event.

# your first pipeline — 3 lines of logic
import beava as bv

@bv.stream
class PageView:
    user_id: str
    path:    str

@bv.table(key="user_id")
def UserStats(e: PageView):
    return e.agg(views_total=bv.count())

bv.App("0.0.0.0:6400").register(PageView, UserStats).serve()

Part 3 · Build the dashboard

Now the real thing: four metrics, rendered live.

Same stream, richer table. Lifetime total views, rolling 24h views, distinct categories over the past 7 days, and the latest timestamp per user. Every push updates every metric atomically. Register to replace your first pipeline, then watch the dashboard populate below.

# a per-customer analytics dashboard
import beava as bv

@bv.stream
class PageView:
    user_id:  str
    path:     str
    category: str

@bv.table(key="user_id")
def Dashboard(e: PageView):
    return e.agg(
        views_total     = bv.count(),                     ← lifetime views
        views_24h       = bv.count(window="24h"),        ← today's activity
        distinct_cats_7d= bv.distinct(e.category, window="7d"), ← category breakdown
        last_seen       = bv.latest(e.ts),                ← "last seen"
    )

bv.App("0.0.0.0:6400").register(PageView, Dashboard).serve()

Part 4 · Cost at scale

How much RAM will this take?

Pick a scale. Every metric is sized per-entity, not per-event — so one billion events on a million users costs exactly the same as two billion events on the same million users.

entities
1M
distinct user_ids on the key axis
memory footprint
858 MB
≈ 900 B × hot entities (weighted avg, window TTL)
aws fit (+50% headroom)
$15/mo
t3.small · 2 GiB RAM
per-user breakdown (~852 B expected, round to 900 B for slack)
views_totali64 counter8 B
views_24h24 hourly buckets (i64 each)192 B
distinct_cats_7d3-tier weighted: 95% ExactArray (128 B) + 5% HashSet (32 KB)480 B
last_seeni64 timestamp8 B
user_id keystring + hash-table slot100 B
row overheadschema ptr + version + tombstone + padding64 B
Events don't add memory. The sketch tiers and window TTL do the work: distinct-count promotes through ExactArray (≤16, ~128 B) → HashSet (≤1024, ~32 KB) → HLL p=12 (~4 KB, constant) per-user as cardinality grows, and any user inactive past the widest window (7d here) ages out of the hot set entirely. One billion events on 1M hot users is still well under a gigabyte.
