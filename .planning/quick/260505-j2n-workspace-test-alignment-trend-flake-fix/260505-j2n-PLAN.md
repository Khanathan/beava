### Post-Phase-13.4 verb-style GET contract (D-01, D-02, D-03)

**Request body:** `POST /get` accepts `{table: string, key: string, features?: string[]}`. Legacy `{keys: string[], features: string[]}` is rejected with HTTP 400 + structured error:
```json
{"error":{"code":"unsupported_request_shape","message":"POST /get expects {table, key, features?}; received legacy {keys, features} shape — see docs/http-api.md#post-get"}}
```

**Response body:** Flat dict, no envelope. The features ARE the response body keys directly:
```json
{"cnt": 10, "total": 42.5}
```
NOT `{"alice": {"cnt": 10}}` (legacy multi-key envelope). NOT `{"result": {...}}`. NOT `{"row": {...}}`.

Cold-start (no events for that key): empty object `{}`.

**Single-feature GET unchanged:** `GET /get/:feature/:key` still returns `{"value": <val>}`.

**Test-rewrite map for GET calls:**

| BEFORE (legacy)                                                                                                  | AFTER (verb-style)                                                                                                            |
|------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------|
| `post_json("/get", &json!({"keys": ["alice"], "features": ["cnt"]}))` <br> assert `body["alice"]["cnt"] == 10`     | `post_json("/get", &json!({"table": "TxCount5m", "key": "alice", "features": ["cnt"]}))` <br> assert `body["cnt"] == 10`        |
| `post_json("/get", &json!({"keys": [k], "features": ["cnt"]}))` <br> assert `body[k]["cnt"]`                       | `post_json("/get", &json!({"table": "TxCount5m", "key": k, "features": ["cnt"]}))` <br> assert `body["cnt"]`                    |

The `table` field is the derivation NAME (the table-typed derivation that the agg lives on). `features` is optional — omit to get all features for the row, or pass a subset.

### Post-Phase-13.4 force=true register contract (D-01)

**Destructive registry changes** (rename, type-change, op removal, agg removal, window-change, key-cols change, **including dropping a previously-registered descriptor**) require `force: true` in the register payload, OR the new payload must include ALL prior nodes (additive-by-replacement: register replaces the prior set verbatim; missing prior nodes count as DROPS = destructive).

Without `force: true`, server returns HTTP 409 with structured body:
```json
{"error":{"code":"force_required","reason":"Destructive registry change requires force=true. See diff for details.","diff":{"additive":[...],"destructive":[{"kind":"rename","from":"X","to":"Y"}, ...]}},"registry_version":<prior>}
```

**Two alignment strategies for failing tests:**

1. **Strategy A (preferred): include all prior nodes** — re-send Transaction + TaggedTx + OnlyBig in one payload when adding OnlyBig. Cleanest because it preserves the test's intent (no force). Used for sc2_with_columns, phase7_restart_cycle SC4 / bonus, phase4_sc5_tcp.

2. **Strategy B (fallback): pipeline 3 register frames as ONE multi-node payload** — for criterion_6_pipelined_registers, the test's INTENT was wire-level pipelining ordering. The legacy interpretation (3 separate register calls additively merging) is dead post-13.4. Re-cast the test as ONE register payload with all 3 nodes; the wire-level pipelining property is now exercised by the existing TCP frame-codec proptest in `beava-core::wire::tests`. Belt-and-suspenders kept.

   Note: Strategy A would change the test's nature (replacing 3 frames with 1), losing the pipelining-on-the-wire signal. To PRESERVE the multi-frame pipelining check, use a different opcode for the secondary frames — `OP_PING` (0x0000) is harmless and proves the connection accepts pipelined frames in order. We'll go with this hybrid: 1 register frame followed by 2 pipelined OP_PING frames; assert all 3 responses arrive in order.

### Trend slope contract (server-side, FROZEN — DO NOT MODIFY)

```rust
// crates/beava-core/src/agg_state_velocity.rs:253-263
impl TrendState {
    pub fn slope(&self) -> Option<f64> {
        if self.n < 2 {
            return None;
        }
        let n = self.n as f64;
        let denom = n * self.sum_xx - self.sum_x * self.sum_x;
        if denom == 0.0 {
            return None;
        }
        Some((n * self.sum_xy - self.sum_x * self.sum_y) / denom)
    }
}
```

`now_ms` is the OLS x-axis. With 1000 events / 3 users pushed in ~1ms wall time, all events for a given user can land at the SAME `now_ms` → `var(t) == 0` → `denom == 0.0` → slope returns `None`. This is the documented engine-correct fallback for a degenerate time-axis. The sibling `test_trend_residual_per_user_high_volume` (same file, line 294-295) already accepts `None` as the contract sentinel:

```python
if resid is None:
    continue
```

Wave 2 fix: relax `test_trend_per_user_high_volume` to the SAME contract — `if slope is None: continue` (mirroring the sibling), then assert finite/bounded ONLY when a value is returned. The test's existing comment block already documents this is engine-correct; the contradiction was between the comment and the assertion.

This is a **test bug** (test contract was internally inconsistent), NOT a server bug. No `crates/beava-core/` changes required.
