# Phase 12.9 — AggOp memory boxing — CONTEXT

**Status:** Inserted 2026-05-03 between Phase 12.8 (memory governance) and Phase 13 (final ship). Triggered by post-Phase-12.8 r8g maxcard bench investigation captured at `.planning/ideas/per-entity-memory-budget.md`.

**Goal:** Close the fraud-team 22 KB / entity vs CLAUDE.md 7 KB budget gap by boxing 7 fat AggOp variants. **Conditional on perf regression < 5%** — if boxing the 7 variants regresses small/tcp throughput by more than 5%, the change is reverted and the v0 pitch is reframed instead (option b from the investigation doc).

**Why a separate phase:** Phase 13 ships the v0 perf-pitch numbers ("~7 KB / entity → 700 GB for 100M entities" per CLAUDE.md). Without this fix, fraud-team users see 3× the budget. Doing the fix as its own phase keeps the boxing change cleanly separable from the SDK polish + packaging work in Phase 13.

---

## Locked decisions

### D-01 — Box exactly the 7 fat variants (not the borderline ones)

Box: `SeasonalDeviation`, `HourOfDayHistogram`, `EventTypeMix`, `GeoVelocity`, `GeoSpread`, `GeoDistance`, `DistanceFromHome`. These all have `size_of` ≥ 80 bytes and together set the inline-payload floor at 600 B.

**Do NOT box:**
- `TrendResidual` (72 B) and `BurstCount` (64 B) — borderline; would only drop the floor by another 8 B, not worth the indirection cost
- `Histogram` (48 B) — under threshold
- `MostRecentN` (48 B) and `ReservoirSample` (40 B) — under threshold

Result: `size_of::<AggOp>()` should drop from 600 B to **72 B** (`TrendResidual` becomes the new floor-setter), an **8× shrink**.

### D-02 — Override the Phase 11 D-08 explicit-no-boxing comment, contingent on perf gate

`crates/beava-core/src/agg_op.rs` lines 405–408 explicitly state:

> *size delta intentionally — every aggregation entity allocates exactly one `Vec<AggOp>` for its slots, and per-feature box indirection would dominate the small per-event update path that Phase 13 needs to keep <300 ns.*

Phase 11 D-08 traded memory-per-entity for per-update CPU cost. **Phase 12.9 reverses this trade if and only if** the perf measurement (Plan 12.9-02) confirms that:

1. The per-event indirection cost on the **7 boxed variants** is small (likely true — fraud-team uses these variants 1–2 times per entity, and only some events match each feature)
2. The cache-locality win from a 8× smaller `Vec<AggOp>` slot offsets the indirection
3. Net throughput regression on small/tcp regression-gate ≤ 5% vs Phase 12.8 baseline (751,498 EPS post-12.7 baseline; -2.5% on 12.8 → ~733k EPS gate)

If the gate fails, **revert the boxing** and adopt option (b) from the investigation doc: reframe the v0 pitch as "~7 KB / entity for simple-shape pipelines; ~22 KB for fraud-shape" with a note that boxing is a v0.0.x experiment.

### D-03 — No `FORMAT_VERSION` bump

`serde::Serialize for Box<T>` is **transparent** — the wire format of `Box<SeasonalDeviationState>` is byte-identical to `SeasonalDeviationState`. bincode uses serde, so WAL records and snapshot files written by post-12.9 code are readable by pre-12.9 code (and vice versa). The schema change is purely in-memory.

This decision overrides the investigation doc's conservative claim that a `FORMAT_VERSION` bump might be needed. Verified by:
- serde's blanket impl in `serde::ser::impls`: `impl<T: Serialize + ?Sized> Serialize for Box<T> { fn serialize(&self, ...) { (**self).serialize(...) } }`
- Same for Deserialize via `Box::new(T::deserialize(...))`

Phase 12.7 just RESET FORMAT_VERSION 2→1 four days ago; bumping again now without a wire-incompatible change would churn the persistence test matrix unnecessarily.

### D-04 — Bundle `bytes_per_entity_p99` dynamic sampling into Plan 12.9-02

Phase 12.8 ships `beava_bytes_per_entity_p99` as a **static placeholder = 7000** in `crates/beava-core/src/agg_state.rs::EntityCountResidentSnapshot`. With actual per-entity bytes now varying significantly post-12.9 (from ~22 KB pre-fix to ~6 KB post-fix), the static placeholder becomes meaningless. Replace with dynamic sampling — small reservoir of recent entity sizes, p99 computed on snapshot.

Folds into Plan 12.9-02 (perf gate) so the metric reflects the post-fix reality from day one of Phase 13 ship work.

### D-05 — Keep the size_of dump test as a permanent regression gate

Promote `crates/beava-core/tests/per_entity_size_dump.rs` (currently an investigation tool) into a **CI-enforced** size_of regression gate by tightening its assertion to `assert!(size_of::<AggOp>() <= 80)`. Future operator additions that exceed the threshold force the contributor to either Box the new variant or explicitly bump the threshold (which surfaces the trade-off for review).

This is the same pattern as the 4 architectural-test CI tripwires Phase 12.8 added (`phase12_8_lifetime_ops_have_bounds`, `phase12_8_cold_entity_eviction`, `phase12_8_unbounded_op_in_lifetime_mode`, `phase12_8_metrics_endpoint`).

### D-06 — Update `crates/beava-core/src/agg_op.rs` doc-comment block (lines 399-408) to reflect the new trade-off

The current block documents the "we accept the 600B size delta" decision. Phase 12.9 reverses that. Update the comment to:


## Anti-decisions (for the record)

- **Did NOT decide to box the windowed inner ops separately** — `WindowedOp` is already boxed at the outer level; its inner `(i64, Box<AggOp>)` buckets benefit transparently from the new smaller `AggOp`.
- **Did NOT decide to bump TrendResidual / BurstCount** — borderline cases; revisit only if a future workload shows them dominating per-entity cost after the 7 fat variants are boxed.
- **Did NOT decide to do the on-EKS r8g rebench** — local M4 throughput regression-gate is sufficient to confirm no perf regression. The 22 KB → 6 KB memory lift is independently verifiable from the size_of test + the fraud-team feature count math; we don't need r8g to confirm a math identity. EKS rebench can happen post-ship if a customer flags it.
