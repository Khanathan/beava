    .join(other, on: str | list[str], within: str | None = None, type: str = "inner") — called on Stream or Table:

    Shape dispatch by (self_type, other_type):
      - Stream ↔ Stream (windowed):
        - `within` REQUIRED; else TypeError("Stream↔Stream join requires within=... (e.g. '30m')")
        - `type` must be in {"inner", "left"}; else "outer joins deferred to v0.1" RuntimeError; also reject "right", "full", "cross"
        - Output: Stream (windowed join produces a stream of joined events)
      - Stream ↔ Table (enrichment):
        - `within` NOT allowed; if provided, TypeError
        - `type` must be "inner" or "left"; outer rejected as above
        - Output: Stream (the left Stream enriched with the Table's current row at event-time)
      - Table ↔ Table (same-key):
        - `within` NOT allowed; TypeError if provided
        - `type` must be "inner" or "left"
        - Keys: both tables' full key lists must match `on` exactly; else RuntimeError("Table↔Table join requires full-key match; Table 'A' key=[...], 'B' key=[...], on=[...]")
        - Output: Table with combined schema, using `_right` suffix on field-name collisions

    Schema inference (common):
      - Non-key fields are unioned; on collision, left keeps its name, right gets `_right` suffix
      - Test asserts a collision case
      - If left and right schemas are identical except for key, the right's non-key fields appear with `_right` suffix and all types preserved

    JoinSpec:
      - Holds left, right, on (normalized to list[str]), within (or None), type

      - `_to_join_json() -> dict` for REGISTER JSON emission in Task 3 (shape: {"op": "join", "right": right._name, "on": [...], "within": "30m"|None, "type": "inner"|"left"})
