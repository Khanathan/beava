## The win

aggregate** over 60s on an 8-client workload. That's the perf gate
number Phase 56 SC-5 was trying to get but couldn't (the scenario
couldn't even boot; server livelocked in semaphore waits).

Peak sustained enrichment rate: **~117,000 enrichments/sec** during the

**~4 million× improvement in server-side processing rate.**

`beava_enrich_cross_shard_total`: absent (zero) — no cross-shard
traffic. `beava_enrich_missing_total`: absent — every lookup hit the
local replicated copy.

## What changed at the code level

- **Default behaviour** of `@bv.source_table(key=...)` is now
  replicated: writes fan out to all shards, enrichment reads are local.
- **`sharded=True`** opt-in preserves Phase 56 partitioned-by-key +
  cross-shard `ShardOp::ReadEntityAt` behaviour for large tables.
- **Write fanout** (`OP_UPSERT_TABLE_ROW` / `OP_DELETE_TABLE_ROW`)
  dispatches via `send_op_await_setok_all_shards` helper for replicated
  tables, single-shard route for `sharded=True`.
- **EnrichFromTable operator** forces `target_shard_idx ==
  input_shard_idx` for replicated source tables — same-shard fast path
  taken for every event; no semaphore wait on cross-shard reply.

  above. `WARMUP>0` needs a server-drain barrier between phases.
