# Phase 59.5: shared-source-tables — Context

**Gathered:** 2026-04-21
**Status:** Ready for planning
**Mode:** Manual (user-requested insertion; grey areas listed below)

<domain>
## Phase Boundary

Redesign `@bv.source_table` from **partitioned-and-cross-shard-read** to
**replicated-by-default (broadcast) with opt-in partitioning** for large
tables. The partitioned path keeps the Phase 56 `ShardOp::ReadEntityAt`
code path exactly as-is — that code is correct, it's just the wrong
default for the 95% case (small dimension tables).

Delivers both the **EnrichFromTable** (stream × table) and **Table × Table
join** code paths operating entirely on local state when both sides are
replicated. Cross-shard reads are eliminated for the default case. This
fixes the performance cliff discovered when profiling the Phase 56 cross-
shard scenario: **shard threads were spending 100% of wall time blocked
in `semaphore_wait_trap` waiting for cross-shard ReadEntityAt replies**,
producing 0.03 enrichments/sec against 2.16M EPS ingest.

**In scope:**
- New default-replicated path for `@bv.source_table`. Writes fan out to all
  shards (upsert + delete). Reads are local on every shard.
- Backward-compat opt-in: `@bv.source_table(key=..., sharded=True)` preserves
  today's Phase 56 behavior (partition by key, cross-shard ReadEntityAt).
- `OP_UPSERT_TABLE_ROW` / `OP_DELETE_TABLE_ROW` wire-level fanout when the
  target is a replicated source_table (vs. single-shard route for `sharded=True`).
- EnrichFromTable operator uses local-only state reads when the right-side
  table is replicated; falls back to `ShardOp::ReadEntityAt` when `sharded=True`.
- Table × Table join: when both inputs are replicated, the join runs
  entirely on local state per shard. When one or both are sharded, apply
  the Phase 56 ReadEntityAt path unchanged.
- Phase 57 retraction flow adapted: retractions on replicated tables fan
  out the same way writes do; each shard locally fans-out to its affected
  downstream aggregations.
- Tests: new replicated-path integration tests; existing Phase 56 cross-
  shard tests annotated with `sharded=True` so they keep covering the
  partitioned code.
- Perf gate re-run of the cross-shard enrichment scenario under the new
  default — this is the gate number that Phase 56 SC-5 was trying to get
  but couldn't.

**Out of scope (explicit):**
- Changes to `@bv.stream` routing or `StreamStreamJoin` — SSJ already
  colocates L+R via `hash(join.on) % N` and works correctly.
- Hot-key salting (Phase 60) — independent axis, paused; this phase
  unblocks it by fixing the cross-shard perf cliff first.
- Auto-detection of broadcast-vs-partitioned based on table size. Keep it
  explicit opt-in (`sharded=True`) in this phase; auto-classification is
  a future enhancement.
- Backward-compat with existing snapshots that have a source_table stored
  under Phase 56 partitioned layout. Acceptable to require re-seeding
  when upgrading, same as prior `v9 → v10` snapshot bumps.
- Binary wire-format changes to OP_UPSERT_TABLE_ROW — Phase 59 already
  established the binary format; we only change dispatch (fanout vs
  single-route), not the frame layout.
- Aggregation-on-source-table (still rejected at register time, same as
  Phase 55 D-B6).

</domain>

<problem_statement>
## What the Phase 56 profile showed

```
Thread_beava-shard-N (10,682 samples = 10.682 s wall):
  thread_start
    ...
    shard_event_loop
      process_shard_event
        push_with_cascade_on_shard
          read_entity_batch_at_shard       ← cross-shard EnrichFromTable read
            block_on(...)
              park()
                semaphore_wait_trap         ← blocked here 100% of time
```

All 10 shards spend 100% of wall time blocked on a semaphore waiting for
another shard to reply to a `ShardOp::ReadEntityAt`. Because every shard
is waiting on every other shard, the system livelocks. `/metrics` after
30 s with 8 clients pushing 2.16M EPS:

```
beava_enrich_intra_shard_total{Countries} = 1
beava_enrich_cross_shard_total{Countries} = 0
beava_enrich_missing_total{Countries}     = 1
```

One successful enrichment in 30 seconds. 2.16M EPS client ingest is
inbox-absorption, not pipeline throughput.

The synchronous-reply cross-shard read is the wrong architecture for
small dimension tables (Countries = 50 rows). The standard streaming-
systems design is **replicate small tables to every worker**; enrichment
reads are then local and cross-shard traffic is eliminated.

Kafka Streams calls this `GlobalKTable`. Flink calls it `BroadcastState`.
Materialize replicates small relations to every worker.

</problem_statement>

<decisions>
## Implementation Decisions

### Area A — SDK surface

**D-A1 (Default changes, no kwarg for the common case):**
```python
@bv.source_table(key="country_code")
class Countries:
    country_code: str
    name: str
```
With no `sharded=` kwarg, this is **replicated**. Writes fan out to all
shards. EnrichFromTable reads are local. This is the path 95% of users
want.

**D-A2 (Opt-in for huge tables):**
```python
@bv.source_table(key="country_code", sharded=True)
class BigCatalog:
    ...
```
With `sharded=True`, the table is partitioned by `hash(key) % N` (today's
Phase 56 behavior). Writes route to the owning shard; enrichment on a
non-owner shard triggers `ShardOp::ReadEntityAt`. Intended for tables too
large to replicate (tens of GB).

**D-A3 (naming — open question, flag for user):**
The `sharded=True` kwarg is a Beava-native term (matches existing shard
terminology) and is unambiguous for users who already understand the
partitioning model. Alternatives considered:
- `partitioned=True` — standard DB term, perhaps more obvious to non-
  Beava users but duplicates "partition" meaning with streams.
- `co_located=True` — describes the join behavior but not the table layout.
- `mode="sharded"` / `mode="replicated"` — explicit mode enum; more
  verbose but eliminates the default-is-hidden surprise.

**Recommendation: `sharded=True`** — matches the codebase's existing
shard vocabulary and is the shortest correct name. Surface this as a
grey area in discuss phase.

### Area B — Wire-level write fanout

**D-B1:** `OP_UPSERT_TABLE_ROW` / `OP_DELETE_TABLE_ROW` dispatch branches
on the target table's `sharded` flag (stored in raw_register_jsons, same
place as the current kind="source_table" marker).
- **Replicated (default):** send the same mutation to ALL shards. Each
  shard's event loop applies the upsert/delete locally under its own
  fjall partition. Ack is returned only after all N shards have applied
  (all-or-nothing — if any shard fails, emit a Phase 57 retraction on
  the ones that succeeded). `source_lsn` is the same on all shards for
  the same upsert.
- **Sharded (opt-in):** single-shard dispatch as today (hash(key) %
  shard_count), no change.

**D-B2 (replication model — synchronous vs quorum):** Synchronous all-
N ack keeps writes simple and matches the current "LSN echoes on ack"
contract. We do NOT introduce async replication / quorum acks in this
phase — that's a v2 concern. On a partial failure (one shard can't
apply), we fail the write and surface a retraction on successful shards.
Phase 57's cross-shard retraction primitives already cover this.

**D-B3 (HTTP batch paths):** `POST /table/{name}/batch` and
`/table/{name}/batch/delete` also fan out — same branching on the
sharded flag.

### Area C — Operator local-read path

**D-C1:** EnrichFromTable operator — when the right-side source_table is
replicated, read from the local shard's state store. No `ShardOp::ReadEntityAt`.
The operator already has a per-shard state reference; just route the
lookup to it when the replicated-flag is set.

**D-C2 (Table × Table join):** when both inputs are replicated, the join
runs entirely on local state per shard. Each shard has both sides fully
materialized; a local hash-join or merge-join emits results locally. When
one side is sharded and the other replicated, route the join to the
sharded side's owning shard (the sharded side's event triggers the
lookup into the local replicated state). When both are sharded, fall
back to Phase 56's `ShardOp::ReadEntityAt` for cross-shard lookups.

### Area D — Retraction fanout

**D-D1:** Phase 57 retractions for source-table deletes — currently the
delete lands on the owning shard, which fans out `ShardOp::RetractDownstream`
to every shard that has an enriched event in its cascade history. With
replicated source tables, every shard receives the delete simultaneously
(Area B fanout), so each shard can fan out retractions to its own local
downstream aggregations without any additional cross-shard hops.
The `retraction_depth` counter and 16-hop cap apply unchanged.

**D-D2 (ContribSet changes):** `ContribSet::SourceTableKeys` already
records which keys a downstream output depends on; this structure is
unchanged. What changes is WHERE the retraction originates — every
shard, not just the owning shard. The retraction is a no-op on shards
that had no matching downstream rows (which is the common case for a
single country_code lookup).

### Area E — Storage layout

**D-E1 (shared partition name):** Each shard stores its replicated
source_table under the same partition key it uses today (name-scoped
fjall partition). No storage-layout migration; existing tests that
upsert via `register_source_table` + `upsert_table_row` continue to
work — they just trigger local upserts everywhere instead of hashing.

**D-E2 (memory cost):** Replicated tables cost `size × N_shards` RAM/disk.
Users declaring a 10 GB source_table with `sharded=False` (default) would
see 10 × N memory use. This is the correctness-for-cost tradeoff; opt-in
`sharded=True` for tables too large to replicate. We should emit a
warning at upsert time when a replicated table exceeds a size threshold
(maybe 100 MB), suggesting the user consider `sharded=True`.

### Area F — Backward compatibility

**D-F1 (Phase 56 tests):** existing cross-shard tests like
`tests/cross_shard_enrich_from_table.rs` exercise the partitioned path.
After this phase, they must opt into `sharded=True` at their fixture's
`register_source_table` call, OR be duplicated so we cover both paths.
Recommendation: duplicate the tests so the new default (replicated) and
the opt-in (sharded) each have coverage.

**D-F2 (Phase 56 perf-smoke):** `tests/crossshard_enrich_perf_smoke.rs`
tests a p99 latency floor for cross-shard enrichment. Under the new
default this becomes an intra-shard p99 test (which should be much
faster). Keep a `sharded=True` variant of the test so the cross-shard
code path stays covered.

**D-F3 (snapshot compat):** Existing snapshots with source_table data
stored under one shard need to be re-seeded after upgrade. Acceptable
per Phase 55 D-A5 precedent — source_table data is CDC-replayable by
upstream. Document in UPGRADE.md / CHANGELOG.

### Area G — Perf gate

**D-G1 (the gate number Phase 56 SC-5 wanted):** re-run the cross-shard
enrichment scenario under the new default. Previous results were:
- Phase 56 baseline: 0.03 enrichments / 30 s (server-side; livelocked)
- Phase 56 cross-shard scenario: gate DNE (couldn't boot, 56-NEXT #6)
- Phase 56 default fraud pipeline (no cross-shard): 1,195,914 EPS

**Expected after this phase:**
- Replicated default: ≥ Phase 55 baseline × 85% = 1,059,261 EPS (the
  same floor Phase 56 SC-5 was trying to hit). Small-table replicated
  enrichment shouldn't be slower than a non-enriched pipeline.
- Sharded opt-in: same as Phase 56 today (correctness unchanged).

</decisions>

<success_criteria>
## Success Criteria

**SC-1 (default replicated upsert + EnrichFromTable round-trip):**
Registering `@bv.source_table(key="country_code")` (no `sharded=` kwarg)
fans out upserts to all shards. A Txns event on shard X enriches against
the local Countries row from shard X's state (zero cross-shard traffic).
Verified by a new integration test `tests/replicated_enrich_from_table.rs`.

**SC-2 (sharded opt-in preserves today's behavior):**
`@bv.source_table(key="country_code", sharded=True)` + Txns routing by
hash(user_id) triggers `ShardOp::ReadEntityAt` cross-shard reads exactly
as today. `tests/cross_shard_enrich_from_table.rs` continues to pass
with the explicit `sharded=True` flag.

**SC-3 (Table × Table join — both replicated):**
`Countries.join(Currencies, on=["currency_code"])` where both sides are
replicated produces correct join results. Join state and reads are
local on every shard. New test `tests/replicated_table_table_join.rs`.

**SC-4 (retraction through replicated source_table):**
Deleting a row from a replicated Countries table retracts downstream
EnrichFromTable outputs on ALL shards (not just the owning-shard).
`tests/replicated_source_table_delete_retraction.rs` verifies.

**SC-5 (N=1 ↔ N=8 parity for replicated tables):**
`tests/sharding_parity` extension: same Countries + Txns workload on
N=1 and N=8 produces byte-identical enriched output (modulo per-shard
ordering, which must still be per-key-deterministic).

**SC-6 (cross-shard perf gate — the real number):**
Bench `BEAVA_ENRICH_CROSSSHARD_SCENARIO=1` under new default achieves
≥ 1,059,261 EPS sustained over 60s on the Phase 56 benchmark harness.
`/metrics` shows cross-shard enrichment counters growing at a rate
proportional to ingest (not flatlining at 0).

**SC-7 (write fanout correctness):**
`OP_UPSERT_TABLE_ROW` with a replicated table writes to all N shards'
state before acking. Partial-failure path surfaces a clear error and
emits Phase 57 retractions on the shards that DID apply. Unit-tested
via a fault-injection test.

</success_criteria>

<grey_areas>
## Grey areas to resolve in discuss-phase

1. **API naming:** `sharded=True` vs `partitioned=True` vs `mode="..."`.
   Recommendation: `sharded=True` (Beava-native terminology).

2. **Write fanout — sync-all vs async-replicate:** this phase lands
   sync-all-N for simplicity. Is that acceptable for 32-shard cluster
   writes? Async-replicate is a v2 knob if needed.

3. **Size-based warning threshold:** emit a WARN when replicated
   `OP_UPSERT_TABLE_ROW` takes the stream past N × 100 MB. Is 100 MB
   the right per-replica ceiling, or should we use cluster-total GB?

4. **Phase 56 test migration:** duplicate existing tests to cover both
   paths, or add `sharded=True` everywhere and write fresh tests for
   the default? Recommendation: duplicate — explicit coverage of both
   paths, no implicit regression risk.

5. **Snapshot migration:** require re-seed on upgrade, or write a
   one-shot migration that broadcasts existing owning-shard data to
   other shards? Recommendation: require re-seed (source_table data is
   CDC-replayable by upstream).

</grey_areas>
