Output: A Cargo.toml with `client`/`server` features, a lib.rs that cfg-gates `pub mod server`, a main.rs that cfg-gates the whole server binary, and a green `cargo test` on defaults.
</objective>

Per CONTEXT.md decisions:
- **D-A1**: Feature-flag on the main crate. `server` is default. `client` strips server-only code via `#[cfg(feature = "server")]`.
- **Specifics §Feature propagation**: Any server-only dependency must move behind `optional = true` + `features = ["server" = ["dep:..."]]`. However, for v0-demo simplicity, this plan does NOT make deps optional yet — it only gates module visibility. Dep-optionality is a follow-on optimization if client build size matters; gating modules is sufficient to prove the split compiles. (Tokio/axum still link in; they just don't get called. Trade-off noted; aligns with "easiest for v0 and demo".)

Decision traceability:
- Per D-A1: `client`/`server` features added to Cargo.toml; `default = ["server"]`.
- Per CONTEXT §code_context: `src/server/**` gated at module level via lib.rs `pub mod server` cfg.
