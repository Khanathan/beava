<decisions>
## Implementation Decisions

### Crate location
- **D-01:** New `crates/beava-bench/` workspace member with a single CLI binary `beava-bench` (subcommand `throughput`). Keeps perf tooling out of the server crate while reusing `beava-server` (testing feature) and the public HTTP/TCP wire.
- **D-02:** Pipeline configs ship as JSON files in `crates/beava-bench/configs/` (small.json, medium.json, large.json). One JSON file per shape — same shape that the server's `/register` accepts.

### Workload runner
- **D-03:** 60s wall-time time-bounded run. Push events at saturation across N parallel client tasks (default N = `min(num_cpus, 8)`). Saturation = each task pushes in a tight loop until `Instant::now() >= deadline`. Capture every push latency in a per-task `hdrhistogram::Histogram` (1 µs precision, 60s max), then merge.
- **D-04:** Every 1s during the run, do a batch `/get` of 100 random keys × all features; record P99 batch-get latency in a separate histogram.
- **D-05:** Peak RSS via `ps -o rss= -p <pid>` polled every 500ms (cross-platform — macOS `task_info` and Linux `/proc/self/status` both differ; shelling out to `ps` keeps the harness OS-agnostic).
- **D-06:** Single-process measurement (harness spawns the server in-process via `TestServer` for isolation; binary mode also supported via `--server-url` for measuring an external `beava` binary). Default = in-process; this gives stable, reproducible numbers without network jitter from container/launchd noise.

### Two transports
- **D-07:** Run each pipeline size against HTTP and TCP separately, in distinct subprocess invocations (or sequential runs within the same harness invocation). 6 rows per ledger entry (3 sizes × 2 transports).
- **D-08:** HTTP uses `reqwest` (already a dev-dep), TCP uses the `TcpClient` helper from `beava-server::testing` (already exists). Both keep persistent connections (HTTP keep-alive, TCP single connection per task).

### Ledger
- **D-09:** `.planning/throughput-baselines.md` mirrors `.planning/perf-baselines.md` format: top-level explanation, `## hw-class: ...` sections, append-only markdown table rows.
- **D-10:** Columns: Phase | Date | Pipeline | Transport | Sustained EPS | P50 push (µs) | P95 push (µs) | P99 push (µs) | P99 batch-get (µs) | Peak RSS (MB) | Commit SHA | Notes

### Per-phase regression contract (CLAUDE.md update)
- **D-11:** Append a new bullet under §Performance Discipline: "Every phase from 8 onward MUST include a 'throughput run' task that re-runs the harness and appends a row to `.planning/throughput-baselines.md`. Plan-checker contract: a Phase 8+ plan without a throughput-run task is rejected. Same 10% warn / 25% block thresholds apply, measured against the simple-fraud (small) shape."

### TestServer flake — likely root cause + fix
- **D-12:** Hypothesis to verify in plan 01: the flake is NOT registry-state propagation (Arc is shared correctly) — it is **WAL-dir collision** between two `TestServer::default()` spawns in the same `#[tokio::test]`. Default `wal_dir` uses `format!("beava-test-wal-{pid}-{uniq}")` — `uniq_id()` is a per-process counter, so different. BUT the two TestServers share `(pid, snapshot_dir)` — the second spawn might recover the first's snapshot/WAL on bind. Fix: ensure tests that need two spawns use explicit per-spawn `wal_dir` + `snapshot_dir` `tempfile::tempdir()` (this is what `phase6_push.rs` does — and it passes). Most likely fix is documenting this in TestServer docs + adding a unit test.

### Phase 7 deferred benches
- **D-13:** Add three benches under `crates/beava-persistence/benches/snapshot_recovery_bench.rs`:
  - `snapshot/serialize_state_1k_features` — bincode encode time per snapshot at 1000 features
  - `snapshot/atomic_write_default_fsync` — full write+fsync+rename round-trip
  - `recovery/replay_wal_10k_records` — replay throughput (events/sec) past a snapshot LSN
- **D-14:** Append rows to `.planning/perf-baselines.md` under a new `### Phase 7 — snapshot + recovery` section.

### Phase 7 deferred restart-cycle smokes
- **D-15:** Once TestServer flake fix lands, write `crates/beava-server/tests/phase7_restart_cycle.rs`:
  - `sc1_snapshot_then_restart_reproduces_state` — push 1000 events → force_snapshot → drop server → respawn with same dirs → assert feature values match
  - `sc2_crash_mid_snapshot_preserves_committed_events` — leverage the existing `BEAVA_CRASH_AT` injection points (per Phase 7 SUMMARY)
  - `sc4_schema_evolution_survives_restart` — register A → push → snapshot → register B → restart → assert both events queryable

### Claude's Discretion
- Whether the harness CLI is invoked as `cargo bench --bench throughput` (criterion-style) or `cargo run -p beava-bench --release -- throughput --pipeline small --transport http`. Choosing the latter — criterion is for microbenches, not 60s sustained workloads.
- HDR histogram precision (defaulting to 1µs / 60s max).
- Number of parallel client tasks (defaulting to min(8, num_cpus)).
- Whether to capture both per-1s rolling EPS and overall EPS (planning to capture both, but reporting overall only in the ledger to keep rows readable).

## Existing Code Insights

### Reusable Assets
- `beava_server::testing::TestServer` + `TestServerBuilder` — already supports HTTP + TCP, `wal_dir`/`snapshot_dir` overrides, `force_snapshot_now`. The harness wraps these.
- `beava_server::testing::TcpClient` — already implements `register_json`, `send_raw`, `read_n_frames`. Need to add `/push` and `/get` equivalents (or push via raw frames using OP_PUSH / OP_GET — check the wire spec).
- `beava-core` exports `Registry`, `wire::*` (frame codec). 
- Phase 6/7 WAL + snapshot path runs unchanged — harness just exercises it end-to-end.
- `hdrhistogram` crate is rust-standard for percentile capture.

### Established Patterns
- All integration tests use `tempfile::tempdir()` for per-test isolation — the harness should do the same so a stale WAL/snapshot doesn't bleed between runs.
- Conventional commits: `test(07.5-NN): subject` → `feat(07.5-NN): subject` → `chore(07.5-NN): subject` per CLAUDE.md §TDD Discipline.
- macOS hw-class fsync is ~7.4 ms P50 (Phase 6 baseline). The first throughput numbers will reflect this — document but do NOT mark as blocker.

### Integration Points
- New crate `crates/beava-bench/` joins workspace `Cargo.toml [workspace.members]`.
- Ledger `.planning/throughput-baselines.md` — new file, mirrors `perf-baselines.md`.

<specifics>
## Specific Ideas

- The 6-row first baseline (small/medium/large × HTTP/TCP) on Apple M4 is the only required hw-class for this phase. Linux baseline lands when CI is wired up (Phase 13 area).
- Pipeline shapes use only Phase 5 operators: count, sum, avg, min, max, variance, stddev, ratio. No decay, no sketches, no geo.
- Each pipeline has 1 entity type (e.g. `user_id`), 1 window (e.g. `1m`), N features for N=1/5/15.
- TCP should win EPS over HTTP (less framing overhead) — if HTTP equals or beats TCP, that's a surprise to flag.

</specifics>
