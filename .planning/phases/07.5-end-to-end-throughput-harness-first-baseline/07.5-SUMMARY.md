### Folded scope landed

**A. TestServer router-state flake — DIAGNOSED + FIXED.** Phase 7's
"axum router-state propagation glitch" hypothesis turned out to be a
misdiagnosis. The actual blocker was a silent WAL-codec round-trip bug:
`RegistryBumpPayload::decode` used bincode 1.x, but PayloadNode contains
`serde_json::Value` fields (in `AggSpec.params` and `OpNode::Fillna.defaults`)
which bincode rejects with `DeserializeAnyNotSupported`. `recovery::
replay_wal_from_lsn` was silently swallowing the decode error via
`tracing::warn!`, making the bug invisible at the integration level for
an entire phase. Two-line fix:

1. `RegistryBumpPayload::{encode,decode}` switched from `bincode` to
   `serde_json` (RegistryBump is per-`/register` cold path; size delta
   irrelevant).
2. `recovery::replay_wal_from_lsn` now returns a hard error on RegistryBump
   decode/apply failure instead of warn-and-skip — the apply-AFTER-fsync
   invariant says: durable means must-replay.

**B. Snapshot/recovery criterion benches — LANDED.** Three benches in
`crates/beava-persistence/benches/phase7_snapshot_recovery.rs`:
- `snapshot/serialize_state_1k_features`: 10.68 µs
- `snapshot/atomic_write_default_fsync`: 8.45 ms
- `recovery/replay_wal_10k_records`: 675.93 µs

Baseline rows appended to `.planning/perf-baselines.md`. Closes the
Phase 7 perf-discipline gate that Plan 07-04 deferred.

**C. Restart-cycle integration tests — LANDED.** Three tests in
`crates/beava-server/tests/phase7_restart_cycle.rs`:
- `sc1_snapshot_then_restart_reproduces_state` (PASS)
- `sc4_schema_evolution_survives_restart` (PASS)
- `snapshot_then_schema_evolution_then_restart` (PASS, bonus)

These flip Phase 7 SC1 + SC4 from PARTIAL to PASS. SC2 (crash mid-snapshot)
was scoped out — needs a subprocess crash-probe binary; the disk-level
atomic-rename invariant is already proven by `snapshot_roundtrip.rs` unit

### First-baseline numbers (Apple-M4 / Darwin-24.3.0 / 10 cores)

| Pipeline | Transport | Sustained EPS | Push P99 | Get P99 | Peak RSS |
|---|---|---:|---:|---:|---:|
| small  | http | 990  | 12.0 ms | 5.3 ms | 30 MB |
| medium | http | 1031 | 11.3 ms | 5.7 ms | 48 MB |
| large  | http | 1007 | 11.3 ms | 7.1 ms | 74 MB |
| small  | tcp  | n/a | — | — | — |
| medium | tcp  | n/a | — | — | — |
| large  | tcp  | n/a | — | — | — |

EPS is fsync-bottlenecked at ~1k/sec across all pipeline sizes — macOS
`F_FULLSYNC` hw-class limit (~7.4ms P50 per Phase 6 baseline). Per-push
aggregation work for 1 vs 5 vs 15 ops is sub-fsync (~225 ns vs 7.4 ms),
which is why the three sizes converge. RSS scales linearly with feature
count.

TCP rows record `n/a (deferred)` because `OP_PUSH` (`0x0010`) on the TCP
wire returns `OP_ERROR_RESPONSE { code: "op_not_implemented" }` — the
opcode is reserved at Phase 2.5 but the handler is owned by a future
phase. The harness CLI is ready; the server-side handler is not.
