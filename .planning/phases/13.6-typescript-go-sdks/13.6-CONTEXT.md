# Phase 13.6: TypeScript + Go SDKs (communicate-only) — Context

**Gathered:** 2026-05-03 (interactive discuss-phase; 4 single-question turns; restarted after major scope change)

<domain>
## Phase Boundary

**REVISED 2026-05-03 per user directive:** TS + Go SDKs are **wire-thin communicate-only clients** — NO pipeline DSL. Pipeline authoring (decorators, expressions, agg helpers, register-time validation) lives in Python (Phase 13.5) only.

**API surface in TS + Go (mirrors Python's `App` only, ~500-800 LOC each — much smaller than original 1800 LOC scope):**
- `App` constructor + URL-scheme dispatch (`http://` / `tcp://` / no-URL embed)
- `app.register(jsonDescriptors)` — accepts pre-compiled JSON (Python authored these via DSL; or hand-written)
- `app.push(eventName, dict)` — fire-and-forget (acks=1)
- `app.pushSync(eventName, dict)` — acks=all (durable; for fsync-validation use cases)
- `app.get(table, entityId)` per-entity + `app.get(table)` global form (TS overloaded signatures; Go separate `GetGlobal` per idiom)
- `app.batchGet([(table, key), ...])`
- `app.reset()` — gated on test_mode per Phase 13.4 D-03
- `app.ping()`
- `app.close()`

**NOT in TS + Go SDKs (Python-only):**
- `@bv.event` / `@bv.table` decorators
- `bv.col` operator overloading + expression DSL
- `bv.lit` (only used inside expressions)
- 53 op helpers (`bv.count`, `bv.sum`, ...)
- `events.filter().group_by().agg()` chained DSL
- `bv.demo("adtech"|"fraud"|"ecommerce")` loader (Python-only; demos are bundled with Python pip install per 13.5 D-02)

**KEEP in TS + Go SDKs (wire-level, not DSL):**
- `force=True` / `dry_run=True` flags as kwargs to `app.register(json, {force: true, dry_run: false})` — these are wire-level concerns; TS/Go users pass them through
- `bv.App(test_mode=True)` constructor kwarg per Phase 13.5 D-05 (mirrors Python; passes to embed binary in embed mode; ignored with warning in network mode)

</domain>

<decisions>
## Implementation Decisions

### D-01 — TS bundle: ESM-only (USER-LOCKED)

- ESM-only output; Node 18+ LTS baseline (current as of 2024)
- Single bundle in `dist/` (no CJS dual-publish)
- ~200 LOC less plumbing than ESM+CJS dual
- `package.json` `"type": "module"` + `"exports": { ".": "./dist/index.js" }`
- Build: `tsc --module esnext --target es2022 --strict`
- **Why:** modern; cleaner; smaller dist; Node 18 LTS is current baseline. Excludes Node 16 + older bundlers — acceptable trade for v0.

### D-02 — Go module: `github.com/beava-dev/beava` monorepo with `sdk/go/` subdir (USER-LOCKED)

- GitHub org: **`beava-dev`** (NOT `beava-io` as previously assumed in ROADMAP.md / docs/sdk-api/go.md — needs updating in 13.6 closure per D-04 doc patches)
- Repo layout: **monorepo** for now — single `github.com/beava-dev/beava` repo houses everything (Rust server, Python SDK, TS SDK, Go SDK, docs)
- Go module path: `github.com/beava-dev/beava/sdk/go`
- TS package source: `github.com/beava-dev/beava/sdk/typescript/` (published to npm as `@beava/sdk`)
- npm publish: from `sdk/typescript/` subdir
- Repo rename: current codename `tally` → public `beava` may happen at v0 ship (Phase 13.8 territory; not 13.6)
- **Why monorepo for now:** simpler release coordination; keep options open for later split if needed (`go get` redirect path is deferred until a real signal that split is needed)

### D-03 — Cross-SDK conformance test = single Python orchestrator (USER-LOCKED)

- Lives at `python/tests/conformance/test_cross_sdk.py`
- Drives all 3 SDKs against the SAME scenario via subprocess:
  - Python: in-process `import beava as bv; ...`
  - TS: `subprocess.run(["node", "tests/conformance/run.ts"])` (or `npx tsx`)
  - Go: `subprocess.run(["go", "test", "./tests/conformance/..."])`
- Single source-of-truth scenario file: `python/tests/conformance/scenario.json` (input events + expected outputs per language)
- Asserts identical outputs across all 3 SDKs (catches wire drift immediately)
- ~200 LOC for orchestrator + ~50 LOC adapter per SDK (TS/Go)
- Phase 13.6 closure ships it; runs in CI on every PR
- **Why single orchestrator:** automated cross-SDK assertion catches drift; manual review of per-SDK fixtures (Option B) misses subtle drift. Orchestrator approach is the right safety net.
