## TL;DR

**The criterion bench measures a registry that is structurally different from fraud-team. Only ~14 features × 0 windowed ops vs fraud-team's 88 features × 60 windowed ops.** The 38× gap is mostly ratio-of-work (~6×, 88÷14) plus a per-call cost-class shift (~6×, ~50 ns hot non-windowed → ~300 ns windowed). The dominant cost driver is **WindowedOp dispatch overhead**: every windowed feature bypasses Plan 19.2-01's pre-extraction and re-pays `row.get()` + `AggOp::update_with_row` + bucket-position scan + per-bucket inner `update`. Hypothesis **(A) plus (B)** is confirmed; (C) and (D) are minor.

## 2. Root cause: hypothesis (A) + (B), with quantitative evidence

| Hyp | Statement | Verdict | Evidence |
|---:|---|---|---|
| **A** | WindowedOp bucket-maintenance dominates | **CONFIRMED — primary** | 60/88 features windowed; top-4 cost drivers all windowed; per-call cost on windowed path is 5–10× the non-windowed equivalent (CountDistinct 435 ns windowed vs ~50 ns non-windowed). |
| **B** | Per-feature row.get cost rebuilt for windowed path | **CONFIRMED — primary** | `agg_op.rs:813-826` shows `Windowed(w) => w.update_with_row(row, ..., where_expr)` which forwards `row` not `extracted`. | **D** | Sketch ops (HLL/UDDSketch/Entropy) are Tier 2/3 and dominate | partially true but secondary | Sketches *are* in the top-4, but the cost dominator is the windowed *wrapping* not the sketch arithmetic itself. ### Why criterion bench misses this

The criterion bench at `crates/beava-core/benches/apply_path_bench.rs:73-375` builds a synthetic 14-agg registry across 3 clusters. **Zero of the 14 use `window_ms`.** The closest to a windowed op is `txn_amount_ewma` with `half_life_ms: Some(3_600_000)` — but that uses `EwmaState` which is decay-based (no bucket ring). The Percentile op (`txn_amount_p50`, line 110-126) has `window_ms: None`, so it goes through `update_pre` directly — UDDSketch insert path with no WindowedOp wrapping.

This means the criterion's `apply_path/warm_key/14_aggs` 362.71 ns measures **the pre-extraction fast path**. fraud-team's apply-thread budget measures **the pre-extraction fast path (28 features) + the WindowedOp slow path (60 features)**. The two benches are not measuring the same workload; they happen to share a name ("14 aggs"), but only one of them resembles fraud-team.

### The wrapping cost decomposed

Per-call cost on the windowed path (typical AggKind=CountDistinct, 435 ns):
1. **Outer dispatch** — `update_with_extracted` match arm `AggOp::Windowed(w) => w.update_with_row(...)` — ~10 ns
2. **`agg_op.rs:801` evaluate where_expr** — None branch, ~5 ns
3. **`WindowedOp::update_with_row` epoch compute** — `event_time_ms.div_euclid(bucket_ms)`, ~5 ns
4. **`position_for_epoch` linear scan** — SmallVec scan over 1–4 active buckets, ~5–15 ns
5. **Per-bucket `AggOp::update_with_row(row, ..., where_expr)`** — ~10 ns dispatch + re-evaluate where (None ~5 ns)
6. **Inner state `update(row, fname, …)`** — does **`row.get(fname)`** (linear scan over **19-field Txn row**) → ~30–50 ns just for the find
7. **Hash + add** for HLL/CountDistinct — ~20–30 ns (hash_value_for_hll uses FxHasher per Plan 19.2-02)
8. **Mode-promotion check** for HLL hybrid — ~10 ns
9. **Lazy bucket-allocate path** taken on first event in new epoch (~80–200 ns amortized; rare)

Items (3)+(4)+(6) — ~50–70 ns — are the WindowedOp tax that the criterion bench never pays. Items (1)+(5) — ~25 ns — are the double-dispatch tax (outer + inner `update_with_row`). Item (7) is the one actually measured by the criterion warm-key.

For windowed Count (181 ns/call), the same fixed 100–120 ns wrapping tax is paid for an "increment a counter" inner op that should cost <10 ns. **Count windowed is 18× slower than Count non-windowed** purely due to the windowed dispatch tax — that's where the 1,996 ns/event from 11 Count calls comes from.

## 4. Phase 19.3 sub-goal proposals

### 19.3-A — Plumb `extracted` + `field_idx` into WindowedOp (highest priority)

**Idea:** Add `WindowedOp::update_at(extracted, field_idx, lat_idx, lon_idx, event_time_ms, where_matched)` that mirrors the non-windowed pre-extracted path. Per-bucket inner op dispatches via `AggOp::update_with_extracted` (already exists) instead of `AggOp::update_with_row`. Eliminates the inner `row.get(fname)` and the inner `evaluate_where_predicate` re-evaluation.

**Predicted lift:** 60 windowed calls × (50 ns row.get + 5 ns where re-eval + 10 ns dispatch) ≈ **3,900 ns/event saved**.
- 14,059 ns → ~10,150 ns agg stage
- 70k EPS → ~95k EPS estimated end-to-end (apply thread is the bottleneck, so per-event saving translates ~1:1)
- Closes ~50% of the gap to Phase 19.2's 100k EPS PASS target.

**Files:** `crates/beava-core/src/agg_windowed.rs:191-211` (new fast-path method), `crates/beava-core/src/agg_op.rs:867` (Windowed arm dispatches new method), `crates/beava-core/src/agg_apply.rs:225-235` (pass through).

### 19.3-B — Specialize windowed Count/Sum (next highest)

**Idea:** Count and Sum are the most-called windowed kinds (11 + 3 = 14 calls/event in fraud-team). Their inner state update is trivial (`n += 1` / `total += v`). Bypass the full `AggOp::update_with_row → AggOp::update → CountState::update(row, …)` chain by inlining: `WindowedOp::update_with_row` matches on `inner_kind` for Count/Sum and writes to the inner state's `n` / `(total, n)` directly. Saves the ~3-level dispatch tax for the highest-frequency kinds.

**Predicted lift:** 14 calls × ~80 ns dispatch elimination ≈ **1,100 ns/event saved**. (Stacks with 19.3-A; 19.3-A's saving for Count is ~50 ns/call from removing the row.get; this saves the *remaining* dispatch chain.)

**Files:** `crates/beava-core/src/agg_windowed.rs:160-211` (specialized arms in `update`/`update_with_row`).

### 19.3-C — Hoist `ExtractedFields` build above the descriptor loop

**Idea:** Currently `extracted` is rebuilt per-desc at `agg_apply.rs:201-205`. Across 5 descs × ~6 distinct fields × ~25 ns/find = ~750 ns wasted on overlapping field reads. Hoist a single per-event ExtractedFields keyed by `(source_event_schema, field)` — registry knows the union of all fields any agg on this source needs, so the apply loop builds one ExtractedFields per event and indexes it via per-agg `field_idx` arrays.

**Predicted lift:** ~600 ns/event saved (eliminates redundant row.get across overlapping cluster fields).
- `entity_key` + ExtractedFields rebuild together = 264 + 1,293 + per-feature dispatch ≈ 2,500 ns/event currently in the shared-entity loop. Down to ~1,800.

**Files:** `crates/beava-core/src/registry.rs` (precompute per-source apply_field_names union; already exists as `EventDescriptor.apply_field_names` per agg_apply.rs context — just unused), `crates/beava-core/src/agg_apply.rs:201-205` (build one extracted per event).

## Closing note for Phase 19.3 planners

Phase 19.2's PASS-WITH-DEFICIT verdict was correct on the apply-loop optimizations themselves — D-01 through D-04b are all real wins on the path they target. The unstated assumption was that the *fraud-team workload's per-event apply cost is dominated by that path*. This investigation shows it isn't: **68% of fraud-team's per-event update calls are windowed and bypass the entire D-01..D-04b chain.** Phase 19.3's first sub-goal (19.3-A above) extends D-01's pre-extraction across the WindowedOp boundary — a single architectural change that unblocks the fraud-team 100k EPS target with the same conceptual model Phase 19.2 already proved out for non-windowed ops.

The criterion bench should be amended for Phase 19.3: add a `apply_path/warm_key/14_aggs_windowed` group whose registry has the same 14 features wrapped in `WindowedOp(window_ms = 24h)`. That bench should currently show ~5 µs (the live trace's per-feature × 14 ratio); after 19.3-A it should drop to ~1 µs. Without this bench, Phase 19.3's sub-goals will repeat Phase 19.2's measurement gap — a microbench that improves while the live workload stays flat.
