## Phase Boundary

Drop fraud-team K=10k zipfian apply-path from 13.4 µs → 6-8 µs/event (77k → ~125-150k EPS) via 7 sub-goals: wrapping reduction (field pre-extraction + hasher optimization), apply-loop refactor (EntityKey cluster + single-u64 fast path), op-specific tuning (UDDSketch BTreeMap → sorted Vec, EventTypeMix allowlist + str_from_row Cow), op-removal cleanup (unique_cells + geo_entropy out of catalogue; bv.entropy gets max_categories cap), and cost-class observability (catalogue docs + feature-gated debug endpoint). Stacks on Phase 19.1's WindowedOp lazy-bucket + GeoSpread Welford lifts.

- No same-key sketch batching (preserves Redis-shaped read-after-write) — `project_no_same_key_batching`
- fraud-team K=10k zipfian is the canonical complex-pipeline tuning bench — `project_fraud_team_primary_bench`
- Devex-first (cost-class catalogue is observability, NOT API split) — `project_v2_devex_first`
- Cross-event arrival-order processing for same `entity_key` preserved in all refactors

<decisions>
## Implementation Decisions

### Wrapping reduction (sub-goals 1+2)

- **D-01:** Field pre-extraction at apply-loop entry uses an **indexed array** (`Vec<&Value>` with field-idx-into-row resolved at register-time). Apply-loop entry does ONE pass over the row's `SmallVec<[(CompactString, Value); 8]>`; aggs hold a `u8` field_idx and do `extracted[idx]` (~5 ns vs ~30-60 ns for `row.get(field_name)`). Register-time validates that all field names referenced by aggs exist in the source schema; missing-field errors reject the registration.

- **D-02a:** AHasher initialization moves from per-call (`ahash::AHasher::default()` in the hot path, which reads thread-local random seed) to **process-static**, initialized at registry-init time via `OnceLock` (same pattern as Phase 18 env-var caching). Saves ~10-20 ns per hash op (Bloom, Entropy, EventTypeMix). Single seed per process; HashDoS resistance preserved at the seed-randomization-per-process level.

- **D-02b:** HLL ops switch from AHasher to **FxHasher** (`rustc-hash`). HLL's internal `mix64` post-processes the input hash for distribution, which repairs FxHasher's weaker statistical properties — this is the standard HLL recommendation in Heule et al. 2013. FxHasher is ~3-5 ns vs AHasher's ~30-50 ns for short strings. ~9 HLL ops on fraud-team × ~30 ns saved each. HashDoS isn't a concern — fraud workloads are operator-controlled, not adversarial input.

### EntityKey cluster + single-u64 fast path (sub-goal 3)

- **D-03:** EntityKey uses **Approach C (Hybrid)**: `enum EntityKeyShape { SingleU64(u64), SingleStr(u64), Multi(SmallVec<...>) }` with two storage maps per AggStateTable. Numeric single-keys (u64/i64/f64-bits) → SingleU64 (zero collision; bit-cast is bijective for non-NaN floats; reject NaN keys at register-time). String single-keys → SingleStr (FxHash of string + stored original; ~10⁻¹⁹ collision at 10M keys). Compound keys → Multi (SmallVec; zero collision). Best coverage: numeric AND string keys both get the fast path.

- **D-04:** Aggs cluster by `group_keys` signature at register-time using a **split-by-agg_id** storage shape: shared `EntityKey::from_row` + shared state-table lookup (one per unique signature), but each clustered agg keeps its own `Vec<AggOp>` in its existing `AggStateTable`. Update time: walk member agg_ids, do `state_tables[agg_id].update(entity_key, ...)`. Simpler than shared-Vec-with-indirection; preserves existing `AggStateTable` structure. fraud-team has 14 aggs across ~3 unique key signatures → ~308 ns lift from cluster dedup alone (separate from the EntityKey single-u64 fast path).

### Op-specific tuning (sub-goals 4 + 5)

- **D-04a (mechanical, no gray area):** UDDSketch `BTreeMap<i32, u64>` → flat sorted `Vec<(i32, u64)>` with binary-search insert. Same α=0.01 accuracy contract, same retraction support, same 2,048-bucket cap. Per-call algorithm floor: 130 ns → ~75 ns. Stacked with wrapping fixes: 963 ns → ~80 ns (~12× speedup).

- **D-04b (mechanical, no gray area):** EventTypeMix two-part fix:
  - Swap `Vec<String>` allowlist for `AHashSet<String>` at `EventTypeMixState::new` (`agg_buffer.rs:312-314`); O(allowed_len) per check → O(1).
  - `str_from_row` and `value_to_key_string` (`agg_state.rs:830-843`) return `Cow<'_, str>` instead of allocating new `String` per `Value::Str(Arc<str>)`; allocation only for derived strings.
  - Combined: 1,127 ns → ~50-100 ns (~10-20× speedup).

### Op removal + state cap (sub-goal 6 — reframed)

- **D-05:** **Remove `bv.unique_cells` and `bv.geo_entropy` from the operator catalogue.** v0 isn't shipped publicly so no migration cost. Users who need unbounded-distinct geo cells use `bv.count_distinct(quadkey(lat, lon))` (HLL via existing `count_distinct`, bounded memory, ±1.6% error). Users who need geo-cell entropy use `bv.entropy(quadkey(lat, lon))` (generic entropy op + max_categories cap, see D-05a).

  **Side effects (cleanup tasks for plan-phase):**
  - Operator catalogue: 55 → 53 ops
  - Phase 11 geo family: 6 ops → 4 ops (geo_velocity, geo_distance, geo_spread, distance_from_home retained)
  - Snapshot format cleanup: drop `UniqueCellsState` + `GeoEntropyState` variants from serialization enums (no on-disk data exists yet — pre-launch)
  - Tests: remove unique_cells/geo_entropy fixtures + Phase 11 microbenches (`crates/beava-core/benches/agg_geo_bench.rs` if separate)
  - Docs: don't add to `docs/operators/` (when that exists post Phase 20); add `count_distinct(quadkey)` + `entropy(quadkey)` recipe pages instead
  - REQUIREMENTS.md operator catalogue list update
  - PROJECT.md geo-family enumeration update
  - STATE.md operator-catalogue line update (currently lists 55 ops in §"Operator catalogue shipped")

- **D-05a:** Apply **max_categories cap + drop-new + cap-hit metric** to `bv.entropy` (the generic op, since unique_cells/geo_entropy are removed). When the BTreeMap hits `max_categories`, ignore novel categories (existing ones still increment). Add Prometheus counter `beava_entropy_categories_capped_total{op,registry_version}`. Default `max_categories=1024` if not specified at register-time. Mirrors existing `EventTypeMix.max_categories` pattern.

### Cost-class observability (sub-goal 7)

- **D-06:** Cost-class metadata lives in a **hand-maintained markdown table** at `docs/operators/cost-class.md`. Three tiers: Tier 1 (≤30 ns/call), Tier 2 (30-100 ns/call), Tier 3 (100-300 ns/call). Source-of-truth is the markdown; review-time discipline keeps it in sync with the audit data. No proc-macro / build-script complexity. Authoritative tier numbers from `.planning/research/operator-update-uniformity-audit.md`.

- **D-07:** `/debug/op-cost` HTTP endpoint **feature-gated behind `BEAVA_DEV_ENDPOINTS=1`** env var. Endpoint only registered if env var is set at process start; production deployments leak no internal latency telemetry by default. Returns the latest TRACE_AGG per_kind output as JSON. Matches Phase 15's pattern for dev/PIT-debug endpoints.

<specifics>
## Specific Ideas

- **Numeric collision floor for SingleU64** (D-03): u64-bit-cast of f64 means two different f64 keys with the same u64 bits is impossible (bit-cast is bijective for non-NaN). NaN keys are an edge case — reject at register-time when `group_keys` includes a column with float dtype, OR reject at push-time when a NaN appears, OR canonicalize to a sentinel u64. Defer exact policy to plan-phase but reject is the v0 default.
- **FxHasher choice rationale** (D-02b): FxHasher is the same hasher rustc uses internally (`rustc-hash`); battle-tested in compiler hot paths. `mix64` post-processing is what HLL papers (Heule et al. 2013) recommend for non-cryptographic streaming hashes anyway.
- **Cluster signature key** (D-04): hash of `(group_keys: Vec<&str>)` sorted lexicographically. Two aggs with `["user_id"]` cluster together; `["user_id", "merchant"]` is a different cluster. Register-time stability matters more than runtime speed of the signature.
- **EventTypeMix Cow refactor** (D-04b): the Cow's borrow lifetime ties to the row's borrow scope, which is the apply-event scope. No allocation on the hot path when the input is already `Value::Str(Arc<str>)`; allocate only for derived strings.
- **Default max_categories=1024 for entropy** (D-05a): aligns with EventTypeMix default; covers ~99% of fraud-team workloads; users override at register-time when their cardinality demands it.
- **TRACE_AGG inner output → /debug/op-cost JSON shape** (D-07): keep the per_kind data as-is (the bench harness already consumes this format); endpoint just exposes the in-memory aggregator's last snapshot.
- **`count_distinct(quadkey(lat, lon, zoom))` recipe** (D-05): `quadkey` is a register-time stateless op (Phase 4 expression DSL family); takes lat, lon, zoom and produces a deterministic integer cell id. If quadkey doesn't already exist as a stateless op, add it as part of the D-05 cleanup work — small scope addition, avoids users having to compose `floor(lat * 2^zoom) * X + floor(lon * 2^zoom)` manually.

</specifics>
