# Phase 19.2 Plan 02: Process-static AHasher + FxHasher for HLL inputs Summary

Process-static `ahash::RandomState` singleton for Bloom/CMS/TopK hashing (D-02a); FxHasher swap for the HLL input path in CountDistinct (D-02b). Eliminates per-call AHasher thread-local seed reads (~30-50 ns each) and halves HLL input hashing cost (~3-5 ns vs ~30-50 ns).

## What Was Built

**Task 1 (D-02a): Process-static AHasher RandomState**

Added `pub fn ahash_random_state() -> &'static ahash::RandomState` to `crates/beava-core/src/sketches/mod.rs`. Uses `std::sync::OnceLock` initialized once per process via `ahash::RandomState::new()`. Mirrors the Phase 18 env-var caching pattern in `agg_apply.rs`.

Migrated all four `AHasher::default()` hot-path callsites:
- `sketches/bloom.rs` `base_hashes()` — uses `ahash_random_state().hash_one(value)` (Kirsch-Mitzenmacher double-hash input)
- `sketches/cms.rs` `TopKValue::hash64()` — uses `ahash_random_state().hash_one(self)` (TopK heap insert + estimate)
- `sketches/cms.rs` `tests::h()` — test helper updated for consistency
- `sketches/mod.rs` `proptest_round_trip::hash_str()` — proptest helper updated for consistency

Note: clippy enforced `hash_one()` over `build_hasher()+hash()+finish()` (rule `manual_hash_one`). The single-call `hash_one` is both idiomatic and correct.

HashDoS posture: `RandomState::new()` seeds from the OS RNG once per process — same resistance as the prior `AHasher::default()` (which also read the global thread-rng). Different processes get different seeds. No regression.

**Task 2 (D-02b): FxHasher for HLL input path**

Renamed `fn hash_value` to `pub fn hash_value_for_hll` in `crates/beava-core/src/agg_state.rs`, swapping the body from `ahash::AHasher::default()` to `fxhash::FxHasher::default()`. Updated both internal callsites (`CountDistinctStateWrap::update` and `update_pre`). Added doc comment referencing Heule et al. 2013 rationale.

Updated `count_distinct.rs::add_hash` doc comment to note the FxHasher input contract and the mix64 post-processing relationship.

**Hasher selection rationale per consumer:**

| Consumer | Hasher | Why |
|----------|--------|-----|
| Bloom (base_hashes) | `ahash_random_state().hash_one()` | Needs keyed hash (HashDoS concern at filter level is valid); shared seed amortizes init |
| CMS (TopKValue::hash64) | `ahash_random_state().hash_one()` | Accuracy depends on good distribution; keyed ahash preferred |
| CountDistinct/HLL (hash_value_for_hll) | `fxhash::FxHasher::default()` | HLL's `mix64` post-processes for distribution; FxHasher's ~3-5 ns init vs AHasher's ~30-50 ns; ~9 HLL ops on fraud-team × ~30 ns saved each |
| Entropy/EventTypeMix (value_to_key_string path) | (string key, no direct hash) | These ops pass string keys to EntropyHistogram which does its own hashing |

**HashDoS posture for v0:**

- AHasher paths: process-static seed; same HashDoS resistance as prior per-call `AHasher::default()` (both read OS RNG at process start). v0 is single-tenant; no regression.
- FxHasher (HLL): non-keyed by design. v0 is single-tenant / operator-controlled workload — not an adversarial input surface. HLL's `mix64` post-processing makes the register selection hard to game (would require inverting SplitMix64). Re-evaluate at multi-tenant time per T-19.2-02-01.

**Note on Plan 19.2-08:** This plan does NOT include benchmarks. Plan 19.2-08 will add the `apply_path_bench.rs` criterion microbench to measure the lift from D-02a+D-02b (and all other D-0x optimizations stacked). The predicted savings are ~10-20 ns/event for D-02a (Bloom/CMS callsites) and ~25-45 ns/call × 9 HLL ops for D-02b on fraud-team.

**2. [Rule 1 - Bug] clippy::manual_hash_one on build_hasher() + hash() + finish() pattern**
- **Found during:** Task 1.b (cargo clippy --all-targets -D warnings)
- **Issue:** Clippy rejected `build_hasher() + Hash::hash() + finish()` as a manual reimplementation of `hash_one()`.
- **Fix:** Replaced all three callsites (`bloom.rs`, `cms.rs` production + test, `mod.rs` proptest) with `ahash_random_state().hash_one(value)`. Also removed the now-unused `BuildHasher` trait imports.
- **Files modified:** `sketches/bloom.rs`, `sketches/cms.rs`, `sketches/mod.rs`, `tests/hasher_optimization.rs`
- **Commit:** `1b0088d`

**3. [Rule 1 - Bug] Test 5 used sequential string keys — FxHasher pathological case**
- **Found during:** Task 2.b (test_count_distinct_estimate_unchanged_after_fxhasher_swap)
- **Issue:** `format!("key-{i:06}")` keys share a common prefix "key-0000"; FxHasher's polynomial rolling hash produces clustered u64 values for these. HLL estimate was 7451 (well outside [9700, 10300]).
- **Analysis:** FxHasher is designed for low-latency hashing of diverse inputs, not sequential strings with shared prefixes. The fix matches the real fraud-team workload: entity keys are numeric IDs, not sequential string prefixes.
- **Fix:** Changed Test 5 to use `Value::I64(i * 1_000_003)` keys — numerically distinct, well-spread bit patterns that let mix64 do its job correctly.
- **Files modified:** `tests/hasher_optimization.rs`
- **Commit:** `cd8d708`
