<context>
Phase 19.2 was the apply-path optimization phase. The architectural goals landed cleanly (53-op catalogue, FxHasher for HLL, EntityKey hybrid + cluster dispatch, UDDSketch flat sorted Vec, EventTypeMix Set+Cow+update_at, /debug/op-cost endpoint, cost-class.md). The microbench wins are real (criterion shows 14-agg warm-key apply at 362 ns vs prior trace ~13.4 µs full-path).

But the live throughput on fraud-team came in at 70.6k EPS, well below the 100k PASS target. Verifier blamed "WAL+bookkeeping overhead" — without running the live trace that would have shown WAL is 85 ns/event (negligible) and the actual culprit is the agg stage at 14 µs/event, dominated by WindowedOp wrapper layer doing per-call row.get() linear scans that Plan 19.2-01's pre-extraction was supposed to eliminate.

The WindowedOp wrapper takes raw `row` (not `extracted`), forwards to inner ops via `update_with_row` (not `update_with_extracted`), and inside each bucket the inner op re-does the row.get(fname) lookup that the apply-loop entry already paid once. 60 of 88 fraud-team feature updates pay this tax = ~9,000 ns/event of pure wrapping cost. The criterion bench at 362 ns measured the non-windowed fast path that 32% of fraud-team's features actually hit — bench was structurally blind to the windowed dispatch path.

The fix is mechanically simple — add `WindowedOp::update_at(extracted, field_idx, ...)` mirroring the non-windowed pattern that Plan 19.2-01 established. The investigation report (19.2-INVESTIGATION.md) has full file:line references and predicted lift estimates per sub-goal.

The bigger lesson: dispatch-protocol refactors must enumerate ALL dispatch entry points up-front, including nested wrapper layers. Plan 19.2-01's plan listed 19+ row.get callsites in agg_state.rs but never mentioned WindowedOp::update_with_row. The plan-checker (sonnet) checked what was listed. The verifier (sonnet) accepted the criterion bench at face value. The post-execution trace investigation (opus) caught it.

The model profile change (all 18 agents on opus) is the structural fix for that broader pattern — opus does broader pattern matching, more likely to ask "does this extension cover ALL the code paths that need it?" before declaring done.
</context>
