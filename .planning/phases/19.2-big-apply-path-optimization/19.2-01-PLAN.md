<objective>
Eliminate per-call `row.get(field_name)` linear scans on the apply-loop hot path by hoisting field extraction to apply-loop entry. Aggs resolve `field_name → field_idx` once at register-time and read pre-extracted values via `extracted[idx]` (~5 ns) instead of scanning the row's `SmallVec<[(CompactString, Value); 8]>` per call (~30-60 ns × ~17 callsites in `agg_state.rs` × N features).

Implements **D-01** locked in 19.2-CONTEXT.md.

Purpose: highest-leverage single fix for fraud-team apply-path (per audit: -800 to -2,500 ns/event saved). Foundation for D-04 (cluster) which assumes fields are already extracted.

Output: register-time field-idx resolution, apply-loop pre-extraction, all 17+ `row.get(field)` callsites in `agg_state.rs` collapsed to indexed lookups, missing-field rejection at register time, RED-then-GREEN tests proving the contract.

Note on geo field_idx scope: this plan validates that `descriptor.ext.lat_field` + `descriptor.ext.lon_field` exist in the source schema (rejection on missing geo fields) but DOES NOT migrate geo ops to compound `lat_idx: u8` + `lon_idx: u8` slots — that work is bundled into Plan 19.2-06 Task 3 alongside the unique_cells/geo_entropy removal (which also touches `agg_geo.rs`). After Plan 06 lands, the four surviving geo ops (geo_velocity, geo_distance, geo_spread, distance_from_home) consume `extracted[lat_idx]` + `extracted[lon_idx]` and `read_lat_lon` is removed.
</objective>
