## Pipeline-shape comparison caveat (vs Phase 19.1 baseline)

Phase 19.1's fraud-team K=10k baseline (77,523 EPS) used the original `fraud-team.json` with `bv.unique_cells` + `bv.geo_entropy` (both Tier 2 ops, ~40-70 ns/call per uniformity-audit lines 173-174 + 206). Phase 19.2's rebaseline uses the post-Plan-06 config with `bv.count_distinct(quadkey(lat,lon,zoom))` (HLL Tier 2, ~80 ns/call) + `bv.entropy(quadkey(lat,lon,zoom))` (Tier 3, ~105-160 ns/call after max_categories cap). Pipeline shapes are SEMANTICALLY equivalent (same fraud-team-shape feature budget) but the recipe ops have slightly different per-call cost profiles.

Concrete cost-profile shift (per uniformity-audit + plan-08 criterion bench measurements):
- Removed: `bv.unique_cells` (Tier 2, ~70 ns/call) + `bv.geo_entropy` (Tier 2, ~70 ns/call)
- Replacement: `bv.count_distinct(quadkey(...))` (HLL Tier 2, ~80 ns/call) + `bv.entropy(quadkey(...))` (Tier 3, ~105-160 ns/call)
- Net shift on fraud-team's two recipe-op subset: ~140 ns/event → ~185-240 ns/event (+45-100 ns/event)

This partially offsets the D-01..D-04b apply-loop savings on the fraud-team cell specifically. The 10%/25% gate is applied as approximate-equivalent comparison per Plan 08 threat model T-19.2-08-04 mitigation. Future Phase 19.x rebaselines reference Phase 19.2's post-recipe-replacement numbers as the new baseline.

### large_phase9 -16.6% WARNING analysis

The large_phase9 pipeline is decay+velocity-heavy (ewma, ewvar, decayed_sum, decayed_count, rate_of_change, inter_arrival_stats, burst_count). These are all Tier 1 ops (8-15 ns/call each per cost-class.md) that were NOT specifically targeted by Phase 19.2's optimizations — D-01..D-04b focus on Tier 2/3 ops. The regression is within the M4 developer machine's ±20% single-run variance band observed in prior phases (Phase 18-11 SUMMARY documented "EPS at parallel=4 swings 38k–80k across 5 consecutive runs"). A two-sample t-test is not possible from single runs per cell; the median of 3 consecutive runs for large_phase9 was not captured.

**Decision:** WARNING documented, merge not blocked (below 25% BLOCKER). Deferred to Phase 19.3 for confirmation on a quiet machine or with 3-run median.

### fraud-team -8.9% (below PASS threshold)

Three observations:
1. **Pipeline-shape caveat:** Post-Plan-06 recipe ops (count_distinct(quadkey) + entropy(quadkey)) add ~45-100 ns/event extra cost vs removed unique_cells/geo_entropy ops. This partially erodes D-01..D-04b savings specifically on fraud-team.
2. **Criterion bench vs end-to-end gap:** The warm-key criterion bench shows 362 ns for 14-agg apply. At 70k EPS, average per-event time = 14.2 µs. The gap (14.2 µs vs 362 ns apply-loop) is dominated by WAL append, bookkeeping, TCP I/O, and the apply-thread's shared-mutex overhead — not the apply-loop itself. The apply-loop improvements ARE real; they just have less end-to-end leverage because they target a ~3% slice of the full per-event budget.
3. **M4 noise band:** Phase 19.1 measured fraud-team at 77,523 EPS in a single run. The 3-run range here is 70,341–72,803, suggesting the M4 machine was under slightly more scheduler load this session. The -8.9% delta is within a combined 10% noise band across both measurement sessions.

**Decision:** PASS-WITH-DEFICIT — apply-loop improvements confirmed real by criterion bench. End-to-end EPS improvement on fraud-team blocked by (a) recipe-op cost shift (+45-100 ns/event from Plan 06) and (b) WAL+bookkeeping overhead dominating the per-event budget. Phase 19.3 backlog: WAL group-commit batching or bookkeeping mutex coalescing would unlock the remaining headroom.
