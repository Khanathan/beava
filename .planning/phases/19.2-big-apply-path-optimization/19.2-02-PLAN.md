The pattern to replace: a process-static `RandomState` that we hash through:
```rust
use std::sync::OnceLock;

pub fn ahash_random_state() -> &'static ahash::RandomState {
    static RS: OnceLock<ahash::RandomState> = OnceLock::new();
    RS.get_or_init(ahash::RandomState::new)
}
```

Then `let mut h = ahash_random_state().build_hasher();` (this still allocates an `AHasher` from the seed, but the seed lookup itself amortizes; build_hasher is ~2-3 ns vs default's ~30-50 ns).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /push | Untrusted: event payload could try to exploit hash collisions for performance attack |
| process boundary | RandomState seed randomized once per process via OnceLock — different processes hash differently |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19.2-02-01 | D (DoS) | FxHasher in HLL hot path (no HashDoS resistance) | accept | v0 is single-tenant, operator-controlled. HLL's mix64 post-processes the input hash; downstream HLL register selection uses bit-extraction so adversary needs to craft collisions in mix64-output space, not FxHasher-output space — same difficulty as cracking a non-cryptographic hash. Documented in code comment. Re-evaluate at multi-tenant time. |
| T-19.2-02-02 | D (DoS) | Process-static AHasher seed reuse within a process | mitigate | OnceLock initializes a fresh `RandomState::new()` per process — same HashDoS resistance as today's `AHasher::default()` (which also reads from the global thread-rng seeded once). No regression. |
| T-19.2-02-03 | T (Tampering) | Determinism across processes | accept | HLL estimates within a single process remain deterministic across pushes (D-06 invariant preserved). Cross-process estimates differ (different seeds), which is the same as today and is OK because snapshot/replay uses the post-hash bucket counts, not the hash inputs. |

No new attack surface — same trusted apply-path inputs. The hasher migration is a constant-factor optimization with security-equivalent guarantees.
