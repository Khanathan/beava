# Phase 19.2 Plan 01: Field-idx resolution + apply-loop pre-extraction Summary

Register-time field-index resolution plus per-event pre-extraction in the apply loop: `AggOpDescriptor.field_idx` populated once at registration, `ExtractedFields` built once per event, `update_with_extracted` replaces `update_with_row` in the hot path.

## What Was Built

**Task 1 (red→green): Register-time field-idx resolution + missing-field rejection**

Added three new fields to the descriptor structs:
- `AggOpDescriptor.field_idx: u8` — pre-resolved index into the per-agg `field_names` list; defaults to `FIELD_IDX_NONE` (`u8::MAX`)
- `AggregationDescriptor.field_names: Vec<String>` — per-agg distinct field list, built at register-time
- `EventDescriptor.apply_field_names: Vec<String>` — cross-agg union for potential future apply-path use

`Registry::resolve_field_indices_for_agg_mut` validates field references against the source schema, rejects missing fields with a clear error message, and populates `field_idx` and `field_names` in-place. Called from `apply_registration` via a static inner helper. Also exposed as `resolve_field_indices_for_agg` (read-only validate only) for test use.

`Row::_take_get_count()` probe added behind `feature = "test-utils"` Cargo feature — incremented on every `Row::get` call, reset and read by integration tests to assert architectural invariants.

**Task 2 (red→green): Apply-loop pre-extraction**

`agg_apply.rs` now builds `ExtractedFields` once per agg per event from `desc.field_names`, then in the features loop looks up `extracted[field_idx]` and calls `AggOp::update_with_extracted(pre_val, ...)`.

`AggOp::update_with_extracted` dispatches:
- Field-bearing ops (Sum, Avg, Min, Max, Variance, CountDistinct, Percentile, TopK, BloomMember, Entropy, First, Last, FirstN, LastN, Lag) → `state.update_pre(pre_val, where_matched)` — no `Row::get` call
- Fieldless ops (Count, Ratio, Recency, Streak, HourOfDayHistogram, etc.) → existing `update(row, ...)` — pre_val unused
- Windowed ops → `WindowedOp::update_with_row` fallback (bucket routing still needs row+field)
- Geo ops → existing `update(row, ...)` fallback (multi-field row access)
- Decay/velocity ops (Ewma, RateOfChange, etc.) → existing `update(row, ...)` fallback (deferred to later plans)

`agg_state.rs` gains `numeric_from_value(Option<&Value>) -> Option<f64>` helper and `update_pre(pre_val, where_matched)` on 15 concrete states.
