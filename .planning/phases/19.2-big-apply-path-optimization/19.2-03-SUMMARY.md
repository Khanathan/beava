**EntityKeyShape 3-variant enum (SingleU64/SingleStr/Multi) replaces EntityKey::from_row on the apply hot path, with cluster_id-keyed shape caching that builds EntityKeyShape once per unique group_keys signature per event**

## Accomplishments

- EntityKeyShape enum with SingleU64(u64), SingleStr(u64, CompactString), Multi(EntityKey) variants covers all Value types; non-NaN F64 and I64/Bool/Datetime route to the alloc-free u64 map
- AggStateTable replaced single `entities` HashMap with 3 sub-maps (single_u64, single_str, multi), each keyed by FxHasher for zero-alloc dispatch on the numeric/string single-key hot paths
- cluster_id assigned at register-time via FxHasher over declaration-order group_keys; apply loop builds EntityKeyShape once per (event, cluster_id) via shape_cache, not once per agg descriptor — fraud-team's 14 aggs across ~3 unique signatures pay ~3 shape-builds per event instead of 14
- F64-typed group_keys rejected at register-time with explicit error; NaN values at push-time produce None (event silently dropped for that agg)

## Decisions Made

- **EntityKeyShape::Multi wraps EntityKey not raw SmallVec**: Value doesn't derive Hash/Eq, so the raw SmallVec type cannot be used as a HashMap key. EntityKey (with its manual Hash+Eq impls over SmallVec contents) is the correct key type for the multi sub-map.
- **Declaration-order cluster signature overrides CONTEXT.md sorted-lex**: The existing EntityKey::from_row is order-sensitive (columns in declaration order). Sorting group_keys for the signature hash would create cluster_id collisions between aggs that produce different EntityKeys at runtime, breaking shared-lookup correctness. FxHasher with 0u8 separators over declaration-order keys prevents prefix collisions.
- **query_feature checks single_str first, then multi fallback**: The HTTP query API always produces Value::Str EntityKey pairs (pipe-separated parsing). Recovery-path inserts go to multi (insert_from_entity_key). The fallback ensures both paths are queryable.
- **test-utils feature gate for integration test counter**: `#[cfg(test)]` visibility is crate-internal; integration tests in tests/ compile as separate crates. The EKS_BUILDS thread-local and _take_entity_key_build_count are gated on `cfg(any(test, feature = "test-utils"))` with `required-features = ["test-utils"]` in Cargo.toml.
