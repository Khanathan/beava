## Q1 — Field pre-extraction storage shape

| Option | Description | Selected |
|--------|-------------|----------|
| Indexed array | `Vec<&Value>` with field-idx-into-row resolved at register-time. Apply-loop entry does ONE pass over the row's `SmallVec<[(CompactString, Value); 8]>`; aggs hold a `u8` field_idx. ~5 ns per lookup. | ✓ |
| Hashmap by field name | `HashMap<&str, &Value>` built per-event at apply entry. ~30 ns per lookup; allocation per event unless pooled. | |
| Hybrid (array + map fallback) | Top-N hot fields in array, cold fields in fallback HashMap. Adds branch complexity. | |

**User's choice:** Indexed array (Recommended)
**Notes:** Best cache locality, smallest per-event allocation, register-time validates field names exist.

## Q2 — HLL hasher choice

| Option | Description | Selected |
|--------|-------------|----------|
| FxHasher | ~3-5 ns vs AHasher's ~30-50 ns. HLL's `mix64` post-processes for distribution, repairing FxHasher's weaker statistical properties. HashDoS isn't a concern for operator-controlled fraud workloads. | ✓ |
| Cached AHasher | Eliminate per-call init via process-static instance. Saves ~10-20 ns/call. Preserves HashDoS resistance. | |
| FxHasher for HLL + cached AHasher elsewhere | Two hasher types matched to each op's theoretical floor. Complexity. | |

**User's choice:** FxHasher (Recommended)
**Notes:** Both fixes apply (D-02a + D-02b): process-static AHasher init for general hashing (Bloom, Entropy, EventTypeMix), FxHasher specifically for HLL inputs. Discussion lead-up explicitly framed these as cooperating fixes.

## Q3 — EntityKey single-u64 collision approach

| Option | Description | Selected |
|--------|-------------|----------|
| Approach C: Hybrid | `enum EntityKeyShape { SingleU64(u64), SingleStr(u64), Multi(SmallVec) }` with two storage maps per AggStateTable. Numeric → zero collision; string → birthday-paradox (~10⁻¹⁹ at 10M keys); compound → zero. | ✓ |
| Approach A: Numeric only | Only u64/i64/f64 keys get fast path. Strings fall back to Multi. Zero collision; loses ~150 ns lift for string-keyed aggs. | |
| Approach B: Hash everything to u64 | All single-keys hash to u64. Birthday-paradox collision domain expands to numerics. Reject. | |

**User's choice:** Approach C: Hybrid (Recommended)
**Notes:** Best coverage — numeric AND string keys both get the fast path.

## Q4 — Cluster storage shape

| Option | Description | Selected |
|--------|-------------|----------|
| Split by agg_id | Each clustered agg keeps its own `Vec<AggOp>` in its existing `AggStateTable`; cluster shares `EntityKey::from_row` + state-table lookup. | ✓ |
| Shared `Vec<AggOp>` | Cluster owns ONE `Vec<AggOp>` covering all member aggs; secondary indirection at update time selects which `AggOp[i]`. Saves ~24 bytes × (M-1) per cluster + one hashmap lookup per cluster. | |
| Shared Vec + per-agg slot table | Hybrid: `Vec<AggOp>` + parallel `slot_idx[]`. Most memory-efficient but most code complexity. | |

**User's choice:** Split by agg_id (Recommended)
**Notes:** Simpler than shared-Vec; preserves existing `AggStateTable` structure; no new dispatch indirection.

## Q5 — UniqueCells/GeoEntropy cap behavior

This question went through three iterations.

### Iteration 1 — initial cap-behavior question

| Option | Description | Selected |
|--------|-------------|----------|
| Drop new (truncate) | Map ignores novel cells once cap hit; existing cells still increment. | |
| HLL fallback at threshold | Swap state into HLL approximation. Lose exact count but preserve unbounded cardinality. | |
| LRU evict | Evict least-recently-touched cell. Output drifts over time. | |
| Reject at register-time only | No runtime cap behavior. Doesn't actually solve unbounded-memory. | |

**User's response:** "what are unique cells and geoentropy" — asked for op-semantics explanation before answering.

### Iteration 2 — re-asked with industry context after explaining op semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Drop new + cap-hit metric | Cap behavior + Prometheus counter `beava_geo_cells_capped_total`. Default `max_cells=1024`. Matches EventTypeMix prior art. | |
| Drop new (no metric) | Same drop-new behavior, less observable. | |
| HLL fallback at threshold | ~3-5 weeks added scope; irreversible state transition. | |
| LRU evict | Output drifts. Industry-rare. | |

**User's response:** "how about just remove that ops" — proposed removal as alternative; opened the question of whether unique_cells/geo_entropy belong in the catalogue at all.

### Iteration 3 — final removal-vs-cap question

| Option | Description | Selected |
|--------|-------------|----------|
| Remove both | Delete `bv.unique_cells` + `bv.geo_entropy`. Document `count_distinct(quadkey)` + `entropy(quadkey)` as canonical recipes. Apply `max_categories` cap to `bv.entropy`. | ✓ |
| Keep both with cap + metric | Preserve geo APIs + apply cap to all three (unique_cells, geo_entropy, entropy). | |
| Make them register-time aliases | Desugar to `count_distinct(quadkey)` and `entropy(quadkey)`. Adds register-time complexity. | |
| Asymmetric (remove unique_cells, keep geo_entropy) | Splits the difference at the cost of inconsistency. | |

**User's choice:** Remove both (Recommended)
**Notes:** v0 not shipped publicly so no migration cost. Operator catalogue: 55 → 53. Sub-goal 6 reframed from "cap UniqueCells/GeoEntropy" to "remove the two ops + apply max_categories cap to bv.entropy". The user's preference for op-removal here is consistent with the broader devex-first principle (memory `project_v2_devex_first`) — fewer overlapping ops is cleaner.

## Q6 — Cost-class doc surface

| Option | Description | Selected |
|--------|-------------|----------|
| Source attribute | `#[cost = "tier1\|tier2\|tier3"]` proc-macro + build-script-generated docs page. Single source of truth; no drift. ~1 day macro work. | |
| Catalogue markdown table | Hand-maintained `docs/operators/cost-class.md`. Zero build complexity. Drifts from source over time; review-time discipline. | ✓ |
| Both: source attribute as truth + docs page generated | Source attribute + generated docs page. Equivalent to option 1 with explicit doc generation. | |
| Runtime-only via /debug/op-cost | Skip docs entirely. Worse devex for capacity planning. | |

**User's choice:** Catalogue markdown table
**Notes:** Simpler. Review-time discipline acceptable. Audit docs already use this format. Source-of-truth is the markdown; if drift becomes an issue post-v0, the source-attribute approach is a clean upgrade path.

## Q7 — `/debug/op-cost` endpoint gating

| Option | Description | Selected |
|--------|-------------|----------|
| Feature-gated behind `BEAVA_DEV_ENDPOINTS=1` | Only registered when env var set. Production deployments leak no internal latency telemetry. Matches Phase 15 pattern. | ✓ |
| Always-on | Endpoint registered unconditionally. Simpler config. Concern: multi-tenant deployments. | |
| Always-on but auth-gated | Adds auth-token middleware. Probably overkill for v0 (single-trust-zone). | |
| Always-on, /metrics-style | Expose through Prometheus `/metrics`. Reuses existing scrape config. | |

**User's choice:** Feature-gated behind BEAVA_DEV_ENDPOINTS=1 (Recommended)
**Notes:** Aligns with "no surprises in prod" principle. Operators set env var explicitly when they want the endpoint exposed.
