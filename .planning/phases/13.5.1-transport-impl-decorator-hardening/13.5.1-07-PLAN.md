## 6 architectural deviations — formalize-or-defer verdict

Each deviation below was a Plan 05 Rule-2/Rule-3 in-plan auto-fix. They MUST get an explicit verdict so the SDK doesn't carry undocumented hacks into v0 GA.

### Deviation 1 — `_table.py::_resolve_upstream_proxies` walks caller-frame `f_locals`
**Site:** `python/beava/_table.py:60-104` (commit `5fb99fe5`)

**Description:** Plan 04 locked D-01 strict TypeError on unannotated `@bv.table` parameters, relying on `get_type_hints(fn)` to resolve types via `fn.__globals__`. The pytest pattern of declaring `@bv.event class Txn:` *inside* a test fn placed `Txn` in the caller's `f_locals`, never in module globals. Plan 05 worked around by walking `inspect.currentframe().f_back` up to 8 frames deep, collecting locals, and passing them as `localns` to `get_type_hints` (with a last-ditch `caller_locals.get(name)` fallback).

**Verdict:** **FORMALIZE-V0** (FIX-NOW)

**Rationale:** Frame-walking with magic depth limits is the textbook example of fragile-on-introspection code. If a user wraps `@bv.event` in `functools.lru_cache` or reflects through a dataclass-decorator stack, the frame depth shifts and silent breakage returns. The right fix: surface a public `bv.event_class(name, ...)` factory that doesn't require class-decorator semantics, OR document the supported declaration patterns and tighten the resolver to a documented contract. This belongs in v0 because every Phase 13.5+ user will hit this if they write tests in the natural pytest style.

**Follow-up plan:** `13.5.1-07e` (deviation formalization cluster) — sub-task: refactor `_resolve_upstream_proxies` to use a documented declaration contract; ship a `docs/sdk-api/python.md` section describing supported `@bv.event` declaration sites; update or add `test_table_decorator_typeerror.py` cases for inner-class, lambda-defined, and lru_cache-wrapped cases.

### Deviation 2 — `_embed.py` per-spawn unique `BEAVA_WAL_DIR` + `BEAVA_SNAPSHOT_DIR`
**Site:** `python/beava/_embed.py:107-126` (commit `5fb99fe5`)

**Description:** Default `./beava-wal/` collided across parallel pytest spawns (`os error 17`); each `spawn_embedded_server` call now pins WAL + snapshot dirs to `$TMPDIR/beava-embed-<pid>-<unix-ms>-<random>/...`.

**Verdict:** **FORMALIZE-V0** (FIX-NOW, but already correct shape — needs only documentation + cleanup hooks)

**Rationale:** This is a correct fix that should ship as-is — it's the right behavior for `test_mode=True`. Two follow-up gaps remain: (a) no cleanup-on-exit registered for the unique tmpdirs (pytest fixtures will leak GBs of disk if the test churns), (b) no documentation that `bv.App(test_mode=True)` allocates per-spawn disk. Both are small.

**Follow-up plan:** `13.5.1-07e` — sub-task: register `atexit.register` cleanup for the per-spawn tmpdir; document the disk allocation in `docs/sdk-api/python.md`'s `App(test_mode=...)` section.

### Deviation 3 — `_app.py::_descriptor_to_node` Python-side schema inference
**Sites:** `python/beava/_app.py:117-241` (`_FIXED_OP_OUTPUT_TYPE`, `_agg_output_type`, `_infer_derivation_schema`, `_infer_event_derivation_schema`, `_parent_wire_schema`, `_parent_wire_schema_from_root`)

**Description:** Server-side `DerivationDescriptor` deserializer requires a `schema` field (no `#[serde(default)]`). Plan 13.5-11 left `_descriptor_to_node` emitting derivation nodes without it; register rejected with `invalid_registration: missing field 'schema'`. Plan 05 added a 53-op output-type table on the Python side that mirrors Rust `output_type_for` (in `crates/beava-core/src/agg_op.rs:1129-1223`).

**Verdict:** **REVERT-NEEDED** (FIX-NOW; this hack is wrong — schema inference belongs server-side)

**Rationale:** Maintaining a parallel 53-op output-type table in Python that has to track Rust verbatim is a long-tail correctness hazard. Every new operator added to the Rust catalogue requires a Python-side update or registration silently breaks for that op. The right fix is server-side: make `DerivationDescriptor.schema` `#[serde(default)]` and have the server infer it from the chain, the way the SDK currently does (since Rust already has `output_type_for`). The Python-side table can then shrink to a compatibility shim or get deleted.

**Follow-up plan:** `13.5.1-07b` (Category 2 server-side schema-propagation cluster — natural co-location since both touch server schema handling). Sub-tasks: (1) make `DerivationDescriptor.schema` defaultable in Rust + infer from chain via existing `output_type_for`, (2) delete or hollow out Python's `_FIXED_OP_OUTPUT_TYPE`, (3) add a regression test in `crates/beava-server/tests/` asserting register succeeds without an explicit `schema` field.

### Deviation 4 — `HttpTransport.send_push` body shape `{event, data}`
**Site:** `python/beava/_transport.py:160-202`

**Description:** PATTERNS.md and `docs/http-api.md:176` documented verb-style `{event_name, fields}`. Live server parser at `crates/beava-runtime-core/src/http_listener.rs:284,301,313` actually expects `{event, data}`. Plan 13.4 verb-style migration apparently didn't finish the rename on the parser side. Plan 05 SDK matches the live impl.

**Verdict:** **DOCUMENT-V0.0.X** (acceptable hack; deviates from documented intent but matches shipped contract)

**Rationale:** The SDK is correct in matching the live server. The fix is a documentation erratum, which Plan 05 SUMMARY already noted is queued. Reverting to `{event_name, fields}` requires a coordinated server + SDK change that doesn't add value at v0 — the wire shape is private (users don't hand-construct `/push` bodies). v0.0.x cleanup pass: pick one (`{event, data}` lock OR rename to `{event_name, fields}` everywhere) and align docs + parser + SDK.

**Follow-up plan:** None for v0. Add an entry to `.planning/ideas/v0.1-deferrals.md` v0.0.x section: "Lock HTTP `/push` body shape: `{event, data}` vs `{event_name, fields}` — pick one and align docs + parser + SDK."

### Deviation 5 — `TcpTransport.send_reset` checks `OP_GET_RESPONSE` for success
**Site:** `python/beava/_transport.py:581-617`

**Description:** PATTERNS.md A5 said `OP_RESET` returns its own opcode `0x0040`. Live server at `crates/beava-server/src/server.rs:2338-2344` returns `OP_GET_RESPONSE = 0x0023` with body `{"reset": true, "registry_version": N}` (per `apply_shard.rs:1538-1539`).

**Verdict:** **DOCUMENT-V0.0.X** (acceptable hack; deviates from documented intent but matches shipped contract)

**Rationale:** Same shape as Deviation 4 — SDK matches the live server, and the wire opcode is private. Reverting requires a coordinated rename. The right answer for v0.0.x is to either standardize on a generic `OP_JSON_RESPONSE` opcode for "any JSON-bodied success response" or add the dedicated `OP_RESET` response opcode. Both are real wire-protocol changes; neither is justified for v0.

**Follow-up plan:** None for v0. Add an entry to `.planning/ideas/v0.1-deferrals.md` v0.0.x section: "Standardize TCP success-response opcodes: either generic `OP_JSON_RESPONSE` or per-op `OP_RESET_RESPONSE` / `OP_PING_RESPONSE`. Document and align SDK + server."

### Deviation 6 — `crates/beava-server/src/server.rs::build_runtime_state` honors `BEAVA_TEST_MODE=1`
**Site:** `crates/beava-server/src/server.rs::build_runtime_state` (commit `5fb99fe5`)

**Description:** Phase 13.4 D-03 (USER-LOCKED) documented `BEAVA_TEST_MODE=1` as one of two opt-in paths to enable OP_RESET. The `build_runtime_state` comment said "the env var is the deployment-knob path (paired with `bind_with_config` in main.rs)" — but `main.rs` calls `bind()`, not `bind_with_config()`. The env var was a dead knob; without the fix, `bv.App(test_mode=True)` couldn't enable reset on a spawned binary, and `test_reset_equivalent[*]` failed 3/3.

**Verdict:** **FORMALIZE-V0** — already-formalized; no further action needed

**Rationale:** The Plan 05 fix wired `build_runtime_state` to honor the env var per the existing Phase 13.4 D-03 contract — this isn't a hack, it's the *completion* of a Phase 13.4 wire-through gap that happened to be discovered during 13.5.1. Phase 13.4.1 SUMMARIES should be cross-referenced to confirm the gap is closed; if 13.4.1's SUMMARY doesn't yet mention it, append a one-line addendum.

**Follow-up plan:** `13.5.1-07e` — micro-task: append a one-line addendum to `.planning/phases/13.4.1-server-wire-spec-verb-style-migration/13.4.1-XX-SUMMARY.md` (whichever is appropriate) noting the `build_runtime_state` env-var wire-through landed in `5fb99fe5` as part of 13.5.1-05; OR drop the addendum if 13.4.1's existing artifacts already capture it.
