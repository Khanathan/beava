## Spike findings (rejected clock-injection)

The orchestrator's prompt called for "spike-then-fix; tolerance route is cheaper if it works." The spike captured failure modes for all 8 tests against the live binary and concluded:

| # | Failure mode | Why clock-injection wouldn't help |
|---|--------------|----------------------------------|
| 1 | `test_age`: age=134, expected_min=186 (50ms slack too tight) | The skew is between client wall-clock (`first_ts` recorded BEFORE push) and engine processing-time (`first_processed_ts` recorded AFTER push arrives). A synthetic engine clock wouldn't fix the client/engine timing-domain gap — only widening the test's tolerance bounds it. |
| 2 | `test_ewma`: diff=14.5 vs abs=10 | EWMA's online EW form is dominated by recent values, not arithmetic mean. Engine-correct under ANY clock (synthetic or real). Tolerance-only fix. |
| 3 | `test_ewvar`: actual=0.01 vs expected=99.91 | EWVar near zero under sub-second pushes (engine-correct online EW form). A fixed clock wouldn't change the EW math. |
| 4 | `test_ew_zscore`: \|z\|=239 vs bound 6 | Direct downstream of #3 — division by tiny EWVar. |
| 5 | `test_trend`: slope=-7.6e-06 vs >0 | OLS slope on degenerate time-axis (var(t)≈0); FP-noise dominated. Synthetic clock with the same ms-clustered timestamps produces the same outcome. |
| 6 | `test_trend_residual`: returns None | Engine returns None for ill-conditioned regressions (valid contract). Skip-on-None fix. |
| 7 | `test_top_k`: dict-shape response | Pure shape-handler bug — no clock dependency. |
| 8 | `test_bloom_member`: TypeError on `capacity` kwarg | Helper-drift Category-1 symptom — fails at decoration time before any clock runs. |

**Conclusion:** Clock-injection is unjustified. The tolerance / weakening / shape-fix route handles all 8 with bounded test-only edits.

## Decisions made

1. **Test-only surface.** `python/beava/_embed.py`, `crates/beava-core/src/time.rs`, and `python/beava/_agg.py` were NOT touched. This preserves the bounded scope explicitly called for in the orchestrator's prompt.

2. **Weak invariants over arithmetic-convergence assertions.** Tests #2-5 originally encoded statistical-textbook expectations (EWMA→arithmetic mean, |z|<6 for std-normal, OLS slope>0 for monotonic input). Under the engine's processing-time-only Redis-shaped semantics (`project_redis_shaped_no_event_time_ever`), those expectations don't hold under sub-second-burst pushes. The fix encodes the engine-correct contract: finiteness, non-negativity (for variance), in-range (for EWMA over `[0,100]` inputs), bounded-magnitude (for OLS slope catching NaN/Inf without rubber-stamping). This matches the existing weak-invariant pattern of `test_decayed_sum`, `test_decayed_count`, `test_twa` in the same files.

3. **Test #1 keeps the lower-bound assertion** but widens slack 50→300ms. Age IS bounded below by `query_start - first_ts - slack`; the only fix needed was acknowledging that a 500-event push burst spreads ≥150ms.

4. **Test #8 (bloom_member) is a misclassification.** Triage labeled it "sketch-warmup timing tolerance" but the actual failure is a Category 1 helper-drift symptom (TypeError at decoration time). Per scope discipline, the fix lands in 07c as a test-side correction (drop unsupported kwargs); if 07a later determines the helper SHOULD accept `capacity`/`fpr`, it can re-add them to both helper and test. This is documented inline in the test.

5. **Test #7 (top_k) is also a misclassification.** Triage labeled it "sketch-warmup timing tolerance" but the failure is a response-shape mismatch (engine returns dicts, test extracted only tuples). Pure shape-handler extension.
