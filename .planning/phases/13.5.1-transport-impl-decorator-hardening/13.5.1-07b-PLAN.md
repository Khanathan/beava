<objective>
Plan 13.5.1-07 triaged 4 `test_lit.py` failures as Category 2 (server schema-prop gaps) + 1 as Category 3 (feature-name collision, fix via test rename). Plan 05 separately marked Deviation 3 as FORMALIZE-V0: revert the Python `_FIXED_OP_OUTPUT_TYPE` mirror and let the server own schema inference. This plan executes both: (1) make `DerivationDescriptor.schema` serde-defaultable + chain-infer at the validator, (2) widen the where-predicate validator to see sibling event-derivation schemas in the same payload, (3) revert Python's `_FIXED_OP_OUTPUT_TYPE` hack, (4) rename test fixtures to use globally-unique feature names. Also adds a Rule-3 blocking-issue fix (`BEAVA_ADMIN_ADDR=:0` in `_embed.py`) without which no v0 pytest can boot the binary at all.
</objective>

<discovery_log>
**Discovered during execution (additional scope beyond parent triage):**

1. `_embed.py` does not set `BEAVA_ADMIN_ADDR`, so the spawned binary defaults to `127.0.0.1:8090`. On any host with a prior `beava` instance running (or parallel tests), this fails with `Address already in use`. This is a Rule 3 blocking-issue auto-fix (without it the test binary can't boot at all; the previous 4 executors likely hit this and misdiagnosed).

2. `test_lit_explicit_filter_literal` ALSO fails with `feature_name_collision` (CountImplicit + CountExplicit both declare feature `n`), not just the where-prop bug the parent plan listed. Same root cause as `test_lit_immutability`. Sub-task 4 expanded to rename both tests' features.

3. `test_lit_value_types` fails with `unknown character '&'` — the test uses Python `&` operator in `bv.col(..) & bv.col(..)`-style expressions but the server expression parser does not accept `&` (only `and`?). This is a pre-existing parser gap completely unrelated to schema propagation. **DEFERRED to v0.0.x backlog** — out-of-scope per Rule SCOPE BOUNDARY (not introduced by this plan's changes; affects parser, not validator).

4. `test_lit_force_float_division` reports `field 'rate' does not exist in upstream schema` — same Category 2 root: the `Rated = Telemetry.with_columns(rate=...)` derivation adds `rate`, but `UserMeanRate(rated: Telemetry)` declares upstream `Telemetry` (no `rate` field). Solved by sub-task 2's union-of-sibling-event-derivations approach.
</discovery_log>

## Sub-task 1 — Rust `DerivationDescriptor.schema` serde-default + chain-infer

**Files:** `crates/beava-core/src/registry.rs`, `crates/beava-server/tests/phase13_5_1_07b_register_no_schema.rs`

### Task 1a (RED)
Add a workspace-level integration test asserting that a register payload with `kind: derivation` AND no `schema` field is accepted by the server (currently rejected with `missing field schema`).

Commit: `test(13.5.1-07b): RED — register fails without explicit schema field on derivation`

### Task 1b (GREEN)
- Add `#[serde(default)]` to `DerivationDescriptor.schema`.
- Server already runs schema-propagation in `validate_expressions` and writes `propagated_schemas` — the chain-inferred schema is written back to the registry post-validation. This means: for a register payload that omits `schema.fields`, the server already has the machinery to fill it in. The failing case is just the deserializer rejecting `missing field`.

Commit: `feat(13.5.1-07b): DerivationDescriptor.schema serde(default) + chain inference`

## Sub-task 2 — Where-predicate validator: union sibling-derivation schemas

**Files:** `crates/beava-core/src/agg_compile.rs`

### Task 2a (RED)
The failing test `test_lit_constant_column` IS the red signal. Skip a separate red-write step here: the test in `python/tests/v0/test_lit.py::test_lit_constant_column` is the contract; running it currently produces `aggregation_invalid_where: where predicate references unknown field 'source'`. Treat as red-by-existence.

### Task 2b (GREEN)
In `resolve_upstream_schema_for_agg`, when the resolved upstream is an event (or an event in the registry), additionally walk the same-payload `nodes` and union in the schema of any `kind=derivation, output_kind=event` whose ONLY upstream is that same event name. Field-name collisions: last-write-wins (sibling additions take precedence over base event fields, mirroring the runtime apply order).

Rationale: when a user writes `Tagged = Click.with_columns(source=...)` then `UserClicks(tagged: Click)`, the SDK records `UserClicks.upstreams = ["Click"]` (correct — the param is annotated `: Click`). At register time, the server should see all "shape-extending" siblings of the Click event in the same payload to validate fields like `where source == 'web'`. This matches the user's intuition that "Tagged is a Click-shaped event with extra columns".

Commit: `fix(13.5.1-07b): propagate event-derivation schema to where-predicate validator`
