## Deferred Issues (PASS-WITH-DEFERRALS)

These items were discovered during execution but are **out-of-scope per Rule SCOPE BOUNDARY** — pre-existing issues not introduced by this plan, and architecturally larger than the schema-prop fix this plan was scoped for.

### test_lit_constant_column + test_lit_force_float_division — runtime sibling-derivation chain not auto-applied

- **Symptom 1:** `test_lit_constant_column` registers `Tagged = Click.with_columns(source=bv.lit("web"))` then `UserClicks(tagged: Click)`. Register passes after sub-task 2; runtime push delivers `Click` events directly to `UserClicks` (upstream `Click`), bypassing Tagged's `with_columns`. Result: `where source == 'web'` evaluates false on every event → `web_only=0`.
- **Symptom 2:** `test_lit_force_float_division` — same shape: `Rated = Telemetry.with_columns(rate=...)` → `UserMeanRate(rated: Telemetry)`; `field='rate'` not visible at apply time → `mean_rate=None`.
- **Root cause:** Plan 13.5.1's data-plane has no concept of "derived event source". Each `@bv.table def F(x: SomeEvent)` reads `SomeEvent` events directly. Sibling event-derivations like `Tagged` are registered nodes but not auto-routed into downstream tables that name `Click` as upstream.
- **Why deferred:** Architectural change — requires deciding whether Beava v0 supports "derived event sources as substitutes for raw events" or whether users must explicitly annotate `def UserClicks(tagged: Tagged)`. The latter is the current contract; the test was authored against an unimplemented former contract.
- **Recommendation:** Add to `.planning/ideas/v0.1-deferrals.md` v0.0.x section: "Decide @bv.table semantics for sibling event-derivation routing — either (a) wire it through the apply path (sticky cost; needs benchmark), or (b) fix the test to annotate `def UserClicks(tagged: Tagged)`."

### test_lit_value_types — server expression parser does not accept `&` operator

- **Symptom:** `[invalid_expression] nodes[1].ops[0].expr: col 13: col 13: unknown character '&'`
- **Root cause:** Test uses Python `&` (bitwise-and) in `(bv.col("n") >= bv.lit(0)) & (bv.col("x") <= ...)`. The `_col.py` overload renders this as `&` in the wire string, but the server's expression parser (`crates/beava-core/src/expr.rs`) does not lex `&`.
- **Why deferred:** Pre-existing parser gap, completely unrelated to schema propagation. Fix would be a 1-line lexer addition + grammar update, but is out-of-scope per Rule SCOPE BOUNDARY (not introduced by this plan; affects parser, not validator).
- **Recommendation:** Add to `.planning/ideas/v0.1-deferrals.md`: "Server expr parser lex `&` / `|` / `~` so Python bitwise overloads in chain predicates roundtrip."

### test_lit_immutability — runtime same as #constant_column case (UseA/UseB filters bypassed)

- After rename, register passes; runtime hits same architectural issue as test_lit_constant_column — `UseA = M.filter(...).named("UseA")` is registered separately but `CountA(filt: M)` reads M directly, so the filter is never applied. Same deferral as item 1.
