<interfaces>
BASELINE.json (from Phase 22-04) defines 9 gated cells:
  small_1c, small_4c, small_8c, medium_1c, medium_4c, medium_8c, large_1c, large_4c, large_8c

Each cell's pass criterion: `delta_pct_vs_baseline &gt;= -5%` (measured eps
within 5% of BASELINE median). Phase 23 already added 2 characterization
cells (join_small_1c, enrich_small_1c) that don't gate.

Phase 24 adds 4 more characterization cells:
- `late_events_small_1c` — 10% of events arrive late-but-within-window (force the watermark compare path on hot loop)
- `tombstone_cascade_small_1c` — 1 in 100 events is a OP_DELETE_TABLE (force the cascade tombstone branch)
- `tt_join_real_1c` — TT-join driven by OP_PUSH_TABLE (not the marker shim from Phase 23); proves storage+cascade real path
- `enrich_with_wm_1c` — Stream↔Table enrichment with `_event_time`-bearing events; exercises watermark parse + expression eval with event_time()

Integration test scaffold: start a tally_server with registered:
  - 1 source Stream (Purchases, keyed on user_id, carries _event_time)
  - 1 source Table (UserProfile, keyed on user_id) fed via OP_PUSH_TABLE
  - 1 derived Stream (EnrichedPurchases = Purchases.join(UserProfile))
  - 1 derived Table (UserStats = EnrichedPurchases.group_by(user_id).agg(count, sum))
  - 1 derived Table (UserView = UserStats.join(UserProfile, on=user_id, type="inner"))

Exercise:
- Push events in-order → UserView has expected rows
- Push an out-of-order event within 5s → lands in correct bucket, UserView updates
- Push a late event past 5s → dropped, `tally_late_events_dropped_total` increments
- OP_DELETE_TABLE on UserProfile → cascade retracts EnrichedPurchases' next lookup, and UserView is tombstoned (Table row state flips)
- After 7d grace (fast-forward via a test-only `gc_tombstones(t+7d+1s)` call), the tombstoned rows are GC'd

This is the biggest, most integrated test Phase 24 produces.
</interfaces>
