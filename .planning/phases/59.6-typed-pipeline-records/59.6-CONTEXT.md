# Phase 59.6: typed-pipeline-records — Context

**Gathered:** 2026-04-21
**Status:** Ready for planning
**Mode:** Manual (follow-on to 59.5 profiling finding)

<domain>
## Phase Boundary

Replace `serde_json::Value` as the in-pipeline event/state representation
with typed, fixed-layout row records derived from the SDK's declared
schema at `@bv.stream` / `@bv.source_table` register time. Wire codec,
engine operators, and state store all work on typed rows. JSON fallback
is retained for dynamically-schema'd / ad-hoc paths but is off the hot
path of registered pipelines.

**Goal:** bring per-event shard-thread cost from 8.5μs to ~1-2μs,
lifting the per-shard ceiling from ~118K EPS to ~500K-1M EPS. At 10
shards, ~5-10M EPS per node — Materialize territory.

**In scope:**
- SDK → schema: `@bv.stream`/`@bv.source_table`/`@bv.table` compile to
  a `RegisteredSchema` (field list, types, offsets) at register time
  and send it alongside the existing REGISTER JSON.
- Server-side typed `Row` representation — `#[repr(C)]` packed struct
  with offset-based field access. Inline short strings (≤ 15 bytes)
  avoid heap alloc; longer strings arena-allocated per row.
- Binary wire codec evolution: OP_PUSH_BATCH payload becomes
  schema-aware typed rows (length-prefixed columns). Phase 59's binary
  wire format is the starting point — this extends it to
  schema-parameterized row layouts.
- Operator specialization: EnrichFromTable, SumOp, CountOp, LastOp,
  FirstOp, AvgOp, MinOp, MaxOp, DistinctCountOp, EmaOp, LagOp,
  StddevOp, PercentileOp, VarianceOp, TopKOp, FirstNOp, LastNOp get
  typed implementations specialized on the column type. Generic
  dispatch preserved for the dynamic-schema fallback path.
- State store: replace `AHashMap<String, Value>` entity state with
  typed row storage. Fjall serialization becomes packed-row memcpy
  instead of serde_json round-trip.
- Cascade + Stream-Stream Join + EnrichFromTable operator-level
  changes: they all consume and produce typed rows.
- A clean fallback: the `Value` path is kept for ad-hoc (REST /debug,
  historical extract, dynamic predicates) — registered pipelines run
  typed.

**Out of scope (explicit):**
- Schema evolution (adding/removing fields from a registered stream).
  For this phase, schema is immutable post-register. Evolution is a
  follow-up once typed infra is stable.
- SQL-style dynamic predicates over typed rows — Phase 71+ if we
  decide we want on-the-fly filter/select in registered pipelines.
- Arrow/Parquet interop — conceptually neighbors this work (typed
  record batches) but out of scope for the perf unlock. Can land as
  an additive format once typed pipeline is in place.
- Python SDK Arrow integration — same reason. Future-compat.
- Changes to the storage layer's LSM structure (we keep fjall/inmem);
  we only change what we store and how we encode it.
- Wire protocol redesign — we extend OP_PUSH_BATCH / OP_REGISTER's
  existing envelope. No new frame ops.
- Hot-key salting (Phase 60) — orthogonal axis; 59.6 makes each event
  cheaper, 60 makes hot keys spread. Both compound.

</domain>

<problem_statement>
## The 8.5μs bottleneck (from 59.5-W3.5 histograms)

Per-event shard-thread cost breakdown at sustained 1.2M EPS:

```
decode phase         avg=0.89μs  (serde_json::from_slice or binary→Value)
lock_acquire phase   avg=0.42μs  (parking_lot RwLock reads, uncontended)
pipeline phase       avg=8.5μs   (push_with_cascade_on_shard — 87.6%!)
─────────────────────────────────────────────
total                avg=9.8μs
```

Inside the 8.5μs pipeline phase, time goes to (estimated):
- Stream-def lookup ~100ns
- EnrichFromTable lookup + enriched-event construction ~650ns
- Cascade dispatch ~200ns
- Aggregation feature updates (3 × vtable dispatch) ~900ns
- Sliding-window state maintenance ~500ns
- State write-back ~300ns
- Watermark observe ~100ns
- Event queue handoff + misc ~200ns
- **`serde_json::Value` cloning / allocation / enum dispatch: ~5-6μs**

The last line is where the refactor targets. `Value::clone()` is
~500ns for a small object. Multiple clones per event stack up. Enum
match on `Value::String/Number/Object` every field access is another
5-20ns per access × dozens of accesses per event.

Materialize at ~2-5μs per event on typed `Row`, Beava at 8.5μs on
`Value`. The gap is structural — not a compiler setting.

</problem_statement>

<decisions>
## Implementation Decisions

### Area A — Schema runtime

**D-A1:** `RegisteredSchema { fields: Vec<FieldSpec>, inline_str_cap: u8 }`
server-side. `FieldSpec { name, ty, offset, nullable }`. `ty` is a
compact enum — `I64`, `F64`, `Bool`, `InlineStr` (up to 15 bytes),
`String` (arena-allocated offset into per-row arena), `Bytes`. Schema
is constructed at OP_REGISTER time, keyed by stream name, stored in
a `SchemaRegistry: HashMap<String, Arc<RegisteredSchema>>`.

**D-A2:** Row type: `#[repr(C)] struct Row { schema_id: SchemaId,
payload: [u8; N] }`. Fields at known offsets. Field access via
`unsafe { std::ptr::read_unaligned(ptr.add(offset)) }` for scalars;
`&str` slice for inline strings.

**D-A3:** For strings longer than inline cap, payload holds a `(start,
len)` pair pointing into a per-row arena allocation. Arena is a single
`Vec<u8>` that lives as long as the row.

### Area B — Wire codec (extend Phase 59 binary)

**D-B1:** OP_PUSH_BATCH gains a `schema_id` prefix. Subsequent rows
are packed according to that schema's layout. Decoder on the server
validates `schema_id` matches a registered schema, then either
memcpys the fixed portion directly into a Row or fails with
`ProtocolError::UnknownSchema`.

**D-B2:** Back-compat: if the first OP_REGISTER did not ship a typed
schema, the stream stays on the JSON path. Mixed-mode deployments
work — one stream typed, another JSON. `SchemaRegistry.get(name)`
returns `Option<Arc<RegisteredSchema>>` and operators branch on it.

**D-B3:** SDK side: `@bv.stream` decorator extracts the class's
type hints at import time and builds a `Schema` object. At
`app.register()`, the schema is compiled into a binary layout and
sent as a `schema: {fields:[{name, ty, offset}], inline_str_cap: 15}`
section of the REGISTER payload alongside the existing `fields:
{...}` metadata. Server reads the `schema` section and constructs
`RegisteredSchema` from it.

### Area C — Operators

**D-C1 (specialization strategy):** Operators are generic over column
type where it helps. E.g., `SumOp<T: AggNum>` with `T` instantiated
to `i64` or `f64` at schema-bind time. The trait `AggOp` gets a
typed variant `TypedAggOp: Send + Sync` that operates on a typed
row. The generic dispatch (`Box<dyn AggOp>`) stays for the dynamic
path.

**D-C2 (EnrichFromTable):** right-side Countries row is already
typed (replicated source table); enrich is a struct-field copy from
source row to enriched row. Generates a new `Row` for the enriched
stream, typed according to the enriched-stream schema (which the
SDK computes at `.join()` time).

**D-C3 (StreamStreamJoin):** both sides route to hash(join.on) % N
already (Phase 56). L and R buffers store typed rows. Join match
constructs a typed enriched row.

**D-C4 (Aggregation state):** `group_by` entity state is a typed
row per entity key. Feature updates mutate specific columns in
place. No allocation per event once the entity state is initialized.

**D-C5 (fallback path):** the existing `Value`-based operator
implementations are kept, feature-gated by a runtime check: if the
stream has no typed schema, route through the generic path. Used for
dynamic queries, REST /debug reads, historical extract. Never on
the hot ingest path of a registered pipeline.

### Area D — State storage

**D-D1:** fjall backend: per-entity state was JSON-serialized. Now
it's memcpy of the row payload (schema_id + fixed bytes + arena). No
serde_json. Encoding is ~10-50× faster per entity.

**D-D2:** inmem backend: `AHashMap<EntityKey, Row>` directly. Row
is 32-256 bytes depending on schema. Memory use drops by removing
Value's boxed/enum overhead.

**D-D3:** Snapshot format: bump to v11. Rows encoded with their
schema_id prefix. On load, if the schema_id is known, decode into
typed Row; if not, reject with "schema missing; re-register pipeline".
No schema migration in this phase.

### Area E — Observability

**D-E1:** Keep the 59.5-W3.5 histograms (`beava_shard_push_phase_seconds`
and `beava_shard_event_process_seconds`) unchanged. After 59.6 lands,
these are the primary signal for "did the refactor achieve its
target?" — we should see `phase="pipeline"` drop from 8.5μs avg to
~1.5μs avg.

**D-E2:** Add one new counter: `beava_typed_row_path_total{stream}`
vs `beava_value_fallback_path_total{stream}`. Lets operators see
which path a given stream took over the last observation window.
Should be 100% typed for any registered pipeline post-landing.

### Area F — Migration strategy

**D-F1 (operator-by-operator):** Ship one operator at a time,
behind a compile-time feature flag or runtime check. Order of
landing based on call volume in the reference workload:
1. EnrichFromTable (hot — every cross-shard scenario event)
2. CountOp + LastOp + FirstOp (simple, covers 80% of aggs in real use)
3. SumOp + AvgOp (arithmetic)
4. MinOp + MaxOp
5. DistinctCountOp + PercentileOp + hybrid sketches
6. EmaOp + LagOp (streaming/recurrence)
7. VarianceOp + StddevOp
8. TopKOp + FirstNOp + LastNOp

**D-F2 (parity gate per operator):** every operator, after typed
rewrite, must produce byte-identical output to its `Value` predecessor
on a 10K-event parity harness. Harness seeds both code paths with
the same stream of events and diffs the emitted entity state.

**D-F3 (wire compat):** a stream registered with a typed schema emits
typed rows; a stream registered without emits JSON. Mixed mode is
the default; ops check per-stream.

### Area G — Python SDK

**D-G1:** `@bv.stream` / `@bv.source_table` decorators walk the class
annotations at definition time and build a `Schema` object. New
public attribute `_beava_schema: Schema` on the descriptor. Existing
`fields: {...}` metadata remains for back-compat.

**D-G2:** `_serialize.py::compile_to_register_json` emits a new
`schema` sub-object:
```json
{
  "name": "Txns",
  "kind": "stream",
  "schema": {
    "inline_str_cap": 15,
    "fields": [
      {"name": "user_id", "ty": "inline_str", "offset": 0, "nullable": false},
      {"name": "country_code", "ty": "inline_str", "offset": 16, "nullable": false},
      {"name": "amount", "ty": "f64", "offset": 32, "nullable": false}
    ],
    "row_size": 40
  },
  "shard_key": "user_id"
}
```

**D-G3:** `client.push_many(Txns, batch)` compiles each event dict
into the declared row layout once (fast) and sends a packed batch.
`push_single` (for debug) stays JSON-capable.

</decisions>

<success_criteria>
## Success Criteria

**SC-1 (schema registry wire):** @bv.stream with typed fields produces
a REGISTER payload with `schema: {...}` block; server parses into
`RegisteredSchema`; `engine.is_typed_stream("Txns")` returns true.
Unit test in `tests/register_typed_schema.rs`.

**SC-2 (typed row ingest):** OP_PUSH_BATCH with schema_id header
decodes into a typed Row on the server without allocating a
`serde_json::Value`. Integration test verifies no `Value` alloc
via counter instrumentation.

**SC-3 (EnrichFromTable typed path):** cross-shard enrichment scenario
runs end-to-end on typed rows. Output is byte-identical to the
Value-path reference. Parity harness test.

**SC-4 (aggregation typed path):** CountOp/LastOp/SumOp/AvgOp on
typed rows produce identical entity state to the Value path after
100K events. Parity harness test.

**SC-5 (perf gate):** same cross-shard enrichment scenario as Phase
56/59.5 (Countries + Txns + UserEnrichedStats), now run on typed
pipeline. Per-event shard-thread cost drops from 8.5μs to
**≤ 2μs avg**, measured via `beava_shard_push_phase_seconds{phase="pipeline"}`.
Aggregate throughput ≥ 3M EPS on the reference macOS 10-core laptop.

**SC-6 (fallback path unregressed):** a stream registered without a
typed schema (old SDK path) continues to work via the Value fallback.
Existing tests like `test_v0_register_roundtrip` stay green.

**SC-7 (mixed mode):** two streams registered, one typed, one
untyped, interoperate via a cross-shard EnrichFromTable. Both paths
coexist.

**SC-8 (state store):** inmem + fjall both store typed rows when the
stream has a schema. Snapshot v11 round-trip test.

**SC-9 (parity battery):** full `tests/sharding_parity` N=1 ↔ N=8
parity test suite runs on typed path for all affected streams;
byte-identical output.

**SC-10 (backward-compat snapshot):** v10 snapshots (Value-based)
load successfully and are transparently re-serialized to v11 on
next snapshot cycle.

</success_criteria>

<grey_areas>
## Grey areas to resolve in discuss-phase

1. **Inline string cap (15 bytes default):** covers user_id, country_
   code, most ISO codes, UUIDs in base58. Too small for free-text
   tags (hashtags, product names). Is 15 the right floor, or bump
   to 23/31? Longer = more bytes per row × higher EPS cost trade-off.

2. **Arena allocation for long strings:** per-row `Vec<u8>` arena is
   simple but has allocation cost. Alternative: a shared thread-local
   bump allocator per shard event loop, freed at end-of-batch.
   Simpler vs faster tradeoff.

3. **Schema evolution:** explicit "out of scope" in this phase. But
   users will want it. Do we design Schema with an opaque version
   field now (forward-compat) even if we don't support evolution?

4. **Generic-over-column-type explosion:** some ops (DistinctCountOp
   with hybrid sketch) have enough implementation complexity that
   monomorphizing over 3 column types (i64/f64/str) might blow
   codegen size. Is type-erasure inside those ops acceptable?

5. **Python SDK codegen vs interpret:** at push_many time, do we
   codegen a row-packer per schema (numba/cython style) or interpret
   the schema field-by-field? Simpler to interpret; faster to codegen.
   Probably interpret first, codegen later if push_many becomes a
   client-side bottleneck.

</grey_areas>

<expected_impact>
## Expected impact (why this is worth the scope)

**Per-event:** 8.5μs → ~1.5μs (≈5.7× reduction)
**Per-shard EPS:** ~118K → ~650K
**10-shard aggregate:** ~1.2M EPS → **~6.5M EPS**
**Hetzner 16-core estimate:** ~10-15M EPS per node

This puts Beava at **Materialize-class throughput** on commodity
hardware with the same correctness story (cross-shard enrichment,
retraction, sliding-window aggregations, sharded + replicated tables).

The JSON-on-the-wire API stays at the SDK boundary. Users write
Python classes; the engine runs typed. That's the whole shape.

</expected_impact>
