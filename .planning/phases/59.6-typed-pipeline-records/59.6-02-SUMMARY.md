## Decoder API

`src/wire/typed.rs`:

```rust
pub fn decode_typed_row_push_batch(
    bytes: &[u8],
    schema: &RegisteredSchema,
) -> Result<(Vec<Row>, u64 /*ack_token*/, usize /*bytes_consumed*/), BeavaError>
```

Input body shape (starts AFTER the opcode + stream_name, which the dispatcher has already consumed):

```
[u32 BE schema_id][u32 BE row_count]
[rows: row_size × row_count bytes]
[u32 BE arena_len][arena_bytes]
[u64 BE ack_token]
```

Rejections:
- Short header at any field boundary → `BeavaError::Protocol`
- `schema_id != 0 && schema_id != registered.schema_id` → rejected with actionable "schema_id N does not match" message
- `row_count * row_size` overflow or exceeds payload cap → T-59.6-02-01 DoS defense

Wave 2 simplifications (baked in):
- **Shared-arena clone:** the single arena payload is cloned into every decoded Row's `arena` field. Wave 4 revises to per-row arena splits using the `(start, len)` pointers already in the packed payload.
- **schema_id = 0 shortcut:** clients that haven't plumbed the REGISTER ack's assigned schema_id back can send 0; the decoder trusts the stream_name lookup. Wave 6 tightens once REGISTER ack is extended.

## Wave-2 Row→Value Bridge (Rationale)

`src/engine/schema.rs::row_to_value` converts a typed `Row` back into a `serde_json::Value` by walking the schema's field list and emitting the appropriate `Value` variants. The server-side dispatcher (`handle_push_typed_batch` in `src/server/tcp.rs`) uses this bridge to render each decoded Row into JSON bytes, then feeds those bytes through the existing `PayloadFmt::Json` shard path.

**Why a bridge?** Wave 2's scope is wire-level correctness, not operator rewrite. Shipping wire-aware schemas without changing operator execution lets us validate:

1. OP_PUSH_TYPED_BATCH frames deserialize correctly against a registered schema.
2. The `typed_row_path_total` counter can be trusted to measure typed-path traffic.
3. Clients can negotiate the typed opcode end-to-end and land the PLAN — all without a single operator change.

**Why not direct handoff?** Operators consume `serde_json::Value` today. Threading a `Row` through would require simultaneously rewriting `SumOp`, `LastOp`, `EnrichFromTable`, etc. — that's Wave 3+'s scope.

**When is the bridge removed?** Wave 3 lands `EnrichFromTable` on the typed path and introduces a `ShardOp::TypedPush { schema_id, payload_bytes, arena }` variant. The listener-side `row_to_value` + JSON re-serialize call sites go away; the shard thread reads field values from `Row` directly via `read_i64`/`read_f64`/`read_inline_str` on the hot path. At that point the `beava_typed_row_path_total` counter reflects genuine serde_json::Value-free traffic.

**Invariant preservation:** the re-serialize goes through `crate::wire::reserialize_value_to_json_bytes` (the single sanctioned re-serialize helper from Phase 59). `scripts/verify-no-tcp-json-reserialize.sh` still exits 0 because it greps for `serde_json::to_vec(payload)` / `serde_json::to_vec(r.payload)` literals, and we avoid those patterns.

## Python SDK v0.3.0 Release Notes

**New public surface:**

- `beava._protocol.OP_PUSH_TYPED_BATCH = 0x19` — opcode constant.
- `beava._protocol.WIRE_TYPED_PIPELINE = 2` — capability bit the server advertises for Wave 2+ support.
- `beava._client.BeavaClient._pack_typed_batch(stream_name, schema_json, events, ack_token, schema_id=0) -> bytes` — byte-for-byte mirror of the Rust decoder.

**Behavioral change (transparent):** `App.push_many(stream, events)` now checks:

1. Did the server advertise `WIRE_TYPED_PIPELINE` during `negotiate_wire_format()`?
2. Does `stream` have a compiled `_beava_schema` attribute (attached by Wave 1's `@bv.stream` / `@bv.source_table` / `@bv.table` decorators)?

If BOTH yes → emit `OP_PUSH_TYPED_BATCH` (0x19). Otherwise → legacy `OP_PUSH_BATCH` (0x0A). Users with un-annotated streams and users talking to pre-59.6 servers continue to work unchanged.

**Version bump:** 0.2.0 → 0.3.0.

### Rule 3 — unblock test compile via salt sweep

- **Found during:** Task 1 verification run.
- **Issue:** `cargo test --release --test binary_push_bytes_passthrough` (and `--test json_over_tcp_still_accepted`) failed with `E0063: missing field 'salt' in StreamDefinition` — the pre-existing Phase 60 salt-sweep leakage documented in Wave 0's `deferred-items.md`. Wave 2's must-haves include these two tests staying GREEN as Phase 59 D-B3 / no-reserialize regression guards.
- **Fix:** Added `salt: None` to the `StreamDefinition { ... }` literals in both test files. Tiny targeted change (2 lines each); leaves the 40+ other `StreamDefinition {}` literals scattered across tests/ and src/ for the Phase 60 executor to sweep whole.
- **Files modified:** `tests/binary_push_bytes_passthrough.rs`, `tests/json_over_tcp_still_accepted.rs`.
- **Commit:** `f3dfd86` (Task 1).

### Rule 2 — handle TypedRow in decode_event_on_shard

- **Found during:** Task 1 first build.
- **Issue:** Adding `PayloadFmt::TypedRow = 2` broke the non-exhaustive `match fmt` in `src/wire/binary.rs::decode_event_on_shard`. Compiler refused to build.
- **Fix:** Added `PayloadFmt::TypedRow` arm routing to the JSON decoder (the listener has already rendered Row → JSON bytes via the Wave-2 bridge, so on-shard decode is identical to `Json`). This keeps the Wave-2 bridge fully transparent to the Phase 59 D-C1 decode-on-shard contract.
- **Files modified:** `src/wire/binary.rs`.

### Rule 2 — schema_id field on every ShardEvent struct-literal site

- **Found during:** Task 1 first build after adding `schema_id: u32` to `ShardEvent`.
- **Issue:** 14 call sites across `src/shard/thread.rs`, `src/engine/pipeline.rs`, `src/engine/cascade_target.rs`, `src/state/eviction.rs` failed to compile — they use struct-literal construction, so a new required field fails the compile.
- **Fix:** `schema_id: 0` added everywhere. Applied via 4 targeted `Edit` calls (two `replace_all` passes at 8-space and 24-space indent, plus two one-off edits for the odd-indent sites). Zero behavior change for non-TypedRow events (they all set schema_id = 0).
- **Files modified:** see above.

### Wave-2 pattern adjustment — Command::PushTypedBatch captures body as owned Vec

- **Found during:** Task 1 design.
- **Issue:** `parse_command` takes `&[u8]` and must not hold a reference into it past the caller frame. For OP_PUSH_BATCH this is fine because every event is decoded synchronously inside parse_command. For OP_PUSH_TYPED_BATCH we want the decode to run in the dispatcher (where the engine read guard is available for the schema lookup), not inside parse_command.
- **Fix:** Introduced a `Command::PushTypedBatch { stream_name, body: Vec<u8> }` variant that carries the post-stream-name body bytes forward to the dispatcher. Single allocation per batch — acceptable at Wave 2 given the JSON-bridge overhead already dominates. Wave 3 replaces this with a `Bytes`-backed zero-copy carrier.
- **Files modified:** `src/server/protocol.rs`.
