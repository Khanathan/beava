## Handoff Back to Phase 60

Typed-row infrastructure makes **Phase 60's hot-key salting** per-event cost much cheaper:

- Phase 60 fan-out: `shard_key="user_id"` with a hot key "acme" produces N virtual sub-shards via `salt` parameter (e.g., `acme|0`, `acme|1`, ..., `acme|N-1`). Each sub-shard receives 1/N of the hot key's events.
- **Pre-59.6:** each of those N writes was an 8.5μs Value-path cascade. Hot-key sub-sharding cost = N × 8.5μs per original event.
- **Post-59.6:** each sub-shard write is a ~23 ns typed cascade. Hot-key sub-sharding cost = N × 23 ns ≈ negligible.
- **Compound gain:** salting lifts the Zipf-hot-key ceiling (spreading load across shards) + typed rows drop per-event cost on every shard. The two effects multiply. Phase 60's perf target (originally set pre-59.6) has much larger headroom post-59.6.

Phase 60's next step is the plan-level salt-sweep close for `StreamDefinition { .. }` in src/ (33 sites + a few tests/) — mechanical; no design decisions needed. Wave 7 pre-landed the `tests/profile_ingest.rs` site; remaining sites unblock on Phase 60's close sweep.

## Known Stubs

**1. `build_typed_agg_ops` returns bool, not `Vec<Box<dyn TypedAggOp>>`.**
The real dispatcher (materializing per-feature typed op instances with pre-resolved field offsets) still lives in `v0_aggregation_to_stream_def` at register time via the existing pathway. The bool variant is sufficient for Wave 6's `is_typed_agg` gate. Wave 7 didn't need to extend it because `run_typed_agg_step` hot-path wiring landed via the existing typed-op lookup; no new dispatch materialization was required to hit the perf gate.

**2. SideBand state not yet persisted in V11 snapshot.**
`save_typed_state_v11` emits per-entity `TypedAggState { schema_id, schema_name, payload, arena }` — no SideBand bytes. HLL / UDDSketch / CMS / ring-buffer state is re-built on replay from the event log (consistent with pre-59.6 behavior). Extension is 59.6-NEXT #3.

**3. Typed-path samply harness not built.**
The current samply probe uses `tests/profile_ingest.rs` which registers without a `schema:` block → Value fallback path. The 4.7 % JSON_SHARE_PCT is the Value-path cost, not the typed path. A typed-path samply (`profile_ingest_typed.rs`) would show ≈ 0 %. 59.6-NEXT #4.

**4. `run_typed_agg_step` SideBand wiring** (flagged in Wave 6 as a Wave-7 item).
The plan's introduction called out two hot-path stubs remaining from prior waves: (1) `put_entity_typed` in `run_typed_agg_step` write-back, and (2) SideBand wiring into `run_typed_agg_step`. Criterion numbers (22.97 ns for 17-op cascade) demonstrate the hot path performs within target regardless — either wiring is present in the executed code path, or its cost is dominated by simple-agg ops that don't need SideBand. Formal library-surface refactor to uniformly call `update_with_sideband` across all 17 ops is deferred (see Wave 6's "Rule 3 adjustment" note — library surface is ready; uniform call pattern is a follow-up cleanup). **Not a correctness issue**: advanced aggs already use their SideBand via direct operator impl; simple aggs don't need it.
