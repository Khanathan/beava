## Design Decisions (inline-string slot width)

Resolved one grey area in CONTEXT.md: the example table shows InlineStr fields at offsets 0 and 16, implying a 16-byte slot per field at `inline_str_cap=15`. Decision: **the slot is always `inline_str_cap + 1` bytes** — the trailing byte holds a NUL terminator so readers can recover strings shorter than the cap without per-row length tracking. This is cemented in both `Rust engine::schema::FieldTy::fixed_width` and Python `_schema_compile._field_width`, with matching unit tests on both sides.

## REGISTER JSON shape change

**Before (pre-59.6, still valid):**
```json
{"name": "Txns", "kind": "stream",
 "fields": {"user_id": {"type": "str", "optional": false}, "amount": {"type": "float", "optional": false}},
 "shard_key": "user_id"}
```

**After (typed SDK, additive):**
```json
{"name": "Txns", "kind": "stream",
 "fields": {...},            // unchanged
 "shard_key": "user_id",     // unchanged
 "schema": {                 // NEW
   "inline_str_cap": 15,
   "fields": [
     {"name": "user_id", "ty": "inline_str", "offset": 0, "nullable": false},
     {"name": "amount",  "ty": "f64",        "offset": 16, "nullable": false}
   ],
   "row_size": 24
 }}
```
