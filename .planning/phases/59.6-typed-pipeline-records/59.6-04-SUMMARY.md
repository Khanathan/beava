key-decisions:
  - "Per-column-type struct specialization over generic-over-T: Min/Max/Sum are written as `*I64` + `*F64` variants rather than a single `MinOp<T: AggNum>` impl. Rationale: D-C1 explicitly warns against monomorphization blowup once the column type matrix grows to 6 (i64, f64, bool, inline_str, string, bytes). Each struct is thin (~30 lines) and strictly #[inline]-friendly in the hot path."
  - "Min/Max empty-state via seen_offset bool, not sentinel comparison on the value: the Value-path operators.rs MinOp/MaxOp report Missing before the first event. Mirroring that required either (a) reserving a sentinel value (f64::NAN or similar) or (b) a side bool. Chose (b) — a 1-byte flag is cheaper than NaN-handling arithmetic and keeps Row.read_f64 semantics identical to scalar reads elsewhere."
  - "Avg uses sum + count in state, returns sum/count at read_feature: mirrors the Value-path AvgOp behavior. Empty state returns FeatureValue::Missing (count == 0 ⇒ no division)."

```

   → Shard thread process_shard_event → ShardOp::PushTypedRow { row, schema_id }
   → engine.push_typed_on_shard(stream_name, row, &schema, shard, ...)
        ├─ cascade empty → Value fallback (row_to_value + push_with_cascade_on_shard)
        ├─ cascade has variant NOT in is_wave4_typed_compatible → Value fallback
        └─ cascade = EnrichFromTable | Count | Sum | Avg | Min | Max | Last | First (Wave-4 set)
             → run_typed_enrich_cascade  (Wave-3 bridge still in use; Wave 5 replaces)
             → (future) run_typed_agg_step
                  ├─ shard.get_or_init_entity_state_typed(stream, key, state_schema, ops)
                  │     ├─ first call: Row::zeroed + for op in ops: op.init_state(state)
                  │     └─ subsequent: AHashMap lookup (no alloc)
                  ├─ for op in ops: op.update_typed(state, schema, event, ev_schema, now)
                  │     → offset read + arithmetic + offset write
                  └─ for op in ops: fmap.insert(op.name(), op.read_feature(state, schema))
```

## Deviations from Plan

### Rule 3 — Trait-object slice shape (`&[&dyn]` vs `&[Box<dyn>]`)

- **Found during:** Task 2 wiring of `get_or_init_entity_state_typed` and `run_typed_agg_step`.
- **Issue:** Plan pseudocode uses `init_ops: &[Box<dyn TypedAggOp>]`. Callers (SC-4 tests, future cascade plan cache) hold concrete op values as locals; boxing them forces per-op heap allocations.
- **Fix:** Accept `init_ops: &[&dyn TypedAggOp]` — a slice of trait-object references. Callers pass arrays like `[&count as &dyn TypedAggOp, &sum, ...]`. No allocation.
- **Impact:** Downstream call sites are marginally cleaner; identical semantics. Documented in the method's docstring.

### Plan-body adjustment — cascade still bridges to Value path

- **Found during:** Task 2 dispatch section.
- **Issue:** Plan says "`push_typed_on_shard`'s `wave3_compatible` check expands to Wave 4: EnrichFromTable | Count | Sum | Avg | Min | Max | Last | First — and the matched ops run via `run_typed_agg_step` directly on `Shard.entity_state_typed`". Wave 4 expands the predicate (done) but still bridges through `run_typed_enrich_cascade` → Value cascade walk. A direct typed cascade walk that threads `Shard.entity_state_typed` into the per-target dispatch requires the cascade-plan pre-derivation work scoped for Wave 5.
- **Fix:** `is_wave4_typed_compatible` widens correctly; `run_typed_agg_step` is exposed as a public method that SC-4 tests drive directly. The cascade walk itself still goes through `run_typed_enrich_cascade` (which Wave 3 established as the Value-bridge) for any Aggregation variant. This keeps Wave 3's SC-3 parity GREEN while letting Wave 4's TypedAggOp implementations be exercised + measured.
- **Impact:** Wave 4 satisfies SC-4 (operator-boundary parity for 7 simple aggs) without landing Wave 5's cascade-plan pre-derivation dependency. Wave 5 replaces the `run_typed_enrich_cascade` bridge with a direct typed cascade walk.
