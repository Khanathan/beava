<objective>
Ship the register-time retraction-impact analyzer. Walk the DAG from every
modifiable `@bv.event` source to every downstream aggregation feature,
classify each feature by its `AggKind::tier()`, and emit structured warnings
or errors per D-07/D-08.

Extend `POST /register` response body with `warnings: [...]` and `errors:
[...]` arrays. Tier B offenders flip the status to 400 and block registration
per D-09.

Python SDK raises `BeavaRetractionWarning` (subclass of `Warning`) via
`warnings.warn()` for every warning entry in the response.

Ship three docs stubs on the website under `/w/<code>` matching the `docs_url`
in the warning payload.

Purpose: users learn about operator-correctness consequences of
`modifiable=True` **before** their first push — not after silent drift shows
up in a production dashboard.

Output: `retraction_analyzer.rs` new module + register response shape +
Python warning class + three docs stubs + integration smokes for all three
tier paths.
</objective>

    Create `crates/beava-core/src/retraction_analyzer.rs`:
    ```rust
    use crate::agg_op::{AggKind, Tier};
    use crate::registry::Registry;

    #[derive(Debug, Clone, PartialEq, Serialize)]
    pub enum Severity { Info, Warn, Error }

    #[derive(Debug, Clone, PartialEq, Serialize)]
    pub struct AnalyzerFinding {
        pub severity: Severity,
        pub code: &'static str,
        pub feature: String,
        pub source_chain: Vec<String>,
        pub why: String,
        pub usually_fine_when: String,
        pub worry_when: String,
        pub action: String,
        pub docs_url: String,
    }

    pub fn analyze_register(reg: &Registry, verbose: bool) -> Vec<AnalyzerFinding> { ... }
    ```

    Algorithm:
    1. For each `EventDescriptor` in the registry where `modifiable=true`:
       - BFS/DFS through derivations whose `upstreams` transitively reach
         this source (reuse existing DAG walk if present; add if not).
       - Collect every aggregation feature whose `source_node_name` is in the
         reachable set.
    2. For each reached feature:
       - Look up its `AggKind`; call `.tier()`.
       - Materialize an `AnalyzerFinding` with the canonical prose per code:
         - `Tier::C` → Info, code `BV-W-AGG-SUBTRACTIVE-OK` (included only if
           verbose).
         - `Tier::A` → Warn, code `BV-W-AGG-APPROX-MODIFIABLE`.
         - `Tier::B` → Error, code `BV-W-AGG-REJECT-MODIFIABLE`.
         - **Override:** if `AggKind::CountDistinct` → always Warn
           `BV-W-AGG-APPROX-MODIFIABLE` with the D-10 prose about adaptive
           promotion.
    3. Populate `source_chain` by walking parent links back to the modifiable
       source. If a feature reaches the modifiable source through multiple
       paths, emit one finding per unique chain (document but unlikely in
       practice).

    Canonical prose dictionaries (static &str) for each code. Include the
    exact `why` / `usually_fine_when` / `worry_when` / `action` strings from
    CONTEXT D-07 — these become authoritative and docs stubs quote them
    verbatim.
  </action>
