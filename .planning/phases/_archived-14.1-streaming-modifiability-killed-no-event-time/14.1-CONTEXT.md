# Phase 14.1 — Streaming Semantics Chunk B (Opt-in Modifiability)

## Purpose

Ship **opt-in** modifiability for `@bv.event` sources. Users who declare
`@bv.event(modifiable=True, modification_log_depth=16)` get insert-replay
for out-of-order events that land **within** the Phase 14 tolerance window.
Everyone else pays nothing — `modifiable=False` is the default and the apply
path is unchanged from Phase 14 for those streams.

Ship alongside a **register-time retraction-impact analyzer** that classifies
every operator used downstream of a modifiable stream into one of three tiers
(Subtractive-exact / Approximate / Deterministic-reject) and surfaces a
`BV-W-AGG-*` warning (or registration error) to the user before they ever
push an event.

Out of scope (explicitly): table modifiability (Phase 15/17), event-time PIT
temporal store (Phase 15), `@bv.source` annotation (Phase 16), table aggregation
with tiered modifiability (Phase 17).

## Background — What Phase 14 gives us

Phase 14 ships:
- Per-stream watermark state on `AppState` (inside `borrow_mut`).
- Front-door drop: events with `event_time < stream_watermark - tolerate_delay_ms`
  are silently dropped, counted, and logged (rate-limited).
- Bucket widening `bucket_ms = (window + tolerate_delay) / 64` in
  `WindowedOp::new*` — within-tolerance late events can no longer bucket-reset.
- `tolerate_delay_ms` is threaded through `agg_compile.rs` into every compiled
  `WindowedOp`.

Phase 14 does NOT:
- Allow the apply path to *see* late events for any operator (everything within
  tolerance still goes through a plain `update` — commutative ops are fine,
  order-sensitive ops silently corrupt or produce best-effort results).
- Provide any per-(entity, feature) modification log.
- Surface per-operator correctness guarantees to the user at register time.

14.1 fills those gaps — only for streams whose source author explicitly opts
in via `modifiable=True`.

## Locked Decisions

### D-01 — Modifiability is OPT-IN, default OFF
`@bv.event(modifiable=False, ...)` is the default. Phase 14's front-door drop
remains the canonical behaviour for 99% of streams: out-of-order events within
tolerance go through a plain commutative apply; out-of-tolerance events drop.

Only when a source declares `@bv.event(modifiable=True)` does 14.1 machinery
(modification log, insert-replay, analyzer warnings) activate. The cost budget
for non-modifiable streams is hard: **≤ 5 ns additional apply-path cost per
push vs Phase 14 baseline** (enforced by criterion microbench in Plan 06).

### D-02 — Terminology: user-facing "modifiable" / internal "Retracted"
All user-facing prose, SDK arg names, docs pages, error codes, and warning
messages use **"modifiable" / "change or delete"**. Internal Rust type names
(`RecordType::Retract`, `Retracted` enum variants, `app.retract()` primitive
shipped in Phase 11.5) are implementation detail and preserved as-is to avoid
a gratuitous rename churn. Docs URL namespace is `https://beava.dev/w/<code>`.

Warning codes use the `BV-W-AGG-*` prefix (W = warn-tier, AGG = aggregation
analyzer). User-visible content says "change or delete" wherever possible.

### D-03 — Schema additions: `modifiable` + `modification_log_depth`
`@bv.event` gains two new kwargs:
- `modifiable: bool = False` — opt into the modifiability machinery.
- `modification_log_depth: int = 16` — ring-buffer depth for the per-(entity,
  feature) modification log, Tier 3 operators only. Must be ≥ 1. Hard ceiling
  **256** (rejected at register time — beyond that, memory scales pathologically).

`EventDescriptor` gains two new fields (additive-only — pre-14.1 payloads
deserialize with defaults, preserving Phase 2 registry-version semantics):
```rust
pub struct EventDescriptor {
    // ... existing fields ...
    #[serde(default)]
    pub modifiable: bool,
    #[serde(default = "default_modification_log_depth")]
    pub modification_log_depth: u16, // default 16
}
```

Per-op override (e.g. `bv.ewma(half_life="1h", modification_log_depth=32)`) is
parsed, validated (must be ≥ 1 and ≤ 256), stored in the per-feature descriptor,
and takes precedence over the stream-level default at log-allocation time.

### D-04 — Log allocation scope: per-(entity, feature), lazy, Tier 3 only
The modification log is **per-(entity, feature)** — not per-stream, not per-
entity — and is lazy-allocated on the first event that would need to insert
into it. **Only Tier 3 operators allocate** — Tier 1 operators (commutative)
never touch the log path, Tier 2 operators use their existing sorted/retracting
state and do not allocate a separate log.

Log shape: fixed-capacity ring buffer of `ModificationLogEntry { event_time_ms:
i64, event_id: u64, row: Row }`. Capacity = `modification_log_depth` (default
16). When full, oldest event is evicted. If an out-of-order insert lands
*before* the oldest-retained event, the insert is **rejected at the apply
boundary** (not silently dropped — `beava_modification_log_evicted_total{feature}`
increments and the event is logged at WARN rate-limited like Phase 14 drops).

### D-05 — Three tiers (operator classification catalogue)

The register-time analyzer classifies every `AggKind` used downstream of a
modifiable stream into one of three tiers. See `.planning/phases/14.1-
streaming-modifiability/TIER-CATALOGUE.md` for the full mapping; the canonical
table is reproduced here so CONTEXT is the source of truth.

#### Tier C (Subtractive-exact) — value is exact after insert-replay
Commutative under addition or subtractive-deletion. Insert-replay produces the
same state as in-order arrival. **No per-op redesign needed**; the tier label
is informational only (`BV-W-AGG-SUBTRACTIVE-OK`, verbose-only, suppressed by
default).

| AggKind | Rationale |
|---|---|
| `Count` | Σ 1; commutative |
| `Sum` | Σ x; commutative |
| `Avg` | Sum / Count; both commutative |
| `Variance` / `StdDev` | Welford on aggregated (n, mean, m2) is commutative via parallel-merge identity |
| `Ratio` | num/denom, both counts — commutative |
| `Histogram` / `HourOfDayHistogram` / `DowHourHistogram` | bucket counts; commutative |
| `EventTypeMix` | per-type count; commutative |
| `DecayedSum` / `DecayedCount` | semi-group-mergeable (decay factor applied at query time via last-event-time — see implementation note in D-06 slow path) |
| `BurstCount` (w/ windowed aggregation) | Count over sliding bucket; commutative within bucket |
| `CountDistinct` **in ExactArray/HashSet modes only** (≤ 1024) | set union is commutative |
| `BloomMember` | bit-set union is commutative |
| `Ewma` / `EwVar` / `EwZScore` | **Tier C ONLY when compiled with a subtractive-Cormode variant.** Current `EwmaState` at `agg_state_decay.rs:71` is **Tier A** (time-order sensitive; half-weighted when late event arrives — see `update` line 108). 14.1 does **not** swap the EWMA to subtractive-Cormode; ewma stays in **Tier A** for 14.1. Upgrade to subtractive-exact is a v0.1 follow-on. |
| `Twa` | time-weighted average; Tier C when stored as Σ(value×dt) with explicit last-event-time. Current `TwaState` at `agg_state_decay.rs:326` is Tier C-compatible *if* replay reconstructs `last_event_time_ms` correctly. Covered by the slow-path replay in Plan 05. |

Default analyzer behaviour for Tier C: **no warning surfaced** (informational
note only if `--verbose` is passed at register time). Plan 03 implements the
`BV-W-AGG-SUBTRACTIVE-OK` note but the register response body omits it unless
explicitly requested.

#### Tier A (Approximate, best-effort, bounded drift) — insert-replay OK, error bound grows
Approximate algorithms where insert-replay is well-defined (the sketch accepts
the out-of-order event) but the error bound is slightly worse than pure
in-order arrival. Users get a **WARN** (`BV-W-AGG-APPROX-MODIFIABLE`) at
register time with the specific drift bound.

| AggKind | Rationale |
|---|---|
| `CountDistinct` in **HLL mode** (promoted past 1024) | HLL register updates are commutative set additions — result is idempotent under any arrival order. Drift comes from the promotion boundary itself: if an out-of-order event would have kept the entity in ExactArray mode but promotion already happened, the HLL estimate stands. |
| `Percentile` (UDDSketch) | bucket-keyed sketch; insert is commutative on bucket counts but quantile estimate shifts with bucket granularity; documented ε = 0.01 rel-err |
| `TopK` (SpaceSaving) | counter-promotion is order-sensitive; insert-replay can evict a since-re-inserted tail counter — drift bounded by `min_frequency / k` |
| `Entropy` | Σ p log p over observed types; commutative on counts but floating-point re-aggregation drift (ε ≈ 1e-10) |
| `SeasonalDeviation` | baseline histogram update is commutative; deviation query against mean is stable |
| `Ewma` / `EwVar` / `EwZScore` (as shipped today) | time-order sensitive — late event is half-weighted (see `agg_state_decay.rs:108`). Bounded drift because half-life provides natural attenuation of old observations. Stays Tier A until the Cormode-subtractive upgrade lands (v0.1). |

Default analyzer behaviour: **WARN** with `action: "If approximation is
acceptable for this feature, no change needed. If exact values required on
modifiable streams, use Tier C operator or Phase 17 table aggregation."`

Adaptive `count_distinct` nuance (D-10): because the user doesn't know at
register time whether any given entity will promote to HLL mode, the warning
**always fires** on modifiable streams. Runtime metric
`beava_feature_promoted_to_approx_total{feature}` emits on actual promotion
events.

#### Tier B (Deterministic-reject) — insert-replay is ill-defined
Operators that encode **position or ordering** explicitly. Insert-replay would
require re-reading the full event history from the source — not the per-
(entity,feature) K-event log — so 14.1 rejects these at register time.

| AggKind | Rationale |
|---|---|
| `Min` / `Max` | Current value is `min/max(x_i)` but *time at which it was observed* is part of query semantics via `MinState`/`MaxState` timestamps. Monotonic; insert-replay only works if we re-scan ALL events, not K recent ones. |
| `First` / `Last` | Pinned to first- or last-observed event_time. Late insert would re-pin First; silently invalidates any already-emitted downstream read. |
| `FirstN` / `LastN` | Top-N by time — same reason |
| `Lag` | Value N events ago — position-encoded |
| `MostRecentN` / `ReservoirSample` | Sampling by position/recency |
| `FirstSeen` / `LastSeen` / `Age` / `HasSeen` | Timestamp markers; insert-replay moves timestamps |
| `FirstSeenInWindow` | Same as FirstSeen in window scope |
| `TimeSince` / `TimeSinceLastN` | Derived from last-event timestamp |
| `Streak` / `MaxStreak` / `NegativeStreak` | Sequence-order encoded in state |
| `GeoVelocity` | `Δdistance / Δt` between consecutive events — sequence-order encoded |
| `DistanceFromHome` | Requires first-observed "home" position — insert-replay would move home |
| `Trend` / `TrendResidual` | Linear regression over ordered (t, x) pairs — inserting a middle point invalidates slope |
| `RateOfChange` | `Δx/Δt` between adjacent events |
| `InterArrivalStats` | Mean/stddev of inter-arrival times |
| `DeltaFromPrev` | Previous-event encoded |
| `OutlierCount` | Cumulative deviation count requires ordered scan |
| `ValueChangeCount` | Same |
| `ZScore` | Against running mean/stddev — order-sensitive accumulation |

Default analyzer behaviour: **ERROR** (`BV-W-AGG-REJECT-MODIFIABLE`). Register
returns 400 with a structured diagnostic per offending feature:
```
action: "Remove `modifiable=True` from the source, use a Tier C/A operator,
 or model this as a table in Phase 15."
```

The user must either remove `modifiable=True`, swap the operator, or (if they
understand the consequences) pin to a different source that's not modifiable.

### D-06 — Fast path vs slow path (runtime mechanics)

Inside the push handler's `borrow_mut` on `AppState`, after the Phase 14
front-door drop has passed:

```
fn apply_with_modifiability(event, entity_key, descriptor):
    source = registry.event(source_name)
    if !source.modifiable:
        # Fast path identical to Phase 14. Zero additional cost.
        return apply_event_to_aggregations(row, event_time_ms, ...)

    max_for_entity = entity_row.max_event_time  # O(1) tracked per-entity
    if event_time_ms >= max_for_entity:
        # Fast path even when modifiable=true — normal forward progress.
        # Append to modification log (Tier 3 features only).
        for tier3_feature in desc.tier3_features:
            entity_row.mod_log[feature].push(entry)
        apply_event_to_aggregations(...)  # plain update
        entity_row.max_event_time = event_time_ms
    else:
        # Slow path: event_time_ms < max_for_entity but >= watermark.
        # (Phase 14 already dropped events past watermark; we get here only
        # within the tolerance window.)
        #
        # For Tier 3 features: insert into mod_log, rebuild feature from log.
        # For Tier 1/Tier 2 features: plain update (commutative).
        for feature in desc.features:
            if feature.tier == 3:
                entity_row.mod_log[feature].insert_sorted(entry)
                entity_row.state[feature] = replay_tier3(
                    entity_row.mod_log[feature],
                    feature.descriptor,
                )
            else:
                feature.state.update(...)  # commutative, no replay needed
```

**Cost model:**
- Fast path, `modifiable=False`: identical to Phase 14. Measured budget
  **≤ 5 ns overhead** vs Phase 14 baseline (one extra predicate check on a
  field that's already in-cache).
- Fast path, `modifiable=True`: adds O(log depth) work for the Tier 3 log
  push — typically 1-2 cache lines per log entry, ring-buffer write. Budget
  **≤ 50 ns overhead** per Tier 3 feature per push.
- Slow path: O(depth) replay of Tier 3 features. For `modification_log_depth=16`
  that's 16× the per-op update cost. Budget: **≤ 5 µs per slow-path event per
  Tier 3 feature** on Apple-M4. Documented as an accepted v0 latency spike.

### D-07 — Register-time warnings (`BV-W-AGG-*` taxonomy)

Each warning has: `code`, one-line `subject`, `feature` (name), `source_chain`
(stream→derivation→feature path), `why` (human prose, 1 sentence), 
`usually_fine_when` (prose), `worry_when` (prose), `action` (prose),
`docs_url` (`https://beava.dev/w/<lowercase-code>`).

CLI rendering (5 lines max per warning):

```
BV-W-AGG-APPROX-MODIFIABLE  feature=p99_amount_5m  (stream→users→tx_flat→p99)
  percentile(UDDSketch) is approximate with bounded drift (ε ≈ 0.01) under
  out-of-order insert. Usually fine; worry if downstream tripwires treat the
  quantile as an exact bound. See https://beava.dev/w/bv-w-agg-approx-modifiable
```

Three codes ship in 14.1:

| Code | Tier | Severity | Register response |
|---|---|---|---|
| `BV-W-AGG-SUBTRACTIVE-OK` | C | info | `warnings: [...]` (verbose only) |
| `BV-W-AGG-APPROX-MODIFIABLE` | A | warn | `warnings: [...]` |
| `BV-W-AGG-REJECT-MODIFIABLE` | B | error | `errors: [...]` + 400 |

### D-08 — Analyzer output format (register response body)

`POST /register` response body gains two new fields:

```json
{
  "registry_version": 42,
  "registered_descriptors": [...],
  "warnings": [
    {
      "code": "BV-W-AGG-APPROX-MODIFIABLE",
      "feature": "p99_amount_5m",
      "source_chain": ["tx_stream", "tx_flat", "p99_amount_5m"],
      "why": "percentile is approximate under out-of-order arrival (ε ≈ 0.01)",
      "usually_fine_when": "quantile used as ranking signal, not hard bound",
      "worry_when": "quantile gates downstream tripwires or contract SLAs",
      "action": "Accept ε, or move to Phase 17 table aggregation",
      "docs_url": "https://beava.dev/w/bv-w-agg-approx-modifiable"
    }
  ],
  "errors": []
}
```

On any Tier-B offender, `errors` is non-empty and the response is **400** with
`error.code = "registration_conflict"` + per-feature entries populated in
`errors` (backwards-compatible with Phase 2 error shape — `warnings` is
additive, `errors` replaces the single-diagnostic shape when present).

Python SDK surfaces warnings as `BeavaRetractionWarning(Warning)` subclass
raised via `warnings.warn(...)` after register completes. Errors remain
`BeavaRegistrationError`.

### D-09 — Tier B + modifiable stream = hard error (not warning)

Tier B on a modifiable stream is ERROR, not WARN. Rationale: Tier B ops produce
*silently wrong* results under insert-replay (moving Min to a late event,
re-pinning First, etc.) — the user cannot choose "I accept the wrong value."
They must either remove `modifiable=True`, swap the operator, or model the
source as something immutable.

(Counter-intuition for v0: we could imagine "Tier B = warn, silently drop late
events for these ops" but that's confusing — the same stream would have
per-feature drop policies. Hard error is simpler.)

### D-10 — Adaptive `CountDistinct` analyzer nuance

`CountDistinct` transitions through three modes at runtime:
1. **ExactArray** (≤ 128 uniques) — Tier C, insert-replay exact.
2. **HashSet** (≤ 1024 uniques) — Tier C, insert-replay exact.
3. **HLL** (> 1024 uniques) — Tier A, insert-replay bounded-drift.

At register time the analyzer doesn't know which mode any given entity will
end up in. Policy: **always emit `BV-W-AGG-APPROX-MODIFIABLE` for CountDistinct
on a modifiable stream**, with prose:

```
count_distinct is EXACT while entity cardinality ≤ 1024, then promotes to
HLL (ε ≈ 1.04 / sqrt(m) ≈ 1.6% with default precision). Under modifiable
streams, HLL-mode values are best-effort; ExactArray/HashSet modes are exact.
Runtime metric: beava_feature_promoted_to_approx_total{feature}
```

When an entity actually promotes (ExactArray/HashSet → HLL) inside a modifiable
stream, increment `beava_feature_promoted_to_approx_total{feature}` once per
promotion event. Declared in Plan 06 alongside other Phase 13 metrics.

### D-11 — WAL format: modification log recovery

When an event triggers slow-path insert-replay, the WAL must preserve enough
state for recovery to reproduce the replay. Two options were considered:

- **Option A (rejected):** New `RecordType::Modify = 0x06` that points at a
  prior event's `event_id` and carries the inserted event. Adds two record
  reads per slow-path event on recovery (look up target + apply insert).

- **Option B (accepted):** Re-use the existing `RecordType::Event = 0x01`
  record unchanged. The WAL is a linear log of raw push events; recovery
  replay reconstructs the per-(entity,feature) modification log by re-running
  the same fast/slow-path logic in `apply_with_modifiability`. Out-of-order
  events in the WAL literally **are** the modification history.

Option B is correct because Phase 14's apply path is already a pure function
of the event stream (D-06 of Phase 14). The modification log state is derived,
not primary — no new WAL record variant needed. Phase 11.5's `event_id_index`
is sufficient to tie a downstream `app.retract(event_id)` (table scope) to a
specific upsert; stream retraction (future v1) would still need a new record
variant but that's out of 14.1 scope.

Snapshot format: the per-entity modification logs are part of `AggStateTable`
snapshot state (Phase 7 format). Adds `mod_log: Option<RingBuffer<ModLogEntry>>`
next to each Tier 3 feature state. Snapshot format version bumps one step
(v2 → v3). Pre-v3 snapshots load with empty mod_logs — recovery re-materializes
them from WAL replay.

### D-12 — Memory constraint (documented, not enforced)

A per-(entity, feature) K=16 event log at ~100 B per entry = **~1.6 KB per Tier
3 feature per entity**. For a rich 30-feature pack with ~4-5 Tier 3 features
that's **~7 KB/entity** of modification log on top of the ~6 KB/entity Tier 1+2
state — roughly **13 KB/entity total**, ~1.9× the CLAUDE.md 7 KB/entity budget.

Policy: **document, do not enforce**. Users enabling stream-modifiability
self-select into larger memory footprint; the docs page at
`beava.dev/docs/modifiability` explicitly warns about the budget blow-up and
shows the math. Capacity-estimation tooling in Phase 13 gains a
`modifiable_entities` input parameter.

### D-13 — Phase 14 drop boundary is inherited, unchanged

Insert-replay **only** happens for events `event_time_ms ∈ [watermark,
max_event_time_for_entity)`. Events with `event_time_ms < watermark` are
dropped at the front door by Phase 14 — 14.1 does NOT relax this. Users who
want a longer tolerance horizon on a modifiable stream increase
`tolerate_delay_ms` at the source — Phase 14's watermark check moves with it.

### D-14 — Phase 13.3 apply-path assumption

14.1 lands after Phase 13.3's `Arc<LocalState<RefCell<AppState>>>` refactor.
The modification log and its insert-replay loop live **inside the
`borrow_mut`** on `AppState` — no new synchronization primitive. Slow-path
replay (16× per-op cost) is an accepted latency spike; the Phase 14
operator-family yield_now pattern (if extended to this path) can mitigate.

### D-15 — Ordering of plans is strict serial (Wave 1 → 6)

Every plan depends directly on the prior plan's `files_modified`. No parallel
execution. Rationale: this phase is a tight coupling of schema changes (01)
→ runtime primitive (02) → register analyzer (03) → operator-side state redesign
(04) → end-to-end wire-up (05) → perf + SUMMARY (06). Skipping ahead risks
branch conflicts in `registry.rs`, `agg_apply.rs`, and `app_state.rs`.

## Success Criteria (SC1–SC8)

1. **SC1 — Schema round-trip.** `@bv.event(modifiable=True,
   modification_log_depth=32)` compiles to REGISTER JSON with fields populated;
   server round-trips them into `EventDescriptor.modifiable +
   .modification_log_depth`; `GET /registry` reflects them; additive-only
   diff semantics preserved (setting `modifiable=True` on an existing event
   is a schema change → 409 unless explicit version bump).
2. **SC2 — Lazy log allocation.** Given an event with `modifiable=True` and
   three features (one Tier 1, one Tier 2, one Tier 3), only the Tier 3 feature
   allocates a `ModificationLog` on first push to a given entity. Verified by
   a memory-instrumented test asserting `Option<RingBuffer>` remains `None`
   for Tier 1/2 state.
3. **SC3 — Fast path zero-cost for `modifiable=False`.** Criterion microbench
   `mod_off_vs_phase14` measures per-push overhead ≤ 5 ns vs Phase 14 baseline
   on Apple-M4. `.planning/perf-baselines.md` captures the row.
4. **SC4 — Slow-path insert-replay correctness.** Proptest: generate N events
   with arbitrary `event_time_ms` within `[0, tolerate_delay_ms)` window, apply
   (a) in sorted order, (b) in shuffled order with `modifiable=True`. For every
   Tier 1 and Tier 3 operator in the catalogue, assert state is exactly equal
   (Tier 1) or within ε (Tier A — documented per-op tolerance). Covers
   Count/Sum/Avg/Variance/Histogram (Tier 1), Ewma/Twa/RateOfChange/
   InterArrivalStats/Trend/TrendResidual/DeltaFromPrev/Streak/MaxStreak/
   NegativeStreak/GeoVelocity/DistanceFromHome (Tier 3 replay), and
   CountDistinct/Percentile/TopK (Tier A bounded drift).
5. **SC5 — Register-time analyzer.** Three smokes:
   - Tier C feature on modifiable stream → 200 + empty `warnings` (verbose-off).
   - Tier A feature on modifiable stream → 200 + `warnings: [{code:
     "BV-W-AGG-APPROX-MODIFIABLE", ...}]`. Python SDK raises
     `BeavaRetractionWarning`.
   - Tier B feature on modifiable stream → 400 + `errors: [{code:
     "BV-W-AGG-REJECT-MODIFIABLE", ...}]`. Python SDK raises
     `BeavaRegistrationError`.
6. **SC6 — Adaptive promotion metric.** Under a modifiable stream with
   `CountDistinct`, pushing 1500 unique values promotes an entity past the
   ExactArray → HLL boundary. `beava_feature_promoted_to_approx_total{feature
   ="unique_ips"}` increments from 0 to 1.
7. **SC7 — WAL replay preserves modification state.** Scenario: register
   modifiable event with 2 Tier 3 features, push events 1..5 in shuffled
   order (some in-tolerance late), kill process, restart, assert post-restart
   feature values are byte-identical to pre-restart.
8. **SC8 — Throughput regression.** `beava-bench simple-fraud (small, HTTP,
   BATCH_MS=0)` on `modifiable=False` sources shows **≤ 5%** regression vs
   Phase 14 baseline (same 10%/25% gate applies). `.planning/throughput-
   baselines.md` captures the row. A second bench row with `modifiable=True`
   + Tier 3 features captures the opt-in cost as its own baseline (no
   regression gate — establishes the first number).

## Grey Areas (sign-off before execution)

1. **Which Tier 3 ops genuinely need replay-from-snapshot vs recompute-from-
   last-N-events?** Candidates that could use the simpler "recompute from K
   most recent events" strategy (no per-op `replay_from_log` helper needed —
   just re-run `update` K times from a stored Row log):
   - `Streak` / `MaxStreak` / `NegativeStreak` — state is trivially rebuilt
     from a K-event scan in arrival-time order.
   - `FirstN` / `LastN` / `Lag` / `MostRecentN` — position-encoded; direct
     Row-log replay is simpler than retrofitting a custom `replay_from_log`.
   - `DeltaFromPrev` / `RateOfChange` — two-event window; recompute is cheap.
   - `GeoVelocity` / `DistanceFromHome` — same, few-event state.
   
   Ops that **actually** need a custom `replay_from_log` because their state
   is genuinely accumulator-shaped (not re-derivable from K Rows alone):
   - `Ewma` / `EwVar` / `EwZScore` / `Twa` — decay-weighted accumulators;
     replay-from-start-of-log re-accumulates correctly if we store
     `(event_time_ms, value)` pairs.
   - `Trend` / `TrendResidual` — linear regression state `(Σt, Σx, Σt², Σtx,
     n)`; if log preserves Rows, replay is `new_state(); for e in log {
     update(e); }` — same as a generic helper.
   - `InterArrivalStats` — running mean/stddev of dt; replay same as generic.
   - `OutlierCount` / `ValueChangeCount` / `ZScore` — cumulative counters
     against a running stat; replay same as generic.
   
   **Resolution:** all Tier 3 ops get a **generic `replay_from_log(&[ModLogEntry])`
   default implementation** that does `let mut s = Self::default(); for e in
   log { s.update(...); } s`. Per-op overrides only where the default is
   materially more expensive than a hand-tuned re-derivation (Plan 04 starts
   with the generic default for every op and promotes to a custom impl only
   if bench shows ≥ 2× speedup. Expected: zero custom overrides land in
   14.1.)
2. **Per-feature `modification_log_depth` precedence.** Stream-level
   default from `@bv.event(modification_log_depth=...)` is inherited; per-op
   override on `bv.ewma(modification_log_depth=...)` wins. If a per-op override
   is set without `modifiable=True` on the source, register returns 400 with
   `BV-CONFIG-MODLOG-DEPTH-ON-IMMUTABLE-SOURCE` (informational; the override is
   meaningless without modifiability).
3. **Snapshot v3 back-compat.** Pre-v3 snapshots lack `mod_log` fields; they
   load with `mod_log = None` for every Tier 3 feature. Recovery then
   re-materializes the log from WAL replay — identical outcome. Plan 05
   covers this in the snapshot-format task.
4. **Adaptive CountDistinct promotion metric wire-up.** `beava_feature_
   promoted_to_approx_total{feature}` must increment from the CountDistinct
   state machine's promotion path, which lives in `crates/beava-core/src/
   sketches/count_distinct.rs`. Plan 06 wires this up; naming convention
   matches Phase 13-01's `/metrics` Prometheus registry.
5. **CLI verbose flag for `BV-W-AGG-SUBTRACTIVE-OK` notes.** Assumed: server-
   side a `verbose` query param on `POST /register?verbose=1` opts into
   emitting Tier C info notes. Python SDK exposes via `app.register(...,
   verbose=True)`. Kept simple — no separate log-level knobs.
6. **Slow-path latency spike.** A slow-path replay of 16 events × 5 Tier 3
   ops per entity × 1 entity = 80 `update` calls inside one `borrow_mut` on
   `AppState`. Expected cost ≤ 5 µs (D-06 budget). Documented as an accepted
   v0 cost; if bench shows > 10 µs P99 under realistic workload, Plan 06
   surfaces it and we revisit in a follow-up phase.
