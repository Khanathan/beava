<objective>
Wave 3 ensures the TCP replica-ingest path runs through the new per-shard accept
topology that Waves 1 and 2 established — no separate listener, no tokio::spawn,
no carved-out replica accept thread.

Why a dedicated wave: replica connections invoke `replica_ingest_batch` (a sync
function already) and the `OP_LOG_FETCH` / `OP_SUBSCRIBE` opcodes. In the pre-Wave-1
world these flowed through the same `handle_connection` tokio task that primary
PUSH used, so Wave 1 + Wave 2 ALREADY handle them correctly (as long as the
opcode dispatch inside `handle_connection[_blocking]` stays unified). The risk:
during Wave 1/2 refactor, an executor might accidentally split the accept path
into "primary PUSH" vs "replica" branches — that would silently break replica
ingest or let a per-conn tokio::spawn slip back in.

Wave 3 is small but load-bearing: it (a) verifies the unified dispatch is preserved,
(b) extends the existing Phase-54 `replica_ingest_routing.rs` test with a per-shard
assertion, (c) documents the at-least-once + fast-path assumptions for replica
connections on the new topology.

Output: Replica ingest confirmed to ride the new per-shard accept path on both
platforms. HTTP untouched.
</objective>
