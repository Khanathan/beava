    **Step A — SDK op methods on `EventSource` + `EventDerivation`** (`python/beava/_events.py`):

    Add 8 methods to both classes (the method bodies are identical — same ops list append + new-derivation construction — so factor into a shared mixin or helper if natural). Each method:
    - Constructs a new `EventDerivation` with:
      * `upstream = self` (preserves the chain-parent link).
      * `ops = [*self.ops, new_op_dict]` for EventDerivation (EventSource starts with `ops = []`).
      * Same schema as upstream for now (server propagates; client-side is a placeholder).
    - Returns the new derivation. NEVER mutates `self`.

    Per-method op dict shape (matching the Rust OpNode wire shape):
    - `filter(expr: Expr) -> EventDerivation`: `{"op": "filter", "expr": expr.to_expr_string()}`.
    - `select(*fields: str) -> EventDerivation`: `{"op": "select", "fields": list(fields)}`.
    - `drop(*fields: str) -> EventDerivation`: `{"op": "drop", "fields": list(fields)}`.
    - `rename(**mapping: str) -> EventDerivation`: `{"op": "rename", "mapping": dict(mapping)}`.
    - `with_columns(**exprs: Expr) -> EventDerivation`: `{"op": "with_columns", "exprs": {name: e.to_expr_string() for name, e in exprs.items()}}`.
    - `map(**exprs: Expr) -> EventDerivation`: same as with_columns but `"op": "map"` (SDK-OPS-06 alias; server treats identically; wire shape retains the user's chosen verb).
    - `cast(**type_map: str) -> EventDerivation`: validate every value ∈ {"str", "int", "float", "bool"}; raise `ValueError(f"invalid cast target: {value}")` otherwise. Then `{"op": "cast", "type_map": dict(type_map)}`.
    - `fillna(**defaults) -> EventDerivation`: `{"op": "fillna", "defaults": dict(defaults)}` — values are JSON scalars (int/float/str/bool); raise `ValueError` if any value is None (fillna with null is meaningless).

    **Step B — SDK op methods on `TableSource` + `TableDerivation`** (`python/beava/_tables.py`):

    Same 8 methods as EventSource, but returning `TableDerivation`. Two table-specific behaviors:
    - `.drop(*fields)`: if any `f in fields` is in `self.key`, raise `ValueError(f"cannot drop key field '{f}'")` (SDK-OPS-03 table half). Otherwise proceed normally.
    - `.rename(**mapping)`: if any old key-list entry `k` is in `mapping`, compute the new key list: `[mapping.get(k, k) for k in self.key]`. Attach the new key list to the TableDerivation (SDK-OPS-04 table half). The descriptor's serialized REGISTER payload carries the cascaded key list.

    **Step C — Python reference evaluator** (`python/beava/_eval_reference.py`, NEW ~100–150 LoC):

    Pure-Python evaluator that mirrors Rust's eval semantics. Signature: `def evaluate(expr: Expr, row: dict[str, Any]) -> Any` where `Expr` is the SDK's expression AST (reuse `python/beava/_col.py` types; they already model the same grammar the Rust parser accepts). The evaluator MUST:

    1. **Apply the `(x == null)` → `isnull(x)` rewrite at the AST level before evaluating** — faithfully mirror Plan 04-02's Rust-side parser rewrite. This means: walk the AST, collapse `BinOp("==", e, Literal::Null)` and `BinOp("==", Literal::Null, e)` to `Call("isnull", [e])`, recurse into children. Do this ONCE up front; then the evaluator's BinOp("==") path stays strict-null (returns None when either operand is None).

    2. **Implement three-valued null logic** (CONTEXT.md D-04):
       - Arithmetic on None → None (any None operand poisons the result).
       - Boolean `and`: `True and None` → None; `False and None` → False (short-circuit); `None and None` → None.
       - Boolean `or`: `True or None` → True (short-circuit); `False or None` → None; `None or None` → None.
       - `not None` → None.
       - Comparison with None → None.
       - `isnull(x)` → True if x is None else False (always bool, never None).
