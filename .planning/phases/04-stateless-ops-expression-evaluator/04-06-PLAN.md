<objective>
Rust-side Phase 4 acceptance gate. Land the /dev/apply_ops dev-only endpoint (gated behind BEAVA_DEV_ENDPOINTS=1) and a Rust smoke test that exercises ROADMAP SC1/SC2/SC3/SC5 end-to-end over BOTH HTTP and TCP transports, proving the full loop from `POST /register` → propagated schema in GET /registry → `OpChain::apply(Row)` per event via the dev endpoint (and directly via Registry::compiled_chain for in-process assertions).

This plan is the first of a two-plan acceptance split (Plan 04-06 = Rust side; Plan 04-07 = Python side). The Python SDK op-method surface + SC4 client/server eval-equivalence proptest lands in Plan 04-07 to keep each plan within CLAUDE.md's task-scope budget (one plan should be one concern, not five). Both 04-06 and 04-07 are in wave 6; 04-07 depends on 04-06 so the dev endpoint exists before the Python proptest tries to POST to it.

Because Phase 6 (push + WAL) hasn't landed, we cannot test "server rejects events failing the predicate" via a real push endpoint. The acceptance approach here:
- Register a derivation over HTTP and TCP.
- Call POST /dev/apply_ops (new endpoint) with synthetic rows; assert pass/drop/transform outcomes.
- Assert `Registry::compiled_chain(name)` returns Some(Arc<OpChain>) so Phase 5's apply loop has a clear contract.
- Assert GET /registry shows the propagated (server-computed) schema, not the client-supplied one.
- Send a malformed-expression register payload; assert HTTP 400 + TCP 0xFFFF error frames carry the SC5 shape.

Python-side coverage of SC1–SC5 via the Python SDK + SC4's client/server eval-equivalence proptest lives in Plan 04-07. This split keeps Plan 04-06's scope within the CLAUDE.md task-size budget (one production endpoint + 7 Rust smoke tests + 3 endpoint unit tests) and makes Plan 04-07 the natural home for the 8 SDK op methods, the Python reference evaluator, and the 256-case hypothesis proptest.
</objective>
