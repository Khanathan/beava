<objective>
Implement the Phase 4 expression evaluator — `eval(expr: &Expr, row: &Row) -> Value`. Evaluates parsed ASTs against row values, enforces SQL-style three-valued null propagation, dispatches `cast` / `isnull` builtins via a table-driven registry, and satisfies SRV-APPLY-06 ("Expression evaluator parses `to_expr_string()` canonical form into AST; evaluates per-event in-place"). The parsing half landed in Plan 04-02; this plan is the evaluation half + the extensibility hook Phase 5 will use for `bv.count(where=...)`.

Purpose: This is the closed loop between the SDK's `bv.col(...)` and the server deciding whether an event passes a filter or what value to derive. Plans 04-04 (op chain) and 04-05 (register integration) call `eval()` for every row × every op chain step. Without this plan, filter predicates and with_columns derivations stay as opaque strings.

Note on the `(x == null)` rewrite: Plan 04-02 owns the post-parse AST normalization that collapses `BinOp("==", e, Literal::Null)` (and the commutative form) to `Call("isnull", [e])`. By the time an AST reaches eval.rs, that rewrite has already happened — eval.rs's BinOp("==") branch is strict-null-propagating (returns Value::Null when either operand is Null, per CONTEXT.md D-04) and has NO null-equality special case. This plan's tests MUST mirror that invariant: any test that wants to evaluate `(x == null)` parses the source string first and observes the rewritten Call node, not a BinOp.

Output: `crates/beava-core/src/eval.rs`, `crates/beava-core/src/expr_builtins.rs`, unit tests covering every grammar node kind + every null-propagation edge, and a truth-table-equivalence proptest pairing evaluator output with a Rust-side reference implementation of the same semantics (the "random predicate + random event → deterministic evaluator output" invariant from CONTEXT.md §Specifics).
</objective>

<!-- From Plan 04-02 — Expr variants the evaluator matches on -->

```rust
pub enum Expr {
    Field { name: String, span: Span },
    Literal(Literal, Span),
    BinOp { op: BinOpKind, left: Box<Expr>, right: Box<Expr>, span: Span },
    UnaryOp { op: UnaryOpKind, operand: Box<Expr>, span: Span },
    Call { fn_name: String, args: Vec<Expr>, span: Span },
}

pub enum Literal {
    Null,
    Bool(bool),
    Int(i64),
    Float(f64),
    Str(String),
    BareIdent(String),  // cast(x, float) — 2nd arg after post-parse rewrite
}
```

IMPORTANT: Plan 04-02's parser applies a post-parse AST normalization pass that collapses `BinOp("==", e, Literal::Null)` and `BinOp("==", Literal::Null, e)` into `Call("isnull", [e])`. So by the time `parse("(x == null)")` returns, the Expr variant is `Call`, not `BinOp`. eval.rs relies on this — it does NOT special-case null operands in BinOp("==").

<!-- CONTEXT.md §D-04 null-propagation spec — eval MUST honor: -->

- `null + anything` → Null
- `null * 0` → Null (not 0)
- `null > anything` → Null (not Bool)
- `null == null` → Null (strict SQL semantics; users who want a Bool test use `.isnull()` which becomes the `isnull` Call after Plan 04-02's rewrite)
- Boolean ops: short-circuit per Value::and/or_three_valued (Plan 04-01)
- `isnull(x)` → always Bool (never Null)
- Filter predicate returning Null → row dropped (04-04 op-chain enforces; 04-03 just returns Null)

<!-- CONTEXT.md §D-05 runtime type rules — at eval time: -->
- Arithmetic on non-numeric or mixed types → Null (register-time catches most; eval stays defensive)
- `cast` failures at value level (e.g., `cast('abc', 'int')`) → Null
- i64 overflow: saturate (`i64::saturating_add`, `saturating_sub`, `saturating_mul`); division by zero → Null
- f64: IEEE-754 (NaN/Inf propagate naturally)
