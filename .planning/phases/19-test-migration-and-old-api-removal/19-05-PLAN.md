bench.py pipeline structure (from RESEARCH.md):
- define_small(): 1 keyed @st.stream, 5 features -> @source + @dataset with 5 features
- define_medium(): 2 keyed @st.stream + 1 @st.view, fan-out -> 1 @source + 2 @dataset + 1 view-dataset
- define_large(): 3 keyed @st.stream + 2 @st.view, cascade + fan-out + HLL -> 1 @source + 3 @dataset + 2 view-datasets

Each define_*() returns streams to register + primary stream for push.
In new API: push to the @source (keyless entry point), cascade handles the rest.

From python/tally/_source.py:
- @source creates a SourceDef (keyless stream)
- SourceDef._tally_stream_name = class name

From python/tally/_dataset.py:
- @dataset(depends_on=[...]) creates DatasetDef
- group_by("key").agg(**ops) defines keyed aggregation
- depends_on is required (list of upstream SourceDef/DatasetDef)

Push target: app.push(SourceClass, event) -- push to source, server cascades to datasets.

  <action>
Rewrite bench.py to use the new API:

1. Replace `import tally as st` with `import tally as tl` and `from tally import source, dataset, group_by`

2. **define_small()**: Replace the single @st.stream(key="user_id") with:
   - `@source` class `RawTxns` (keyless entry point)
   - `@dataset(depends_on=[RawTxns])` class with `features = group_by("user_id").agg(...)` containing the 5 features
   - Return value: both classes for registration, RawTxns as push target

3. **define_medium()**: Replace 2 @st.stream + 1 @st.view with:
   - `@source` class for the raw event entry
   - 2 `@dataset(depends_on=[Source])` classes with group_by().agg() for keyed aggregation
   - 1 `@dataset(depends_on=[...])` with derive-only for the view equivalent
   - Fan-out: if the old API used fan-out (event has both user_id and merchant_id), the source receives the event and cascades to both keyed datasets. Check if the new API supports fan-out to multiple keyed datasets from one source.

4. **define_large()**: Same pattern as medium but with 3 keyed datasets and 2 view-datasets, including HLL (distinct_count) and cascade chains.

5. Update the harness code:
   - `app.register(*streams)` -> `app.register(*all_classes)` (register sources + datasets)
   - `app.push(primary_stream, event)` -> `app.push(source_class, event)` (push to source)
   - Keep all timing/reporting code unchanged

6. **Critical for benchmark accuracy**: The old define_small() had NO cascade overhead (direct push to keyed stream). The new API adds a source -> dataset cascade hop. This may add ~1-5% overhead for small pipeline. This is expected and acceptable as long as the 8-client aggregate stays within -5% of 1.1M eps. Medium and large already had cascade, so those should be equivalent.

7. Run a quick sanity check: `python3 benchmark/tally-throughput/bench.py --pipeline small --mode sync --clients 1 --events 1000` to verify it works.
