<decisions>
## Implementation Decisions

### D-01 — SSG: integrate into existing beava-website/ + Pagefind (USER-LOCKED)

- DO NOT spin up MkDocs Material from scratch
- USE existing `beava-website/` framework (hand-rolled static HTML + bespoke design system + Pagefind search via `build-search.mjs`)
- Build a small markdown → HTML conversion script (~200-400 LOC; Python or Node; planner picks language) that:
  - Reads `docs/*.md` files (output of Phase 13.0)
  - Renders to HTML matching beava-website's existing design (uses `beava-design-system/` tokens)
  - Writes to `beava-website/project/docs/<...>/index.html`
- Extend `beava-website/build-search.mjs` to include new docs pages in Pagefind index
- Pagefind handles client-side search; no backend required
- **Why:** zero design re-work; matches existing site visually; preserves existing search infrastructure; minimal new dependencies. NOT MkDocs Material — that would force re-doing the visual identity that already exists.

### D-02 — Visual identity: use existing beava-design-system/ (USER-LOCKED)

- Source-of-truth tokens: `beava-design-system/colors_and_type.css` + `beava-design-system/ui_kits/`
- New docs pages pull these tokens (presumably via @import or CSS custom properties; planner picks based on existing pattern in beava-website)
- Zero new design work in 13.7
- If any token gaps for docs-specific elements (e.g., code blocks, callouts, tables): planner adds minimally to `beava-design-system/` (with user heads-up; could be deferred to a designer pass post-launch)

### D-03 — Vertical guides DEFERRED + Quick-start tone for any 13.7-shipped guide content (USER-LOCKED)

- 3 vertical guides (adtech / fraud / ecommerce) → **DEFERRED**, user-authored follow-up with interactive embeds
- Phase 13.7 does NOT ship vertical guide content
- Phase 13.7 DOES ship `/docs/quickstart.md` polish (existing doc from Plan 13.0-14; render into site + add screenshot/GIF)
- `/guide/` pages stay as existing placeholders OR get "coming soon" stubs (planner picks; minimal effort)
- **Why deferred:** user wants vertical guides to be interactive (embedded REPL or similar), which requires v0.1+ infrastructure (browser-WASM `@beava/browser` per `v0.1-deferrals.md`). Don't ship static prose now and re-author later.

### D-04 — Deploy: Cloudflare Pages (USER-LOCKED)

- Cloudflare Pages connected to `beava-website/` directory
- Build command: `cd beava-website && npm run build` (or whatever the existing build script is)
- Output dir: `beava-website/dist/` or `beava-website/project/` (planner verifies)
- Custom domain: beava.dev (Cloudflare DNS already controls; just add DNS record)
- Auto-deploy on push to `v2/greenfield` (or `main` post-merge)
- Preview deploys on every PR
- Free tier sufficient for v0 traffic
- **Why:** generous free tier; built-in analytics; native DNS integration with beava.dev; matches static-first + Pagefind stance.

### Claude's Discretion (planner picks)

- Plan/wave shape (much smaller now: ~3-4 plans across 2 waves):
  - Wave 1: markdown→HTML converter + theme integration + Pagefind extend
  - Wave 2: cross-link audit + quickstart polish + Cloudflare deploy config + closure
- Markdown→HTML converter language (Python or Node; planner picks based on existing tooling in beava-website/)
- Animated GIF tooling (asciinema + agg) vs screenshots-only (less effort)
- Whether to add a build-time link checker (recommended) or rely on manual review
- Cloudflare Pages config exact mechanics (CLI vs dashboard; default CLI for reproducibility)
- Whether to add "coming soon" stubs for /guide/ vertical pages or leave existing content untouched

</decisions>
