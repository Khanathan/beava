## Decisions Made

- **Function name discovery: `parse_agg_kind`, not `op_str_to_agg_kind`.** The plan body's `<interfaces>` block hypothesized a function name `op_str_to_agg_kind` — execution found the actual name in `agg_compile.rs:253` is `parse_agg_kind`. The test extractor was written to call `extract_fn_body(&stripped, "parse_agg_kind")`. The plan body's name was illustrative; the test adapts to the actual code.
- **Match-arm extractor look-ahead-for-`=>`-or-`|` rule.** The `lifetime_bound_for_op_str` body contains both pattern strings (e.g. `"count" | "sum" => OpLifetimeBound::O1`) AND RHS expression strings (e.g. `OpLifetimeBound::BoundedByRequiredKwarg("buckets")`). A naive "every quoted string is a pattern" extractor would falsely classify `"buckets"` / `"n"` / `"samples"` / `"k"` / `"max_categories"` as op-strings. The look-ahead rule (the next non-whitespace tokens after the closing quote must be either `=>` directly OR a `|` chaining to another quoted alternative) correctly excludes RHS strings.
- **Drift-guard threshold: 30.** Picked conservatively low (the actual count is 54) so a deliberate op removal — should v0.1+ ever reduce the catalogue below 54 — doesn't trip the guard accidentally. The real signal the guard is built for is "near-zero count, function moved/renamed" rather than "exactly 54".
- **Single test: prefix commit per TDD §Note 4.** No paired RED→GREEN here — Plan 04 already provides the GREEN behaviour. The architectural test is a CI-tripwire, not a feature implementation. Single `test(12.8-05): ...` prefix commit.
- **GREEN-by-default (NO `#[ignore]`).** The test runs on every `cargo test --workspace`. Plan 04 having landed first means the bound table is populated; the test passes GREEN-on-arrival. Plan 05 was bumped from Wave 2 to Wave 3 per the planner-checker W-01 fix specifically so it executes AFTER Plan 04 — running concurrently with Plan 04 in Wave 2 would have produced a false-RED architectural test against Plan 01's all-Unbounded placeholder.
- **Single-file (NOT a companion pair).** Per CONTEXT.md `<decisions>` D-locked-autonomously: "Architectural test pair vs single test: Single test sufficient — 12.6/12.7 used companion file-absence tests because there were files to assert deleted; 12.8 has no analogous deletion target." The file-absence half of the pair-pattern is unnecessary; only the symbol-coverage half applies.

## Lessons Learned

- **Single-file architectural test pattern works for positive-coverage invariants; pair pattern was over-engineered for this case.** 12.6 and 12.7 used companion file-absence tests (`phase12_6_legacy_axum_killed.rs` / `phase12_7_legacy_table_handlers_killed.rs`) because they had deleted files to assert-stay-deleted. 12.8 has no analogous deletion target — every classification arm in `lifetime_bound_for_op_str` is "addition" rather than "removal". A single architectural test asserting subset-of-source-of-truth is sufficient. CONTEXT.md correctly anticipated this; the plan honored it.
- **Match-arm-pattern extraction needs a structural rule, not a regex.** The naive "every quoted string in the function body" extractor would have falsely classified RHS arguments to `BoundedByRequiredKwarg("...")` and `BoundedByConfig("...", N)` as op-strings. The look-ahead-for-`=>`-or-`|` rule correctly distinguishes patterns from RHS literals because Rust's match-arm grammar requires `=>` (or `|` chaining) to immediately follow a pattern token. This generalizes: any future test that needs to extract pattern-strings from a Rust match block can reuse the `extract_op_strings_from_match_body` helper verbatim.
- **Plan-body function names are hypotheses; verify in execution.** The plan referenced `op_str_to_agg_kind` — actual name is `parse_agg_kind`. Cost to verify: 30s. The plan's `<read_first>` block explicitly directed the executor to grep `reservoir_sample\|count_distinct` in `agg_compile.rs`, which surfaced the correct function name on first inspection. Pattern: when a plan references a function by name, include a grep-to-verify instruction in `<read_first>` so the executor catches drift before writing the test.
- **Wave reordering for architectural-test-against-table-data.** The W-01 checker fix that bumped Plan 05 from Wave 2 to Wave 3 prevented a false-RED test against Plan 01's placeholder. General principle: an architectural test that asserts "table T contains classifications for all entries in source S" must execute strictly AFTER the plan that populates T, otherwise it fails for the wrong reason during the parallel-wave window. CONTEXT.md `<decisions>` D-locked-autonomously surfaces this implicitly via "the architectural test runs GREEN-on-arrival post Plan 04"; the W-01 fix made it explicit in the plan-graph.
