<objective>
Wire 5 new Prometheus-format metric families onto the existing `/metrics` admin endpoint:

1. `beava_cold_entity_evictions_total{source}` (counter) — Plan 03's eviction sites
2. `beava_lifetime_op_cap_hit_total{op}` (counter) — wraps existing `EntropyStateWrap::categories_capped_count` + new hooks for top_k / histogram cap-hits where existing internals expose them
3. `beava_entity_count_resident{source}` (gauge) — sum of `AggStateTable` HashMap sizes per source
4. `beava_bucket_reclaim_total{op}` (counter) — `WindowedOp::evict_oldest_bucket` site (already in `crates/beava-core/src/agg_windowed.rs` line ~270)
5. `beava_bytes_per_entity_p99{source}` (gauge) — sampled estimate; v0 ships a best-effort placeholder (log how to compute, populate with rough static estimate) — Phase 13 ship-gate may upgrade to dynamic sampling

All 5 families ride `crates/beava-server/src/http_admin.rs::metrics_handler` (lines 83-121) — extends the existing format-string output. The endpoint stays on tokio admin sidecar per `project_phase18_no_dual_runtime`.

**No new threads** in apply path (per CONTEXT D-04 and `project_no_sharded_apply`). All counters use `AtomicU64::fetch_add(1, Relaxed)` for the apply-side write; `metrics_handler` uses `.load(Relaxed)` for the admin-side read.

**Per-source labelling design:** the metrics atomics are NOT per-source-fanned-out at compile time. Instead, the metrics state is a `HashMap<Arc<str>, MemoryGovMetricsPerSource>` keyed by source name, populated lazily on first event for that source. This avoids the "you have to know all sources at startup" problem and matches the existing Phase 18 pattern of registering metrics on demand.

Output: `/metrics` exposes the 5 new families. Test scrapes /metrics, exercises eviction path, scrapes again, asserts the counter incremented.

**Scope discipline:** `bytes_per_entity_p99` is the riskiest metric. Per CONTEXT.md: "rough estimate sampled periodically (no per-entity cost on hot path)". v0 ships a STATIC placeholder (e.g. 7000 bytes/entity per the PROJECT.md "Memory" budget line). A periodic re-sampler is out of scope for this plan; if Phase 13 ship-gate needs accuracy, a follow-up plan is filed. Documented in CLAUDE.md § Memory Governance Invariant block (Plan 09).
