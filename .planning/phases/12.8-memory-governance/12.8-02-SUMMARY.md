# Phase 12.8 Plan 02: cold_after decorator + EventDescriptor.cold_after_ms — Summary

Add the v0 surface for D-01 (per-source cold-entity TTL via
`@bv.event(cold_after='7d')`) without any eviction logic. Decorator-time
parsing, range validation [1s, 365d], wire JSON emission, registry struct
field, equivalence-relation inclusion. Eviction logic lands in Plan 12.8-03;
this plan only establishes the surface so Plan 03 has something to read.

## What Shipped

**Python decorator (`python/beava/_events.py`):**

- New `cold_after: str | None = None` kwarg on `@bv.event`
- Reuses Phase 5 `validate_duration_string` + `duration_to_ms` (same
  `s|m|h|d|ms` suffixes the existing `dedupe_window` / `keep_events_for`
  parsers accept — CONTEXT.md mentioned `w` weeks but the actual parser
  has no such suffix; Plan 02 follows the parser shape, not CONTEXT.md)
- Range: `1s ≤ cold_after ≤ 365d` inclusive, both bounds enforced at
  decoration time with forward-looking error framing ("must be ≥ 1s",
  "must be ≤ 365d in v0") per CONTEXT D-01 / 12.7 D-02
- `'forever'` is REJECTED at decoration time — would defeat the cold-TTL
  purpose; `cold_after=None` (omit) is the canonical "never expire"
- `EventSource.__init__` gains `cold_after_ms: int | None = None` parameter
  stored as `self._cold_after_ms`
- `EventSource._to_register_json()` emits `"cold_after_ms": <int|None>`
  alongside `dedupe_window_ms` / `keep_events_for_ms`
- Default omitted (`cold_after=None`) preserves existing no-expiry
  behavior — zero impact on legacy code that doesn't add the kwarg

**Rust struct (`crates/beava-core/src/registry.rs`):**

- `pub cold_after_ms: Option<u64>` field added with `#[serde(default)]`
  (forward-compat with older Python payloads that omit the key entirely)
- `equiv_ignoring_version` now includes the new field — diff engine
  picks up cold-TTL config changes across re-registers (locks the field
  into the conflict-detection set alongside `dedupe_key`,
  `dedupe_window_ms`, `keep_events_for_ms`)

**Cascade:** 60 `EventDescriptor { ... }` literal-construction sites
updated across 14 files (initial `cargo build` flagged 51 missing-field
errors; some sites behind `cfg-test` were caught by the wider Python-script
sweep). Files touched: 4 src files in `beava-core`, 2 benches, 5 test
files in `beava-core/tests/`, 1 bench in `beava-persistence`, 1 test file
in `beava-server`. All sites now carry `cold_after_ms: None,`.

## Deviations from Plan

### Plan-vs-CONTEXT reconciliation

- **CONTEXT.md says "duration suffixes `s|m|h|d|w`"; the actual Phase 5
  parser has no `w` (weeks).** Plan 02 line 52 noted this discrepancy
  upfront and chose to follow the existing parser shape, not CONTEXT.md.
  Per `feedback_logistics_autonomy` this kind of reconciliation is
  autonomous. The cold_after kwarg accepts `s|m|h|d|ms` (the `ms` is
  rejected by the >= 1s lower-bound check, but it's parser-legal so
  failure is delegated to the range guard with a clear "must be ≥ 1s"
  message rather than a parser-level "unknown unit"). No change needed
  to `validate_duration_string` itself.

### Auto-fixed Issues

None. The plan executed exactly as written.

### Stricter contract than the underlying parser

- `cold_after='forever'` is rejected even though `validate_duration_string`
  accepts `'forever'` (used by `keep_events_for`). The plan flagged this
  in the `<behavior>` section (Test 9 explicitly verifies the rejection
  with rationale "would defeat the cold-TTL purpose"). The error message
  steers users to `cold_after=None` (omit the kwarg) for unbounded
  retention. Documented as a Plan 02 decision in frontmatter.

## Lessons Learned

- **Adding an `Option<u64>` `#[serde(default)]` field to a hand-rolled
  Rust struct cascades to `N` test fixtures + bench harnesses.** For
  `EventDescriptor` the cascade was 60 sites across 14 files — discovered
  by `cargo build` flagging 51 missing-field errors initially (some
  cfg-test sites only surfaced via the wider Python-script sweep). Budget
  ~5-10 minutes for the mechanical insertion when planning a similar field
  addition. The script-driven approach (Python `re.compile` + insert after
  `keep_events_for_ms:` line) was robust because the new field is
  invariant-cousin to the existing one. A `Default::default()` impl on
  `EventDescriptor` would have made this even cheaper, but the existing
  struct doesn't carry one — adding it is out of scope here.

- **CONTEXT.md mentioning a wider parser surface than the source of
  truth is a recurring planning-vs-impl mismatch.** Plan 02 flagged this
  upfront in the objective and decided autonomously per
  `feedback_logistics_autonomy`. Future planners gathering CONTEXT can
  cross-check claims like "duration parser supports X" against the
  actual `_UNIT_TO_MS` / `_DURATION_RE` table to avoid landing
  contradictions in a plan.

- **Stricter contract on `cold_after` than the underlying parser** —
  rejecting `'forever'` even though `validate_duration_string` accepts
  it — is the right choice for v0 because it preserves the explicit
  `cold_after=None` semantic from D-01. Pattern: when reusing a parser
  whose accepted set is wider than the new feature's accepted set,
  add an explicit guard upstream of the parser call rather than narrowing
  the parser itself (which would break other callers).
