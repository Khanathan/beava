# Phase 12.8: Memory Governance — Context

**Gathered:** 2026-05-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Two-tier memory hygiene + lifetime-aggregation contract before final ship (Phase 13). Closes the operational gap where v0 users with long-lived high-cardinality fraud workloads see unbounded memory growth without explicit governance.

**In scope:**

1. **Tier 1 — entity-level cold-entity TTL (opt-in, default OFF):**
   - New `cold_after` parameter on `@bv.event(cold_after='7d')` decorator (and class-form equivalent).
   - Default = `None` → no expiry. Users opt in per-source.
   - Duration string parser reused from Phase 5 (`s|m|h|d|w` suffixes; min 1s, max 365d).
   - Lazy expiry: on event arrival for an entity, check `now_ms - entity.last_seen_ms > cold_after_ms`; if expired, treat as fresh entity (FRESH STATE on resurrect — locked permanent, Redis TTL pattern).
   - `app.delete(entity_key)` Python verb for explicit removal (already exists or needs wire if missing).
   - No background scan thread (preserves `project_no_sharded_apply` single-thread invariant).

2. **Tier 2 — bucket-level reclaim within active entities (always on):**
   - Existing per-event eviction during `update_at(now_ms)` is the canonical mechanism — already trims trailing buckets as windowed ops advance their head.
   - **No additional scanning of idle entities.** Idle entities hold their state until either (a) Tier 1 cold-TTL evicts them, or (b) next event triggers normal `update_at()` cleanup.
   - Tier 2 is "always on" because the existing per-event reclaim is always on; this phase clarifies + tests the contract, doesn't add new mechanism.

3. **Lifetime aggregation contract (`[first_seen, now)` forever-window):**
   - `windowed=` omission means lifetime — implicit semantics (no new SDK kwarg per D-02).
   - Each of the 54 operators declares a finite per-entity memory ceiling for lifetime mode at register-time.
   - **Memory-bound classes:**
     - **O(1):** count, sum, avg, min, max, variance, stddev, ratio, streak, max_streak, negative_streak, first_seen, last_seen, age, has_seen, time_since
     - **Bounded sketch (op-config defines size):** count_distinct (HLL), percentile (DDSketch), top_k (k cap), bloom_member, entropy (max_categories per Phase 19.2 D-05a)
     - **Bounded by required `n` kwarg:** first_n, last_n, lag, most_recent_n, time_since_last_n
     - **Bounded by required size kwarg:** reservoir_sample
     - **Bounded by config (audit + cap-where-missing):** histogram (buckets), event_type_mix (max_categories — already capped), seasonal_deviation, dow_hour_histogram (24×24 already O(1)), hour_of_day_histogram (24 already O(1))
   - **Hard reject at register-time** for any op without a memory-bound proof — new JSON-prelude shim (D-03).

4. **New register-time shim — `pre_check_unbounded_op_in_lifetime_mode`:**
   - Fourth JSON-prelude shim in `crates/beava-core/src/register_validate.rs` alongside `pre_check_removed_ops` (12.6-04), `pre_check_legacy_event_time_keys` (12.6-06), `pre_check_unsupported_node_kind` (12.7-01).
   - Walks the registered DAG; for each operator instance, checks `windowed=` is set OR a memory-bound parameter is present.
   - Rejects with structured error code `unbounded_op_in_lifetime_mode` + message naming the op + suggested cap kwarg.
   - Error framing: forward-looking ("requires explicit memory bound in v0"), NOT retrospective ("removed").

5. **Metrics — Prometheus-compatible:**
   - `cold_entity_evictions_total{source=...}` — counter; increments on lazy eviction at event arrival.
   - `lifetime_op_cap_hit_total{op=..., source=...}` — counter; increments when an op hits its per-entity cap (e.g., HLL precision saturated, top_k displaces, histogram bucket dropped).
   - `entity_count_resident{source=...}` — gauge; current resident entity count per source.
   - `bytes_per_entity_p99{source=...}` — gauge; rough estimate sampled periodically (no per-entity cost on hot path).
   - `bucket_reclaim_total{op=...}` — counter; increments on bucket trim during `update_at()`.
   - All wired into existing `/metrics` endpoint (Plan 13-01 — already partially shipped on `phase-13-ship`).

6. **Architectural test — `phase12_8_lifetime_ops_have_bounds.rs`:**
   - Mirrors 12.7 D-03 architectural-test-as-grep pattern.
   - Walks the operator catalogue (`crates/beava-core/src/agg_op.rs` and per-op modules) and asserts every op declares either `windowed=` requirement OR a finite memory-bound parameter.
   - Sanity allowlist test confirms grep isn't a no-op.
   - Locks the lifetime-bounds invariant in CI.

7. **Microbench + throughput rebaseline (Phase 8+ contract):**
   - One microbench cell measuring per-event apply cost with cold-entity TTL enabled (vs disabled) — confirm <5% per-event regression.
   - Throughput rebaseline (8-cell `small/medium/large/fraud-team × http+tcp`) — regression-gate cell `small/tcp` vs Phase 12.7 baseline (751,498 EPS), 10% warn / 25% block per CLAUDE.md.

**Out of scope (decided autonomously per `feedback_logistics_autonomy`):**

- **Cold TTL granularity finer than per-source.** Per-feature TTL ("this aggregation for this entity hasn't been touched") is too complex for v0. Entity-level only.
- **Eviction-aware contract for resurrected entities.** Locked: fresh state (Redis TTL pattern). No partial-state preservation; no opt-in to "remember" a subset of features.
- **Memory-pressure backpressure.** No automatic action when entity count × bytes exceeds threshold. Metric emission only — operators wire their own alarms. Refusing new entities or emergency LRU eviction is a v0.0.x or v0.1+ concern.
- **Per-event-type cold logic.** Per-source uniform (one `cold_after` per `@bv.event` source). Multi-event-type sources use the source's TTL.
- **SSD overflow / tiered eviction.** Locked OUT permanently per `project_no_sharded_apply` + PROJECT.md.
- **Resurrection with persisted history.** Would need disk to remember evicted entities; would conflict with v0's in-memory-only commitment. Deferred indefinitely.
- **Background reclaim thread.** Violates `project_no_sharded_apply`; rejected.
- **Periodic compact pass during snapshot writer.** Considered and rejected (Area 4 Option B): per-event reclaim during `update_at()` plus optional cold-entity TTL covers all cases without snapshot-thread overhead.
- **`bv.session()` revival via bucket-reclaim primitive.** Forward-compat note only; Phase 25 stays archived.
- **New operators beyond declaring lifetime semantics on existing 54.** Pure governance phase, no catalogue additions.
- **Plan 12-10 push-and-get on mio.** Still deferred per Phase 12.6 D-04.

</domain>

<decisions>
## Implementation Decisions

### Cold-entity TTL config surface
- **D-01 — Per-source decorator only.** `@bv.event(cold_after='7d')` (and class-form equivalent). Default = `None` → no expiry. Reuses Phase 5 duration parser (`s|m|h|d|w` suffixes).
  - **Rationale:** Granular per-source policy; declarative + survives restart via durable registry; no env-var/decorator config drift; aligned with existing `@bv.event(dedupe_window=...)` style.
  - **No `BEAVA_COLD_TTL` env var.** No runtime API. Single surface.
  - **`cold_after=None`** (omitted) is the explicit "never expire" — no implicit default change for existing users.

### Lifetime aggregation SDK shape
- **D-02 — Implicit lifetime via `windowed=` omission.** No new SDK kwarg.
  - **Rationale:** Existing behavior already treats `windowed=` omission as lifetime. Adding `lifetime=True` would be a 54-op signature change with two surfaces to document. The contribution Phase 12.8 makes is the **per-op memory-bound contract at register-time** (D-03), not new syntax.
  - **Operator docs** (Phase 13) will explicitly document: "Omitting `windowed=` means lifetime aggregation. Each op declares its lifetime memory ceiling — see catalog table."

### Memory-bound enforcement for unbounded ops
- **D-03 — Hard reject at register-time.** Fourth JSON-prelude shim `pre_check_unbounded_op_in_lifetime_mode` in `register_validate.rs`.
  - Walks the registered DAG; for each operator instance, checks the op's lifetime memory-bound contract.
  - Rejects with structured error code `unbounded_op_in_lifetime_mode` + message naming the op + suggested cap kwarg (e.g., `"histogram requires explicit num_buckets in lifetime mode; suggested: histogram(num_buckets=256, ...)"`).
  - **Catalog audit during plan phase:** verify every op in `crates/beava-core/src/agg_op.rs` declares its bound; cap kwargs added where missing (likely candidates: `histogram(num_buckets=...)`, possibly `seasonal_deviation`).
  - **Error framing:** "requires explicit memory bound in v0" (forward-looking, NOT "feature removed").
  - **Aligned with 12.6/12.7 strict-deny posture.**

### Bucket reclaim trigger
- **D-04 — Per-event only via existing `update_at(now_ms)` mechanism.** No additional scanning of idle entities.
  - **Rationale:** Trailing-bucket eviction during `update_at()` already happens during normal operation. Idle entities hold state until either (a) cold-TTL evicts (D-01, opt-in), or (b) next event triggers cleanup. Zero new hot-path cost. Preserves `project_no_sharded_apply` single-thread invariant.
  - **Tests verify the contract:** entity receives event at T=0 with windowed=60s, no events for 120s, then event at T=120s — assert all old buckets evicted, only new bucket present.
  - **Rejected alternatives:**
    - Periodic compact pass during snapshot writer (snapshot duration grows linearly with entity count; per-event already covers it).
    - Eager scan via apply-thread heartbeat (overkill given D-01 + per-event reclaim).
    - Background reclaim thread (violates `project_no_sharded_apply`).

### Locked autonomously (per `feedback_logistics_autonomy`)
- **Eviction-aware resurrect contract:** FRESH state on resurrect (Redis TTL pattern). No partial-state preservation. Locked permanent — reviving requires explicit user override + new ADR.
- **Memory-pressure backpressure:** Metric emission only. No automatic refuse-new-entity or emergency LRU. Operators wire alarms.
- **Cold TTL granularity:** Entity-level only (one `cold_after` per source). No per-feature or per-event-type granularity.
- **Implementation: lazy expiry on event arrival.** No background scan. Check `now_ms - last_seen_ms > cold_after_ms` when entity is touched.
- **Storage of `last_seen_ms`:** Already part of entity state (used by `last_seen` op). No new field.
- **Architectural test pair vs single test:** Single test sufficient (`phase12_8_lifetime_ops_have_bounds.rs`). The companion-file pattern from 12.6 Plan 10 / 12.7 D-03 was for ASSERTING DELETED FILES STAY DELETED — there are no such files in 12.8.
- **REQUIREMENTS.md positive anchors:** `V0-MEM-GOV-01` (cold-entity TTL opt-in), `V0-MEM-GOV-02` (lifetime ops have bounds), `V0-MEM-GOV-03` (per-event bucket reclaim).
- **Section-ownership:** Closure plan owns STATE.md / ROADMAP.md / CORRECTNESS-PATH.md / CLAUDE.md / 12.8-SUMMARY.md / 12.8-VERIFICATION.md. Earlier plans don't touch these (matches 12.6 Plans 09+13, 12.7 Plan 10).

### Claude's Discretion (planner decides)
- **Plan ordering / wave shape.** Suggested: Wave 1 = register-time shim (D-03) + `cold_after` decorator parsing; Wave 2 = cold-entity TTL implementation + per-op bound declarations; Wave 3 = metrics wiring + architectural test; Wave 4 = perf rebaseline + closure.
- **Per-op cap kwarg naming.** `histogram(buckets=...)` vs `histogram(num_buckets=...)` etc. Planner picks based on Pythonic + existing SDK conventions.
- **Sampling strategy for `bytes_per_entity_p99` gauge.** Sample every N events (planner-chosen N), or compute on snapshot trigger only. Affects metric freshness vs hot-path cost.
- **`cold_after` decorator parser tests.** Standard test surface; planner picks coverage.
- **Test file naming.** `phase12_8_*` prefix; specific names planner-chosen.

<specifics>
## Specific Ideas

- **`@bv.event(cold_after='7d')` is the canonical v0 surface.** Self-documenting; per-source granularity; declarative; durable via registry.
- **Implicit lifetime is the SDK contract.** `bv.count()` without `windowed=` IS lifetime. `bv.count_distinct()` IS lifetime + bounded sketch. Users learn one rule.
- **Hard reject at register-time is the v0 commitment.** Aligns with "users size their box": memory budget must be explicit at registration. No silent defaults; no auto-cap.
- **Per-event reclaim during `update_at()` is the canonical mechanism.** Already implemented; D-04 verifies + tests.
- **Architectural test pair pattern is overkill for 12.8** — single architectural test (`phase12_8_lifetime_ops_have_bounds.rs`) sufficient. 12.6/12.7 used companion file-absence tests because there were files to assert deleted; 12.8 has no analogous deletion target.
- **No new wire opcodes; no new error codes beyond `unbounded_op_in_lifetime_mode`** — Phase 12.8 is a SHIM + CONTRACT phase, not a wire-shape change.
- **Estimated scope:** ~6–10 plans across 3–4 waves. Mostly contract additions + metrics + tests. No deletion sweep (unlike 12.6/12.7).

## SDK example after Phase 12.8
```python
@bv.event(cold_after='30d')
class Transaction:
    user_id: str
    amount: float
    merchant: str

# Lifetime aggregations (windowed= omitted) — each declares a memory bound:
total_count = bv.count()                              # O(1) lifetime
distinct_merchants = bv.count_distinct(col='merchant') # HLL ~12KB lifetime
top_5 = bv.top_k(col='merchant', k=5)                 # SpaceSaving ~1KB lifetime
hist = bv.histogram(col='amount', num_buckets=256)    # 256-bucket lifetime — REQUIRED
last_5 = bv.last_n(col='amount', n=5)                 # bounded n=5 lifetime

# Windowed aggregation — explicit window means trailing-bucket eviction during update_at:
avg_60s = bv.avg(col='amount', windowed='60s')        # rolling 60s; reclaims as it advances

# After 30 days of no events for a user_id, that user's state is evicted on next event.
# No code change needed; opt-in via cold_after on the source.

# Register — register-time validator runs the 4 JSON-prelude shims:
app.register(Transaction, total_count, distinct_merchants, top_5, hist, last_5, avg_60s)
```
