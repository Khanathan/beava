# Phase 12.8 Plan 03: Cold-entity TTL eviction on apply hot path — Summary

Wires Plan 02's `EventDescriptor.cold_after_ms` field into the mio apply hot
path. When the source has `cold_after_ms = Some(N)` and the apply path
reaches an entity whose `last_seen_ms` is more than N ms ago, the entity's
prior state is CLEARED (FRESH state on resurrect, Redis TTL pattern, locked
permanent per CONTEXT D-04). When `cold_after_ms = None` (default), the
eviction check is a single `Option::is_some` branch — zero per-event cost
over the pre-12.8-03 baseline.

## What Shipped

**AggStateTable additions (`crates/beava-core/src/agg_state_table.rs`):**

- 3 new public sidecar HashMaps mirroring the existing 3-shape dispatch:
  ```rust
  pub last_seen_u64: HashMap<u64, u64, FxBuildHasher>,
  pub last_seen_str: HashMap<CompactString, u64, FxBuildHasher>,
  pub last_seen_multi: HashMap<EntityKey, u64, FxBuildHasher>,
  ```
- `evict_entity_by_shape_if_cold(shape, now_ms, cold_after_ms) -> bool`:
  reads the entity's `last_seen_ms` from the appropriate sidecar; if
  `now_ms.saturating_sub(last_seen) > cold_after_ms`, REMOVES both the
  state Vec<AggOp> AND the sidecar entry; returns `true` (evicted) /
  `false` (warm or absent). `saturating_sub` handles wall-clock skew.
- `record_last_seen_by_shape(shape, now_ms)`: stamps the sidecar after
  each apply call. Uses `raw_entry_mut::from_key` on str/multi shapes to
  skip key-cloning on the warm path (the common case once an entity is
  warm); single `insert` on u64 shape (fast-no-alloc Copy).

**apply_event_to_aggregations signature change (`crates/beava-core/src/agg_apply.rs`):**

```rust
pub fn apply_event_to_aggregations(
    source_name: &str,
    row: &Row,
    now_ms: i64,
    _event_id: u64,
    registry: &Registry,
    state_tables: &mut StateTables,
    cold_after_ms: Option<u64>,  // <-- NEW (Plan 12.8-03)
) {
```

Inside the per-aggregation loop, between `state_tables[agg_idx]` access and
`get_or_init_by_shape`:

```rust
if let Some(ttl_ms) = cold_after_ms {
    let _evicted = table.evict_entity_by_shape_if_cold(shape, now_ms as u64, ttl_ms);
    // _evicted ignored here; Plan 06 wires the metric counter via this signal.
}
let entity_row = table.get_or_init_by_shape(shape, &desc);
```

After the feature-update loop (and before the timing measurement):

```rust
if cold_after_ms.is_some() {
    table.record_last_seen_by_shape(shape, now_ms as u64);
}
```

## Per-event Cost (informal — Plan 08 microbench measures formally)

- `cold_after_ms = None` (the default, sources that don't opt in): one
  `Option::is_some` branch (~1 ns); branch predicted not-taken in steady
  state. **Zero cost over pre-12.8-03 baseline.**
- `cold_after_ms = Some + warm` (entity recently active): one branch + one
  HashMap lookup (~10-15 ns) + integer comparison + `record_last_seen`
  HashMap upsert (~10-15 ns). Total ~25-35 ns per touched aggregation.
- `cold_after_ms = Some + cold` (entity past TTL — rare event, dominated by
  memory-free overhead): one branch + lookup + comparison + 2 HashMap
  removes (state map + sidecar). Cold path is by definition rare.

## Lessons Learned

- **Sidecar HashMap pattern keeps the Vec<AggOp> value type stable.** The
  alternative — augmenting the value to `(Vec<AggOp>, u64 last_seen_ms)` —
  would have churned ~30 sites including snapshot encoding (`iter_sorted`),
  query path (`query_feature`), recovery path (`insert_from_entity_key`),
  and the legacy apply oracle. Sidecar's blast radius was 1 file (struct
  definition) + the apply hook. Cost is ~16 bytes/entity for sources that
  opt in; zero for sources that don't (the field is `HashMap::new()`-empty
  and unread until `cold_after_ms.is_some()`).

- **Adding an `Option<T>` parameter to a hot-path function cascades widely
  in test fixtures.** `apply_event_to_aggregations` had 8 caller sites
  beyond the plan-named 3. The remediation is mechanical (`None` everywhere
  in tests), but underestimating the cascade by a factor of 3 is a planning
  trap: future plans changing this signature should expect ≥10 sites.

- **`raw_entry_mut::from_key` is the right primitive for "stamp this key,
  zero-clone if present".** The naive `insert(s.clone(), now_ms)` always
  clones; `raw_entry_mut` skips the clone on the warm path. On the
  `record_last_seen` hot path, this matters — the warm path is invoked
  ~1 per event per opt-in source.

- **Cold-TTL eviction inline at the apply path's per-aggregation loop is
  the correct layering** — it sits between the per-aggregation shape
  derivation and the `get_or_init_by_shape` call, naturally inheriting
  the cluster-cache and shape-dispatch machinery. Lifting it to the
  `apply_shard.rs` orchestration layer would have duplicated the
  shape-derivation logic and broken the "single owner of state mutation"
  invariant.
