## Executive summary

Phase 12.8 delivers the memory governance commitments locked by `project_no_sharded_apply`: **per-entity memory ceilings are now explicit at registration**. Two-tier hygiene + lifetime aggregation contract land before final ship (Phase 13).

**Tier 1 — entity-level cold-entity TTL** (Plans 02 + 03): opt-in via `@bv.event(cold_after='7d')` decorator (range `[1s, 365d]`, `s|m|h|d` duration parser); default = `None` (no expiry) preserves existing user behavior. Lazy expiry on event arrival in `apply_shard.rs::dispatch_push_sync`: if `now_ms - last_seen_ms > cold_after_ms`, entity state RESETS (FRESH state on resurrect — Redis TTL pattern, locked permanent per `project_no_sharded_apply`). Sidecar `last_seen_ms: HashMap<EntityKey, u64>` per source on `AggStateTable`. Per-event check cost: ~5-15 ns when None, ~30-50 ns when set.

**Tier 2 — bucket-level reclaim within active entities** (Plan 03 + 06): existing per-event eviction during `update_at(now_ms)` is canonical; `BucketReclaimCounter::inc` increments on bucket trim. No additional scanning of idle entities; no background thread. Verified via test.

**Lifetime aggregation contract** (Plans 01 + 04 + 05): explicit `windowed=` omission semantics for `[first_seen, now)` forever-window. Each of the 53 AggKind variants (54 op-strings via `ema` alias) classifies as `O1` / `BoundedSketchN` / `BoundedByRequiredKwarg(name)` / `BoundedByConfig(name, default)`. The 4th JSON-prelude shim `pre_check_unbounded_op_in_lifetime_mode` in `register_validate.rs` rejects `Unbounded` ops at register-time with structured error code `unbounded_op_in_lifetime_mode` + suggested cap kwarg. Forward-looking framing ("requires explicit memory bound in v0", NOT "feature removed").

**Per-call env-var read** (Plans 01 + 06 — per checker B-02 fix): `memory_gov_enforce_enabled()` reads `std::env::var("BEAVA_MEMORY_GOV_ENFORCE")` on every register dispatch (cold path). Avoids `OnceLock<bool>` that would lock first observed value across all tests in a binary. Plan 06 flipped the default from OFF to ON; the explicit escape hatch is `BEAVA_MEMORY_GOV_ENFORCE=0`.

**5 Prometheus metric families** (Plan 06): `cold_entity_evictions_total`, `lifetime_op_cap_hit_total`, `entity_count_resident`, `bytes_per_entity_p99`, `bucket_reclaim_total` — all wired into existing `/metrics` admin sidecar (tokio). UNLABELED in v0; per-source labels deferred to v0.0.x.

## Lessons learned

These patterns emerged worth saving for future phases:

### 1. 4-shim cumulative pattern in `register_validate.rs` (Plans 12.6-04 + 12.6-06 + 12.7-01 + 12.8-01)

Four sequential JSON-prelude shims now coexist in `register_validate.rs`:
1. `pre_check_removed_ops` (12.6-04 — joins/unions)
2. `pre_check_legacy_event_time_keys` (12.6-06 — event_time_field/tolerate_delay)
3. `pre_check_unsupported_node_kind` (12.7-01 — table)
4. `pre_check_unbounded_op_in_lifetime_mode` (12.8-01 — unbounded ops)

**Pattern is mature.** Each shim walks `body.get("nodes")?.as_array()?`, inspects per-node fields, returns structured error code + message. Reusable for any future v0.x or v0.1+ feature retirement / contract addition. Proven across 4 shims now.

### 2. Per-call env-var read for register-time gates (Plans 01 + 06 — per checker B-02)

`std::env::var("BEAVA_MEMORY_GOV_ENFORCE")` read on every register dispatch (cold path; ~10-100 events/sec at production rates). The OnceLock-cached pattern from elsewhere in the codebase (e.g. `apply_shard.rs:269-273` for trace flags) is the WRONG pattern for register-time gates because it locks the first observed env value for the lifetime of the process — breaks per-test env flips in integration test binaries.

**Lesson:** OnceLock optimization is justified for hot-path env reads (per-event); per-call reads are correct for cold-path env reads (per-request). Don't paste-pattern OnceLock blindly.

### 3. Soft default for backward compatibility (Plan 04 `top_k`)

`top_k` classified as `BoundedByConfig("k", 10)` rather than `BoundedByRequiredKwarg("k")` to preserve backward-compat with ~10 existing tests passing `k.unwrap_or(10)`. Documented under Claude's Discretion in CONTEXT.md.

**Lesson:** New strict contracts can include explicit soft defaults to ease migration. The trade-off is the contract is slightly weaker; users who rely on the default can hit the cap silently. The metric `lifetime_op_cap_hit_total{op="top_k"}` exposes when the soft default is in play. Future Phase 13 hardening item: consider promoting `top_k` to `BoundedByRequiredKwarg` with explicit migration message.

### 4. Process-static counter pattern (Plan 06)

Process-static `AtomicU64` / `AtomicUsize` counters (e.g., `ColdEntityEvictionCounter::inc()`) are simpler than threading metric handles through `AdminState` to every call site. Phase 19.2 D-05a's `EntropyStateWrap::categories_capped_count` set the precedent; Plan 06 extended to 4 sibling counters. Avoids 10+-file `AdminState` plumbing cascade.

**Lesson:** For aggregate-only metrics (no per-source labels), process-static counters are the cleanest pattern. When v0.0.x adds per-source labels, the counter shape changes (probably HashMap of AtomicU64 keyed by source_id) but the lookup pattern stays similar.

### 5. Single-file architectural test sufficient when no deletion targets (Plan 05)

Single-file `phase12_8_lifetime_ops_have_bounds.rs` is sufficient; no companion needed.

**Lesson:** Architectural-test pair pattern is for "deleted things stay deleted" scenarios. Pure additions need only a single contract test. Don't reflexively pair-pattern when the cost of the second file isn't paying for itself.
