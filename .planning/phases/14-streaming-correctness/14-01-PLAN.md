  </behavior>
  <action>
    Wire the drop into `push.rs` step 8 (inside the existing `borrow_mut` scope once Phase 13.3 lands; currently inside the `app.dev_agg.state_tables.lock()` scope pre-13.3 — rebase onto 13.3 before this task if not merged).

    Order inside `borrow_mut` (post-13.3):
    1. Compute `tolerate_delay_ms` for this event's descriptor: `descriptor.tolerate_delay_ms.unwrap_or(DEFAULT_TOLERATE_DELAY_MS)`.
    2. `if app_state.watermarks.is_late(event_name, event_time_ms, tolerate_delay_ms) { ... drop ... return PushOutcome::LateDrop { ack_lsn } }` — still succeed the push (client sees 200 + ack_lsn); drop happens silently.
    3. Drop branch: `watermarks.record_drop(event_name)`; increment Prometheus counter; `if watermarks.should_log_now(event_name, now_ms_unix()) { tracing::warn!(stream=event_name, event_time_ms, watermark_ms=?, drift_ms=?, "late event dropped") }`.
    4. Non-drop branch: `watermarks.observe(event_name, event_time_ms as u64)` THEN call `apply_event_to_aggregations(...)` (unchanged).

    **Critical:** the watermark is observed ONLY on non-dropped events. A dropped event must not advance the watermark — that would mean the drop was for nothing.

    In `metrics.rs` (Phase 13 `/metrics` wiring): declare
    ```rust
    pub static EVENTS_DROPPED_LATE_TOTAL: Lazy<IntCounterVec> = Lazy::new(|| ...);
    ```
    (use existing IntCounterVec pattern from Phase 13 Plan 13-01).

    Rate-limited log impl: `should_log_now` is already the contract (Task 1). Pass `now_ms_unix()` — wall-clock is fine for log rate-limiting (not deterministic-state-affecting).

    **Note on WAL:** the event is still WAL-appended (step 7 runs before step 8). This matches D-07 — replay reproduces the drop deterministically because the same watermark check runs on replay. See Plan 14-03 Task 2.
