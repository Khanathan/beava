Validator surface (register_validate.rs): the validator already produces structured errors per Phase 2. Add a new error code `tolerate_delay_exceeds_window` returned as HTTP 400 / TCP error_response.

Recovery surface (recovery.rs:18,49,66-69,121,159,171 — see head of that file): the replay loop already updates `max_event_time_ms`. This plan adds a call to `app_state.watermarks.observe(event_name, event_time_ms as u64)` at the same site, per-event, so per-stream watermarks rebuild in lockstep with the existing global `max_event_time_ms`.

Important subtlety: replay MUST replay drops too. If an event was dropped at live-runtime (late), the same event on WAL replay must STILL be dropped at the same spot so post-restart aggregation state matches pre-restart. This works automatically IF the replay loop applies the same `is_late` check before `apply_event_to_aggregations` — which it gets for free if we reuse the Plan 14-01 push-path drop check.

Two viable impls for "apply the drop check on replay":
- A: refactor Plan 14-01's drop block into a free function `maybe_apply_event(...)` and call it from both push.rs and recovery.rs. Preferred.
- B: re-implement the is_late check inline in recovery.rs. Rejected — too easy to drift.
