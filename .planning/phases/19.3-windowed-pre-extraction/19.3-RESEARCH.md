**Researched:** 2026-04-28

**Confidence:** HIGH for code-reference findings (verified by grep + file read); MEDIUM for predicted-lift translation (depends on Phase 19.2 investigation §1 measurements which were captured at trace cost — investigation footnotes a 2,526 ns scaffolding gap between per-AggKind sum and outer features-stage that the planner should be aware of when validating sub-goal lifts).

## 1. Summary

Four open questions resolve cleanly from the source. (1) The per-thread `Vec<Value>` for hoisted ExtractedFields belongs as a local variable in `apply_event_to_aggregations` next to the existing `shape_cache` — no new struct needed; there is no `AggApplyContext` today and `StateTables = Vec<AggStateTable>` is a bare type alias, so introducing one for a single buffer would be over-engineering. (2) `WindowedOp::update_with_row` has zero production callers outside `agg_op.rs:817,868` and the `WindowedOp::update_with_row` recursion at `agg_windowed.rs:202,209`; all other refs are tests or the `windowed_op_init` cold bench. The protocol extension reaches everything that matters by changing only `AggOp::update_with_extracted::Windowed(…)` arm and adding a sibling `update_at` on `WindowedOp` that delegates to `AggOp::update_with_extracted` (already exists). (3) The criterion bench helper `build_fraud_team_synthetic_registry` does NOT support `window_ms` — every one of the 14 features hard-codes `window_ms: None`. Plan 19.3-01 needs to add a parallel `build_fraud_team_synthetic_registry_windowed(window_ms: u64)` builder (or a single parametrized builder) whose 14 features all carry `window_ms: Some(window_ms)`. (4) **There is no `RetractableOp` in the codebase.** `WindowedOp` is the SOLE wrapper variant of `AggOp` (search `enum AggOp` returns 41 variants; the only one that wraps another `AggOp` is `Windowed(Box<WindowedOp>)`). Phase 19.3's enumeration obligation is satisfied by covering WindowedOp alone — the wrapper-bypass anti-pattern from `feedback_dispatch_refactor_enumerate_wrappers` is satisfied trivially because there are no other wrappers.

**Welford state cross-check:** `VarianceState::update_pre` (`agg_state.rs:357-369`) and the windowed-inline arm Plan 19.3-03 introduces must produce bit-identical `(n, mean, m2)` after the same event sequence. The non-windowed reference algorithm is the 5-line Welford at `agg_state.rs:364-368`; the inline arm must match line-for-line (delta is computed BEFORE mean update, delta2 AFTER). The bit-identical test compares `VarianceState` field-equal across a non-windowed `VarianceState::update_pre` loop and a `WindowedOp` (window_ms larger than the event timespan, so all events fall into one bucket) calling the new specialized arm.

**Primary recommendation:** Plan 19.3-02 (the 19.3-A landing) should reshape `AggOp::update_with_extracted` Windowed arm to call a new `WindowedOp::update_at(extracted, field_idx, lat_idx, lon_idx, event_time_ms, where_matched)` that mirrors the bucket-routing logic of `update_with_row` but, on the per-bucket call, dispatches via `AggOp::update_with_extracted` (already on the protocol) instead of `AggOp::update_with_row`. The `where_matched` boolean is computed ONCE in the outer `update_with_extracted` (line 860-863) and threaded down — eliminating the per-bucket `evaluate_where_predicate` re-evaluation. Predicted lift held; the protocol surface is small and matches existing non-windowed pattern.

## 2. Open Question Resolutions

### Q1 — Where should the per-thread `Vec<Value>` ExtractedFields buffer live structurally?

**Findings:**
- `crates/beava-core/src/agg_apply.rs:90-96` — `apply_event_to_aggregations` takes `state_tables: &mut StateTables` and `registry: &Registry`; it is a free function, not a method on a context struct.
- `crates/beava-core/src/agg_state_table.rs:383` — `pub type StateTables = Vec<AggStateTable>;` is a bare type alias. There is no struct hosting per-thread scratch buffers today.
- `agg_apply.rs:118-152` — per-event scratch (`t_*` durations, `per_kind`, `shape_cache`) is declared as **local variables inside the function**. The `shape_cache: Vec<Option<Option<EntityKeyShape>>>` (line 152) is the closest precedent: per-event allocated, indexed by cluster_id, lives on the stack frame.
- Searched for `AggApplyContext|ApplyContext` across `crates/beava-core/src/` — zero hits.
- Searched for `thread_local!` in apply-path — found at `agg_state_table.rs:359` (test-only `EKS_BUILDS` counter) and `row.rs:526`, `bloom.rs:110` (op-internal use). None on the apply hot path.

**Recommendation: Option (e) — Keep it as a local variable in `apply_event_to_aggregations`, declared once at the top of the function alongside `shape_cache`.**

Specifically:
```rust
// At apply_event_to_aggregations entry, after shape_cache (line 152):
let mut hoisted_extracted: Vec<Option<&Value>> = Vec::new(); // sized lazily on first use per cluster
```

Then build it ONCE per event (the "C" hoist) using the union-of-fields list. There are two sub-options for the union list:
- **(e1) Build per-source from `EventDescriptor.apply_field_names`** — that field exists at `registry.rs` (declared at line 74) but is currently `vec![]` everywhere it is constructed. **Plan 19.3-04 needs a registration-time pass to populate `apply_field_names` as the union of all `agg.field_names` across aggs sourced from this event.** The hoist would then be: `for (i, fname) in src_event.apply_field_names.iter().enumerate() { hoisted_extracted[i] = row.get(fname); }`, and per-agg `feat.descriptor.field_idx` would index into this single per-event array (instead of indexing into the per-agg `desc.field_names` rebuild at lines 201-205).
- **(e2) Hoist the per-agg rebuild but dedupe across descs in the same event** — simpler but smaller win because aggs with non-overlapping fields still pay full cost.

**Recommend (e1).** The investigation §3 explicitly calls out `EventDescriptor.apply_field_names` as "already exists per agg_apply.rs context — just unused at the per-event level". This is half-true: the *field* exists; the registration logic that populates it does not yet. Plan 19.3-04 is one ~30-line registration pass + one apply-loop hoist + a re-indexing of `field_idx` (which currently indexes into `agg.field_names` — must shift to index into `event.apply_field_names`).

**Why not the other options:**
- (a) `AggApplyContext` — no such struct; introducing one is bigger surface than the problem warrants. The only state needed is one `Vec<Option<&Value>>`.
- (b) `thread_local!` — possible but worse: lifetime-erases the borrowed `&Value` references and forces `unsafe` or owned `Value` clones, which defeats the purpose. Local stack frame avoids this entirely (the `hoisted_extracted` borrows live as long as `row` lives, which is the function scope).
- (c) New `ApplyState` struct — same over-engineering as (a). One buffer doesn't justify one struct.
- (d) On `AggStateTable` — would couple per-event scratch to per-agg storage (one of N tables); wrong shape.

**Acceptance criterion (concrete):** `cargo expand` (or just visual inspection) of `apply_event_to_aggregations` shows exactly ONE `Vec<Option<&Value>>` allocation per call when `apply_field_names` is non-empty, replacing the current N×per-desc rebuild at lines 201-205. Line 201-205's `let extracted: ExtractedFields = desc.field_names.iter().map(|f| row.get(f.as_str())).collect();` is deleted; the per-feature `feat.descriptor.field_idx` looks up against the hoisted buffer instead.

**Implementation hazard:** `ExtractedFields<'_>` is currently a `SmallVec<[Option<&'a Value>; 8]>` (`agg_op.rs:232`). Plan 19.3-04 must either widen the SmallVec inline cap (because the per-event union is larger than the per-desc subset — fraud-team has up to ~12 distinct fields per source event) or accept a heap allocation when the union exceeds 8. Recommendation: bump inline cap to 16; benchmark in Plan 19.3-01's bench harness.

### Q3 — Does `apply_path_bench.rs` already support windowed registries?

**Finding (read of `apply_path_bench.rs:43-408`):** No. `build_fraud_team_synthetic_registry` constructs 14 `AggOpDescriptor` entries, every single one of which sets `window_ms: None`. Specific lines: `apply_path_bench.rs:79, 95, 111, 132, 148, 169, 188, 226, 242, 259, 274, 311, 327, 343` — all 14 features hard-code `window_ms: None`. The closest to "windowed" is `txn_amount_ewma` with `half_life_ms: Some(3_600_000)` (line 135), which is decay-based (no bucket ring) and goes through `EwmaState`, NOT `WindowedOp` (per investigation §2.1).

**Recommendation: smallest-diff approach for Plan 19.3-01.** Refactor `build_fraud_team_synthetic_registry` to take a `window_ms: Option<u64>` parameter and thread it through every feature's `window_ms: window_ms` slot. Then add two callers:
```rust
fn build_fraud_team_synthetic_registry() -> Arc<Registry> {
    build_fraud_team_synthetic_registry_inner(None)
}
fn build_fraud_team_synthetic_registry_windowed(window_ms: u64) -> Arc<Registry> {
    build_fraud_team_synthetic_registry_inner(Some(window_ms))
}
```
And add bench groups `apply_path/cold_key/14_aggs_windowed` + `apply_path/warm_key/14_aggs_windowed` that call the windowed builder with `window_ms = 86_400_000` (24h, matching the predicted-lift table assumptions).

**Caveat (HIGH confidence; verified by reading `agg_op.rs:467-477`):** `AggOp::new` only wraps in `WindowedOp` if `desc.kind.supports_windowed_wrap()` is true. Per `agg_windowed.rs:464` and `:473`, "Phase 8 + 9 + 11 ops are never wrapped in WindowedOp." This means the bench's current 14 features need to be filtered when running the windowed variant — `BloomMember` (`apply_path_bench.rs:341`) is windowless-only and will panic if `window_ms` is set (`agg_windowed.rs:97-100`: `assert!(!matches!(kind, AggKind::BloomMember), ...)`), and `EventTypeMix`, `Ewma` are not in the supports_windowed_wrap set either.

**Plan 19.3-01 should:** EITHER (a) drop the non-windowable 3 features and add 3 substitute Tier-1 windowable features (e.g., 3× more `Count` or `Sum` variants — keeps the bench at 14 features so the cold/warm comparison is direct); OR (b) accept a 11-feature windowed bench and document the subset in a code comment. **Recommend (a)** — preserves the "14_aggs" naming parity with the non-windowed bench so the criterion comparison column-aligns.

The CONTEXT D-04 acceptance criterion already names the new bench `apply_path/warm_key/14_aggs_windowed` — keeping 14 features keeps the criterion-comparison side-by-side legible.

**`supports_windowed_wrap` reference (`agg_op.rs` near AggKind impl, line ~181):** Plan 19.3-01 must read this to confirm exactly which Tier-1 kinds qualify. Per investigation §2 the answer is: Count, Sum, Avg, Min, Max, Variance, StdDev, Ratio, CountDistinct, Percentile, TopK, Entropy. **Twelve windowable kinds.** The bench's current 14 features collapse: drop BloomMember, EventTypeMix, Ewma; replace with 3 of {windowed StdDev, windowed Ratio, windowed Min — anything in the supported set that exercises Tier-1 inline path that 19.3-B specializes}.

### Welford bit-identical cross-check test design

```rust
// New test in crates/beava-core/src/agg_windowed.rs
#[test]
fn variance_windowed_inline_matches_non_windowed() {
    // Construct two states. The non-windowed VarianceState is the reference.
    let mut reference = VarianceState::default();
    // The WindowedOp wraps Variance with a window large enough that all events
    // fall in a single bucket — the windowed-inline arm runs against the same
    // logical state as the reference.
    let window_ms = 100_000_000_u64; // > the timespan of all events below
    let mut windowed = WindowedOp::new(AggKind::Variance, window_ms);

    // Deterministic event sequence: 100 amounts with realistic floating-point spread
    // (large + small mixed) to exercise Welford's numerical stability path.
    let amounts: Vec<f64> = (0..100).map(|i| {
        // Mix small + large + negative to exercise mean drift.
        let base = (i as f64) * 1.7;
        if i % 3 == 0 { base * 1e6 } else if i % 3 == 1 { -base } else { base * 1e-3 }
    }).collect();

    for (i, &x) in amounts.iter().enumerate() {
        let row = Row::new().with_field("amount", Value::F64(x));
        let pre_val = Some(&Value::F64(x)); // pre-extracted
        // Update reference via Plan 19.2-01's pre-extracted path:
        reference.update_pre(pre_val, true);
        // Update windowed via the new specialized arm (Plan 19.3-03):
        let extracted: ExtractedFields = smallvec::smallvec![pre_val];
        windowed.update_at(&extracted, 0_u8, FIELD_IDX_NONE, FIELD_IDX_NONE,
                           1_000_000 + i as i64, true);
    }

    // Extract the windowed VarianceState from the (single) bucket.
    assert_eq!(windowed.buckets.len(), 1, "all events should land in one bucket");
    let bucket_op: &AggOp = &windowed.buckets[0].1;
    let inner = match bucket_op {
        AggOp::Variance(s) => s,
        _ => panic!("expected Variance inner state"),
    };

    // Bit-identical: same n, same mean (f64 bit pattern), same m2 (f64 bit pattern).
    assert_eq!(reference.n, inner.n, "n diverged");
    assert_eq!(reference.mean.to_bits(), inner.mean.to_bits(),
               "mean diverged: ref={} new={}", reference.mean, inner.mean);
    assert_eq!(reference.m2.to_bits(), inner.m2.to_bits(),
               "m2 diverged: ref={} new={}", reference.m2, inner.m2);
}
```

The `to_bits()` comparison is the strict criterion — any reordering of additions or any algebraic re-arrangement (e.g., computing `delta2 = x - new_mean` BEFORE storing the new mean vs AFTER, or `m2 += delta * (x - mean)` vs `m2 += delta * delta2`) shows up as a different bit pattern. **Plan 19.3-03 must inline Welford line-for-line per `agg_state.rs:364-368`** — no algebraic shortcuts.

## 5. Open Risks

### R1 (HIGH) — `ExtractedFields<'_>` SmallVec inline cap of 8 may spill to heap on per-event union (19.3-C)
**Source:** `agg_op.rs:232` — `pub type ExtractedFields<'a> = smallvec::SmallVec<[Option<&'a crate::row::Value>; 8]>;`
**Why it matters:** the per-agg `extracted` rebuild today is sized to the per-agg distinct-field count (typically ≤ 8). The per-event hoisted union across all aggs sourced from this event is larger — fraud-team's Txn source has ~12 distinct fields aggregated across all aggs. Inline cap 8 → spill → 19.3-C's "zero per-event allocations" acceptance criterion fails on fraud-team specifically.
**Mitigation:** Plan 19.3-04 widens the inline cap to 16 (or makes it generic; current callers can stay on the 8-cap alias). Validate via `peak_alloc` in the criterion warm-key bench. If the widened cap regresses cache behavior on small-field-count workloads, fall back to 12 (covers fraud-team Txn) or accept a one-time spill on the first event of a hot key.

### R2 (MEDIUM) — Per-AggKind trace cost itself is non-trivial (~2,526 ns scaffolding, investigation §1)
**Source:** investigation §1 footnote — the per-kind sum is 15,192 ns/event but outer features-stage is 17,718 ns; the 2,526 ns delta is dispatch + ExtractedFields build + match-arm scaffolding NOT inside the per-op timer.
**Why it matters:** Plan 19.3 verification compares pre-19.3 trace (with the same scaffolding) to post-19.3 trace.
