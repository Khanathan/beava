## Domain

Close the fraud-team apply-stage WindowedOp dispatch tax. 60 of 88 fraud-team feature updates per event currently bypass Plan 19.2-01's field pre-extraction protocol because `WindowedOp::update_with_row` forwards `&Row` (not `&ExtractedFields`) and re-does `row.get(fname)` inside each bucket update. Phase 19.3 extends the `update_at(extracted, field_idx, …)` protocol across the WindowedOp wrapper, drops the agg-stage from 14,059 ns → ~8,450 ns, and lifts end-to-end EPS on fraud-team K=10k zipfian from 70k → ~115-125k.

Scope is FIXED by ROADMAP. Three stacked sub-goals (19.3-A → 19.3-B → 19.3-C) plus mandatory bench/trace-verification work. New capabilities (group-commit batching, sharding, etc.) are out of scope and belong in their own phases.

## Decisions

### D-01: Specialize ALL trivial Tier-1 windowed kinds (19.3-B scope expansion)

**Decision:** Inline the inner-state fast-path in `WindowedOp` for **Count, Sum, Min, Max, Avg, Variance** — all 6 Tier-1 kinds with windowed wrapping. Bypass the full `AggOp::update_with_row → AggOp::update → InnerState::update(row, …)` dispatch chain by matching on `inner_kind` in the windowed update entry and writing inner state directly.

**Why:** Originally scoped as Count+Sum only (~1,100 ns/event saved); user expanded to full Tier-1 windowed coverage. Predicted lift ~2,000 ns/event vs the ~1,100 ns of the narrower scope.

**Implementation notes for planner:**
- Count: `n += 1`
- Sum: `total += value` (Value::F64 / Value::I64 type-dispatch)
- Min: `cur = cur.map_or(v, |c| c.min(v))`
- Max: symmetric to Min
- Avg: maintain `(total, n)` tuple; defer division to query time (already the existing pattern)
- **Variance: Welford state** (`(n, mean, m2)`) — the inline path is non-trivial. Planner must verify the Welford update is correctly inlined, not reimplemented inconsistently with the non-windowed `VarianceState`. Acceptance criterion: cross-check that `update_with_extracted` and the new windowed-inline path produce bit-identical state for the same event sequence.

**Acceptance criteria for this decision:**
- `WindowedOp::update_with_row` (or new fast-path method) contains a match arm on `inner_kind` for {Count, Sum, Min, Max, Avg, Variance} that bypasses `AggOp::update_with_row`.
- Per-AggKind live trace shows windowed Count cost drops from ~180 ns/call → ≤ 30 ns/call; Sum ≤ 30 ns/call; others ≤ 60 ns/call (Welford has slightly more arithmetic).
- `cargo test --workspace` proves bit-identical state between windowed-specialized arms and the non-windowed equivalents under the same event sequence.

### D-02: Sequential landing — A → measure → B → measure → C

**Decision:** Land 19.3-A (WindowedOp::update_at fast-path) alone first. Re-run live trace + EPS measurement. Only proceed to 19.3-B once 19.3-A's lift matches predicted (~3,900 ns/event saved → ~95k EPS). Same gate between 19.3-B and 19.3-C.

**Why:** Phase 19.2 shipped microbench wins that did not translate to end-to-end EPS because the apply-loop optimizations targeted ~3% of the per-event budget. Sequential landing gates against repeating that pattern. Each sub-goal carries an explicit measurement contract; if measured lift undershoots predicted by >25%, planning re-evaluates before proceeding to the next sub-goal.

**Plan structure implication:** Phase 19.3 plans should be at minimum 4 plans (one per sub-goal A/B/C plus the bench amendment + rebaseline plan), each with its own verification gate. Plans for B and C list "preceded by A measurement gate" / "preceded by B measurement gate" as an explicit precondition.

### D-03: Reused per-thread `Vec<Value>` for hoisted ExtractedFields (19.3-C storage)

**Decision:** Single `Vec<Value>` on the apply thread. `clear() + resize_with(max_field_count, || Value::Null)` at the top of `apply_event_to_aggregations`, then populate by `field_idx` from registry. Reuse across events; no per-event allocation.

**Why:** Apply thread is single-threaded forever per memory `project_no_sharded_apply`, so per-thread storage is sufficient. Per-source-schema cached buffer (rejected) adds per-source state without measurable benefit since `resize_with` on an already-sized Vec is essentially free. Per-event arena (rejected) is overkill for a fixed-size buffer.

**Implementation notes for planner:**
- Buffer lives on the apply-thread context (`AggApplyContext` or equivalent — researcher must confirm where it should live; not in `EventDescriptor`).
- Pre-size during register based on max field-count across all sources (cheap, monotonic — only grows).
- Indexed by per-agg `field_idx` arrays (already exist post-Plan 19.2-01; just unused at the per-event level per investigation §3).

**Acceptance criteria for this decision:**
- Allocator profile (criterion `peak_alloc` or instrumentation): zero per-event allocations for ExtractedFields on the warm path.
- `extracted` rebuild count drops from `5 × per-event` (one per descriptor) → `1 × per-event`.

### D-04: `WindowedOp::update_at` signature + warm-key bench scope (19.3-A signature)

**Decision (signature):**
```rust
fn update_at(
    &mut self,
    extracted: &ExtractedFields,
    field_idx: u8,
    lat_idx: Option<u8>,
    lon_idx: Option<u8>,
    event_time_ms: i64,
    where_matched: bool,
)
```
Mirrors the existing non-windowed `update_at` signature. Per-bucket inner op dispatches via `AggOp::update_with_extracted` (already exists from Phase 19.2-01) instead of `AggOp::update_with_row`. Eliminates inner `row.get(fname)` linear scan AND inner `evaluate_where_predicate` re-evaluation.

**Decision (bench scope):** Add `apply_path/warm_key/14_aggs_windowed` criterion group ONLY. Same 14 features as the existing non-windowed bench, wrapped in `WindowedOp(window_ms = 24h)`. **No cold-key sibling for this phase.**

**Why (signature):** Forwarding `&ExtractedFields` + `field_idx` matches the non-windowed protocol exactly — same interface for outer + nested layers. Bucket-local field_idx recompute (rejected) pays the lookup tax we are trying to remove. The `lat_idx`/`lon_idx` Option pair preserves geo-op support.

**Why (bench scope only warm-key):** Matches Phase 19.2 bench shape. Cold-key (lazy-bucket-allocate) is exercised by the integration trace + small/medium/large throughput cells; a dedicated cold-key criterion group adds CI time without distinct signal Phase 19.2 already covers via `apply_path/cold_key`. The warm-key windowed bench is the new measurement instrument; cold-key sibling can be added in a future phase if a regression demands it.

**Acceptance criteria for this decision:**
- `WindowedOp::update_at` exists with the signature above; called from `AggOp::update_with_extracted::Windowed(…)` arm in `agg_op.rs`.
- `grep -c 'row.get(' crates/beava-core/src/agg_state.rs` ≤ 5 on apply-time hot paths (was ~30 callsites pre-19.3); any remaining call has explicit grandfathering rationale in code comment.
- `apply_path/warm_key/14_aggs_windowed` exists; baseline measurement is committed BEFORE 19.3-A's implementation (i.e. the bench is added in a "test:" red-commit per CLAUDE.md TDD rule, showing the pre-19.3 cost ~5 µs, then 19.3-A's "feat:" commit drops it to ~1 µs).
