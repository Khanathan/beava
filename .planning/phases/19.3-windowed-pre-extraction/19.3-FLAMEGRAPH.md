# Phase 19.3 — Apply-Path Flamegraph Deep-Dive

## TL;DR

A real flamegraph (samply 1500 Hz, 3,954 samples on `beava-apply` thread, 2.64 s CPU time) reveals **three high-leverage optimization opportunities the cost-model investigation missed** — all of which substantially shift the predicted-lift table:

1. **CountDistinct's `HashSet` mode uses `std::HashSet<u64>` with SipHash** even though the input `u64` is already a hash. SipHash hashing the u64 again costs ~358 samples = **9.0% of apply-thread CPU = ~1,180 ns/event** for fraud-team's 9 windowed CountDistinct features. **Switching to `hashbrown::HashSet<u64, BuildHasherDefault<FxHasher>>` (or just letting CountDistinct skip the wrapping HashSet entirely and write to a `Vec<u64>` until promotion) saves nearly all of it.** This is *bigger than the entire predicted lift of 19.3-B (1,394 ns/event)* and lands as a 1-file change (~10 LOC).

2. **TopK uses `BTreeMap::entry` for the Exact-mode counter** — 5.7% self-time = ~750 ns/event for 2 windowed TopK features. fraud-team's K=10k cardinality blows past TopK's exact-threshold quickly, so most calls are in Hybrid mode where the cost is `CountMinSketch::insert + heap.insert_or_bump`. But the Exact arm's BTreeMap allocator + balance is still on the hot path. Pre-allocate the spine, or switch to a flat `Vec<(TopKValue, u64)>` capped at exact-threshold (typical 64) for ~500 ns/event.

3. **`ExtractedFields` SmallVec[8] inline cap is wrong for fraud-team** — TxnByUser cluster has **10 distinct fields** referenced across its features, so the SmallVec spills to heap on every Txn event. `RawVecInner::reserve` and `RawVec::with_capacity_in` together appear at 4.0% inclusive = ~530 ns/event of allocator traffic on the hot path. **Widening the inline cap from 8 to 16 (one-line change) eliminates most of it.** This is 19.3-C-adjacent but covers a separate cost item the COST-MODEL report also missed.

These three changes together predict **~2,100 ns/event saved** — comparable in size to the originally-predicted 19.3-A lift, achievable in a single afternoon, and at **lower complexity than 19.3-B**. Combined with the existing 19.3-C plan they push fraud-team K=10k zipfian toward the **90-100k EPS** range — putting the original Phase 19.3 PASS gate of 100k EPS back in reach **without** Phase 19.4's WAL group-commit work.

**Recommendation:** add these three changes to Phase 19.3 as `19.3-D` (sized as a small follow-up plan, not a full sub-goal), land them BEFORE 19.3-C because they're independent and lower-risk, and re-measure before deciding whether 19.3-B is needed at all.

### 4.2 Why fraud-team CountDistinct is mostly in HashSet mode

K=10k zipfian distinct cardinality across 9 windowed CountDistinct features × 10k entities. With `EXACT_THRESHOLD=16` and `HASH_THRESHOLD=1024`, hot zipf-head entities promote to HLL quickly, but tail entities stay in HashSet mode for a long time. Sampling the steady-state distribution: roughly 60% of CountDistinct calls land in HashSet mode (matches the 56% HashSet leaf + 5% mode-check + 8% pre-hash distribution).

### 4.3 Mode-promotion check itself

`count_distinct.rs:58` is `if let Err(pos) = values.binary_search(&hash)` — that's only the ExactArray arm. Line 71-72 is the HashSet arm. Line 84 is the Hll arm. The trace shows leaf samples distributed across all three — confirming a healthy mix in fraud-team.

**Implication:** lowering `HASH_THRESHOLD` from 1024 to 64 (or some value where the HashSet mode is rarely entered) would push more calls into HLL mode where `hll.add_hash` (~50 ns) is much faster than `HashSet<u64>::insert` (~140 ns). This is a knob that doesn't require any code surgery — only a const change. Trade-off: HLL is approximate (relative error ~1.5% at p=12); for fraud-team's CountDistinct features used in scoring, that's almost certainly fine. **See §7 Optimization #1 for the SipHash fix which is even simpler.**

## 7. Optimization proposals (top 5, ROI-ranked)

### #1 — Replace `std::HashSet<u64>` in CountDistinct with `hashbrown::HashSet<u64, BuildHasherDefault<FxHasher>>` OR a flat `Vec<u64>` until promotion

**Hypothesis:** `CountDistinctState::HashSet { hashes: std::collections::HashSet<u64> }` rehashes the input u64 with SipHash on every insert. Since the input IS already a hash from FxHasher, a no-op identity hasher (or just a flat Vec until the threshold is hit) would skip SipHash entirely.

**Predicted lift:** **~1,180 ns/event** = 56% of CountDistinct cost = 9% of apply-thread CPU. Sets the EPS up by `1 / (1 - 1180/13180) = 9.8%` ≈ **80,000 EPS** on its own.

**Specific code-level change:**

`crates/beava-core/src/sketches/count_distinct.rs:24-27`:

```rust
// BEFORE
#[serde(rename = "v0_count_distinct_hash_set")]
HashSet {
    hashes: std::collections::HashSet<u64>,
},

// AFTER (option A — minimal: identity hasher)
#[serde(rename = "v0_count_distinct_hash_set")]
HashSet {
    hashes: hashbrown::HashSet<u64, BuildHasherDefault<U64IdentityHasher>>,
},
```

Define a one-screen `U64IdentityHasher` (returns the input u64 directly via `Hasher::finish`). Alternative (option B): use `BuildHasherDefault<FxHasher>` — FxHasher is already a fast hash mixer.

**OR (option C, even better):** **Eliminate the HashSet mode entirely**. Use a sorted `Vec<u64>` with `binary_search` (the same data structure as `ExactArray`) up to a higher threshold (~512), then promote directly to HLL. That removes the per-insert HashMap probe + hasher call entirely — every insert is a binary_search + insert (cache-friendly, no allocations after Vec::with_capacity). This is what the existing ExactArray code already does at line 58. Just lift the threshold.

**Risk + side effects:**
- Snapshot-stability: changing the field type breaks bincode round-trip on existing snapshots. Need a migration path or accept a snapshot break (acceptable pre-Phase 13 ship).
- Bit-identical: NO (mode promotion thresholds may shift slightly — different intermediate state representation, but same query output).
- Memory: option C uses less memory (Vec<u64> is denser than HashSet<u64>); options A/B same.
- Correctness: option A and B preserve exact set semantics. Option C does too — `binary_search + insert` keeps the Vec sorted, queries via `len()`.

**Effort:** **2-3 hours** (write hasher impl, update CountDistinct, run hash test suite, run criterion bench, run live trace).

**ROI ranking:** **#1 — highest leverage, lowest risk.** This single change saves more ns/event than 19.3-A and 19.3-B combined and keeps the architecture cleaner.

### #2 — Widen `ExtractedFields` SmallVec inline cap from 8 to 16

**Hypothesis:** TxnByUser cluster references **10 distinct fields** across its features. SmallVec[8]+1 → spill to heap on every Txn event. Per-event ExtractedFields build allocates ~10×16 bytes = 160 bytes from the global allocator at agg_apply.rs:205.

**Predicted lift:** **~530 ns/event** (the visible RawVec/allocator traffic). Small but free.

**Specific code-level change:**

`crates/beava-core/src/agg_op.rs:232`:

```rust
// BEFORE
pub type ExtractedFields<'a> = smallvec::SmallVec<[Option<&'a crate::row::Value>; 8]>;

// AFTER
pub type ExtractedFields<'a> = smallvec::SmallVec<[Option<&'a crate::row::Value>; 16]>;
```

**Risk + side effects:**
- Stack frame footprint: each `ExtractedFields` grows from `8 * 16 = 128 B` to `16 * 16 = 256 B` on the stack — the apply loop holds one per-desc, so ~5 descs × 256 B = 1.3 KB extra stack use. Negligible vs the main thread's 8 MB stack.
- Cache footprint: 256 B is still 4 cache lines; one prefetch covers it.
- Correctness: zero risk — pure capacity widening.
- ABI: type-alias change; recompile required but no API change.

**Effort:** **30 minutes** (one-line change + verify criterion + verify live trace).

**ROI ranking:** **#2 — best effort/lift ratio.** Free perf, near-zero risk.

### #3 — Resolve `lat_idx`/`lon_idx` for fraud-team geo features (or fallback to skip the row.get scan more aggressively)

**Hypothesis:** `agg_geo::read_lat_lon` at agg_geo.rs:24-29 (the row.get-based fallback path) is hit 157 samples = ~520 ns/event = 3.97% inclusive. This means Plan 19.2-06's `lat_idx`-based fast path is **NOT being used** for fraud-team's 4 geo features. Either lat_idx is unresolved at register-time, or the registry isn't populating field_idx for `lat`/`lon` field names.

**Predicted lift:** **~350 ns/event** (the pure row.get scan cost; `read_lat_lon_from_extracted` is ~10 ns/call × 4 = 40 ns/event, leaving ~480 ns to remove).

**Specific code-level change:**

### #4 — TopK Exact-mode: replace BTreeMap with a flat sorted Vec + linear-search OR pre-allocate the BTreeMap spine

**Hypothesis:** TopK's Exact mode at `top_k.rs:64` does `*counts.entry(value).or_insert(0) += 1`. With 222 leaf samples + 133 memcmp leaf samples = ~1,180 ns/event of TopK-Exact cost. The BTreeMap allocator + node split + memcmp during traversal are the cost.

**Predicted lift:** **~500 ns/event** if the Exact-mode threshold is small enough (≤16) to make a flat scan faster than a BTree traversal.

**Specific code-level change:**

`crates/beava-core/src/sketches/top_k.rs:11-50`:

```rust
// BEFORE
TopKState::Exact {
    counts: BTreeMap<TopKValue, u64>,
    k, threshold, hybrid_width, hybrid_depth,
}

// AFTER
TopKState::Exact {
    // Flat Vec, linear-search for entry. Bounded by threshold (typ 16-64).
    counts: Vec<(TopKValue, u64)>,
    ...
}
```

Then `insert` does:
```rust
match counts.iter_mut().position(|(k, _)| k == &value) {
    Some(i) => counts[i].1 += 1,
    None => {
        counts.push((value, 1));
        // promote if over threshold
    }
}
```

**Risk + side effects:**
- Snapshot-stability: changes serialized representation. Bincode round-trip break (same caveat as #1).
- Bit-identical: query output identical; intermediate state shape changes.
- Performance regression risk if `threshold` is large (>64): linear search degrades. Use a tunable threshold; default 32 works in fraud-team.

**Effort:** **3-4 hours** (rewrite Exact mode, run topk test suite, criterion + live trace).

**ROI ranking:** **#4 — moderate lift, moderate complexity.** Lower priority than #1-3.

### #5 — EventTypeMix: collapse the double `contains_key`/`entry` into a single `entry`

**Hypothesis:** `agg_buffer.rs:421-428` does `if !allow_set.contains(...)` (HashSet probe) then `if !self.counts.contains_key(...)` (HashMap probe) then `self.counts.entry(cat_string).or_insert(0)` (HashMap probe AGAIN, with ownership transfer). Three hashmap/hashset probes per call when one would do.

**Predicted lift:** **~150 ns/event** (the redundant probe + the `Cow::into_owned` clone).

**Specific code-level change:**

`crates/beava-core/src/agg_buffer.rs:421-428`:

```rust
// BEFORE
if let Some(allow_set) = &self.allowed_set {
    if !allow_set.contains(cat_cow.as_ref()) { return; }
} else if !self.counts.contains_key(cat_cow.as_ref())
    && self.counts.len() >= self.max_categories {
    return;
}
let cat_string = cat_cow.into_owned();
let entry = self.counts.entry(cat_string).or_insert(0);
*entry = entry.saturating_add(1);

// AFTER
if let Some(allow_set) = &self.allowed_set {
    if !allow_set.contains(cat_cow.as_ref()) { return; }
}
match self.counts.get_mut(cat_cow.as_ref()) {
    Some(c) => { *c = c.saturating_add(1); return; }
    None if self.allowed_set.is_none() && self.counts.len() >= self.max_categories => return,
    None => {
        self.counts.insert(cat_cow.into_owned(), 1);
    }
}
```

**Risk + side effects:**
- Logic equivalence: yes (a `get_mut` + `insert` covers same cases as `contains_key` + `entry`).
- Allocation: one fewer `String::clone` per call (Cow::into_owned only when actually inserting).

**Effort:** **2 hours** (test + measure).

**ROI ranking:** **#5 — small lift, contained.** Could fold into 19.3-D as a nicety.

## 9. Honest verdict on EPS ceiling

**Per-event apply-thread CPU minimum (theoretical):**

If we removed ALL of the following (pure thought experiment, not realistic):
- All hot-loop allocator traffic (~700 ns/event)
- All redundant SipHash work (~320 ns/event)
- All double hashmap probes (~150 ns/event)
- All row.get scans on geo (~350 ns/event)
- All scaffolding overhead (~700 ns/event)

= 2,220 ns/event removed → 13,180 - 2,220 = **10,960 ns/event minimum** with current architecture.

EPS ceiling at that minimum = `1e9 / 10,960 = ~91,000 EPS`. That's the **realistic optimization ceiling** for fraud-team K=10k zipfian on the current architecture **without** WAL group-commit, batched apply, or apply-loop SIMD.

To go beyond ~100k EPS sustainable, the next-largest lever is **reducing the per-event WAL roundtrip cost** (Phase 19.4 idea) or **batched apply across multiple events sharing a cluster** — neither is in scope for Phase 19.3.

**Realistic targets after Phase 19.3:**

## 10. Decision support

### (a) Are 19.3-B + 19.3-C still good ideas?

**19.3-C (ExtractedFields hoist):** **YES, still highest-leverage of the original three.** The flamegraph confirms ~387 ns/event in the per-desc rebuild path (agg_apply.rs:205 self-time) and ~530 ns of allocator traffic, totaling ~900 ns/event addressable by the hoist. Cost-model's prediction of -1,800 ns/event was somewhat optimistic; flamegraph suggests **~900-1,200 ns/event is more realistic**, but still solid lift.

**19.3-B (Tier-1 specialize):** **MARGINAL given new evidence.** The flamegraph shows the dispatcher (`update_with_extracted_no_where`) takes 12.6% self-time = 1,659 ns/event total. Specializing 6 of 33 arms saves at most a fraction of that. The cost-model's -1,394 ns/event estimate is plausible but **my proposals #1 and #4 are larger lifts at lower complexity** — recommend doing them FIRST and re-evaluating whether 19.3-B is still needed afterward.

### (b) Is there a NEW higher-leverage optimization?

**YES — proposal #1 (CountDistinct std::HashSet → identity hasher or Vec) is the single biggest win available, and was completely missed by the cost-model investigation** because it doesn't show up in BEAVA_TRACE_AGG_TIMING (which measures per-AggKind ns/call, not where INSIDE the kind time goes). Only a flamegraph reveals it.

This is exactly the case the `feedback_cost_model_from_flamegraph` memory was designed for: the trace-counter cost model identified CountDistinct as the #1 driver but couldn't tell us that 56% of CountDistinct cost is a fixable design bug.

### (c) Is there a fundamental cost-model issue that needs a different phase?

**Partially.** The HLL approach is fine; the choice to wrap u64 hashes in `std::HashSet<u64>` for the intermediate mode is a v0 implementation shortcut that's ripe for a focused fix. This isn't "different phase" territory — it's a 3-hour edit. After that fix lands, the per-AggKind cost ranking changes substantially: CountDistinct drops from #1 to maybe #4-5, and TopK + Percentile sketches become the dominant remaining cost (which IS structural and would need different sketches to dent further, but that's beyond v0 scope).
