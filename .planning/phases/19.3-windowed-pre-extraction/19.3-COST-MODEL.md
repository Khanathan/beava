# Phase 19.3 — Cost-Model Refinement (post-19.3-A)

## TL;DR

Three structural mistakes in `19.2-INVESTIGATION.md` made the predicted lifts for sub-goals 19.3-A/B/C **2.5-3× too optimistic**:

1. **Wrong windowed call count.** Investigation §1 said "60/88 windowed". Actual fraud-team config has **43/88 windowed** features per event (verified by parsing `fraud-team.json`). All "60 × X ns saved" arithmetic in §4 was off by a 1.4× factor at the source.
2. **Per-call cost in real workload is ~30-50 ns lower than investigation conjectured.** Investigation derived `~50 ns row.get + 5 ns where re-eval + 10 ns dispatch ≈ 65 ns/call savings` from microbench arithmetic. Post-19.3-A measurement on the live workload shows the windowed fast path is **30 ns/feature in criterion warm-key, with no measurable per-call ns/call delta vs Phase 19.2 in the live trace** (per-AggKind ns/call actually went UP a few ns across the board; the savings live entirely in scaffolding/instrumentation).
3. **The criterion bench's `14_aggs_windowed` bench is structurally insensitive to dispatch tax**, because all 14 features fit in one warm bucket and the inner state's `update_pre` already short-circuited row.get for many Tier-1 kinds. The criterion baseline of 463 ns was already paying ~33 ns/feature, not the 350+ ns/feature investigation §2 conjectured for windowed inner-state cost.

The post-19.3-A binary measures **+4.4% EPS** (73,743 vs 70,639) on fraud-team K=10k zipfian — within ±5% measurement noise of "no real improvement at all". The architectural change is correct (D-01..D-04 stand) but the **predicted lift was wrong, and the same prediction methodology applied to 19.3-B and 19.3-C will produce predictions that are similarly 2-3× too optimistic.**

**Recommended phase target revision:** drop the 100k EPS PASS gate to **85k EPS** (PASS-WITH-DEFICIT acceptable). Realistic post-19.3 ceiling with all three sub-goals landed: **80-90k EPS**.

## 4. NEW cost model — sub-goals 19.3-B and 19.3-C

The numbers below come from **measured per-AggKind ns/call in the post-19.3-A live trace** + **measured call frequency from BEAVA_TRACE_AGG_TIMING** + **honest accounting of per-call overhead the optimization itself adds**. No conjectural per-call ns × call-count arithmetic on top of a microbench number.

### Sub-goal 19.3-B (Tier-1 specialize Count/Sum/Min/Max/Avg/Variance)

D-01 expanded the original Count+Sum scope to all 6 Tier-1 windowed kinds. Per-row table:

| AggKind | Site (file:line) | calls/evt (windowed only) | Measured ns/call (post-19.3-A windowed avg)¹ | Estimated ns/call after specialization² | Optimization overhead³ | Net Δ ns/event |
|---|---|---:|---:|---:|---:|---:|
| Count    | `agg_windowed.rs:160-211` (windowed `update`) | 10 | ~190 | ~80 | +5 (new match arm) | -1,050 |
| Sum      | same | 2 | ~210 | ~100 | +5 | -210 |
| Min      | same | 1 | ~115 | ~80 | +5 | -30 |
| Max      | same | 0 | ~46 | n/a | n/a | 0 (no windowed Max in fraud-team) |
| Avg      | same | 1 | ~184 | ~100 | +5 | -79 |
| Variance | same | 1 | ~145 | ~110 (Welford has irreducible cost) | +10 (bit-identical req'd) | -25 |
| **Sum-of-rows** | | **15 windowed Tier-1 calls/event** | | | | **-1,394 ns/event** |

¹ Measured ns/call on the **non-windowed** version of each kind in the trace as a proxy for "what the inner state cost is on its own". Windowed contribution is the tax above this base.

² Estimated by removing the `WindowedOp::update_at → update_with_extracted_no_where → match arm → state.update_pre` dispatch chain (~5-level call) and replacing with an inline state mutation in the bucket SmallVec. Specialized cost ≈ position_for_epoch (10 ns) + inline state op (5-30 ns) + bucket push amortized (1 ns/call).

³ Each new match arm in the windowed entry point costs a few ns of branch-predictor pressure + code-cache footprint. For 6 specialized kinds, the dispatch table grows from "1 generic arm" to "7 arms (6 specialized + 1 fallback)". Mostly free on hot path due to BTB caching, but accounted as ~5 ns to be honest.

**Net 19.3-B prediction: -1,394 ns/event saved** (was investigation's "-2,000 ns/event saved" with all 6 — ~1.4× over).

**Key pinpoint of why this is smaller than predicted:** the per-call savings is ~50-110 ns, NOT the 80-150 ns assumed in CONTEXT.md ("Count cost drops from ~180 ns/call → ≤30 ns/call"). The "≤30 ns/call" target is unrealistic — bucket lookup alone (`position_for_epoch`) costs 10-15 ns; even an inlined `n += 1` lands at 50-80 ns total because of the SmallVec indirection + Box deref. Plan 19.3-03's acceptance criterion of "≤30 ns/call" should be **revised to ≤80 ns/call**.

## Appendix B — Trace data parsing reproducibility

Single-pass parser used for §2 numbers:

```python
import re
from collections import defaultdict
totals = defaultdict(lambda: [0, 0])
sums = defaultdict(int); n = 0
with open('/tmp/19.3-02-trace.txt') as f:
    for line in f:
        if not line.startswith('TRACE_AGG ns:'): continue
        n += 1
        for k in ('descs','feat_updates','registry_call','entity_key','table_lookup','entity_row_init','features','TOTAL'):
            m = re.search(fr'\b{k}=(\d+)', line)
            if m: sums[k] += int(m.group(1))
        idx = line.find('per_kind=')
        if idx == -1: continue
        for entry in line[idx+9:].strip().split(','):
```
