## Hypotheses for the gap

1. **Sketch-update cost dominates dispatch tax on the synthetic 14-feature
   criterion bench.** Plan 19.3-01's deviation already documented this:
   the synthetic windowed bench was only 1.46× the non-windowed sibling
   (vs the predicted 3×), because percentile/topk/entropy/countdistinct
   sketches dominate the per-event budget. Eliminating the row.get scan
   (~50 ns) and per-bucket where-eval (~5 ns) saves ~55 ns × 14 features
   = ~770 ns per event budget — versus the criterion's 463 ns total. That
   already exceeds the criterion's total cost, which means the prediction
   was internally inconsistent with the synthetic bench's structure.

2. **fraud-team.json's actual windowed-update count per event was
   probably overestimated.** The 19.2-INVESTIGATION §4 cost model assumed
   60 windowed updates × 65 ns/call = 3,900 ns/event. If many events hit
   lazy-bucket-allocate paths, or if the windowed inner ops were already
   fast (Count via update_pre is ~10 ns), the achievable lift is
   bounded below the prediction. Live trace shows -1,526 ns/event — about
   40% of predicted, suggesting maybe 24-30 effective windowed-tax events
   per event rather than 60.

3. **WindowedOp::update_at still has overhead the prediction missed.**
   The new method allocates an empty `Row::new()` (stack, but still
   ~20 ns of init) and computes `pre_val` from `extracted[field_idx]` per
   call. The bucket dispatch then re-enters `update_with_extracted_no_where`
   with a 9-arg signature. Some of the predicted savings (eliminating
   row.get) are offset by this added per-bucket dispatch + indexing cost.
   Investigation should profile to see whether the bucket-dispatch indirection
   itself is now the dominant cost.

4. **Phase 19.2 PASS-WITH-DEFICIT context.** Phase 19.2's investigation
   already noted (footnote, §1) a 2,526 ns scaffolding gap between
   per-AggKind sum and outer features-stage. The 19.3-A prediction did
   not account for this scaffolding still being present after the
   refactor; if the per-AggKind cost moves but the scaffolding stays,
   end-to-end agg-stage drop is bounded by per-AggKind drop only.

## Next-step direction (HALT — Plan 19.3-03 BLOCKED)

Per CONTEXT D-02: "if measured lift undershoots predicted by >25%, planning
re-evaluates before proceeding to the next sub-goal." All three thresholds
trip by >25%. The proper next step is NOT Plan 19.3-03 (Tier-1 specialization)
but a re-investigation:

1. **Re-trace fraud-team.json with per-call windowed counters** to verify
   how many `update_at` calls actually fire per event (vs the predicted
   60). If the count is materially lower, update the cost model.

2. **Profile `WindowedOp::update_at` directly** with a flamegraph or
   `perf` (Linux) / `samply` (macOS) to identify where the new method's
   ~420 ns/call goes. Likely candidates: `position_for_epoch` linear scan,
   `Row::new()` stack allocation, the `update_with_extracted_no_where`
   re-dispatch.

3. **Reconsider whether Tier-1 specialization (Plan 19.3-03 / 19.3-B)
   would ALSO underdeliver.** The prediction model that yielded the
   19.3-A floor assumed dispatch tax was the dominant cost. If 19.3-A
   shows dispatch tax is NOT dominant, 19.3-B's predicted lift (~2,000 ns)
   is also probably overestimated. Plan 19.3-B's measurement gate would
   trip the same way.

4. **Consider whether 19.3-C (event-level ExtractedFields hoist) should
   land first or in parallel** — if the per-desc rebuild at agg_apply.rs:201-205
   is a bigger cost than the per-bucket dispatch, doing 19.3-C first may
   give a cleaner measurement floor for 19.3-A/B re-evaluation.

5. **The end-to-end ship gate (Phase 13: ≥ 3M EPS/core) is far above
   even the predicted ~95k EPS.** Phase 19.3 was sized as ~25% improvement
   on fraud-team's 70k EPS toward an unstated intermediate goal. Whether
   the achieved ~4% is acceptable or whether a different leverage point
   (WAL group-commit, Phase 19.4) is needed should be discussed with the
   user before continuing.
