<objective>
Wire the Plan 05-01/02/03 aggregation machinery into the `POST /register` + TCP `op=register` pipeline. Add Rule 11 (analogous to Phase 4's Rule 10 for op-chain expression validation) that:

1. Compiles every `OpNode::GroupBy { keys, agg }` in a derivation's `ops` into an `AggregationDescriptor`
2. Validates:
   - Every `group_by` key exists in upstream schema (error: `unknown_field`)
   - Every operator's `field` (for sum/avg/min/max/variance/stddev) exists in upstream schema
   - Every operator's `where` predicate's `referenced_fields()` are all in upstream schema
   - `window=` duration string parses (regex `\d+(ms|s|m|h|d)` or literal `forever`)
   - Aggregation-on-Table → reject with `aggregation_on_table_not_supported` (SDK-AGG-05)
3. Caches compiled `Arc<AggregationDescriptor>` in `RegistryInner.compiled_aggregations` map
4. Surfaces errors over HTTP (400 with `{error: {code, path, reason}}`) and TCP (`OP_ERROR_RESPONSE`) with matching code strings

Per D-07 + Phase 4 wire shape. `AggSpec { op, params }` already exists in `OpNode::GroupBy` from Phase 2 (`op_node.rs`); Plan 05-04 decodes `params` into `AggOpDescriptor` with `where_expr` + `window_ms` + `field`.

Purpose: complete the registration side of Phase 5. Apply-loop hook (Plan 05-05) and HTTP query endpoint (Plan 05-06) both read `compiled_aggregations`. Python SDK (Plan 05-07) produces the REGISTER JSON this plan parses.
</objective>
