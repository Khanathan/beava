# Phase 12.7 — v0 Table Strip

## Goal

Strip `@bv.table` and the entire table surface from v0. Beava v0 commits to **events-only**: `@bv.event` + the 54-op catalogue + `/push` + `/get` + `/register` + WAL/snapshot durability + Python SDK. Tables return in v0.1+ alongside joins / aggregation if justified by demand.

## Why now

Phase 12.6 (closed 2026-04-30 PASS-WITH-WARN) deleted joins, unions, event-time, watermarks, PIT — and wired up mio `/upsert /delete /retract` endpoints in its Wave-1.5 gap closure (Plan 14). But:

1. With joins gone, tables have no consumer pattern. The only remaining use case for `@bv.table` in v0 was as a join target.
2. With aggregation deferred (Phase 17 archived-indefinitely), tables-as-aggregation-output also has no v0 use case.
3. Plan 14's mio table handlers were added to satisfy existing tests — but those tests describe a feature that is now out of v0 scope.
4. Net: ~14 table-related tests + Plan 14's mio handlers + `temporal_http.rs` helpers + `TemporalStore` + `app.retract(event_id)` + `@bv.table` Python decorator are all dead surface for v0.

This phase walks back the table surface to give v0 a clean, narrow shape. **Approximately ~10 plans** estimated.

## Architectural commitment

Per `project_v0_events_only_scope` (memory): v0 ships as pure events. Tables/joins/aggregation return together in v0.1+ if/when there's user demand. The simplification is permanent for v0; it does not preclude v0.1+ revival.

## NOT in scope (keep)

- `@bv.event` and the 54-op catalogue (counters, sketches, windowed, recency, geo, decay, velocity)
- `/push`, `/get`, `/register` endpoints
- WAL + snapshot durability (the WAL still records events; tables-related records get a clean migration path or version bump)
- Mio data plane + tokio admin sidecar
- Python SDK event surface

## Open product decisions (resolve in `/gsd-discuss-phase 12.7`)

1. **WAL/snapshot schema bump** — does removing the table record types require a v2→v3 schema bump, or is v2 forward-compatible if we just stop emitting table records? (Probably need v3 since old WALs may have table records that the new code can't deserialize.)
2. **Phase 11.5 artifact handling** — keep its SUMMARY.md and add a banner, or rename to `_archived-...` and replace with a stub? (Suggest: keep + banner — preserves traceability.)
3. **Test deletion vs migration** — for the 14+ tests, all delete (table feature gone) or keep the no-tokio architectural test that happens to test table surface? (Mostly delete; phase18_07_no_tokio_dataplane keeps its non-table assertions if any.)
4. **Migration message in WAL/snapshot loader** — old WAL with table records: error with structured message pointing to the v0 → v0.1+ migration story, or silently skip records?
5. **TestServer `tcp_max_frame_bytes` builder** — Plan 15 added env-var coupling for table tests; does it still need the builder if no table tests remain? (Likely yes — generic TCP test infra.)

## Architectural anti-patterns to reject

- Do NOT preserve `TemporalStore` as a "deprecated stub" (HARD remove)
- Do NOT keep `app.retract` as a no-op (raise on use; it's a removed verb)
- Do NOT add a `compat_v0_tables` feature flag (clean break)
- Do NOT silently parse-and-strip `@bv.table` — register-time hard error with structured code `feature_removed_no_tables_v0`
