## Implementation Decisions

### WAL / snapshot schema
- **D-01 — Reset FORMAT_VERSION to 1 across all 3 schemas; no bump-and-tail.**
  - `record.rs::FORMAT_VERSION: u32 = 2 → 1`.
  - `snapshot_body.rs::SNAPSHOT_BODY_FORMAT_VERSION: u16 = 2 → 1`.
  - Snapshot header: `SNAPSHOT_FORMAT_VERSION: u16 = 2 → 1`.
  - **Rationale:** v0 isn't released. Nothing to be backward-compatible with. Pre-12.7 dev WALs (which carried `v=2` after 12.6 Plan 06) fail at the version-byte check with `SchemaVersionMismatch`/`UnsupportedVersion` (existing error paths). v0 launches at `version=1` across the wire/WAL/snapshot. **More aggressive than 12.6's bump pattern — it RESETS rather than bumps.** No migration shim, no compat path, no parse-and-strip. Operators clear `.beava/wal` + `.beava/snapshots` before booting the new binary.
  - **RecordType variants:** `TableUpsert (0x03)` / `TableDelete (0x04)` / `Retract (0x05)` deleted from the enum. `from_u8(0x03|0x04|0x05)` returns `UnknownRecordType` (existing generic variant; not a new table-specific code).

### Strictness boundaries — error framing
- **D-02 — Hard errors at all layers, framed as "not supported in v0", NOT "feature removed".**
  - **Rationale:** v0 is the FIRST public release. Users never knew tables existed in v0. Retrospective `feature_removed_no_*_v0` framing (12.6's pattern) confuses fresh users; forward-looking "not supported" framing is correct for v0.
  - **Python SDK:**
    - `@bv.table` → `AttributeError` (or `TypeError` if accessed but called) with message `"@bv.table is not supported in v0; v0 ships events-only via @bv.event"`. Decorator-time error per the 12.6 Plan 08 pattern (raise at decoration, not push time).
    - `app.retract(event_id)` → `AttributeError: 'App' object has no attribute 'retract'` (natural Python error from method removal). No no-op stub; no `retract` attribute on `App`.
    - `app.push_table` / `app.delete_table` → same — methods removed; natural `AttributeError`.
  - **Server register-time:** Pre-deserialize JSON-prelude shim in `register_validate.rs` rejects `{"kind": "table", ...}` register payloads with structured error code `unsupported_node_kind` (not `feature_removed_no_tables_v0`) and message listing supported kinds (`["event"]`). Same shim pattern as 12.6 Plan 04 (JSON-prelude before strict serde) but with v0-correct error code naming.
  - **Mio HTTP routes:** Deleted `/upsert/{name}` / `/delete` / `/retract` / `/table/{name}` routes return plain **404** (mio's default for unknown routes). No special deny-handler; no `410 Gone` body. Stale curl-callers see "not found" — same as any unknown route.
  - **WAL recovery:** `RecordType::from_u8(0x03|0x04|0x05)` → `UnknownRecordType` (existing generic error). No table-flavored error code in persistence.

  - Both tests run on every `cargo test --workspace` per 12.6 Plan 10's CI-gate convention. Locks `project_v0_events_only_scope` at the same architectural level as `project_phase18_no_dual_runtime`.
  - **CLAUDE.md update:** Add `§Conventions § Events-Only Invariant (locked Phase 12.7)` block alongside the existing `§ mio-only Hot-Path Invariant (locked Phase 12.6)` block. Document that reviving `@bv.table` requires explicit user override + new ADR overturning `project_v0_events_only_scope`.

## Specific Ideas

- **D-01 is more aggressive than 12.6's bump pattern.** User explicit: "Just make it version 1 no need to care backward compat". Reset is the canonical move for an unreleased product; bump is the canonical move for a released product. v0 falls in the former bucket.
- **D-02 framing — "not supported in v0", NOT "feature removed".** User explicit: "error but no mention of remove". The error messages, structured codes, and docstrings should read as if tables simply aren't a v0 capability — not as if they were taken away. This applies to **all** error paths: Python `AttributeError` messages, server `unsupported_node_kind` code (NOT `feature_removed_no_tables_v0`), test assertion messages, and the architectural test's failure messages.
- **D-03 reuses 12.6 Plan 10's exact two-file architectural-test scaffold.** No new pattern; same allowlist + sanity-test convention. Cheap to land because the prior pattern is mature.
- **D-04's positive requirement (`V0-EVENTS-ONLY-01`)** anchors the events-only commitment as a testable invariant in REQUIREMENTS.md. The D-03 architectural test satisfies it. Forward-compat: when v0.1+ revives tables, they'll need to either retire `V0-EVENTS-ONLY-01` or add a `V0.1-TBL-*` family that overrides it — explicit ADR territory.
- **Phase 11.5 retro-descope banner is LOAD-BEARING for the audit trail.** Without it, future readers landing on `11.5-SUMMARY.md` would think temporal tables shipped successfully — they did, then got walked back. The banner is mandatory; Phase 11.5 dir stays in place per `feedback_logistics_autonomy` (no rename to `_archived-...`).
- **Estimated scope:** ~6–10 plans across 3–4 waves. Smaller than 12.6 (15 plans) because 12.7 is mostly deletion + REQUIREMENTS.md sweep + 11.5 banner. No new architecture beyond the tiny architectural-test pair.
