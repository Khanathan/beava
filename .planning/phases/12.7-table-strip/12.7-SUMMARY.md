## Executive summary

Phase 12.7 is the architectural moment where Beava v0's surface narrows to **events-only** per `project_v0_events_only_scope` (locked 2026-04-30). The entire table / temporal / retraction surface is **deleted** — `@bv.table` Python decorator + `POST /upsert/{table_name}` + `POST /delete` + `POST /retract` + `GET /table/{name}` mio handlers + `temporal_http.rs` (~756 LOC) + `temporal.rs` (~394 LOC; `TemporalStore` / `MvccVersion` / retention sweep) + `app.retract` SDK verb + 12 table-related tests. Walks back Phase 11.5 (temporal MVCC + retract primitive) and Phase 12.6 Plan 14's mio table handlers.

The **persistence schema RESET** (D-01) is more aggressive than 12.6's bump pattern — it RESETS `FORMAT_VERSION 2→1` (rather than bumping 2→3) across `record.rs` + `snapshot_body.rs` + `snapshot_header.rs`. v0 isn't released; nothing to be backward-compatible with. Pre-12.7 dev WALs (carrying `v=2` after 12.6 Plan 06) fail at the existing version-byte check via `SchemaVersionMismatch` / `UnsupportedVersion` error paths. v0 launches at version=1 across the wire/WAL/snapshot. Operators clear `.beava/wal` + `.beava/snapshots` before booting the new binary.

The **error framing** (D-02) is **forward-looking** ("not supported in v0"), NOT retrospective ("feature removed"). v0 is the FIRST public release; users never knew tables existed in v0. Server register-time uses structured error code `unsupported_node_kind` (not `feature_removed_no_tables_v0`); deleted mio HTTP routes return plain 404 (mio's default for unknown routes — no special deny-handler); Python `bv.table` raises `AttributeError` naturally from import-side absence; `App.upsert(...)` / `App.delete(...)` / `App.retract(...)` raise `AttributeError` naturally from method removal; WAL recovery returns the existing generic `PersistError::UnknownRecordType` for deleted discriminants. `GroupBy.agg()` is a method-shape stub that raises `RuntimeError("...not supported in v0; ...events-only...")` at call time — keeps SDK-surface stable for legacy import paths but always-error semantics on call.

The **architectural test pair** (D-03) — `phase12_7_no_table_surface.rs` (symbol grep, 286 LOC, 3 tests) + `phase12_7_legacy_table_handlers_killed.rs` (file/symbol absence, 338 LOC, 6 tests) — locks the events-only commitment at the same architectural level as `project_phase18_no_dual_runtime` (12.6 Plan 10's `phase12_6_mio_only_dataplane.rs`). ## Lessons learned

These patterns emerged from Phase 12.7's pure-deletion sweep, worth saving as memories for future architectural-deletion phases:

### 1. Architectural-test inversion — RED at write-time via NEGATIVE assertions

**Pattern:** When a phase's goal is to DELETE a surface, write the architectural test FIRST in Wave 1 with NEGATIVE assertions (symbol absent / file absent). The test starts RED; subsequent waves' deletions turn it GREEN incrementally. The test failure list IS the deletion roadmap.

**Phase 12.7 application:** Plan 02 wrote `phase12_7_no_table_surface.rs` (symbol grep across 5 walk dirs × 18 forbidden patterns) + `phase12_7_legacy_table_handlers_killed.rs` (6 file/symbol-absence tests with sanity-positive surviving-variant assertions). At Plan 02's commit, the symbol-grep failure list reported 32 (file, pattern) pairs across 13 production files; the per-file tests reported 6 RED. Plans 03/04/05/06 each turned specific tests GREEN; the 32 → 0 count served as a literal progress map.

**Different from forward-state TDD** (where the impl makes the test GREEN). Here, the deletion makes the test GREEN. Reusable for any future architectural commitment that expresses itself as a deletion (e.g., when stripping any feature for a future v-release). Pair with `#[ignore]`-management (Lesson 2) to keep the workspace green during the RED interlude.

### 2. RED-state architectural test management via `#[ignore]` + descriptive ignore-reason

**Pattern:** When an architectural test is RED at end of Wave 1 because subsequent plans land its dependencies, mark it `#[ignore]` with a string naming the unblocking plan (`Wave 1 RED state; turns GREEN after Plan 12.7-XX`). Closure plan (last wave) removes the `#[ignore]` markers as the final tests-pass moment.

**Phase 12.7 application:** Plan 02 marked all 9 architectural tests `#[ignore]` with comments naming Plan 10 closure as the unblock point. Manual `cargo test -- --ignored` could verify the RED state at any point during Waves 2-3. Plan 10's `5645ead` commit fixed misattributed comments (originally "Plan 12.7-09 closure" — Plan 09 was perf-only) and dropped the `#[ignore]` annotations. Tests now run BY DEFAULT.

**Comment-attribution drift caveat:** the original `#[ignore]` comments said "Plan 12.7-09 (closure)" — but Plan 09 turned out to be perf-only (microbench + throughput rebaseline), not the closure plan. Plan 10 became the actual closure plan and owns the `#[ignore]` removal. Documented openly in 12.7-09 SUMMARY's "Tangential observations" so the audit trail captures the correction.

### 3. Pure-deletion phase produces measurable performance lift

**Pattern:** When a phase consists entirely of deletions (no new code on hot path), expect neutral-to-slight-lift on throughput and significant-lift on microbench. Architectural rationale: smaller dispatch tables → tighter icache + better branch prediction + tighter LLVM jump tables. Document the expected magnitude up front so the perf-rebaseline verdict isn't surprising.

**Phase 12.7 application:** Plan 09 measured -25 to -30% across 3 microbench cells (apply hot path) and +7.3% on the small/tcp throughput regression-gate cell. Architectural rationale (per `12.7-09-SUMMARY.md`): the `dispatch_one` match table shrunk by 4 arms (HttpUpsert/HttpDelete/HttpRetract/HttpTableGet); the `WireRequest` enum's discriminant range shrunk; the `RecordType::from_u8` lookup shrunk; `temporal_http.rs` and `temporal.rs` are no longer compiled. Net effect: smaller icache footprint + better branch prediction + tighter LLVM-emitted jump tables. NO offsetting headwind (Path X SystemTime::now() from 12.6 was already baked into both 12.6 and 12.7 measurements).

**Microbench / throughput asymmetry:** microbench shows the pure server-side lift (-25 to -30%); throughput shows bench-client-bound numbers (mean +0.9% across 8 cells; the Apple-M4 ceiling is bench-client-bound at ~700-750k EPS regardless of server speed). Both signals point in the same direction (faster); at different scales. Phase 13 ship-gate on Hetzner Linux EPYC-Genoa will see closer-to-microbench EPS lift because the kernel TCP stack on Linux is faster than Darwin's loopback.

### 4. Method-shaped stub vs method DELETION vs method REPLACEMENT

**Pattern:** When deleting a Python SDK method whose import path is referenced in legacy fixtures, choose between:

- **Method DELETION** (e.g., `App.upsert(...)` in Plan 06): natural `AttributeError` on call. Breaks any test that imports + calls the method. Best when callers should fail loudly at the new v0 surface.
- **Method-shaped stub** (e.g., `GroupBy.agg(...)` in Plan 06): retains signature so legacy import paths import cleanly; raises `RuntimeError` at call time with v0 framing. Best when the import-time failure would cascade to fixture-collection breakage and the call-time failure is sufficient diagnostic.
- **Method REPLACEMENT** (NOT used in 12.7): silently changes semantics. Bad — never in v0 because users haven't seen v0 yet.

**Phase 12.7 application:** Plan 06 used DELETION for `App.upsert/delete/retract` (natural AttributeError) and method-shaped STUB for `GroupBy.agg()` (RuntimeError at call time, signature preserved). The choice was driven by import-time fixture preservation: 5+ Python tests import `from beava import bv` + chain `events.group_by().agg(...)`; deleting `agg()` would break import-time. Stub keeps fixtures importable, RuntimeError fires at call.

### 5. Forward-looking error framing for first-release deletions

**Pattern:** When deleting a feature in a project's FIRST public release (v0), use forward-looking framing ("not supported in v0") NOT retrospective ("feature removed / `feature_removed_no_*_v0`"). Users never knew the feature existed; retrospective framing confuses fresh users.

**Phase 12.7 application:** Plan 01 chose `unsupported_node_kind` (not `feature_removed_no_tables_v0`) for the JSON-prelude shim. Plan 06 chose `RuntimeError("not supported in v0")` (not "feature removed"). Plan 03 chose plain 404 (not `410 Gone` with retrospective body). All error paths read as if tables simply aren't a v0 capability — not as if they were taken away.

**Distinct from 12.6's pattern:** 12.6 Plans 04 / 06 / 08 used retrospective framing (`feature_removed_no_joins_v0`, `unknown_field_event_time_v0`, `feature_removed_no_unions_v0`) because joins/unions/event-time WERE features that existed in pre-pivot codebase. v0 had brief exposure to them. v0 tables NEVER had public exposure, so the framing flips.

**Future similar decisions:** when retiring a feature in v0.x or v0.1+, choose retrospective framing if the feature shipped publicly; choose forward-looking framing if it never did.

### 6. Section-ownership contract for shared docs across many waves

**Pattern (from 12.6 Plans 09 + 13 + 12.7 Plans 07 + 08 + 10):** When 3+ plans across different waves edit the same doc, partition the doc by SECTION up front and call out the partition in each plan's frontmatter. Verify the boundary at commit time via `git diff` inspection BEFORE the commit lands.

**Phase 12.7 application:** REQUIREMENTS.md was edited by Plan 07 (Wave 3) and the 11.5-* files were edited by Plan 08 (Wave 3); STATE.md / ROADMAP.md / CORRECTNESS-PATH.md / CLAUDE.md were edited by Plan 10 (Wave 4); perf-baselines.md / throughput-baselines.md were edited by Plan 09 (Wave 4). The contract worked perfectly — no collisions across all 10 plans, no merge conflicts, no overlap. Plan 10's frontmatter explicitly listed which sections it OWNED and which earlier-plan-owned sections it did NOT touch.

### 7. Module-deletion produces natural Python AttributeError (no explicit deny stub needed)

**Pattern:** When deleting a Python source module, the `from .X import Y` re-export in `__init__.py` goes with it; subsequent `bv.X` access raises `AttributeError: module 'beava' has no attribute 'X'` naturally. NO explicit deny stub at the namespace level. Different from 12.6 Plan 08's TypeError-raising decorator-stub pattern (which was for keyword-arg removal, not module removal).

**Phase 12.7 application:** Plan 06 deleted `python/beava/_tables.py` entirely (~502 LOC). The `from ._tables import table` line in `python/beava/__init__.py` went with it; the `"table"` token in `__all__` was removed. `bv.table` access fails naturally. NO explicit deny stub needed at the namespace level (per CONTEXT D-02 §"Integration Points"). Same posture for `App.upsert` / `App.delete` / `App.retract` method removal — natural `AttributeError`, no no-op stubs.

### 8. Foreign-phase retro-descope banner-stamp without dir rename

**Pattern (from 12.6 Plan 09 + 12.7 Plan 08):** When a later phase walks back an earlier phase's landed work, banner-stamp the earlier phase's SUMMARY+VERIFICATION+CONTEXT triple with `> blockquote` + horizontal rule, citing the locked-memory-file architectural commitment. Original content preserved verbatim below. Phase directory NOT renamed (per `feedback_logistics_autonomy`) — banner-stamp preserves traceability without churning paths.

**Phase 12.7 application:** Plan 08 banner-stamped `11.5-SUMMARY.md` + `11.5-VERIFICATION.md` + `11.5-CONTEXT.md` with byte-identical uniform text. Phase 11.5 directory remains at `.planning/phases/11.5-temporal-tables-retraction-primitive/`; the banner explicitly cites `project_v0_events_only_scope` as the architectural commitment that walks back the original work. Audit trail is now: any reader landing on `11.5-SUMMARY.md` sees the banner first, then the original content.

### 9. Plan-step instruction lag — closure-plan task scope discovered via prior commits

**Pattern:** Plan 10's prompt described "remove `#[ignore]` markers" as a primary closure task — but those markers were already removed (alongside the dead `temporal_throughput.rs` binary cleanup) in commit `5645ead` BEFORE this Plan 10 execution started. This is a closure-plan-prompt-vs-actual-state drift; document it openly so the audit trail captures what was already done vs what this Plan 10 execution did (doc-only closure work).

**Phase 12.7 application:** Plan 10 verified the precondition (`#[ignore]` already removed; temporal_throughput.rs already deleted; architectural test pair GREEN by default), noted the drift in this SUMMARY, and proceeded directly to the doc-only closure work (VERIFICATION + SUMMARY + CLAUDE.md + STATE/ROADMAP/CORRECTNESS-PATH).
