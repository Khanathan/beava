## Decisions Made

1. **GroupBy.agg() = method-shaped stub with always-raise body.** Per D-04 + PATTERNS.md §Pattern 9: the SDK still imports cleanly, `Tx.group_by("k")` returns a `GroupBy` stub, but `.agg(...)` raises `RuntimeError` with v0 framing instead of constructing a `TableDerivation`. No `TableDerivation` construction anywhere — the import is gone, the class is gone. This is more invasive than CONTEXT.md initially flagged (CONTEXT only described decorator-level stubs). The `RuntimeError` message contains both required tokens (`"not supported in v0"` + `"events-only"`) per the RED test contract.

2. **wire.rs OP_PUSH_TABLE/OP_DELETE_TABLE deleted, not retained-as-stubs.** The wire surface resets to v0 events-only. Future v0.1+ revival adds NEW constants (or re-introduces these at the same numeric values) only if/when justified by demand + explicit ADR overturning `project_v0_events_only_scope`. The 4 in-module tests previously asserting their reserved status were updated to assert absence (`opcode_name(0x0013) == None`).

3. **Test files outside the plan's frontmatter list were stripped where blocking imports demanded.** Per Rule 3 (auto-fix blocking issues): `test_app.py`, `test_phase3_smoke.py`, `test_validate.py`, `test_package_init.py`, `bench_register_compile.py` all had module-scope `@bv.table(...)` decorators or `from beava._tables import ...` imports that would fail at pytest collection time. Each was surgically stripped; events-only fixtures preserved their non-table assertions.

4. **`test_phase5_smoke.py` was NOT touched.** It tests aggregation features (Phase 5 scope). Per `project_v0_events_only_scope`, aggregation is not supported in v0; the v0 RuntimeError now fires at fixture-build time. These tests **legitimately** fail because they're testing v0.1+ functionality that v0 doesn't ship. Since they were already failing in baseline (server-connection issues), the new failure mode is the equivalent gate (just earlier in the call stack). The architectural-equivalent gate on the server side lives in `phase12_7_unsupported_node_kind.rs` from Plan 03.

5. **`_VALID_BEAVA_KINDS` narrowed `{event, table, derivation}` → `{event, derivation}`.** With `@bv.table` gone, no descriptor can legitimately have `_beava_kind="table"`; if a synthetic one shows up via reflection, the `bad_return_type` rule rejects it.
