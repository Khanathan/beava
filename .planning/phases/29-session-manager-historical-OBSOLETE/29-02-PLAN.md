<objective>
Implement the bootstrap half of the historical replica state machine. Given a
ready `ClientSession` (from 29-01) and a client `ClientStateStore` (from
Phase 28), issue `OP_SNAPSHOT_FETCH`, decode the server's streamed snapshot
entry-by-entry, apply each to the store, and capture the terminal HWM seq.
That HWM is the handoff to the catchup loop (29-03).

Purpose: This is the "hydrate from baseline" step. It replaces what would
otherwise be a cold-start apply_event-from-zero by accepting a server-produced
compacted snapshot, so the client catches up in O(scope size) not O(full log).
It's also the only plan in Phase 29 that touches streaming-decode memory
behavior — the key non-functional correctness property.

Output:
- `src/client/bootstrap.rs` — `run_bootstrap` + `BootstrapError`.
- Integration tests including a heap-bound property test (single-entry peak,
  not N-entry peak).

Assumption to validate at execution time: Phase 27's snapshot wire format —
whatever byte layout 27-01 lands for snapshot entries + terminal frame — is
the source of truth. This plan uses whatever `SnapshotEntry`/entry decoder
Phase 27 exposes. If Phase 27 only exposed an encoder-side and the decoder
isn't factored out, the executor moves the decode into a shared module first,
then wires this bootstrap loop against it. Do not reimplement a second codec.

Assumption to validate at execution time: Phase 28's `ClientStateStore`
exposes `apply_event(&mut self, event: &Event)` (or equivalent). If the exact
signature differs, adapt the caller — do not widen the client StateStore API
for this plan.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server snapshot stream → client decoder | Server is semi-trusted: may emit malformed frames, oversized entries, or truncated terminals. Client must not OOM, panic, or apply garbage to the store. |
| decoded event → ClientStateStore.apply_event | Event has been produced by Phase 27's scope filter — trusted content. But apply_event itself may error (schema mismatch); surface cleanly. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-02-01 | Denial of Service | Unbounded snapshot buffering | mitigate | Streaming decode — one entry in flight; heap-bound test enforces. `recv_frame` inherits max-frame-size from 29-01. |
| T-29-02-02 | Tampering | Malformed SNAPSHOT_TERMINAL payload | mitigate | Strict length check (must be exactly 8 bytes) → Protocol error, no HWM assumed. |
| T-29-02-03 | Information Disclosure | Server error message leaked to logs with sensitive info | accept | Server controls its own error messages; client just forwards. Post-v0 could redact. |
| T-29-02-04 | Elevation of Privilege | Malicious server forcing apply_event with crafted event to trigger store panic | mitigate | apply_event is expected to return Result (Phase 28 contract). run_bootstrap wraps errors, never unwraps. If apply_event panics, that's a Phase 28 bug, not a Phase 29 one — but run_bootstrap should be caller-panic-safe via tokio task boundary (document in SUMMARY). |
| T-29-02-05 | Repudiation | Partial-apply state after mid-stream server error is invisible | mitigate | Log info "bootstrap: applied {n} entries before error" before returning BootstrapError::Server. Caller (29-03 wiring) decides to discard the store. |
</threat_model>
