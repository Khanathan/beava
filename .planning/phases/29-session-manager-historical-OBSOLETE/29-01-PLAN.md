<objective>
Stand up the client-side TCP session: a persistent, opcode-agnostic framed
connection to a Tally server with reconnect-with-handshake semantics. This is
the transport foundation for the Phase 29 bootstrap (29-02) and catchup (29-03)
loops. No replica business logic in this plan — just "a socket that stays up,
speaks the Phase 27 wire format, and re-announces its scope after a blip."

Purpose: Decouple reconnect + framing + scope handshake from the two replica
state-machine steps that follow, so 29-02 and 29-03 can be pure
request/response code against a known-good session handle.

Output:
- `src/client/session.rs` — `ClientSession` with `connect`, `handshake`,
  `send_frame`, `recv_frame`.
- `src/client/reconnect.rs` — pure `ReconnectPolicy` + async
  `run_with_reconnect` helper.
- `src/client/frame.rs` — shared 4-byte-length-prefix frame codec,
  feature-gated for both client and server builds (one implementation).
- `src/client/error.rs` — `SessionError` enum + `OutOfScopeError` re-export.
- Rust unit + integration tests covering reconnect math, jitter distribution,
  scope-handshake byte equality across reconnects, and client-side scope
  validation.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client CLI → Tally server TCP | Client is trusted (same operator), but admin token must transit correctly and server must validate. Untrusted-user boundary is the server's auth gate, not the client's session code. |
| server reply → client decoder | Server is semi-trusted. A buggy or compromised server could emit oversized frames or malformed scope errors; client must fail closed, not panic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-01-01 | Tampering | ClientSession handshake bytes on reconnect | mitigate | Test asserts byte-for-byte equality of handshake bytes across attempts. Scope is held immutable in `ClientSession` between reconnects. |
| T-29-01-02 | Information Disclosure | Token in reconnect logs | mitigate | `SessionError` Debug/Display must never print token. Token field is `Option<String>` held in ClientSession; `Debug` derive excludes it via manual `impl Debug` or `secrecy::Secret` wrapper if already in deps — document choice in code. |
| T-29-01-03 | Denial of Service | Oversized frame from server | mitigate | `read_frame` enforces a max frame size (default 256 MiB, const in `frame.rs`). Exceeding → `SessionError::HandshakeFailed` / connection drop, not OOM. |
| T-29-01-04 | Denial of Service | Infinite reconnect loop | mitigate | B1: max_attempts=5, then `GaveUp` — client exits, does not spin forever. |
| T-29-01-05 | Spoofing | Server impersonation (MITM) | accept | Phase 29 does not add TLS — `TALLY_HOST` is trusted by policy; TLS is deferred to post-v0 ops hardening. Documented in objective. |
| T-29-01-06 | Repudiation | Reconnect attempts untraceable | mitigate | warn-level log per attempt with attempt number; error-level log on GaveUp. No structured JSON per CONTEXT — human-readable only. |
