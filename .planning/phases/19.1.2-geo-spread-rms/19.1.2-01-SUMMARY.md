## Accomplishments

- **GeoSpreadState shape rewritten** (CONTEXT D-01): dropped `samples: Vec<(f64, f64)>` and `max_km: f64`; added `m2_lat: f64` and `m2_lon: f64`. Memory cost dropped from O(n) (16 bytes/sample × millions of samples per entity → up to 16 MB/entity at 1 M events) to O(1) (16 fixed bytes regardless of history depth).

- **query() now returns RMS dispersion** (CONTEXT D-03): `sqrt(var_lat_km² + var_lon_km²)` with local-mean-latitude cos-correction; `Null` for n<2 (variance undefined for <2 samples — aligned with `VarianceState`/`StdDevState` contract).
- **5 new unit tests landed** (4 from plan + 1 strong-RED extra): null-for-n<2, unit-square RMS, permutation invariance, monotone-in-scatter, asymmetric-cluster-NYC-vs-LA-RMS-vs-max divergence.

## Decisions Made

- **Population vs sample variance (m2 / n vs m2 / (n-1)):** chose population (`m2 / n`). Rationale: per CONTEXT D-03 explicit math, and because RMS dispersion as a fraud feature should not jump discontinuously between n=2 and n=3 (Bessel correction multiplies by n/(n-1) which is 2.0 at n=2, 1.5 at n=3, etc — that discontinuity is undesirable for behavioral features). Aligns with `np.std(coords)` default behavior.
- **`Value::Null` returned for n<2 (was Null only for n=0):** consistent with `VarianceState::query_variance` and `query_stddev` (existing pattern in `agg_state.rs::VarianceState`). Rules out a degenerate F64(0.0) result for n=1 that downstream consumers might mistake for "user is centred".
- **Smoke baseline test preserved:** the old `geo_spread_returns_max_distance_from_centroid` test (4 corners at ±0.5° around origin) returns ~78.7 km under both old max-from-mean and new RMS semantics — coincidence of the symmetric input. Renamed to `geo_spread_unit_square_at_origin_smoke` and kept as a baseline; it doesn't distinguish RED from GREEN but does verify both code paths agree on this canonical input.

## Snapshot Format Compatibility

**BREAKING — pre-fix snapshots fail loud on deserialize.**

`GeoSpreadState` lost two fields (`samples: Vec<(f64,f64)>`, `max_km: f64`) and gained two fields (`m2_lat: f64`, `m2_lon: f64`). Serde does not gracefully migrate this — pre-fix bincode/JSON snapshots will fail with `missing field: m2_lat` (correct, loud failure mode).

**Operator workaround if needed:**
1. Stop beava
2. Discard the snapshot file under the configured snapshot dir
3. Restart — beava will replay WAL from start to rebuild agg state from raw events (WAL stores raw events, not agg state, so it is not affected by this change)

**Why this is acceptable now:**
- v0 is not yet shipped publicly per user memory `project_beava_product`
- No production data migration path needed
- Per CONTEXT D-07: "WAL replay is unaffected"; loud failure is the correct mode pre-public-release
- Snapshot version constant: not bumped — `beava-persistence` does not currently expose a version field for individual agg states. This will be addressed if/when a unified agg-state version field is added (out of scope for this hotfix per CONTEXT D-08).

## Polar-Latitude Limitation

The new `query()` uses `km_per_deg_lon = 111.32 * cos(mean_lat_rad)` — the standard equirectangular approximation. At extreme polar latitudes (`|lat| > ~85°`) the cos-correction degenerates and the longitude variance contribution becomes inaccurate (cos approaches 0, but actual longitude separation in km also approaches 0 — the approximation still has the correct sign and magnitude direction, just with degraded precision near the singular pole).

Per CONTEXT D-11: fraud workloads do not routinely involve sub-arctic transactions. Acceptable for v0; documented as a known limitation. Future v0.1 work could swap to geodesic Haversine variance if a sub-arctic use case surfaces.
