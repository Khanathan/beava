<objective>
Fix the pre-existing HTTP parser buffer-cap bug discovered during Plan 19.1-02's smoke run. `parse_http_request` rejects ANY HTTP request larger than 8 KiB because `MAX_HEADER_BYTES` is checked against the whole buffer. fraud-team.json's 15 KiB register POST trips this on the second TCP packet. After fix: 8 KiB cap applies to header bytes only (up to `\r\n\r\n`); body bytes parse cleanly via `Content-Length` up to `MAX_BODY_BYTES = 4 MiB`.

This plan unblocks Phase 19.1 Wave 2 (lazy buckets, WAL bump) and Wave 3 (rebaseline) — both need fraud-team.json to register against the live bench server.
</objective>

<threat_model>
## Phase 19.1.1-01 Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Header-bomb attack via large headers | high | Existing `MAX_HEADER_BYTES = 8 KiB` cap retained; applied to header-bytes only (pre-`\r\n\r\n` region). Any unbounded reader that reads 8 KiB with no `\r\n\r\n` boundary returns `ParseError::TooLarge`. |
| Body-bomb attack via large body | high | `MAX_BODY_BYTES = 4 MiB` cap (existing at line 143) gates `Content-Length`. Already-correct code; this plan makes it reachable. |
| Slowloris-style partial reads | low | Out of scope; not changing the read loop's idle-timeout behavior. Existing connection-level timeouts apply. |
| Bypass of body cap via missing/bogus Content-Length | medium | Same as before — `Content-Length` check is mandatory before body read. No new bypass. |

Net new attack surface: **none**. The fix RESTORES the intended `MAX_BODY_BYTES = 4 MiB` body cap that was unreachable due to a prior bug; header-bomb defense is preserved verbatim.

</threat_model>

<action>
Edit `crates/beava-runtime-core/src/http_listener.rs:69-74`. The current code is:

```rust
let probe = if buf.len() > MAX_HEADER_BYTES {
    return Err(ParseError::TooLarge);
} else {
    buf.as_ref()
};
```

The fix: gate the cap on whether `\r\n\r\n` has been found. Use httparse's `Status::Partial` vs `Status::Complete` (or scan for the boundary directly). Replace with logic equivalent to:

```rust
// Find the header-end boundary. If we have NOT found `\r\n\r\n` AND the buffer is
// already > MAX_HEADER_BYTES, this is a header bomb — reject. If we HAVE found the
// boundary, the body bytes that follow are gated by Content-Length at line 143.
let header_end = buf.windows(4).position(|w| w == b"\r\n\r\n");
let probe = match header_end {
    Some(_) => buf.as_ref(),  // Headers complete; body parsing will use Content-Length cap.
    None => {
        if buf.len() > MAX_HEADER_BYTES {
            return Err(ParseError::TooLarge);  // Header bomb: 8 KiB with no header end.
        }
        buf.as_ref()  // Still reading headers; below header cap.
    }
};
```

(Exact syntax may differ — match the existing style. Key invariant: only reject when buffer is too big AND no `\r\n\r\n` found yet.)
