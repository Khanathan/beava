# Phase 19.1.1 ‚Äî HTTP buffer-cap hotfix ‚Äî Context

Single-decision hotfix mini-phase inserted between Phase 19.1 Wave 1 and Wave 2. Discovered during Plan 19.1-02 smoke-run: `crates/beava-runtime-core/src/http_listener.rs:69-74` rejects ANY HTTP request larger than 8 KiB ‚Äî including bodies ‚Äî because `MAX_HEADER_BYTES` is checked against the whole buffer instead of just header bytes. fraud-team.json's 15 KiB register POST trips this on the second TCP packet, before `Content-Length` parsing happens.

`MAX_BODY_BYTES = 4 MiB` exists at line 143 but is unreachable because the early-return at line 69-74 fires first.

Phase 19.1 Wave 2 (lazy buckets + WAL bump) and Wave 3 (rebaseline) need fraud-team.json to register against the live bench server. 19.1.1 fixes the cap split so the rest of 19.1 can run.

## Phase Boundary

**In scope:**
- `crates/beava-runtime-core/src/http_listener.rs:69-74` ‚Äî split the buffer-size cap into:
  1. Header-only cap (up to `\r\n\r\n` boundary, ‚â§ `MAX_HEADER_BYTES = 8 KiB`)
  2. Body cap via `Content-Length` (‚â§ `MAX_BODY_BYTES = 4 MiB`)

<decisions>
## Implementation Decisions

### Buffer cap split

- **D-01:** Two separate caps. `MAX_HEADER_BYTES = 8 KiB` applies only to bytes BEFORE the `\r\n\r\n` boundary (httparse's `header_end`). `MAX_BODY_BYTES = 4 MiB` applies to body bytes via `Content-Length` header check (already at line 143; just make it reachable).
- **D-02:** Keep the early-return semantics for genuinely-too-large headers. If 8 KiB are read with no `\r\n\r\n` found, reject with `ParseError::TooLarge` (header bomb defense).
- **D-03:** Body-cap behavior unchanged from existing line 143 logic ‚Äî return `ParseError::TooLarge` if `Content-Length > MAX_BODY_BYTES`. Don't relax this.
- **D-04 (Claude's discretion):** Whether to expose `MAX_HEADER_BYTES` / `MAX_BODY_BYTES` via env-var tunables. Defer ‚Äî defaults are fine; a 4 MiB body cap covers the realistic-fraud register payload by 250√ó.

### TDD shape

- **D-05:** RED test: synthesize a 15 KiB POST body, send through `parse_http_request`, assert `Ok(_)`. Test fails on current code (`Err(ParseError::TooLarge)`). Commit: `test(19.1.1-01): RED ‚Äî parse_http_request accepts 15 KiB body`.
- **D-06:** GREEN: split the buffer-cap. Commit: `feat(19.1.1-01): GREEN ‚Äî split MAX_HEADER_BYTES vs body cap via Content-Length`.
- **D-07:** Header-only-cap regression test (additive): assert that a 9 KiB header bomb still gets rejected with `ParseError::TooLarge`. Lives in same test file, separate test fn.

### Threat model

- **D-08:** New attack surface: none. The fix RESTORES intended behavior (body up to 4 MiB) that was unreachable due to a prior bug. Header-bomb defense is preserved (8 KiB cap on header bytes only). No new code paths reachable by adversary.

</decisions>

<code_context>
## Existing Code Insights

- `parse_http_request` uses httparse for header parsing. After `\r\n\r\n`, body is read separately based on `Content-Length`.
- The early-return at lines 69-74 is a header-bomb defense (httparse stack-allocs ~32 headers √ó 256 B each = 8 KiB). The defense is correct in intent but incorrectly applied to the whole buffer.
- Line 143 has the Content-Length check that should gate bodies ‚Äî it's the right place; just unreachable due to the early-return.

</code_context>

<specifics>
## Specific Ideas

- The fix can probably be implemented as: change the condition from `buf.len() > MAX_HEADER_BYTES` to a "have we found `\r\n\r\n` yet?" check. If no, AND buf.len() > MAX_HEADER_BYTES, fail (header bomb). If yes, proceed to body parsing path which has its own cap.
- Use `httparse::Status::Partial` vs `httparse::Status::Complete` to distinguish "still reading headers" from "headers done; reading body".

</specifics>
