# Plan 18-02 — Lock-free WAL with 3-buffer state machine + dedicated writer/fsync thread

## Goal

Eliminate **all per-event synchronization overhead** on the apply thread's WAL path. Replace the Phase 6.1 mpsc-to-fsync-worker hop AND avoid the Mutex-per-append pattern entirely, by giving apply sole ownership of the active WAL buffer and using a 3-buffer state machine to hand sealed buffers off to a dedicated writer/fsync thread.

This is the single biggest expected uplift in Phase 18 — the 43% reactor cost samply measured on Phase 13.3 was largely the mpsc + tokio task-switch pattern. Removing both AND avoiding a Mutex on the hot path puts apply CPU at ~3-5% on the WAL append (down from ~10% Mutex pattern, ~23% Redis-style inline-flush pattern at 3M EPS).

**Architecture decided 2026-04-25** after walking through Cassandra (mmap-write, accepts SIGBUS), LMDB (chose write() default, opt-in mmap), Redis (single-thread inline write, ~23% CPU at 3M EPS), and PostgreSQL (group-commit Mutex, ~15% CPU at 3M EPS). Ping-pong + lock-free single-writer matches Disruptor/io_uring/AF_XDP rings semantically, sized per PostgreSQL/MySQL defaults.

## Scope: ~400 LoC

(Was ~300 in original; +100 for 3-buffer state machine + watermark tracking + per-buffer LSN range bookkeeping.)

## Architecture

### Buffer ownership state machine (no Mutex)

Each WAL buffer (16 MiB) is in exactly one state at any time:
- **active** — apply thread is sole writer. Apply memcpys records into this buffer + bumps an atomic position cursor.
- **sealed** — handed off to writer thread. Apply does NOT touch.
- **flushing** — writer thread is calling `write(fd, ...)` + `fsync(fd)`. Writer thread owns.
- **free** — safe to reuse as the next active buffer.

Transitions are atomic-pointer swaps. Per-event apply cost: just memcpy + atomic position bump = ~15 ns. No Mutex on hot path.

### Four-watermark discipline (LSN tracking)

Tracked as four atomic `u64`s in `WalLsn`:
- `committed_lsn` — bytes accepted into in-memory buffer (apply set this; valid for `/push` Periodic acks)
- `written_lsn` — `write(fd)` to kernel page cache completed (writer thread set this)
- `synced_lsn` — `fsync(fd)` returned (writer thread set this; valid for `/push-sync` acks)
- `acked_lsn` — derived; the watermark for whatever durability policy the client requested

Per-buffer LSN range tracked in `BufferState { lo: Lsn, hi: Lsn, state: ActiveSealedFlushingFree }` so when buffer A becomes synced, we know which `/push-sync` waiters to wake.

### Threads

- **apply thread** (existing, single-thread event loop) — sole writer to the active buffer
- **writer + fsync thread** (NEW, single `std::thread`) — owns sealed buffers, calls `write()` + `fsync()` per tick or per buffer-full event, advances `written_lsn` and `synced_lsn`

Writer + fsync are merged into ONE thread for v0 simplicity. Sequential per buffer: `write(A) → fsync(A) → free(A)`. If perf gates show fsync latency hides write throughput, split into two threads in Plan 18-05/18-06. Pipelining is a Phase 18 v2 optimization, not a v0 must.

### File handle

WAL segment file opened with `O_APPEND | O_WRONLY | O_CREAT` (Phase 6.1 currently uses `O_WRONLY | O_CREAT`). O_APPEND is defense-in-depth — single-writer architecture means atomicity isn't strictly needed, but cost is zero so we add it. Refuse to start if WAL directory is on a network FS (NFS/SMB/FUSE) — atomic appends don't compose reliably there.

### Tunables

| Env var | Default | Purpose |
|---------|---------|---------|
| `BEAVA_WAL_BUF_BYTES` | `16777216` (16 MiB) | Per-buffer size |
| `BEAVA_WAL_FSYNC_TICK_MS` | `5` | Fsync cadence |
| `BEAVA_WAL_BUFFER_COUNT` | `3` | Number of buffers in state machine (min 2) |

**2.2.b (GREEN):**

  - States encoded as `u8`: ACTIVE=0, SEALED=1, FLUSHING=2, FREE=3

**Edge cases to test:**
- Active full mid-append: append returns to caller indicating "swap and retry"; caller swaps and writes to new active
- All free buffers exhausted (writer thread fell 3+ ticks behind): apply blocks (sleeps Condvar) until writer frees a buffer. **This is the only blocking case on apply hot path** and only fires under sustained overload. Document as a backpressure mechanism.
- Concurrent reads: writer thread takes ownership of sealed buffer; apply thread must not touch it (verified by test using TSan-equivalent or runtime invariants)

### Task 2.3 — Writer + fsync thread (~110 LoC)

**2.3.a (RED):** test that:
- Writer thread starts on `WalWriter::spawn()`, runs in its own `std::thread`
- Writer pops sealed buffers from queue, calls `write(fd, &buf.bytes[..buf.pos])`, advances `written_lsn`
- Writer then calls `fsync(fd)`, advances `synced_lsn`
- Writer returns sealed buffer to free pool (state transition)
- Fsync tick (5ms default) triggers seal even if active buffer not full
- Shutdown: writer drains queue + final fsync before exit

**2.3.b (GREEN):**
- New file `crates/beava-runtime-core/src/wal_writer.rs` with:
  - `pub struct WalWriter { fd: File, ring: Arc<WalBufferRing>, lsn: Arc<WalLsn>, tick_ms: u64 }`
  - `pub fn spawn(self) -> JoinHandle<()>` — moves into std::thread
  - Worker loop:
    1. `std::thread::sleep(tick_ms.into_duration())`
    2. `ring.seal_active_if_due()` (force seal even if not full)
    3. While `let Some(buf) = ring.sealed_queue.pop()`:
       - `write(fd, &buf.bytes[..buf.pos.load()])`
       - `lsn.mark_written(buf.lsn_hi)`
       - `fsync(fd)` (or `fdatasync` where available)
       - `lsn.mark_synced(buf.lsn_hi)` + Condvar wake
       - `ring.return_to_free(buf)` (state transition)
- File: open with `OpenOptions::new().append(true).write(true).create(true).custom_flags(O_APPEND)` — explicit O_APPEND for clarity

**Edge cases to test:**
- High event rate: 1 M events/sec sustained for 30s, assert no event loss; `synced_lsn` advances within `2 × tick_ms` latency of `committed_lsn`
- Disk full / EIO: write returns error; writer thread sets `WAL_BROKEN` atomic; apply checks this on next append → rejects with 503 Service Unavailable
- macOS F_FULLSYNC: fsync can take 5-10ms (longer than tick); ensure writer doesn't wedge — if fsync took > 2× tick, log warning
- Shutdown: SIGTERM → flush + final fsync before process exit
- File rotation: when WAL file exceeds `BEAVA_WAL_SEGMENT_BYTES` (default 1 GiB), writer rotates to new segment file (fsync parent dir for crash safety)

## Risks & mitigations

- **Buffer state-machine atomic ordering bugs** — The active→sealed→flushing→free transitions involve multiple atomic operations across two threads. Easy to get memory ordering wrong. Mitigation: encapsulate in `WalBuffer::try_append` and `WalBufferRing::seal_active` with documented invariants; loom-test or fuzz-test concurrent semantics; document Acquire/Release pairing on each atomic.
- **Sealed-buffer queue grows under burst** — If writer thread can't keep up (slow disk), sealed queue grows; apply blocks waiting for free buffers. Mitigation: 3 buffers gives 15ms (3 × 5ms tick) of burst absorption at 3M EPS; document the soft-overload behavior; surface via `beava_wal_buffer_stalls_total` Prometheus counter (Plan 13's middleware).
- **Network FS catastrophic failure modes** — O_APPEND atomicity isn't reliable on NFS/SMB; mmap WAL would be even worse there. Mitigation: refuse-on-network-FS at startup via `statfs()` check; document local-block-storage requirement.
- **macOS F_FULLSYNC > tick** — fsync can take longer than the 5ms tick. Writer falls behind; sealed queue grows. Mitigation: 3-buffer headroom; warning log if fsync > 2× tick; document macOS dev-only constraint (Linux NVMe much faster).
- **Crash safety regression** — Keep all phase6.1 crash tests in the suite; run them on every Stage 18.2 commit. Add new crash tests for the 3-buffer state (kill at each transition point).
- **/push-sync waiter starvation** — If many waiters pile up at high LSN and writer thread is slow, waiters time out. Mitigation: `BEAVA_PUSH_SYNC_TIMEOUT_MS` (default 5s) returns 503 with retry hint; surface via Prometheus.

## Plan 18-07 implications

The new `WalBufferRing` + `WalWriter` replace Phase 6.1's `WalSink::append_event_with_mode` async API on the hand-rolled runtime path. Plan 18-07 (tokio data-plane removal) will:
1. Remove the legacy `WalSink` async API and its mpsc machinery (no longer used after data-plane is hand-rolled-only)
2. Migrate any remaining tokio-path callers (admin endpoints don't append to WAL; they only read state)
3. Update Phase 6.1 tests to exercise the new ring directly (tokio path's tests become hand-rolled tests)
