# Plan 18-02 — Inline WAL + pthread fsync worker

## Goal

Eliminate the 2 task switches per request that came from mpsc-to-fsync-worker on the tokio path. WAL append becomes a direct `Rc<RefCell<WalWriter>>` call from the apply thread. fsync runs on a separate `std::thread` with no tokio coordination.

This is the single biggest expected uplift in Phase 18 — the 43% reactor cost from samply was largely this hop pattern.

## Scope: ~300 LoC

## Tasks (TDD red-green per task)

### Task 2.1 — `Rc<RefCell<WalWriter>>` direct call from apply thread (~150 LoC)

**2.1.a (RED):** test in `phase18_02_inline_wal_test.rs` that the apply thread can call `wal.borrow_mut().append(record)` directly and get back an LSN; no async/await on the path.

**2.1.b (GREEN):**
- `crates/beava-runtime-core/src/wal_inline.rs`: `WalWriter` struct (re-exported from `beava-persistence` or wrapped) holding the in-memory append buffer + segment file handle.
- Method: `WalWriter::append_inline(&mut self, record: WalRecord) -> Result<Lsn, WalError>`. Synchronous. Updates the buffer, returns next LSN.
- Glue: `runtime_core_glue.rs` apply path calls `state.wal.borrow_mut().append_inline(record)` instead of `state.wal_sink.append_event_with_mode(...).await`.
- Remove the mpsc-to-fsync-worker hop entirely from the hand-rolled hot path.

**Edge cases to test:**
- Buffer overflow (segment rotation): assert apply still returns successfully; rotation happens inline.
- Concurrent fsync read of buffer: assert fsync thread can read while apply is writing (use a single `Mutex` between them OR atomic-snapshot pattern; pick the simpler one).

### Task 2.2 — fsync worker on `std::thread` (no tokio) (~100 LoC)

**2.2.a (RED):** test that fsync worker periodically calls `File::sync_data()` and updates the durable-LSN watermark; apply thread is NOT blocked during fsync.

**2.2.b (GREEN):**
- `crates/beava-runtime-core/src/fsync_worker.rs`: `FsyncWorker` running on `std::thread`.
- Loop: `std::thread::sleep(tick_ms)`, then take a snapshot of the WAL buffer, fsync, update the durable-LSN atomic.
- Coordination: WAL writer → fsync worker via `Arc<Mutex<WalBuffer>>` for buffer access; durable-LSN via `Arc<AtomicU64>`.
- `tick_ms` configurable; default 5ms (Redis-style group-commit).

**Edge cases to test:**
- High event rate: 1M events/sec for 30s, assert no event loss; durable-LSN advances within tick_ms latency.
- Idle periods: tick fires, no events to flush, no errors logged.
- Shutdown: fsync worker drains buffer + final fsync before exit.

### Task 2.3 — `/push-sync` (PerEvent mode) integration (~50 LoC)

**2.3.a (RED):** test that `/push-sync` blocks until fsync acknowledges the request's LSN.

**2.3.b (GREEN):**
- For PerEvent mode: apply queues a `oneshot::Sender<Lsn>` to fsync worker's "pending" list (keyed by LSN).
- fsync worker, after each fsync syscall, fires all oneshots whose LSN ≤ durable-LSN.
- Apply thread waits on the oneshot (this IS a sync wait, but only on PerEvent path; never on Periodic path).
- `Periodic` mode unchanged — apply returns immediately after WAL buffer append.

## Perf gate 2.1 (Apple-M4, INFORMATIONAL)

After all tasks, run beava-bench with hand-rolled runtime + Periodic WAL:
```bash
cargo bench -p beava-bench --features hand-rolled-runtime -- --pipeline small --transport tcp --batch-ms 0 --duration-secs 60
```

Expected:
- TCP single-push (Periodic): **300-500k EPS/core** on M4
- HTTP single-push (Periodic): 100-150k EPS/core (JSON parse floor still)
- WAL append microbench: ≤ 200ns/event

If gate fails: investigate. The 2-task-switch removal SHOULD give us ~5-10× lift over Stage 18.1.

## Perf gate 2.2 (Apple-M4, INFORMATIONAL)

samply profile:
- WAL-related CPU: ≤ 5% (was ~10% in Phase 13.3 BufWriter::flush_buf inline)
- fsync syscall does NOT appear on apply thread's flamegraph (it should be on the fsync_worker thread only)
- No mpsc/oneshot machinery showing on hot path

## Verification

Plan complete when:
- WAL append happens inline; no mpsc-to-fsync-worker on hot path
- Periodic mode: zero await on apply; PerEvent mode: oneshot wait works
- M4 perf gates 2.1 + 2.2 met
- Existing phase6 + phase6.1 + phase7 tests still pass (durability invariants preserved)
- WAL crash tests still pass (kill before fsync, restart, verify event missing for Periodic; present for PerEvent)

## Risks & mitigations

- **Lock contention on `Arc<Mutex<WalBuffer>>`**: apply writes, fsync worker reads. Brief lock holds (~100ns each). Mitigation: if profiling shows contention, switch to `crossbeam_queue::ArrayQueue` (lock-free SPSC). Probably unnecessary at v0.
- **Buffer overflow during high write burst**: bounded buffer with backpressure on apply (block apply briefly until fsync drains). Default buffer size: 4MB.
- **Crash safety regression**: keep all phase6.1 crash tests in the suite; run them on every Stage 18.2 commit.
