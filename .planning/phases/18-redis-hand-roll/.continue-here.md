<decisions_made>
1. **Phase 13.3 REJECTED — single-threaded data plane forever** (locked 2026-04-26):
   - Rationale: in-process apply sharding (lockless RefCell + LocalSet) would force cross-shard queries to scatter-gather inside the process, breaking atomic semantics and complicating API surface
   - Path to >1M EPS aggregate: multiple Beava instances + per-key sharding (Redis-cluster pattern)
   - Per-instance throughput ceiling = single apply thread (~1M EPS small / ~400k medium / ~150k large_phase9 on Linux Xeon)
   - Memory: `project_no_sharded_apply.md`

2. **TRACE_AGG_TIMING split from TRACE_APPLY_TIMING** — inner eprintln was contaminating outer agg measurement; clean separation lets us measure each independently

3. **Pipelining is client-side opt-in, not automatic** — TCP wire is FIFO + no request_id (Redis-style); SDKs need explicit pipelining infra. Documented for future SDK improvements.

4. **3M EPS/core ship-gate REFRAMED post-Phase-13.3-rejection** — was "single-instance via in-process sharding"; now "per-instance peak achievable on Linux Xeon with all 18-04.7 + 18-04.8 + 18-12 + future 18-05 io_uring + OP_PUSH_MANY"

5. **Burst pipelining baseline preserved** — `--pipeline-depth N` flag landed in bench-v18; demonstrated 6-8× EPS lift on M4 loopback at p=16/pd=256. Continuous pipelining (queued) will replace it with constant load.
</decisions_made>
