# Plan 18-07 — Hot-path consolidation (tokio removal + Phase 12.5 routes + Phase 16 rename)

## Goal

Collapse the dual-runtime architecture into the single hand-rolled runtime, lock in the new Phase 12.5 wire surface, and absorb Phase 16's HTTP rename. After Plan 18-07 lands, Phase 18's perf gates measure the SINGLE production runtime — no feature-flag confusion, no parallel code paths, no reverse-bench artifacts.

This plan absorbs three workstreams:

**(a) Tokio data-plane removal** — Per user decision 2026-04-25 ("remove the tokio we can revert it if not working"). The `--features hand-rolled-runtime` flag was an intermediate scaffold; this plan rips it out, deleting the tokio data-plane code paths entirely. Tokio stays for admin endpoints on port 8081 (matches Phase 18 CONTEXT D-01).

**(b) Phase 12.5 routes** — Per user decision 2026-04-25, the new combined-endpoint routes (`/push-and-get/:event`, `/push-sync-and-get/:event`) land in 18-07's router rather than waiting for Phase 12.5's separate executor.

**(c) Phase 16 HTTP rename** — Per user decision 2026-04-25, the `/push-table → /upsert` and `/delete-table → /delete` rename absorbs from Phase 16 Plan 02. Phase 16 will mark Plan 02 as "absorbed by Plan 18-07" in its CONTEXT.md.

**7.3.b (GREEN):**

- `crates/beava-runtime-core/src/wire.rs`: extend `WireRequest` enum with `PushAndGet { event, body }` and `PushSyncAndGet { event, body }`. Extend `parse_http_to_wire` to dispatch the new routes.
- `crates/beava-runtime-core/src/wire.rs`: add TCP opcode constants `OP_PUSH_AND_GET = 0x0015` and `OP_PUSH_SYNC_AND_GET = 0x0016`. Extend `parse_wire_frame` to handle them.
- `crates/beava-server/src/runtime_core_glue.rs`: add `dispatch_push_and_get` and `dispatch_push_sync_and_get` methods. Each:
  1. Parses `{row, query}` from body
  2. Calls `wal_append_periodic` (or `_per_event` for the sync variant) — captures committed_lsn
  3. Calls `app_state.apply_event` with the row → updates state
  4. Calls `app_state.query_features(query)` — gets values; reflects the just-applied event
  5. Returns response: `{ack_lsn, registry_version, features, warnings}`
  6. The atomicity is by virtue of running on the single apply thread between push and query — no other event can interleave
- For unknown features in `query.features`: return 200 with that feature mapped to `null` and warning code `unknown_feature` (matches Phase 12.5 SC5).
- For unknown event source: return 400 with `unknown_event` (matches /push behavior).

- **`cargo build --workspace` cache rebuild** — removing the feature flag changes the dependency graph; first build after 18-07 will be slow but subsequent are normal.
