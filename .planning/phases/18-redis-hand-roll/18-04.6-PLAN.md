# Plan 18-04.6 — Real EventLoop integration into ServerV18::serve()

## Goal

**Wire the components that Plans 18-01..04 built into a single working serve loop.** Currently those plans shipped pieces in isolation: hand-rolled mio EventLoop (18-01), lock-free WAL with 3-buffer state machine (18-02), I/O thread pool for reads (18-03) and writes (18-04). The just-shipped `ServerV18::serve()` (commit 6f457ce) does NOT use any of them — it's an axum/tokio shim that bridges to the legacy async WalSink, which is why the first M4 measurement came back at 234 EPS (6× SLOWER than the 1,413-EPS legacy path).

This plan replaces that shim with a real serve loop that:
1. Runs mio Poll loop on a dedicated `std::thread` (NOT inside a tokio task)
2. Calls `EventLoop::tick()` in the canonical Redis-shaped pattern: poll → IoPool read distribute → join → execute apply → IoPool write distribute → join
3. Uses `WalBufferRing::append()` synchronously on the apply path (NO async/await, NO mpsc to fsync worker)
4. Keeps the tokio admin server running on a separate thread + port (unchanged)

Scope was implicitly assumed by Plans 18-05/18-06 but no plan actually owned it — surfaced 2026-04-25 when the v18 bench harness measurement showed 234 EPS instead of the 300-500k Plan 18-02 floor.

## Architectural decisions (locked at plan-write time, 2026-04-25)

### D-1 — AppState ownership: single-writer Mutex (not Rc<RefCell>)

Phase 18 CONTEXT D-02 originally said `Rc<RefCell<AppState>>` direct ownership. That is the v0 IDEAL but requires Phase 13.3's lockless-apply refactor (currently in progress on `phase-13.3-lockless-apply` worktree, not merged).

**For Plan 18-04.6:** keep the existing `Arc<Mutex<AppState>>` (or equivalent `ConcurrentAppState`) wrapper but use it from the SINGLE apply thread only. The Mutex is uncontended (single writer), so lock+unlock cost is the uncontended fast path (~10-20ns). When Phase 13.3 lands and gets merged, a follow-up plan can drop the Mutex entirely.

This decision avoids pulling Phase 13.3's whole scope into Phase 18.

### D-2 — Async-to-sync bridge: removed from data plane

The 6f457ce serve() bridges `dispatch_wire_request` (async) → `WalSink::append_event_with_mode` (mpsc-based). Plan 18-04.6 deletes that bridge. The new apply path is purely synchronous:
1. Apply thread reads parsed `WireRequest` from per-client buffer (set by IoPool worker in read phase)
2. Apply thread calls `state.lock().unwrap()` (uncontended), runs the dispatch logic synchronously, including `wal_buffer_ring.append(record_bytes)` which is also sync (just memcpy + atomic position bump per Plan 18-02)
3. Apply thread enqueues response into `client.output_queue` (per Plan 18-04)
4. IoPool workers handle write phase

The async WalSink mpsc machinery becomes dead code on the data plane (still used by admin endpoints if any need it, which they shouldn't — admin is read-only).

### D-3 — runtime_core_glue: keep async API, add sync API

The existing `dispatch_wire_request(...)` is async (used by the 6f457ce shim and by integration tests). DON'T remove it — those callers still need it. Add a NEW sync method `dispatch_wire_request_sync(...)` for the serve loop. The async method becomes a thin wrapper that calls the sync version inside `tokio::task::spawn_blocking` (or similar) for the non-hot-path callers (admin endpoints if any, integration tests).

This keeps 18-04.6 backward-compatible with everything that existed before, while providing the sync hot path the apply thread needs.

### D-4 — Threading model

| Thread | Role | Type |
|--------|------|------|
| 1 apply thread | mio Poll + EventLoop::tick + dispatch + WalBufferRing append | std::thread |
| N IoPool workers | parallel read parse + write serialize | std::thread (Plans 18-03/04) |
| 1 WalWriter thread | drain sealed buffers + write() + fsync() | std::thread (Plan 18-02) |
| 1 tokio runtime | admin endpoints (/metrics /health /ready /registry) | tokio + axum |

Total: 1 + N + 1 + tokio-pool. Default N = num_cpus()-1 per Plan 18-03 default.

### D-5 — Bench harness migration

`beava-bench-v18` (commit 9c87bb0) currently calls `ServerV18::serve_with_dirs()` which is the shim. After 18-04.6 lands, the bench needs to call the new `serve_with_dirs()` (same name, new impl) — no API change needed at the bench level. The smoke test (commit bb82dc0) similarly continues to work.

## Risks & mitigations

- **AppState lock contention** — even with single-writer pattern, if some background task or admin endpoint also locks the state, the apply thread can stall. Mitigation: audit lock callers; if found, add a "dataplane" Mutex variant that only the apply thread takes. If impossible, this is the Phase 13.3 problem and 13.3 needs to land first.
- **mio + tokio in same process** — running mio on its own thread alongside tokio's runtime should be fine on macOS (kqueue used by both), but watch for syscall conflicts. Mitigation: separate threads + separate fd pools.
- **Existing tests use TestServer** — don't break legacy tests. ServerV18 is a parallel API; TestServer + legacy Server stay around for backward compat. Plan 18-04.6 only changes ServerV18's serve impl.
- **6f457ce shim cleanup** — the prior shim has dead code paths after 4.6.3. Don't delete in this plan; flag for cleanup in Plan 18-08 (post-verification housekeeping).
