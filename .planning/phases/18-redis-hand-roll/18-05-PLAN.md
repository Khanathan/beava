# Plan 18-05 — Per-worker continuous loops + IoBackend trait (Valkey 8 model)

## Goal

Eliminate the last `IoPool::join_all` spin barrier (the WRITE phase, post-18-13) and prepare the architecture for io_uring on Linux. The reshape is **not** a "swap kqueue for io_uring under the existing IoPool model" — it's a switch to **per-worker continuous-loop workers**, which is the Valkey 8 architecture validated to deliver +230% throughput at the same hardware (360k → 1.19M RPS).

After this plan lands:
- Apply thread is purely a dispatcher (drains `read_rx`, calls `ApplyShard::dispatch_*`, pushes to `write_tx[w]`, polls listeners). Never touches client sockets.
- Each worker thread owns a disjoint subset of clients (`slot_idx % N`), runs its own mio::Poll, parses + serializes + writes for its clients independently.
- Channels carry all coordination. mio Waker handles cross-thread wakes. No spinning, no join_all.

This is the architectural milestone that closes the read+write gap on macOS and unlocks io_uring on Linux.

## Architecture (post-18-05 target state)

```
                 ┌──────────────────────────────┐
                 │  Apply thread                 │
                 │  - polls 2 listeners          │
                 │  - drains read_rx             │
                 │  - calls ApplyShard           │
                 │  - pushes to write_tx[w]      │
                 │  - wake worker[w]             │
                 │  - accepts; hands off to      │
                 │    new_client_tx[w]           │
                 └─┬───┬──────────────┬──────────┘
                   │   │              │
        new_client_tx[w]  write_tx[w]
                   │   │              │ wake worker[w]
                   ▼   ▼              ▼
           ┌─────────────┐ ──read_tx──► (apply)
           │  Worker w   │
           │  - owns mio::Poll          │
           │  - owns Vec<ClientState>   │
           │  - reads + parses → push to read_tx
           │  - drains write_rx → encode + flush
           │  - mio::Waker registered with own Poll
           └─────────────┘
           ... (N workers, default 5) ...
```

## Key decisions (locked from design walkthrough 2026-04-26)

| Decision | Value | Rationale |
|---|---|---|
| Default IO threads | **5 fixed** | Beava's parse:apply ratio is 4:1; 5 workers balance with apply at 1M EPS theoretical. `max(2, num_cpus/4)` was per-tick-era heuristic. |
| Idle behavior | **mio Waker** | Sleep in `mio::poll(Some(LONG_TIMEOUT))`; wake via Waker.wake() from cross-thread. Zero CPU when idle, ~10-50 µs wake latency. |
| CPU pinning | **defer** | Revisit during Plan 18-05.5 Linux Xeon tuning if needed. |
| State sync | **Mutex preserved (Path A)** | Maintains G1 (per-event multi-feature atomicity) + G2 (cross-aggregation atomicity). Lock-free path (Plan 18-17) deferred to v0.1+ as opt-in. |
| Encode location | **on worker** | Distributes encode CPU; keeps apply focused on dispatch. |
| Worker → client routing | **`slot_idx % N`** | Deterministic, cache-friendly, simpler than dynamic load-balancing. |
| Listener polling | **on apply** | Apply has its own small `mio::Poll` for the 2 listeners + accepts. Workers don't see listeners. |
| Channel sizes | read_tx **16384**, write_tx **4096** each, new_client_tx **256** each | Absorb bursts; simple back-pressure if filled. |
| Migration | **outright replacement** | v18 path is internal; no public API to preserve. Maintain one path. |
| Linux Xeon HARD GATE | **deferred to 18-05.5** | Requires HW; build-verify here, measurement when HW available. |

**5.2.b (GREEN):**
- New module `crates/beava-runtime-core/src/io_thread_worker.rs` with `worker_main_loop<B: IoBackend>(...)`. Worker holds:
  - `B` (the backend, with own Poll + Waker)
  - `clients: HashMap<u64, ClientState>` (own subset)
  - `new_client_rx: crossbeam_channel::Receiver<NewClient>` (apply hands off)
  - `write_rx: crossbeam_channel::Receiver<(u64, GlueResponse)>` (apply pushes responses)

- Worker loop sketch (per design note):
  ```rust
  loop {
      while let Ok(nc) = new_client_rx.try_recv() { backend.add_client(...); clients.insert(...); }
      while let Ok((slot, resp)) = write_rx.try_recv() {
          if let Some(c) = clients.get_mut(&slot) { encode_glue_response_*(resp, &mut c.write_buf); }
      }
      backend.poll(Some(Duration::from_secs(1)), &mut events)?;
      for ev in events.drain(..) {
          match ev {
              IoEvent::Readable(s) => parse_and_push(s, &mut clients, &mut backend, &read_tx),
              IoEvent::Writable(s) => flush_write_buf(s, &mut clients, &mut backend),
              IoEvent::Closed(s)   => { clients.remove(&s); backend.close(s); }
              IoEvent::WakerSentinel => continue, // re-loop, drain channels
          }
      }
      if stop.load(Acquire) { break; }
  }
  ```
- Apply thread refactor: removes the per-tick `read_set` / `write_set` classification, no more `IoPool::publish + join_all`. Apply's main loop:
  ```rust
  loop {
      listener_poll.poll(Some(short_timeout), &mut listener_events)?;
      for ev in listener_events { accept_loop_round_robin(); }
      while let Ok(item) = read_rx.try_recv() {
          let resp = apply_shard.dispatch_wire_request_with_row(item.request, item.parsed_row);
          let w = (item.slot_idx as usize) % n_workers;
          for r in resp { write_txs[w].send((item.slot_idx, r))?; worker_wakers[w].wake()?; }
      }
      if stop.load(Acquire) { break; }
  }
  ```

## Risks & mitigations

- **Architectural blast radius**: This is a major refactor (~1500 LoC across 8 files). Risk of subtle bugs in cross-thread coordination. Mitigation: TDD per task; comprehensive `phase18_05_continuous_workers_test.rs` covering each design decision (round-robin routing, mio Waker wake, channel backpressure, shutdown).

- **Channel deadlock**: Apply blocks on `write_tx[w].send()` if worker w's queue is full. Worker blocks if `read_tx.send()` is full and apply is slow to drain. Mitigation: bounded(16384) on read_tx, bounded(4096) per write_tx — generous capacity; `try_send` with yield-and-retry on the rare full case. Document the backpressure invariant: apply must drain `read_rx` continuously; workers must drain `write_rx` continuously.

- **macOS perf regression**: New architecture might be SLOWER on M4 than 18-13's already-fast post-merge ceiling (527k EPS). Mitigation: TRACE_APPLY measurement post-Task 5.2 should show gap drop further (the write phase join_all removed). If regression observed, debug before continuing.

- **io_uring crate API drift**: `io-uring 0.6.x` API may change. Pin to `=0.6.4` exact version to prevent surprise breaks.

- **Compile-time blow-up from generics**: `IoBackend` trait with monomorphization may bloat compile time. Mitigation: keep `EventLoop<B>` generic in tight scope; switch to `&dyn IoBackend` at outer boundary if compile time grows >20%.
