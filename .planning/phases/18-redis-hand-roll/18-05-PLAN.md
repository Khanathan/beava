# Plan 18-05 — io_uring on Linux (HARD GATE)

## Goal

Replace `mio` (epoll on Linux, kqueue on macOS) with **io_uring** on Linux for
the data-plane TCP + HTTP I/O path. io_uring eliminates two-syscall-per-event
overhead (`epoll_wait` + `read`) by batching submission and completion through
shared rings, and supports zero-copy operations (`IORING_OP_RECV_MULTI`).

This is the FIRST hard-gated stage in Phase 18 (per D-14, D-16). Apple-M4 is
informational; the ≥3M EPS/core target is measured on the Linux Xeon runner
established in Stage 18.4.5.

The architecture stays identical to Stages 18.1-18.4: same `IoPool`, same
apply thread, same `WireResponse` flow. Only the backend abstraction (poll +
read + write syscalls) changes per OS. macOS retains `mio` (`kqueue`).

Reference: `18-redis-research.md` §"io_uring future" + `18-rust-translation.md`
notes on Send/Sync rules.

Document the SQE/CQE flow in a comment block at the top of `io_uring.rs`:
```text
Per-tick lifecycle for io_uring backend:
  1. submit_read pre-armed recv SQEs at client registration time (multi-shot)
  2. backend.poll(timeout) drains completion queue → IoEvent vec
  3. main loop dispatches Read events to io_pool
  4. io_pool workers do parsing (no syscall — bytes already in BytesMut from CQE)
  5. apply thread runs
  6. for each output: submit_write enqueues IORING_OP_WRITEV SQE
  7. backend.poll on next tick drains write completions
```

## Risks & mitigations

- **HARD GATE may not be met on first try**: this is the entire reason Phase
  18 exists. Tuning SQ depth, SQPOLL, multi-shot recv may take days. Budget 1
  week of tuning before escalation. Pre-emptive mitigation: have Stage 18.6
  micro-optimizations (zero-copy argv, hand-rolled response formatters)
  ready as additional levers.
- **`io-uring` crate API changes**: pin to `0.6.x` exact version; document
  any patch-version migrations. Track upstream for any breaking changes
  before Phase 18 ship.
- **Kernel version requirement**: io_uring needs Linux 5.6+; multi-shot recv
  needs 5.18+. Document required kernel in `18-04.5-runner-setup.md`.
  Ubuntu 24.04 LTS ships 6.8 → fine.
- **SQPOLL kernel thread CPU cost**: SQPOLL spawns a kernel thread that
  spins on the SQ. On a busy machine this competes for CPU; on a dedicated
  bench host with isolated cores it's a clear win. Measure both; the
  decision is empirical.
- **`io_uring` security CVE history**: io_uring has had several escalation-of-
  privilege CVEs (most recently 2024). For OSS Apache 2.0 v0, this is on the
  operator (their kernel patch level). Document in
  `18-05-linux-baseline.md` deployment notes.
- **`unsafe` density rises**: the io-uring crate exposes a safe wrapper but
  buffer lifetime management for in-flight SQEs is delicate. Use the crate's
  `BufRing` mechanism where possible; document any custom unsafe in
  `// SAFETY:` blocks. Reviewer must spot-check.
- **Trait abstraction monomorphization compile-time blow-up**: keep the
  `EventLoop<B>` generic in tight scope; do NOT propagate the generic into
  every `impl` block. If compile time grows >20%, switch to `&dyn IoBackend`
  at the top of `tick()` and accept one virtual call per phase boundary
  (~6 calls per tick — negligible cost vs millions of EPS).
