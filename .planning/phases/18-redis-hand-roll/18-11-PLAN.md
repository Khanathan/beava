# Plan 18-11 — Hot-path optimization: HashMap AggStateTable + Row variant D + Arc<EventDescriptor>

## Context (read first)

After Plan 18-10, microbench + TRACE_APPLY profiling on M4 (parallel=1, n=27,646) revealed:

```
PUSH per-stage breakdown (post-18-10, mean ns):
  parse        911   17.7%   body bytes → Row deserialize
  lookup       299    5.8%   registry.read() + descriptor clone
  validate     114    2.2%   schema check + dedupe extract + event_time
  wal_build     88    1.7%   v=2 binary header serialization
  wal_append   158    3.1%   lock-free WAL ring
  agg        3,191   61.9%   ← apply_event_to_aggregations + Mutex
  bookkeeping  394    7.6%   counters + event_id_index + dedupe cache
  TOTAL      5,154  100.0%
```

Drilling into the 3,191 ns `agg` stage (n=8,857):

```
AGG sub-stages (mean ns):
  registry_call          98    3.8%   second registry.read() + filter + collect
  entity_key             96    3.7%   EntityKey::from_row
  table_lookup           85    3.3%   state_tables.entry(node_name.clone())
  entity_row_init     2,147   82.0%   ← BTreeMap<EntityKey, Vec<AggOp>>::entry(key.clone())
  features               57    2.2%   Count::update_with_row (the actual work)
  TOTAL agg          2,617  100.0%
```

**The container is 37× the work it contains.** The Count operator does 57 ns of useful work; the BTreeMap traversal + EntityKey clone takes 2,147 ns to set up that work.

Spike data (`body_to_row_variants.rs`, `53a9198`):
- Variant D (SmallVec<[(CompactString, ValueC); 8]>) for Row body→Row: **146 ns msgpack / 184 ns json** (vs 408/403 ns baseline)
- Variant E (Vec<Value> by column id) was slower than D — descriptor scan adds ~16 ns

This plan implements **all four** optimizations identified in the body-to-row-spike report and the agg-stage breakdown:

1. **Variant D for Row** — SmallVec storage + CompactString
2. **HashMap for AggStateTable** — FxHashMap + raw_entry_mut to eliminate `key.clone()`
3. **EntityKey storage upgrade** — SmallVec<[(CompactString, Value); 2]> matching Row's inline-string discipline
4. **Arc<EventDescriptor>** — eliminate per-push descriptor clone in `dispatch_push_sync`

Plus a free win discovered along the way:

5. **Per-source aggregation index** — replace `compiled_aggregations` BTreeMap linear scan with `aggregations_by_source: HashMap<String, Vec<Arc<AggregationDescriptor>>>` for O(1) lookup, eliminating the redundant registry RwLock acquire inside `compiled_aggregations_for_source`.

## Architecture decisions

### D-1 — `Row.0`: `SmallVec<[(CompactString, Value); 8]>` (was `BTreeMap<String, Value>`)

```rust
use compact_str::CompactString;
use smallvec::SmallVec;

pub struct Row(pub SmallVec<[(CompactString, Value); 8]>);
```

Most events have ≤8 fields → inline storage, **zero heap allocs for the row container**. CompactString does inline-storage for strings ≤24 bytes (most field names + most short str values).

`Row::get(&str)`: linear scan over the SmallVec (≤8 elements, cache-friendly). Faster than BTreeMap O(log N) for small rows.

`Row::with_field(field: &str, value: Value)`:
- Convert `&str` to `CompactString` once at the API boundary.
- If field already exists, replace value in-place.
- Else push to SmallVec.

For the `Row::Deserialize` hot path, use `with_field_owned(key: CompactString, value: Value)` to skip the conversion when the key already comes from `next_key::<&str>()` → `CompactString::from(&str)`.

### D-2 — `Value::Str(CompactString)` (was `Value::Str(String)`)

```rust
pub enum Value {
    Null,
    Str(CompactString),  // was String
    I64(i64),
    F64(f64),
    Bool(bool),
    Bytes(Vec<u8>),
    Datetime(i64),
    Json(serde_json::Value),
    List(Vec<Value>),
    Map(BTreeMap<String, Value>),  // unchanged — only used as agg output, not in Row fields
}
```

CompactString implements `Deref<Target=str>` so most string-comparison sites work unchanged via auto-deref. Sites that explicitly took ownership of a `String` (`Value::Str(s) => s` consuming pattern) need to switch to `Value::Str(s) => s.into_string()` or similar.

`Value::Map`'s inner `BTreeMap<String, Value>` is left alone — it's only used as an aggregation output (e.g. histogram, event_type_mix) and isn't on the per-push hot path.

### D-3 — `Row::Deserialize` via direct SmallVec push

```rust
fn visit_map<M>(self, mut access: M) -> Result<Row, M::Error> {
    let mut row = Row(SmallVec::with_capacity(8));
    while let Some(key) = access.next_key::<CompactString>()? {
        let value: Value = access.next_value_seed(BeavaValueSeed)?;
        row.0.push((key, value));
    }
    Ok(row)
}
```

Both `serde_json` and `rmp_serde` support `next_key::<CompactString>` if CompactString impls `Deserialize` (it does, via the `serde` feature). No String allocation per key.

`BeavaValueVisitor::visit_str`/`visit_borrowed_str`/`visit_string` all return `Value::Str(CompactString::from(v))`. CompactString chooses inline-vs-heap automatically.

### D-4 — `AggStateTable.entities: FxHashMap` + `raw_entry_mut`

```rust
use fxhash::FxBuildHasher;

pub struct AggStateTable {
    pub entities: HashMap<EntityKey, Vec<AggOp>, FxBuildHasher>,
}

pub fn get_or_init<'a>(
    &'a mut self,
    key: &EntityKey,
    descriptor: &AggregationDescriptor,
) -> &'a mut Vec<AggOp> {
    use std::collections::hash_map::RawEntryMut;
    match self.entities.raw_entry_mut().from_key(key) {
        RawEntryMut::Occupied(entry) => entry.into_mut(),
        RawEntryMut::Vacant(slot) => {
            let new_row = descriptor
                .features
                .iter()
                .map(|f| AggOp::new(&f.descriptor))
                .collect();
            slot.insert(key.clone(), new_row).1
        }
    }
}
```

Key observations:
- **`raw_entry_mut().from_key(key)`** takes a borrowed key for the lookup; only clones into the map on insertion (cold path). Hot path: no `key.clone()`.
- **FxBuildHasher** is non-cryptographic, ~3× faster than the default SipHasher for short keys (we don't need DoS resistance — apply path is single-writer single-process).
- `std::collections::HashMap` exposes `raw_entry_mut` as nightly... actually as of Rust 1.50 it's stable on the experimental API. **Use `hashbrown::HashMap` directly** if std HashMap raw_entry_mut isn't accessible — hashbrown is what std::HashMap wraps internally and exposes raw_entry_mut on stable.

### D-5 — `EntityKey` storage: `SmallVec<[(CompactString, Value); 2]>`

```rust
pub struct EntityKey(pub SmallVec<[(CompactString, Value); 2]>);
```

Most aggregations group by 1-2 keys. SmallVec inline storage for ≤2 entries → no heap alloc on construction.

`EntityKey::from_row` reads group_keys from the row directly into the SmallVec without intermediate Vec allocation.

`EntityKey::Hash` and `EntityKey::PartialEq` impls walk the SmallVec; equality is O(N) where N is the number of group keys (typically 1-2).

### D-6 — `Arc<EventDescriptor>` in `dispatch_push_sync`

```rust
// Today (apply_shard.rs:251):
let descriptor = match inner.events.get(event_name).cloned() {  // ← clones EventDescriptor
    Some(d) => d,
    ...
};

// After:
let descriptor = match inner.events.get(event_name) {
    Some(d) => Arc::clone(d),  // ← refcount bump only
    ...
};
```

Requires `Registry::events: HashMap<String, Arc<EventDescriptor>>` (currently HashMap<String, EventDescriptor>). Migration: wrap each EventDescriptor in Arc at registration time. Downstream callers that took `EventDescriptor` by value now take `&EventDescriptor` (deref through Arc) or `Arc<EventDescriptor>`.

`validate_row_against_descriptor(&descriptor, &row)` already takes `&EventDescriptor` — compatible.

### D-7 — Per-source aggregation index

Add a precomputed index to `Registry::inner`:

```rust
struct RegistryInner {
    ...
    compiled_aggregations: BTreeMap<String, Arc<AggregationDescriptor>>,  // unchanged
    aggregations_by_source: HashMap<String, Vec<Arc<AggregationDescriptor>>>,  // NEW
}
```

`Registry::compiled_aggregations_for_source` becomes:
```rust
pub fn compiled_aggregations_for_source(&self, source_name: &str) -> Vec<Arc<AggregationDescriptor>> {
    self.inner
        .read()
        .aggregations_by_source
        .get(source_name)
        .cloned()
        .unwrap_or_default()
}
```

Built-up in `apply_registration` whenever a new derivation lands. The `.cloned()` here clones a `Vec<Arc<...>>` — for typical small numbers of aggs per source this is tens of ns. Eliminates the linear scan and the per-call `.values().filter().collect()` allocation.

**Better still:** return `Vec<Arc<AggregationDescriptor>>` *cloned* but the aggregations vector is built once at registration time, so `cloned()` is just `Vec::clone()` of an already-small vector. For most apps this Vec has 1-3 entries.

### D-8 — Snapshot determinism preservation

`AggStateTable` switching from BTreeMap → HashMap loses iteration order. The **D-06 invariant** says the apply path must be deterministic given the input event stream. Iteration order matters for:

- **Snapshot writes** (Phase 7) — the snapshot serialises entities; if order varies between runs, byte-identical snapshots would differ
- **Debug routes** (`/dev/list_entities`, `/tables`) — observable iteration order

Fix: at snapshot write time, materialise a **sorted Vec<(EntityKey, Vec<AggOp>)>** by EntityKey before serialising. One-time O(N log N) cost on snapshot (cold path, infrequent). Hot path (per-push apply) gets HashMap O(1).

Add a method `AggStateTable::iter_sorted(&self) -> impl Iterator<Item = (&EntityKey, &Vec<AggOp>)>` that returns entries in EntityKey-sorted order. Snapshot + debug routes use this.

EntityKey already implements `Ord` (via `Vec<(String, Value)>` lexicographic order) — verify it still works after the SmallVec change.

### D-9 — Backward-compat at the API/wire layer

The wire protocol bytes are unchanged. WAL records v=1 (legacy) and v=2 (Plan 18-09) replay to the same Row shape; only the Row's *internal storage* differs. WAL replay → Row → apply_event_to_aggregations works through the new types.

Python SDK is unaffected — wire format unchanged.

All Phase 6.1, 7, 11.5, 18-04.6, 18-07, 18-09, 18-10 integration tests must pass unchanged.

## Risks & mitigations

- **CompactString cascade is large** — every `Value::Str(s)` pattern arm in the codebase needs review. Some sites consumed `s` as owned String. Mitigation: clippy will surface incompatibilities; fix mechanically. Estimated 30-50 sites.
- **`hashbrown::HashMap::raw_entry_mut` API surface** — verify the exact method signature in hashbrown 0.15. If `from_key` needs a `Q: Borrow<K>` constraint, EntityKey already satisfies via `Borrow<EntityKey>`. If the API differs, fall back to `entry()` and accept the `key.clone()` cost — still ~100 ns vs the 2,147 ns BTreeMap walk, so even the fallback is a win.
- **Snapshot format change** — if the snapshot encodes Row contents byte-for-byte, the SmallVec storage change might alter the encoding. Mitigation: Row's Serialize impl should produce the same wire format as before (ordered map of field name → value). If it doesn't, write a custom Serialize impl that produces the historical encoding.
- **EntityKey ordering for snapshot** — old BTreeMap implicitly sorted by EntityKey lex order. New iter_sorted must produce the same order so snapshot bytes don't drift. Verify with a round-trip determinism test (Task 11.9.a covers this).
- **Plan size** — 12 tasks, ~1,200-1,500 LoC. Risk of mid-execution checkpoint or context exhaustion. Mitigation: each task is independently verifiable; if executor hits a blocker, plan supports incremental landing.
