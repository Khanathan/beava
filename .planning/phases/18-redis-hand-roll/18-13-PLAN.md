# Plan 18-13 — SPSC ring between IoPool and apply (closes the gap on macOS)

## Context

Plan 18-04.7 wired the IoPool worker thread to publish parsed events to the apply thread via a per-tick `Vec<WireRequest>` + `Vec<Option<Row>>` side-channel on `MioClient`. Each tick:

1. IoPool polls mio (kqueue on macOS, epoll on Linux)
2. IoPool parses N envelopes + body→Row deserialize
3. IoPool fills the `parsed_requests` + `parsed_rows` Vecs
4. `tokio::join_all` spin barrier → apply phase begins
5. Apply thread drains the batch
6. `join_all` again → IoPool encodes + writes responses
7. Loop

The `join_all` spin barriers + batch-boundary semantics produce a measurable per-event gap on macOS:

- 98.8% of events: gap < 1 µs (apply running back-to-back within a single tick)
- 1.2% of events: gap = ~218 µs **stall every ~128 events** (waiting for next IoPool tick to publish)
- Amortized: ~1.4-2 µs/event of gap at p=16/pd=256

For the small workload (1.04 µs apply work + 1.4 µs gap = 2.44 µs cycle = 410k EPS), the gap is **134% of the apply work itself**. The apply thread is gap-bound, not work-bound.

This plan replaces the batched Vec-publish + `join_all` with **per-IoPool-worker SPSC ring buffers**. IoPool pushes events one-by-one as they parse; apply consumes one-by-one as the rings fill. No batch boundary, no spin barrier — events flow continuously.

The architecture remains compatible with the io_uring backend (Plan 18-05): the SPSC rings sit between the IoPool worker thread and the apply thread, regardless of which kernel poll primitive the IoPool uses.

## Architecture decisions

### D-1 — SPSC ring primitive: `rtrb` (production-quality lock-free SPSC)

`rtrb` is an audio-grade single-producer single-consumer ring buffer:
- Cache-line-padded heads/tails (no false sharing)
- 10-30 ns producer.push / consumer.pop overhead
- Bounded capacity (we'll use `2 * pdepth_max = 8192` to absorb burst)
- Drops the `Send` bound in a clean way (split into producer + consumer halves)

Alternative considered: `crossbeam-channel::bounded` — easier to use but ~80 ns send/recv overhead. `tokio::sync::mpsc` — async-friendly but unnecessary (we're sync between threads). `rtrb` is the right primitive for SPSC at our event rates.

### D-2 — One ring pair per IoPool worker → apply

The current architecture has multiple IoPool worker threads (default `max(2, num_cpus/4)`). Each worker owns a subset of clients. We give each worker:
- **Inbound ring** (worker → apply): `rtrb::Producer<WorkItem>`
- **Outbound ring** (apply → worker): `rtrb::Producer<ResponseItem>`

The apply thread polls all N inbound rings round-robin; when nothing's available across all rings, it briefly yields (`std::thread::yield_now()` then back to poll). This avoids futex/condvar for the hot path while still backing off when truly idle.

```rust
struct WorkItem {
    request: WireRequest,
    parsed_row: Option<Row>,
    client_handle: u64, // identifies which IoPool worker to send response to
}
struct ResponseItem {
    response: GlueResponse,
    client_handle: u64,
}
```

### D-3 — Apply-thread main loop

```rust
fn apply_loop(rings: Vec<rtrb::Consumer<WorkItem>>, response_rings: Vec<rtrb::Producer<ResponseItem>>) {
    let mut idle_spins: u32 = 0;
    loop {
        let mut did_work = false;
        for (i, ring) in rings.iter_mut().enumerate() {
            while let Ok(item) = ring.pop() {
                let resp = dispatch_one(item.request, item.parsed_row);
                response_rings[i].push(ResponseItem { response: resp, ... }).ok();
                did_work = true;
            }
        }
        if did_work {
            idle_spins = 0;
        } else {
            idle_spins = idle_spins.saturating_add(1);
            if idle_spins < 100 {
                std::hint::spin_loop();
            } else if idle_spins < 1000 {
                std::thread::yield_now();
            } else {
                std::thread::sleep(std::time::Duration::from_micros(10));
            }
        }
        if shutdown.load(Ordering::Relaxed) { break; }
    }
}
```

The progressive backoff (spin → yield → sleep 10µs) keeps latency low at high load (spin/yield) and frees CPU at low load (sleep).

### D-4 — IoPool worker pushes to ring instead of returning batched Vec

Each IoPool worker processes its socket events synchronously:

```rust
fn io_worker_loop(producer: rtrb::Producer<WorkItem>, ...) {
    let mut events = Vec::with_capacity(64);
    loop {
        backend.poll(timeout, &mut events)?;
        for event in &events {
            match event.kind {
                IoEventKind::ClientReadable => {
                    // Read from socket, parse envelope + body→Row
                    let item = parse_one(...);
                    producer.push(item).ok();
                    // No batch barrier — apply thread sees the event immediately
                }
                IoEventKind::ClientWritable => { drain outbound ring + write socket; }
            }
        }
    }
}
```

### D-5 — Fallback when ring is full (backpressure)

If `producer.push` returns `Err(Full)`, the IoPool worker briefly waits (yields once) and retries. At our throughputs (~400k EPS) and ring capacity (8192), this should never happen except under pathological load.

### D-6 — Backward compat: keep `MioClient.parsed_requests` + `parsed_rows` for legacy tests

The two integration tests `phase18_04_6_integration_test.rs` and `phase18_04_8_body_on_iopool_test.rs` use the existing Vec-based publish API. We keep the old API as a back-door for those tests (they don't exercise the SPSC path), and add the SPSC primitives alongside. New code paths use SPSC; legacy tests stay green without modification.

## Risks

- **rtrb buffer-lifetime gotchas**: rtrb uses `&mut self` on producer/consumer. We must hold the producer in the IoPool worker thread, the consumer in the apply thread. No cross-thread sharing of either half. Standard SPSC pattern.
- **Spin-loop CPU usage at low load**: progressive backoff (spin → yield → sleep 10µs) ensures we don't burn a core when idle. Worst case: 100 spin_loop iterations + 900 yields + sleep_us(10) = ~10 µs idle latency before the apply thread fully sleeps. Acceptable.
- **Per-event SPSC push overhead**: 10-30 ns per push. At 1M EPS = 10-30 ns/event = 1-3% overhead. Net gain (closing the 1.4-2 µs gap) is far larger.
- **Order preservation across multiple IoPool workers**: each connection is owned by ONE worker. Within a connection, FIFO is preserved. Across connections, ordering is undefined (and was undefined before this change too). No regression.
