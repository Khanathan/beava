# Plan 18-04.7 — IoPool wiring into serve_with_dirs

## Context (read first)

Plans 18-03 + 18-04 built the IoPool infrastructure:
- `IoPool::new(n)` — N `std::thread` workers with per-slot `AtomicUsize` spin-barrier (Release/Acquire ordering, 3-tier idle backoff)
- `IoPool::publish(items: Vec<WorkItem>)` — round-robin distribution
- `IoPool::join_all()` — Acquire barrier waits for all current items to complete
- `parse_client_from_buf` (read-phase work item)
- `serialize_into` (write-phase work item, BytesMut BufMut zero-alloc encoding)
- `Client::output_queue` / `write_buf` / `write_offset` — partial-write resume support
- `Client::pending_parse_input` / `parsed_requests` / `parse_error` — read coordination slots
- `WireResponse` enum (TcpAck / TcpError / HttpStatus / HttpStaticOk)

Plan 18-04 included `test_per_tick_lifecycle_read_apply_write` which **already verifies the per-tick lifecycle works in isolation**: 4-thread IoPool, 16 mock clients, recorded phase log `["read_dist","read_join","apply","write_dist","write_join"]`.

What's MISSING: `ServerV18::serve_with_dirs` doesn't call any of this. The 18-04.6 SUMMARY explicitly noted: *"D-4: 1 apply thread (mio) + WalWriter + 1 tokio (admin only); IoPool not yet layered."*

Today, the apply thread does parse + dispatch + encode inline per mio event. Per BEAVA_TRACE_SRV_TIMING measurements:
- parse stage: ~303-911 ns (post-18-10 / pre-18-11)
- encode stage: ~209-462 ns
- Combined: ~500-1,400 ns per event ON THE APPLY THREAD

Moving these stages off the apply thread:
- Apply-thread per-event work drops by ~500-700 ns
- Parse and encode run **concurrently with the previous event's apply** (per-tick batch), not serially
- Single-thread theoretical EPS: lifts proportionally
- Multi-client (parallel=N) throughput: lifts more, because parse/encode for events 2..N happen while event 1's apply runs

## Architecture decisions

### D-1 — Per-tick batch lifecycle in serve_with_dirs

Replace the current per-event inline pipeline with a per-tick batch pipeline:

```rust
loop {
    // Phase 1: poll mio events
    poll.poll(&mut events, Some(Duration::from_millis(0)))?;

    // Phase 2: classify events into read-ready vs write-ready clients
    let mut read_clients = Vec::new();
    let mut write_clients = Vec::new();
    for event in &events {
        match event.token() {
            tcp_listener_token => accept_new_clients(&mut clients, &tcp_listener)?,
            http_listener_token => accept_new_clients(&mut clients, &http_listener)?,
            client_token => {
                let client_idx = client_token_to_idx(token);
                if event.is_readable() { read_clients.push(client_idx); }
                if event.is_writable() && clients[client_idx].has_write_work() { write_clients.push(client_idx); }
            }
        }
    }

    // Phase 3: read distribution → IoPool workers do socket.read + parse
    if !read_clients.is_empty() {
        let read_items: Vec<WorkItem> = read_clients.iter().map(|&idx| {
            let client_ptr = clients.as_mut_ptr();  // safe: IoPool waits at join_all before apply touches clients
            Box::new(move || {
                let client = unsafe { &mut *client_ptr.add(idx) };
                read_and_parse_client(client);
            }) as WorkItem
        }).collect();
        io_pool.publish(read_items);
        io_pool.join_all();  // Acquire barrier
    }

    // Phase 4: apply (single-threaded, on this thread)
    for &idx in &read_clients {
        let client = &mut clients[idx];
        for parsed in client.drain_parsed_requests() {
            let responses = apply_shard.dispatch_wire_request_sync(parsed);
            for resp in responses {
                client.enqueue_response(resp);
            }
        }
    }

    // Phase 5: write distribution → IoPool workers do serialize + socket.write
    let writable_clients: Vec<usize> = read_clients.iter().chain(write_clients.iter())
        .filter(|&&idx| clients[idx].has_write_work())
        .copied().collect();
    if !writable_clients.is_empty() {
        let write_items: Vec<WorkItem> = writable_clients.iter().map(|&idx| {
            let client_ptr = clients.as_mut_ptr();
            Box::new(move || {
                let client = unsafe { &mut *client_ptr.add(idx) };
                serialize_and_write_client(client);
            }) as WorkItem
        }).collect();
        io_pool.publish(write_items);
        io_pool.join_all();
    }
}
```

Key invariants:
- **`io_pool.join_all()` is the synchronization point.** Apply thread is guaranteed to see all parse results when join_all returns (Release/Acquire barrier).
- **Apply thread NEVER touches a client's read_buf or write_buf while IoPool is active.** The two phases are strictly serialized — read_phase ends with join_all, then apply runs, then write_phase starts.
- **Clients vec must not be re-allocated during a tick** (IoPool workers hold raw pointers). Pre-allocate to capacity at server startup; never resize within a tick.

### D-2 — Configurable IoPool thread count

```rust
fn default_io_threads() -> usize {
    std::env::var("BEAVA_IO_THREADS")
        .ok()
        .and_then(|s| s.parse().ok())
        .unwrap_or_else(|| std::cmp::max(2, num_cpus::get() / 4))
}
```

Default ratio: 1 IoPool thread per 4 cores (conservative — IoPool threads spin briefly, don't burn full cores). Operator can override via env var.

For M4 (10 cores): default = 2 IoPool threads.
For Linux Xeon 16-core: default = 4 IoPool threads.

### D-3 — Single mio Poll vs per-IoPool-thread Poll

Today the apply thread runs the single `mio::Poll`. After this plan, the **apply thread keeps the only mio::Poll** — IoPool workers just run socket syscalls (`read`/`write`) on file descriptors handed to them; they don't poll. This matches Redis 6 I/O threads architecture.

Plan 18-05 (io_uring HARD GATE on Linux) will move to per-IoPool-thread `io_uring` for batched async I/O. That's a separate plan.

### D-4 — No protocol-specific bypass

Both HTTP and TCP read+parse work items go through the IoPool. The HTTP frame parser (`parse_http_request`) and the TCP wire parser (`parse_wire_request`) are both safe to call from arbitrary threads on a `&mut Client`. Mixed HTTP+TCP traffic at parallel=N benefits equally.

### D-5 — Backpressure / overflow handling

If the IoPool is saturated (publish would exceed slot capacity), the apply thread waits at the next `join_all()` for current work to drain. Do NOT spawn additional threads under load — the spin-barrier design assumes a fixed worker count. If sustained load exceeds N IoPool threads' capacity, the queue depth (mio events backing up) is the natural backpressure signal; the kernel TCP buffer absorbs spike traffic.

## Risks & mitigations

- **`Vec<Client>` mutation across IoPool workers via raw pointers** — UB if Vec is resized mid-tick. Mitigation: pre-allocate `Vec::with_capacity(MAX_CONCURRENT_CLIENTS)` at startup; track `Client` slots by index; never call `Vec::push` outside the apply phase. Use `Vec::set_len` discipline if needed.
- **Read+parse error propagation** — if a client's frame is malformed, the IoPool worker writes to `client.parse_error`; apply phase reads this and emits an error response. Mitigation: existing `parse_error` slot in Client (Plan 18-03) handles this. Test surfaces error path correctness.
- **IoPool slot exhaustion** — if `read_clients.len() > io_pool.slot_count()`, `publish` returns failure or blocks. Mitigation: spec says round-robin with N items into M slots works for any N (workers loop; one worker handles multiple items). Verify via test with parallel=64 (64 clients into 2 IoPool threads).
- **Worker-thread-local TLS confusion** — `parse_client_from_buf` is a free function that doesn't depend on TLS, but if any inner code uses thread-locals, results may differ when called from IoPool thread vs apply thread. Mitigation: strict integration test ensures parse/encode return identical Row contents when called from arbitrary thread.
- **Coordination with running Plan 18-11** — 18-11 is in flight in a separate worktree. Both base off `409675f`. This plan's strict file scope (D-7) avoids 18-11's territory. If a merge conflict still arises, resolve manually at merge time.
- **Plan 18-11 hasn't landed yet** — if Plan 18-11 fails or rolls back, this plan's measurements (EPS targets) reference the post-18-11 baseline. Mitigation: targets are stated as relative lifts; absolute numbers depend on 18-11's actual delivery.
