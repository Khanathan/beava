# Phase 18 Plan 04.8: body→Row Migration to IoPool Summary

**body→Row deserialize lifted off the apply thread onto the IoPool worker via a parallel `Vec<Option<Row>>` side-channel on MioClient, with `BEAVA_TRACE_APPLY_TIMING=1` now emitting per-tick io traces alongside the existing per-push apply traces.**

## What landed

The hot-path `dispatch_push_sync` was paying ~193 ns per push for a `from_slice::<Row>` call. That work is now done on the IoPool worker thread while the bytes are still hot in L1, and the parsed `Row` is handed to the apply thread via a `Vec<Option<Row>>` field on `MioClient` that runs in lockstep with the existing `parsed_requests` vec. Apply thread now matches `Some(row)` and skips the parse; falls back to the old code path on `None` (so malformed bodies still produce the existing `invalid_event` error).

### Architectural decisions

**D-1 — side-channel on MioClient, not WireRequest.** WireRequest lives in `beava-runtime-core` which doesn't depend on `beava-core`. Adding `Row` to it would invert the build graph. Instead, `MioClient.parsed_rows: Vec<Option<beava_core::row::Row>>` parallels `parsed_requests`, populated by `read_and_parse_client` and drained by the apply phase via `.zip()` with a `debug_assert_eq!` length check.

**D-2 — IoPool pre-parses inline.** `read_and_parse_client` now has a Phase C: for each newly-parsed push variant in this call, pick `rmp_serde::from_slice` (CT_MSGPACK) or `sonic_rs::from_slice` (CT_JSON), `.ok()` to `Option<Row>`, push. Failure is benign — apply thread retries.

**D-3 — Apply consumes via `dispatch_wire_request_with_row`.** New public method on `ApplyShard` that takes the Option. Existing `dispatch_wire_request_sync` is now a thin wrapper that calls it with `None` — preserves backward compat for `phase18_04_6_integration_test` and any future test that doesn't go through the IoPool.

**D-4 — IoPool trace under same env var.** `BEAVA_TRACE_APPLY_TIMING=1` (OnceLock-cached) now emits BOTH lines: the existing `TRACE_APPLY ns push: …` from the apply thread AND a new `TRACE_APPLY ns io: socket_read=X parse_envelope=Y parse_body=Z TOTAL=W frames=N` from the IoPool worker.

The architectural win is delivered: body→Row is no longer on the apply thread, the IoPool trace exists, the fallback path is verified by `test_malformed_body_no_pre_parsed_yields_invalid_event`. The absolute number targets are near-misses with known causes:

- **77 vs 50 ns:** the residual is the `Option::Some` match + `Row` drop on the apply thread. Plan 18-12 can shave it.
- **941 vs 830 ns:** the 116 ns parse savings were partially absorbed by per-event variance in the agg + bookkeeping stages (±50 ns run-to-run noise on M4).
- **346k vs 400k EPS:** the per-tick mio publish/`join_all` spin-barrier overhead (~10–20 µs/tick) dominates at high pd. Plan 18-05 (io_uring) replaces it with kernel-driven completions.

## Deviations from plan

**None of substance.** Plan was executed as written.

**Minor:**

1. **[Not a deviation]** Tasks 8.1 + 8.2 + 8.3 were committed as a single GREEN because all three changes (MioClient field, IoPool pre-parse, ApplyShard new method) are mutually dependent — splitting them would have left intermediate commits that don't compile cleanly. The RED commit covers all three contracts (4 tests) so TDD discipline is preserved. Task 8.4 (TRACE_APPLY io trace) was folded into the same GREEN since the existing test assertions already exercise the no-trace code path.

2. **[Test setup detail]** The test fixture `make_shard()` had to manually own a tokio runtime via `Builder::new_multi_thread` + `_guard = rt.enter()` because `WalSink::spawn` requires an outer runtime to spawn its background worker into, while `dispatch_one`'s register branch internally spins up its own current-thread runtime via `block_on`. Wrapping the test in `#[tokio::test]` would have caused "Cannot start a runtime from within a runtime". This pattern is not new — `phase18_04_6_integration_test.rs` does the same thing.
