# Plan 18-16 — `state_tables` from BTreeMap to Vec-by-agg_id

## Context

`state_tables: Arc<Mutex<BTreeMap<String, AggStateTable>>>` is in the apply hot path. Every push event does:
```rust
let table = state_tables.entry(desc.node_name.clone()).or_default();
```
which costs ~30-70 ns per lookup (3-7 string compares + tree walk). For a 15-aggregation pipeline at 500k EPS:
- 15 × 50 ns × 500k = 375 ms/sec = **37% of one core** spent in BTreeMap lookups

Plan 18-11 D-7 already converted the **inner** AggStateTable's `entities` map to `hashbrown::HashMap+FxBuildHasher`. The **outer** `state_tables` map remained BTreeMap. This plan completes the optimization by making the outer lookup O(1) array index (~1-2 ns per lookup), saving ~36% of one core on complex pipelines.

The Mutex stays — Path A from the architecture walkthrough preserves G1 (per-event multi-feature atomicity) and G2 (cross-aggregation atomicity).

## Architecture

### Stable agg_id assignment

`AggregationDescriptor` gains `pub agg_id: u32`. The registry's `RegistryInner` carries `next_agg_id: u32` (monotonic counter). At registration, each new aggregation gets `agg_id = next_agg_id; next_agg_id += 1;`. IDs are stable for process lifetime. Beava's additive-only registration means IDs never get reassigned.

### Vec-by-id state storage

`state_tables` becomes `Arc<Mutex<Vec<AggStateTable>>>` of size MAX_AGGREGATIONS=256 (pre-allocated at server init via `Vec::resize_with(256, AggStateTable::default)`). Apply indexes by `desc.agg_id as usize`; readers do the same. Vec never resizes at runtime (256 is plenty for v0; future could push higher if needed).

### Snapshot interaction

Snapshot writer iterates the Vec, pairs each populated entry with its name (via `registry.find_name_by_agg_id(id)`). Cold path — sort cost negligible.

Snapshot reader replays registrations in order via WAL replay; registry assigns IDs deterministically (same monotonic order). Then for each (name, state) entry from snapshot, locates the now-assigned ID via `registry.compiled_aggregation_id(name)` and writes `state_tables[id] = state`.

### What stays the same (preserves all guarantees)

- `Mutex<...>` wrapping the Vec — same lock semantics; G1 + G2 hold
- `AggStateTable` internals (per-entity HashMap with hashbrown+FxBuildHasher) — Plan 18-11 unchanged
- ApplyShard, WAL, snapshot, recovery flow — same shape; only the lookup primitive changes
- Public API (`POST /get`, `app.get(...)`, etc.) — unchanged; internal optimization only

## Risks

- **Snapshot/recovery determinism**: WAL replay must reconstruct registrations in same order to ensure agg_ids match the snapshot's expected order. Beava's WAL is single-writer + append-only, so replay order is deterministic. Mitigation: explicit test in 16.3.

- **Vec pre-allocation memory**: 256 × sizeof(AggStateTable) ≈ 256 × few KB = ~1 MB at startup. Trivial.

- **MAX_AGGREGATIONS=256 cap**: If a user registers >256 aggregations, the Vec index is OOB. Defensive: assert in `apply_registration` and return a 4xx if exceeded. Most users will have <50 aggregations; 256 is a comfortable headroom.

- **Hidden BTreeMap iteration order assumptions in tests**: Several tests likely iterate `state_tables` and assume key order. Need to update them to sort or to use explicit names.
