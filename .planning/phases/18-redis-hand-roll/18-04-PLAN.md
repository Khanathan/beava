# Plan 18-04 — I/O threads for writes

## Goal

Match Redis 6.0's full I/O threading pattern by extending the I/O thread pool
to also handle the write phase. After apply produces responses, the apply
thread enqueues raw `WireResponse` enums into each client's
`output_queue: VecDeque<WireResponse>`; the I/O threads then serialize each
enum to bytes and drain the kernel write buffer via `write_vectored()` in
parallel.

This closes the second-largest remaining cost after parse parallelisation:
the `serde_json::to_vec` + `write()` syscall on every response. With responses
serialized off the apply thread, the apply thread spends more wall-clock time
on actual apply work, lifting aggregate throughput meaningfully.

Reference: `18-redis-research.md` §"I/O threads write phase" +
`18-rust-translation.md` rows 11-12.

**4.1.b (GREEN):** Implement:

  - Replace `write_queue: VecDeque<Bytes>` with `output_queue: VecDeque<WireResponse>` (raw responses, not bytes).
  - Add `write_buf: BytesMut` (the staging buffer that I/O thread serializes into).
  - Add `write_offset: usize` for partial-write resume.
- `crates/beava-runtime-core/src/response.rs`:

- `crates/beava-runtime-core/src/io_thread.rs`:
  - Extend `WorkItem` enum: `Read(ClientRef) | Write(ClientRef)`.

  - `WouldBlock` is normal (kernel buffer full); leave bytes, return; main re-arms `Interest::WRITABLE` for that client.
- `crates/beava-runtime-core/src/event_loop.rs`:
  - Add `pub fn run_write_phase(&mut self, clients: &mut [Client])` exposed for test harnesses; production `tick()` calls it inline after the apply phase.

### Task 4.2 — Per-tick lifecycle: poll → reads → apply → writes → join (~70 LoC)

**4.2.a (RED):** Write `tests/io_threads_write_test.rs::test_per_tick_lifecycle_read_apply_write`
that:
- Constructs an end-to-end loop with 4 I/O threads + 16 mock clients
- Each client sends a `ping` frame, then immediately the next tick should:
  1. Read+parse the frames (parallel on I/O threads)
  2. Apply (serial on apply thread, produces 16 `WireResponse::TcpAck`)
  3. Write the responses (parallel on I/O threads)
- Asserts 16 acks delivered after exactly 1 tick (or 2 if partial-write path triggers, which it shouldn't on hermetic mocks)
- Asserts the order of phases via a shared `Vec<&'static str>` log: `["read_dist", "read_join", "apply", "write_dist", "write_join"]`

Run test → must FAIL. Commit:
`test(18-redis-hand-roll-18-04): per tick lifecycle read apply write phases`

**4.2.b (GREEN):** Refactor `EventLoop::tick`:
```text
1. poll.poll(&mut events, timeout)?
2. Walk events; mark readable + writable clients; collect ready_read, ready_write.
3. READ PHASE:
   - publish Read work items for ready_read to io_pool
   - join_all
   - drain parsed_requests into the apply input queue
4. APPLY PHASE (serial, on apply thread):
   - drain apply input queue, dispatch each to AppState
   - enqueue WireResponse into each client's output_queue
   - inline WAL append (Stage 18.2)
5. WRITE PHASE:
   - publish Write work items for clients with non-empty output_queue OR pending write_buf to io_pool
   - join_all
   - if any client has remaining bytes (kernel buffer full), re-arm Interest::WRITABLE
6. before_sleep:
   - flush WAL to kernel (Stage 18.2)
   - check time events (snapshot rotation, idle cache)
7. loop
```
- Both `READ PHASE` and `WRITE PHASE` reuse the same `IoPool` via the unified `WorkItem` enum from Task 4.1.
- If `IoConfig::io_threads == 0`: both phases run inline on apply thread (legacy path; matches Stage 18.2).

Run test → must PASS. Commit: `feat(18-redis-hand-roll-18-04): per-tick lifecycle with read+write i/o thread phases`

## Perf gate 4.2 (Apple-M4, INFORMATIONAL — tail latency)

Under steady 1M EPS load (16 clients * 64k EPS each * 1 minute):
- p50: ≤ 0.5ms
- p99: ≤ 5ms
- p999: ≤ 50ms (allowing for occasional GC-style pauses from `BytesMut::reserve` reallocs)

If p99 > 5ms: investigate. Likely culprits:
- I/O thread parker wake latency (lower the park timeout from 100µs → 50µs)
- Single client monopolizing apply thread (add per-client read budget per tick: max 16 frames)
- WAL fsync tick blocking apply (verify Stage 18.2 fsync runs on pthread, not apply thread)

## Risks & mitigations

- **`output_queue` is per-client `VecDeque<WireResponse>` — no shared lock**:
  ownership is exclusive-per-tick (Task 3.2 invariant extends here). Same
  `unsafe impl Send for ClientRef` justification applies.
- **`write_vectored` IoSlice lifetime**: the `IoSlice<'_>` borrows from
  `write_buf: BytesMut`. Since the work item runs synchronously inside the
  worker before the join, and main waits on the join before touching the
  client, the borrow is sound. Document in `// SAFETY:` block.
- **Serialization correctness regression**: existing `WireResponse` byte
  outputs are golden-tested in `crates/beava-server/tests/wire_format_test.rs`
  (Phase 2.5). Reuse those goldens in `tests/io_threads_write_test.rs` to
  catch any drift from inline → off-thread serialization paths.
- **Partial-write deadlock if mio never reports writable again**: kernel
  re-asserts EPOLLOUT/kqueue write event when buffer drains. mio relays
  faithfully. Backstop: if a client has pending writes for >100 ticks,
  log warning + close connection (treat as slow-client liveness issue).
- **HTTP keep-alive + write phase ordering**: keep-alive responses must be
  written in the exact order their requests were processed by apply.
  `output_queue: VecDeque` preserves FIFO order; serialization is FIFO too.
  Add explicit test in 4.2.
