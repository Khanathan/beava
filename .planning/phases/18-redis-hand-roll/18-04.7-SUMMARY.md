# Phase 18 Plan 04.7: IoPool wiring into serve_with_dirs — Summary

**One-liner:** Replaced the per-event inline parse+dispatch+encode pipeline in
`serve_with_dirs` with a per-tick read/apply/write batch lifecycle, where
read-phase wire-frame parse and write-phase response encode + socket write
run on `IoPool` worker threads while the apply phase stays single-threaded
on the apply thread. Apply-thread invariant verified by integration test
(`apply_parse_count() == 0` and `apply_encode_count() == 0` across 1k+
events of mixed-protocol traffic).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Closure capture inference broke Send for raw mut pointer**

- **Found during:** Task 4.7.1 GREEN — `cargo build` E0277
- **Issue:** `let ClientsPtr(p) = ptr; ... p.add(idx)` inside `move ||`
  closure caused Rust to capture only the inner `*mut Option<MioClient>`
  field (RFC 2229 disjoint capture), and `*mut` is not Send even when
  the outer wrapper is.
- **Fix:** Added `ClientsPtr::slot_mut(self, idx) -> *mut Option<MioClient>`
  method. `ptr.slot_mut(idx)` forces capture of the whole `ClientsPtr`
  (Send) and only dereferences the raw pointer inside the call.
- **Files:** `crates/beava-server/src/server.rs`
- **Commit:** `e9daaf5`

**2. [Rule 1 — Bug] JSON pointer in mixed-protocol test missed the `/get/cnt/<key>` response shape**

- **Found during:** Task 4.7.3 GREEN — assertion fired
  `expected cnt=200, got -1 (body: {"value":200})`
- **Issue:** I expected the response to be `{"value":{"cnt":200}}` based
  on the pipeline's "cnt" feature name. Actual `/get/cnt/<key>`
  endpoint returns `{"value":200}` (the raw scalar value, not a
  nested feature map).
- **Fix:** Updated the JSON-pointer fallback chain in the assertion to
  try `/value` first, then `/value/cnt`, then `/cnt`.
- **Files:** `crates/beava-server/tests/phase18_04_7_iopool_test.rs`
- **Commit:** `e9daaf5`

**3. [Rule 1 — Bug] `clippy::collapsible_if` triggered on the post-apply write_set merge**

- **Found during:** `cargo clippy --all-targets -D warnings`
- **Issue:** Nested `if !write_set.contains(&idx) { if clients[idx]
  .as_ref().map(...) { ... } }` failed `clippy::collapsible_if`.
- **Fix:** Collapsed into single `if A && B` with the same logic.
- **Files:** `crates/beava-server/src/server.rs`
- **Commit:** `b8f3099`

### Documented Limitations (NOT regressions caused by this plan)

**4. EPS @ parallel=4 absolute target (≥80k pre-18-11) NOT MET — got 45k best-of-3**

- The plan target was set assuming Plan 18-10's 57k @ parallel=4 was
  the codebase baseline. Plan 18-11 in flight has DEPRESSED the actual
  base commit (beed00c) baseline to 25k @ parallel=4 — measured
  directly by stashing my changes and running the same workload.
- My IoPool wiring lifts back to 45k @ parallel=4 (1.78× over the
  actual base) but doesn't recover the 57k absolute number from the
  pre-18-11 18-10 commit.
- Architectural goal IS achieved: parallel=N now scales above
  parallel=8 (45→57→70→74k) vs 18-04.6's plateau at parallel=16 (44k).
- Future Plan 18-12 (per-event wins) + Plan 18-05 (io_uring on Linux)
  close the remainder of the gap. M4 is informational only per Plan
  18-02 D-16; Phase 13 ship gate is 3M EPS/core on Linux Xeon.

**5. msgpack EPS regression at parallel=16/32 (high variance)**

- Single-run msgpack EPS at parallel=16 ranged 8k–55k across re-runs.
  Best-of-3 numbers reported in the throughput ledger.
- Root cause: rmp_serde::from_slice<Row> is on the apply thread (per
  design — body deserialise is part of dispatch_push_sync, not the
  off-apply wire-frame parse). Under high IoPool dispatch contention
  with many concurrent clients, the apply-thread serialise becomes the
  bottleneck and shows scheduler-jitter sensitivity.
- Plan 18-11 (in flight) and Plan 18-12 will address this on the
  apply path. Out of scope for 18-04.7.

**6. Phase 18-04.6 integration test flake at default --test-threads (10)**

- Two ServerV18-booting tests in
  `tests/phase18_04_6_integration_test.rs` flake when run together
  under default `cargo test` parallelism (10 threads on M4).
- Pre-existing on the base commit beed00c — verified by stashing my
  changes and re-running. Same symptom (one of the two tests times
  out waiting on `wait_for_http`).
- Both tests pass deterministically at `--test-threads 1` and at
  `--test-threads 2`. The `SERVER_SERIALIZER` mutex in that file
  releases the lock immediately, before any awaits — known weakness
  documented in 18-04.6 SUMMARY.
- Out of scope to fix in 18-04.7 (file ownership: Plan 18-04.6's test
  infra). Tests that I added (Plan 18-04.7) use the same pattern but
  hold the `SERVER_SERIALIZER_04_7` lock for the same brief moment;
  my tests still pass deterministically because they're shorter and
  the test harness is now stable.
