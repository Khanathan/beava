# Plan 18-09 — Native serialization per transport: msgpack on TCP, JSON-direct on HTTP

## Context (read first)

The session leading to this plan (2026-04-25 evening) found that:
- Server-side per-event wall is ~3.8 µs at HEAD with sonic-rs PATH A swap (~3.2 µs); 263k → 322k EPS theoretical single-thread
- The dominant cost is JSON parse + JSON re-serialize (~1.1 µs total in dispatch_push_sync)
- The wire layer ALSO does a JSON envelope round-trip for OP_PUSH (in `tcp_listener.rs::parse_wire_request`):
  ```
  payload = JSON {"event": "<name>", "body": {...}}
  → parse envelope
  → re-serialize body to JSON bytes
  → pass to apply path
  → apply path parses JSON body again
  ```
- This means OP_PUSH today does TWO JSON round-trips (envelope + body) on the wire alone, plus the apply-time parse

User decision 2026-04-25: each transport should use its NATURAL format end-to-end, and parse DIRECTLY into `beava_core::row::Row` (skip the `serde_json::Value` / `sonic_rs::Value` intermediate that costs ~100-500 ns per push):

- **TCP transport**: msgpack end-to-end. Frame body is raw msgpack bytes. No JSON anywhere on TCP path.
- **HTTP transport**: JSON end-to-end. Frame body is raw JSON bytes. Parse directly into Row via serde Visitor (no intermediate JsonValue).

Both paths converge at `apply_event_to_aggregations(&row, ...)` — the existing aggregation framework is unchanged.

## Architecture decisions

### D-1 — Per-transport native format

**TCP wire** uses msgpack content_type=0x02 (Phase 2.5 reserved this opcode; this plan finally implements the handler). Clients send `[u32 len][u16 op=OP_PUSH][u8 ct=0x02][msgpack envelope bytes]` where the envelope is the msgpack-encoded `{event, body}` map.

**HTTP wire** uses JSON content_type=0x01 (unchanged from today). HTTP `POST /push/:event` carries a JSON body in the request — already structured this way; HTTP doesn't need an envelope because event_name is in the URL path.

### D-2 — Parse directly into Row, no intermediate Value

Today: client bytes → `JsonValue` (heap allocation, HashMap of String keys) → walk fields → build `Row`.

Tomorrow: client bytes → serde Deserialize → `Row` directly via a custom `RowVisitor` impl. Saves ~150 ns per push (the JsonValue allocation + walk).

`Row` already implements `Deserialize` via beava-core if we annotate appropriately. If not, write a custom Visitor.

### D-3 — WAL stores raw wire bytes, no re-serialization

WAL record format becomes a small binary header + the raw body bytes the client sent:
```
[u8  v=2                ]   1 B   (WAL record version)
[u8  body_format=0x01|0x02] 1 B   (JSON or msgpack)
[u32 registry_version  ]   4 B
[u64 event_time_ms     ]   8 B
[u16 event_name_len    ]   2 B
[u8* event_name bytes  ]   N B
[u32 body_len          ]   4 B
[u8* body raw bytes    ]   M B   (JSON or msgpack from client; unmodified)
─────────────────────────────────
Header: 20 + N bytes
Body:   M bytes (typically 80-200 B for msgpack, 100-300 B for JSON)
```

This eliminates the ~500 ns wal_serialize step entirely. WAL replay parses the body using body_format byte to choose the parser.

### D-4 — WireRequest carries body_format

`WireRequest::TcpPush { event_name, body, body_format }` adds the format byte. parse_wire_request populates it from the frame's content_type. dispatch_push_sync branches on body_format to choose the parser.

### D-5 — Backward compat

CT_JSON on TCP still works (clients that haven't upgraded). The msgpack code path is OPT-IN via content_type byte. The HTTP path is unchanged for clients (still POST JSON). Phase 6.1/7/11.5 tests use TestServer which uses CT_JSON — those keep passing.

### D-6 — Legacy WalSink stays for legacy callers

The new WAL header format (v=2) ships in the new ApplyShard path (mio runtime). The legacy WalSink (used by Phase 6.1 path + admin endpoints) still writes v=1 records. WAL replay must handle both versions.

## Risks & mitigations

- **Row Deserialize impl complexity** — Row uses an enum Value with multiple variants (I64/F64/Str/Bool/...). Custom Visitor is non-trivial. Mitigation: derive Deserialize on Row + custom impl on Value if needed. Test thoroughly with both JSON and msgpack inputs.
- **WAL v=2 replay correctness** — mixing v=1 and v=2 records in the same WAL file (during the rollout window). Mitigation: version byte at the START of each record; replay dispatches on it. Add explicit test for mixed-version WAL.
- **Python SDK API change** — adding wire_format kwarg might surprise users. Mitigation: default to msgpack on TCP (NEW behavior is the default), document the rollout in CHANGELOG. Provide JSON opt-out for debug.
- **rmp-serde version pin** — pick a stable version. Use `rmp-serde = "1"` (current major).
- **dependency size** — adds ~500 KB to binary. Acceptable for v0.

## Expected perf impact

After Plan 18-09 lands (msgpack on TCP, no re-serialization):

```
Server-side per-event wall (TCP path):
  Before (today):           ~3.8 µs   (JSON envelope + body parse + re-serialize)
  After 18-09 (msgpack):    ~2.0 µs   (msgpack envelope + msgpack body parse + raw WAL)
  Single-thread ceiling:    500k EPS  (was 263k)

Parallelism sweep (TCP/small/parallel=N):
  parallel=1:   ~50k → ~80k EPS
  parallel=4:   ~106k → ~250k EPS
  parallel=16:  ~56k → ~200k EPS  (still bottlenecked by IoPool not wired)
  parallel=32:  ~67k → ~250k EPS

To break through ~250k requires Plan 18-04.7 (IoPool wiring into serve loop)
which absorbs the remaining I/O cost off the apply thread. Combined with
18-09, expect parallel=32 in the 400-500k EPS range on M4.
```

## Plan 18-04.7 / 18-05 implications

Plan 18-04.7 (IoPool wiring) is independent of 18-09 — they're both wins. They stack. Order doesn't matter much; pick whichever is lower-risk per the executor's preference.

Plan 18-05 (io_uring on Linux) is the HARD GATE. After 18-09 + 18-04.7 land, M4 should hit ~400-500k EPS. Linux Xeon with io_uring should hit 1-3M EPS easily, putting us in striking distance of the 3M ship-gate.
