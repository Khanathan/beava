# Plan 18-04.8 — body→Row migration to IoPool + IoPool runtime timing

## Context

Plans 18-04.7 (IoPool wiring) + 18-11 (per-event optimization) merged onto v2/greenfield. Current TRACE_APPLY breakdown at parallel=4/pd=64 (post-merge, env-cached):

```
parse        193 ns   ← body→Row deserialize ON APPLY THREAD (rmp_serde/sonic_rs from_slice)
lookup        31 ns
validate      32 ns
wal_build     33 ns
wal_append    43 ns
agg          473 ns
bookkeeping  169 ns
TOTAL        974 ns
```

The "parse" stage (193 ns) is body bytes → Row, called inside `dispatch_push_sync` on the apply thread. Plan 18-04.7 moved envelope parsing to IoPool but body→Row stayed on apply because Row is a beava-core type and the migration path wasn't trivially obvious. This plan does it.

Same pattern as the parse_wire_request migration: do work on the IoPool worker thread, hand pre-processed result to the apply thread, apply thread skips redundant work.

## Architecture decisions

### D-1 — Where to store the pre-parsed Row

`WireRequest` lives in beava-runtime-core which does not depend on beava-core. Adding `Row` to `WireRequest` would create a cross-crate dependency and reorder the build graph. Avoid that.

Instead, add a parallel side-channel on `MioClient` (in beava-server/src/server.rs):
```rust
struct MioClient {
    ...existing fields...
    parsed_requests: Vec<(WireRequest, bool)>,
    parsed_rows: Vec<Option<beava_core::row::Row>>,  // NEW: parallel to parsed_requests
}
```

`parsed_rows[i]` holds the Row corresponding to `parsed_requests[i]` (if it's a TcpPush/HttpPush and IoPool worker succeeded in parsing it). For non-push variants (Ping, Register, GetSingle, etc.), `parsed_rows[i]` is None.

### D-2 — IoPool worker pre-parses

In `read_and_parse_client` (the IoPool worker function), after `parse_wire_request` / `parse_http_request` returns a `WireRequest`, inspect it. If it's `TcpPush { body, body_format, .. }` or `HttpPush { body, body_format, .. }` or `HttpPushSync` or `HttpPushBatch`, attempt body→Row:

```rust
let row_opt: Option<Row> = if body_format == CT_MSGPACK {
    rmp_serde::from_slice::<Row>(&body).ok()
} else {
    sonic_rs::from_slice::<Row>(&body).ok()
};
```

Push to both vecs:
```rust
client.parsed_requests.push((req, keep_alive));
client.parsed_rows.push(row_opt);
```

The Option handles malformed bodies — if pre-parse fails, fall through to apply thread, which will retry and emit `invalid_event` per existing error path.

### D-3 — Apply thread consumes pre-parsed Row

In `dispatch_wire_request_sync` / `dispatch_one`, the push-handling branches need to receive the pre-parsed Row. Cleanest path: add a new method on `ApplyShard` that takes both the WireRequest and the pre-parsed Row option:

```rust
pub fn dispatch_wire_request_with_row(
    &self,
    req: WireRequest,
    pre_parsed_row: Option<Row>,
) -> Vec<GlueResponse> {
    // For push variants, use pre_parsed_row if Some; else parse from body inside dispatch_push_sync
}
```

In `dispatch_push_sync`:
```rust
fn dispatch_push_sync(
    &self,
    event_name: &str,
    body: Bytes,
    body_format: u8,
    pre_parsed_row: Option<Row>,
) -> GlueResponse {
    let row: Row = match pre_parsed_row {
        Some(r) => r,
        None => {
            // Fallback: pre-parse failed on IoPool, retry here
            if body_format == CT_MSGPACK {
                rmp_serde::from_slice::<Row>(&body)?
            } else {
                sonic_rs::from_slice::<Row>(&body)?
            }
        }
    };
    // ... rest unchanged
}
```

Apply phase in serve_with_dirs zips the two vecs together when calling dispatch:
```rust
for ((req, keep_alive), row_opt) in client.parsed_requests.drain(..).zip(client.parsed_rows.drain(..)) {
    let resps = apply_shard.dispatch_wire_request_with_row(req, row_opt);
    ...
}
```

### D-4 — IoPool timing trace under BEAVA_TRACE_APPLY_TIMING (same env var)

Add timing instrumentation in `read_and_parse_client`:

```rust
fn read_and_parse_client(client: &mut MioClient) {
    fn trace_apply_enabled() -> bool {
        use std::sync::OnceLock;
        static FLAG: OnceLock<bool> = OnceLock::new();
        *FLAG.get_or_init(|| {
            std::env::var("BEAVA_TRACE_APPLY_TIMING").ok().as_deref() == Some("1")
        })
    }
    let trace = trace_apply_enabled();
    let t0 = if trace { Some(std::time::Instant::now()) } else { None };

    // ... existing socket.read loop ...
    let t_read = t0.map(|t| t.elapsed());

    // ... existing parse_wire_request loop, but capture frame count ...
    // ... AND body→Row pre-parse for push frames ...
    let t_envelope = t0.map(|t| t.elapsed());
    // (separate t_body for body→Row work specifically)
    let t_body = t0.map(|t| t.elapsed());

    if let (Some(t0_inst), Some(read), Some(env), Some(body)) = (t0, t_read, t_envelope, t_body) {
        let total = t0_inst.elapsed();
        eprintln!(
            "TRACE_APPLY ns io: socket_read={} parse_envelope={} parse_body={} TOTAL={} frames={}",
            read.as_nanos(),
            (env - read).as_nanos(),
            (body - env).as_nanos(),
            total.as_nanos(),
            frame_count
        );
    }
}
```

`frame_count` is the number of frames parsed in this call (could be 0 if no complete frame, 64 if pipelined batch arrived).

## Risks & mitigations

- **`Vec<Option<Row>>` allocation per push** — Row is now constructed in IoPool and Option-wrapped. Memory overhead per push: small (Row is SmallVec inline ≤8 fields, mostly stack). No mitigation needed.
- **Apply thread fallback path complexity** — if IoPool pre-parse fails AND apply thread also fails, two error paths. Mitigation: explicitly test both paths.
- **Cross-crate type usage** — MioClient now references `beava_core::row::Row`. Already true (server.rs imports beava-core in many places). No new dep.
- **Strict order — drain parsed_rows in lockstep with parsed_requests** — using `.zip()` enforces this. If lengths drift by bug, panic immediately on assert.
