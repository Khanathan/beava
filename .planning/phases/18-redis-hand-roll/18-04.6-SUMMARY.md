# Phase 18 Plan 04.6: Real EventLoop Integration into ServerV18::serve() Summary

Replaced the tokio/axum shim (234 EPS, slower than legacy) with a real mio event loop on a dedicated `std::thread`. The new serve loop uses `ApplyShard::dispatch_wire_request_sync` called inline per tick, `WalBufferRing::append` on the hot path, and keeps the tokio admin server unchanged. First measurement: **44k EPS TCP/small** — a 190x improvement over the shim baseline.

## Decisions Made

**D-1 (AppState ownership):** Kept `Arc<Mutex<AppState>>` with the apply thread as the single writer. The Mutex is uncontended so lock+unlock is the ~10-20ns fast path. Phase 13.3 will remove it via lockless-apply refactor.

**D-2 (No async bridge on data plane):** Deleted the `tokio::task::spawn_blocking` + `WalSink::append_event_with_mode` path from serve(). Apply path is purely synchronous: parse → lock → update state → WalBufferRing::append → unlock → encode response.

**D-3 (Keep async API, add sync):** Existing `dispatch_wire_request` async fn kept for backward compat (tests, admin). New `dispatch_wire_request_sync` added to `ApplyShard` and to `runtime_core_glue` for the hot path.

**D-4 (Threading model):** 1 apply thread (mio Poll + EventLoop::tick inline apply) + 1 WalWriter thread + 1 tokio runtime (admin only). IoPool parallelism from Plans 18-03/18-04 not yet layered on top — deferred to a follow-up plan.

**D-5 (Bench API unchanged):** `beava-bench-v18` calls `serve_with_dirs` — same name, new implementation. No bench-side changes needed.

## Performance Results (Task 4.6.6)

Harness: `beava-bench-v18`. Date: 2026-04-25. hw-class: Darwin-24.3.0 / 10 cores (Apple M4).

| Transport | Parallel | EPS | p50 µs | p99 µs | vs 18-05.1 shim |
|-----------|----------|-----|--------|--------|-----------------|
| TCP | 16 | 44,379 | 137 | 4,723 | 190x improvement |
| TCP | 32 | 60,033 | 242 | 4,855 | 257x improvement |
| HTTP | 16 | 43,642 | 146 | 3,913 | 197x improvement |

**vs plan target (50k EPS TCP/parallel=16):** 44,379 = 11% below the 50k sanity floor. This is a WARNING (10-25% range), not a BLOCKER. The 50k target was an "M4 informational sanity floor" — the real Plan 18-02 target is 300-500k EPS/core, which requires IoPool parallelism (Plans 18-03/18-04) layered on top. Current bottleneck: single apply thread + AppState Mutex per event on the hot path.

**Root cause of gap to 300-500k floor:** IoPool parallelism not yet wired into the serve loop (in-scope for a follow-up plan). The apply path runs entirely inline on the single mio thread. Lock contention is negligible (single writer) — the bottleneck is throughput of the single-threaded apply + encode-response pipeline.
