# Plan 18-01 — Hand-rolled event loop with HTTP + TCP listeners

## Goal

Stand up `beava-runtime-core` crate with a `mio`-based event loop. Add HTTP/1.1 + framed TCP listeners both in the same loop. Bridge to existing `AppState` apply path. Behind feature flag so tokio path remains the default until cutover.

## Scope: ~2200 LoC (TCP-only would be ~1500; HTTP+TCP is ~2200)

## Tasks (TDD red-green per task)

### Task 1.1 — `beava-runtime-core` crate scaffold (~200 LoC)

**1.1.a (RED):** test in `event_loop_smoke.rs` that `EventLoop::new()` constructs and returns; trivial unit assertion.

**1.1.b (GREEN):**
- New crate at `crates/beava-runtime-core/` with `Cargo.toml` declaring deps: `mio = "1"`, `bytes = "1"`, `httparse = "1.10"`, `crossbeam-channel = "0.5"`, plus `beava-core` dependency.
- `src/lib.rs` re-exports `EventLoop`, `Client`, `Listener`, `Response`.
- `src/event_loop.rs`: `EventLoop` struct wrapping `mio::Poll` + `mio::Events`. `EventLoop::new()`, `EventLoop::register_listener()`, `EventLoop::tick(timeout: Option<Duration>)`.

### Task 1.2 — TCP framed listener + client state machine (~400 LoC)

**1.2.a (RED):** test in `event_loop_smoke.rs` that a TCP-framed listener accepts a connection, reads a `ping` frame (Phase 2.5 wire format), produces a parsed `WireRequest::Ping` event.

**1.2.b (GREEN):**
- `src/tcp_listener.rs`: `TcpListener` wrapper; binds to addr; accepts via `mio::TcpListener`.
- `src/client.rs`: `Client` struct with per-client `read_buf: BytesMut`, `write_queue: VecDeque<Bytes>`, state enum {`Reading`, `Parsing`, `Writing`, `Closing`}.
- Frame parser: `parse_frame(&[u8]) -> Option<(WireRequest, usize)>` returning the parsed request + bytes consumed. Reuses `crates/beava-core/src/wire.rs` opcode constants (Phase 2.5).
- Integration: `EventLoop::tick` reads ready TCP clients, invokes parser, dispatches to `apply_callback`.

### Task 1.3 — HTTP/1.1 listener + per-client HTTP state machine (~700 LoC)

**1.3.a (RED):** test that an HTTP listener accepts a connection, parses `POST /push/foo HTTP/1.1` with a JSON body, produces a `WireRequest::HttpPush` event with `(event_name, body_bytes)`.

**1.3.b (GREEN):**
- `src/http_listener.rs`: HTTP/1.1 listener using `httparse::Request` for headers; body read by Content-Length OR chunked encoding (use `httparse::CHUNK_SIZE_LINE` helper).
- `src/router.rs`: simple match dispatch on path. Recognized routes: `/push/:event`, `/push-sync/:event`, `/push-batch/:event`, `/get`, `/get/:feature/:key`, `/upsert/:table`, `/delete/:table`, `/retract`. Unknown path → 404. Method mismatch → 405.
- HTTP-specific client state: extends per-client state with `http_state: HttpClientState { request_line, headers, body_pos, body_len, keep_alive }`. Reset between requests on same connection (keep-alive).
- Pipelining: process multiple requests from one buffer if available.

**Edge cases to test (red-green pairs each):**
- Chunked transfer encoding: send a `/push-batch` body in 3 chunks; assert correct concatenation.
- Keep-alive pipelining: send 3 `/push` requests on one connection without waiting; assert all 3 dispatched.
- Malformed header: missing `\r\n\r\n` boundary; assert connection closed cleanly.
- Connection: close header: assert connection closed after response.

### Task 1.4 — Apply-path integration via `runtime_core_glue.rs` (~300 LoC)

**1.4.a (RED):** integration test that pushes through the hand-rolled HTTP listener and reads back via `Server::http_get` — assert apply happened.

**1.4.b (GREEN):**
- `crates/beava-server/src/runtime_core_glue.rs`: glue layer that takes `WireRequest` from `beava-runtime-core` and calls existing `AppState::apply_event` / `AppState::query_feature`.
- For TCP: `WireRequest::TcpPush { event, payload }` → MessagePack decode → apply.
- For HTTP: `WireRequest::HttpPush { event, body }` → `serde_json::from_slice` → apply.
- Response building (`src/response.rs` in runtime-core): Pre-formatted byte-string templates for common responses (`HTTP/1.1 204 No Content\r\n...`, TCP ack frames). NO `serde_json::to_vec` for hot path.

### Task 1.5 — Server boot integration (feature-flagged) + admin tokio (~400 LoC)

**1.5.a (RED):** test that `Server::bind_v18(http_addr, tcp_addr, admin_addr)` returns successfully with all three listeners; HTTP and TCP go through hand-rolled loop, admin (port 8081) goes through tokio.

**1.5.b (GREEN):**
- Feature flag `--features hand-rolled-runtime` in `crates/beava-server/Cargo.toml`.
- `crates/beava-server/src/server.rs`: under feature flag, spawn dedicated OS thread for hand-rolled event loop; admin endpoints bound to separate port via tokio runtime in another thread.
- `crates/beava-server/src/http_admin.rs`: tokio/axum app handling `/metrics`, `/health`, `/ready`, `/registry`. Read-only access to shared state via `Arc<RwLock<RegistrySnapshot>>` updated on register.

### Task 1.6 — samply profiling under load (~100 LoC test infrastructure)

**1.6.a (RED):** test that asserts samply profile shows reactor/scheduler cost < 15% of total CPU on a 30s bench run. (Skipped on CI; manual run required.)

**1.6.b (GREEN):**
- Document procedure in `.planning/phases/18-redis-hand-roll/18-01-perf-profile.md`: how to run samply on the hand-rolled bench, what to compare against the Phase 13.3 baseline.
- Capture initial M4 numbers; commit a baseline snippet.

## Perf gate 1.1 (Apple-M4, INFORMATIONAL)

After Task 1.5 lands, run:
```bash
cargo bench -p beava-bench --features hand-rolled-runtime -- --pipeline small --transport tcp --duration-secs 30
cargo bench -p beava-bench --features hand-rolled-runtime -- --pipeline small --transport http --duration-secs 30
```

Expected:
- TCP: 15-50k EPS/core (sanity range — Phase 18.1 is setup, real uplift starts in 18.2)
- HTTP: 8-25k EPS/core
- Match Phase 13.3 tokio baseline within ±20% (must not catastrophically regress; large regressions indicate a setup bug)

## Perf gate 1.2 (Apple-M4, INFORMATIONAL)

samply profile under 64-client load:
- Reactor + scheduler cost: < 15% of CPU (down from 43% on tokio Phase 13.3)
- Per-event task switches: ≤ 1 per request (down from 4 on tokio)

If gate 1.2 fails: investigate before Stage 18.2. The whole Phase 18 thesis rests on removing tokio's reactor cost.

## Verification

Plan complete when:
- `beava-runtime-core` crate exists, builds, tests green
- HTTP + TCP listeners both serve from the same event loop
- Apply path produces expected results through both transports
- Admin tokio path on port 8081 still works (smoke test all admin endpoints)
- M4 perf gates 1.1 + 1.2 met
- Samply profile documented in `18-01-perf-profile.md`

## Risks & mitigations

- **httparse edge cases** (chunked, trailers): `httparse` is mature; cover the cases that matter (chunked + keep-alive pipelining) with explicit tests.
- **Cross-thread state sharing for admin** (`/metrics`, `/health`, `/registry`): admin reads shared atomics + `Arc<RwLock<RegistrySnapshot>>`; no write-back path.
- **tokio dependency stays** (admin endpoints) — acceptable per D-01.
