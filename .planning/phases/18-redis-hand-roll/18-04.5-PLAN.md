### Task 4.5.1 — Decide runner platform + provision (~½ day)

**checkpoint:human-action** required — this is a billing/access decision.

Decision matrix (record outcome in `18-04.5-runner-setup.md`):

| Option | Cost | Pros | Cons |
|---|---|---|---|
| GitHub Actions self-hosted runner on dedicated bare-metal Xeon | one-time hw + electricity | full control, no per-CI-run cost, repeatable | hardware procurement, must self-host |
| Hetzner / OVH bare-metal Xeon dedicated server (~€50-100/mo) | monthly | good price/perf, dedicated cores, no neighbours | can't easily share with other GH workflows; ops overhead |

7950X3D OR EX130-S with Xeon Gold 6442Y, 24 physical cores, 256GB RAM,
~€100-150/mo). Linux Xeon language is preserved in the gate even if we end up
on AMD EPYC; what matters is "Linux x86_64 server-class with isolated cores
+ tuned kernel" — the Phase 13 spec reads "Xeon" as a class label for the
hw-class column in `.planning/perf-baselines.md`.

Provision the machine. Capture in `18-04.5-runner-setup.md`:

### Task 4.5.2 — Tune kernel + boot for low-jitter benchmarking (~½ day)

Edit `/etc/default/grub`, set `GRUB_CMDLINE_LINUX_DEFAULT` to include:
- `isolcpus=2-15` — reserve cores 2..15 for benava processes only
- `nohz_full=2-15` — turn off scheduler tick on isolated cores when only one task runs
- `rcu_nocbs=2-15` — push RCU callbacks off isolated cores
- `transparent_hugepage=never` — predictable allocation latency
- `intel_pstate=performance` (or `cpufreq.default_governor=performance` on AMD) — pin frequency
- `mitigations=off` — disable spectre/meltdown mitigations on this dedicated bench host (NOT on prod hosts)

Run `update-grub && reboot`. After reboot:
- `taskset -c 2-15 cargo bench ...` to pin benches to isolated cores
- `cat /sys/devices/system/cpu/cpu2/cpufreq/scaling_governor` → must show `performance`
- `cat /sys/kernel/mm/transparent_hugepage/enabled` → must show `[never]`

Verify low jitter: run `cyclictest -p 99 -t 4 -i 1000 -l 10000 -a 2-5` for 10s,
assert max latency < 50µs. If higher, investigate (typically: another tenant
on the box, or BIOS power-saving still on — Hetzner support ticket if needed).

Document the verified-tuned state in `18-04.5-runner-setup.md`.

### Task 4.5.5 — CI workflow `.github/workflows/perf-linux.yml` (~1 hour)

Create a manual-trigger GitHub Actions workflow that runs on the self-hosted
runner:

```yaml
name: perf-linux
on:
  workflow_dispatch:
    inputs:
      stage:
        description: "Phase stage to bench (18-3, 18-4, 18-5, 18-6)"
        required: true
        type: choice
        options: ["18-3", "18-4", "18-5", "18-6", "all"]

jobs:
  bench:
    runs-on: [self-hosted, linux, xeon, isolcpus]
    timeout-minutes: 120
    steps:
      - uses: actions/checkout@v4
      - name: Build release
        run: cargo build --workspace --release --features hand-rolled-runtime
      - name: Run benches
        run: bash scripts/run-perf-${{ inputs.stage }}.sh
      - name: Upload baseline diff
        uses: actions/upload-artifact@v4
        with:
          name: perf-baseline-${{ inputs.stage }}
          path: .bench/**
```

Mirror the bench commands from Task 4.5.4 into `scripts/run-perf-18-3.sh`,
`scripts/run-perf-18-4.sh`, etc. Stages 18.5 + 18.6 will use the same
infrastructure with their stage-specific bench shapes added in those plans.

Workflow does NOT auto-fail on regression yet; it produces artifacts that the
phase verification step inspects. (We can add diff-against-baseline assertions
in Stage 18.6.)

## Risks & mitigations

- **Provisioning delay**: Hetzner AX/EX provisioning typically <2 hours but can
  spike to 1-3 days. If urgent, fall back to Equinix Metal (provisioning <15
  min, ~3x cost). Document chosen vendor in `18-04.5-runner-setup.md`.
- **Self-hosted runner registration token leakage**: token goes in 1Password
  vault, NOT in the repo or workflow file. Workflow references the runner by
  label only.
- **Bench variance from shared resources**: dedicated bare-metal eliminates
  most of this. If we end up on cloud, document expected variance (typically
  ±5-10% for cloud bare-metal vs ±1-2% for dedicated). Phase 18 perf gates
  use ±10% warn / ±25% block per CLAUDE.md §Performance Discipline, well
  inside cloud variance.
- **Kernel tuning revert on package upgrade**: pin GRUB config; document
  `apt-mark hold linux-image-*` if needed.
- **No SSE4.2/AVX2 features autodetected**: Rust release builds with
  `-C target-cpu=native` for Phase 18 benches. Document this in the runbook;
  ensure CI workflow runs with `RUSTFLAGS="-C target-cpu=native"` when on the
  bench runner.
