# Plan 18-03 — I/O threads for reads

## Goal

Match Redis 6.0's I/O threading pattern: a pool of `std::thread` workers do
`read()` + frame/HTTP parse in parallel, while the apply thread orchestrates
the dispatch and processes parsed commands serially. The apply thread remains
the sole owner of `Rc<RefCell<AppState>>` — I/O threads never touch app state.

This stage parallelises the per-request CPU cost of parsing (httparse on HTTP,
frame decode on TCP) which Phase 18.2 numbers will show is the biggest
remaining hot-path cost after WAL was inlined.

Reference: `18-redis-research.md` §"I/O threads" + `18-rust-translation.md`
rows 5-8.

Run `cargo test -p beava-runtime-core io_pool_spin_barrier` → must FAIL (no
implementation). Commit:

**3.1.b (GREEN):** Implement:
- `crates/beava-runtime-core/src/config.rs`: `pub struct IoConfig { pub io_threads: usize }` with `Default::default()` returning `num_cpus::get().saturating_sub(1).max(1)`.
- `crates/beava-runtime-core/src/io_pool.rs`:
  - `pub struct IoSlot { pending: AtomicUsize, work: Mutex<Vec<WorkItem>>, parker: Parker }`
  - `pub struct IoPool { slots: Vec<Arc<IoSlot>>, handles: Vec<JoinHandle<()>>, stop: Arc<AtomicBool> }`
  - `IoPool::new(n: usize) -> Self` spawns N `std::thread`s each running `io_worker_loop(slot, stop)`.
  - `IoPool::publish(&self, items: Vec<WorkItem>)` distributes round-robin into slots, sets each `slot.pending.store(slot_count, Release)`, unparks worker.
  - `IoPool::join_all(&self)` spins on `slot.pending.load(Acquire) == 0` for every slot. Spin uses `std::hint::spin_loop()`; after 1024 idle iters, `thread::yield_now()`; after 65536 idle iters, `thread::park_timeout(Duration::from_micros(50))`. Workers `parker.unpark()` when they finish to wake the joiner faster.
  - `WorkItem`: alias for `Box<dyn FnOnce() + Send + 'static>`.
- `crates/beava-runtime-core/src/io_thread.rs`:
  - `pub(crate) fn io_worker_loop(slot: Arc<IoSlot>, stop: Arc<AtomicBool>)` — loops: park if no work, drain `slot.work` lock, execute each `WorkItem`, then `slot.pending.store(0, Release)` to signal done.
  - Backoff: same spin pattern as join.

Run test → must PASS. Commit: `feat(18-redis-hand-roll-18-03): io_pool spin barrier with release/acquire`

After Task 3.5 lands, run beava-bench across the scaling sweep:

```bash

  cargo bench -p beava-bench --features hand-rolled-runtime -- \
    --pipeline small --transport tcp --io-threads $N --duration-secs 30 \
    | tee bench-io-$N.log
done
```

Expected (Apple-M4):
- `io_threads = 0`: 300-500k EPS/core (matches Stage 18.2 baseline)
- `io_threads = 2`: 600-900k EPS/core aggregate
- `io_threads = 4`: **1-1.5M EPS/core aggregate** (informational target)
- `io_threads = 8`: 1-1.5M EPS/core aggregate (no super-linear past physical core count; possible mild regression from hyperthread contention)

Append per-row to `.planning/perf-baselines.md` keyed by Apple-M4 hw-class with
columns `(stage, io_threads, transport, EPS/core, p99-ms)`.

If gate fails (4 threads doesn't reach ≥1M EPS/core aggregate): investigate
parse cost. Likely culprits:
- `httparse` re-parsing headers on every chunk (should reuse `Request` between chunked-body reads)
- `BytesMut::reserve` reallocating mid-parse (pre-size the read buffer to 8KB)
- Atomic contention on `slot.pending` (move to per-thread cacheline padding)

## Risks & mitigations

- **`unsafe impl Send for ClientRef`**: justify in code comment block; main
  thread calls `join_all` before touching the client again, so the per-tick
  exclusive-access invariant is sound. Reviewer should sanity-check.
  Alternative: clone the `BytesMut` into the work item (avoids unsafe but
  costs an alloc per ready client per tick — likely too expensive).
- **Spin-wait CPU burn at idle**: addressed by Task 3.4 backoff. Park timeout
  100µs balances wake latency vs CPU cost.
- **mio Poll-not-thread-safe**: only main thread touches `mio::Poll`. I/O
  threads operate on the `Client` slice handed to them; they never call
  `poll.poll()`. Documented in `event_loop.rs`.
- **Connection sticky-per-tick, not per-connection**: deliberately matches
  Redis pattern. A client's reads can land on different I/O threads across
  ticks; this is fine because state stays in the `Client` struct, not the
  thread.
- **Hyperthread contention on M4**: M4 has 4 perf cores + 6 efficiency cores.
  Past 4 I/O threads, scheduling on E-cores hurts. Document in
  `18-03-perf-scaling.md`. Linux Xeon (Stage 18.4.5+) has many more perf
  cores; scaling continues there.
- **Senior review checkpoint**: per spec, atomic-coordination correctness
  should be reviewed before Stage 18.5 hard gate. Flag this in
  `18-RESUME.md` after 18-03 lands.
