# Plan 18-12 — Eliminate per-push String alloc in bookkeeping

## Context

Post-18-04.8 trace breakdown will show:
```
parse        ~50 ns  (after 18-04.8: body→Row on IoPool)
lookup        31 ns
validate      32 ns
wal_build     33 ns
wal_append    43 ns
agg          473 ns
bookkeeping  169 ns  ← still: event_id_index.lock() + insert + event_name.to_string()
TOTAL       ~830 ns
```

The bookkeeping String allocation is `event_name.to_string()` per push, used as the `EventIdEntry::Stream.event_name` field for retract routing. Mutex acquire is uncontended (~10 ns); the String alloc is the real cost (~50-100 ns plus the value copy).

## Architecture decisions

### D-1 — Intern event_name as Arc<str> at registration

`EventDescriptor` gets a new `pub name_arc: Arc<str>` field, populated at registration via `Arc::from(name.as_str())`. Existing `name: String` field stays for backward compat (many readers iterate it as &str via Deref).

### D-2 — EventIdEntry::Stream uses Arc<str>

```rust
pub enum EventIdEntry {
    Stream { event_name: Arc<str> },  // was String
    Table { ... },
    ...
}
```

This propagates to consumers — but the only hot-path producer is dispatch_push_sync. Test/admin consumers can convert via `.as_ref()` (free, &str via Deref).

### D-3 — dispatch_push_sync uses descriptor.name_arc.clone()

Inside the bookkeeping block:
```rust
// Before:
EventIdEntry::Stream { event_name: event_name.to_string() }   // ~50-100 ns String alloc

// After:
EventIdEntry::Stream { event_name: descriptor.name_arc.clone() }  // ~5 ns refcount bump
```

`descriptor` is already in scope post-18-11 (it's the `Arc<EventDescriptor>` from the registry lookup).

### D-4 — File scope

Three files:
- `crates/beava-core/src/registry.rs` — add `name_arc: Arc<str>` to `EventDescriptor`, populate in registration paths
- `crates/beava-server/src/registry_debug.rs` — change `EventIdEntry::Stream.event_name` from `String` to `Arc<str>`
- `crates/beava-server/src/apply_shard.rs` — bookkeeping site uses `descriptor.name_arc.clone()`

If consumers of `EventIdEntry` exist outside these files (e.g. /admin endpoints reading the index), update mechanically — `Arc<str>` derefs to `&str`, so most call sites work unchanged.

## Risks

- **Cross-crate Arc pollination** — Arc<str> doesn't break Serialize/Deserialize because it serializes as the underlying str. WAL replay path may need handling if it reconstructs EventIdEntry from disk; verify.
- **&str-via-Deref behavior** — most consumers are fine; some may need `.as_ref()` annotation for explicit conversion.
