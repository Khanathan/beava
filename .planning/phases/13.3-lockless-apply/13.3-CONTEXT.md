# Phase 13.3 — Lockless apply via `RefCell` + `LocalSet` (Option 0)

**North star:** replace the `parking_lot::Mutex<BTreeMap<String, AggStateTable>>` on the apply hot path with a `RefCell<StateTables>` owned by a single `LocalSet` task. Redis-shaped single-thread event loop. Target ≤ 80 ns/event (aim 60 ns) inside the apply closure. ~500 LoC refactor concentrated in `crates/beava-server/src/apply/` (new module), the runtime wiring in `main.rs`, and the four callsite surfaces (`push.rs`, `feature_query.rs`, `snapshot_task.rs`, `recovery.rs`).

Phase 13.2 (ApplyBuffer / batch coalesce) is ABANDONED — the Mutex is removed outright, not amortized.

---

## Locked decisions

### Architecture & ownership

**D-01 — `RefCell` owns state, single task owns the `RefCell`.**
A new `ApplyActor` task owns `RefCell<ApplyState>` inside a `tokio::task::LocalSet`. The `RefCell` is NEVER shared across tasks — all access goes through the actor. This is Option 0 from the idea doc; Options A/B/C are out of scope for this phase.

**D-02 — `ApplyState` is the union of everything that was under `state_tables.lock()` today.**
Specifically: `BTreeMap<String, AggStateTable>` (aggregation state), `HashMap<u64, EventIdEntry>` (event_id → entry index, Phase 11.5 D-10), `HashMap<String, TemporalStore>` (temporal tables, Phase 11.5 D-01), and monotonic counters (`next_event_id`, `max_event_time_ms` — kept atomic for now, see D-14). All four are moved into `ApplyState` (the counters stay `AtomicU64` for lock-free read access, they're commutative).

**D-03 — `LocalSet` runs on the existing `current_thread` runtime.**
`main.rs` wraps `rt.block_on(LocalSet::new().run_until(async { … }))`. No multi-threaded runtime. The single-thread mental model from PROJECT.md is preserved. Tests use `#[tokio::test(flavor = "current_thread")]` + `LocalSet`; a small `spawn_local_test` helper lives in `testing.rs`.

**D-04 — `AppState` becomes `Rc<AppState>`.**
Replaces today's `Arc<AppState>`. Handlers clone the `Rc` into their closures. Axum's `with_state` accepts `!Send` state when the whole service runs on a `LocalSet`. The WAL sink handle (`WalSink`) stays `Arc`-shaped internally because it owns an `Arc<Mutex<…>>` for the fsync worker queue — that's orthogonal (WAL's lock is not on the apply hot path).

### Channel shape & command protocol

**D-05 — Channel is `tokio::sync::mpsc` bounded, capacity 8192.**
Bounded for back-pressure; 8192 is empirically ~100× the current 17k EPS / 50 µs pace = ~340 in-flight at steady state, giving ~20× slack. NOT `unbounded` (OOM under burst) and NOT a `LocalSet::spawn_local`-per-event pattern (spawn cost dominates at ns budgets). The bounded `mpsc` receiver lives on the actor task; every HTTP/TCP handler holds a `clone` of the sender.

**D-06 — Commands are `ApplyCommand { … , ack: oneshot::Sender<Result<…>> }`.**
Enum variants for the initial cutover: `ApplyEvent { event_name, row, event_time_ms, ack_lsn, ack }`, `ApplyTableUpsert { … , ack }`, `ApplyTableDelete { … , ack }`, `ApplyRetract { event_id, ack }`, `Snapshot { reply: oneshot::Sender<SnapshotBody> }`, `InstallSnapshot { body, ack }`, `ReplayWal { records, ack }`, `RegistryUpdate { new_registry_arc, ack }`. Query variants (`GetFeature`, `BatchGet`, `GetTemporalVersion`) may go through the actor (D-11) or via `ArcSwap` (D-11). Phase 13.3 Plan 03 decides.

**D-07 — Drain many per wake.**
Actor loop: `rx.recv_many(&mut buf, 4096).await` then iterate synchronously. Natural coalescing of back-to-back events without any explicit `ApplyBuffer` primitive (Phase 13.2's abandoned approach). The loop never `.await`s inside the drain — pure synchronous state mutation.

### Read-path semantics

**D-08 — `/push-sync` preserves read-your-writes by ACKing AFTER apply.**
ACK ordering: client receives 200 only after `oneshot::Sender::send(Ok(ack_lsn))` fires, which happens after the actor has mutated state. Any subsequent `/get` for the same key sees the new value — identical semantics to today.

**D-09 — `/push` (Periodic mode) preserves today's semantics.**
Today: WAL returns after in-memory append (pre-fsync), then apply happens under the lock, then HTTP 200 is returned. New: WAL returns after in-memory append, then command is sent through the actor channel, actor applies and ACKs via `oneshot`, handler returns 200. The relative ordering (WAL append → apply → response) is preserved.

**D-10 — Back-pressure: 503 with `Retry-After: 0` when channel is full.**
If `tx.send(cmd).await` would block because `mpsc` is at capacity, handlers use `tx.try_send(cmd)` first; on `Full`, return `503 Service Unavailable` with JSON `{"error":{"code":"apply_queue_saturated"}}` and `Retry-After: 0` header. Clients are expected to retry. This is Redis-shaped — shed load rather than unbounded queue. Phase 13 perf gate will tune the 8192 capacity if needed.

**D-11 — Read path: `GET /get` and batch-get go through the actor too (linearizable, simplest).**
Rationale: the actor is the unique writer, so a `GetFeature` command serialized behind pending applies gives linearizable reads for free. Throughput cost is ~200 ns oneshot roundtrip per query, acceptable because `/get` is not the hot bottleneck (push is). `ArcSwap` snapshot approach is deferred to a future phase if read latency needs it. The `max_event_time_ms` used by `compute_query_time_ms` stays atomic (lock-free read, no roundtrip cost).

### Registry, snapshot, shutdown

**D-12 — Registry updates route through the actor via `RegistryUpdate { new_registry_arc }`.**
Today: `register.rs` does `registry.apply_registration(…)` under a `Registry`-internal `RwLock` that's orthogonal to `state_tables.lock()`. New: `register.rs` still does the `Registry::apply_registration` call (keeps its own `RwLock`), then sends `RegistryUpdate { new_registry_arc: self.registry.clone() }` through the actor channel so the actor observes the new registry via `Arc::clone`. This guarantees register + pending applies are serialized at the actor — a newly-registered aggregation sees only events pushed AFTER the registration ACK.

**D-13 — Snapshot: actor produces serialized `SnapshotBody` via `Snapshot { reply }`.**
`snapshot_task` sends `Snapshot { reply }` through the actor; actor responds by serializing the state synchronously into a `SnapshotBody` (bincode encode stays on the actor thread — snapshot is a synchronous copy-out of `ApplyState`). Snapshot task then writes bytes to disk via `spawn_blocking` (unchanged). Preserves: (a) snapshot-write never blocks the apply loop beyond the serialize step, (b) snapshot represents a consistent point-in-time view.

**Trade-off accepted:** snapshot serialize (~10 µs for 1k features per current baseline; linear in state size) runs on the actor thread. At production scale (1M entities) this could be ~10 ms — a hiccup for `/push` latency. Phase 13.3 accepts this; a future phase can introduce copy-on-write if measured problematic.

**D-14 — Counters stay atomic.**
`next_event_id: AtomicU64`, `max_event_time_ms: AtomicU64`. Both are commutative (monotonic fetch_max / fetch_add). Lock-free reads from `compute_query_time_ms` preserved. Actor writes to these atomics inside its apply closure — same memory cell, just no longer under a Mutex.

**D-15 — Shutdown drains.**
On `shutdown_signal`: (1) close the `mpsc::Sender` (via dropping the master `tx` held by `AppState` — handlers' clones are already being dropped as HTTP shutdown runs), (2) actor loop sees `recv` return `None`, drains any residual commands, (3) performs a final snapshot via `SnapshotTriggerTx`, (4) task exits. `serve()` awaits the actor `JoinHandle` before returning. All in-flight `oneshot::Sender`s resolve before ACK futures are dropped.

### Testing

**D-16 — Tests run under `LocalSet` via helper.**
`testing::spawn_local_test` wraps `LocalSet::run_until`. Existing tests using `#[tokio::test]` with default flavor on a multi-thread runtime need to migrate to `#[tokio::test(flavor = "current_thread")]` + `LocalSet`. The test harness (`TestServer::spawn`) already uses `current_thread` (confirmed via `main.rs`), so the migration is contained.

**D-17 — `RefCell` double-borrow panic is a test contract.**
If any code path reaches `state.borrow_mut()` while another borrow is live (would indicate an `.await` inside the apply closure), the panic IS the signal. A `cfg(debug_assertions)` assertion in `ApplyState::apply_event` wrapper documents "no `.await` allowed here." Release builds keep the `RefCell` check (it's ~1 ns).

4. Read-your-writes semantics preserved — `/push-sync` acceptance smokes + crash UAT unchanged.

## Grey areas / assumptions that may need revision

- **Axum `!Send` state**: confirmed feasible at doc level (axum supports `!Send` handlers via `Router::with_state` + `serve` on `LocalSet`), but the first touchpoint in Plan 01 is to write a failing test that actually boots axum with `Rc<AppState>` and fails on the `Send` bound. If the bound can't be satisfied without forking axum, fall back to `Arc<RefCell<…>>` with `SafeCell`-style `unsafe` (documented). Likelihood low; contingency noted.
- **`tokio::test(flavor = "current_thread")` + `LocalSet`**: test helper pattern is documented in tokio but not in existing beava tests. Plan 01 Task 0.a writes a smoke test that proves the harness works before any production code lands.
- **Snapshot serialize blocking apply**: accepted for 13.3; revisited if `large` pipeline p99 push latency regresses beyond +25% of current.
- **HLL / CMS / sketch ops inside actor**: these allocate on `update` (CMS grows, sketches merge). Allocation inside a `RefCell::borrow_mut` block is fine (we're not yielding) but may push per-event cost above 80 ns on sketch-heavy pipelines. Plan 04's criterion bench measures both "1 count op" and "1 sketch op" cases; if sketches push the p50 above 80 ns the 60 ns target is downgraded to per-op-family in VERIFICATION.

## Not in scope for Phase 13.3

- Option A (atomic slot specialization for commutative ops) — future phase.
- Option B (sharded actors) — future phase, only if 13.3 single-actor throughput is inadequate for the Phase 13 ship gate.
- Multi-thread runtime — locked out per PROJECT.md.
- Stream retraction — still Phase 12 / v1 territory.

## Plan structure

- **13.3-01-PLAN.md** — Scaffold `ApplyActor` + `ApplyState` + `LocalSet` wiring; state ownership moves into the task; operations still route through the old Mutex-shaped API wrapping a channel send. Existing tests still pass. (Foundation; no behavior change.)
- **13.3-02-PLAN.md** — Cutover the apply path (`/push`, `/push-sync`, TCP `OP_PUSH`, table upsert/delete, retract, register). Remove the Mutex. Preserve read-your-writes + fsync-on-spawn_blocking.
- **13.3-03-PLAN.md** — Cutover the read path (`GET /get`, `POST /get` batch, temporal lookups) to route through the actor. Atomic counters retained for lock-free `compute_query_time_ms`.
- **13.3-04-PLAN.md** — Criterion microbench + full throughput run + SUMMARY + VERIFICATION. Appends rows to `.planning/perf-baselines.md` and `.planning/throughput-baselines.md`.
