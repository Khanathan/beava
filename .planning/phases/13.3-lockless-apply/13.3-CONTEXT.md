# Phase 13.3 — Lockless apply via `Rc<RefCell<AppState>>` + `LocalSet` (Option B)

**North star:** replace the `parking_lot::Mutex<BTreeMap<String, AggStateTable>>` on
the apply hot path with a `Rc<RefCell<AppState>>` shared into every handler via
axum's `State<_>` extractor. Redis-shaped single-thread event loop. One OS
thread runs accept + parse + apply + response. `borrow_mut` → mutate → drop →
respond. The `borrow_mut` scope is synchronous, no `.await` inside.

Phase 13.2 (ApplyBuffer / batch coalesce) is ABANDONED — the Mutex is removed
outright, not amortized.

---

## Hot path shape

```
axum handler task (on LocalSet, !Send)
  └─ State<Rc<RefCell<AppState>>>
       └─ state.borrow_mut()          ~1 ns RefCell check (non-atomic)
       └─ s.apply(event)               ~60 ns (op-dependent)
       └─ drop(borrow)
       └─ respond                      (fsync fan-out for acks=all only)
```

Zero channel on the apply path. No mpsc, no oneshot, no actor, no back-pressure
queue. Reads do `state.borrow()` — linearizable by construction (single thread,
cooperative), zero overhead.

---

## Locked decisions

### State ownership & runtime

**D-01 — State ownership: `Rc<RefCell<AppState>>`.**
One `AppState` struct holds registry, aggregations (`BTreeMap<String, AggStateTable>`),
temporal tables (`HashMap<String, TemporalStore>`), event_id index
(`HashMap<u64, EventIdEntry>`), idempotency cache, and the in-memory WAL buffer.
`AppState` is cloned into every handler via axum's `State<Rc<RefCell<AppState>>>`.
NOT `Arc` — `Rc` is correct because we are single-thread; this saves ~9 ns per
handler clone (atomic CAS vs non-atomic refcount). See D-16 for measurements.

**D-02 — Runtime: current-thread tokio + LocalSet.**
```rust
let rt = tokio::runtime::Builder::new_current_thread().enable_all().build()?;
let local = tokio::task::LocalSet::new();
rt.block_on(local.run_until(async move {
    let state: Rc<RefCell<AppState>> = Rc::new(RefCell::new(AppState::new(...)));
    // spawn_local the listener — see D-03
}))?;
```
Everything on the apply path runs inside the `LocalSet`. `tokio::task::spawn_local`
(not `spawn`) for any work that touches `Rc<RefCell<AppState>>`.

**D-03 — Axum integration (the only non-trivial glue).**
Axum's default `Router::into_make_service()` returns a `Service` bound by `Send`,
which `Rc<RefCell<_>>` violates. Solution: hand-written accept loop using
`hyper::server::conn::http1::Builder::new().serve_connection(io, tower_service)`
where `tower_service` is `axum::extract::connect_info::IntoMakeService` wrapped
via `tower::ServiceExt` so the outer future is `!Send`, and the whole thing is
driven inside `tokio::task::spawn_local`. This is ~40 LoC in
`crates/beava-server/src/main.rs` (or a new `serve_local.rs` module). Pattern
reference: axum's `examples/serve-with-hyper` + `examples/unix-domain-socket`
demonstrate the `!Send` service idiom; both compile against axum 0.7 / hyper 1.x,
which matches our workspace deps.

**Plan 01 Task 1 MUST prototype this before Plan 02 starts.** If axum-on-LocalSet
proves impossible (hyper 1.x requires `Send` somewhere untangleable), the fallback
is `Arc<RefCell<AppState>>` with an `unsafe impl Send` on a newtype wrapper —
safe because we only ever access it from the single LocalSet thread, but this
requires explicit user sign-off. It is NOT pre-authorized.

### Read + write paths

**D-04 — Read path: direct borrow.**
```rust
async fn get_handler(
    State(state): State<Rc<RefCell<AppState>>>,
    ...,
) -> Response {
    let s = state.borrow();
    let v = s.state_tables.get(&agg).and_then(|t| t.query_feature(...));
    drop(s);
    json_response(v)
}
```
No roundtrip, no snapshot, no `ArcSwap`. Linearizable because single-thread
cooperative: any writer that would mutate had to finish before this `await`-free
borrow block started.

**D-05 — Write path: direct borrow_mut.**
```rust
async fn push_handler(
    State(state): State<Rc<RefCell<AppState>>>,
    ...,
) -> Response {
    // WAL append first (under borrow_mut), then apply under same borrow.
    let ack_lsn = {
        let mut s = state.borrow_mut();
        let lsn = s.wal_buffer.append_inmem(&record);  // fast; no I/O
        s.apply_event(event_name, row, event_time_ms, lsn);
        lsn
    }; // borrow released here
    // For acks=all, wait for fsync; see D-06.
    respond(ack_lsn)
}
```
The `borrow_mut` scope MUST NOT contain any `.await` point. This is enforced by
convention (code review + the `clippy::await_holding_refcell_ref` lint from Plan
03). Long batch handlers (`push_many`, `mset`) take the borrow once per chunk
of N=128 events, release, `tokio::task::yield_now().await`, retake. This is the
Redis "SLOWLOG" model — fairness is cooperative, per-chunk.

**D-06 — WAL fsync: unchanged from Phase 13.1.**
The in-memory WAL buffer lives inside `AppState`. Handler appends to buffer under
`borrow_mut` (fast), then releases the borrow. A separate `fsync_worker` task
drains the buffer via `tokio::task::spawn_blocking` and runs fsync. The worker is
`Send` (it only holds a `tokio::sync::mpsc::Receiver<FsyncCmd>` and a file handle),
so it runs on the main runtime, not on the LocalSet.

- `/push` (acks=1 / Periodic): handler appends to in-memory buffer, drops
  borrow, returns 202. Fsync happens async on the worker.
- `/push-sync` (acks=all / PerEvent): handler appends to buffer, drops borrow,
  sends `oneshot::Sender` over a `tokio::sync::mpsc` to the fsync worker, then
  `.await`s the oneshot for the durable-commit ACK. **This is the ONLY
  cross-thread `.await` on the hot path, and only for acks=all semantics.** It
  does NOT hold a borrow across the await (the borrow was dropped first).

**D-07 — Registry updates: yield between DAG phases.**
`POST /register` is rare but expensive (DAG validation + schema propagation +
install can touch 10k nodes). Pattern:
```rust
{
    let mut s = state.borrow_mut();
    s.registry.parse_new_definition(&body)?;
} // drop borrow
tokio::task::yield_now().await;  // let accept loop + pending /get run
{
    let mut s = state.borrow_mut();
    s.registry.topo_sort_and_validate()?;
} // drop borrow
tokio::task::yield_now().await;
{
    let mut s = state.borrow_mut();
    s.registry.propagate_schema()?;
} // drop borrow
tokio::task::yield_now().await;
{
    let mut s = state.borrow_mut();
    s.registry.install()?;
}
```
Each phase is one synchronous `borrow_mut` scope; yields between phases keep the
accept loop responsive during big registers.

**D-08 — Snapshot: `spawn_local` + borrow-while-serialize.**
`snapshot_task` is spawned via `spawn_local` (it needs `Rc<RefCell<AppState>>`).
Acquires `state.borrow()` at a quiescent moment, calls
`bincode::serialize_into(file, &*s)` directly on-thread. At 1M entities this
blocks writes for ~10 ms — **documented as accepted for v0 ship gate**; v0.1
roadmap item is incremental / copy-on-write snapshot. Plan 04's measurement
confirms real impact on the small/medium/large benches.

**D-09 — Back-pressure: OS TCP accept queue.**
No app-level 503. No queue. If the single apply thread falls behind, incoming
connections queue at the kernel; clients time out naturally. Tune
`net.core.somaxconn` and the listener's `SO_REUSEADDR`; document the sysctl
recommendations in `13.3-SUMMARY.md`. This is Redis-shaped — shed load by letting
the kernel do it, not by adding a 503 path in application code.

### Fairness & discipline

**D-10 — Per-event cost targets (microbench).**
`criterion` bench at `crates/beava-server/benches/phase13_3_apply.rs` measures
`state.borrow_mut().apply_event(...)` per event for the 5 most-common op
families. Targets on Apple-M4:

| Op family | Target | Notes |
|---|---:|---|
| count | ≤ 80 ns | scalar, 1 branch + 1 add |
| sum | ≤ 80 ns | scalar add |
| avg | ≤ 80 ns | scalar add + count |
| ewma / ewvar | ≤ 200 ns | decay math, no alloc |
| hll / cms / sketch | ≤ 400 ns | inherent allocator pressure |

Exceeding a scalar target (count/sum/avg) is a **BLOCKER** (that's the
Redis-shape promise). Exceeding a sketch target is a **WARNING** (sketches are
inherently more work; noted in VERIFICATION).

**D-11 — Fairness / multi-connection.**
Single-thread cooperative means a slow handler blocks all others. Enforce via
code review gates:
- `apply_event` is O(1) per event — no loops over state.
- Batch handlers (`push_many`, `mset`) yield per 128-event chunk.
- Register handler yields per DAG phase (D-07).
- No unbounded `for … in …` over state_tables in any handler without an
  internal yield.

These are review-only gates, not runtime gates. Add a written checklist to
`13.3-SUMMARY.md` so reviewers remember.

### Mutex removal inventory (D-12)

**Must be deleted from the apply hot path** (exhaustive; verified via grep at
Plan 02 close):

| Site | Today | After 13.3 |
|---|---|---|
| `DevAggState.state_tables: Arc<Mutex<BTreeMap<…>>>` | Mutex | moved into `AppState.state_tables: BTreeMap<…>` (no Mutex; `RefCell` of the outer struct serializes) |
| `DevAggState.temporal_stores: Arc<Mutex<HashMap<…>>>` | Mutex | moved into `AppState.temporal_stores` |
| `DevAggState.event_id_index: Arc<Mutex<HashMap<u64, EventIdEntry>>>` | Mutex | moved into `AppState.event_id_index` |

**May remain** (orthogonal, not on apply hot path):
- `Registry` internal `RwLock` — register.rs writes it rarely; handlers clone
  `Arc<Registry>` out of `AppState` under `borrow()` and drop the borrow before
  doing registry work.
- `IdemCache` internal `RwLock` — sits on push hot path BUT is a bounded LRU with
  sharded internals; leave it for a future phase if measured problematic.
- `WalSink` fsync worker queue `Mutex` — orthogonal, cross-thread.
- `/metrics` counter atomics — lock-free by construction (D-15).

### Send boundary (D-13)

| Task | `Send` bound | Runtime |
|---|---|---|
| axum accept loop | `!Send` (holds `Rc<RefCell<AppState>>`) | LocalSet (`spawn_local`) |
| HTTP handlers | `!Send` | LocalSet |
| TCP frame handlers | `!Send` | LocalSet |
| snapshot_task | `!Send` | LocalSet (`spawn_local`) |
| fsync_worker | `Send` (mpsc Receiver + File) | main runtime (`spawn`) |
| WAL disk-IO (spawn_blocking closures) | `Send` | blocking pool |

Any `tokio::spawn` (as opposed to `spawn_local`) on code that touches
`Rc<RefCell<AppState>>` is a bug. CI grep check in Plan 04 verification.

### Read-your-writes & metrics

**D-14 — Read-your-writes preserved by construction.**
Handler does `borrow_mut → apply → drop → (optional fsync await for acks=all) →
borrow → read → respond`. All on the same thread in order. Nothing else can run
between borrow and borrow because we only yield at `.await` points and the
apply scope has no `.await`.

**D-15 — `/metrics` counters stay `AtomicU64`.**
`beava_http_requests_total`, `beava_operator_updates_total`,
`beava_apply_events_total`, etc. — all bumped directly from the handler without
touching the RefCell. The counters are commutative; atomicity is enough.
`next_event_id: AtomicU64`, `max_event_time_ms: AtomicU64` — same deal;
`fetch_max` / `fetch_add` are monotonic. These DO NOT go inside `AppState`'s
RefCell — they live on `AppState` as plain `AtomicU64` fields (the RefCell
guards the non-atomic collections only). `compute_query_time_ms` reads
`max_event_time_ms` via `load(Relaxed)` with no borrow at all.

### Rationale

**D-16 — Rc vs Arc measured cost.**
`Rc::clone` is ~1 ns (non-atomic refcount increment). `Arc::clone` is ~10 ns
(atomic CAS). Every axum handler runs `State::<T>::from_request` which clones
the state. At 100k req/s that's 1 ms/s saved by `Rc` vs `Arc`. Small but free.
Also: `Rc<RefCell<_>>` vs `Arc<Mutex<_>>` — borrow is ~1 ns, Mutex lock is
~20 ns uncontended and ~hundreds ns contended. At the throughputs we're
targeting, the RefCell is ~20× cheaper per access.

### Testing (D-17)

All apply-path tests use `#[tokio::test(flavor = "current_thread")]` and the
`LocalSet::run_until(async { ... })` pattern. Add a helper in
`crates/beava-server/src/testing.rs`:
```rust
pub async fn with_local_set<F, Fut, T>(f: F) -> T
where
    F: FnOnce() -> Fut,
    Fut: std::future::Future<Output = T>,
{
    let local = tokio::task::LocalSet::new();
    local.run_until(async move { f().await }).await
}
```
Existing tests that use `#[tokio::test]` default flavor on a multi-thread
runtime need to migrate. The `TestServer` harness already uses current_thread
(confirmed in `main.rs`), so the migration is scoped to a handful of ad-hoc
tests that spawned `tokio::spawn` on Send state.

## Grey areas / assumptions to verify

- **Axum `!Send` on LocalSet (D-03).** Documented as feasible via axum 0.7 +
  hyper 1.x + hand-written accept loop (axum examples `serve-with-hyper` and
  `unix-domain-socket`). Plan 01 Task 1 prototypes this against a trivial
  `Rc<RefCell<()>>` handler to confirm the `!Send` future compiles and serves.
  If infeasible, fallback to `Arc<RefCell<AppState>>` + `unsafe impl Send`
  newtype requires explicit user sign-off. NOT pre-authorized.

- **`await_holding_refcell_ref` lint availability.** The clippy lint exists
  (nightly) but stability varies. Plan 03 attempts to enable it workspace-wide;
  if unstable, falls back to a code-review checklist + optional custom lint.
  The runtime check (RefCell panic on double-borrow) remains the safety net.

- **Snapshot serialize blocking writes (D-08).** At 1M entities, ~10 ms borrow
  held. Plan 04 measures the p99 push latency impact and ledgers the result.
  If the impact exceeds +25% on large pipeline, revisit as v0.1 roadmap item;
  not a 13.3 blocker.

- **Sketch op per-event cost (D-10).** HLL / CMS allocate on `update` and merge.
  Target is ≤ 400 ns; if we miss, WARNING not BLOCKER, and we downgrade the
  target in VERIFICATION per op family. The scalar targets (count/sum/avg ≤ 80 ns)
  are non-negotiable — those ARE the Redis-shape promise.

- **`tokio::test(flavor = "current_thread")` + LocalSet in existing tests.**
  Migration cost contained: `TestServer::spawn` already uses current_thread;
  handful of ad-hoc tests may need the `with_local_set` helper. Plan 01 Task 0
  establishes the pattern.

## Not in scope for Phase 13.3

- Atomic-slot specialization for commutative ops — v0.1 phase if needed.
- Multi-thread runtime / sharded state — locked out per PROJECT.md single-thread
  mental model.
- Stream retraction — Phase 12 / v1 territory.
- `ArcSwap`-style snapshot reads — D-04 direct borrow is already lock-free; the
  v0.1 candidate if snapshot-serialize-blocking-writes becomes a real problem.
