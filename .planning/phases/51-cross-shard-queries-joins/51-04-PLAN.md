<objective>
Implement `JoinShardKeyMismatch` register-time fatal error — TPC-CORR-04.

Purpose: When a stream participates in a join operator and its declared shard_key disagrees with the peer stream's shard_key, the registration must fail immediately with a structured error before any shard is activated. The exact D-12 error message text is locked for grep-testability. A warning is also emitted to /debug/warnings.

Output:
- `src/engine/join_validator.rs` — `validate_shard_keys` function; `JoinShardKeyMismatch` error struct with exact D-12 message format
- `src/engine/pipeline.rs` — `register()` calls `validate_shard_keys` before inserting stream; propagates error to caller
- `src/server/signals.rs` — `JoinShardKeyMismatchWarning` signal type + emission helper
- Tests (TDD): mismatched shard_key → error with locked message; matching shard_key → no error; /debug/warnings entry appears

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SDK caller → PipelineEngine::register | Registration input is trusted (server-side); validation is a correctness guard, not a security gate |
| engine → SignalRegistry | Write to shared RwLock; called only at REGISTER time (not hot path) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-51-04-01 | Denial of Service | SignalRegistry::record in register() | accept | Called only at REGISTER (rare); record() performs zero I/O; no hot-path risk |
| T-51-04-02 | Tampering | join_validator — peer stream lookup | accept | Peer lookup uses existing streams map; missing peer = skip (not an error); no state mutation on skip |
| T-51-04-03 | Repudiation | JoinShardKeyMismatch signal | mitigate | Signal id is stable (stream_a:stream_b); second attempt with same pair refreshes last_seen, does not duplicate; operator can see history in /debug/warnings |
</threat_model>
