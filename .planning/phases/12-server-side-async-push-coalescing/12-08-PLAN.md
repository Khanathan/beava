<objective>
Cut apply-thread per-event orchestration overhead from 53% (push) / 91% (read) to under 25% by eliminating per-event kernel wake-up coordination, eliminating apply's idle gaps, amortizing IO-worker wake-up cost, and eliminating response BytesMut allocator churn.

**What this fixes:** post-12-07 samply/perf traces show the apply thread spending only ~10% of its budget on real per-request work during read-heavy load (196k r/s) and ~47% on real aggregation work during saturated push (51k EPS Linux, 92k EPS macOS). The remaining 50–90% is per-request orchestration overhead split across `run_mio_event_loop` outer body (~10%), `crossbeam_channel::Sender::send` (~3%), kernel `try_to_wake_up` (~5%), `epoll_wait` + eventfd I/O (~3%), `libc::malloc` + `cfree` (~4%), syscall entry/exit (~4%), kernel spin locks (~3%). Apply is also idle ~42% of wall-time during push, blocked in `epoll_wait` waiting for IO worker to enqueue events (~6 µs of idle per event).

**What this delivers (per `12-08-SCOPE.md` D-A through D-D, locked):**

1. **D-A — Adaptive busy-poll on apply.** Replace `event_loop.tick(timeout)` blocking with a `read_rx.try_recv()` busy-poll loop with K=10,000 spin-iteration counter; fall back to `read_rx.recv_timeout(Duration::from_micros(50))` when truly idle. Apply CPU: ~100% under load, drops to <5% when truly idle (>50 µs no events).

2. **D-D — Drain-until-empty pattern.** Apply's per-iteration loop drains ALL queued items (no DRAIN_CAP=1024 cap) before checking timers + idle fall-through. Already partially landed in Phase 18-13 / Plan 12-07 but the cap forces re-entering the outer mio loop more often than necessary.

3. **D-B — Response batch with hybrid threshold.** Apply collects responses into a per-iteration `response_batch: SmallVec<[(slot_u64, WriteEncoder); 16]>`. Flush when batch has 16 entries OR `batch_started_at.elapsed() ≥ 100µs`. The 100µs timer is checked on each spin iteration via `Instant::now()` (apply busy-polls fast enough to see the deadline). At low load: timer dominates → flush per request → no batching artifact. At high load: batches of ~16 → 1 wake-up per 16 responses → drops `try_to_wake_up` cost ~16×.

4. **D-C — Per-IO-worker BytesMut pool.** Each IO worker maintains a thread-local `BytesMutPool` (fixed-cap `crossbeam_queue::ArrayQueue<BytesMut>`, default 256 buffers × 4 KiB capacity = 1 MiB per worker). Response build path acquires from pool (or allocs if pool empty), serialises into the buffer, freezes. Reclamation: a custom `RecyclableBytes` wrapper whose `Drop` returns the buffer to the pool. Eviction: pool grows past 512 buffers → drop excess.

Purpose: closes the "blocking the Plan 12-09 push-and-get latency target (P50 < 300 µs HTTP)" status from `12-08-SCOPE.md`. Without 12-08, push-and-get lands at ~600 µs because of the 5 µs/event overhead Plan 12-08 removes.

Output: apply-thread loop running busy-poll + response batch + per-thread pool; criterion microbench for the loop; throughput rebaseline across small/medium/large/fraud-team × tcp/http × push+read on Apple-M4 + Hetzner; Plan 12-08 SUMMARY.
</objective>

Key rewrite points (Waves 1, 2, 3):
- Wave 1 (D-A): replace `Some(Duration::from_millis(50))` blocking branch with a `read_rx.recv_timeout(Duration::from_micros(50))` call; the listener-poll cadence stays on `event_loop.tick(0ms)` (non-blocking) but is ONLY exercised every `LISTENER_POLL_EVERY` iterations or after a wake.

`WorkerClient.write_buf: BytesMut` is the per-client output buffer. Each encoder call extends it with serialized JSON / TCP frame bytes. The pool's job (Wave 4) is to:
1. Provide a freshly-cleared 4 KiB BytesMut to each encoder call
2. Reclaim it once the bytes are written through `flush_write_buf`'s `client.write_buf.split_to(n)` consumption
3. Avoid the per-call `BytesMut::with_capacity(4096)` alloc that today's `write_buf` extends implicitly via `BufMut::put_slice`

Implementation choice (locked 2026-04-29 — performant variant): the pool is a thread-local on the IO worker — no cross-thread coordination, no contention. The encoder fills a pooled `BytesMut`, then **wraps the frozen Bytes in a `RecyclableBytes` smart pointer whose `Drop` impl returns the underlying `BytesMut` allocation to the pool** instead of freeing it. ZERO malloc per response on the hot path (until pool capacity is exceeded; then falls back to direct alloc).

Sketch:
```rust
pub struct RecyclableBytes {
    inner: bytes::Bytes,                       // immutable view exposed to consumers
    reclaim: Option<RawBytesMut>,              // raw allocation for reclamation
    pool: Arc<BytesMutPool>,
}
impl Drop for RecyclableBytes {
    fn drop(&mut self) {
        if let Some(raw) = self.reclaim.take() {
            // Only reclaim if we hold the LAST refcount; otherwise let Bytes drop normally.
            if Arc::strong_count(&self.inner) <= 1 { /* push back to pool */ }
        }
    }
}
```

`WorkerClient.write_buf` writes consume the RecyclableBytes via `extend_from_slice` then drop the wrapper, returning the alloc to the pool. Net effect: per-response `Bytes::from(serialized)` malloc is gone on the steady-state hot path. If `RecyclableBytes` reference-counted reclamation is too tricky to make sound under all paths, fall back to a `Vec<u8>`-backed `BytesMut` pool that the encoder uses directly + `BufMut::extend_from_slice` into `write_buf` (the simpler design); the planner-executor decides per Wave 4.b's verify gate (must achieve `libc::malloc + cfree` < 0.5% of apply CPU per the must_haves).

    ///

    /// `buf_capacity` — capacity each buffer is constructed with on first alloc.
