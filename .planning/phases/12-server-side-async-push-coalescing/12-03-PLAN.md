    Add a Rust integration test that:
    1. Starts a test server on 127.0.0.1:0 with the `medium` pipeline registered
    2. Spawns task A that opens a TCP client and pushes 5,000 OP_PUSH_ASYNC frames as fast as it can, then sends FLUSH
    3. Concurrently task B opens a second TCP client and, while task A is running, sends 100 OP_PUSH sync frames with 500µs gaps, recording the wall-clock latency of each sync response
    4. After both complete, compute sync p50 and p99 from task B's latencies
    5. Assert a **shape-based ceiling** — all three of:
        - `sync_p99_us < 200.0` (absolute ceiling — pathological regression catch)
        - `sync_p50_us < sync_p99_us` (sanity: p50 strictly less than p99)
        - `sync_p99_us < 3.0 * sync_p50_us` (distribution shape — p99 not more than 3× p50, catches tail-blow-up regressions that a single absolute bound would miss)

    Rationale for shape-based bound: in-process `cargo test` runs vary 2-3× wider than a dedicated bench run because the test harness shares the tokio runtime. A single absolute 150µs bound is either too tight (flaky) or too loose (misses regressions). The shape check `p99 < 3 × p50` catches the pathological "async saturation pushes sync p99 into the 500µs+ range" case regardless of the baseline noise floor, while `p99 < 200.0` is a hard pathological ceiling. The TIGHT 91.4µs gate lives in bench.py; the Rust test's job is to catch regressions before we ever run bench.
