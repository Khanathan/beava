<objective>
Production-ready /get on HTTP+TCP through the mio data plane, plus migration of the production binary `target/release/beava` from the legacy axum tokio path to `ServerV18` (mio).

**What this fixes:** the production binary `crates/beava-server/src/main.rs` currently boots `Server::bind` (axum tokio path), which mounts `feature_query_router` ONLY when `BEAVA_DEV_ENDPOINTS=1`. Phase 19.4's `python/benches/read_bench.py` doesn't set that env var, so every POST /get returns 404. Separately, the mio HTTP parser (`http_listener.rs`) already routes `/get` and `/get/:feature/:key` into `WireRequest::HttpGet` / `HttpGetSingle`, but `apply_shard.rs` dispatches them into `runtime_core_glue::dispatch_get_batch` which is a STUB returning `{"result": {}}`. And TCP `OP_GET` / `OP_MGET` / `OP_GET_MULTI` are reserved opcodes that today fall through to `WireRequest::Unknown { op }`.

**What this delivers:**

1. WireRequest variants `TcpGet { body, body_format }`, `TcpMGet { body, body_format }`, `TcpGetMulti { body, body_format }` for framed TCP /get
2. `tcp_listener::parse_wire_request` arms for OP_GET / OP_MGET / OP_GET_MULTI gated by content_type (CT_JSON / CT_MSGPACK)
3. `ApplyShard::dispatch_one` arms for the new TCP variants, reusing `dispatch_get_single_sync` / `dispatch_get_batch_sync`
4. Real `dispatch_get_batch` impl in `runtime_core_glue.rs` (mirrors `dispatch_get_single`, iterates `keys × features`, returns `{"result": {key: {feature: value}}}`)
5. TCP response encoder support for `GlueResponse::QueryResult` and `GlueResponse::QueryNotFound`, plus a new `OP_GET_RESPONSE = 0x0023` opcode
6. `main.rs` migration: `Server::bind(&cfg, dev_endpoints).await?.serve(shutdown_signal()).await` becomes `ServerV18::bind(http_addr, tcp_addr, admin_addr).await?.serve_with_dirs(shutdown_signal(), wal_dir, snapshot_dir).await`
7. Tests proving each layer: HTTP single, HTTP batch, TCP single, TCP batch, read-your-writes through apply serialization, production binary running unmodified `read_bench.py` to a non-zero `ok` count
8. Criterion microbench for the read-path; row appended to `perf-baselines.md`
9. Throughput run rebaselined post-migration; rows appended to `throughput-baselines.md`

Purpose: closes the "blocking Phase 19.4 read-bench measurement runs" status from `12-07-SCOPE.md`. Aligns the production binary with the architectural directive (mio data plane only) per memory `project_phase18_no_dual_runtime` and the user's 2026-04-29 directive (reads must be serialized through the apply thread; /get must work over both HTTP and TCP).

Output: working /get on production binary, a real `dispatch_get_batch`, three new TCP variants, working TCP /get end-to-end, microbench + throughput-run baselines, Plan 12-07 SUMMARY.

pub fn parse_wire_request(
    buf: &mut BytesMut,
    max_frame_bytes: u32,
) -> Result<Option<WireRequest>, beava_core::wire::FrameError> {
    let frame = match decode_frame(buf, max_frame_bytes)? { Some(f) => f, None => return Ok(None) };
    let req = match frame.op {
        OP_PING => WireRequest::Ping,
        OP_REGISTER => WireRequest::Register { payload: frame.payload },

    };
    Ok(Some(req))
}
