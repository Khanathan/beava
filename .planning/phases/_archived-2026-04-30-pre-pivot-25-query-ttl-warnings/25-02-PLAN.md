<objective>
Ship v0 retention: apply the 30d/90d TTL defaults at registration time,
track TTL evictions + eviction-then-reinit via a per-Table bloom filter,
surface recommendations over HTTP + CLI + startup advisory. Per
`v0-restructure-spec.md §7` and Phase 25 CONTEXT.md §TTL defaults +
§Suggestion engine (both LOCKED).

Purpose: Users should never tune TTLs by guessing. The engine observes
the key reinitialization pattern (evicted then re-seen → TTL is too
short) and hands back copy-pasteable decorator overrides. Without this,
TTL is a footgun — too short and users silently lose state, too long
and memory bloats unobservably.

Output:
- TTL defaults enforced at REGISTER (30d Table / 90d Stream / 7d tombstone)
- EvictionTracker with per-Table bloom filter + reinit counter
- `tally_ttl_evictions_total` / `tally_ttl_eviction_then_reinit_total` /
  `tally_history_compacted_total` / `tally_history_backfill_misses_total`
  on /metrics
- `/debug/config-recommendations` HTTP endpoint
- `tally suggest-config` binary in `src/bin/`
- Startup advisory one-line-per-knob if recommendations exist at boot
</objective>

pub struct EvictionTracker {
    // Per-table bloom filters; 7d rolling via pair of bloom filters
    // rotated daily (classic "generational" bloom pattern).
    per_table: DashMap<String, GenerationalBloom>,
}
impl EvictionTracker {
    pub fn new(capacity_per_table: usize, fp_rate: f64) -> Self;
    pub fn record_eviction(&self, table: &str, key: &str);
    pub fn check_reinit(&self, table: &str, key: &str) -> bool; // true → key seen recently
    pub fn rotate_generation(&self, now: SystemTime);  // daily tick
}

// src/engine/pipeline.rs
pub struct ConfigRecommendation {
    pub knob: String,             // e.g., "UserProfile.ttl"
    pub current: String,          // e.g., "30d"
    pub suggested: String,        // e.g., "60d"
    pub confidence: f64,          // 0.0..1.0
    pub reason: String,
    pub evidence: serde_json::Value,
    pub copy_paste: String,       // "@tl.table(key=\"user_id\", ttl=\"60d\")"
}

                        tracker: &EvictionTracker) -> Vec<ConfigRecommendation>;

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-25-02-02 | Denial of Service | Per-table bloom filters at 1-2 MB × N tables → memory amplification | mitigate | Cap at 256 Tables per instance (matches GET_MULTI cap); document ~500 MB max at 256-table limit; add `tally_bloom_memory_bytes` metric for observability |
