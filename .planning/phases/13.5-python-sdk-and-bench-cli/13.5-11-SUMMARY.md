## What shipped

### Task 11.b — mypy --strict pass + Transport API alignment (LANDED)

`mypy --strict` on `python/beava/` now exits 0 with `Success: no issues found in 19 source files`.

Resolved 12 mypy errors that surfaced a deeper SDK integration gap from Plans 02-07:
- `App.register` was calling `t.send_register(descriptors=, force=, dry_run=)` but `Transport.send_register(payload_json: bytes)` only accepted bytes.
- `App.push` / `App.get` / `App.batch_get` / `App.reset` were calling Transport methods that didn't exist.

**This commit:**
1. Extends the `Transport` base class with the 4 missing method stubs (`send_push`, `send_get`, `send_batch_get`, `send_reset`). Stubs raise `NotImplementedError`. Subclasses override.
2. Adds `python/beava/_app.py::_to_register_json()` — module-level helper that converts `@bv.event` / `@bv.table` descriptors into the wire-spec `{"nodes": [...]}` JSON shape. This is the contract `python/tests/v0/test_lit.py` references.
3. Updates `App.register` to build the JSON payload from descriptors and call `t.send_register(payload_bytes)` with positional bytes arg. `force` / `dry_run` flags now ride inside the JSON payload.
4. Aligns `App.push` / `App.get` / `App.batch_get` / `App.reset` to the new Transport surface.
5. Updates `TcpTransport.send_push` signature from `(event_name, body_dict, *, wire_format)` to `(*, event_name, fields)` per the new base-class contract.
6. Test fix: `python/tests/internal/test_app_lifecycle.py::test_register_calls_transport_with_force_dry_run_kwargs` updated to assert the new positional-bytes-args contract; `force` / `dry_run` are now extracted from the JSON payload.

**Internal tests:** 145/145 GREEN against the new API.

### Task 11.c — 68 v0 acceptance tests against real engine (DEFERRED)

**Status: NOT LANDED** — surfaces deeper SDK integration gaps that exceed Phase 13.5's scope.

**Root cause:** `HttpTransport` / `TcpTransport` / `EmbedTransport` need full implementations of `send_push`, `send_get`, `send_batch_get`, `send_reset` (currently base-class `NotImplementedError`). Each requires:
- Wire-format envelope construction (`{"event": ..., "body": ...}` for push; `{"feature": ..., "key": ...}` for get).
- HTTP method routing (POST `/push/<event_name>` / GET `/get/<feature>/<key>` / POST `/batch_get` / POST `/reset`).
- TCP frame encoding with the right opcodes (OP_PUSH=0x10, OP_GET=0x20, OP_BATCH_GET=0x24, OP_RESET=0x40).

**Additional gap discovered:** v0 tests like `test_count_per_user_high_volume` use the pattern:
```python
@bv.table(key="user_id")
def UserClicks(clicks):                           # <- no annotation
    return clicks.group_by("user_id").agg(...)
```
The current `_table.py::_resolve_upstream_proxies` resolves unannotated parameters to `inspect.Parameter.empty` (rendered as `<class 'inspect._empty'>`), which then doesn't have `.group_by()` — causing `AttributeError`. The DSL needs a generic event-stream proxy for unannotated parameters.

**Test count baseline at Plan 11 close:** 0/68 v0 acceptance tests passing (all RED at SDK transport level).

**Disposition:** Plan 11.c work moves to v0.1+ as **issue:13.5-11-deferred**. Phase 13.6+ (TS/Go SDK plans) and Phase 13.8 (packaging+GA) gate on this work landing. Recommend a dedicated plan in v0.1 for end-to-end transport implementation before Phase 13.8 ships.

### Bonus — Phase 12.7 tripwire alignment with ADR-001 (LANDED)

The Phase 12.7 tripwire `phase12_7_legacy_table_handlers_killed::python_bv_table_re_export_deleted` forbade ANY `"table"` token in `__all__`, but ADR-001 (locked 2026-05-03) revives `@bv.table` for aggregation-output. Updated the tripwire:
- Forbid `from ._tables import` (plural — legacy upsert/delete) — kept active.
- Allow `"table"` in `__all__` (the new aggregation-output decorator surface).
- Sanity-positive: assert `from beava._table import table` (singular — ADR-001 surface) is re-exported.
