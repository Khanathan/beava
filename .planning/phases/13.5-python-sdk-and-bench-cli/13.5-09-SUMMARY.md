### Decision: backed by config files, not hand-coded payloads

Workloads delegate to existing `crates/beava-bench/configs/*.json` files
to avoid wire-shape drift. fraud uses `fraud-team.json` directly
(`project_fraud_team_primary_bench`); adtech / ecommerce reuse a smaller
config and synthesize family-coverage labels via `op_chain` for the
smoke test.

The Plan 09 description envisaged hand-rolled adtech / ecommerce builders
with bespoke event-generator iterators (Impression/Click/Conversion etc.).
Phase 13.6+ may upgrade those if the Phase 13 ship gate doesn't hit other
operator-coverage requirements first. For v0 the config-backed shape
ships ≥ 5 op-family coverage (fraud-team) and 53-op exposure across all
3 datasets combined.

## Notes

Workload `event_generator` is a closure that captures the config's
`event_name` + `key_field` + `extra_fields`. Synthetic events are
deterministic with seed `0xCAFE_BABE` (matches legacy bench).

For Plan 11, when v0 acceptance tests run against the engine, these
workloads do NOT participate — the v0 tests use their own
`@bv.event` / `@bv.table` declarations. Workloads here serve the bench
CLI only.
