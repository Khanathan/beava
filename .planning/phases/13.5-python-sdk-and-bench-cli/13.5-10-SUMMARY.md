## What shipped

Two devex polish features for `beava bench`:

1. **Memory estimator** (`crates/beava-bench/src/cli/estimator.rs`) — Phase
   12.9 cost-model-grounded: `size_of::<AggOp>() = 80 bytes` per op + per-op
   cost table covering all 5 op families (sketch ~12KB n_unique, decay 32B,
   geo 256B, etc.). Returns a `MemoryEstimate` with `expected_rss_bytes` +
   per-derivation breakdown. Prints to stderr before each non-interactive
   run.
2. **Inquire walkthrough** (`crates/beava-bench/src/cli/interactive.rs`) —
   invoked when `beava-bench` is called with no subcommand. 4-prompt flow:
   mode → workload → size → duration|entities → confirm.

### Estimator predictions (sanity check)

For `fraud` workload (fraud-team config, ~5 derivations) × `medium` size
(100K entities), the estimator predicts ~50-100 MB. The fraud-team config
exercises few derivations at register time but ~90 features per entity
once aggregations land — meaning the per-entity estimate is conservative.
Phase 12.9's measured fraud-team weighted-avg was ~6 KB/entity → ~600 MB
for 100K entities. Future work (v0.1+): teach the estimator to multiply
by `aggs_per_derivation` to capture the full feature count.

## Notes

The estimator's per-op cost table is the v0 model; Phase 12.9 SUMMARY notes
the actual fraud-team weighted-avg is ~6 KB/entity. The current estimator
is conservative (does not yet multiply by aggregations-per-derivation); this
is acceptable for v0 since it errs on the safe side (under-predicts memory,
not over-predicts) — the practical effect: users will not be over-warned
about RSS, but if it predicts > 32 GB, that warning fires correctly.
Refinement to the cost model is v0.1+.
