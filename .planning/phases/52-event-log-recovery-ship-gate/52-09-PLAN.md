  <action>
    Create `docs/architecture-tpc.md` with the following 8 sections. Audience: new
    contributors + operators. Tone: technical but accessible. No marketing language.

    **Section 1 — Motivation**
    Explain the DashMap contention problem: pre-v1.2, all entity state lived in a single
    `DashMap<Entity, Row>` shared across threads. At N>4 cores, cache-line bouncing on
    DashMap's internal sharding buckets became the throughput ceiling (~300k EPS observed
    vs 1.5M target). The TPC architecture gives each shard thread exclusive ownership of a
    disjoint key range — zero cross-thread contention on hot paths.

    **Section 2 — Shard model**
    Include the target-architecture ASCII diagram from `TPC-SHARD-DESIGN.md` § "Target
    architecture" verbatim (read the file to extract it). Explain: N shards, each owns a
    `Shard` struct (AHashMap, WatermarkState, EventLog handle, per-shard metrics). Events
    route by `hash(event.key) mod N`. BEAVA_SHARDS defaults to physical core count on
    release builds.

    **Section 3 — Routing**
    Explain `shard_hint` flow: EventSource computes hint on push; TCP/HTTP parsers forward
    it; shard router uses it to skip rehash when upstream_N == downstream_N. Fallback: if
    shard_key is undeclared, route to shard 0 (emits ShardKeyMissingWarning at N>1).
    Hash function: ahash (deterministic across restarts for same input).

    **Section 4 — Joins**
    Explain co-location requirement: two streams joined must declare the same `shard_key=`.
    Mismatched keys produce `JoinShardKeyMismatch` at registration time (fatal, actionable
    error with both stream names + suggested fix). Why: a join event for key K must arrive
    at the same shard on both streams — only possible if hash(K) routes identically.

    **Section 5 — Recovery**
    Explain parallel per-shard recovery: on boot, N recovery threads each replay their own
    `data/shard-N/streams/*/log.bin`. Main thread coordinates via the boot-barrier
    "recovered" sub-state. `/ready` returns 503 until all N shards pass recovered. `/health`
    stays 200 throughout. Performance: 7.0s → 1.5s on N=8 (linear with core count).

    **Section 6 — Reshard workflow**
    Document `tally reshard --from N --to K --data-dir PATH --output PATH [--replace]`.
    Steps: (1) server must be stopped; (2) tool acquires data-dir lock, rehashes all keys
    to new K-shard layout, writes new v8 snapshot; (3) with `--replace`, swaps dirs
    atomically; (4) restart with BEAVA_SHARDS=K. Downtime = tool runtime. Pointer to
    `docs/operations.md` for sizing guidance.

    **Section 7 — Fork/replica**
    Explain re-hash on ingest: replica always computes `hash(key) mod downstream_N` on
    arrival. Upstream shard_hint is a fast-path optimization only (safe when N matches).
    LSN dedup: every entry has a u64 LSN (upstream_shard_id:8 | stream_ord:16 | seq:40).
    Replica tracks max_lsn_seen per (stream, upstream_shard) in snapshot v8. On reconnect,
    events with LSN <= max_lsn_seen are dropped — closes the rolling-restart double-emit
    window.

    **Section 8 — Ship-gate rationale**
    List the 3 merge-to-main criteria and explain each:
    (1) Standard cells within -5% at N=1: confirms the TPC migration does not regress
        single-shard performance — existing operators upgrading to v1.2 see no regression.
    (2) complex-c8-x8 >= 3x at N=CPU_COUNT: validates that the architecture delivers the
        promised multi-core scaling benefit.
    (3) pareto-c8-x8 cross_shard_fraction <40%: confirms that hot-key (Zipf) workloads do
        not overwhelm any single shard due to routing skew — routing is architecturally fit.

  <name>Task 2: Update docs/operations.md — Shard Sizing & Hot-Shard Diagnosis section</name>

  <action>
    Append a new H2 section `## Shard Sizing & Hot-Shard Diagnosis` to `docs/operations.md`
    with these 6 subsections:

    **1. Choosing BEAVA_SHARDS**
    - Release builds default to `num_cpus::get_physical()` (all physical cores).
    - On an 8-core box: BEAVA_SHARDS=4 is a safe starting point (leaves headroom for OS
      and network threads). On 16+ core boxes: use full physical count.
    - Rule: set BEAVA_SHARDS to be a power of 2 when possible (ahash mod is consistent
      regardless, but powers of 2 aid reasoning about key distribution).

    **2. Metrics to watch**
    - `beava_shard_keys_owned{shard}`: entity count per shard. Imbalance > 2x mean
      indicates a hot-shard condition.
    - `reactor_utilization{shard}`: CPU time fraction on shard reactor. Values > 0.85
      sustained indicate saturation.
    - `shard_probe` `cross_shard_fraction`: fraction of events requiring cross-shard
      routing. Sustained values > 40% indicate shard_key misalignment.

    **3. BEAVA_HOT_SHARD_THRESHOLD tuning**
    - Default: 1.5× fleet mean `keys_owned`. When any shard exceeds this, `/debug/shards`
      includes a `hot_shards` array and a warning log is emitted.
    - Lower to 1.2× for latency-sensitive workloads. Raise to 2.0× to suppress warnings
      on naturally skewed key distributions (user explicitly accepting imbalance).

    **4. Hot-shard diagnosis flow**
    - Step 1: `GET /debug/shards` → inspect `hot_shards` array.
    - Step 2: Run `shard_probe` on the reported hot shard — observe `cross_shard_fraction`
      and `keys_owned`.
    - Step 3: If hot-shard persists > 5 minutes: check stream `shard_key=` declarations.
      Misaligned joins or missing shard_key declarations (routing to shard 0) are the most
      common causes.
    - Step 4: If workload is inherently skewed (e.g., Zipf key distribution): consider
      increasing BEAVA_SHARDS to spread the load, then run `tally reshard`.

    **5. Reshard workflow**
    - When shard count needs changing: stop the server, run `tally reshard --from N --to K
      --data-dir PATH --output PATH`, restart with `BEAVA_SHARDS=K`.
    - See `docs/architecture-tpc.md` § "Reshard workflow" for full CLI reference.
    - Note (CONTEXT D-02): do not manually write to `data/logs/` — this directory is
      managed by Beava's migration tooling. Write to `data/shard-N/streams/` via the server.

    **6. Three ship-gate criteria as production health indicators**
    - These criteria were validated before v1.2 shipped; they serve as ongoing health
      baselines:
      (1) N=1 throughput within -5% of v1.1 baseline → no regression on single-shard
          deployments.
      (2) complex-c8-x8 >= 3x at N=CPU_COUNT → multi-core scaling is delivering.
      (3) pareto cross_shard_fraction <40% → routing handles skewed workloads safely.
    - Run the benchmark suite against any major config change to verify these hold.
  </action>
