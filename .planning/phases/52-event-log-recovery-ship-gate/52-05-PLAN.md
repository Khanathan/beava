<objective>
Implement fork/replica ingest re-hash on arrival (TPC-CORR-06, CONTEXT D-08). On every
OP_LOG_FETCH and OP_SUBSCRIBE event received from an upstream, the replica computes
`target_shard = hash(event.key) mod downstream_N`. The upstream `shard_hint` in
OP_LOG_FETCH metadata is used ONLY as a fast-path optimization when `upstream_N ==
downstream_N` — never as a routing constraint. No `--reshard-from` CLI flag is introduced.

Purpose: A replica with a different shard count than its upstream can ingest correctly
without any out-of-band coordination. This enables the N=1→N=8 fork pattern and closes
the correctness gap identified in TPC-CORR-06: upstream shard_hint is metadata, not a
routing authority.

Output: Updated `src/server/replica.rs` ingest paths; full TDD suite in
`tests/test_replica_rehash.rs` covering routing, fast-path, compile-time flag absence,
and fork parity.
</objective>

| Boundary | Description |
|----------|-------------|
| upstream wire → replica ingest | upstream_shard_hint is untrusted operator metadata; must never override the rehash path when N differs |
| upstream_n negotiation | upstream_N from handshake could be spoofed; fast path only activates when N values match exactly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-52-05-02 | Information disclosure | feature parity test | accept | Test uses in-process ephemeral data; no production data at risk |
| T-52-05-03 | Denial | rehash_to_shard panic | mitigate | shard_count=0 panics on modulo; add assert!(shard_count > 0) in rehash_to_shard at entry |
</threat_model>
