# Phase 13.5.2 — Decorator-only derivation surface + runtime chain-op execution

**One-liner:** Locked the public derivation-declaration surface to exactly two decorator forms (`@bv.event class | @bv.event def | @bv.table[(key=)] def`); rejected raw chain expressions at register-time AND at decoration time with sharp diagnostics; AND wired runtime chain-op execution into the agg apply path so the documented `bv.lit("web")` constant-column flow works end-to-end. 5 test_lit failures GREEN; v0 acceptance 73-74/89 → 81/89.

## What we shipped

### D-01 (USER-LOCKED, Plan 03) — `App.register` rejects EventDerivation instances

`python/beava/_app.py::App.register` now iterates `*descriptors` BEFORE transport access and rejects any raw `EventDerivation` chain with:

```
RegistrationError: at index N — 'EventDerivation' is not a registerable
descriptor. Wrap the chain in @bv.event:
    @bv.event
    def Foo(click: Click):
        return click.with_columns(...)
```

The check uses `getattr(d, "_is_bv_event_function", False)` to skip legitimate `@bv.event def` outputs (which carry the marker set by `_make_event_derivation`).

### D-02 (USER-LOCKED, Plan 04) — Decorator-only public surface + annotation-time rejection

Both decorator resolvers (`python/beava/_table.py::_resolve_upstream_proxies` and `python/beava/_events.py::_make_event_derivation`) raise `TypeError` at decoration time if a parameter annotation resolves to a raw `EventDerivation` instance. Catches the bug earlier than register-time, with the same "wrap in @bv.event" hint.

`_make_event_derivation` also got the same closure-cell + caller-frame walking that `_resolve_upstream_proxies` got in 13.5.1 — so function-local `@bv.event class` definitions resolve correctly under `from __future__ import annotations`. The marker handshake (`result._is_bv_event_function = True`) connects D-01 + D-02.

### D-04 (USER-LOCKED, Plan 05) — Mechanical rewrite of test_lit.py + ADR-003 examples

5 tests in `python/tests/v0/test_lit.py` rewritten to use `@bv.event def Tagged(click: Click): return click.with_columns(...)` form. Three `bv.lit` examples in `.planning/decisions/ADR-003-global-aggregation-and-bv-lit.md:177-183` rewritten to the same form. (`docs/sdk-api/python.md` already showed `bv.lit` only as a simple expression node `bv.lit(42)`; no chain-form examples there to rewrite.)

### Discovered scope expansion: runtime chain-op execution

During Plan 05 execution, discovered that the runtime apply path didn't actually transform events through chain ops (with_columns / select / drop / rename / cast / fillna) — they were register-time schema-propagation only. Events arrived at aggregations as raw source rows; any agg field reference or where predicate targeting a chain-added column silently saw `None`.

**Fix landed in three coordinated changes:**

1. **`crates/beava-core/src/agg_apply.rs`** — per-desc, look up `registry.compiled_chain(&desc.node_name)` and apply via `OpChain::apply(row.clone())` BEFORE evaluating the agg. Fast path (no chain registered) stays zero-overhead. The `pre_val` resolver also gained a slow-path fallback for `field_idx == FIELD_IDX_NONE` (typical for chain-added fields not in the source event's `apply_field_names` union) that uses `agg_row.get(field)`.

2. **`crates/beava-core/src/register_validate.rs::validate_expressions`** — when a derivation has `GroupBy` ops, also compile the chain prefix (with non-GroupBy ops filtered out) so `compiled_chains[derivation_name]` is populated. Previously, GroupBy presence skipped chain compilation entirely.

3. **`python/beava/_app.py::_descriptor_to_node`** — SDK chain-flatten: when a `@bv.table` descriptor's parent chain contains an `@bv.event def` output, set the wire upstream to the ROOT `EventSource`. Required because the server's apply-time routing index `aggregations_by_source` keys aggregations by their direct upstream NAME — events at the root EventSource only trigger aggregations whose declared upstream IS the root.

Plus a small SDK fix in `python/beava/_col.py`: serialize Python `&` / `|` boolean combinators as the expr-grammar `and` / `or` keywords (per `crates/beava-core/src/expr.rs:461`). Previously sent as `&` / `|` which the server's expr parser rejected with `unknown character '&'`.
