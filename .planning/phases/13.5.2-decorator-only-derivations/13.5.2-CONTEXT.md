# Phase 13.5.2: Decorator-only derivation surface (kill chain registration) - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Lock the public derivation-declaration surface to **exactly two decorator forms** and reject everything else with sharp diagnostics:

1. `@bv.event class X:` ‚Äî event source (class form)
2. `@bv.event def X(upstream: Source): return upstream.<chain>` ‚Äî event derivation (function form)
3. `@bv.table[(key=...)] def X(upstream): return upstream.group_by(...).agg(...)` ‚Äî aggregation table (function form)

Anything else ‚Äî most importantly the chained `EventDerivation.named("X")` + register-as-sibling pattern that the `bv.lit()` ADR-003 examples currently use ‚Äî is rejected at register-time AND at `@bv.table` annotation-resolution time with a `RegistrationError` / `TypeError` whose message points at the canonical `@bv.event def X(...)` rewrite.

This collapses the "sibling event-derivation routing" architectural blocker (named in `RESUME-2026-05-04-PM.md` as the open question for 13.5.2) by removing the chain-form siblings entirely. Under the new contract, parameter annotation is the single source of truth for upstream binding ‚Äî there is no second mechanism to disambiguate.

**In scope:**
- `app.register()` rejects any `EventDerivation` instance at the top level with the sharp "wrap in @bv.event" hint.
- `_resolve_upstream_proxies` (in `python/beava/_table.py`) rejects any parameter annotation that resolves to an `EventDerivation` instance with the same hint at decoration time.
- `_resolve_upstream_proxies` for `@bv.event` (in `python/beava/_events.py`) gets the same rejection.
- Rewrite the 5 `python/tests/v0/test_lit.py` tests to use the canonical `@bv.event def Tagged(click: Click): return click.with_columns(...)` form.
- Update `docs/sdk-api/python.md` lines 177-183 (the three `bv.lit` examples that currently show `events.with_columns(...)` chain form) to use the `@bv.event def` form.
- Update `.planning/decisions/ADR-003-global-aggregation-and-bv-lit.md` lines 177-183 (same examples) to match.
- Add paired RED‚ÜíGREEN unit tests for both rejection paths.

**Out of scope:**
- Removing `.named()` itself from the public API (kept callable for internal `__derived_N` auto-naming; not a public surface).
- Auditing `examples/`, `beava-website/`, `blog/` for residual `.named()` patterns (Option C in Area 4 ‚Äî DEFERRED to a v0.0.x cleanup pass).
- Adding a dedicated `test_decorator_contract.py` lockdown class (Option B in Area 4 ‚Äî DEFERRED; the rejection-path unit tests cover the regression risk for v0).
- Triage of the other 10 v0 acceptance failures (`*_per_user_high_volume`, `test_reset_equivalent`) ‚Äî separate follow-up plan after this lands.

</domain>

<decisions>
## Implementation Decisions

### Routing semantics

- **D-01 (USER-LOCKED, Area 1) ‚Äî Hard-error at register; `.named()` callable internally.**
  `app.register(*descriptors)` validates each descriptor at the top of the call: anything that is an `EventDerivation` instance (i.e. a chain expression like `Click.with_columns(...).named("Tagged")`) is rejected with:
  ```
  RegistrationError: at index N ‚Äî 'EventDerivation' is not a registerable
  descriptor. Wrap the chain in @bv.event:
      @bv.event
      def Tagged(click: Click):
          return click.with_columns(...)
  ```
  `EventDerivation.named(...)` stays callable internally ‚Äî used by `_make_derivation` in `python/beava/_events.py:145-148` for auto-naming intermediates as `Foo__derived_N`. Public API surface = decorators only.

  **Why:** The user's directive verbatim ‚Äî *"this is too confusing please dont allow this pattern the only pattern should be @bv.table and @bv.event."* The chain-form sibling-derivation pattern created an ambiguity (which sibling does an annotation-by-raw-source-class bind to?) that has no clean answer; killing the pattern removes the ambiguity entirely. Kept `.named()` callable internally so the existing `__derived_N` auto-numbering keeps working without surgery on `_events.py`.

### Decorator surface contract

- **D-02 (USER-LOCKED, Area 2 ‚Äî collapsed by user's tighter directive) ‚Äî Decorator-only public surface; rejection at decoration AND register.**
  `@bv.event` accepts only:
    - a class (`inspect.isclass()` true) ‚Üí event source
    - a non-class callable (`callable(x) and not inspect.isclass(x)`) ‚Üí event derivation function
  `@bv.table[(key=...)]` accepts only a non-class callable ‚Üí aggregation table function.

  Both decorators' parameter resolvers (`python/beava/_table.py::_resolve_upstream_proxies` and the analogous `python/beava/_events.py` resolver for `@bv.event def`) raise `TypeError` if any parameter annotation resolves to an `EventDerivation` instance:
  ```
  TypeError: @bv.table function 'F' parameter 'tagged' annotation resolves
  to an EventDerivation instance (a raw chain). Annotate with an
  @bv.event-decorated class or function instead ‚Äî e.g.
      @bv.event
      def Tagged(click: Click): return click.with_columns(...)
      @bv.table(key="user_id")
      def F(tagged: Tagged): return tagged.group_by("user_id").agg(...)
  ```

  This catches the bug at decoration time (earlier in the dev loop than register), AND `app.register()` still rejects per D-01 as a defense-in-depth net for any chain that escapes the decorator path. Two enforcement sites; same hint.

  **Why:** The user's second directive ‚Äî *"@bv.event and @bv.table should only be tagged on Class and function"* ‚Äî explicitly tightens the decorator surface beyond the routing question. Combined with D-01 this defines the ENTIRE valid derivation-declaration surface: there are no other shapes.

### Multi-sibling policy (Area 3 ‚Äî collapsed by D-01+D-02)

- **D-03 (DERIVED) ‚Äî Multi-sibling ambiguity is N/A under the new contract.**
  Under D-01+D-02, multiple `@bv.event def Tagged1(c: Click)` and `@bv.event def Tagged2(c: Click)` are independent named derivations. Each is bound by its own parameter annotation at the consumer's `@bv.table`. There is no chain-form "sibling on the same source" concept anymore, so there is no ambiguity to resolve. No new validation required.

### Test + doc rewrite scope

- **D-04 (USER-LOCKED, Area 4) ‚Äî Minimum-scope rewrite.**
  Ship only what is required to green the tests + correct the documented `bv.lit` examples:
    - Rewrite the 5 tests in `python/tests/v0/test_lit.py` (`test_lit_constant_column`, `test_lit_explicit_filter_literal`, `test_lit_force_float_division`, `test_lit_value_types`, `test_lit_immutability`) to wrap each chained derivation in an `@bv.event def Tagged(click: Click): return click.<chain>` function.
    - Update `docs/sdk-api/python.md:177-183` ‚Äî the three `bv.lit` examples currently show `events.filter/with_columns(...)` as bare chain expressions; rewrite to show them inside `@bv.event def` bodies.
    - Update `.planning/decisions/ADR-003-global-aggregation-and-bv-lit.md:177-183` to match.
    - Add paired RED‚ÜíGREEN unit tests for both new rejection paths (D-01's register-time + D-02's annotation-time).
  **Estimated wall-clock:** ~1 day.

  **Why:** User explicitly chose Option A in Area 4. Option B (dedicated `test_decorator_contract.py` lockdown class) and Option C (audit all docs/examples for residual `.named()` chains) are DEFERRED, not rejected ‚Äî they're sensible v0.0.x cleanup passes once the OSS launch is live and we have time to harden.

</decisions>

## Specific Ideas

- **The user's tightening was deliberate.** Reading their two responses together: the public API surface is DELIBERATELY minimized. There are exactly three valid decorator shapes and nothing else. This isn't a "minimum viable" cut ‚Äî it's the load-bearing devex commitment. Plans should not add escape hatches, opt-in flags, or "advanced mode" exceptions to the rule.
- **The hint message must show the rewrite**, not just describe the problem. The user's verbatim example (Option C preview in Area 1) shows a multi-line code-block in the error message body. Plans should preserve that ‚Äî `RegistrationError.__str__` has to render the suggested fix verbatim.
