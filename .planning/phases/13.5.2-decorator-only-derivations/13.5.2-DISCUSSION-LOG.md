## Pre-discussion framing

The architectural blocker named in `RESUME-2026-05-04-PM.md`: when `def UserClicks(tagged: Click)` is registered alongside `Tagged = Click.with_columns(source=bv.lit("web")).named("Tagged")`, the Python decorator binds the parameter to `Click` (the raw event), not `Tagged` (the augmented derivation). The server then rejects the registration with `aggregation_invalid_where: where predicate references unknown field 'source'` because the where-predicate validator uses Click's schema, not Tagged's.

This is the failure mode behind 5 `python/tests/v0/test_lit.py` failures and the documented `bv.lit("web")` "constant column" example in `docs/sdk-api/python.md:182` and ADR-003.

Codebase scan: only `python/tests/v0/test_lit.py` uses the chain-form sibling pattern in production code (`python/tests/test_phase4_smoke.py` is an internal smoke test). Backwards-compat cost is low for any direction.

---

## Area 1 — Routing semantics

| Option | Description | Selected |
|--------|-------------|----------|
| A: Strict-explicit — annotation IS the binding | Annotation `tagged: Click` means parent=Click, period. To consume Tagged, write `tagged: Tagged`. Sharp register-time error if where/select references a sibling-only field. Aligns with current docs verbatim. 5 test_lit tests get mechanical rewrite. Mypy-friendly. No magic. | |
| B: Auto-route to sibling on same source | Runtime walks the registration: if param annotates `Click` AND exactly one `.named(...)` derivation of `Click` exists in the same call, silently rebind parent to the sibling. Test passes as written. Magic; multi-sibling needs explicit policy; mypy disagrees with runtime. | |
| C: Hybrid — annotation for mypy, param-name for binding | Annotation = mypy type hint only. Param NAME does case-insensitive lookup vs `.named(...)` derivations in scope. `tagged` matches `Tagged`. Two sources of truth; users learn the matching rule. | |

**User's choice:** None of the above. User typed in the freeform-Other slot:
> *"this is too confusing please dont allow this pattern the only pattern should be @bv.table and @bv.event"*

**Notes:** This is a sharper architectural cut than any of the three options. Rather than choose between routing rules for chain-form derivations, the user is removing the chain-form derivation pattern entirely from the public surface. Under the new contract, every named derivation must be declared via `@bv.event def Tagged(click: Click): return click.with_columns(...)`. Annotation = upstream descriptor (always; no magic; no second mechanism). The "sibling routing" question is N/A by construction.

## Area 1 follow-up — Enforcement mechanism

Asked because the user's freeform answer required nailing down HOW the chain-form rejection fires.

| Option | Description | Selected |
|--------|-------------|----------|
| A: Delete `.named()` entirely from EventDerivation | Remove `.named()` method from `_events.py`. Any code calling `chain.named('X')` raises `AttributeError`. `app.register(EventDerivation_instance)` raises a sharp `RegistrationError`. Cleanest cut. Hard break for any downstream code. | |
| B: Soft-deprecate — warn at register, keep `.named()` working in v0.0.x | Keep `.named()` returning a usable EventDerivation; emit `DeprecationWarning` when register sees one; full removal in v0.1. Lets the test suite migrate gradually. | |
| C: Hard-error at register, leave `.named()` callable internally | `.named()` still returns an EventDerivation (some internal code may use it) but `app.register()` rejects EventDerivation instances at the top level with a clear "use @bv.event" error. Smaller surgery than A. | ✓ |

**User's choice:** **C — Hard-error at register, leave `.named()` callable internally.**

**Notes:** Internal `_events.py:_make_derivation` uses `.named(f"__derived_{counter}")` for auto-numbering chained intermediates. Option C keeps that working without surgery. Option A would require rewriting the auto-numbering path; Option B leaves a public-but-deprecated surface that conflicts with the user's "only @bv.event and @bv.table" directive.

## Area 2 — Diagnostic UX

| Option | Description | Selected |
|--------|-------------|----------|
| A: Register-only (minimal) | Only `app.register()` rejects EventDerivation instances. Other entry points emit existing errors unchanged. Smallest surface. | |
| B: Also reject at @bv.table annotation resolution | `_resolve_upstream_proxies` raises `TypeError` immediately at decoration time if a parameter annotation resolves to an `EventDerivation` instance. Catches the bug earlier in the dev loop. | |
| C: Aggressive — also reject `.named()` if called on a chain | Above plus deprecate-and-warn `.named()` calls. Internal code uses private `_set_name`. | |

**User's choice:** None of the above. User typed in the freeform-Other slot:
> *"@bv.event and @bv.table should only be tagged on Class and function"*

**Notes:** This is a tightening of the decorator surface contract that goes beyond the diagnostics question. Combined with Area 1's answer, the user is locking the entire valid derivation-declaration surface to exactly three shapes:
1. `@bv.event class X:` — event source (class form)
2. `@bv.event def X(...):` — event derivation (function form)
3. `@bv.table[(key=...)] def X(...):` — aggregation table (function form)

Anything else — including any `EventDerivation` instance leaking into either `app.register()` or `_resolve_upstream_proxies` — is rejected. Reading the user's tightening direction, the diagnostic policy resolves to **Option B** (catch at BOTH `app.register()` AND `_resolve_upstream_proxies`); Option C's `.named()` deprecation is incompatible with the Area-1-follow-up choice (`.named()` callable internally). Locked as **D-02** in CONTEXT.md.

## Area 3 — Multi-sibling policy

**Not asked.** Collapsed by the combination of D-01 (no chain-form siblings reach register) + D-02 (no chain-form siblings reach annotation resolution). Multiple `@bv.event def Tagged1(c: Click)` / `@bv.event def Tagged2(c: Click)` are independent named derivations, each bound by its own annotation. No ambiguity to resolve.
