```rust
pub fn fan_out_retraction_for_source_table(
    &self, sibling_shards, input_shard, input_shard_idx,
    table_name: &str, table_key: &str, source_lsn: u64, n_shards: usize,
) -> Result<(), BeavaError> {
    const DEPTH: u8 = 1;
    let reason = RetractReason::SourceTableDelete {
        table_name: table_name.to_string(),
        table_key: table_key.to_string(),
        source_lsn,
    };
    for downstream_def in self.enrich_downstreams_of(table_name) {
        // Scan rows in downstream_def where contributing_inputs.source_table_keys
        // contains table_key. O(N_dirty_rows) since we only scan the per-stream
        // snapshot — full-table scan is acceptable only on DELETE (cold path).
        for row_key in input_shard.dirty_set_for_stream_snapshot(&downstream_def.name) {
            let Some(entity) = input_shard.read_entity(&row_key) else { continue };
            let Some(ci) = &entity.contributing_inputs else { continue };
            if !ci.source_table_keys.iter().any(|k| k == table_key) { continue };
            let target_shard_idx = if n_shards <= 1 {
                input_shard_idx
            } else {
                crate::routing::shard_hint::hash_to_shard(&row_key, n_shards) as usize
            };
            self.retract_downstream_at_shard(
                sibling_shards, target_shard_idx, input_shard, input_shard_idx,
                &downstream_def.name, &row_key, reason.clone(), DEPTH,
            )?;
        }
    }
    Ok(())
}
```

Trade-off note on scan scope: `dirty_set_for_stream_snapshot` from Wave 2 is batch-scoped. For source-table DELETE, the affected downstream rows may NOT be in the current batch's dirty set. Option A: scan the full downstream partition (O(N), acceptable on DELETE cold path). Option B: maintain a secondary index (source_table_key → downstream_row_keys). Wave 3 chooses Option A for simplicity; Option B goes on the 57-NEXT list if perf demands.

## SSJ tombstone fan-out

For SC-2, a primary stream entity tombstone (e.g., tombstone L:u1) must walk the __ssj__ buffer on hash(join.on)%N shard and retract every joined output that included u1's event_id.

Implementation:
- In the primary stream tombstone path (same site as Wave 2's primary_event fan-out), also iterate registered SSJ joins whose side involves the tombstoned stream.
- For each such join, compute target_shard_idx = hash(join_key) % N. Since ALL joined outputs live in per-shard partitions indexed by join_key (not event_id), we need a secondary scan: fetch the __ssj__ entity at hash(join_key) % N, iterate its buffered events, find ones with primary_event_id == tombstoned event_id, and retract the CORRESPONDING joined outputs (which live in the downstream stream that CONSUMES the join's output — identified via topo-sort).

Simpler pragmatic Wave-3 implementation: at JOIN-emit time, we already know the joined output's row_key. Record a per-join side-map `{primary_event_id → [joined_output_row_key]}` in the __ssj__ entity's SideMap field (new). Tombstone path reads SideMap, retracts each listed row_key.

Add to ContribSet on the __ssj__ entity state (buffer-level, not row-level):
```rust
#[derive(Debug, Clone, Default, Serialize, Deserialize)]
pub struct SsjSideMap {
    pub left_event_id_to_joined_rows: AHashMap<u64, Vec<String>>,
    pub right_event_id_to_joined_rows: AHashMap<u64, Vec<String>>,
}
```

This is stored in the EntityState at the __ssj__ slot as part of the buffer bookkeeping; apply_ssj_insert updates it at emission time.

        let now_ms = /* millis since epoch */;

        if let Some(existing) = self.retraction_beyond_history.iter_mut()
