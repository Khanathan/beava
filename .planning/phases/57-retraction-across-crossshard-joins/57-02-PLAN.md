```rust
fn make_primary_event_id(shard_id: u16, epoch_ms: u64) -> u64 {
    // Pack 16 bits of shard_id + 48 bits of epoch_ms (u48, not u64).
    // Collisions across shards are irrelevant — each shard owns its id-space.
    assert!(epoch_ms < (1u64 << 48), "epoch_ms overflow u48");
    ((shard_id as u64) << 48) | (epoch_ms & ((1u64 << 48) - 1))
}
```

### Step 2 — Tombstone detection + fan-out (src/engine/pipeline.rs)

Identify the path that handles primary-event tombstone (today: `delete_entity` or `tombstone_static` invoked from TCP DELETE opcode or a PUSH with a deletion marker). After the tombstone is applied locally, call a new helper `PipelineEngine::fan_out_retraction_for_primary`:

```rust
fn fan_out_retraction_for_primary(
    &self,
    sibling_shards: Option<&[ShardHandle]>,
    input_shard: &mut Shard,
    input_shard_idx: usize,
    primary_stream: &str,
    primary_event_id: u64,
    n_shards: usize,
) -> Result<(), BeavaError> {
    const DEPTH: u8 = 1;  // root retraction; apply_retraction will refuse on >=16
    for downstream_def in self.cascade_downstreams_of(primary_stream) {
        let dirty_rows = input_shard.dirty_set_for_stream_snapshot(&downstream_def.name);
        for dirty_row_key in dirty_rows {
            let Some(entity) = input_shard.read_entity(&dirty_row_key) else { continue };
            let Some(ci) = &entity.contributing_inputs else { continue };
            if ci.primary_event_id != Some(primary_event_id) { continue };
            let target_shard_idx = if n_shards <= 1 {
                input_shard_idx
            } else {
                (crate::routing::shard_hint::hash_to_shard(&dirty_row_key, n_shards) as usize)
            };
            let reason = crate::shard::thread::RetractReason::PrimaryEventRetract {
                stream_name: primary_stream.to_string(),
                event_id: primary_event_id,
            };
            self.retract_downstream_at_shard(
                sibling_shards, target_shard_idx, input_shard, input_shard_idx,
                &downstream_def.name, &dirty_row_key, reason, DEPTH,
            )?;
        }
    }
    Ok(())
}
```

### Step 5 — Recursive cascade (depth handling)

Each call to retract_downstream_at_shard passes depth + 1; apply_retraction in turn, upon retracting a row, may itself need to fan-out to rows that depend on THAT row (downstream-of-downstream). For Wave 2 scope, Stream→Table cascades can only nest through further Stream→Table cascades (no EnrichFromTable / SSJ dependencies yet — Wave 3). So the cascade chain is:
  primary_stream → TT1 → TT2 → ... → TTn

The fan-out is recursive: apply_retraction on TT1 must itself call `fan_out_retraction_for_this_row` on the TARGET shard, which dispatches to TT2's owner, and so on. Implement this by having `apply_retraction` carry a post-apply hook OR return the list of downstream rows to retract; the dispatcher arm (in src/shard/thread.rs ShardOp::RetractDownstream) then re-dispatches. Choose the pattern that keeps shard apply_retraction pure + lets the dispatch arm drive recursion.

Pragmatic: after apply_retraction returns Retracted, the dispatch arm itself iterates `dirty_set_for_stream_snapshot` for downstreams of `stream_name` and emits further RetractDownstream ops with `depth + 1`. Cap at 16.
