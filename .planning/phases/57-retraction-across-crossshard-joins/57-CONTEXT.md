## Phase Boundary

Retractions propagate end-to-end through cross-shard joins and cascades. Every emitted
downstream row tracks the input events that contributed to it: left/right event ids for
StreamStreamJoin, source-row key for EnrichFromTable, primary event id for Stream→Table.
Tombstones / deletes on any tracked input trigger retraction tombstones on the owning
shards of affected downstream outputs.

Source-table DELETE's pending-retraction markers (written by Phase 55) are consumed here.

**Out of scope (explicit):**
- Rewriting history outside the `history_ttl` window. Late retractions warn + skip.
- Adding retractions to NEW operator kinds (anything Phase 58+ introduces is out of scope).
- Perf micro-optimization (that's Phases 58–63).
- UI / CLI surfaces for inspecting retraction graphs — deferred.

<decisions>
## Implementation Decisions

### Area A — Contributing-Input Tracking

- **D-A1 (tracking scope):** Per emitted downstream row, store a `contributing_inputs: ContribSet` structure:
  - Stream→Table: `{primary_event_id: u64}` (single id).
  - EnrichFromTable: `{primary_event_id: u64, source_table_keys: SmallVec<[String; 2]>}` (usually 1 source-table row read; 2+ when a downstream has multiple enrichments).
  - StreamStreamJoin: `{left_event_id: u64, right_event_id: u64}` (pair).
- **D-A2 (storage):** Co-located with the downstream row in its fjall partition. Serialized alongside the row's `EntityState` / `JoinOutput` record. Same single-writer invariant (Phase 53) — no cross-partition locking.
- **D-A3 (primary_event_id assignment):** Generated by source shard at ingress — `(shard_id: u16, epoch_ms: u48)` packed into a `u64` for monotonic-per-shard ordering. Collisions across shards are irrelevant (each shard owns its own id-space). Compact, log-friendly.
- **D-A4 (cardinality control):** SmallVec-inline `[String; 2]` for source_table_keys covers the common case (1–2 enrichments per event) without heap. Overflow-to-heap handles wider pipelines. No explicit cap — perf test gauges it.
- **D-A5 (back-compat):** Pre-Phase-57 rows (written by P55/P56) have no `contributing_inputs` field. Loading them under v10 snapshot format returns `None`, and retraction logic treats `None` as "cannot-retract" (same as late-retraction beyond history_ttl). No snapshot migration — just a `schema_version` bump to v10.

### Area B — Retraction Dispatch

- **D-B1 (trigger primitive):** New `ShardOp::RetractDownstream { target_shard, stream_name, row_key, reason: RetractReason, reply: oneshot::Sender<RetractAck> }`. Source shard emits this for every downstream row whose `contributing_inputs` intersects the retracted event.
- **D-B2 (retract reasons enum):** `RetractReason::{SourceTableDelete(table_key), EntityTombstone(entity_key), PrimaryEventRetract(event_id)}` — keeps evidence clear for observability.
- **D-B3 (dispatch pattern):** Same as P56's SsjInsert — `try_send` + `crossbeam::channel::bounded(1)` oneshot + blocking `recv` + `BeavaError::ShardOverload` on Full. Retraction is on the write path of the retracting event, not a background job; backpressure at ingress if targets are full.
- **D-B4 (idempotency):** RetractDownstream is idempotent on the target — if the row has already been retracted (or never existed), the op is a no-op that still returns `RetractAck::Ok`. Necessary because (a) source shards may retry on `ShardOverload`, and (b) multiple tombstones on the same event may fan out to the same downstream.
- **D-B5 (cascade of retractions):** When a downstream row is retracted, its OWN `contributing_inputs` list is walked: any row downstream of this row also needs retraction. Bounded by pipeline depth (typically 2–3 hops in fraud workloads). Guard: a per-ShardOp `retraction_depth` counter caps fan-out at 16 hops and raises `BeavaError::RetractionDepthExceeded` — alarms via `beava_retraction_depth_exceeded_total`.

### Area C — Late Retractions & history_ttl

- **D-C1 (window):** Retractions apply only to events whose `primary_event_id` is within `history_ttl` of the current watermark on the target shard. Events older than that are not retractable — the retraction skips with a `tracing::warn!("RetractionBeyondHistory: ...")` and increments `beava_retraction_beyond_history_total{operator}`.
- **D-C2 (history_ttl source):** Read from `StreamDefinition.history_ttl` (existing config, no new knob). Default unchanged from P55.
- **D-C3 (warning dedup):** Warnings deduped per (operator, retract_reason_class) at 60s cadence (same pattern as Phase 51 /debug/warnings). Surface via `/debug/warnings.retraction_beyond_history`.

<specifics>
## Specific Ideas

- **Symmetry with P56 cross-shard dispatch:** Retraction dispatch should read structurally identical to `ssj_insert_at_shard` — same 5 lines shape (same-shard fast path, accumulate per-target, coalesce, try_send, await oneshot).
- **Idempotency is non-negotiable:** Both source-side retries and fan-out duplicates can deliver the same RetractDownstream twice. Target-side must no-op on already-retracted rows — no partial-state visible.
- **Cascade bound:** 16-hop depth cap is generous for fraud workloads (typical 2–3 hops). The cap is a safety-net for accidentally self-referential pipelines, not a hard constraint on real use.

</specifics>
