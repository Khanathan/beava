## Implementation Decisions

### D-01 — Wire-spec rigor: Markdown + JSON Schema + per-endpoint JSON examples

**Decision.** `docs/wire-spec.md` is authored as Markdown prose + frame layout diagrams. **Each of the 6 core opcodes** (REGISTER, PUSH, GET, BATCH_GET, RESET, PING) ships:

1. A **JSON Schema** (machine-readable, Draft 2020-12 or compatible) for both request and response bodies.
2. **1–2 worked JSON examples** per endpoint, committed as `examples/wire/<opcode>-{request,response}.json` files (and `examples/wire/<opcode>-error-<code>.json` for the principal error responses).

**Why:** SDK porters in Phase 13.6 (TypeScript + Go) can write a JSON-Schema-based validator that immediately rejects malformed requests/responses against the contract. This eliminates the "Slack message" rate during implementation. Aligns with TDD/contract-first preference. The schemas are live-checkable artifacts (`ajv-cli` or equivalent) that the engine in 13.4 also validates against.

**How to apply:**

- Schemas live under `examples/wire/schemas/<opcode>.{request,response}.schema.json` — treated as part of the spec, not an afterthought.
- Examples live under `examples/wire/<opcode>-<scenario>.json` — naming scheme: `register-fraud-team.request.json`, `register-conflict.error.json`, `push-success.response.json`, etc.
- 13.4 (engine) MUST validate inbound JSON against schemas in CI via a test that loads `examples/wire/schemas/*.schema.json` and asserts every `examples/wire/*.json` validates.
- 13.5/13.6 SDK porters use the schemas + examples as the source of truth (NOT the Python source, which is the canonical implementation but not the canonical contract).

### D-02 — Operator catalog: one page per op (53 pages)

**Decision.** `docs/operators/<op-name>.md`, one file per operator, total 53 files across 8 family subdirectories: `core/` (8) / `sketch/` (5) / `point-ordinal-recency/` (15: 11 point/ordinal + 4 recency, including `streak/max_streak/negative_streak`) / `decay/` (7) / `velocity-trend/` (9: 8 velocity + 1 z-score) / `bounded-buffer/` (7) / `geo/` (6) / `zscore/` (1).

**Each op page contains** (mandatory section template):
- **Signature** — Python signature with all kwargs, defaults, and types
- **Semantics** — what it computes, mathematically and informally
- **Return type** — scalar / list / dict / windowed (Vec<bucket>)
- **Complexity** — per-event CPU (O(...)), memory per entity (bytes), cross-link to `docs/operators/cost-class.md`
- **Worked example** — 1–2 short Python snippets, plus the JSON wire form (cross-linked to `examples/wire/`)
- **Edge cases** — empty stream, NaN inputs, lifetime mode (`windowed=` omitted), structured-error code if applicable
- **Family page** — short overview at `docs/operators/<family>/index.md`; index.md for each family lists all members with one-line summaries and links

**Why:** Highest devex per `project_v2_devex_first` — feature engineers (target: Priya, YC fintech CTO) google `bv.percentile` and land directly on a self-contained page. SEO-friendly: each op gets its own URL on beava.dev. Matches DuckDB / Polars conventions. Family overview pages give SDK porters a quick scan; per-op pages give end users depth. Total surface: ~5–7 KLOC docs, but mechanical authoring (signature extraction from `python/beava/_agg.py` + boilerplate + 1 worked example).

**How to apply:**

- Plan one wave per family OR one plan per family (planner's call) so authoring can parallelize.
- Signature extraction: planner produces a one-shot intermediate file (`docs/operators/_signatures.tsv` or similar) that downstream per-op writers fill in narrative around. Alternatively, planner generates 53 pages from a template in one wave, then per-family waves polish.
- `docs/operators.md` (the single mega-table) is REMOVED. Replaced by `docs/operators/index.md` (master list with per-family subsections + a one-row-per-op summary table at the top that links to per-op pages).
- Cross-links: each op page links to its wire JSON example, its Python source line, and the cost-class entry.

### D-03 — Examples & demos: runnable against language-local mock during 13.0; re-verified against real engine post-13.4

**Decision.** All 9 vertical demo files (`examples/{python,typescript,go}/{adtech,fraud,ecommerce}.{py,ts,go}`) are **runnable** files. During Phase 13.0:

### D-04 — Pipeline-DSL compilation rules: per-chain-operator worked examples + ambiguity matrix

**Decision.** `docs/pipeline-dsl/compilation-rules.md` ships:

1. **One worked example per chain method** — Python source → JSON wire → server-side semantics. Methods covered:
   - `events.filter(expr)` — single + chained filters
   - `events.select(*cols)`
   - `events.drop(*cols)`
   - `events.rename(**mapping)`
   - `events.with_columns(**mapping)`
   - `events.cast(**type_map)`
   - `events.fillna(**fill_map)`
   - `events.group_by(*keys)` (returns `GroupBy`, intermediate)
   - `groupby.agg(name=bv.<op>(...), ...)` (returns derivation)
   - `bv.col(...)` operator overloading (`+ - * /`, comparisons, `& | ~`, `.isnull()`, `.cast()`, `.alias()`)
   - `windowed=` kwarg semantics (lifetime mode when omitted, sliding-window when present)
   - `@bv.table` aggregation-output decorator (function-form per ADR-001)

2. **Ambiguity matrix** — explicit ALLOWED / FORBIDDEN / UNDEFINED rulings on edge cases:
   - Chained filters: `e.filter(a).filter(b)` — equivalent to `e.filter(a & b)`? (ALLOWED, equivalent)
   - `select` before `group_by`: trims columns before keying — ALLOWED
   - Expressions in `agg(name=...)` — `agg(big_purchases=bv.count(filter=bv.col("amount") > 100))` — ALLOWED, filter expr compiles to JSON
   - Multi-agg: `agg(a=bv.count(), b=bv.sum("x"))` — ALLOWED
   - `bv.col` arithmetic in agg argument: `bv.sum(bv.col("x") * 2)` — ALLOWED
   - Boolean-sum trick: `bv.sum(bv.col("is_fraud").cast(int))` — DOCUMENTED as the recommended pattern for conditional counts
   - `with_columns` after `group_by`: FORBIDDEN (group_by returns GroupBy, not Event/Table)
   - Nested `group_by`: FORBIDDEN
   - Cross-event aggregation: FORBIDDEN per `project_redis_shaped_no_event_time_ever`

**Why:** SDK porters writing the TS+Go compilers (Phase 13.6) need unambiguous rules. The Python compiler (already exists in `python/beava/_agg.py` + `_col.py`) is the reference but isn't a spec. Without an ambiguity matrix, every porter has to reverse-engineer the same edge cases independently. Aligns with `feedback_verify_plan_decisions` — explicit per-rule decisions are auditable and testable.

**How to apply:**

- Each worked example shows: Python source code (5–10 lines) → JSON wire (the full register payload for that descriptor) → 1-line note on server-side semantics.
- Ambiguity matrix lives as a Markdown table at the bottom of `compilation-rules.md` with columns: Pattern / Verdict / Rationale / Test fixture link.
- Test fixtures: each ALLOWED row links to a JSON file in `examples/wire/` (or `examples/dsl/`) showing the compiled output. Each FORBIDDEN row links to the structured error code SDK porters MUST raise.
- ~12–15 worked examples + ambiguity matrix of ~20 rows.

DuckDB and Polars ship per-op pages and are the reference voice (`feedback_beava_website_voice`).
- **D-03 mock + post-13.4 re-verification doubles as integration regression test.** If a 13.0 example breaks against the real engine post-13.4, that's a contract bug — surfaces immediately rather than at v0.0.0 ship.
- **D-04 ambiguity matrix is auditable.** Each row links to a fixture or error code; planner-can-test-it; SDK porters can grep the matrix during their compiler work.
- **`@bv.table` is REVIVED in v0 as aggregation-output only** (function-form `@bv.table(key=...) def Name(ev) -> bv.Table: return ev.group_by(...).agg(...)`).
