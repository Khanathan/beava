### docs/concepts/events-vs-tables.md (~120 LOC)

H1: `# Events vs Tables`

Sections:
- ## Overview — beava has two kinds of nodes in the registered DAG: events (immutable append-only streams) and tables (aggregation output)
- ## @bv.event — declares an event source (input); class form + function form
- ## @bv.table — declares an aggregation output (per ADR-001 partial overturn); function form ONLY in v0; wraps a `events.group_by(...).agg(...)` chain into a named keyed entity
- ## What @bv.table is NOT in v0
  - NOT a mutable upserted store (no `app.upsert(table, key, ...)`)
  - NOT a tombstoned delete store (no `app.delete(...)`)
  - NOT a temporal MVCC table (no time-travel queries)
  - NOT a retraction-aware aggregation (no retraction propagation)
  - These were stripped Phase 12.7; ADR-001 ONLY revives the aggregation-output use case
- ## When to use @bv.event vs @bv.table
  - Push semantics: events arrive via `app.push(event_name, fields)`; tables are READ-ONLY (computed)
  - Read semantics: `app.get(table_name, key)` returns row-shape dict for the table; events are not directly queryable
  - Lifecycle: events accumulate in WAL + are processed; tables hold per-entity aggregation state
- ## Cross-references
  - [ADR-001](../../../.planning/decisions/ADR-001-bv-table-partial-overturn.md)
  - CLAUDE.md `§ Events-Only Invariant` (locked Phase 12.7)
  - [docs/pipeline-dsl/overview.md](../pipeline-dsl/overview.md)

### docs/concepts/embed-mode.md (~80 LOC)

H1: `# Embed Mode`

Sections:
- ## Overview — `bv.App()` with no URL spawns a local `beava` binary as a subprocess on ephemeral ports
- ## Binary discovery — 4-step search per `python/beava/_embed.py::discover_binary()`: `BEAVA_BINARY` env var → `PATH` → `./target/debug/beava` → `BinaryNotFoundError`
- ## Lifecycle — spawn at App creation; teardown at `app.close()` or context-manager exit; subprocess stdout/stderr captured for debugging
- ## When to use
  - Quickstart / `bv.demo()` (no server install needed)
  - pytest fixtures (`bv.test.fixture(reset_each=True)`)
  - Notebooks / scripts (no remote infrastructure)
- ## When NOT to use
  - Production (use `tcp://` or `http://` to a running `beava` server)
  - Multi-process workloads (each process spawns its own embed)
  - Persistent state (embed mode resets when the App closes; for persistence use `bv.App(persist_dir=...)` per Phase 13.4)
- ## Cross-references — sdk-api/shared.md transports section, sdk-api/python.md App.__init__

### docs/concepts/lifetime-aggregation.md (~100 LOC)

H1: `# Lifetime Aggregation`

Sections:
- ## What lifetime aggregation means — when an aggregation operator omits the `window=` kwarg, it accumulates over the entity's entire history (no rollover; one bucket; lifetime mode)
- ## When to use lifetime mode — first_seen, last_seen, age, has_seen are typically lifetime; first_n / last_n / lag with required `n` kwarg also work in lifetime mode
- ## V0-MEM-GOV-02 contract — every lifetime aggregation operator declares a finite memory ceiling at register time per `[V0-MEM-GOV-02](../../../.planning/REQUIREMENTS.md)`
- ## Memory bound classifications:
  - `O1`: count, sum, mean, min, max, var, std, ratio, first, last, has_seen, first_seen, last_seen, age, time_since, streaks, decay-family, etc.
  - `BoundedSketch`: n_unique (HLL), quantile (DDSketch), bloom_member
  - `BoundedByRequiredKwarg("n")`: first_n, last_n, lag, time_since_last_n, most_recent_n
  - `BoundedByConfig("max_categories", 256)`: entropy, event_type_mix
  - `BoundedByConfig("k", 10)`: top_k
  - `BoundedByConfig("samples", 100)`: distance_from_home
- ## Register-time enforcement
  - JSON-prelude shim in `crates/beava-core/src/register_validate.rs::pre_check_unbounded_op_in_lifetime_mode` returns structured error `unbounded_op_in_lifetime_mode` if a required kwarg is missing
  - Default-ON via `BEAVA_MEMORY_GOV_ENFORCE` env-gate (escape hatch: `BEAVA_MEMORY_GOV_ENFORCE=0`)
- ## Cross-references — CLAUDE.md `§ Memory Governance Invariant`, REQUIREMENTS.md V0-MEM-GOV-02, docs/operators/index.md

### docs/concepts/processing-time-only.md (~80 LOC)

H1: `# Processing-Time Only (No Event-Time)`

Sections:
- ## Overview — beava's time semantics are server-side processing time (`now_ms()` at apply); event-time is permanently rejected per `project_redis_shaped_no_event_time_ever` (locked 2026-04-30)
- ## What this means
  - All windowed ops use server-side `now_ms()` for bucket assignment
  - Recency ops return server arrival timestamps (NOT event timestamps)
  - No `event_time_ms` field on the wire; no `event_time_field` decorator kwarg; rejecting these returns `code="event_time_not_supported_in_v0"`
  - No watermarks, no late-event handling, no out-of-order reordering
- ## Why no event-time
  - Permanent architectural commitment — Beava is "Redis-shaped, processing-time only"
  - Event-time + watermarks add complexity that doesn't justify the v0 scope
  - For backfill / historical replay use cases, see `.planning/ideas/v0.1-historical-extraction-engine.md` (deferred to v0.1+)
- ## Implications for users
  - Push events as soon as they happen (don't batch-replay historical events)
  - If you must replay history, the result reflects current `now_ms()` not the event's original timestamp
  - For event-time semantics, wait for v0.1+ historical extraction engine
- ## Cross-references — CLAUDE.md (Memory) `project_redis_shaped_no_event_time_ever`, [docs/error-codes.md](../error-codes.md) `event_time_not_supported_in_v0`

### docs/architecture/single-thread-apply.md (~110 LOC)

H1: `# Single-Threaded Apply`

Sections:
- ## Overview — beava processes ALL events on a single OS thread (the apply loop); auxiliary threads handle WAL fsync, HTTP accept, snapshot writer
- ## Why single-thread
  - Per `project_no_sharded_apply` (locked 2026-04-26): rejecting in-process apply sharding (Phase 13.3 was REJECTED)
  - For higher per-instance throughput, run multiple beava instances sharded at the entity-key level (Redis-cluster pattern)
  - Per-instance ceiling: ~1M EPS for simple counters, ~400k for medium aggregations on Linux Xeon
- ## Apply loop responsibilities
  - Receive pushed events (mio event loop polls TCP + HTTP listeners)
  - For each event: validate against registered schema, apply to all affected derivations, update per-entity state
  - WAL append happens inline (synchronous fsync handed off to spawn_blocking thread per Phase 13.1)
- ## What the apply loop does NOT do
  - No async / no tokio in the data plane (per `project_phase18_no_dual_runtime`)
  - No locks on hot path (per-entity HashMap + Arc<EventDescriptor>)
  - No cross-entity coordination (cross-entity ops are out of scope per Phase 25 deferral)
- ## Cross-references — CLAUDE.md `§ mio-only Hot-Path Invariant`, mio-data-plane.md

### docs/architecture/mio-data-plane.md (~110 LOC)

H1: `# mio Data Plane`

Sections:
- ## Overview — mio is the only data-plane runtime; admin sidecar (tokio/axum) serves /health, /ready, /metrics, /registry on cfg.admin_addr
- ## Why mio (not tokio for data plane)
  - Per `project_phase18_no_dual_runtime` (locked 2026-04-23): hand-rolled hot path, no dual stack
  - Phase 12.6 deleted legacy axum data plane (~7,475 LOC); architectural test `phase12_6_mio_only_dataplane.rs` enforces this at CI level
- ## What mio does
  - Polls TCP + HTTP listeners on a single OS thread
  - Per-tick lifecycle: poll → distribute_reads → join → apply → distribute_writes → join (per Phase 18-04.7 IoPool design)
  - Routes inbound frames to `crates/beava-server/src/apply_shard.rs::dispatch_one` (single hot-path entry)
- ## Admin sidecar
  - tokio/axum runtime bound on cfg.admin_addr (separate port from data-plane)
  - Routes: GET /health, GET /ready, GET /metrics, GET /registry
  - The ONLY legitimate axum use site post-12.6
- ## How enforced — `crates/beava-server/tests/phase12_6_mio_only_dataplane.rs` walks workspace and fails if forbidden symbols (Router/Json/Extension/extract/routing/http/body) appear outside http_admin.rs
- ## Cross-references — CLAUDE.md `§ mio-only Hot-Path Invariant (locked Phase 12.6)`, single-thread-apply.md

### docs/architecture/wal-snapshot.md (~120 LOC)

H1: `# WAL + Snapshot`

Sections:
- ## Overview — write-ahead log (WAL) for crash recovery + periodic snapshots for fast restart
- ## WAL format (post-12.7 RESET)
  - `FORMAT_VERSION = 1` (`crates/beava-persistence/src/record.rs`)
  - Frame: `[u32 length][u32 crc32c][u64 lsn][u8 record_type][payload]`
  - Magic header: `b"BEAVAWAL"`
  - RecordType variants (post-12.7): `Event = 0x01`, `RegistryBump = 0x02` (Table* variants stripped)
- ## Snapshot format (post-12.7 RESET)
  - `SNAPSHOT_FORMAT_VERSION = 1` (header)
  - `SNAPSHOT_BODY_FORMAT_VERSION = 1` (body)
  - Snapshot writes complete in-memory state + registry to disk
  - Periodic (default 30s, configurable)
- ## Recovery on boot
  - Load latest snapshot (LSN N)
  - Replay WAL from LSN N+1 to present
  - RTO target: < 30s at 10GB state on NVMe
- ## Sync modes (Phase 6.1)
  - `SyncMode::Periodic` (default; group-commit fsync every 1-5ms or 1MB; acks=1 Kafka-style; ~15× EPS lift)
  - `SyncMode::PerEvent` (acks=all via `/push-sync`)
- ## Cross-references — CLAUDE.md (Constraints), Phase 6 + 7 SUMMARYs, wire-spec.md ack_lsn semantics

### docs/architecture/memory-budget.md (~150 LOC)

H1: `# Memory Budget`

Sections:
- ## Overview — beava is in-memory only (no SSD overflow); users size their box; budget is ~7 KB per entity for a rich 30-feature pack
- ## Verified Phase 12.9 numbers (CITE EXPLICITLY)
  - `size_of::<AggOp>() = 80 B` (post-Phase 12.9 boxing of 7 fat variants)
  - Verified by CI tripwire `crates/beava-core/tests/per_entity_size_dump.rs::aggop_size_within_cap`
  - fraud-team weighted-avg per-entity ~6 KB (was ~22 KB pre-12.9 boxing); clears the 7 KB budget with headroom
- ## Per-entity memory math
  - For a 30-feature aggregation on a single entity:
    - 30 × `size_of::<AggOp>` (= 80 B) = 2.4 KB for the agg-op tags
    - + per-op state (varies by op family — see [docs/operators/cost-class.md](../operators/cost-class.md))
    - + entity-key overhead (~16 B for SmallVec<[u8; 16]>)
    - + HashMap bucket overhead (~24 B per entry in hashbrown)
  - Total: ~3-7 KB depending on op mix
- ## Capacity calc — 7 KB × 100M entities = ~700 GB → fits on a 1 TB box with headroom
- ## What CAN blow the budget
  - Lifetime aggregations without bounded kwargs (V0-MEM-GOV-02 enforces this at register-time)
  - Cold entities accumulating without TTL (V0-MEM-GOV-01 — opt-in `cold_after=` kwarg)
  - Sketch ops with very high cardinality (DDSketch / HLL state grows by log factor; bounded but real)
- ## V0-MEM-GOV invariants (cite all 3)
  - V0-MEM-GOV-01: opt-in `@bv.event(cold_after=...)` for cold-entity TTL eviction
  - V0-MEM-GOV-02: every lifetime op declares a finite per-entity memory ceiling at register-time
  - V0-MEM-GOV-03: per-event bucket reclaim within active entities (always-on Tier 2)
- ## /metrics observable — 5 Prometheus metric families (Phase 12.8 Plan 06):
  - `beava_cold_entity_evictions_total`, `beava_lifetime_op_cap_hit_total`, `beava_entity_count_resident`, `beava_bucket_reclaim_total`, `beava_bytes_per_entity_p99` (gauge static = 7000 placeholder per Phase 12.8 PLANNER-SURFACED CONCERN; dynamic sampling deferred to Phase 13.4)
- ## Cross-references — CLAUDE.md `§ Memory Governance Invariant`, REQUIREMENTS.md V0-MEM-GOV-01/02/03, observability.md
