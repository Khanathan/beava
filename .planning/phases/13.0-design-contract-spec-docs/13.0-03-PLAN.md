H1: `# Beava HTTP API`

## Overview

1-2 paragraphs:
- v0 ships HTTP/1.1 + JSON for curl/LB/WAF reach
- All data-plane endpoints are POST + JSON body (verb-style routes; no path-query lookup; lookup args go in body)
- Admin sidecar (health/readiness/metrics/registry) is on a SEPARATE port (`cfg.admin_addr`) per Phase 12.6 mio-only invariant
- Cross-reference: this doc is the HTTP transport spec; the wire-level body shapes live in `docs/wire-spec.md`. Every route in this doc has a corresponding `OP_<opcode>` in the wire spec
- Note that 13.0 declares the post-13.4 target state; the current code (post-12.7) uses event-name-suffixed routes like `POST /push/{event_name}`. 13.4 renames to verb-style `POST /push` with `event_name` in the body

## Authentication / Headers

- v0 has NO authentication — public OSS launch is unauthenticated
- `Content-Type: application/json` is REQUIRED on all data-plane endpoints; mismatch returns 415
- `X-Trace-Id` header is OPTIONAL; propagated to logs
- No CORS headers in v0 (single-origin assumption); cross-origin support is v0.1+

## Data plane endpoints

For each of the 6 endpoints (`/register`, `/push`, `/get`, `/batch_get`, `/reset`, `/ping`), include this table format:

```
### POST /<route>

| Field | Value |
|-------|-------|
| Method | POST |
| Path | `/<route>` |
| Wire opcode | `OP_<NAME>` (`0x<hex>` per docs/wire-spec.md) |
| Content-Type | `application/json` |
| Auth | None (v0) |

**Request body:** see [docs/wire-spec.md `OP_<NAME>` request schema](./wire-spec.md#op_<name>) — fully specified at [examples/wire/schemas/<name>.request.schema.json](../examples/wire/schemas/<name>.request.schema.json).

**Response body (success):** see [docs/wire-spec.md `OP_<NAME>` response schema](./wire-spec.md#op_<name>) — fully specified at [examples/wire/schemas/<name>.response.schema.json](../examples/wire/schemas/<name>.response.schema.json).

**HTTP status codes:**

| Status | When |
|--------|------|
| 200 | Success |
| 400 | Validation error (malformed JSON, schema mismatch, etc.) |
| 404 | Unknown endpoint |
| 409 | Registration conflict (additive-only violation; suggest force=true) |
| 415 | Wrong Content-Type |
| 500 | Server error (internal) |

**Curl example:**

\`\`\`bash
curl -X POST http://localhost:7380/<route> \\
  -H 'Content-Type: application/json' \\
  -d @examples/wire/<name>-<scenario>.request.json
\`\`\`

**Errors specific to this endpoint:**

| Code | When | Recovery |
|------|------|----------|
| ... | ... | ... |
```

Specific endpoint contents:

### POST /register

- Wire opcode: OP_REGISTER (0x0001)
- Body: descriptors array + `force?` + `dry_run?` flags
- Response: `{status, registry_version, added, removed?, changed?, diff?}`
- Errors: `registration_conflict` (409) / `schema_mismatch` (400) / `cycle` (400) / `missing_upstream` (400) / `unsupported_node_kind` (400)
- Cite ADR-001 for `kind: "table"` aggregation-output payloads
- Curl example uses `examples/wire/register-fraud-team.request.json`

### POST /push

- Wire opcode: OP_PUSH (0x0010)
- Body: `{event_name, fields: {...}}` — note: in 13.4 the event_name moves into the body (was URL-suffix in pre-13.4 routes)
- Response: `{ack_lsn, registry_version}` (acks=1 per Phase 6.1 SyncMode::Periodic)
- Errors: `unknown_event` (400), `schema_mismatch` (400), `validation_failed` (400)
- Curl example uses `examples/wire/push-success.request.json`

### POST /get

- Wire opcode: OP_GET (0x0020)
- Body: `{table, key, features?}`
- Response: row-shape dict (cold-start returns `{}`, NOT 404 — empty result is success)
- Errors: `unknown_table` (400), `invalid_key` (400)
- Cite that key may be string OR array (composite key)

### POST /batch_get

- Wire opcode: OP_BATCH_GET (0x0024)
- Body: `{requests: [{table, key, features?}, ...]}` — heterogeneous (different tables across entries supported)
- Response: `{results: [row-shape dict, ...]}` (ordering matches request; per-entry cold-start is `{}`, not error)
- Errors: `batch_too_large` (400) if total entries > server cap (planner discretion: 10000 per current SRV-API-08 — confirm via grep of register_validate.rs or similar)

### POST /reset

- Wire opcode: OP_RESET (0x0040)
- Body: `{}` empty
- Response: `{status: "ok"}` after WAL truncation completes
- Cite: DESTRUCTIVE — wipes in-memory state + WAL; useful only for tests
- Errors: typically none (server-side fault returns 500)

### POST /ping

- Wire opcode: OP_PING (0x0000)
- Body: `{}` empty (or absent)
- Response: `{server_version, registry_version}`
- Errors: typically none

## Admin sidecar endpoints (separate port, GET-shaped)

Document that the admin sidecar binds on `cfg.admin_addr` (NOT the data-plane port) per Phase 12.6 mio-only invariant; admin lives on tokio/axum.

```
### GET /health

| Field | Value |
|-------|-------|
| Method | GET |
| Path | /health |
| Port | cfg.admin_addr |
| Auth | None (v0) |

Response: 200 OK with body `{status: "ok"}` when server is up. Cheap probe; does NOT confirm registry is loaded.

### GET /ready

200 only after recovery is complete (snapshot loaded + WAL replayed). 503 during recovery.

### GET /metrics

Prometheus exposition format. See `docs/architecture/observability.md` (Plan 13.0-13) for metric family list. Includes the 5 Phase 12.8 memory-governance counters.

### GET /registry

Registry-version snapshot. Optional `?version=N` query for a historical version (debugging).
```

## Error envelope format

Every 4xx and 5xx response carries the standard error envelope per `examples/wire/schemas/error.schema.json`:

```json
{
  "code": "<structured-code>",
  "path": "<JSON-or-DAG-path>",
  "message": "<forward-looking framing per Phase 12.7 D-02>"
}
```

Cite `docs/error-codes.md` (Plan 13.0-12) as the canonical structured-code list with HTTP status mappings.

## Connection lifecycle

- Single TCP connection per `App` instance + auto-reconnect on drop (no pooling in v0)
- Keepalive: respect server's TCP keepalive settings; v0 doesn't pin a value
- Idempotency: per Phase 6.1, push idempotency via `dedupe_key` + `dedupe_window` registered on the event source

- Pre-13.4 (current state): `POST /push/{event_name}` style routes (event_name in path)
- Post-13.4 (this doc's target): `POST /push` with event_name in body — verb-style (Polars/DuckDB API convention)
- 13.4 ships the rename mechanically; 13.0 documents the post-13.4 target
