## Overview
1-2 paragraphs explaining: dual transport (HTTP/JSON + custom-framed TCP); Redis-style FIFO correlation; opcode discovery vs JSON kind discriminators; the spec is authoritative — `examples/wire/schemas/*.schema.json` are machine-validatable contracts.

## Frame Format

Document the TCP frame layout with a labeled byte diagram:

```
+--------+-----+--------------+--------------+
| length | op  | content_type | payload      |
| u32 BE | u16 | u8           | length-3     |
|  4 B   | 2 B | 1 B          | bytes...     |
+--------+-----+--------------+--------------+
```

State that:
- `length` covers op + content_type + payload (NOT itself); minimum frame is 8 bytes total (4-byte length prefix + 4-byte body when payload is empty)
- `content_type`: `0x01` = JSON, `0x02` = msgpack (TCP-only; v0 HTTP is JSON-only)
- Correlation: Redis-style strict-FIFO on a connection (NO request_id field)
- Errors: server replies with `OP_ERROR_RESPONSE = 0xFFFF` and a JSON error envelope payload
- HTTP transport: each opcode has a corresponding verb-style POST route (see `docs/http-api.md` Plan 13.0-03)

## Opcode Table

A markdown table listing opcodes (all 6 v0 ops + reserved) — exact values:

```
| Op name | Value | Direction | Body | Notes |
|---------|-------|-----------|------|-------|
| OP_PING | 0x0000 | client→server | `{}` | Health probe; response carries server_version + registry_version |
| OP_REGISTER | 0x0001 | client→server | DAG payload | Discriminated union: kind=event \| table \| derivation |
| OP_PUSH | 0x0010 | client→server | `{event_name?, fields}` | Sync push; ack=1 (Kafka-style); body schema below |
| OP_GET | 0x0020 | client→server | `{table, key, features?}` | Returns row-shape (dict of feature → value) |
| OP_BATCH_GET | 0x0024 | client→server | `[{table, key}, ...]` | Heterogeneous batch lookup; per-request response order matches request order |
| OP_RESET | 0x0040 | client→server | `{}` | Wipes all in-memory state + truncates WAL; useful for tests |
| OP_GET_RESPONSE | 0x0023 | server→client | row-shape body | Response for OP_GET / OP_BATCH_GET |
| OP_ERROR_RESPONSE | 0xFFFF | server→client | error envelope | Universal error reply |
| OP_PUSH_SYNC | 0x0011 | RESERVED (not v0) | — | Reserved for future acks=all push (v0.1+) |
| OP_PUSH_MANY | 0x0012 | RESERVED (not v0) | — | Reserved for future batch push (v0.1+) |
| Future set/mset | 0x0030-0x003F | RESERVED | — | Reserved range for future direct-feature-write opcodes |
```

For `OP_REGISTER`, the request body is a discriminated union on `kind`:
- `kind: "event"` — register a `@bv.event` source descriptor (schema fields, optional `keep_events_for`, `cold_after`, `dedupe_key/dedupe_window`)
- `kind: "table"` — register a `@bv.table(key=...)` aggregation-output node (per ADR-001; output_kind=table)
- `kind: "derivation"` — register a derived event/table node (filter/select/with_columns/group_by chains; `output_kind` field disambiguates event vs table; `ops` array carries the chain)

For `OP_PUSH`:
- Body: `{fields: object}` — flat dict of field-name → value matching the registered event schema
- The event_name comes from the URL path (HTTP) or a separate header byte (TCP framed body) — represent the wire body as `{fields: {...}}`
- Cite the ADR-002 op-rename note: pushed events are events, not aggregations, so op renames don't affect push body shapes

For `OP_GET`:
- Body: `{table: string, key: string | array, features?: string[]}`
- Response: `{...}` row-shape — one feature → value pair per available feature; cold-start returns `{}`

For `OP_BATCH_GET`:
- Body: `[{table, key}, ...]` heterogeneous list (different tables across entries supported)
- Response: list-of-dicts, ordering matches request order; per-entry cold-start is `{}` not error
- Cite that this opcode is NEW in v0 (post-12.7) per ROADMAP §13.4

For `OP_RESET`:
- Body: `{}` (empty object)
- Response: `{status: "ok"}` after WAL truncation completes
- Use case: testing fixtures (`bv.test.fixture` resets between tests)
- Note that OP_RESET is destructive; the spec doc reminds operators to call it only on a beava instance bound to test data

For `OP_PING`:
- Body: `{}`
- Response: `{server_version, registry_version}`

```python
#!/usr/bin/env python3
"""
Phase 13.0 wire-spec validation harness.

Loads every JSON Schema in examples/wire/schemas/ and asserts every
worked example in examples/wire/<name>-<scenario>.<request|response|error>.json
validates against its corresponding schema.

Filename → schema inference:
- examples/wire/<op>-<scenario>.request.json → examples/wire/schemas/<op>.request.schema.json
- examples/wire/<op>-<scenario>.response.json → examples/wire/schemas/<op>.response.schema.json
- examples/wire/<op>-<scenario>.error.json → examples/wire/schemas/error.schema.json (shared envelope)
- examples/wire/<op>-request.json (no scenario tag) → examples/wire/schemas/<op>.request.schema.json
- examples/wire/<op>-response.json (no scenario tag) → examples/wire/schemas/<op>.response.schema.json

Exits 0 if all examples validate; exits 1 with diagnostics otherwise.

Used as a CI gate by Phase 13.4's Rust validation harness (different
implementation, same fixtures).
"""
import json
import pathlib
import sys

try:
    from jsonschema import Draft202012Validator
except ImportError:
    print("Install jsonschema: pip install 'jsonschema>=4.0'", file=sys.stderr)
    sys.exit(2)


WIRE_DIR = pathlib.Path(__file__).parent
SCHEMAS_DIR = WIRE_DIR / "schemas"


def schema_for_example(example_path: pathlib.Path) -> pathlib.Path | None:
    """Infer the schema path for a given example file."""
    name = example_path.stem  # e.g. "register-fraud-team.request"
    if name.endswith(".error"):
        return SCHEMAS_DIR / "error.schema.json"
    # Pattern: <op>-<scenario>.<kind>
    if "." not in name:
        # e.g. "ping-request" or "reset-response"
        if name.endswith("-request"):
            op = name[: -len("-request")]
            return SCHEMAS_DIR / f"{op}.request.schema.json"
        if name.endswith("-response"):
            op = name[: -len("-response")]
            return SCHEMAS_DIR / f"{op}.response.schema.json"
        return None
    base, kind = name.rsplit(".", 1)  # "register-fraud-team", "request"
    op = base.split("-", 1)[0]  # "register"
    return SCHEMAS_DIR / f"{op}.{kind}.schema.json"


def main() -> int:
    errors: list[str] = []
    examples = sorted(p for p in WIRE_DIR.glob("*.json") if p.parent == WIRE_DIR)
    if not examples:
        print("No examples found under examples/wire/")
        return 1

    for example_path in examples:
        schema_path = schema_for_example(example_path)
        if schema_path is None or not schema_path.exists():
            errors.append(
                f"NO SCHEMA for {example_path.name} (looked for {schema_path})"
            )
            continue
        with open(schema_path) as f:
            schema = json.load(f)
        with open(example_path) as f:
            instance = json.load(f)
        validator = Draft202012Validator(schema)
        validation_errors = sorted(
            validator.iter_errors(instance), key=lambda e: e.path
        )
        if validation_errors:
            for ve in validation_errors:
                path_str = "/".join(str(p) for p in ve.path) or "<root>"
                errors.append(
                    f"INVALID {example_path.name} (schema={schema_path.name}) "
                    f"at {path_str}: {ve.message}"
                )

    if errors:
        print("FAIL — wire-spec example validation:", file=sys.stderr)
        for e in errors:
            print(f"  {e}", file=sys.stderr)
        return 1

    print(f"OK — all {len(examples)} examples validate against their schemas")
    return 0


if __name__ == "__main__":
    sys.exit(main())
```
