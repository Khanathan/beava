broken=0

        # Strip ../ traversals to a final path relative to repo root

done

# ADR-003: First-class global aggregation + public `bv.lit` export

## Status

ACCEPTED 2026-05-03 (Phase 13.0 mid-execution scope addition).

Implementation deferred to Phase 13.4 (engine sentinel routing — ~30 LOC) + Phase 13.5 (Python SDK — ~110 LOC: bv.lit export + global-table arity overload + group_by() empty allowance) + Phase 13.6 (TS + Go SDK overloads).

## Context

Phase 13.0 mid-execution (2026-05-03) the user surfaced two API gaps not covered by any prior decision:

1. **No global aggregation.** Beava v0 currently rejects `events.group_by()` (empty args) at the SDK layer (`python/beava/_events.py:170-172`) with `ValueError: "group_by() requires at least one key"`. `@bv.table(key=...)` requires a key. There is NO way for users to express "total events / sec across all entities", "p95 latency globally", "top 10 hottest pages across the platform" — every aggregation is per-entity.
   - Workaround would be: add a literal column with `bv.lit("global")`, group_by it. But `bv.lit` is also not exported.
   - Real use cases blocked: monitoring dashboards (total throughput, current entity count, global p95), anomaly detection (is the GLOBAL rate spiking?), operator-facing health summaries.
   - First-impression criticism risk for v0 launch: "you can't even count total signups today?"

2. **`bv.lit(value)` not in public namespace.** The `_Literal` AST node exists at `python/beava/_col.py:195` and is used internally by operator-overloading coercion (e.g., `bv.col("x") > 100` constructs an implicit `_Literal(100)`). But users cannot construct literals explicitly. Use cases blocked:
   - Constant columns: `events.with_columns(source=bv.lit("web"))`
   - Explicit literal coercion in expressions where ambiguity matters: `bv.col("a") / bv.lit(10.0)` (force float division)
   - Pattern compositions like the global-agg workaround.

Both gaps are small mechanical exposures of existing internal capability. Both have high user-visible value for monitoring and dashboard use cases. v0 launch ship-criticism is real.

## Decision

**Ship both `bv.lit` and first-class global aggregation in v0.**

### Decision A — Public `bv.lit(value)` export

Expose the existing internal `_Literal` AST node as a public factory function `bv.lit(value)`:

```python
# Public API:
def lit(value: int | float | str | bool | None) -> _Literal:
    """Construct an explicit literal expression for use in bv.col chains and with_columns()."""
    return _Literal(value)
```

Mirror surface in TypeScript SDK (`bv.lit(value)`) and Go SDK (`bv.Lit(value)`). Wire-level: literals were already serialized via the existing operator-overloading path; no wire change required.

### Decision B — First-class global aggregation

Three coordinated SDK + engine + wire surface additions:

1. **SDK — Python (lands Phase 13.5):**
   - Allow `events.group_by()` with empty args (no `*keys`) → returns a `GroupBy` instance with empty `_keys`.
   - Add `events.agg(**aggs)` shorthand (no `group_by` required) — equivalent to `events.group_by().agg(**aggs)`.
   - Allow `@bv.table` decorator without `key=` kwarg → declares a global table (no per-entity dimension).
   - Add `App.get(table_name)` arity overload (no entity_id arg) → returns the global feature dict; `app.get("GlobalTable") == {}` until first event.
   - All 53 ops work with both per-entity and global aggregation — same op semantics, different state-keying dimension.

2. **Engine — Rust (lands Phase 13.4):**
   - Sentinel entity_id `""` (empty string) routes global-table state. Existing per-entity hashmap machinery handles it as "just another entity" with key = "".
   - Register-time validation: `kind="table"` with `key_cols=[]` is the wire-level signal for global. Validates that `key_cols` is either non-empty (per-entity) or empty (global) — never null.
   - GET path: `entity_id=""` on the wire returns the global state for that table.
   - Cold-start: same as per-entity (empty dict before first event).

3. **Wire spec — JSON (patches Phase 13.0 docs):**
   - Register payload: `{"kind": "table", "name": "GlobalCounter", "key_cols": [], "spec": {...}}` — empty `key_cols` array signals global.
   - GET request: `{"table": "GlobalCounter", "entity_id": ""}` — empty string entity_id retrieves global state.
   - Existing JSON Schemas need a one-line update to permit `key_cols: []` (already allowed by the `array` type; just clarify in description).
   - 3 new example fixtures: `register-global-counter.request.json`, `get-global.request.json`, `get-global.response.json`.
