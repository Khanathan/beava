# Decisions made during execution
decisions:
  - id: D-EXEC-1
    title: HTTP status table coverage scope
    decision: Per-endpoint status tables cover 200, the 4xx codes that apply (400, 403, 404, 409 as relevant), 415 (Content-Type mismatch — applies to all data-plane endpoints), and 500 (server error catch-all). 503 added to GET /ready (during recovery). Other 4xx/5xx that don't apply per-endpoint are omitted to keep the per-endpoint table tight; the canonical mapping lives at docs/error-codes.md (Plan 13.0-12, forward reference).
    rationale: Plan acceptance asks for "HTTP status mapping table" per endpoint; tightening to actually-applicable codes per route makes each table actionable rather than a copy-pasted boilerplate. The full alphabetical code → status mapping is owned by 13.0-12.
  - id: D-EXEC-2
    title: Curl examples reference real fixtures, not inline JSON
    decision: Each curl example uses `-d @examples/wire/<scenario>.request.json` to load a real fixture file. Where the post-13.4 form differs from what the existing fixture carries (push body needs `event_name` inside the body but the fixture only has `fields`), an additional `jq`-based example demonstrates the manual transform with explanation.
    rationale: Aligns with D-03 from the phase context — "examples are runnable; if a 13.0 example breaks against the real engine post-13.4, that's a contract bug." Prevents copy-paste rot. The two-form push example (jq + plain @file) flags the transient mismatch explicitly.
  - id: D-EXEC-3
    title: Pre-13.4 vs post-13.4 form documented inline rather than in a separate appendix
    decision: The pre-13.4 → post-13.4 migration is documented (a) as a top-of-doc paragraph clarifying the post-13.4 target framing, (b) in the POST /push section as a blockquoted note, and (c) as a dedicated "Note on event-name routing" section with side-by-side table + rationale bullets at the bottom. No separate appendix file.
    rationale: One-doc-one-topic. The migration is fundamental enough that readers need to see it at the start (so they understand the doc's framing), in context of /push (so they know which exact route is changing), and as a closing reference (so they have the rationale digest). An appendix would split the explanation across two files.
  - id: D-EXEC-4
    title: Admin sidecar /health and /ready back-compat note included
    decision: Document the data-plane port back-compat mirroring of /health and /ready (Plan 12.6-01 carry-over) explicitly under each admin endpoint, marking the canonical location as the admin sidecar. Note that the data-plane mirroring "may be removed in v0.0.x".
    rationale: Test files across the codebase (Phase 1 / 7 / etc.) currently poll `ts.base_url()` for readiness — operators reading this doc will hit the data-plane port if they follow legacy examples. Documenting the back-compat path explicitly avoids "but this works on /:7380/health, why does the doc say /:9090/health?" confusion. The "may be removed in v0.0.x" framing matches the existing comments in `crates/beava-runtime-core/src/router.rs` and signals operators not to depend on the data-plane mirroring.

# Phase 13.0 Plan 03: docs/http-api.md (verb-style POST routes) Summary

**One-liner:** Authored the canonical HTTP route table for v0 — 6 verb-style POST data-plane endpoints + 4 admin sidecar GET endpoints, declaring the post-13.4 target state with full per-endpoint status / curl / error coverage and cross-links to wire-spec.md and the JSON Schema fixtures.

## Goal

Per ROADMAP §13.4 Wave 1, v0 ships verb-style POST routes for all 6 data-plane operations (`/register`, `/push`, `/get`, `/batch_get`, `/reset`, `/ping`) with JSON body. The doc declares the **post-13.4 target state**; Phase 13.4 implements the route rename mechanically (mainly `POST /push/{event_name}` → `POST /push` with `event_name` in the body), and 13.0 documents what the routes will look like so downstream SDK ports (Plan 13.5 Python, Plan 13.6 TS + Go) can author against a stable contract.

The doc also documents the admin sidecar (4 GET endpoints on `cfg.admin_addr`) per the Phase 12.6 mio-only invariant — making it explicit that the data-plane port is hand-rolled mio + non-blocking I/O while admin lives on tokio + axum.

## What was built

`docs/http-api.md` (693 lines) authored from scratch. The previous `docs/http-api.md` (1137-line axum-era file flagged in 13.0-CONTEXT.md as STALE) was already nuked by an earlier plan; this doc is a clean rewrite, not a delta.

### Verbs and routes

| HTTP route | Wire opcode | Body |
|------------|-------------|------|
| `POST /register` | `OP_REGISTER` (`0x0001`) | `{descriptors: [...], force?, dry_run?}` |
| `POST /push` | `OP_PUSH` (`0x0010`) | `{event_name, fields: {...}}` (post-13.4) |
| `POST /get` | `OP_GET` (`0x0020`) | `{table, key, features?}` |
| `POST /batch_get` | `OP_BATCH_GET` (`0x0024`) | `{requests: [...]}` |
| `POST /reset` | `OP_RESET` (`0x0040`) | `{}` |
| `POST /ping` | `OP_PING` (`0x0000`) | `{}` |

### Admin sidecar (separate port `cfg.admin_addr`)

| HTTP route | Purpose |
|------------|---------|
| `GET /health` | Cheap liveness — admin server up; does NOT confirm registry loaded. |
| `GET /ready` | Readiness — `200` after recovery completes; `503` during recovery. |
| `GET /metrics` | Prometheus exposition format; lists 9 metric families exposed in v0. |
| `GET /registry` | Registry snapshot dump (current version + node count); `?version=N` reserved for v0.0.x. |
