Each H3 section has these subheadings:

```
### events.filter(expr)

#### Python source

```python
big = Txn.filter(bv.col("amount") > 100)
```

#### JSON wire

```json
{
  "kind": "derivation",
  "name": "BigTxn",
  "upstreams": ["Txn"],
  "ops": [
    {"op": "filter", "expr": "(amount > 100)"}
  ],
  "output_kind": "event"
}
```

#### Server semantics

Per-event predicate evaluation. Events with amount > 100 flow downstream; others are dropped. Schema unchanged. Composes left-to-right with subsequent ops. Chained filters compose as conjunction: `e.filter(a).filter(b)` is equivalent to `e.filter(a & b)`.
```

After all 12 methods, add the ## Boolean-sum trick subsection:

```
## Boolean-sum trick (recommended pattern for conditional counts)

Per [Q1 Path B locked answer](../phases/13.0-design-contract-spec-docs/13.0-CONTEXT.md), v0 keeps `bv.sum(field: str)` only — the field arg accepts a string column name, NOT an `_ExprAST`. To implement a "count where condition" semantic, use a two-stage pattern:

```python
@bv.event
class Txn:
    user_id: str
    is_fraud: bool

@bv.table(key="user_id")
def UserFraud(txn) -> bv.Table:
    return (
        txn.with_columns(is_fraud_int=bv.col("is_fraud").cast("int"))
           .group_by("user_id")
           .agg(fraud_count_1h=bv.sum("is_fraud_int", window="1h"))
    )
```

This pattern is verified to work because `with_columns` accepts `_ExprAST` (per `_events.py::with_columns`) and produces a new column the agg can sum.

Inline boolean-sum like `bv.sum(bv.col("is_fraud").cast(int))` is FORBIDDEN in v0 — the SDK raises a `RegistrationError(code="schema_mismatch")` at register-time when the field arg is not a string.
```

Step 1 — Use the Write tool to create `docs/schema-evolution.md` (~100-180 LOC):

H1: `# Schema Evolution`

Sections:
- ## Overview — registration is additive-only by default; destructive changes require `force=True`; `dry_run=True` returns diff without applying
- ## What's additive (default behavior — no flag needed)
  - Adding new event source descriptors
  - Adding new derivation nodes
  - Adding new fields to an existing event schema (with default value or as Optional)
  - Result: 200 OK with version bump
- ## What's destructive (requires `force=True`)
  - Removing existing event sources
  - Removing existing derivations
  - Changing field types on an existing event (e.g. f64 → i64)
  - Renaming fields
  - Result without force: 409 Conflict with structured `{diff: {added, removed, changed}}`; with force: 200 OK with version bump
- ## dry_run=True flag
  - Returns the same diff envelope but does NOT apply
  - No version bump
  - Useful for "what would happen if I register this DAG?"
- ## Diff matrix (table)
  - Rows: Add event / Remove event / Add field (Optional) / Add field (required, with default) / Add field (required, no default) / Change field type / Rename field / Add derivation / Remove derivation / Change derivation chain
  - Columns: Default behavior / With force=True / Recovery from prior state
- ## Examples — register-fraud-team.request.json (additive), register-conflict.error.json (destructive without force), register-force.request.json (destructive with force), register-dry-run.request.json (dry run)
- ## Server-side enforcement — JSON-prelude shim in `crates/beava-core/src/register_validate.rs`; cite Phase 12.6 + 12.7 + 12.8 shim pattern
