**Researched:** 2026-05-03
**Domain:** Documentation engineering (spec-doc authoring conventions, ADR templates, JSON Schema dialects, docstring patterns) — informing how to author 14 doc files + 53 per-op pages + 9 example files + 2 ADRs without inventing conventions on the fly
**Confidence:** HIGH (D-01..D-04 are locked; this research informs HOW to execute, not WHAT to decide)

## Summary

Phase 13.0 produces ~70+ documentation artifacts across `docs/`, `examples/`, and `.planning/decisions/`. The 4 design decisions (D-01..D-04) are already locked in CONTEXT.md; this research answers the **convention-level** questions the planner needs:

- **JSON Schema dialect:** Use **Draft 2020-12** because it aligns with OpenAPI 3.1, has full support across the Python (`jsonschema` lib), TypeScript (Ajv with the `2020` import), and Go (`santhosh-tekuri/jsonschema`) validators that 13.4/13.5/13.6 will use. Rust support is partial in the `jsonschema` crate but full in the `boon` crate; Phase 13.4 picks `boon` for the engine-side CI test. [VERIFIED: web search 2026-05-03]
- **ADR template:** Use **Nygard 5-section** (`Title / Status / Context / Decision / Consequences`) for ADR-001 + ADR-002. ~1 page each. Lightweight, well-known, matches the repo's existing decision-recording style (CONTEXT.md "D-NN — title — Decision/Why/How" mirrors Nygard's flow). MADR is structurally heavier and adds Considered Options + Pros/Cons fields that don't earn their keep for these two specific ADRs. [CITED: martinfowler.com/bliki/ArchitectureDecisionRecord.html, github.com/joelparkerhenderson/architecture-decision-record]
- **Per-op page template:** ~120-LOC Markdown per file. Sections: `Signature / Semantics / Parameters / Returns / Complexity / Examples / Wire / Edge cases / See also`. Mirrors Polars' `Expr.<op>` page IA + DuckDB's per-function entry style. SEO-friendly (one URL per op), Priya-google-able, mechanical to author (signatures extractable from `python/beava/_agg.py`). [VERIFIED: codebase grep + web search]
- **Pipeline-DSL compilation rules:** Per-method H3 sections (one heading per chain operator), each with three subheadings: **Python source** / **JSON wire** / **Server semantics** + a final ambiguity-matrix Markdown table. ~12 worked examples confirmed against `_agg.py` + `_col.py` exports; the `windowed=` and `@bv.table` items are **the only cases where the doc's normative form differs from the current codebase** (both v0-target shapes that 13.4/13.5 will implement, not retrospective documentation).
- **Codebase pattern map:** All canonical surfaces are present in the Python SDK; `OP_PUSH = 0x0002` opcode bug confirmed (Rust uses `0x0010`). 14 stale `docs/*.md` files audited — **12 marked NUKE** (they predate the Tally→Beava rename, the no-event-time pivot, and the table strip), **1 KEEP** (`docs/operators/cost-class.md` — alive Phase 19.2 artifact), **1 KEEP-AS-REFERENCE** (`docs/github-repo-surface-runbook.md` — non-overlapping launch-ops content).
- **Mock-backend strategy:** Python reuses `bv.App()` embed mode (`_embed.py` already spawns the binary); during 13.0 the binary doesn't yet honor the new wire so examples may be re-routed to a fixture-replay shim (`examples/python/_mock.py`, ~80 LOC). TypeScript and Go each ship a ~100-LOC `_mock.{ts,go}` that fakes the wire by replaying `examples/wire/*.response.json` fixtures over an in-memory channel. No real network in 13.0.
- **Plan/wave shape:** Refined to **3 waves, 11 plans**. The biggest single block (53 per-op pages, D-02) splits into **6 family plans** in Wave 2 to enable parallelism without explosion. Wave 1 has explicit `depends_on` chains: `wire-spec.md` → `sdk-api/*.md` and `ADR-002` → operator pages.
- **Industry tone:** Copy DuckDB's per-function template verbatim — `## Syntax / ## Description / ## Aliases / ## Examples / ## Notes` — with a beava-specific addition (`## Wire`) for the JSON cross-link D-02 mandates. Matches Priya target user expectations.

**Primary recommendation:** Generate all 53 per-op pages mechanically from a single scaffold script (Wave 2, Plan 13.0-05) that reads `python/beava/_agg.py` AST + `crates/beava-core/src/register_validate.rs::lifetime_bound_for_op_str` + `docs/operators/cost-class.md`, then split into 6 family-polish plans (Wave 2, Plans 13.0-06..11) so authors can write the Worked Example + Edge Cases prose in parallel. This is the largest authoring block and benefits most from the scaffold-then-polish pattern.

## 1. JSON Schema Dialect (D-01)

**Recommendation: Draft 2020-12** for `examples/wire/schemas/*.schema.json`, declared via `$schema: "https://json-schema.org/draft/2020-12/schema"`.

### Rationale (verified against tooling support)

| Validator | Lib | Draft 7 | Draft 2020-12 | Notes |
|-----------|-----|---------|---------------|-------|
| Python (engine + SDK tests) | `jsonschema` (PyPA) | ✓ | ✓ | Both fully supported; Python is the canonical SDK in 13.5 |
| TypeScript (13.6) | `ajv` v8+ | ✓ (default) | ✓ (`import Ajv2020 from "ajv/dist/2020"`) | Ajv splits 2020 into a separate import to avoid bundling bloat |
| Go (13.6) | `santhosh-tekuri/jsonschema/v6` | ✓ | ✓ | v6 release line supports both [VERIFIED: github.com/santhosh-tekuri/boon README] |
| Rust (13.4 engine CI) | `boon` crate | ✓ | ✓ | Recommended over `jsonschema` crate because the latter has only partial 2020-12 support [CITED: docs.rs/boon] |

### Why 2020-12 (not Draft 7)

1. **OpenAPI 3.1 alignment.** OpenAPI 3.1 dropped its custom JSON Schema dialect in favor of Draft 2020-12. This matters because if v0.1+ ever adds an OpenAPI surface (hosted-API consumers, generated client SDKs in other languages), the wire-spec schemas already conform. [CITED: learn.openapis.org/upgrading/v3.0-to-v3.1.html]
2. **`oneOf` discriminator support.** D-01 needs opcode-discriminated unions (the same `register` payload has different sub-shapes for `kind: "event"` vs `kind: "table"` per ADR-001). 2020-12's `oneOf` + `if/then/else` handles this cleanly; Draft 7 also does, but 2020-12 has the additional `unevaluatedProperties: false` keyword that catches typos in nested objects (a real risk with hand-written `register` payloads).
3. **`prefixItems`** for tuples (used in `examples/wire/batch_get-request.json` where the body is `[(table, key), ...]` heterogeneous list).
4. **Future-proofing.** Draft 7 is "safe and broadly supported" but new validator releases focus on 2020-12. Choosing 2020-12 today doesn't strand us.

### Concession: Draft 7 fallback path

If a downstream SDK port (community-driven Java in v0.1+) hits a validator that only supports Draft 7, the schemas degrade gracefully — beava's wire shapes don't use any 2020-12-only keyword that's load-bearing. The `$schema` line can be flipped to `draft-07` and the only feature lost is `unevaluatedProperties` (which becomes a soft-validation gap, not a correctness gap).

## 2. ADR Template (ADR-001 + ADR-002)

**Recommendation: Nygard 5-section template.** ~1 page each. Sections, in order:

```markdown
# ADR-NNN: <Title>

## Status

<Accepted | Proposed | Superseded by ADR-NNN | Deprecated>
Date: YYYY-MM-DD

## Context

<What's the situation? What forces are pushing toward a decision?>

## Decision

<What did we decide? Phrased as an active commitment.>

## Consequences

<What becomes easier or harder? What new constraints? What's a follow-on action?>
```

### Why Nygard, not MADR

| Criterion | Nygard | MADR | Verdict |
|-----------|--------|------|---------|
| Length | ~1 page | ~2-3 pages | Nygard wins for these 2 ADRs |
| Considered Options table | No | Yes | ADR-001 has only 1 option (kept @bv.table aggregation-output, killed everything else); ADR-002 has only 1 option (rename to Polars conventions). No "considered alternatives" matrix to fill. MADR's strength is wasted here. |
| Decision Drivers | No | Yes | The drivers ARE the Context; redundant in MADR. |
| Pros/Cons per option | No | Yes | Same as above. |
| Matches CONTEXT.md style | Partial | No | CONTEXT.md uses `D-NN — title — Decision/Why/How to apply` which is closer to Nygard's `Decision/Context` than MADR's option-matrix. |
| Future ADR ergonomics | Equal | Equal | Both work; Nygard's lower fixed cost wins. |

[CITED: martinfowler.com/bliki/ArchitectureDecisionRecord.html for Nygard rationale; github.com/adr/madr for MADR comparison]

### Concrete starter for ADR-001

```markdown
# ADR-001: `@bv.table` aggregation-output revival (partial overturn of v0 events-only scope)

## Status

Accepted
Date: 2026-05-03

## Context

Phase 12.7 (closed 2026-05-01, PASS) stripped the entire `@bv.table` surface from
beava v0 — no `bv.table` decorator in the public namespace, no `OpNode::Table*`,
no `app.upsert/delete/retract`, no `TemporalStore`, no MVCC, no `temporal_http`.
The strip removed ~5,500 LOC and locked the events-only commitment via
`project_v0_events_only_scope` and the architectural test
`crates/beava-server/tests/phase12_7_no_table_surface.rs`.

The 2026-05-03 v0-launch design session re-examined what the public Python API
should look like. The team locked Polars-style chained syntax for feature
authoring:

```python
@bv.event
class Txn:
    user_id: str
    amount: float

@bv.table(key="user_id")
def UserFeatures(txn) -> bv.Table:
    return (
        txn.group_by("user_id")
           .agg(tx_count_1h=bv.count(window="1h"),
                tx_sum_1h=bv.sum("amount", window="1h"))
    )
```

The `@bv.table` decorator is the natural attachment point for the
`group_by().agg(...)` aggregation output — the user names the result, declares
the partition key, and beava registers it as a derivation. There's no upsert,
no delete, no retract, no MVCC, no temporal versioning. It's a function-form
decorator that wraps an aggregation chain into a named derivation node.

This is a strictly narrower revival than the pre-12.7 surface. Joins, table
mutation, retraction propagation, and session windows REMAIN killed.

## Decision

Revive `@bv.table` as an aggregation-output decorator in the public `bv`
namespace. Specifically:

- `@bv.table(key=...)` decorates a function that returns
  `events.group_by(...).agg(...)`. The decorator captures the chain, names the
  result, and emits a derivation node with `output_kind=table`.
- `app.upsert(...)`, `app.delete(...)`, `app.retract(...)` REMAIN absent.
- `OpNode::TableUpsert`, `OpNode::TableDelete`, `OpNode::TableRetract` REMAIN
  absent. `OpNode::Table*` is permitted ONLY for derivation `output_kind=table`.
- `TemporalStore`, `MvccVersion`, `temporal_http`, and the `RecordType::Table*`
  WAL variants REMAIN absent.
- The architectural test `phase12_7_no_table_surface.rs` is amended to permit
  `OpNode::Table*` when it appears as the output kind of a derivation node, and
  to continue rejecting it everywhere else. The amendment lands in Phase 13.4
  alongside the engine-side `output_kind=table` support.
- `project_v0_events_only_scope` memory is updated with a partial-overturn
  pointer to this ADR.

## Consequences

**Easier:**
- Polars-style aggregation syntax is the canonical Python API (locked from the
  design session).
- Feature authors declare aggregations as named, keyed entities — natural
  conceptual model for fraud / ad-tech / behavioral analytics.
- The wire spec carries `kind=table` derivations; SDK porters in 13.6 (TS, Go)
  implement aggregation output without designing a parallel mechanism.

**Harder:**
- The architectural-test allowlist needs careful scoping. Permitting
  `OpNode::Table*` as a derivation `output_kind` while keeping it forbidden as
  a top-level register node + at all other call sites requires a per-AST-context
  check, not a global symbol grep.
- Future contributors might assume `@bv.table` revives the full Phase 11.5
  surface. Counter-discipline: this ADR is normative; the test enforces.

**Follow-on actions:**
- Phase 13.4 Plan: implement engine-side `output_kind=table` for derivations +
  amend `phase12_7_no_table_surface.rs` allowlist.
- Phase 13.5 Plan: implement Python `@bv.table` function-form decorator that
  wraps the aggregation chain into a derivation node.
- Phase 13.6 Plans: TypeScript + Go SDKs each support `bv.table()` as the
  aggregation-output equivalent (TS: builder-pattern wrapper; Go: function
  returning a table-derivation struct).
- CLAUDE.md `§ Events-Only Invariant` block gets a footnote pointing at this
  ADR.
- REQUIREMENTS.md `V0-EVENTS-ONLY-01` gets a sub-bullet documenting the
  permitted aggregation-output exception.
```

### Concrete starter for ADR-002

```markdown
# ADR-002: Rename core aggregation ops to Polars conventions

## Status

Accepted
Date: 2026-05-03

## Context

Beava's operator catalogue originally chose op names that read naturally in
SQL prose: `avg`, `variance`, `stddev`, `count_distinct`, `percentile`. The
2026-05-03 design session locked Polars-style chained syntax as the canonical
Python API, and Polars uses different names for the same operations: `mean`,
`var`, `std`, `n_unique`, `quantile`.

Polars conventions are documented, well-known to data-engineering audiences,
and consistent with pandas / dplyr / sqlglot. Mixing the two name families in
one codebase (Polars-style chains calling beava-named ops) creates persistent
friction at the language boundary.

The current names are baked into:
- `crates/beava-core/src/agg_compile.rs::parse_agg_kind` (op-string → AggKind)
- `crates/beava-core/src/register_validate.rs::lifetime_bound_for_op_str`
- `python/beava/_agg.py` (helper function names)
- `crates/beava-bench/configs/{small,medium,large,fraud-team}.json`
- `docs/operators/cost-class.md`
- ~60 test fixtures across the workspace

`FORMAT_VERSION` for WAL + snapshot stays at 1 (the rename does not change
the binary record format — only string constants in JSON-shaped op fields).

## Decision

Rename five core operators across server, SDK, and config artifacts:

| Old name | New name | Rationale |
|----------|----------|-----------|
| `avg` | `mean` | Polars / NumPy / pandas convention |
| `variance` | `var` | Polars convention |
| `stddev` | `std` | Polars convention |
| `count_distinct` | `n_unique` | Polars convention |
| `percentile` | `quantile` | Polars convention |

Implementation:
- Phase 13.4 lands the server-side rename: `parse_agg_kind` accepts the new
  names; `AggKind` variants stay as Rust enum names (`AggKind::Avg`,
  `AggKind::Variance`, etc.) but the public string mapping changes.
- Phase 13.4 also updates the four `crates/beava-bench/configs/*.json` files
  and `docs/operators/cost-class.md`.
- Phase 13.5 lands the Python SDK rename in `_agg.py` (helper functions
  `avg/variance/stddev/count_distinct/percentile` are renamed; old names
  remain as deprecation aliases that log a warning and dispatch to the new
  helper). `bv.ema` stays as an alias for `bv.ewma` (existing convention).
- Phase 13.6 ships TypeScript + Go SDKs with the new names from day one (no
  legacy aliases; v0 is unreleased).
- Phase 13.0 (this phase) per-op pages cite both names: heading + signature use
  the NEW name, with a "previously called `<old>`" note in the Edge Cases
  section.

## Consequences

**Easier:**
- Polars-experienced users transition immediately; signatures match.
- `bv.col(...).mean()` reads naturally in chains (no surprise from `bv.col(...).avg()`).
- Cross-language SDK consistency: TS `pipeline.col("x").mean().over("1h")` matches
  Python `bv.col("x").mean().over("1h")`.

**Harder:**
- One-time refactor across server + Python + 4 bench configs + ~60 test
  fixtures. Mechanical but multi-touchpoint.
- Pre-13.0 dev users (small group) need to rename in their local pipelines.
  Mitigation: Python SDK keeps deprecation aliases for one v0 minor (v0.1
  removes them entirely).

**Follow-on actions:**
- Phase 13.4 Plan: server-side rename + bench-config update.
- Phase 13.5 Plan: Python SDK rename + deprecation aliases.
- Phase 13.6 Plans: TS + Go SDKs use new names from day one.
- Phase 13.0 (this phase) per-op pages document both names.
- ADR-002 is referenced in CLAUDE.md as the Polars-naming locked decision.
```

**Confidence: HIGH** — Nygard template is the canonical industry-standard ADR shape; the two starter ADRs are constructed from CONTEXT.md decisions (locked) plus the existing code surfaces (verified by grep).

### Why this template (industry references)

| Reference | Section IA | Adopted? | Why |
|-----------|-----------|----------|-----|
| **DuckDB function pages** | `Syntax / Description / Aliases / Examples / Notes` | YES (with `Wire` added) | Closest match: per-function pages, signature + description + examples + notes is the minimum ergonomic shape. [CITED: duckdb.org/docs/current/sql/functions/text]. We add `Wire` (cross-link to `examples/wire/`) per CONTEXT.md D-02 mandatory section. |
| **Polars `Expr.<op>` pages** | `Parameters / Returns / Examples / See Also` (NumPy-docstring style) | PARTIAL | Polars uses Sphinx auto-generated pages. We're hand-writing — the explicit `Description / Complexity` sections add value Polars gets implicitly from being open-source-readable. [CITED: docs.pola.rs/api/python/dev/reference/expressions/api/polars.Expr.cum_sum.html] |
| **pandas API reference** | `Parameters / Returns / See Also / Notes / Examples` | PARTIAL | Same NumPy-docstring shape as Polars; same trade-off. |
| **dplyr verb docs** | `Usage / Arguments / Value / Examples` (R-style) | NO | R-doc conventions don't translate; sections are less explicit than what beava needs. |

### One fully-worked example: `quantile.md`

```markdown
# bv.quantile

> Approximate quantile of a numeric field over a window or lifetime, backed by DDSketch.

## Signature

```python
bv.quantile(
    field: str,
    q: float,
    *,
    window: str | None = None,
    where: bv.Col | None = None,
    exact_threshold: int = 256,
    hybrid_alpha: float = 0.01,
) -> AggDescriptor
```

> Previously called `bv.percentile`. Renamed to `quantile` per [ADR-002](../../../.planning/decisions/ADR-002-polars-op-rename.md) for Polars-convention consistency. The old name remains as a deprecation alias in v0.0.x and is removed in v0.1.

## Description

`bv.quantile` estimates the q-th quantile (0 ≤ q ≤ 1) of a numeric field across
events that match the optional `where=` predicate. Backed by a hybrid
exact-then-DDSketch state: while the entity has fewer than `exact_threshold`
distinct values, the state holds them exactly and returns the precise q-th
order statistic. Once the threshold is crossed, the state promotes to a DDSketch
with relative accuracy `hybrid_alpha` (default 0.01 = 1% relative error).

Returns a single `f64` per entity. Use `bv.quantile(field, q=0.99, window="1h")`
for "the 99th-percentile transaction amount over the last hour".

The hybrid mode is transparent at the API: callers always read a single float.
Promotion happens server-side without observable behavior change beyond a
small accuracy tradeoff.

## Parameters

| Name | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `field` | `str` | Yes | — | Name of the numeric field (`f64` or `i64`) to take the quantile of. |
| `q` | `float` | Yes | — | The quantile to compute, in `[0, 1]`. `0.5` is the median. |
| `window` | `str` | No | `None` (lifetime) | Duration string, e.g. `"5m"`, `"1h"`, or `"forever"`. Must match `\d+(ms\|s\|m\|h\|d)` or `"forever"`. |
| `where` | `bv.Col` | No | `None` | Boolean expression on event fields; only matching events contribute. |
| `exact_threshold` | `int` | No | `256` | Distinct-value count below which exact mode is used. |
| `hybrid_alpha` | `float` | No | `0.01` | DDSketch relative-error parameter once promoted. |

## Returns

A single `f64`. When the entity has seen zero matching events, the result is
`null` (returned as Python `None`).

## Complexity

| Resource | Bound |
|----------|-------|
| CPU per event | Tier 2 (Exact mode, 8 ns floor / ~35 ns measured) — see [cost-class.md](../cost-class.md#tier-2-moderate) |
|  | Tier 3 (DDSketch mode, post-promotion, ~130 ns floor / ~180 ns measured) |
| Memory per entity | Bounded sketch — up to ~few KB (DDSketch buckets) regardless of stream length |
| Lifetime mode | Allowed (`BoundedSketch` per Phase 12.8 memory governance — fixed structural cap) |

## Examples

### Example 1: 99th-percentile transaction amount per user, hourly

```python

class Txn:
    user_id: str
    amount: float

@bv.table(key="user_id")
def UserAmountStats(txn) -> bv.Table:
    return (
        txn.group_by("user_id")
           .agg(p99_amount_1h=bv.quantile("amount", q=0.99, window="1h"))
    )

# Push events
app.push("Txn", {"user_id": "alice", "amount": 12.50})
app.push("Txn", {"user_id": "alice", "amount": 1500.00})

# Query
result = app.get("UserAmountStats", "alice")
# result == {"p99_amount_1h": 1500.0}
```

- **No matching events:** result is `null`. (Cold-start returns `{}`, not an error.)

## 4. Pipeline-DSL Compilation-Rules Format (D-04)

**Recommendation:** Per-method H3 sections, each with three subheadings: **Python source** / **JSON wire** / **Server semantics**. Final ambiguity-matrix as a Markdown table at the bottom.

### Format template

```markdown
# Pipeline DSL Compilation Rules

> How beava's chained Python syntax compiles to the JSON wire shape sent over `/register`.

## How to read this doc

For each chain method (`events.filter()`, `events.group_by()`, etc.) we show:

1. **Python source** — the literal call as you'd write it.
2. **JSON wire** — what the SDK sends to the server in the `register` payload.
3. **Server semantics** — what the apply loop does with it.

Plus an [Ambiguity Matrix](#ambiguity-matrix) at the bottom rules out edge cases.

---

## events.filter(expr)

### Python source

```python
@bv.event
class Txn:
    user_id: str
    amount: float

big = Txn.filter(bv.col("amount") > 100).named("BigTxn")
```

### JSON wire

```json
{
  "kind": "derivation",
  "name": "BigTxn",
  "upstreams": ["Txn"],
  "ops": [
    {"op": "filter", "expr": "(amount > 100)"}
  ],
  "output_kind": "event"
}
```

### Server semantics

Per-event predicate evaluation. Events with `amount > 100` flow downstream;
others are dropped. Schema unchanged. Composes left-to-right with subsequent
ops.

**Chained filters compose.** `e.filter(a).filter(b)` is equivalent to
`e.filter(a & b)` — both compile to two ops or one op with a conjunctive
predicate; the server's evaluator collapses them at apply time.

Schema-narrowing: output schema is the listed fields in order. Other fields

import json, pathlib
from typing import Any

class MockApp:
    """In-memory, fixture-replay App for Phase 13.0 examples.

    Drop-in for bv.App() during 13.0 verification. Replaced by real
    bv.App() once Phase 13.4 lands (single-line edit in each demo file).
    """

        self.fixtures = pathlib.Path(fixtures_dir or "examples/wire")
        self._registered = set()
        self._state: dict[str, dict[str, Any]] = {}  # table_name -> key -> row

        # Replay register-success.response.json (D-01 fixture)
        for d in descs:
            self._registered.add(d._name)

        # Apply registered derivations against fixture rules
        # (fixture-driven — read examples/wire/push-success.response.json)
        return {"ack_lsn": 1, "registry_version": len(self._registered)}

        return self._state.get(table, {}).get(key, {})

    def batch_get(self, requests):
        return [self.get(t, k) for (t, k) in requests]

    def reset(self):
        self._state.clear()

    def close(self):
        pass

    def __enter__(self):
        return self

    def __exit__(self, *_):
        self.close()

### The convention to copy verbatim

DuckDB's per-function entry shape, slightly extended for beava's wire-doc cross-link:

```markdown
## Syntax

`function_name(arg1, arg2[, optional_arg])`

## Description

What it does, in 1-3 paragraphs. Mathematical definition if useful.

## Aliases

- `function_name_alias` — same op, different name

## Examples

Multiple worked examples with code blocks, each with clear intent.

## Notes

Edge cases, deprecations, performance notes.
```

Beava's per-op template (§3 above) reuses this shape with H2 heading names: `Signature / Description / Parameters / Returns / Complexity / Examples / Wire / Edge cases / See also`. The added `Wire` section is non-DuckDB but mandatory per CONTEXT D-02 (cross-link to `examples/wire/`).

- DuckDB is the canonical "OSS dev tool" voice — used by data engineers similar to Priya's hiring target.

## Open Questions for Planner

These could not be fully resolved during research; surface to user during plan-phase.

1. **Boolean-sum trick mechanics.** CONTEXT D-04 says `bv.sum(bv.col("is_fraud").cast(int))` is the recommended pattern. But `python/beava/_agg.py::sum` signature is `sum(field: str, *, window: str, where: Any = None) -> AggDescriptor` — `field` is a string, not an `_ExprAST`. Two paths:
   - **Path A:** Phase 13.5 extends `bv.sum` (and similar helpers) to accept `_ExprAST` for the field arg. v0 surface explicitly supports `bv.sum(expr)`.
   - **Path B:** v0 keeps `bv.sum(field: str)` only; the recommended pattern becomes `events.with_columns(flag_int=bv.col("flag").cast(int)).group_by(...).agg(c=bv.sum("flag_int", ...))`. The compilation-rules doc shows this two-stage pattern instead of inline boolean-sum.
   - **Recommendation:** Path B — keeps the v0 surface narrow; aligns with current code; the two-stage pattern is fully expressive. Phase 13.5 + 13.6 implement only `bv.sum(field: str)`.

2. **Python SDK example mock backend choice for D-03.** Three options for Phase 13.0 verification of `examples/python/*.py`:
   - **(a)** `bv.App()` embed mode runs the real binary which uses OLD wire (pre-13.4 routes). Demos work today but break when 13.4 ships.

   - **(c)** Skip Python verification in 13.0; defer to 13.5.
