## Wire-spec rigor & format (D-01)

| Option | Description | Selected |
|--------|-------------|----------|

## Operator catalog (53 ops) — depth (D-02)

| Option | Description | Selected |
|--------|-------------|----------|
| One page per op (53 pages) | `docs/operators/<op>.md` per op. Each ~50–150 lines: signature + semantics + return type + complexity + worked example + edge cases + cross-link to wire JSON. | One page per family (8 pages) | `docs/operators/{core,sketch,point,decay,velocity,buffer,geo,zscore}.md`. Each family page contains members as sub-sections. ~50% of per-op authoring lift. SDK porters scroll family pages. | |

**User's choice:** One page per op (53 pages) (Recommended)
**Notes:** Heaviest authoring lift, picked deliberately for devex. Priya (target user per `project_beava_website_ia`) googling `bv.percentile` lands directly on a self-contained page. DuckDB and Polars ship per-op pages — the reference voice.

## Pipeline-DSL compilation rules — coverage (D-04)

Plus an ambiguity matrix (chained filters / select-before-group_by / expressions in agg names / boolean-sum trick / nested group_by / etc.) with ALLOWED / FORBIDDEN / UNDEFINED rulings + test fixture or error code link per row. ~12–15 worked examples + ~20-row matrix. | ✓ |

**Notes:** SDK porters in 13.6 (TS+Go) get unambiguous compilation rules without reverse-engineering Python source. Each ambiguity-matrix row links to a fixture or error code — auditable + testable. Aligns with `feedback_verify_plan_decisions`.
