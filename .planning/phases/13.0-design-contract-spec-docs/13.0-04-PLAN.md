H1: `# Shared SDK Semantics`

## Overview

1-2 paragraphs:
- JSON wire (per docs/wire-spec.md) is the contract; per-language SDKs are thin compilers from idiomatic syntax to the wire
- All 3 v0 SDKs (Python, TypeScript, Go) MUST implement the surfaces in this doc with semantic parity
- This doc is normative for cross-language behavior; per-language idioms (naming style, async/sync, error patterns) are documented in the per-language docs

## Wire transports

Three transports map to one URL scheme each:
- `http://host:port` / `https://host:port` — HTTP/JSON transport (verb-style POST routes per docs/http-api.md)
- `tcp://host:port` — Custom-framed TCP per docs/wire-spec.md (frame: `[u32 length][u16 op][u8 content_type][payload]`)
- No URL (e.g., `App()` in Python, `BeavaApp()` in TS, `NewApp(ctx, "")` in Go) — embed mode (spawn local `beava` binary on ephemeral ports)

URL-scheme dispatch is part of every SDK's contract.

## Window grammar

Document the universal window-string format:
- Pattern: `\d+(ms|s|m|h|d)|forever`
- Examples: `100ms`, `30s`, `5m`, `1h`, `7d`, `forever`
- All 3 SDKs reject malformed windows at decorator/build time

## Key shape

Document the entity-key shape:
- Single key: string
- Composite key: list (or array) of `[string | number | boolean]` items
- Hash-based equality across languages (FxHash on the server, normalized through stringification)

## Field types

Field-type vocabulary (per python/beava/_types.py — `py_type_to_field_type`):

| Wire name | Python | TypeScript | Go |
|-----------|--------|------------|-----|
| `str` | `str` | `string` | `string` |
| `f64` | `float` | `number` | `float64` |
| `i64` | `int` | `number` (note: TS has no native int64; use bigint or document precision floor) | `int64` |
| `bool` | `bool` | `boolean` | `bool` |
| `bytes` | `bytes` | `Uint8Array` | `[]byte` |
| `datetime` | `datetime` | `Date` | `time.Time` |

Document `Optional[T]` semantics: nullable field; SDK marks as optional in the registered schema.

## FeatureResult shape

The result of `app.get(...)` is a row-shape dict (per docs/wire-spec.md OP_GET response). All SDKs surface this as:
- Python: `dict[str, Any]` (or attribute-access via a thin wrapper)
- TypeScript: `Record<string, any>` (or generic typed result `<T>`)
- Go: `map[string]any` (or strongly-typed result via codegen — v0.1+)

Cold-start returns `{}` (empty dict / object / map) — NOT an error.

## ValidationError envelope

Cross-language schema for the validation-error envelope (matches `python/beava/_errors.py::ValidationError`):

```json
{
  "kind": "<one-of: cycle | missing_upstream | schema_mismatch | bad_return_type | unknown_field_type | table_key_invalid | registration_conflict | duplicate_name | unsupported_node_kind>",
  "path": "<DAG/JSON path>",
  "message": "<human-readable, forward-looking framing per Phase 12.7 D-02>"
}
```

The 9 `kind` values are frozen for v0; new kinds require an ADR.

## Schema evolution

Document the `force=true` and `dry_run=true` flags as cross-language register knobs (per docs/wire-spec.md OP_REGISTER):

| Flag | Type | Default | Behavior |
|------|------|---------|----------|
| `force` | bool | false | Permits destructive register (e.g. field type change). Without force, returns 409 conflict. |
| `dry_run` | bool | false | Returns the diff without applying. Body: `{added, removed, changed, diff}`. No version bump. |

## Cross-language naming convention

Spec-level normative:
- Wire JSON keys: snake_case (e.g. `event_name`, `registry_version`)
- Python public API: snake_case (matches PEP 8)
- TypeScript public API: camelCase (matches JS convention; e.g. `app.batchGet(...)`, `eventName: string`)
- Go public API: PascalCase (matches Go convention; e.g. `App.BatchGet(...)`)
- All 3 languages serialize/deserialize automatically at the transport layer; users write idiomatic code

## Error semantics

- Cold-start: `{}` (NOT error)
- Schema mismatch on push: ERROR (raises in Python, throws in TS, returns error in Go)
- Unknown event/table on register validation: ERROR with `kind=missing_upstream` or `kind=unknown_field_type`
- Batch atomic: per-entry errors do NOT abort the batch — each entry returns its own success/error envelope; a transport-level batch failure (e.g. malformed JSON) returns a single batch-level error

## Lifetime aggregation rules (cross-language register-time validation)

Per V0-MEM-GOV-02 (Phase 12.8):
- Lifetime aggregations (windowless mode) MUST declare a finite per-entity memory ceiling at register-time
- Server-side JSON-prelude shim returns `code: "unbounded_op_in_lifetime_mode"` if violated
- All 3 SDKs MUST validate this client-side (not strictly required, but recommended) so the user gets fast feedback

### @bv.table (function form, per ADR-001)

```python
@bv.table(key="user_id")
def UserTxnFeatures(txn: Txn):
    return (
        txn.group_by("user_id")
           .agg(
               tx_count_1h=bv.count(window="1h"),
               tx_sum_1h=bv.sum("amount", window="1h"),
               tx_p99_1h=bv.quantile("amount", q=0.99, window="1h"),
           )
    )
```

Document:
- Function form ONLY in v0 — class form is v0.1+
- key: string OR list of strings (composite key)
- Function body returns `events.group_by(...).agg(...)` chain
- The decorator extracts the chain, names it, and registers as a derivation node with `output_kind=table`
- Cite ADR-001 for the partial-overturn context
