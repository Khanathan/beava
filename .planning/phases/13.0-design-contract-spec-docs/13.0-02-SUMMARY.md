## What Landed

### docs/wire-spec.md (420 lines)

Sections, in order:

1. **Overview** — dual transport (HTTP/JSON + custom-framed TCP), Redis-style FIFO correlation, schema-as-source-of-truth contract, validation harness deferred to Phase 13.4.
2. **Frame Format** — byte-layout diagram for the TCP transport (`[u32 length][u16 op][u8 content_type][payload]`); minimum frame = 8 bytes total (4-byte length prefix + 4-byte body when payload empty); `length` covers `op + content_type + payload`, NOT itself; big-endian for length and op; HTTP transport notes; max-frame-size + framing-error behaviour; correlation = strict-FIFO (no `request_id`).
3. **Opcode Table** — markdown table listing all opcodes with values, direction, body shape, and notes. v0 actives: PING (0x0000), REGISTER (0x0001), PUSH (0x0010), GET (0x0020), GET_RESPONSE (0x0023), BATCH_GET (0x0024), RESET (0x0040), ERROR_RESPONSE (0xFFFF). v0.1+ reserved: PUSH_SYNC (0x0011), PUSH_MANY (0x0012), 0x0030-0x003F range.
4. **Content Types** — `CT_JSON = 0x01` (v0 implementation), `CT_MSGPACK = 0x02` (TCP-only, reserved for v0.1+), `unsupported_content_type` error code.
5. **Per-opcode body shapes** — 6 v0 opcodes documented (PING / REGISTER / PUSH / GET / BATCH_GET / RESET) with the same template per opcode (Request body shape + JSON Schema link + Worked example link + Response body shape + Errors table).
6. **Error Envelope** — `error.schema.json` cross-link; forward-looking framing per Phase 12.7 D-02; same envelope on both transports.
7. **ADR cross-references** — ADR-001 (`@bv.table` aggregation-output revival; mutation paths stay killed); ADR-002 (Polars op renames; `mean / var / std / n_unique / quantile`).
8. **Validation harness (Phase 13.4)** — Rust `boon` crate for engine-side; `examples/wire/_validate_examples.py` for cross-language reference; Ajv2020 for TS; santhosh-tekuri/jsonschema for Go.
9. **Stable contract guarantees** — locked vs not-locked surfaces.
10. **Plan-level traceability** — downstream plan consumers.

### 13 JSON Schemas (Draft 2020-12)

All declare `$schema = "https://json-schema.org/draft/2020-12/schema"` and use `unevaluatedProperties: false` for typo-catch. All have `$id` of form `https://beava.dev/schemas/<name>.schema.json`.

- `register.request.schema.json` — discriminated union via `oneOf` over `EventDescriptor` / `TableDescriptor` / `DerivationDescriptor` defs (each branched on `kind: const` value). `AggOpSpec` nested type documents Polars-rename op-name semantics. ~140 LOC.
- `register.response.schema.json` — `{status, registry_version, added?, removed?, changed?, diff?}`.
- `push.request.schema.json` — `{fields: object}` per Q1 Path B (no `_ExprAST` in field args).
- `push.response.schema.json` — `{ack_lsn, registry_version, idempotent_replay?}`.
- `get.request.schema.json` — `{table, key: string|array, features?}`.
- `get.response.schema.json` — open object (row-shape).
- `batch_get.request.schema.json` — `{requests: [{table, key, features?}, ...]}` heterogeneous.
- `batch_get.response.schema.json` — `{results: [object, ...]}`.
- `reset.request.schema.json` + `reset.response.schema.json` — `{}` and `{status: "ok"}`.
- `ping.request.schema.json` + `ping.response.schema.json` — `{}` and `{server_version, registry_version}`.
- `error.schema.json` — common envelope `{code, path?, message}` (Phase 12.7 D-02 forward-looking framing in description).

### 17 Worked Example Fixtures

- `register-fraud-team.{request,response}.json` — fraud-team-style payload using NEW op names (`mean`, `n_unique`, `quantile` per ADR-002).

- `register-dry-run.request.json` + `register-force.request.json` — `dry_run=true` and `force=true` flag examples.

- `reset-{request,response}.json` + `ping-{request,response}.json` — empty-body opcodes.

### examples/wire/_validate_examples.py (105 LOC)

Walks `examples/wire/*.json`, infers schema via filename pattern, validates each against `Draft202012Validator(schema).iter_errors(instance)`. Exits 0 on all-clean, exits 1 with diagnostic block on failure. Used as cross-language reference; Phase 13.4 ships the equivalent Rust harness using the `boon` crate.

Filename inference rules:
- `<op>-<scenario>.error.json` → `error.schema.json`
- `<op>-<scenario>.request.json` → `<op>.request.schema.json`
- `<op>-<scenario>.response.json` → `<op>.response.schema.json`
- `<op>-request.json` (no scenario) → `<op>.request.schema.json`
- `<op>-response.json` (no scenario) → `<op>.response.schema.json`

## Notes for Downstream Plans

**For Plan 13.0-03 (`docs/http-api.md`):** The opcode table in `wire-spec.md` is the source of truth; HTTP routes mirror this set. Use the same body schemas (cross-link `examples/wire/schemas/`) — HTTP and TCP carry identical JSON bodies in v0 (the only difference is framing).

**For Plan 13.0-04 (`docs/sdk-api/*.md`):** SDK porters consume the schemas from `examples/wire/schemas/` for client-side validation. The Python SDK uses `jsonschema` (already in `requirements`); TS uses `Ajv2020`; Go uses `santhosh-tekuri/jsonschema/v6`.

**For Plan 13.0-12 (`docs/error-codes.md`):** The codes referenced in this spec (`unsupported_node_kind`, `registration_conflict`, `schema_mismatch`, `unknown_event`, `unknown_table`, `feature_not_in_table`, `key_shape_mismatch`, `frame_too_large`, `unsupported_content_type`, `op_not_implemented`, `unknown_op`, `dedupe_replay`, `reset_disabled`, `wal_truncate_failed`) are inputs for the alphabetised list. Adding more is fine; renaming an existing one is a breaking-change ADR.

**For Phase 13.4 (engine prep):** Ship `crates/beava-server/tests/wire_spec_validates.rs` using the `boon` crate. The test loads every `examples/wire/schemas/*.schema.json`, then walks `examples/wire/*.json` (excluding the `schemas/` subdirectory), infers schema by filename, asserts all validate. The Python harness `examples/wire/_validate_examples.py` is the reference behaviour.

**For Phase 13.5 (Python SDK):** SDK unit tests should re-use `examples/wire/_validate_examples.py` as a fixture for client-side request-body validation (catch typos in user code before they hit the server). The schemas are the contract for both directions of the wire — the SDK validates outbound requests; the server validates inbound; they share the same files.

**For Phase 13.6 (TS+Go SDKs):** Same fixtures, language-native validators (`ajv` v8+ `2020` import for TS; `santhosh-tekuri/jsonschema/v6` for Go). The cross-language guarantee is that all four validators (Python / Rust / TS / Go) agree on the same fixture-vs-schema verdict — divergence is a contract bug.
