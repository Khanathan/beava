<interfaces>
Existing opcode constants (src/server/protocol.rs and python/tally/_protocol.py):

```
OP_PUSH        = 0x01
OP_GET         = 0x02
OP_SET         = 0x03
OP_MSET        = 0x04
OP_REGISTER    = 0x05
OP_MGET        = 0x06
OP_PUSH_ASYNC  = 0x07
OP_FLUSH       = 0x08
OP_PUSH_BATCH  = 0x0A
```

Gap at 0x09 (between OP_FLUSH and OP_PUSH_BATCH). Locked assignment for Phase 24:

```
OP_PUSH_TABLE   = 0x0B   (next free after OP_PUSH_BATCH)
OP_DELETE_TABLE = 0x0C
```

Rationale: 0x09 was deliberately left as a gap after OP_FLUSH; keeping new
opcodes contiguous after 0x0A is cleaner than back-filling. Document this
decision in the opcode block comment.

Existing App methods that must NOT change:
- `app.push(stream_class, event)` — fires OP_PUSH_ASYNC; used by the entire Stream source path
- `app.set(key, features)` / `app.mset(entries)` — still drives OP_SET / OP_MSET against static_features

New App methods (from CONTEXT.md §Python SDK):
- `app.push(table_source, key, fields)` — 3-arg form dispatches to OP_PUSH_TABLE
  **Disambiguation:** inspect the first arg's decorator marker (e.g.
  `getattr(table_source, "_tally_kind", None) == "table"`) to branch
  between the existing Stream `push(stream_class, event)` path and the
  new Table path. Fall back on arg count (2 vs 3) for explicit disambig.
- `app.delete(table_source, key)` — always OP_DELETE_TABLE

StateStore methods from plan 01 (already shipped):
- `upsert_table_row(key, table_name, fields, now)`
- `tombstone_table_row(key, table_name, now)`
- `get_table_row(key, table_name)`

Merged GET view semantics (LOCKED — CONTEXT.md §Storage model):
- `app.get(key)` returns union of:
  (a) Live operator features from `streams` (existing behavior)
  (b) Live `table_rows[*].fields` flattened as `{table_name}.{field_name}` entries
     (e.g. `table_rows["UserProfile"].fields["email"]` → `"UserProfile.email"` in response)
  (c) `static_features` (existing behavior)
- Tombstoned rows NOT included in the merged view (they're filtered)
- Collision rule: static_features overlay table_rows on identical emitted-key — document in doc-comment; should not occur in v0 (table_rows use prefixed names, static_features use raw names)
