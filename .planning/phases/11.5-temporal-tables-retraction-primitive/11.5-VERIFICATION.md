## Gaps / follow-ups for Phase 12+

1. **WAL recovery replay** for the three new record types is currently a **stub** (`crates/beava-server/src/recovery.rs` line 181 logs the record and advances LSN). Full replay into `temporal_stores` + `event_id_index` rebuild lands in a follow-up plan (`11.5-02` or rolled into Phase 12). Current behaviour: temporal state is lost across restarts — acceptable for the v0 milestone since Phase 7's snapshot wires for stream/agg state, and snapshot v2 (D-13/D-14) adds chains.
2. **Snapshot v2 round-trip** of MVCC chains (D-13/D-14): NOT shipped. Recovery uses WAL replay only once #1 lands.
3. **Python SDK `app.retract`** helper (D-18/D-19): NOT shipped. Python clients call HTTP directly via `requests.post(...)` until the SDK lands.
4. **`POST /push-table/{name}` for non-temporal tables** writes to `event_id_index` but NOT to a non-temporal table state (Phase 12 scope).
5. **`POST /retract` of `TableDelete` events**: scaffolded in `event_id_index` but not exercised; returns 501 with TODO note.
6. **Composite multi-field keys** in `?key=...` query: only single-field key supported; multi-field requires query-string extension. Not blocking for Phase 12.
7. **Stream retraction (v1)**: documented forward-compat path is the same `Retract` record type + a different replay branch. v0 explicitly rejects with `501 stream_retraction_unimplemented`.
8. **Periodic retention sweep**: `sweep_retention` exists; only the per-write soft cap (1024 versions per entity) is wired automatically. Snapshot-time + timer-driven sweeps deferred.
9. **Throughput row 409-on-already-retracted**: harness counts 409 toward retract-success (race with concurrent retract is treated as idempotent at the harness boundary).
