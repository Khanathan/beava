# Phase 11.5: Temporal tables + retraction primitive — Context

**Gathered:** 2026-04-23
**Status:** Ready for planning
**Mode:** `--auto` (orchestrator-driven, recommended-defaults locked-in)

<domain>
## Phase Boundary

Server-side **MVCC storage for `@bv.table(temporal=True, retention=...)` tables**, plus an **`app.retract(event_id)` primitive scoped to table upserts/deletes** in v0. Wires the `as_of=<lsn>` kwarg the SDK already ships so Phase 12 joins can resolve event-time PIT lookups against table history. Stream retraction is **explicitly deferred to v1** — but the WAL + aggregation format land with stable event-IDs so stream retraction is additive later, not a breaking change.

**In scope:**
- New WAL record types: `TableUpsert`, `TableDelete`, `Retract` (each carries a stable `event_id: u64`).
- New `event_id` field on the existing `Event` record (always populated; non-temporal use ignores it; future stream retraction reads it).
- `temporal: bool` + `retention_ms: Option<u64>` flags on `TableDescriptor`.
- New per-table MVCC store: `BTreeMap<EntityKey, BTreeMap<Lsn, MvccVersion>>` with `MvccVersion = Live(Row) | Tombstone | Retracted { restored_to: Lsn }`.
- HTTP: `POST /push-table/{table_name}`, `POST /delete-table/{table_name}`, `POST /retract` (body `{event_id}`), `GET /table/{table_name}?key=...&as_of=<lsn>`.
- `GET /registry` reports `temporal: bool` per table.
- Snapshot v2 round-trips MVCC chains.
- Recovery replays `TableUpsert`/`TableDelete`/`Retract` records in LSN order.
- Retention sweep — wall-clock-driven background task that drops versions older than `retention_ms`.
- Python SDK: `app.retract(event_id)` client method.
- Throughput run: temporal-table workload variant appended to bench harness; HTTP-only (TCP push not yet wired — Phase 8 sibling).
- At least one criterion microbench (`temporal_store_upsert`, `temporal_store_as_of_lookup`, `retract_apply`).

**Out of scope (explicitly deferred):**
- Stream-event retraction: `POST /retract` against a stream `event_id` returns **501** with `{"error": "stream_retraction_unimplemented", "see": "phase-11.5-summary"}`. Stream retraction with operator-state rewind is a v1 capability.
- Aggregation on temporal tables (already deferred per SDK-AGG-05; v0.1 problem).
- Cross-region MVCC, replicated history (v1+).
- Retention units other than wall-clock duration (e.g., LSN-window retention) — future opt-in.

</domain>

<decisions>
## Implementation Decisions

### MVCC storage shape
- **D-01: `BTreeMap<EntityKey, BTreeMap<Lsn, MvccVersion>>` per table.** Outer map keyed by entity primary-key (composite keys serialized to a stable byte-string). Inner map keyed by LSN (monotonic, unique per record). Justification: `as_of=<lsn>` lookup is `versions.range(..=as_of).next_back()` — O(log n). Insert is O(log n). Sweep is O(versions_per_entity). Memory overhead per version ≈ 24B map node + payload. Acceptable for v0; B+tree / arena allocators are v1.x optimizations.
- **D-02: `MvccVersion` enum:** `Live(Row)` (upsert), `Tombstone` (delete), `Retracted { undo_of: Lsn }` (retraction marker). Three variants because we want `as_of` queries to distinguish "row was deleted at LSN K" from "row was retracted at LSN R" (the latter means the upsert at `undo_of` should be ignored for lookups at LSN >= R).
- **D-03: Composite keys serialized as `length-prefix(field_1) || length-prefix(field_2) || ...`** — total-order-preserving byte serialization so `BTreeMap` ordering remains meaningful for range scans (future-use). v0 only does point lookups, but the encoding is set up for future range queries without a migration.

### Retraction strategy: tombstone-style (D-04 chosen)
- **D-04: Tombstone-style retraction wins.** When `POST /retract {event_id}` succeeds, we **append a `Retracted { undo_of: original_lsn }` version at `retract_lsn`**. We do NOT remove the original `Live`/`Tombstone` version. Rationale:
  - **`as_of` queries straddling the retract are correct**: a query `as_of = retract_lsn - 1` still sees the upsert (history before retraction is preserved); a query `as_of = retract_lsn` sees the prior live state (retraction has taken effect).
  - **Audit trail**: WAL replay is bit-identical regardless of retraction.
  - **Idempotent retraction**: retracting an already-retracted event_id is a no-op (we detect the existing `Retracted` marker pointing at the same `undo_of`).
  - Memory cost: one extra `MvccVersion` per retraction. Negligible for sane retention windows.
  - **Lookup logic**: `lookup(entity, as_of)` walks back from `as_of` collecting versions. If the most recent version is `Retracted { undo_of: K }`, treat the version at `K` as if it never existed and continue walking from `K-1`. Multiple retractions chain naturally.

### Retention enforcement: hybrid (snapshot-time + on-write soft cap) (D-05)
- **D-05: Snapshot-time sweep (primary) + on-write soft cap (secondary).** Rationale:
  - Snapshot already walks every table at fsync — adding a "drop versions where `now_wall_ms - version_wall_ms > retention_ms`" sweep is free I/O-wise.
  - On-write soft cap: if a single entity's chain length exceeds `MAX_VERSIONS_PER_ENTITY = 1024` (configurable, hard cap to prevent pathological retract-loops from blowing memory before the next snapshot), drop the oldest non-tombstone version. Logged as a metric `temporal_versions_capped_total{table=...}`.
  - **Per-version metadata**: each `MvccVersion` carries `wall_ms: u64` (set when the version was written) so the sweep can compare against `now_wall_ms` without walking the WAL. 8 extra bytes per version is acceptable — beats keeping a separate side-index.
- **D-06: Periodic background sweep is NOT in v0.** A background sweep timer adds complexity (single-thread invariant, lock contention) for marginal gain — snapshot-time + on-write cap is enough. v1 can add it if the snapshot interval grows large.

### `as_of` semantics
- **D-07: `as_of=<lsn>` is exclusive of `lsn+1`** — `versions.range(..=as_of_lsn).next_back()`. The version at `lsn=K` is visible to queries with `as_of >= K`. This matches "what would I have seen if I queried at the moment LSN K was applied?" semantics.
- **D-08: `as_of` requires the table be temporal.** Querying `GET /table/foo?as_of=123` against a non-temporal table returns **400** `{"error": "as_of_requires_temporal", "table": "foo"}`. This is a contract-design choice: silently ignoring `as_of` on non-temporal tables would create silent join-correctness bugs in Phase 12.
- **D-09: `as_of` clamping**: if `as_of > current_max_lsn`, treat as "current" (no clamp error). If `as_of` falls before the retention window's earliest preserved version, return **404** `{"error": "as_of_outside_retention", "earliest_lsn": E}`. (Phase 12 join code will catch this and emit `null` for left-joins.)

### Event-ID strategy
- **D-10: `event_id: u64` = WAL LSN.** No separate ID space. The LSN is already monotonically unique per record, durable, and observable to clients via the existing `ack_lsn` response field. Adding a parallel ID space (UUIDs, ULIDs) would double the metadata footprint for zero additional capability in v0.
- **D-11: Existing `Event` record gains an implicit event_id = its LSN.** No format change needed — the LSN field already exists. We just document that `lsn` doubles as the stable event_id. The `Retract` record body carries the `target_event_id: u64` (= target LSN).
- **D-12: Forward-compat for stream retraction**: when stream retraction lands in v1, it will use the same `Retract` record type but the recovery path will route stream-event retracts to operator-state rewind code (which doesn't exist in v0). v0 hard-rejects stream retraction at the `POST /retract` handler with **501**.

### Snapshot integration
- **D-13: Snapshot v2 adds a `temporal_tables` section.** Format: per-table, list of `(entity_key_bytes, [(lsn, wall_ms, version_kind, payload)])`. Backwards-compatible: snapshots without this section recover with empty MVCC stores (which is correct — non-temporal tables don't use MVCC).
- **D-14: Snapshot version bump from v1 → v2.** Recovery code accepts v1 (treats all tables as non-temporal) and v2 (full MVCC restore). One-way: writers always emit v2.

### Single-writer discipline
- **D-15: All MVCC mutations go through the same single-writer lock as aggregations** (`AppState.dev_agg.state_tables` lock). MVCC stores live alongside `state_tables` in `dev_agg`. No new lock; no new thread. Maintains the project's "single thread for event processing" invariant.

### Retention duration parsing
- **D-16: `retention="7d"` parser handles `s|m|h|d|w` suffixes.** Reused from existing window-spec parser (Phase 5). Min retention = 1 second, max = 365 days. Validation at registration time.

### Retract API surface
- **D-17: `POST /retract` body shape:** `{"event_id": <u64>}`. Response on success: `{"ack_lsn": <retract_lsn>, "target_event_id": <id>, "table": "<name>", "entity_key": [...], "restored_to_lsn": <prior_lsn or null>}`. Errors:
  - **404** `{"error": "event_id_not_found"}` — event_id has no WAL record.
  - **409** `{"error": "event_id_outside_retention", "earliest_lsn": E}` — event was upserted but its version was already swept by retention.
  - **409** `{"error": "event_id_already_retracted"}` — idempotency: retracting an already-retracted ID returns 409 (NOT 200 — clients should know they're double-retracting; this is rarely benign).
  - **501** `{"error": "stream_retraction_unimplemented", "see": "phase-11.5-summary"}` — event_id points at a stream `Event` record.
  - **400** `{"error": "table_not_temporal", "table": "<name>"}` — event_id points at a TableUpsert/Delete on a non-temporal table (those don't preserve undo state).

### Python SDK retract
- **D-18: `app.retract(event_id: int) -> RetractAck` — sync HTTP POST.** Mirrors `app.push` shape. No fire-and-forget variant in v0 (retraction is rare and side-effectful; user almost always wants the ack).
- **D-19: `RetractAck` dataclass** has fields: `ack_lsn`, `target_event_id`, `table`, `entity_key`, `restored_to_lsn`. Serialization symmetric with the HTTP response.

### Throughput-run scope
- **D-20: New harness shape: `temporal-fraud`.** Same fields as the existing `simple-fraud` shape but with: 1 temporal table (`@bv.table(temporal=True, retention="1h")`), 70% upserts / 25% reads / 5% retracts, HTTP transport only. Row appended to `.planning/phases/11.5-name/11.5-throughput-row.md` (NOT canonical `.planning/throughput-baselines.md` — orchestrator owns that).
- **D-21: macOS fsync ~7.4 ms** — expect upsert throughput similar to push throughput (~1k EPS local). This is the **first table-write throughput baseline** captured; no prior comparison.

### Bench scope
- **D-22: Three criterion benches**: 
  1. `temporal_store_upsert` — measures pure in-memory MVCC insert (no WAL, no fsync) at chain depths 1, 10, 100, 1000.
  2. `temporal_store_as_of_lookup` — measures `range(..=lsn).next_back()` at the same depths.
  3. `retract_apply` — measures `Retracted` insertion + lookup-with-skip at depth 100.
- **D-23: Bench rows go in `.planning/phases/11.5-name/11.5-perf-row.md`** (per-phase scratch file). Orchestrator merges to canonical `.planning/perf-baselines.md` later.

- **Bench harness** (`crates/beava-bench`): pipeline-shape arg already supports custom shapes; new `temporal-fraud` shape slots in.

### Established patterns
- **Conventional commits with TDD** (`test:` before `feat:`) is enforced.
- **Append-after-fsync** for durability — applies to TableUpsert/TableDelete/Retract identically to Event.
- **Single-thread event processing** — no async surprises; MVCC mutations stay on the writer thread.
- **Schema v1 → v2 bump pattern** seen in Phase 2 — reuse the same approach for snapshot v1 → v2.

### Integration points
- `POST /push-table/{name}` and `POST /delete-table/{name}` slot into the existing axum router alongside `/push/{event_name}`.
- `GET /table/{name}` slots alongside the existing feature query routes (`feature_query.rs`).
- `GET /registry` already serializes the registry — a new field on `TableDescriptor` flows through automatically.
- Recovery in `recovery.rs` already iterates all WAL records; new variants are routed by `RecordType` match.
