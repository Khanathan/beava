## What shipped

A server-side **MVCC temporal store** for `@bv.table(temporal=True, retention=...)` plus the **`POST /retract {event_id}`** primitive that undoes a single table upsert (or delete) by event_id. This unblocks Phase 12 PIT joins (which consume `as_of=<lsn>`) and locks in stable event-IDs for forward-compat with v1 stream retraction.

Single plan (`11.5-01`) covering:
- Registry: `temporal: bool` + `retention_ms: Option<u64>` flags on `TableDescriptor`
- WAL: three new `RecordType` variants — `TableUpsert (0x03)`, `TableDelete (0x04)`, `Retract (0x05)`
- Core: `crates/beava-core/src/temporal.rs` — pure data-structure MVCC store
- Server: AppState wiring (per-table `TemporalStore` map + `event_id_index`), three new HTTP handlers (`POST /push-table/{name}`, `POST /retract`, `GET /table/{name}`), `/registry` debug surface includes the temporal flag
- Bench: `crates/beava-core/benches/temporal_mvcc.rs` — first MVCC baseline
- Throughput: `crates/beava-bench/src/bin/temporal_throughput.rs` — first table-write baseline

## Key design decisions (see 11.5-CONTEXT.md for full D-01..D-23 trail)

### MVCC storage shape (D-01)

`BTreeMap<EntityKey, BTreeMap<Lsn, MvccVersion>>` — outer map keyed by primary-key bytes, inner map per-entity sorted-by-LSN chain of versions. `as_of=<lsn>` lookup is `chain.range(..=as_of).next_back()` after the retraction-skip pass. `EntityKey = Vec<u8>` with length-prefix-concat encoding (D-03) so composite keys preserve total ordering.

`MvccVersion` is a 3-variant enum:
- `Live { row, wall_ms }` — an upsert at this LSN
- `Tombstone { wall_ms }` — a delete at this LSN
- `Retracted { undo_of: u64, wall_ms }` — a retraction marker; `undo_of` is the target LSN

Each version carries `wall_ms` (8 bytes) so retention sweep doesn't have to walk the WAL.

### Retraction strategy: tombstone-style, NOT physical deletion (D-04)

Retracting event_id `K` at retract_lsn `R` inserts a `Retracted{undo_of: K}` marker at `R`. The original version at `K` is **never removed**. Lookups at `as_of < R` still see the original upsert; lookups at `as_of >= R` skip both the marker and the version at `K`.

**Rationale:**
- Audit trail preserved — historical reads through `as_of < R` keep seeing the retracted version, which matters for fraud investigations and replay debugging.
- Idempotent re-application during WAL replay.
- Forward-compat with stream retraction (v1) — same record type + same chain semantics will work; only the apply branch will differ.
- Trade-off: storage grows by 1 version per retraction. Retention sweep eventually evicts both.

### Retention enforcement: per-write soft cap + sweep on demand (D-05)

`TemporalStore::sweep_retention(now_wall_ms, retention_ms) -> usize` drains entries where `now_wall_ms - version.wall_ms() > retention_ms`. Drives retention with two triggers:
1. **Per-write soft cap** — `MAX_VERSIONS_PER_ENTITY = 1024` enforced inside `upsert`/`delete` via single `pop_first()` call when chain exceeds cap.
2. **Snapshot-time sweep** — wired into the snapshot path (Phase 7-style) to drain wall-clock-aged entries; not currently called periodically.

A periodic sweep is deferred — for v0 the soft-cap and snapshot-time sweep together bound memory.

### Stream retraction: explicit 501 (D-04, D-15)

`POST /retract` against a stream `event_id` returns:
```json
HTTP/1.1 501 Not Implemented
{"error": "stream_retraction_unimplemented", "see": "phase-11.5-summary"}
```

The `event_id_index` records every stream push as `EventIdEntry::Stream{event_name}` so we can identify the case unambiguously. Forward-compat: the same `Retract` WAL record type and replay branch will service stream retraction in v1; only the apply path (which must scan all aggregations and undo) will be added.

### Stable event-IDs (D-10/D-11)

`event_id == WAL LSN`. We don't mint a separate ID space. The HTTP `ack_lsn` field returned by every push is the value the client passes back to `POST /retract`. This satisfies the forward-compat requirement and saves a side-table.

## Throughput summary

See `11.5-throughput-row.md`. First table-write baseline. macOS fsync (~7.4 ms F_FULLSYNC hw-class limit) bounds upsert EPS at ~1000/sec, mirroring the Phase 7.5 stream-push baseline. Reads are pure in-memory MVCC and run in the µs range.
