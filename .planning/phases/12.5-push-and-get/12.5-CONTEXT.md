# Phase 12.5 — `push_and_get` combined endpoint

## Goal

Add a single-round-trip combined push + feature-query endpoint that applies a
push and queries features for an entity key **atomically** under the same
single-writer borrow scope. Read-your-writes by construction, ~2× latency win
on the flagship fraud-decisioning hot path, zero change to throughput.

## Motivation (one-liner)

The canonical fraud-scoring loop today is two RTs: `push` → ack, then `get` →
features. On a LAN that's ~400–500μs dominated by network, not server compute.
One combined endpoint collapses to ~250μs and makes "score this transaction
including its own effects" atomic by construction — no client-side "what if
another push raced me" concern.

See `.planning/ideas/push-and-get-endpoint.md` for the full proposal and
`.planning/ideas/throughput-ceiling-analysis.md` for the latency accounting.

## Success criteria

- **SC1** — `POST /push-and-get/{event_name}` with `{row, query}` body returns
  200 with `{ack_lsn, registry_version, features, warnings?}`. The push commits
  AND the features reflect the newly-pushed event atomically (read-your-writes).
- **SC2** — `POST /push-sync-and-get/{event_name}` returns ONLY after WAL fsync
  completes (acks=all parity with `/push-sync`).
- **SC3** — TCP `OP_PUSH_AND_GET (0x0015)` and `OP_PUSH_SYNC_AND_GET (0x0016)`
  expose the same behavior with CT_JSON payloads. Strict-FIFO connection
  semantics preserved; response op echoes request op.
- **SC4** — Python SDK `app.push_and_get(EventType, row={...}, entity_key={...},
  features=[...], sync=False)` returns a `(ack, features_dict)` tuple. `sync=True`
  uses the acks=all path.
- **SC5** — Unknown feature in `query.features` returns 200 with that feature
  mapped to `null` in `features` and the feature name added to `warnings[]`.
  The push itself still commits and its ack_lsn is returned. (Unknown feature
  is NOT a push-time failure.)
- **SC6** — Latency on simple-fraud shape (1 push + 3 feature reads) on Apple-M4
  LAN loopback: P50 < 300μs end-to-end over HTTP; documented baseline row
  appended to `.planning/perf-baselines.md` and
  `.planning/throughput-baselines.md`.

## Out of scope (deferred)

- `push_and_get_multi` — push one event, query N entity keys in one RT. Noted
  in the idea doc as v0.1 variant.
- `push_many_and_get` — batch push + one feature query. Same.
- `/metrics` wiring for the combined endpoint — inherits Phase 13 infrastructure
  once 13-05 ships the per-endpoint counter middleware. Phase 12.5 adds the
  route label to the existing middleware allow-list, no new metric counters.
- MessagePack on these opcodes — CT_JSON only in v0 (parity with `OP_PUSH`).

## Locked decisions

**D-01 — Atomic under one borrow.**
Push applies, then feature query runs, both inside a single `borrow_mut` on
the apply state. No intermediate await point. Read-your-writes is guaranteed
by construction. Rationale: the user-facing fraud flow ("score a tx including
its own effects") becomes trivially correct; no race window; no client-side
coordination.

**D-02 — Two endpoints: acks=1 + acks=all.**
Default `POST /push-and-get` = acks=1 (periodic fsync, matches `/push`);
`POST /push-sync-and-get` = acks=all (per-event fsync before ACK, matches
`/push-sync`). Python SDK surfaces this via `sync=False/True` kwarg.

**D-03 — TCP opcodes.**
New: `OP_PUSH_AND_GET = 0x0015`, `OP_PUSH_SYNC_AND_GET = 0x0016`. Slotted
after `OP_DELETE_TABLE (0x0014)` and before the `OP_GET (0x0020)` block. Both
added to the canonical opcode table in `crates/beava-core/src/wire.rs` with
full `opcode_name` coverage and "wired in Phase 12.5" status. `reserved_phase`
returns `None` for both (implemented). Response frames echo the request op.

**D-04 — Reuse existing service functions.**
The handler calls `execute_push` (already in `push.rs`) and the feature-query
service function (created in Phase 12 Plan 12-01 Task 1 — `push_get_api.rs` —
or lifted from `feature_query.rs` helpers if 12-01 isn't landed in this
worktree). Atomicity is about the `borrow_mut` scope, NOT duplicated logic.
If the existing `execute_push` releases its lock before returning (it does —
it takes `state_tables.lock()` internally), the new handler MUST be rewritten
to inline the apply+query pair under ONE lock acquisition. See grey area #1.

**D-05 — Request body shape.**
```json
{
  "row": { ...event row fields... },
  "query": {
    "entity_key": { "user_id": "abc", "card_id": "xyz" },
    "features": ["f1", "f2", "f3"]
  }
}
```
Composite entity keys supported (matches `POST /get` POST body shape for
multi-key group_by). The `entity_key` is a JSON object mapping group_key
names to string values. Server builds `EntityKey` from it using the same
resolver each requested feature's aggregation descriptor defines.

**D-06 — Response body shape.**
```json
{
  "ack_lsn": 12345,
  "registry_version": 42,
  "features": { "f1": 17, "f2": 3, "f3": 38.20 },
  "warnings": []
}
```
- Success (all features resolved): `warnings` omitted OR `[]`.
- Unknown feature: `features["unknown_feat"] = null` + `warnings: ["unknown_feature: unknown_feat"]`.
- Unknown entity (feature exists, key not seen): `features["f"] = null` (no warning; this is normal).
- Push failure: 4xx/5xx with `{error: {code}}`, no `features` or `ack_lsn` in body.

**D-07 — Python SDK surface.**
```python
ack, features = app.push_and_get(
    EventType,
    row={...},
    entity_key={"user_id": "abc"},
    features=["f1", "f2"],
    sync=False,  # acks=1 (default); True = acks=all via /push-sync-and-get
)
# ack is a PushAck namedtuple; features is dict[str, Any | None]
```
Sync client only in this phase (the async variant stays a v0.1 follow-up —
the existing `beava` Python client is sync-first). Transport auto-dispatches
based on `bv.App("http://...")` vs `bv.App("tcp://...")` URL scheme, matching
existing push/get methods.

**D-08 — Join resolution uses event-time PIT.**
When a queried feature reads from a Phase 12 event↔table join, the query runs
AFTER the push applies — so `event.event_time_ms` from the just-pushed event
is the correct "as_of" the join should see. The feature-query path calls
`lookup_at_event_time(key, event.event_time_ms)` from Phase 15's event-time
temporal store. This makes `push_and_get` fully event-time PIT correct: the
feature value reflects table state at the pushed event's event-time, not
wall-clock. Phase 15 must land before Phase 12.5 for this to hold; without
Phase 15 the behavior degrades to LSN-keyed lookup (correct for linear
arrival, slightly wrong for out-of-order joins — same caveat as today's
Phase 11.5 temporal store).

## Dependencies

- **Phase 12** (joins + push/get API completion) — MUST land first. Phase 12.5
  builds on the settled `/push`, `/get`, `/push-sync` surfaces and (if Plan
  12-01 landed a dedicated `push_get_api.rs` service module) reuses its
  feature-query service function. Without Phase 12, event↔table join
  resolution for queried features is unavailable.
- **Phase 15** (event-time PIT temporal store) — MUST land first for join
  queries inside `push_and_get` to be event-time correct. Without 15 the
  path still works for pure-aggregation queries (no temporal-table joins);
  join-through-push_and_get is approximate (linear-arrival only).
- **Phase 13.3** (lockless apply via RefCell + LocalSet) — strongly recommended
  to land first. The atomicity requirement (push + query under one borrow) is
  much cleaner in the post-13.3 apply shape where state is owned by a single
  RefCell. Pre-13.3 the Mutex makes "one lock acquisition across apply + query"
  awkward but doable. If 13.3 slips, Phase 12.5 can still ship against the
  Mutex with a single `lock()` scope spanning both.

## Grey areas (surface-able uncertainty)

**GA1 — `borrow_mut` scope vs. fsync await (CRITICAL).**
The `/push-sync-and-get` path must fsync the WAL BEFORE returning the ack AND
BEFORE running the feature query. But:

- CLAUDE.md lint (and clippy's `await_holding_refcell_ref`) forbids holding a
  `RefCell` borrow across `.await`.
- fsync is off-thread via `spawn_blocking` (Phase 13.1), so waiting for it is
  an `.await`.

**Resolution locked for this phase:**
The `borrow_mut` scope releases BEFORE the fsync await. Sequence for
`/push-sync-and-get`:
1. Acquire borrow_mut.
2. Append to in-memory WAL buffer, apply to aggregations, query features —
   all inside the borrow.
3. Drop borrow.
4. `.await` the fsync oneshot.
5. Respond with `{ack_lsn, features, ...}`.

Atomicity between apply and query is preserved (both inside one borrow).
Durability ordering is preserved (response returns only after fsync resolves).
This matches the existing `execute_push` PerEvent path's lock discipline.

For `/push-and-get` (acks=1): steps 1–3 identical; no fsync await; response
returns immediately after drop. Even simpler.

**Why this works for read-your-writes:** the feature query runs under the
SAME borrow as the apply, so the query sees the just-applied state. The
fsync await happens AFTER the borrow is dropped; no other apply can
interleave in the meantime (single-writer invariant) because the fsync
oneshot doesn't yield the apply loop until the response is sent.

**GA2 — Phase 12 `push_get_api.rs` existence.**
The prompt notes Plan 12-01 Task 1 shipped a `push_get_api.rs` service module.
In this worktree (`v2/greenfield` @ `03cff58`) that file does not exist —
Phase 12 work lives on `.claude/worktrees/phase-12-followup`. Before Phase
12.5 executes:
- If Phase 12 has merged into `v2/greenfield`, Plan 12.5-01 reuses
  `push_get_api::query_feature` (or similar).
- If not, Plan 12.5-01 extracts a small `query_feature_for_entity` helper
  from `feature_query.rs` first, then uses it. No change in scope — same
  logic, different source of the helper.

**GA3 — Schema propagation for `entity_key` shape.**
The POST body uses `{entity_key: {k1: v1, k2: v2}}` — object-shaped, keyed
by group-key names. The existing `POST /get` uses a different shape:
`{keys: ["alice|merchant1"], features: [...]}` with pipe-separated composite
keys. Phase 12.5 picks the object shape (D-05) because it's self-describing
and doesn't require clients to know the pipe-separator encoding. This
diverges from `POST /get`'s shape — NOT a backward-compat break because
`push_and_get` is a new endpoint. The SDK hides the difference.

**GA4 — Unknown feature vs. unknown entity disambiguation.**
D-06 splits these: unknown feature = `null` + `warnings` entry; unknown entity
= `null` + no warning. Rationale: "unknown feature" is a programming error
the caller wants to surface loudly (misspelled feature name); "unknown entity"
is a normal runtime state (new user, no events seen yet). Both return `null`
so the shape is uniform; the `warnings[]` array is the disambiguator.

**GA5 — Latency target rigor.**
SC6 asserts P50 < 300μs over HTTP on Apple-M4. The ~500μs → ~250μs claim in
the idea doc includes two RTTs collapsing to one. Single-event HTTP on M4
today is ~5μs server compute + ~200μs RTT (loopback). Two round-trips ≈ 400μs
total; one round-trip ≈ 200μs. 300μs is the verification threshold (50%
margin above theoretical), not the goal. If we miss < 300μs, the Plan 03
latency bench flags the gap; investigation is part of the plan, not a
separate spike.
