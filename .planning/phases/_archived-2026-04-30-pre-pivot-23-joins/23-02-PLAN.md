<objective>
Ship Stream↔Stream symmetric interval joins (inner + left) with per-key event-time-indexed buffers and within-bounded eviction.

Purpose: This is the most state-heavy of the three join shapes — it requires bidirectional per-key buffers and probe-on-arrival semantics. We land it after 23-01 because 23-01 establishes the `v0_join_to_stream_def` plumbing and the `EnrichFromTable` cascade pattern. 23-02 swaps in a new FeatureDef variant and OperatorState variant while preserving the dispatch skeleton.

For v0 (no watermarks yet — Phase 24), we use wall-clock arrival order as an approximation of event-time ordering. The research (join-outer-needed.md §4, "retraction doubling") confirms outer is deferred; this plan ships inner+left only. Phase 24 will wire the 5-second watermark to finalize late-event retraction semantics; for now we accept that a 5s-late event can trigger a retroactive match (test explicitly covers this) but DO NOT yet emit retractions for inner joins (retraction propagation lands in Phase 24).

Output: A Stream derivation registered with shape='stream_stream' correctly emits joined events as they arrive, buffers bounded by `within`, evictions tested, retroactive matches tested.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TCP PUSH → StreamJoinBuffer insert | Event-time field (untrusted) determines buffer index; malformed event_time must not crash or DoS |
| Buffer growth vs eviction | Unbounded growth if eviction doesn't keep pace with inserts |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-05 | Denial of Service | StreamJoinBuffer per-key growth | mitigate | Eviction runs on every insert (floor = max_seen - within). At 100k eps × 30m window × 2 sides = ~360M events worst case PER KEY — memory bounded by `within` only, not by total event volume. Document in 23-02-SUMMARY that extreme per-key fan-in could still OOM; Phase 25 (/debug/warnings) will surface |
| T-23-06 | Tampering | `_event_time` field from client | mitigate | `parse_event_time` (Phase 22) already handles missing/invalid → wall-clock fallback. Numeric-bounds check: clamp to `[0, 2^63-1]` to avoid BTreeMap index overflow. Negative or non-numeric event_time falls back to wall-clock (existing behavior) |
| T-23-07 | Repudiation | Retroactive match double-emit | accept | v0 emits both the null-pair and the later matched pair for left joins without retraction. This is a KNOWN v0 limitation, documented in task `ss_retroactive_match` and the SUMMARY. Phase 24 fixes with watermark-bounded retractions |
</threat_model>
