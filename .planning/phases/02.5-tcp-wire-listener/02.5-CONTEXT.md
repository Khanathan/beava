# Phase 2.5: TCP wire listener + framing + full opcode table - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning
**Mode:** Interactive discuss under `/gsd-autonomous --interactive`

<domain>
## Phase Boundary

Add a second server listener — a custom-framed binary TCP protocol — alongside the existing HTTP/JSON listener from Phase 2. Ship a full v0 opcode table (so Phase 6 / 12 / 13 fill in handlers without touching the codec), but only implement the `register` + `ping` handlers now. Every other opcode returns a structured `op_not_implemented` response frame.

What this phase delivers:
- Rust TCP listener bound on a configurable port (default `7380`), default-on alongside HTTP (`7379`)
- Frame codec for `[u32 length][u16 op][u8 content_type][payload]` with round-trip proptest coverage
- Full v0 opcode table defined in one place (shared `u16` constants): `ping`, `register`, `push`, `push_sync`, `push_many`, `get`, `mget`, `set`, `mset`, + dedicated `error_response` opcode
- `register` handler shares the Phase 2 validation + diff + apply engine — no logic duplication
- `ping` handler returns server `registry_version` + build version
- Redis-style strict FIFO on one connection: pipelining comes for free (client sends N, server responds N in order); no `request_id` in the frame because order guarantees correlation
- Single tokio `current_thread` runtime hosts accept loop + per-connection read/decode/execute/write — all one OS thread (matches Beava's single-apply-thread constraint)
- Integration smoke: `phase2_5_smoke.rs` — spawn server with both ports, exercise ping + register + unknown-op over TCP

Out of scope (explicitly deferred):
- `push` / `push_sync` / `push_many` / `get` / `mget` / `set` / `mset` handler bodies — reserved opcode, handler returns `op_not_implemented` response frame (picked up in Phases 6 + 12)
- MessagePack payload encoding — wire reserves content-type `0x02`; Phase 6 / 12 wire it in for hot-path ops. Phase 2.5 only implements `0x01` (JSON)
- TLS on the TCP listener — reverse-proxy terminates; no rustls dep (matches HTTP stance)
- Threaded-I/O (Redis 6+ `io-threads N` style) — deferred to Phase 13 if perf gates fail
- Subscribe / streaming-push opcodes — reserved opcode range, not wired
- v1's `beava._protocol.py` TCP framing — NOT reused; v2 is its own frame shape

</domain>

<decisions>
## Implementation Decisions

### D-01: Frame envelope format (LOCKED — drives the whole codec)

```
[u32 length-big-endian][u16 op-big-endian][u8 content_type][payload: length - 3 bytes]
```

- `length` = total bytes of `op + content_type + payload` (i.e. everything after the length field itself)
- `op` = opcode; see D-02
- `content_type` = 1-byte discriminator for the `payload` encoding; `0x01` = JSON (v0); `0x02` reserved for MessagePack (v0.x); `0xFF` reserved (internal test / fuzz marker, unused in v0)
- `payload` = `content_type`-encoded request or response body; same JSON shape as the HTTP equivalent for v0
- NO `request_id` field. Strict FIFO on one connection; client correlates by order (Redis RESP style). This removes a 4-byte-per-frame footgun (duplicate IDs) and keeps the frame minimal.

Max frame size: `4 MiB` default (configurable via `tcp.max_frame_bytes`). Oversize → the server writes a `frame_too_large` error response, then closes the connection (client can reconnect).

All multi-byte integers are big-endian (network byte order), matching standard practice and `tokio_util::codec`'s built-in `LengthDelimitedCodec`.

### D-02: Opcode table (LOCKED — full v0 space designed now)

Opcodes are `u16` constants defined once in `crates/beava-core/src/wire.rs`. Request opcodes are even (`0x0000`..`0x7FFE`); responses are NOT distinguished from requests by opcode — the client-server direction is implicit, and errors use a dedicated opcode (see D-03).

| Opcode | Name | Wired in this phase? | Notes |
|--------|------|----------------------|-------|
| `0x0000` | `ping` | Yes | Request: empty payload (or ignored). Response: `{server_version, registry_version}` JSON |
| `0x0001` | `register` | Yes | Request: same JSON DAG as `POST /register`. Response: same 200/400/409 JSON shape, all over opcode `0x0001` for success; errors via D-03 |
| `0x0010` | `push` | Reserved (Phase 6) | Fire-and-forget enqueue; no response frame (server may drop or ACK on success — Phase 6 decides) |
| `0x0011` | `push_sync` | Reserved (Phase 12) | Push + return FeatureResult synchronously |
| `0x0012` | `push_many` | Reserved (Phase 12) | Batched push |
| `0x0013` | `push_table` | Reserved (Phase 12) | Table upsert |
| `0x0014` | `delete_table` | Reserved (Phase 12) | Table tombstone |
| `0x0020` | `get` | Reserved (Phase 12) | Single-key feature read |
| `0x0021` | `mget` | Reserved (Phase 12) | Batched feature read |
| `0x0022` | `get_multi` | Reserved (Phase 12) | Multi-descriptor batched read |
| `0x0030` | `set` | Reserved (Phase 12) | Direct feature write |
| `0x0031` | `mset` | Reserved (Phase 12) | Batched direct write |
| `0xFFFF` | `error_response` | Yes | See D-03 |

Reserved opcode handlers return a frame with `op=0xFFFF` and payload `{error: {code: "op_not_implemented", message: "opcode <hex> is reserved for <phase>"}, registry_version: N}`. Connection stays open — client can try other opcodes.

Unknown opcodes (not in the table) return a `error_response` with `code: "unknown_op"`.

### D-03: Error response framing (LOCKED)

Any failure — frame-level, validation, conflict, server error — writes ONE frame with `op=0xFFFF`, `content_type=0x01` (JSON), and a payload matching the HTTP error body shape from Phase 2:

```json
{"error": {"code": "invalid_registration", "path": "nodes[2].upstreams[0]", "reason": "..."}, "registry_version": 3}
```

Error codes reuse HTTP codes where applicable (`invalid_registration`, `registration_conflict`, `unsupported_media_type` → becomes `unsupported_content_type` for TCP since there's no HTTP "content-type" header). New codes for TCP-specific frame issues:
- `op_not_implemented` — reserved opcode not wired yet (Phase 6 / 12 / 13)
- `unknown_op` — opcode not in the table
- `frame_too_large` — exceeds `max_frame_bytes`; server closes connection after writing this frame
- `malformed_frame` — length prefix + content_type + payload don't parse
- `unsupported_content_type` — `content_type` byte unknown (e.g., MessagePack before Phase 6 implements it)

Rationale: SDK has ONE error parser, ONE `ValidationError`-equivalent type, ONE code path for both HTTP and TCP. Shared Phase 2 error shape means `curl` debugging parity.

### D-04: Concurrency + threading model (LOCKED — Redis-style)

- One `tokio::runtime::Builder::new_current_thread()` runtime per `beava` process (unchanged from Phase 1 / 2)
- HTTP listener: existing axum task on the same runtime (unchanged from Phase 2)
- TCP listener: new task spawns an accept loop on the same runtime
- Each accepted connection: one task reads frames, dispatches to the appropriate handler, writes response frames — all on the SAME OS thread. Registry is shared as `Arc<Registry>` (which wraps `parking_lot::RwLock<RegistryInner>` — kept from Phase 2; transitioning to a single-thread `RefCell` is a separate tightening that can happen when the apply loop lands in Phase 5, not here)
- Strict FIFO per connection: the read loop awaits the handler completing before reading the next frame → natural serialization of responses without out-of-order replies. Pipelining = client can send N frames without waiting; server reads/dispatches/writes one at a time on that connection. Redis RESP model exactly.
- Cross-connection interleaving: multiple connections are independent tasks on the single OS thread; tokio schedules them cooperatively. No cross-connection ordering guarantee (intentional — same as Redis).
- Shutdown: graceful. Existing Phase 1 shutdown signal drains in-flight HTTP requests; Phase 2.5 extends it to drain in-flight TCP handlers. New connections rejected with `SERVER_SHUTTING_DOWN` error response, then TCP close.

Threaded-I/O (Redis 6+ `io-threads N` style) is deferred to Phase 13 only if the perf harness shows network as the bottleneck. For v0 register+ping workloads (admin-path, not hot), it wouldn't help anyway.

### D-05: Payload encoding per op (LOCKED, with staged rollout)

- `content_type=0x01` = JSON — implemented in Phase 2.5 for `ping` and `register` (matches HTTP wire exactly; same `serde_json` codec)
- `content_type=0x02` = MessagePack — **reserved** in Phase 2.5; NOT implemented. Phase 6 wires it in for `push` / `push_sync` / `push_many`; Phase 12 for `get` / `mget` / `get_multi` / `set` / `mset`. Choice rationale: hot-path ops benefit from ~3× byte reduction; admin-path ops stay on JSON for debugging parity with curl
- Server behavior when it receives `content_type=0x02` in v0: respond with `error_response { code: "unsupported_content_type", reason: "MessagePack payload encoding ships in Phase 6 / 12" }`. Server never rejects the connection — only the frame

The SDK (Phase 3) picks content type per opcode automatically; users never touch this.

### D-06: Default port + default-on (LOCKED)

- `beava` binds BOTH listeners at startup by default
- HTTP: `127.0.0.1:7379` (unchanged; Phase 1 default)
- TCP: `127.0.0.1:7380` (new)
- Config keys (YAML + env var):
  - `tcp.enabled` (default `true`) / `BEAVA_TCP_ENABLED=1`
  - `tcp.host` (default `127.0.0.1`) / `BEAVA_TCP_HOST`
  - `tcp.port` (default `7380`) / `BEAVA_TCP_PORT`
  - `tcp.max_frame_bytes` (default `4194304` = 4 MiB) / `BEAVA_TCP_MAX_FRAME_BYTES`
- Disable with `tcp.enabled: false` or `BEAVA_TCP_ENABLED=0`
- NO TLS in v0 — terminate at reverse proxy (nginx / Envoy / Cloudflare). Matches HTTP stance and "single binary, zero surprises" aesthetic
- Startup log emits two lines (one per listener) at INFO; both include `addr` field

### D-07: Wire-contract location + code organization (LOCKED)

- `crates/beava-core/src/wire.rs` — new module. Contains:
  - `pub const OP_PING: u16 = 0x0000;` etc. (the opcode table)
  - `pub const CT_JSON: u8 = 0x01;` `pub const CT_MSGPACK: u8 = 0x02;`
  - `Frame` struct: `{ op: u16, content_type: u8, payload: Bytes }`
  - `pub fn encode_frame(frame: &Frame, out: &mut BytesMut)` and `decode_frame(buf: &mut BytesMut) -> Result<Option<Frame>, FrameError>` — uses `tokio_util::codec::LengthDelimitedCodec` or hand-rolled equivalent
  - Proptest coverage: `(op, ct, payload)` → bytes → parsed frame is byte-identical
- `crates/beava-server/src/tcp.rs` — new module. Contains:
  - `pub struct TcpListener { ... }` with `bind(config, registry) -> Result<Self>` and `accept_loop()` spawning per-connection tasks
  - Per-connection loop: read frame → dispatch → write response; strict FIFO, no parallelism
  - Handler dispatch: match on `op` → call `handle_ping`, `handle_register`, or return `op_not_implemented`
- `crates/beava-server/src/register.rs` — existing. Extract the validation + diff + apply logic into a pure function callable from BOTH the HTTP handler and the TCP `handle_register` — no duplicated endpoints. Likely ends up as a module-level `pub(crate) async fn execute_register(registry: &Registry, payload: RegisterPayload) -> RegisterOutcome` that both sides call
- Config: extend `crates/beava-core/src/config.rs` with a new `TcpConfig` struct (port, host, enabled, max_frame_bytes)
- Test harness: `crates/beava-server/src/testing.rs` — add `TestServerBuilder::tcp_port()` (OS-assigned 0 by default), `TestServer::tcp_addr()` accessor, and a `ts.tcp_client() -> TcpClient` helper with `send_frame(op, ct, payload).await -> Response` semantics
- Integration smoke: `crates/beava-server/tests/phase2_5_smoke.rs` — new file, exercises both ports

### D-08: Applied devex-first naming principles (project-wide, not just this phase)

- All user-facing names are plain English; streaming jargon purged
- Sensible defaults baked in everywhere — `@bv.event` works zero-config, tables must declare `key`, every soft knob has a default
- Applied retroactively to Phase 2 wire in commit `26daa41`; doc catch-up in `b534c97`
- Phase 2.5 + Phase 3 must keep this discipline — if a new knob is introduced, document the default in `crates/beava-core/src/defaults.rs`

### Claude's Discretion

Locked by the decisions above are the load-bearing choices. Claude decides:
- Tokio codec framework: `tokio_util::codec::LengthDelimitedCodec` vs hand-rolled. Recommend `LengthDelimitedCodec` for robustness unless it costs measurable perf — in which case drop to hand-rolled `poll_read` + `poll_write`
- Byte buffer type inside handlers: `Bytes` / `BytesMut` vs `Vec<u8>`. Recommend `Bytes` — it's already a tokio dep
- How the TCP accept loop reports listener errors vs per-connection errors in `tracing` — follow Phase 1 patterns
- Which exact handler wire format for `ping` response JSON (e.g., `{server_version: "beava/0.3.0", registry_version: 7}` vs `{v: "0.3.0", ver: 7}`). Pick the verbose form for debuggability
- Shutdown-signal plumbing internals (extend existing `CancellationToken` or new one per listener)
- Whether to write a single connection-state machine struct or inline the read-loop inside a tokio task body — whichever keeps the tcp.rs module < 400 LoC

None — Phase 2.5 was inserted mid-milestone; the parking-lot backlog is untouched.

<specifics>
## Specific Ideas

- **Frame codec smoke target**: proptest generates `(op: u16, content_type: u8, payload: Vec<u8> of length 0..10_000)`, round-trips through `encode_frame` + `decode_frame`, asserts byte-identical. Edge cases: empty payload, payload exactly at `max_frame_bytes`, `max_frame_bytes + 1` (expect `frame_too_large`).
- **Test harness TCP client**: a tiny async helper struct `TcpClient { stream: TcpStream, read_buf: BytesMut }` with `ping() -> PingResponse`, `register(payload: serde_json::Value) -> RegisterResponse`, `send_raw(op, ct, payload) -> RawResponse`. Lives in `testing.rs`. Keep it simple; the production SDK client lives in Python (Phase 3).
- **The `phase2_5_smoke.rs` acceptance gate** should prove the 8 success criteria from ROADMAP.md Phase 2.5 end-to-end — double-binding, round-trip proptest, ping returning `registry_version`, TCP-register equivalent to HTTP-register, unknown-op error, pipelining with N=3 registers on one connection, oversize-frame behavior, graceful shutdown.
- **Don't write a TCP benchmark in Phase 2.5** — Phase 13 owns perf gates. Phase 2.5 smoke cares about correctness: frame parses, handlers fire, errors frame correctly, server shuts down cleanly.
- **Don't pre-implement MessagePack** — the ct byte and error path is enough. Phase 6 / 12 add the msgpack codec when they need it.
- **Document the opcode table in one canonical place** — `crates/beava-core/src/wire.rs` module doc comment. Later phases update it (flip "reserved" to "implemented in Phase N"). SDK pulls from this doc for its own docs.

</specifics>

<deferred>
## Deferred Ideas

- **MessagePack payload codec** — Phase 6 (for push/push_sync/push_many) or Phase 12 (for get family)
- **Request-ID multiplexing on one connection** — v0.x or later. Frame has no slot for it; adding one requires a wire-version bump, which we're intentionally putting off
- **Subscribe / streaming-push opcodes** — v0.x; reserved opcode range `0x0040..0x00FF`
- **TLS on TCP listener (rustls)** — v0.x; `tcp.tls = { cert, key }` config stub not yet implemented
- **Threaded I/O (`io-threads N`)** — Phase 13 if perf gates fail. Not a Phase 2.5 concern
- **TCP keepalive tuning** — tokio defaults stand; operators can tune via OS
- **Connection-level authentication** — no auth in v0 OSS (matches HTTP); commercial tier adds it
- **Bidirectional streaming (server pushes without client request)** — out of scope forever in v0; subscribe ops cover the one legitimate use case
- **Protocol buffers / FlatBuffers** — explicitly excluded in PROJECT.md Out-of-Scope
- **Phase 2's original "v0 is HTTP-only" positioning** — reviewed by the 2026-04-23 amendment; Phase 2.5 supersedes it
