The crucial finding: the original Phase 13.1 "10× regression" was an
apples-to-oranges comparison. Phase 6.1's 15.7k EPS used `--parallel 64`;
the post-merge rerun used `--parallel 8`. At equal scale (parallel=64) the
post-merge codebase was already hitting Phase 6.1 throughput levels even
before the spawn_blocking fix (the bench harness uses tokio multi_thread,
which masks the inline-fsync runtime stall by giving it a different
worker thread to block).

The spawn_blocking fix is still correct and necessary for the *production*
server, which runs on `current_thread`. Verified via the unit-level
`fsync_non_blocking::fsync_does_not_block_runtime_tasks` test, which
goes from FAIL to PASS.
