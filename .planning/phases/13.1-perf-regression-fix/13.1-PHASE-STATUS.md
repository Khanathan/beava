## Investigation summary

The "10× /push throughput regression" documented in
`.planning/throughput-baselines.md` § "POST-MERGE quiescent rerun" turned
out to have **two distinct root causes** that were conflated:

1. **Apples-to-oranges measurement scale.** Phase 6.1's reported 15,672 EPS
   (small/HTTP) was measured at `--parallel 64`. The post-merge quiescent
   rerun used `--parallel 8`. The Periodic-mode group-commit ceiling for
   the in-process `TestServer` benchmark is ~2,200 EPS at parallel=8 and
   ~17,000 EPS at parallel=64 — the post-merge rerun was simply
   parallelism-bound, not regressed. Re-measuring the post-merge codebase
   at `--parallel 64` yields 16,996 EPS (small/HTTP), already at the
   Phase 6.1 regime.

2. **Inline fsync stalls the production runtime.** A separate, real bug:
   the Phase 6.1 documented deviation ("Inline fsync instead of
   `spawn_blocking` … current_thread runtime used in tests has no
   blocking pool") rests on a false premise. tokio's `current_thread`
   runtime DOES have a blocking pool. The production server runs
   `current_thread` (`crates/beava-server/src/main.rs:37`), so the inline
   `writer_ref.sync_data()` call (`crates/beava-persistence/src/fsync_worker.rs:439`
   pre-fix) blocks the executor thread for the duration of `F_FULLSYNC`
   (~7 ms on macOS). During that window no other tokio task on the
   runtime can make progress — including the HTTP push handler racing
   to ACK the next request. The benchmark harness uses tokio
   `multi_thread`, so the bug is invisible in benches, but real users
   running the production binary feel it immediately.

## Profile tool used

samply (release build with `CARGO_PROFILE_RELEASE_DEBUG=line-tables-only`
and `RUSTFLAGS="-C force-frame-pointers=yes"`). The captured profile
showed the bench saturated multiple `tokio-rt-worker` threads with
roughly even sample distribution — consistent with the multi_thread
runtime masking the inline-fsync stall and the bench actually being
parallelism-bound at parallel=8.

The decisive evidence came from a more targeted unit-level reproducer:
`crates/beava-persistence/tests/fsync_non_blocking.rs` runs the
fsync_worker on a `current_thread` runtime alongside a 1ms-tick
counter task, then issues 200 PerEvent appends. Pre-fix the ticker
fires only ~75% as often as wall-clock ticks (clear runtime stall).
Post-fix the ticker keeps perfect time.

## Root cause

`crates/beava-persistence/src/fsync_worker.rs:439` (pre-fix):

```rust
// fsync inline. (See historical note in Phase 6: spawn_blocking
// conflicts with the current_thread runtime used in tests; the worker
// IS its own task in production.)
let sync_result = writer_ref.sync_data().map_err(PersistError::Io);
```

`sync_data()` calls `self.file.flush()` (cheap) then
`self.file.get_mut().sync_data()` (the actual `F_FULLSYNC` syscall on
macOS, ~7 ms blocking). On `current_thread` this stalls the executor.

## Fix applied

Approach: **`tokio::task::spawn_blocking` for the fsync syscall.**

- `WalWriter::flush_buffer()` — cheap in-memory flush, runs on the
  runtime thread.
- `WalWriter::try_clone_file()` — produces an OS-cloned `File` handle
  (shares the same kernel file description, so a fsync on the clone
  durably persists previously-flushed bytes).
- `fsync_worker::blocking_sync_data()` — async helper that flushes,
  clones, then dispatches `cloned.sync_data()` via `spawn_blocking`.
  Falls back to inline fsync if clone fails (preserves correctness on
  FD exhaustion).
- `fsync_worker::fsync_batch()` becomes async; all `worker_loop` call
  sites updated to `.await`.

Diff: 3 files, +62/-16 lines. Commit: `2f3a092`.

## Before/after numbers

See `13.1-throughput-row.md` for the full per-cell table.

| Pipeline | Transport | Parallel | Pre-fix EPS | Post-fix EPS |
|---|---|---:|---:|---:|
| small  | http | 8  | 1659 | 2269 |
| small  | tcp  | 8  | 1944 | 2242 |
| small  | http | 64 | 16996 | 17071 |

At parallel=8 the modest improvement (1659 → 2269, 1.37×) reflects
that the bench harness is parallelism-bound there; the spawn_blocking
fix doesn't materially change a workload that has nothing else to do
during the fsync. At parallel=64 the bench was already saturated by
the multi_thread runtime giving fsync its own blocked thread; the fix
preserves that throughput while also being correct for production
`current_thread`.

## Follow-ups for Phase 13 ship-gate

1. **Update the canonical throughput baseline** to use `--parallel 64`
   consistently. The `--parallel 8` rerun confused the diagnosis. The
   Phase 13 ship gate's 3M EPS/core target needs a saturating
   parallelism setting anyway.

2. **Single-conn ceiling at parallel=1 (286 EPS, P50 3ms)** is still
   real and reflects per-RTT serialization through the Periodic timer.
   Phase 13.2's coalescing work should help; Phase 13's TCP fast-path
   pipelining (if pursued) would also help.

3. **Apply-loop lock contention** is the next bottleneck once fsync is
   off the critical path. At parallel=64 we plateau at ~17k EPS; the
   in-memory state-table mutex (`crates/beava-server/src/push.rs:303`)
   serializes all 64 in-flight pushes through the apply step. A
   per-key sharded lock or an apply-batch handoff would unlock further
   throughput. **This is the right next investigation for Phase 13's
   3M EPS/core target.** Out of scope for 13.1.
