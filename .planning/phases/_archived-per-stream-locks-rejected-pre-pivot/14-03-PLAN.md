**Gate criteria:**
- PASS if 4-client async medium eps > 28k (Phase 12 baseline) — ideally 2-4x improvement
- PASS if 1-client async medium eps >= 128k (within -10% of 142k baseline)
- PASS if 1-client sync p99 <= 99us (within -10% of 90us baseline)
- Any FAIL requires investigation — do NOT paper over regressions

**If single-client regression is observed (>10% drop):**
Likely cause: DashMap per-access overhead (~5-10ns hash+shard lookup) adds up at 142k eps. Check if the overhead is acceptable or if the hot path needs optimization (e.g., caching the DashMap shard reference across operator pushes within the same entity).

**If multi-client shows NO improvement:**
The tokio runtime may still be single-threaded. Check that main.rs uses `#[tokio::main]` with default multi-thread flavor (not `flavor = "current_thread"`). The DashMap concurrency only helps if tokio actually schedules tasks on multiple OS threads.
