<objective>
Replace the global `Arc<Mutex<AppState>>` with a concurrent `ConcurrentAppState` that uses individually-locked fields: `DashMap<EntityKey, StreamEntityState>` per stream for entity-level concurrency, `parking_lot::RwLock` for PipelineEngine, and separate small locks for metrics/snapshot/throughput/latency state. Refactor ALL hot-path handlers (handle_push_batch, handle_push_core_ex, handle_sync_command for Get/Set/Mget/Mset/Register) and main.rs startup to use the new structure.

Purpose: Eliminate the single global lock that serializes ALL connections, enabling concurrent processing of events targeting different streams and different entity keys.
Output: Compiling, all-tests-green codebase with ConcurrentAppState and zero `state.lock()` calls remaining.
</objective>

**Step 2: Define ConcurrentAppState in src/server/tcp.rs** (replacing AppState and SharedState)

Replace the existing `AppState` struct and `SharedState` type alias with:

```rust
use dashmap::DashMap;
use parking_lot::{Mutex as PLMutex, RwLock};

/// Per-stream entity storage. Each registered stream gets one.
pub struct StreamStore {
    pub entities: DashMap<String, StreamEntityState>,
    pub dirty_keys: PLMutex<AHashSet<String>>,
    pub deleted_keys: PLMutex<AHashSet<String>>,
}

/// Concurrent application state. Replaces Arc<Mutex<AppState>> (D-01).
/// Each field is independently lockable — no global mutex.
pub struct ConcurrentAppState {
    /// Stream definitions — RwLock: many concurrent reads on hot path,
    /// write only on REGISTER (D-04)
    pub engine: RwLock<PipelineEngine>,

    /// Per-stream entity state (D-02, D-03).
    /// Key: stream name. Value: StreamStore with DashMap<EntityKey, StreamEntityState>
    pub stream_stores: DashMap<String, StreamStore>,

    /// Static features from SET/MSET — independent DashMap
    pub static_store: DashMap<String, AHashMap<String, StaticFeature>>,

    /// Per-stream event log writers (D-05)
    pub event_log: Option<ConcurrentEventLog>,

    /// Metrics — can use AtomicU64 fields or small Mutex (D-11)
    pub metrics: PLMutex<Metrics>,

    /// Snapshot coordination
    pub snapshot_path: std::path::PathBuf,
    pub snapshot_cycle: PLMutex<u64>,
    pub snapshot_seq: PLMutex<u64>,
    pub last_base_seq: PLMutex<u64>,
    pub previous_base_seq: PLMutex<u64>,

    /// Backfill tracking
    pub backfill_tracker: Arc<BackfillTracker>,
    pub backfill_complete: PLMutex<HashSet<(String, String)>>,

    /// Throughput and latency — separate small locks
    pub throughput: PLMutex<ThroughputTracker>,
    pub latency: PLMutex<LatencyTracker>,
}

pub type SharedState = Arc<ConcurrentAppState>;
```

For `ConcurrentEventLog`, create a wrapper struct in `src/state/event_log.rs` (or alongside in tcp.rs for now) that wraps the existing `EventLog` fields with per-stream `PLMutex<BufWriter<File>>` writers. If the existing EventLog is hard to split, wrap the entire EventLog in `PLMutex<EventLog>` as a transitional step — the critical win is removing it from the global AppState mutex. The per-stream writer split can be refined in Wave 2 (Plan 02).

**Step 3: Refactor StateStore in src/state/store.rs**

The key transformation: iterate `self.entities`, and for each `(entity_key, EntityState)`, distribute the `streams` map entries into per-stream `StreamStore` DashMaps, and collect `static_features` into `static_store`.

**Hot path refactors (tcp.rs):**

1. **handle_push_core_ex** (tcp.rs ~line 556): Currently locks global state, calls `engine.push(...)`, updates store, logs events, tracks metrics. Refactor to:
   - `let engine = state.engine.read();` (parking_lot RwLock read)
   - Get stream definition from engine
   - `state.stream_stores.entry(stream_name).or_insert_with(StreamStore::new)` to get/create StreamStore
   - `stream_store.entities.entry(key).or_default()` to get/create entity via DashMap
   - Push to operators (mutate StreamEntityState via DashMap RefMut)
   - Drop RefMut BEFORE event log write
   - Event log: `state.event_log.as_ref().map(|el| el.append(...))` with per-stream lock inside
   - Metrics: `state.metrics.lock()`, `state.throughput.lock()`, `state.latency.lock()` — each a separate small lock

2. **handle_push_batch** (tcp.rs ~line 902): Currently takes one global lock for entire batch. Refactor to:
   - `let engine = state.engine.read();`
   - Group events by stream (already done in current code)
   - For each stream group: access that stream's DashMap, push events
   - Drop engine read guard after processing
   - Metrics updates via individual locks

3. **handle_sync_command(Get)** (tcp.rs ~line 1059): Currently locks to call `store.get_all_features()`. Refactor to:
   - `let engine = state.engine.read();`
   - Iterate `state.stream_stores`, for each stream: `entities.get_mut(key)` to read operators (get_mut needed because `op.read(now)` mutates per RESEARCH.md)
   - Overlay `state.static_store.get(key)`
   - Evaluate derives/views from engine

4. **handle_sync_command(Set)** (tcp.rs ~line 1079): Refactor to `state.static_store.entry(key).or_default().insert(name, StaticFeature{...})`

5. **handle_sync_command(Mset)** (tcp.rs ~line 1173): Same pattern as Set but in a loop with cooperative yielding

6. **handle_sync_command(Register)** (tcp.rs ~line 1109): `state.engine.write()` for exclusive access during registration. Create StreamStore entries for newly registered streams.

7. **handle_sync_command(Mget)** (tcp.rs ~line 1173): Scatter-gather: for each key, read from all stream DashMaps + static_store

8. **Fan-out** in handle_push_core_ex: When processing fan-out targets, access each target stream's DashMap independently. Per D-07, if sorting is needed, sort target stream names alphabetically. However, since each stream has its own independent DashMap (not a shared lock), alphabetical ordering is a safety defense, not strictly required — DashMap per-instance shards are independent (RESEARCH.md cross-stream analysis confirms LOW deadlock risk).

**HTTP handler refactors (http.rs):**

15. All 13 HTTP handlers: Replace `state.lock()` with appropriate individual lock access. Most are reads: `state.engine.read()`, `state.stream_stores.get(...)`, `state.metrics.lock()` etc. Write handlers: `engine.write()` for register/delete, `state.snapshot_*` locks for trigger_snapshot.

**PipelineEngine interaction (pipeline.rs):**

16. The engine's `push()`, `get_features()`, `push_with_cascade()` methods currently take `&mut self` + `&mut StateStore`. They need to be refactored to work with the new DashMap-based stores. Two approaches:
   - **Option A (recommended):** Make engine methods take `&self` for read access and accept `&DashMap<String, StreamStore>` + `&DashMap<String, AHashMap<String, StaticFeature>>` instead of `&mut StateStore`. This is a larger refactor but cleaner.
   - **Option B:** Keep engine methods signature, reconstitute a temporary mutable StateStore view for each call. This is wasteful.
   - **Option C (pragmatic):** Move the push/get logic into standalone functions in tcp.rs that work directly with ConcurrentAppState, bypassing the engine's store methods. The engine becomes purely a definition store (stream schemas, DAGs, fan-out targets) and the tcp handlers do the state mutations directly.

   Choose the approach that minimizes code churn while achieving the concurrency goal. Option C is likely fastest since handle_push_core_ex already orchestrates the push pipeline inline.

**Test migration:**

**IMPORTANT:** Keep the `#![deny(clippy::await_holding_lock)]` attribute at the top of tcp.rs. It now also catches parking_lot MutexGuards held across await points (clippy recognizes parking_lot). This is the C-7 compile-time gate.

**Anti-patterns:**
- Do NOT hold a DashMap Ref/RefMut while acquiring another DashMap Ref/RefMut on the same DashMap instance (deadlock risk from internal shard collision). Always drop one before acquiring the next.
- Do NOT hold engine.read() while trying to acquire engine.write() in the same scope.
- Do NOT call `state.stream_stores.iter()` on the hot path — only use `.get()` with known stream names.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> TCP handler | Untrusted events cross into state mutation |
| DashMap shard boundary | Concurrent access to different keys within same stream |
| engine RwLock | REGISTER (write) vs PUSH/GET (read) contention |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | D (Denial of Service) | DashMap RefMut hold time | mitigate | Keep RefMut hold times minimal — drop before event log write or metric update. Verified by C-7 clippy gate for async contexts. |
| T-14-02 | T (Tampering) | Race condition on dirty_keys | accept | Per-stream dirty_keys behind parking_lot::Mutex; snapshot is eventually consistent per existing contract ("lose ~30s on crash") |
| T-14-03 | D (Denial of Service) | REGISTER blocking PUSH | accept | REGISTER is rare (once at startup); parking_lot RwLock write blocks reads briefly. Acceptable for cold-path operation. |
| T-14-04 | T (Tampering) | DashMap internal shard collision blocking concurrent ops | accept | DashMap uses num_cpus shards by default; shard collisions are statistical, not a correctness issue. Keep RefMut hold times short. |
</threat_model>
