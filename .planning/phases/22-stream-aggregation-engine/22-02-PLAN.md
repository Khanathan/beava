# PLAN 22-02: Linear + order-sensitive operators

## Goal

Fill in the 11 operator bodies that 22-01 stubbed: `count`/`sum`/`avg` (reuse existing v2.0 impls, confirm they still slot into the new dispatch), `min`/`max` (bucket-granular; gap-fix existing if needed), `variance`/`stddev` (Welford per-bucket so we can retract on bucket expiry), `first`/`last` (by event-time with a `_event_time` fallback to wall-clock ŌĆö event-time wiring shipped in 24), `first_n`/`last_n` (bounded buffers), `ema` (per-entity exponential average with ╬▒ derived from `half_life`), `lag` (bounded VecDeque of the last N values). This plan closes everything except the three hybrid sketches, which are isolated in 22-03 because they carry the perf-gate risk.

## Success Criteria

1. For each of the 11 operators, pushing a hand-constructed stream and reading via GET produces the mathematically correct value within documented tolerance (1e-9 for doubles; exact for ints).
2. `variance` and `stddev` use Welford's online algorithm stored per-bucket (three RingBuffers: count, mean, m2). Merging buckets on read uses Chan's parallel combination formula so bucket boundaries don't introduce error.
3. `min` and `max` are bucket-granular: the per-bucket min/max is tracked, and the window min/max is the min/max across non-expired buckets. Per-event retract within a live bucket is documented as unsupported (per the retraction literature survey ŌĆö fundamental limitation).
4. `first` / `last` expose `_event_time` if the event carries it, else wall-clock; ties broken by arrival order. The event-time field name is hard-coded to `_event_time` to match Phase 24's contract.
5. `first_n` stores the N earliest events by event-time; `last_n` stores the N most-recent (VecDeque with bounded capacity).
6. `ema` computes ╬▒ = 1 - 2^(-step/half_life); state is a single `(value: f64, last_update: SystemTime)` pair ŌĆö not windowed.
7. `lag(field, n)` returns the value from exactly N events ago ŌĆö state is a `VecDeque<FeatureValue>` of size n+1; rejects at register-time if n exceeds a safety cap (default 10_000; configurable via a const).
8. Registration of `ema` / `lag` against an upstream whose `kind` is `table` in the REGISTER payload returns a named error. (Phase 21 already rejects this at the SDK level; this is defense-in-depth since the engine ultimately owns correctness.)
9. All existing v2.0 operator tests that cover count/sum/avg/min/max continue to pass unchanged ŌĆö proves the 22-01 dispatch wiring didn't break the old paths.
10. Snapshot save/load of an `EntityState` containing one of each new operator variant round-trips cleanly (values equal pre-save after load).
11. Benchmark smoke: `python bench.py --small --ops count,sum,avg` runs within 5% of the v2.0 baseline (recorded delta to PR description). No formal perf gate on this plan ŌĆö that lives in 22-03.

## Implementation Steps

1. **Reuse existing impls.** Read `src/engine/operators.rs` and `src/engine/hll.rs` end-to-end. `CountOp`, `SumOp`, `AvgOp`, `MinOp`/`ExactMinOp`, `MaxOp`/`ExactMaxOp`, `EmaOp`, `LagOp`, `FirstOp`, `LastOp`, `LastNOp`, `PercentileOp`, `StddevOp` are already referenced in `pipeline.rs` imports ŌĆö many of these bodies exist from v2.0. Audit each: if it still produces the correct output under the new dispatch, leave alone. If it relies on removed v2.0 glue, patch it.
2. **Welford per-bucket variance.** For `VarianceOp`:
   - Bucket state: `{count: u64, mean: f64, m2: f64}` ŌĆö Clone + Default so it fits `RingBuffer<T>`.
   - `push`: standard Welford update on the current bucket.
   - `read(now)`: merge all non-expired buckets pairwise via Chan's algorithm (╬┤ = mean_b ŌłÆ mean_a; combined_count = a.count + b.count; combined_mean = a.mean + ╬┤ * b.count / combined_count; combined_m2 = a.m2 + b.m2 + ╬┤┬▓ * a.count * b.count / combined_count). Result variance = combined_m2 / (combined_count - 1) for sample variance. Return `FeatureValue::F64(variance)`.
   - `StddevOp`: wrap `VarianceOp`; `read` = sqrt of variance's read.
3. **min/max (bucket-granular).** If the existing `MinOp` / `MaxOp` already track per-bucket min/max via `RingBuffer<MinBucket>` (there are hints in `window.rs`'s Clone-bound doc comment), reuse verbatim. If not, implement: per-bucket slot is `Option<f64>` (None = no event in bucket); on push, current bucket slot = min(current, new_value); on read, min across all Some() slots. Same for max with the flipped comparator.
4. **first / last by event-time.**
   - State: `{value: FeatureValue, event_time: SystemTime}` (with `event_time` optional ŌĆö None sentinel before any event).
   - `push`: extract `event.get("_event_time")`; if present and parseable as ISO-8601 or unix-millis, use it; else use `now`. For `first`: retain the earliest event_time. For `last`: retain the latest.
   - `read`: return the stored value (or Null if no event).
   - Document the `_event_time` protocol in the doc-comment; Phase 24 formalizes it but we honor the field today so tests don't have to change later.
5. **first_n / last_n.** `first_n`: `Vec<(SystemTime, FeatureValue)>` sorted by event_time, truncated to `n`; on push, binary-search insert only if new event_time < current max in the vec (or len < n). `last_n`: `VecDeque<(SystemTime, FeatureValue)>` with `pop_front` when len == n and new event pushed.
6. **ema.** ╬▒ = 1 ŌłÆ 2^(ŌłÆdt/half_life) where dt = now ŌłÆ last_update (seconds). State: `(value: f64, last_update: SystemTime)`. On push: if state.None, initialize to current value + now. Else: ╬▒ = ..., state.value = ╬▒ * new_value + (1-╬▒) * state.value; state.last_update = now.
7. **lag.** State: `VecDeque<FeatureValue>` with capacity `n+1`. On push, `push_back(new)`, if len > n+1, `pop_front`. On read: if len < n+1, return Null; else return `self.history.front()` (the value from n events ago).
8. **Snapshot serde.** For each new OperatorState variant, add:
   - A `Serializable*OpState` struct (matching the operator's internal state) to `src/state/snapshot.rs`.
   - `From<&OperatorState> for SerializableOperatorState` arm.
   - `TryFrom<SerializableOperatorState> for OperatorState` arm.
   - Round-trip test in `test_snapshot_v0_ops.rs`.
9. **Registration defense for ema/lag.** In `pipeline.rs::register_v0`, when `feat.type` is `"ema"` or `"lag"` and the aggregation source's `kind` is `"table"`, return `TallyError::InvalidRegisterPayload { reason: format!("{} operator requires a Stream source; '{}' is a Table", feat.type, source_name) }`. Phase 21 prevents this at the SDK layer; this is belt-and-suspenders.
10. **Unit tests** (`test_operators_v0.rs`):
    - `test_count_bucket_expiry` ŌĆö 10 events in bucket 0, advance past window, countŌåÆ0.
    - `test_sum_numerical_precision` ŌĆö sum a stream of 1e-9 values; assert no drift beyond 1e-6 over 1M events (Kahan-ish check; document if sum uses naive accumulation).
    - `test_avg_empty_window` ŌĆö no events ŌåÆ Null.
    - `test_variance_welford` ŌĆö known-answer: [1,2,3,4,5] ŌåÆ variance 2.5 ┬▒ 1e-9.
    - `test_variance_bucket_merge` ŌĆö split [1..10] across 3 buckets, assert same variance as unbucketed.
    - `test_stddev_sqrt_relation` ŌĆö stddev┬▓ == variance within 1e-12.
    - `test_min_bucket_granular` ŌĆö min across buckets; expire oldest bucket ŌåÆ window min updates.
    - `test_max_tie` ŌĆö two events with same max value ŌåÆ max returned.
    - `test_first_by_event_time` ŌĆö push out-of-order events with `_event_time`; assert the one with earliest event_time wins.
    - `test_last_wall_clock_fallback` ŌĆö push events without `_event_time`; assert arrival order.
    - `test_first_n_bounded` ŌĆö push 100 events, n=5; assert len(state)==5 and contents are 5 earliest.
    - `test_last_n_deque` ŌĆö same, bounded to 5 most recent.
    - `test_ema_half_life_decay` ŌĆö with half_life=60s, after 60s ╬▒ Ōēł 0.5; push 100ŌåÆreadŌåÆwait sim 60sŌåÆpush 0ŌåÆassert Ōēł 50.
    - `test_lag_returns_n_ago` ŌĆö push 10 events, lag(2) reads event[-3] (0-indexed: event 7 when 10th was just pushed).
    - `test_lag_insufficient_history` ŌĆö fewer than n events pushed ŌåÆ Null.
    - `test_ema_on_table_source_rejected` ŌĆö simulate a register call with upstream kind=table ŌåÆ InvalidRegisterPayload.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Welford bucket-merge rounding accumulates over many buckets | Use f64 everywhere; unit-test 60-bucket merge against the single-pass reference on a random stream. |
| `_event_time` field format (ISO-8601 vs epoch-ms) not yet locked by Phase 24 | Accept both; doc the preference order: (1) `_event_time` as int unix-ms, (2) `_event_time` as string ISO-8601, (3) wall-clock fallback. Phase 24 can tighten to one. |
| `first_n` sort on every event is O(n) ŌĆö worst case n=1000 means a tight push-heavy loop does n comparisons per event | Document the O(n) per-push cost; cap n at 1000 at registration (document limit; users needing more can use a windowed operator). |
| `lag(n)` with n=10000 bloats per-entity memory | Hard cap in `_agg_ops.py::_Lag` ŌĆö add to Phase 21 or defer to here. Rust side: saturate at 10_000 with a runtime warning via `/debug/warnings` (Phase 25 wires). For now: hard error at register-time if n > 10_000. |
| v2.0 operator bodies subtly mismatch the new dispatch's assumptions (e.g., `FeatureValue` encoding, error propagation) | First pass: run the full v2.0 test suite after 22-01 merge. Any red test in operators is the mitigation signal. |
