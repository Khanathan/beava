# Plan 44-01 — Historical extraction at timestamps (one-pass replay)

## Goal

Data scientist runs `tl.fork(..., extract_at=[T1, T2, T3], pipelines=[...])`. One replay streams events from `since` forward; as events cross each extraction timestamp, the replica snapshots state for scope keys. After replay, `fork.extract_history()` returns a dict of `{timestamp: {key: feature_map}}`.

## Task T1 — Server-side extract_at snapshot hook

**Files:** `src/main.rs` (ReplicaBootConfig), `src/server/tcp.rs` (AppState field), `src/server/replica_client.rs` (snapshot loop).

**Subtasks:**
- Add `extract_at_millis: Vec<u64>` to `ReplicaBootConfig`. CLI parses `--replica-extract-at T1,T2,...` (comma-separated), each entry parsed via existing `parse_replica_since` helper. Sort ascending at boot.
- Add `pub extracted_history: DashMap<u64, DashMap<String, serde_json::Value>>` to `ConcurrentAppState`. Initialize empty at construction.
- In `replica_client.rs` historical-catchup loop, maintain `cursor: usize = 0`. Before applying each event E:
  ```rust
  while cursor < config.extract_at_millis.len()
      && event_ts_millis > config.extract_at_millis[cursor]
  {
      self.snapshot_extract(config.extract_at_millis[cursor]);
      cursor += 1;
  }
  ```
- After LOG_FETCH END, capture any remaining `extract_at_millis[cursor..]` from the CURRENT state.
- `snapshot_extract(ts)`:
  - Iterate `scope.keys` if set, else iterate `state.store.entity_keys()`.
  - For each key, compute the per-key feature map using the same logic as `debug_key` HTTP handler (engine-level feature evaluation).
  - Insert into `state.extracted_history[ts].insert(key, feature_map_as_json)`.

**Done when:** unit test in replica_client spawns a fake server pushing events in timestamp order, verifies snapshots captured at expected cursor points.

## Task T2 — /extracts HTTP endpoint

**Files:** `src/server/http.rs`.

**Subtasks:**
- Add `async fn debug_extracts(State(state): State<SharedState>) -> impl IntoResponse`:
  ```json
  {
    "extracts": {
      "2026-03-05T10:00:00Z": {"u1": {"count": 3, "total": 60.0}, "u2": {...}},
      "2026-03-15T10:00:00Z": {...}
    }
  }
  ```
- Iterate `state.extracted_history`, sort by timestamp, format each as ISO-8601, collect per-key feature maps.
- Route `.route("/extracts", get(debug_extracts))` on the public router.
- Also add `.route("/extracts", get(debug_extracts))` on the admin router if different.

**Done when:** curl after replay returns expected JSON.

## Task T3 — CLI + Python surface

**Files:** `src/main.rs` (fork subcommand), `python/tally/_fork.py`.

**Subtasks (CLI):**
- In `fork` subcommand parser, add `--extract-at T1,T2,...` flag. Translate directly to `--replica-extract-at` when invoking the server startup path.
- `--help` text updated.
- Unit tests: parse_fork_args_from accepts the new flag.

**Subtasks (Python):**
- Add `extract_at: list[str | int | datetime] | None = None` kwarg to `tl.fork(...)`.
- Serialize to comma-separated string: datetime → ISO-8601, int → unix millis.
- Pass as `--extract-at ...` argv.
- New method `ForkedReplica.extract_history() -> dict[str, dict[str, dict]]`:
  - GET `http://127.0.0.1:{http_port}/extracts` with admin token.
  - Parse JSON, return `{iso_ts: {key: features}}`.
- Update `_fork.py` unit tests for new kwarg parsing + extract_history happy path (mock HTTP).

**Done when:** both CLI and Python pass the value through the stack; Python unit tests green.

**Done when:** 3× consecutive runs green.

## Stop-and-report triggers

- `debug_key` feature lookup path isn't easily reusable as a snapshot helper — flag; may need a shared `compute_features_for_key(engine, state, key)` function.
- Timestamp parsing ambiguity in `--extract-at` (ISO vs millis)  — match `--replica-since`'s behavior exactly.
- Memory model of nested DashMap<u64, DashMap<String, Value>> causes lifetime headaches with serde — fall back to `Mutex<HashMap<...>>` for `extracted_history` since write-path is cold (one write per extraction timestamp), much simpler.
- Scope-less snapshot iterates all entities: what if state has 100k+ keys? Not a concern for v0 demo (scope declares keys); log if exceeded.
