### WARN-capture mechanism note

Each WARN-asserting test (Test 3 + Test 4) builds a per-test scoped subscriber via:

```rust
let capture: Arc<Mutex<Vec<u8>>> = Arc::new(Mutex::new(Vec::new()));
let _guard = install_warn_capture(capture.clone());
// ... run server + request ...
let logs = String::from_utf8_lossy(&capture.lock().unwrap()).to_string();
```

`install_warn_capture` builds a `tracing_subscriber::fmt()` with:
- `with_writer(TestWriter(capture))` — the `MakeWriter` impl drains every emitted line into the shared `Arc<Mutex<Vec<u8>>>`.
- `with_max_level(tracing::Level::WARN)` — only WARN+ events are captured.
- `with_ansi(false)` — no ANSI escapes contaminating the substring matches.
- `tracing::subscriber::set_default(subscriber)` — installs as **per-thread default** (returns `DefaultGuard`); guard drops at end of test scope.

Per-thread scope is critical: the 4 tests run on a multi-threaded tokio runtime and would race over a single global subscriber. The `set_default` approach (vs the process-global `tracing_subscriber::fmt::init()`) gives each test its own isolated capture buffer.

### Strategy recommendation for Plan 04

**Pick STRICT (alias-only WARN), per PATTERNS.md cross-cutting §4.** Test 4 asserts the EXACT D-04 deprecation message MUST NOT appear in the captured WARN buffer when the canonical `key` field is used. Under PRAGMATIC always-warn this fails (the canonical path would also emit the deprecation message text). Under STRICT it passes — the WARN is only emitted when `entity_id` was the source field of the parse.

Implementation hint: add a sentinel field on `BatchGetReqEntry` like `#[serde(rename = "entity_id", default, skip_serializing)] entity_id_alias_used: ()` is NOT how serde aliases work. Instead use a custom deserialise impl that sets a separate `bool` field on the parsed struct, OR (simpler) deserialise the request into an intermediate `serde_json::Value`, inspect each per-entry object's keys for `"entity_id"` (and absence of `"key"`), then deserialise the canonicalised value into `BatchGetReqEntry`. The per-entry inspection naturally walks the `requests` array exactly once.
