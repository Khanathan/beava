### Q1 — `/batch_get` response shape

**Options:**
1. Flatten (Recommended) — flat feature dicts, matches `examples/wire/batch_get-heterogeneous.response.json` and 13.5.1 Plan 03 test expectations; updates ~3 server-side tests.
2. Keep envelope — preserve `[{table, entity_id, features:{...}}, ...]`; SDKs must unwrap.

**User selection:** Flatten (Recommended)

**Captured as:** D-03 — FLAT row response shape on `/batch_get`.

### Q2 — `entity_id` → `key` rename aggressiveness

**Options:**
1. One-release alias + deprecation warning (Recommended) — `#[serde(alias = "entity_id")]` + WARN log; remove in v0.0.x.
2. Strict — no alias; rename only.
3. Permanent alias — keep both forever (incompatible with v0 hygiene per `project_v0_events_only_scope`).

**User selection:** One-release alias + deprecation warning (Recommended)

**Captured as:** D-04 — `entity_id` → `key` rename with one-release `serde(alias)` deprecation.

### Q3 — Legacy request-shape rejection mode

**Options:**
1. Structured 400 with doc hint (Recommended) — `{"error": {"code": "unsupported_request_shape", "message": "...see docs/http-api.md#post-get"}}`.
2. Bare 400, no hint — minimal but unhelpful.
3. Permissive auto-translate — server sniffs and rewrites; hides migration, adds compat layer.

**User selection:** Structured 400 with doc hint (Recommended)

**Captured as:** D-05 — Structured 400 rejection of legacy request shapes.

### Q4 — Missing-feature filter behavior

**Options:**
1. Omit the missing key from the response (Recommended) — Redis HMGET-style; matches existing axum-side `feature_query.rs:232-234` semantics.
2. Return null for the missing key — explicit but introduces JSON null into typed feature dicts.
3. Error — `feature_not_found` — strict; too aggressive for per-entity sparsity.

**User selection:** Omit the missing key from the response (Recommended)

**Captured as:** D-06 — Missing-feature filter behavior: OMIT from response (with separate "feature not in registry" upfront-error path preserved).
