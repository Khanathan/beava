<domain>
The locked Phase 13.0 wire-spec (`docs/wire-spec.md` + `docs/http-api.md` + `examples/wire/*.json`) requires verb-style request bodies on `POST /get`, `OP_GET`, `POST /batch_get`, and `OP_BATCH_GET`:

- Single-row GET body: `{"table": "...", "key": "...", "features"?: [...]}`
- Batch GET per-entry: same shape, repeated under `{"requests": [...]}`
- Response: FLAT rows — one feature dict per request, no `{table, entity_id, features:{...}}` envelope. When `features` is present, response is narrowed to those keys only.

Phase 13.4 was responsible for migrating the server to this contract but landed with the legacy Phase 12.x shapes still in place:

- `crates/beava-server/src/apply_shard.rs:453` — `WireRequest::HttpGet` routes to `dispatch_get_batch_sync` which deserializes `{keys: Vec<String>, features: Vec<String>}` (the 2D-cell legacy shape from Plan 12-07/12-09).
- `crates/beava-server/src/apply_shard.rs:468-504` — `WireRequest::TcpGet` parses `{feature: String, key: String}` (legacy single-feature shape).
- `crates/beava-server/src/apply_shard.rs:~1180` (`dispatch_batch_get_sync` + `BatchGetReqEntry`) — uses `entity_id` (not `key`) and has no per-entry `features` field; response is wrapped `{table, entity_id, features:{...}}`.
- `crates/beava-server/src/runtime_core_glue.rs:302-410` — `dispatch_get_batch` shares the legacy 2D-cell shape with `BatchGetBody { keys, features }`.

The 13.5.1 SDK transport impls send verb-style bodies. Without this fix-up phase, all 68 v0 acceptance tests would 500 at the server boundary regardless of SDK correctness.

`/batch_get` response wrapping is a separate but related issue: the SDK and the v0 acceptance tests expect flat rows. Either the server flattens (this phase) or the SDK unwraps (13.5.1) — server-flat-rows is the cleaner choice because flat rows align with the `examples/wire/batch_get-heterogeneous.response.json` fixture and avoid SDK churn across Python / TS / Go.
</domain>

<phase_boundary>
**In scope:**
- Migrate `WireRequest::HttpGet` body parser from `{keys, features}` → `{table, key, features?}` (single-row form).
- Migrate `WireRequest::TcpGet` body parser from `{feature, key}` → `{table, key, features?}` (single-row form).
- Migrate `BatchGetReqEntry` to `{table, key, features?: Vec<String>}`. Add `#[serde(alias = "entity_id")]` on `key` for one-release back-compat (callers using `entity_id` continue to work but a deprecation warning is logged).
- Migrate `runtime_core_glue.rs::dispatch_get_batch` (and `dispatch_get_single` / their sync wrappers) to the new request shapes. Update the per-row response constructor to emit FLAT rows (drop the `{table, entity_id, features}` envelope). When the request entry's `features` field is present, narrow the response dict to those keys only.
- Update `examples/wire/*.json` fixtures (if any are inconsistent post-migration; most should already match the locked spec).
- Add structured rejection of legacy shapes — receiving `{keys: [...], features: [...]}` on `POST /get` returns `400 Bad Request` with error code `unsupported_request_shape` plus a one-line hint pointing at `docs/http-api.md#post-get`.

- The legacy `*_get_single` SDK helpers (Phase 13.5.1 D-04 owns the rename).

</phase_boundary>

<draft_decisions>

### DR-01 — Verb-style request shape on `POST /get` + `OP_GET` (DRAFT — confirm in /gsd-discuss-phase)

`POST /get` body: `{"table": "...", "key": "...", "features"?: [...]}`. `OP_GET` frame body identical (CT_JSON or CT_MSGPACK per existing codec selector). Single-row only — multi-key batch is on `/batch_get`.

**Rationale:** Locked in `docs/http-api.md` (Phase 13.0 Plan 13.0-03) + `docs/wire-spec.md` (Plan 13.0-02). The 68 v0 acceptance tests assume this shape via `App.get(table, key, features?=[...])`.

### DR-02 — Verb-style per-entry shape on `POST /batch_get` + `OP_BATCH_GET` (DRAFT)

`POST /batch_get` body: `{"requests": [{"table": "...", "key": "...", "features"?: [...]}, ...]}`. `OP_BATCH_GET` frame body identical. Per-entry `features` filter is independent — different entries can request different feature subsets.

**Rationale:** D-03 in Phase 13.5.1 CONTEXT.md mandates per-entry `features` for `App.batch_get`. Server must do the filtering (SDK only marshals the request).

### DR-03 — FLAT row response shape (DRAFT)

`/batch_get` response: `{"results": [<flat_dict_for_request_1>, <flat_dict_for_request_2>, ...]}`. Each entry is the raw feature-name-to-value dict. No `{table, entity_id, features}` envelope. Matches `examples/wire/batch_get-heterogeneous.response.json` (Phase 13.0). Error entries: `{"error": {"code": "...", "message": "..."}}` in place of the flat dict. Index-aligned with the request `requests` array.

**Rationale:** SDK and 68 v0 acceptance tests expect flat rows. Aligns with `App.batch_get` returning `list[dict[str, Any]]`.

### DR-04 — `serde(alias)` back-compat on `entity_id` → `key` (DRAFT)

`BatchGetReqEntry::key` field deserializer accepts both `key` (canonical) and `entity_id` (legacy alias) for one release. Log a `WARN` log line on `entity_id` use; remove alias in v0.0.x cleanup. No alias on `WireRequest::HttpGet` or `WireRequest::TcpGet` — those are NEW fields, not renames.

**Rationale:** `entity_id` was the Phase 12.x batch_get key name; minimizes break for any existing internal benchmarks or tests that haven't been migrated yet. Matches the 13.5.1 D-04 pattern (private + scheduled removal).

### DR-05 — Structured rejection of legacy shapes (DRAFT)

`POST /get` receiving `{keys: [...], features: [...]}` returns 400 with `{"error": {"code": "unsupported_request_shape", "message": "POST /get expects {table, key, features?}; received legacy {keys, features} shape — see docs/http-api.md#post-get"}}`. Equivalent for `OP_GET` over TCP — error frame with `code=0x0`. Same for `{feature, key}` legacy single-feature shape.

**Rationale:** Visible structured failure beats silent 500. Hint points the consumer at the right doc.

</draft_decisions>
