<domain>
The locked Phase 13.0 wire-spec (`docs/wire-spec.md` + `docs/http-api.md` + `examples/wire/*.json`) requires verb-style request bodies on `POST /get`, `OP_GET`, `POST /batch_get`, and `OP_BATCH_GET`:

- Single-row GET body: `{"table": "...", "key": "...", "features"?: [...]}`
- Batch GET per-entry: same shape, repeated under `{"requests": [...]}`
- Response: FLAT rows — one feature dict per request, no `{table, entity_id, features:{...}}` envelope. When `features` is present, response is narrowed to those keys only.

Phase 13.4 was responsible for migrating the server to this contract but landed with the legacy Phase 12.x shapes still in place:

- `crates/beava-server/src/apply_shard.rs:453` — `WireRequest::HttpGet` routes to `dispatch_get_batch_sync` which deserializes `{keys: Vec<String>, features: Vec<String>}` (the 2D-cell legacy shape from Plan 12-07/12-09).
- `crates/beava-server/src/apply_shard.rs:468-504` — `WireRequest::TcpGet` parses `{feature: String, key: String}` (legacy single-feature shape).
- `crates/beava-server/src/apply_shard.rs:~1180` (`dispatch_batch_get_sync` + `BatchGetReqEntry`) — uses `entity_id` (not `key`) and has no per-entry `features` field; response is wrapped `{table, entity_id, features:{...}}`.
- `crates/beava-server/src/runtime_core_glue.rs:302-410` — `dispatch_get_batch` shares the legacy 2D-cell shape with `BatchGetBody { keys, features }`.

The 13.5.1 SDK transport impls send verb-style bodies. Without this fix-up phase, all 68 v0 acceptance tests would 500 at the server boundary regardless of SDK correctness.

`/batch_get` response wrapping is a separate but related issue: the SDK and the v0 acceptance tests expect flat rows. Server-flat-rows is the cleaner choice (D-03 below) because flat rows align with the `examples/wire/batch_get-heterogeneous.response.json` fixture and avoid SDK churn across Python / TS / Go.
</domain>

<decisions>

### D-01 — Verb-style request shape on `POST /get` + `OP_GET` (USER-CONFIRMED)

`POST /get` body: `{"table": "...", "key": "...", "features"?: [...]}`. `OP_GET` frame body identical (CT_JSON or CT_MSGPACK per existing codec selector). Single-row only — multi-key batch is on `/batch_get`.

**Rationale:** Locked in `docs/http-api.md` (Phase 13.0 Plan 13.0-03) + `docs/wire-spec.md` (Plan 13.0-02). The 68 v0 acceptance tests assume this shape via `App.get(table, key, features?=[...])`.

**Implementation site:** `crates/beava-server/src/apply_shard.rs::WireRequest::HttpGet` + `WireRequest::TcpGet` body parser blocks; `crates/beava-server/src/runtime_core_glue.rs::dispatch_get_single` request shape + downstream call signature.

### D-02 — Verb-style per-entry shape on `POST /batch_get` + `OP_BATCH_GET` (USER-CONFIRMED)

`POST /batch_get` body: `{"requests": [{"table": "...", "key": "...", "features"?: [...]}, ...]}`. `OP_BATCH_GET` frame body identical. Per-entry `features` filter is independent — different entries can request different feature subsets.

**Rationale:** D-03 in Phase 13.5.1 CONTEXT.md mandates per-entry `features` for `App.batch_get`. Server must do the filtering (SDK only marshals the request).

**Implementation site:** `crates/beava-server/src/apply_shard.rs::BatchGetReqEntry` struct definition + the dispatch loop in `apply_shard.rs::dispatch_batch_get_sync` (currently around line ~1200-1330).

### D-03 — FLAT row response shape on `/batch_get` (USER-CONFIRMED)

`/batch_get` response: `{"results": [<flat_dict_for_request_1>, <flat_dict_for_request_2>, ...]}`. Each entry is the raw feature-name-to-value dict — NO `{table, entity_id, features:{...}}` envelope. Matches `examples/wire/batch_get-heterogeneous.response.json` (Phase 13.0). Error entries: `{"error": {"code": "...", "message": "..."}}` in place of the flat dict. Index-aligned with the request `requests` array.

**Rationale:** SDK and 68 v0 acceptance tests expect flat rows. Aligns with `App.batch_get` returning `list[dict[str, Any]]`. SDK-side unwrap was the alternative; server-flat is cleaner because it pushes the schema cohesion to ONE point (server) rather than three (Python + TS + Go SDKs).

**Migration impact:** ~3 internal Rust tests in `crates/beava-server/tests/phase13_4_op_batch_get.rs` assert the old envelope; this phase flattens their expected payloads in the same green commits that change the response constructor.

### D-04 — `entity_id` → `key` rename with one-release `serde(alias)` deprecation (USER-CONFIRMED)

`BatchGetReqEntry::key` field deserializer accepts both `key` (canonical) and `entity_id` (legacy alias) for one release. Log a `WARN`-level log line on `entity_id` use ("BatchGetReqEntry: deprecated 'entity_id' field name; rename to 'key'; alias removed in v0.0.x"). Remove the alias in v0.0.x cleanup. Track in `.planning/ideas/v0.1-deferrals.md`.

No alias on `WireRequest::HttpGet` or `WireRequest::TcpGet` — those are NEW fields, not renames; legacy callers were sending the entirely different `{keys, features}` / `{feature, key}` shapes which D-05 rejects with a structured error.

**Rationale:** `entity_id` was the Phase 12.x batch_get key name. One-release alias minimizes break for any existing internal benchmarks or tests that haven't been migrated yet, while WARN logs surface the migration pressure. Strict-no-alias was the alternative; user picked the one-release path because the cutover risk on internal benchmarks (which we control but might miss in this phase) is non-zero.

**Implementation site:** `BatchGetReqEntry` struct definition. Use `#[serde(alias = "entity_id")]` on `key: String`. Add a runtime hook in `dispatch_batch_get_sync` that detects when `entity_id` was the source field (track via custom deserialize or a sentinel) and emits the WARN log. (Pragmatic alternative: emit the WARN unconditionally on every batch_get request that lands during v0 if exact source-field tracking is too fiddly — refine in planning.)

### D-05 — Structured 400 rejection of legacy request shapes (USER-CONFIRMED)

`POST /get` receiving the legacy `{keys: [...], features: [...]}` shape returns `400 Bad Request` with body:

```json
{
  "error": {
    "code": "unsupported_request_shape",
    "message": "POST /get expects {table, key, features?}; received legacy {keys, features} shape — see docs/http-api.md#post-get"
  }
}
```

Equivalent for `POST /get` receiving `{feature, key}` (legacy single-feature shape) — message points at the verb-style schema with the same doc hint. Equivalent for `OP_GET` over TCP — error frame with `code=0x0` (or whatever the existing TCP error code convention is; refine in planning by reading `apply_shard.rs::GlueResponse::InternalError` / structured error path).

`POST /batch_get` receiving the legacy `{keys, features}` shape (the 2D-cell form) returns the same `unsupported_request_shape` error.

**Rationale:** Visible structured failure beats silent 500. Hint points the consumer at the right doc. Permissive auto-translation (alternative considered) was rejected because it hides the migration and adds a compat layer that has to be maintained for the lifetime of v0.

**Implementation site:** New body-shape detection in `apply_shard.rs::WireRequest::HttpGet` parsing branch — first try the new shape, on parse failure attempt to detect the legacy shape and return the structured error rather than a generic `InternalError { reason: <serde error> }`.

### D-06 — Missing-feature filter behavior: OMIT from response (USER-CONFIRMED)

When a request includes `features=["c", "missing"]` and the entity has only `c`, the response is `{"c": 5}` — `"missing"` is silently omitted from the dict. Mirrors the existing axum-side `feature_query.rs:232-234` semantics ("Omit keys with no matching state"). Mirrors Redis `HMGET`-style absent → silent omission.

This is distinct from "feature name not present in registry at all" — that case should still error upfront (`feature_not_found: missing=[...]`) per the existing `dispatch_get_batch::registry.resolve_feature` upfront-check at `runtime_core_glue.rs:382-394`. The distinction:

- **Feature not in registry** (e.g., typo, `"foo"` doesn't exist) → reject the whole request with `feature_not_found` error
- **Feature in registry but not present on this entity** (e.g., entity hasn't been touched by that derivation) → omit silently from THIS row's response

**Rationale:** Per-entity sparsity is normal in fraud-shape workloads. Returning `null` (alternative) introduces JSON-null into feature dicts that previously had only typed values, complicating SDK type contracts. Erroring on per-entity absence (alternative) is too strict — it would force callers to query each feature individually to handle sparsity.

**Implementation site:** The per-row dict construction in `dispatch_get_batch` (and `dispatch_get_single`) inside the features-filter narrowing pass. After narrowing, drop any keys whose value would be `None` / sentinel-absent.

</decisions>
