# Phase 13.4.1 — Server-side wire-spec verb-style migration — SUMMARY

**Closed:** 2026-05-04
**Verdict:** ✅ **PASS**

**HEAD at closure:** `526c9963` (Plan 04 GREEN cluster head; Plan 05 closure adds the artefacts on top)

## Phase Goal

Migrate server-side `POST /get`, `OP_GET`, `POST /batch_get`, and `OP_BATCH_GET` to the locked verb-style wire contract from Phase 13.0 (`docs/wire-spec.md` + `docs/http-api.md`). Single-row body `{table, key, features?}` on POST /get + OP_GET; per-entry `{table, key, features?}` on batch_get + OP_BATCH_GET. Response shape FLAT ROWS — one flat dict per entry, no `{table, entity_id, features:{...}}` envelope. Apply `features` filter at server (omit-on-absent for per-entity sparsity, whole-batch reject for registry-typo). `serde(alias = "entity_id")` on the renamed `key` field for one-release back-compat with `tracing::warn!` on alias use.

## Deficit Origin

Phase 13.4 (engine prep) was responsible for migrating the server to the verb-style contract but landed with the legacy Phase 12.x shapes still in place:
- `WireRequest::HttpGet` parsed legacy `{keys, features}` (the 2D-cell shape from Plan 12-07/12-09).
- `WireRequest::TcpGet` parsed legacy `{feature, key}` (legacy single-feature shape).
- `BatchGetReqEntry` used `entity_id` (not `key`) and had no per-entry `features` field.
- `dispatch_batch_get_sync` response was wrapped `{table, entity_id, features:{...}}` — not flat.

**Discovery:** During `/gsd-plan-phase 13.5.1` goal-backward audit on 2026-05-04 — the 13.5.1 SDK transports were planned to send verb-style bodies, which would 500 at the server boundary. This phase (Phase 13.4.1) was inserted to fix the server-side gap before Phase 13.5.1 Plan 05 (SDK transport impl) executes.

### `crates/beava-server/src/apply_shard.rs`
- `WireRequest::HttpGet { body }` arm migrated to verb-style three-step ladder: (a) try new `{table, key, features?}` shape strict-deserialize; (b) on parse fail, detect legacy shapes (`{keys, features}` 2D-cell or `{feature, key}` single-feature) and return `UnsupportedRequestShape` with route-specific hint; (c) on truly malformed JSON fall through to `InternalError`.
- `WireRequest::TcpGet { body, body_format }` arm migrated identically with CT_JSON / CT_MSGPACK codec dispatch (D-A/D-B carry-forward from Plan 12-09).
- `BatchGetReqEntry` struct rewritten with custom `Deserialize` impl tracking `from_alias: bool` flag (D-04 STRICT alias detection).
- `dispatch_batch_get_sync` per-row response constructor flattened: `Value::Object(feature_map)` push instead of `json!({"table": ..., "entity_id": ..., "features": Object(...)})` wrap (D-03). Per-entry features filter applied (D-06 narrowing pass). Per-tuple error tuples flattened. Upfront feature-not-in-descriptor validation preserves `feature_not_found` distinction from per-entity sparsity.
- `tracing::warn!` emitted on alias use with verbatim D-04 message text: `"BatchGetReqEntry: deprecated 'entity_id' field name; rename to 'key'; alias removed in v0.0.x"`.
- Legacy arms preserved untouched: `WireRequest::HttpGetSingle`, `WireRequest::TcpMGet`, `WireRequest::TcpGetMulti`, `WireRequest::HttpBatchGet`. v0.0.x cleanup window owns their removal.
