### Evidence Test 4 enforces all three structural assertions for whole-batch reject

Per the iter-2 plan-checker fix called out in the orchestrator prompt, Test 4's predicate locks Plan 04's whole-batch-reject disposition to match the single-row verb-style path. The test's three assertions (verbatim from `phase13_4_1_verb_style_batch_get.rs:480-497`):

```rust
let resp_status = resp.status().as_u16();
let body: serde_json::Value = resp.json().await.expect("json");
let body_text = body.to_string();
assert!(
    resp_status >= 400,
    "expected 4xx for feature-not-in-registry per D-06 (whole-batch reject); got status {resp_status}; body: {body_text}"
);
assert!(
    body_text.contains("feature_not_found"),
    "body should mention feature_not_found per D-06; body: {body_text}"
);
// Whole-batch rejection per D-06; NOT per-tuple error rows in a 200
// response. The structural difference between the two dispositions is
// that whole-batch reject has NO `results` key in the body (the request
// never reached the per-entry dispatch loop).
assert!(
    !body_text.contains("\"results\""),
    "expected whole-batch rejection, not per-tuple error rows in 200 response; body: {body_text}"
);
```

This is the single test that distinguishes D-06's two dispositions structurally:
- **Registry-typo (`does_not_exist`)** → whole-batch reject (Test 4): 4xx + `feature_not_found` + NO `results` array
- **Per-entity sparsity (`hit_count` for alice without Hit events)** → silent per-row omit (Test 3): 200 + `results` array with `{}` for that row

Plan 04 must wire these as separate code paths — registry-typo upfront-resolve before the per-entry dispatch loop (mirrors `runtime_core_glue.rs:382-396`); per-entity absence at the `feature_map.insert` site (omit when `query_feature` returns None).
