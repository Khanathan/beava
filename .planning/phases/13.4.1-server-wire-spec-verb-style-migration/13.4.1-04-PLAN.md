**Decision called out for Plan 04 to make explicit:**

1. **OP_MGET (0x0021) and OP_GET_MULTI (0x0022) disposition** (PATTERNS.md §`runtime_core_glue.rs::dispatch_get_batch` lines 337-348). These opcodes exist in the codebase (`crates/beava-runtime-core/src/wire_request.rs:62, 67` + `apply_shard.rs:513-577`) but are NOT in the locked v0 wire-spec (`docs/wire-spec.md` lines 23-39 list ONLY `OP_GET=0x0020`, `OP_BATCH_GET=0x0024`, `OP_GET_RESPONSE=0x0023`, no MGET/GET_MULTI). Per Phase 13.4.1 scope (`<phase_boundary>` "Out of scope: New aggregation operators / new wire endpoints"), this plan does NOT delete `OP_MGET` / `OP_GET_MULTI` — that's a separate v0.0.x cleanup. **What this plan DOES do:** leaves the existing `WireRequest::TcpMGet` / `WireRequest::TcpGetMulti` arms in `apply_shard.rs:513-577` UNTOUCHED — they continue to call `dispatch_get_batch` (the legacy 2D-cell shape). The `dispatch_get_batch` function in `runtime_core_glue.rs` is preserved AS-IS for those callers. The verb-style `WireRequest::HttpGet` and `WireRequest::TcpGet` are the ONLY arms migrated to the new single-row dispatch. The existing `WireRequest::HttpGetSingle { feature, key }` (the legacy `GET /get/:feature/:key` path-encoded route) also stays untouched and continues to call the existing `dispatch_get_single_sync(feature, key, body_format)`.

2. **`dispatch_get_single` signature evolution.** The current signature is `(feature, key, body_format)`. Adding a third parameter (`features: Option<&[String]>`) AND switching the lookup semantics from "feature-name OR aggregation-node-name" to "table-name only" risks breaking the `HttpGetSingle` legacy path. **Strategy:** introduce a NEW function `dispatch_get_single_verb_style(app, table, key, features, body_format)` in `runtime_core_glue.rs` that does ONLY the table-lookup + features-filter case. Leave the existing `dispatch_get_single(app, feature, key, body_format)` UNTOUCHED (legacy `HttpGetSingle` keeps calling it). The new verb-style entry points (`WireRequest::HttpGet` + `WireRequest::TcpGet`) call the NEW function. This avoids the back-compat-break risk and keeps the diff localised.

3. **D-04 alias-detection strategy (STRICT vs PRAGMATIC vs SAMPLED).** Plan 03 recommends STRICT. Implementation: a custom `Deserialize` impl on `BatchGetReqEntry` that tracks which source field name was present (`key` vs `entity_id`) and stores a `from_alias: bool` flag. After deserialise, in the `dispatch_batch_get_sync` loop, before the per-row dispatch, check `entry.from_alias` and emit `tracing::warn!` once per alias-using entry. (Or: use a `serde::Deserialize` impl that captures the source field at parse time via a `MapAccess` walk — see https://serde.rs/deserialize-map.html for the standard recipe.) Allowed alternative: a thin custom-deserialize visitor that records a sidecar `bool` per entry. The pragmatic always-WARN strategy is rejected because Plan 03 Test 4 explicitly asserts that the canonical path does NOT emit the deprecation log.

4. **D-05 detection ladder.** Three-step ladder per PATTERNS.md cross-cutting §1: (a) attempt strict-deserialize of the new shape; (b) on `serde_json::Error` — attempt to deserialize as `serde_json::Value` and detect: `value["keys"].is_array() && value["features"].is_array()` (legacy 2D-cell), or `value["feature"].is_string() && value["key"].is_string()` (legacy single-feature). On detection match → return `GlueResponse::UnsupportedRequestShape { hint: <route-specific message> }`; (c) on no-match → fall through to existing `GlueResponse::InternalError { reason: <serde error> }` so genuinely malformed JSON still surfaces clearly.

These decisions are LOCKED — implement them as described above without re-deciding.

                    let mut key: Option<String> = None;
                    let mut features: Option<Vec<String>> = None;
                    let mut from_alias = false;

                                }

                                if key.is_some() {
                                    return Err(de::Error::custom(
                                        "BatchGetReqEntry: cannot specify both `key` and `entity_id` (entity_id is a deprecated alias)"
                                    ));
                                }
                                key = Some(map.next_value()?);
                                from_alias = true;
                            }
                            "features" => {
                                if features.is_some() {
                                    return Err(de::Error::duplicate_field("features"));
                                }
                                features = Some(map.next_value()?);
                            }
                            other => {
                                let _ignored: serde::de::IgnoredAny = map.next_value()?;
                                let _ = other; // tolerate forward-compat fields
                            }
                        }
                    }
                    Ok(BatchGetReqEntry {
                        table: table.ok_or_else(|| de::Error::missing_field("table"))?,
                        key: key.ok_or_else(|| de::Error::missing_field("key"))?,
                        features,
                        from_alias,
                    })
                }
            }

            deserializer.deserialize_map(EntryVisitor)
        }
