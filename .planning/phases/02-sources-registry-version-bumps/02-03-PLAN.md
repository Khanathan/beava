- If `changed` is non-empty → respond 409 with the diff body shape from CONTEXT.md
- Else if `added` is non-empty → install, bump version, respond 200
- Else (only already_present) → respond 200 with no version bump

```rust
// registry_diff.rs

use crate::registry::{
    DerivationDescriptor, EventDescriptor, RegistryInner, TableDescriptor,
};
use serde::{Deserialize, Serialize};

/// A parsed node from the `POST /register` payload, post-`kind` discrimination but
/// pre-`registered_at_version` assignment. The HTTP handler (Plan 05) produces these.
/// The validation pass (Plan 04) guarantees each descriptor is structurally valid
/// before compute_diff runs.
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(tag = "kind", rename_all = "snake_case")]
pub enum PayloadNode {
    Event(EventDescriptor),
    Table(TableDescriptor),
    Derivation(DerivationDescriptor),
}

impl PayloadNode {
    pub fn name(&self) -> &str {
        match self {
            Self::Event(e) => &e.name,
            Self::Table(t) => &t.name,
            Self::Derivation(d) => &d.name,
        }
    }
    pub fn kind_str(&self) -> &'static str {
        match self {
            Self::Event(_) => "event",
            Self::Table(_) => "table",
            Self::Derivation(_) => "derivation",
        }
    }
}

#[derive(Debug, Clone, PartialEq, Serialize)]
pub struct RegistryDiff {
    pub added:           Vec<String>,
    pub already_present: Vec<String>,
    pub changed:         Vec<ConflictDetail>,
}

#[derive(Debug, Clone, PartialEq, Serialize)]
pub struct ConflictDetail {
    pub name:    String,
    pub reason:  DiffReason,
    pub details: String,           // human-readable explanation, safe to include in 409 body
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum DiffReason {
    KindMismatch,       // payload says event; current has a table/derivation with that name
    SchemaMismatch,     // field type changed or field added/removed
    EventTimeFieldMismatch,
    PrimaryKeyMismatch,
    TtlMismatch,        // history_ttl_ms / ttl_ms / idempotency_ttl_ms / watermark_lateness_ms differ
    IdempotencyKeyMismatch,
    OpsMismatch,        // derivation ops differ
    UpstreamsMismatch,
    OutputKindMismatch,
    TableModeMismatch,
    TablePrimaryKeyMismatch,
    OtherFieldMismatch, // catch-all if something slips past specific checks
}

/// Pure function: given current registry state and a validated payload, classify each node.
/// Input order of `payload` is preserved in all output vectors.
pub fn compute_diff(current: &RegistryInner, payload: &[PayloadNode]) -> RegistryDiff;
```

Response JSON shape (CONTEXT.md §Registration response on 409):
```json
{
  "error": {
    "code": "registration_conflict",
    "diff": {
      "added": ["NewEvent"],
      "removed": [],
      "changed": [
        {"name": "Transaction", "reason": "schema_mismatch", "details": "field 'amount' type changed from f64 to i64"}
      ]
    }
  },
  "registry_version": 3
}
```

Note: `removed` is always `[]` in v0 — the server never removes (CONTEXT.md §Additive-only diff engine). The endpoint in Plan 05 surfaces `removed: []` as a literal empty list in the response body, NOT by querying the diff engine. So the RegistryDiff struct does NOT need a `removed` field. Plan 05 adds the literal `"removed": []` at response-serialization time.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Tampering | ConflictDetail.details string | mitigate | Details are server-constructed from descriptor data only — no client-supplied strings are reflected verbatim (avoids log-injection). Test 4 asserts the details are deterministic. |
| T-02-03-02 | DoS | Unbounded payload size | accept | Phase 2 HTTP body limit (Plan 05) caps payload at 1MiB; diff engine's complexity is O(n * m) where n=payload, m=current. For n,m ≤ 10k, compute_diff is <10ms in practice. |
| T-02-03-03 | Information disclosure | Diff reason details | accept | Details include descriptor names + field types the client submitted; returning them to the client is intentional (SDK asserts on them). No secrets exposed. |
| T-02-03-04 | Spoofing | Equivalence via equiv_ignoring_version | mitigate | Helper ignores ONLY registered_at_version; Prop 4 + Test 18 confirm this. If a future refactor mistakenly ignores other fields, proptest catches it. |
</threat_model>

## For Plan 02-04 (validation)
- Use `PayloadNode` as the input type to your validator: `validate_payload(current: &RegistryInner, payload: Vec<PayloadNode>) -> Result<ValidatedPayload, Vec<ValidationError>>`.
- ValidatedPayload can be a simple `Vec<PayloadNode>` (since validation is structural + semantic, not a type change) — or a newtype wrapper if you want the type system to enforce "compute_diff only accepts post-validation nodes". Newtype recommended.

## For Plan 02-05 (endpoint)
- Happy path: parse JSON → PayloadNode vec → validate → compute_diff → if only added+already_present, install and bump version; else 409 with diff.
- Response JSON: build from RegistryDiff. `removed` field is the literal `"removed": []` (not from the diff engine). Map `DiffReason` to the snake_case serde-derived string in the response body.
- On 409: the `registry_version` in the response body is the CURRENT version (pre-submission), since no bump happened.
- On 200 with all already_present: NO version bump. The response still includes `registry_version` (current value).

## For Phase 3 (Python SDK tests)
- SDK tests that assert on diff-engine output should use the snake_case reason strings: `schema_mismatch`, `event_time_field_mismatch`, etc.
- SDK can call `app.validate(...)` locally (zero network) — that's Phase 3's job to implement, reusing the validation rules but not the diff engine. The diff engine is server-only.

## Stability notes
- `DiffReason` variants can be ADDED without breaking SDK consumers (unknown variant → SDK treats as OtherFieldMismatch-like). Removing variants IS breaking.
- Details string format is human-readable, NOT a stable wire contract. Tests SHOULD assert `details.contains("amount")` not `details == "field 'amount' type changed from f64 to i64"`.
</handoff>
