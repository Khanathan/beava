// In crates/beava-server/src/testing.rs

impl TestServer {
    /// POST arbitrary JSON body to `path` on the test server. Uses the shared reqwest
    /// client with 5s timeout. Caller asserts on status + body via `.status()` and
    /// `.json::<Value>().await`.
    pub async fn post_json<B: Serialize + ?Sized>(
        &self,
        path: &str,
        body: &B,
    ) -> Result<reqwest::Response, reqwest::Error> {
        let url = format!("{}{}", self.base_url, path);
        reqwest::Client::new()
            .post(&url)
            .header("Content-Type", "application/json")
            .json(body)
            .timeout(Duration::from_secs(5))
            .send()
            .await
    }

    /// GET `path`; parse response body as JSON. Panics if non-2xx OR non-JSON body —
    /// test harness, not production code. Use `.get_raw(...)` for error-path tests.
    pub async fn get_json(&self, path: &str) -> serde_json::Value {
        let url = format!("{}{}", self.base_url, path);
        let resp = reqwest::Client::new()
            .get(&url)
            .timeout(Duration::from_secs(5))
            .send()
            .await
            .expect("GET request");
        assert!(resp.status().is_success(), "GET {} returned {}", path, resp.status());
        resp.json().await.expect("JSON body")
    }

    /// Raw GET that doesn't assert on status. For 404 / 503 / error-path tests.
    pub async fn get_raw(&self, path: &str) -> reqwest::Response { /* ... */ }
}

### GET /registry endpoint

Gating model: at `Server::bind` time, read `std::env::var("BEAVA_DEV_ENDPOINTS")` ONCE. If `== "1"`, pass a `DevEndpointsConfig { enabled: true }` to the router, which mounts the route. Otherwise, don't mount the route (returns 404 naturally).

Rationale for bind-time read (vs. per-request): avoids a per-request syscall; matches CONTEXT.md §GET /registry note ("gate behind BEAVA_DEV_ENDPOINTS=1 env var"); allows tests that want the endpoint to set the env var before spawning.

```rust
// crates/beava-server/src/registry_debug.rs

use axum::{extract::State, routing::get, Json, Router};
use beava_core::registry::Registry;
use serde::Serialize;
use std::sync::Arc;

#[derive(Clone)]
pub struct RegistryDebugState {
    pub registry: Arc<Registry>,
}

#[derive(Debug, Serialize)]
struct RegistryDump {
    version: u64,
    events: std::collections::BTreeMap<String, beava_core::registry::EventDescriptor>,
    tables: std::collections::BTreeMap<String, beava_core::registry::TableDescriptor>,
    derivations: std::collections::BTreeMap<String, beava_core::registry::DerivationDescriptor>,
    _dev_only: bool,     // always true — sentinel flag
}

pub fn registry_debug_router(state: RegistryDebugState) -> Router {
    Router::new()
        .route("/registry", get(get_registry))
        .with_state(state)
}

async fn get_registry(State(state): State<RegistryDebugState>) -> Json<serde_json::Value> {
    let inner = state.registry.snapshot();
    let dump = RegistryDump {
        version: inner.version,
        events: inner.events,
        tables: inner.tables,
        derivations: inner.derivations,
        _dev_only: true,
    };
    Json(serde_json::to_value(dump).unwrap())
}
```

## Trust Boundaries

| Boundary | Description |

| `BEAVA_DEV_ENDPOINTS` env var | Operator-supplied config flag — trusted as operator intent, but the endpoint reveals full registry contents including descriptor fields |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06-01 | Information disclosure | GET /registry in prod | mitigate | Route only mounts when `BEAVA_DEV_ENDPOINTS=1`. Default posture: 404. `_dev_only: true` sentinel warns SDK authors. Phase 13 docs will call this out as "never set in production". |
| T-02-06-02 | Tampering | Env-var race in tests | mitigate | DEV_ENV_MUTEX serializes env-mutating TestServer builders (reuses the `ENV_MUTEX` pattern established in Phase 1 config.rs tests). |
| T-02-06-03 | DoS | GET /registry cost on large registries | accept | v0 cap: no registry size limit. Registrations are low-volume admin-path; a 10k-descriptor registry dumps ~500KB — fast. If it becomes a hot-spot, add pagination (deferred). |
| T-02-06-04 | Elevation of privilege | Bind-time env read permanently enables endpoint | accept | Endpoint enablement is a per-process startup decision. Operators wanting to disable must restart the server. Acceptable for a dev-only route. |

## For Phase 6 (WAL)

- `apply_registration` in `beava-core` is the atomic install point. Phase 6 wraps it with:
  1. Construct WAL record (registration payload + pre-bump version)
  2. fsync (group-commit)
  3. Call apply_registration (version bumped, in-memory installed)
  4. Respond
- Crash between (2) and (3) → WAL replay at recovery re-applies the registration. Crash between (1) and (2) → WAL entry is not durable; client sees connection error and retries. Phase 2's in-memory-only semantics become durable.
