```rust
// register_validate.rs

use crate::registry::RegistryInner;
use crate::registry_diff::PayloadNode;
use serde::Serialize;

/// The nine categories from CONTEXT.md §Validation pass. Response shape per
/// SRV-API-12 is `{error: {code, path, reason}}`; `code` below is the stable
/// machine-readable identifier.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize)]
#[serde(rename_all = "snake_case")]
pub enum ErrorCode {
    InvalidRegistration,       // fallback bucket
    NameDuplicate,             // rule 1
    NameReservedPrefix,        // rule 2
    NameBadPattern,
    NameEmpty,
    NameTooLong,
    EventTimeFieldMissing,     // rule 3
    EventTimeFieldWrongType,
    EventSchemaEmpty,
    TablePrimaryKeyEmpty,      // rule 4
    TablePrimaryKeyTooLong,
    TablePrimaryKeyUnknownField,
    DerivationUpstreamUnknown, // rule 5
    DerivationSchemaEmpty,     // rule 6
    RegistrationCycle,         // rule 7
    TopologicalOrderViolation, // rule 8
    IdempotencyKeyUnknownField,// rule 9
    IdempotencyTtlNonPositive,
    DerivationOutputKindTableMissingPrimaryKey, // derivation output_kind=Table requires table_primary_key
}

#[derive(Debug, Clone, PartialEq, Serialize)]
pub struct ValidationError {
    pub code:   ErrorCode,
    pub path:   String,        // e.g., "nodes[2].upstreams[0]"
    pub reason: String,        // human-readable; safe to reflect in 400 body
}

/// Newtype: only returned by `validate_payload(Ok)`. compute_diff's type signature
/// (in Plan 03) currently accepts `&[PayloadNode]`; Plan 05 extracts inner via
/// `ValidatedPayload::into_inner()` for the diff call. This keeps the type guard
/// at the boundary between validate → compute_diff in the endpoint handler.
#[derive(Debug)]
pub struct ValidatedPayload(pub(crate) Vec<PayloadNode>);

impl ValidatedPayload {
    pub fn as_slice(&self) -> &[PayloadNode] { &self.0 }
    pub fn into_inner(self) -> Vec<PayloadNode> { self.0 }
}

pub fn validate_payload(
    current: &RegistryInner,
    payload: Vec<PayloadNode>,
) -> Result<ValidatedPayload, Vec<ValidationError>>;
```

### Path format (used in every test)

- Top-level node: `nodes[i]`
- Node field: `nodes[i].name`, `nodes[i].event_time_field`, `nodes[i].primary_key[0]`
- Schema field: `nodes[i].schema.fields.<field_name>`
- Derivation ops: `nodes[i].ops[j]` (Phase 4 refines further)
- Derivation upstream: `nodes[i].upstreams[0]`

### Validation order + fail-soft semantics

For each node, run checks in this order and COLLECT errors (do not short-circuit on first). Final Err vec is the concatenation of per-node errors in node-index order, plus cross-node errors (uniqueness, cycles, topological order) appended after.

Exception: if `nodes` is empty, return Ok(ValidatedPayload(empty)) — an empty payload is valid and results in a no-op at the endpoint (CONTEXT.md §Empty nodes array).

<handoff>
## For Plan 02-05 (endpoint)

On 400 response, the endpoint should return THE FIRST ValidationError in the Err vec as the primary error (matching the CONTEXT.md §Registration response on 400 shape with a single `{error: {code, path, reason}}`). Rationale: HTTP 400 traditionally returns one error. If we want to return all errors, we can extend to `{errors: [...]}` post-v0, but v0 matches CONTEXT.md verbatim.

Alternative: return the full Vec as `{error: {...first...}, additional_errors: [...]}` — skip this in v0, keep the shape simple.

**Decision (locked):** Plan 05 returns the first error in `{error: {code, path, reason}}` shape, logs the full Vec via tracing at WARN level, and responds 400. SDK gets one error per round-trip; user re-submits after fixing; subsequent errors surface on next attempt.

## For Phase 3 (Python SDK `app.validate`)

SDK-APP-03 requires `app.validate(*descriptors)` to run zero-network validation and return `list[ValidationError]`. Phase 3 can replicate the same rules in Python OR (better) invoke a dry-run HTTP endpoint. For v0 Phase 3, replicate in Python — the rules are small enough and Python needs local validation for test-without-server workflows.

## Stability notes

- `ErrorCode` variants are STABLE for v0. Adding variants is non-breaking (SDK treats unknown as `InvalidRegistration`); removing/renaming is breaking.
- `path` format (`nodes[i].field.subfield`) is the public API — lock it. SDK regex-parses these for pretty-printing.
- `reason` strings are NOT stable — they can be improved without bumping SDK versions. Tests SHOULD assert via `.contains()`, not `==`.
</handoff>
