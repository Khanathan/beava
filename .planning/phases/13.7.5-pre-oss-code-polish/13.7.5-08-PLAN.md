    2. Classify each comment per D-01.
    3. **Examples-specific KEEP-bias rules (different from internal-code plans):**
       - **Top-of-file module docstring or banner-comment** explaining what the demo shows + how to run it: KEEP. ("This is the fraud-detection demo. It registers Click + Login events, computes count/rate/distance features per user, then pushes a synthetic stream and shows feature evolution.") This is the user's first read.
       - **Inline comments next to a `bv.<op>(...)` call** that explain WHY the parameter values were chosen ("# half_life=300 = 5min — short enough to react to bursts, long enough to ignore single-event noise"): KEEP.
       - **Comments labeling a workflow step** ("# Step 1: register the pipeline" / "# Step 2: push synthetic events") in a tutorial flow: this is borderline. CLAUDE.md heuristic says default to no comments, but in tutorial code the cognitive cost of reading + the value of stepwise framing is higher. **Planner's choice: KEEP if the comment names a phase of the workflow that an OSS user would search for in docs ("# Push events", "# Query features"); DELETE if it just paraphrases the next line ("# Now we push the event").**
    4. **Examples-specific DELETE rules:**
       - **Phase / plan references** ("# updated for Phase 13.5"): DELETE.
       - **AI-tell phrases**: DELETE.
       - **Restated signature in inline comment**: DELETE.
       - **Closing-bracket echoes** (`}  # end of dict literal` in Python): DELETE.

    5. Remove orphaned blank lines.

    6. **No logic / API changes.** Same workload distribution. Same op choices. Same flag values.

    7. Verify the demo still runs:

       cd examples/python
