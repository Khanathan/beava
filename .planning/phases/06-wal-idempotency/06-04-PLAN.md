Criterion benches (all use `tokio::runtime::Builder::new_current_thread().enable_all().build()`):

1. `wal/append_nofsync` — measures serialization + CRC + write (no sync_data). Construct a WalWriter directly (not WalSink) in a tempdir; `bench.iter(|| writer.append(&record))`. Payload = 256-byte JSON.

2. `wal/append_fsync_default_coalesce` — measures single-writer append+await through WalSink with default config (fsync_interval_ms=2, fsync_bytes=1MB, segment_bytes=128MB). `bench.iter(|| rt.block_on(sink.append_event(payload.clone())))`. This is the headline P50 number for success criterion #3.

3. `wal/append_fsync_burst_1k` — measures amortized fsync per push under load. Spawn 1000 concurrent `append_event` futures and await them all: `bench.iter(|| rt.block_on(async { let futs: Vec<_> = (0..1000).map(|_| sink.append_event(payload.clone())).collect(); for f in futs { f.await.unwrap(); } }))`. Divide criterion's reported time by 1000 to get per-push cost; note this in the SUMMARY.

phase6_crash_probe binary (crates/beava-server/src/bin/phase6_crash_probe.rs):
- Reads env: BEAVA_WAL_DIR (required), BEAVA_CRASH_MODE ("before_fsync" | "after_ack"), BEAVA_WAL_FSYNC_INTERVAL_MS.
- Starts a minimal axum server bound to an ephemeral port; prints the port to stdout; listens for POST /push/Test.
- BEAVA_CRASH_MODE="before_fsync": sets fsync_interval_ms=999_999_999 (never coalesces); accepts ONE push, returns 202 (accepted, not-yet-durable) instead of 200, then the parent kills it. Actually simpler: the probe doesn't return at all because the handler await never resolves (fsync never fires). The parent waits for stdout PORT, sends one push (with short client-side timeout), SIGKILLs when the push request times out.
- BEAVA_CRASH_MODE="after_ack": uses default fsync_interval_ms=2; accepts push, returns 200 normally; waits 100ms; `std::process::exit(0)` OR parent SIGKILLs after ACK.

Actually, simpler: the probe just runs the server. The TEST controls the CRASH MODE via env var. The probe:
- Reads BEAVA_WAL_DIR, BEAVA_WAL_FSYNC_INTERVAL_MS from env.
- Reuses `beava_server::server::Server::bind` (pointing at a fresh registry with a single registered Test event so pushes work).
- Writes "PORT=<n>\n" to stdout on ready.
- Blocks indefinitely until signal (using the existing shutdown.rs signal handling).

Test flow:
- Test `wal_kill_before_fsync_drops_event`:
  * tempdir T. Spawn probe with BEAVA_WAL_DIR=T, BEAVA_WAL_FSYNC_INTERVAL_MS=999999999.
  * Wait for "PORT=N" on child stdout.
  * Start a reqwest client; call POST http://127.0.0.1:N/push/Test with a 200ms timeout.
  * Timeout fires (handler is awaiting fsync that never fires).
  * SIGKILL the child: `libc::kill(pid, SIGKILL)`.
  * Wait for process exit (wait_timeout 1s).
  * Open T/ with WalReader. Count Event records = 0.

- Test `wal_kill_after_ack_preserves_event`:
  * tempdir T. Spawn probe with BEAVA_WAL_DIR=T, BEAVA_WAL_FSYNC_INTERVAL_MS=1.
  * Wait for "PORT=N".
  * POST /push/Test with {...}. Assert 200 returned.
  * SIGKILL the child.
  * Wait for exit.
  * Open T/ with WalReader. Count Event records ≥ 1.
