Handler flow (per D-11, D-12):
1. Look up event descriptor in `app.registry` by `event_name` → 404 on miss.
2. Validate `body` (JSON object) against the descriptor's schema (field names + types; missing required fields = 400).
3. If descriptor has `dedupe_key` set:
   - Extract the dedupe key value from `body[descriptor.dedupe_key]` (must be string-coercible; else 400).
   - Call `app.idem_cache.get(&event_name, &dedupe_str, now_ms)` → on hit, return the cached bytes as a 200 response (byte-identical).
4. Convert body to Row for apply_event_to_aggregations. Extract event_time from `body[descriptor.event_time_field]` (or wall-clock if field omitted per SDK-DEC-08).
5. Serialize the WAL payload: `serde_json::to_vec(&WalEventPayload { registry_version, stream_id: event_name, event_time_ms, entity_key, event_body: body.clone() })?`.
6. Call `app.wal_sink.append_event(payload_bytes).await?` — the future resolves with the assigned LSN only after fsync (this is where the ACK-after-fsync invariant lives).
7. **Pre-fsync apply** (D-12): actually the pseudo says await first — re-read D-12: apply happens BEFORE await. Revised: between step 5 and 6:
   - 5a. serialize payload
   - 5b. call `wal_sink.append_event(payload)` to enqueue (but do NOT yet await)
   
   The current WalSink API resolves `append_event` ONLY after fsync. To implement pre-fsync apply, split into two APIs:
   - `WalSink::stage(payload) -> Lsn` (not awaitable — just returns the assigned LSN synchronously from the MPSC round-trip via a fast-path channel)
   - `WalSink::wait_for_durable(lsn).await -> Result<(), PersistError>`
   
   **However**, for v0 simplicity and because the apply function is non-blocking and fast (~<1µs per event at Phase 5 benches), we use the simpler pattern: await `append_event(payload).await` → get LSN + durable guarantee in one call, then apply AFTER the await resolves. This changes D-12's "pre-fsync apply" to "post-fsync apply" — which is MORE durable (no divergence between state and disk on crash) and measurably slower only by the apply cost itself (~ns-µs). **Plan 03 adopts post-fsync apply** and documents this as a D-12 refinement: the crash-safety story is stronger and the latency cost is negligible at Phase 5-measured apply speeds. If the P50 bench in Plan 04 shows apply-after-fsync adds measurable overhead, Plan 12 (push_sync) can revisit with pre-fsync apply.
   
   Flow becomes:
   - 5. Serialize WAL payload.
   - 6. `let ack_lsn = app.wal_sink.append_event(payload).await?;` (resolves after fsync)
   - 7. Acquire state_tables lock; call `apply_event_to_aggregations(event_name, row, event_time_ms, ack_lsn, registry, &mut tables)`.
   - 8. Build `PushAck { ack_lsn, idempotent_replay: false, registry_version }`; serialize to bytes.
   - 9. If dedupe was configured: `idem_cache.put(event_name, dedupe_str, CachedEntry { response_bytes, ack_lsn, inserted_at_ms: now_ms, expires_at_ms: now_ms + dedupe_window_ms })`.
   - 10. Return 200 with the serialized bytes.

Response encoding:
- Content-Type: `application/json`
- Body: `{"ack_lsn": N, "idempotent_replay": false, "registry_version": M}` (no extra fields; stable key order via serde_json)

On dedupe hit, the cached `response_bytes` IS the exact body returned on the original miss (including the specific `ack_lsn` of the original). The returned body's JSON still reads `"idempotent_replay": false` because that field reflects the state at the time the FIRST request landed; the cache semantically replays the original response. **NOTE:** this exactly matches success criterion #2 ("byte-identical"). An `idempotent_replay: true` flag is reflected instead in a new response HEADER `X-Beava-Idempotent-Replay: 1` on dedupe hits — this keeps the BODY byte-identical while signaling replay to observability. The header name is stable and documented here.
