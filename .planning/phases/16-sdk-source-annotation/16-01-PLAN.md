<threat_model>
Phase 16 adds no new trust boundaries. Existing boundaries are unchanged:

| Boundary | Description |
|----------|-------------|
| Python SDK user code → Python SDK library | Same-process; user controls decorator inputs. Phase 16 adds stricter validation, reducing surface for silent mis-registration. |
| Python SDK → beava server (HTTP/TCP) | Untrusted input crosses here. Phase 16 does NOT change wire format; server-side validators added in Plan 02 handle out-of-band (curl) push-to-derivation attempts. |

## STRIDE threat register

| ID | Category | Component | Disposition | Mitigation |
|----|----------|-----------|-------------|------------|
| T-16-01 | Tampering | `_beava_is_source` flag on descriptor | accept | In-process Python flag; user can monkey-patch. Server-side `cannot_push_to_derivation` (Plan 02) is the authoritative gate. SDK check is defense-in-depth + fail-fast UX, not a security boundary. |
| T-16-02 | Information disclosure | Error messages naming descriptor / field names | accept | Descriptor names are user-chosen identifiers, not secrets. Matches existing validate_descriptors message pattern. |
| T-16-03 | Denial of service | DAG root-source walker with deeply nested derivations | mitigate | Walker uses memoized set; max depth bounded by existing cycle-detection Rule 3 (cycles caught before walker runs). Add a defensive visited-set limit of 10_000 in the walker. |
</threat_model>
