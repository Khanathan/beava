<objective>
Ship `app.upsert(target, row)` and `app.delete(target, *, key={...})` as
the first-class Python-SDK verbs for writing to source tables, AND
rename the server HTTP paths to match. Enforce D-05 at two layers:
(1) client-side `BV-E-PUSH-TO-DERIVATION` guard before network I/O;
(2) server-side 400 `cannot_push_to_derivation` for out-of-band
callers. Execute the **hard-break HTTP rename (D-09)** —
`/push-table/{name}` → `/upsert/{name}` and
`/delete-table/{name}` → `/delete/{name}`; old paths return 404.

Rationale for folding D-09 into this plan: the verbs (`upsert` /
`delete`) and the wire paths (`/upsert` / `/delete`) must ship together
to avoid an intermediate state where SDK and server disagree. One plan,
one commit sequence, one coherent surface.

Purpose: Give v0 users the named, symmetric verbs they reason about
(`upsert` / `delete`) at every layer (SDK, HTTP, TCP wire) — not just
the Python API.

Output: 2 new App methods; 2 new Transport methods; server routes
renamed; 2 new server integration smokes (derivation reject + hard-break
404); 5 REQ-IDs closed.

**Scope note:** Per CONTEXT GA-2, we do NOT add `app.push_table` /
`app.delete_table` deprecation aliases — they never shipped.
</objective>
