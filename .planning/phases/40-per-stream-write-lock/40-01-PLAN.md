# Plan 40-01 — Per-stream event-log write lock

## Goal

Move the lock around event-log writes from a single global mutex to a per-stream `Mutex<BufWriter>` inside a `DashMap`. Two streams pushed concurrently no longer block each other. Demonstrably scales with `TALLY_WORKER_THREADS` × number of streams.

## Task T1 — EventLog interior-mutability refactor

**Files:** `src/state/event_log.rs` (extensive), unit tests in same file.

**Subtasks:**
- Change struct:
  ```rust
  pub struct EventLog {
      data_dir: PathBuf,
      writers: DashMap<String, PLMutex<BufWriter<File>>>,
      // ... any other fields, unchanged
  }
  ```
- `append(&self, stream_name: &str, ...) -> Result<...>`:
  - `let entry = self.writers.entry(stream_name.to_string()).or_try_insert_with(|| open_writer(&self.data_dir, stream_name).map(PLMutex::new))?;`
  - `let mut guard = entry.lock(); guard.write_all(...)?; guard.flush_buf_only_if_needed();` — match existing flush semantics.
- `append_many(&self, stream_name: &str, entries: &[...]) -> Result<...>`: same pattern, one lock acquisition for the whole batch.
- `read_entries(&self, stream_name: &str, ...) -> Result<...>`: takes `&self`; opens a fresh `File::open` for reading (or briefly locks the writer to fsync first if Phase 35's helper requires it). Read path doesn't share state with writer except for the on-disk file itself.
- `fsync_all(&self)`: `for writer in self.writers.iter() { writer.lock().get_mut().sync_data()?; }`. Order doesn't matter.
- Existing tests in `event_log.rs` — update to use `&self` calls; many won't need changes since `let log = EventLog::new(...)` works the same.
- New unit test `parallel_writes_to_different_streams_do_not_serialize`:
  - Construct EventLog in a temp dir.
  - Spawn 2 threads. Thread A pushes 1000 entries to stream `A`. Thread B pushes 1000 entries to stream `B`. Use a barrier to ensure both start at the same instant.
  - Time the wall clock; assert `wall < 1.5 * single_thread_time` (tolerant — proves they didn't fully serialize).
  - Verify each stream's log file has 1000 entries.

**Done when:** `cargo test -p tally --lib state::event_log` green. New parallel test passes.

## Task T2 — Drop outer mutex + update call sites

**Files:** `src/server/tcp.rs` (multiple call sites), `src/server/replica_client.rs`, `src/server/replica.rs`, possibly `src/state/snapshot.rs`, `src/main.rs` (event_log construction).

**Subtasks:**
- Change `ConcurrentAppState.event_log` field:
  ```rust
  pub event_log: Option<Arc<EventLog>>,
  ```
- Construction site: wrap in `Arc` once.
- Update all `state.event_log.lock()` + `*event_log` patterns. Sed-like substitution:
  ```rust
  // Old:
  let mut event_log = state.event_log.lock();
  if let Some(ref mut log) = *event_log { log.append(...)?; }
  // New:
  if let Some(ref log) = state.event_log { log.append(...)?; }
  ```
- ~10 call sites in `tcp.rs` per the grep done in Phase 40 CONTEXT. Plus replica_client.rs's append calls. Plus replica.rs LOG_FETCH handler's `fsync_all` (Phase 35 added it).
- Snapshot save path: if it currently locks event_log to read HWM or coordinate with writers, adapt to the lock-free outer / per-writer-lock inner pattern.
- Verify `cargo build` succeeds after each call-site batch — easier to debug than fixing all at once.

**Done when:** `cargo build` + `cargo test` green. `scripts/check-feature-builds.sh` green.

## Task T3 — Multi-stream benchmark + scaling validation

**Files:** `benchmark/tally-throughput/bench_multistream.py` (new), optionally `RESULTS.md` update.

**Subtasks:**
- Write `bench_multistream.py` modeled on `bench_v0.py`'s structure but registers N streams (default 4: `S1`, `S2`, `S3`, `S4`) and pushes events round-robin across them.
- CLI: `--streams N` (default 4), `--clients C` (default 8), `--events E` (default 30000), `--host`, `--label`, `--out`.
- Each client thread picks a stream (round-robin or hash-based on client_id) and pushes asynchronously. Final flush before measuring wall time.
- Output: total events/sec, per-stream events/sec, p50/p99 latency.
- Run matrix scenario:
  - Baseline: `--streams 1 --clients 8` (single-stream, all clients fight for same writer lock).
  - Scaling: `--streams 4 --clients 8` (4 streams, 8 clients, 2 per stream — per-stream lock contention only, ~4x parallel writes).
  - Stress: `--streams 8 --clients 8` (8 streams, 8 clients, 1 per stream — minimal contention).
- Run with `TALLY_WORKER_THREADS=8` against the same release binary.
- Print a comparison table at the end. Save JSON to `--out`.
- Document expected delta in SUMMARY: 4-stream cell should be ~3-4x baseline 1-stream throughput; 8-stream cell should be even higher (limited by `min(streams, worker_threads, cores)`).

**Validation:**
- Re-run `bench_v0.py --matrix --events 30000` after the refactor → expect same numbers as before (single-stream benchmark).
- Run `bench_multistream.py --streams 1 --clients 8` → matches single-stream baseline.
- Run `bench_multistream.py --streams 4 --clients 8` → expect ~3-4x.
- If multi-stream scaling doesn't materialize, stop and report — likely indicates contention I missed (snapshot lock, metrics lock, etc.).

**Done when:** Benchmark JSON written; SUMMARY documents the actual numbers vs expected.

## Stop-and-report triggers

- The current event_log API has shutdown semantics (drop, take, reset) that don't trivially port to `Arc<EventLog>` — flag and discuss.
- Snapshot-save path interacts with event_log in a way more intricate than CONTEXT anticipates — flag with the specific interaction.
- After the refactor, single-stream benchmark numbers regress > 5% — flag (could be DashMap entry overhead vs HashMap; might need to tune).
- Multi-stream scaling doesn't materialize after T3 — flag with the actual numbers and a hypothesis (likely a different mutex is the contention point now; profile if possible).
