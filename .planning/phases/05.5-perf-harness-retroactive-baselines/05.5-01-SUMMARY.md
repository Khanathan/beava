# Phase 5.5 Plan 01: Criterion harness setup + perf-baselines scaffold + §Performance Discipline Summary

**One-liner:** Criterion 0.5 wired as workspace dev-dep with hw-class-keyed `.planning/perf-baselines.md` scaffold and CLAUDE.md §Performance Discipline convention codifying 10%/25% regression gates.

## What Was Built

This plan is scaffolding-only — no benchmark numbers, no measurements. It creates the infrastructure that plans 02–05 fill in and plan 06 populates with baseline numbers.

### criterion workspace wiring

`Cargo.toml` gains `criterion = { version = "0.5", default-features = false, features = ["html_reports"] }` under `[workspace.dependencies]`. The `rayon` default feature is explicitly off because the project invariant is single-threaded event processing — bench workers can still use threads but we don't want criterion spinning up a rayon pool by default.

### Bench target declarations

Both hot-path crates now declare `[[bench]]` targets with `harness = false`:

- `crates/beava-core`: `phase4_expr` and `phase5_agg` (plans 03/04 populate)
- `crates/beava-server`: `phase25_wire` (plan 02 populates)

Stub files contain only `fn main() {}` so `cargo bench --workspace --no-run` compiles today. Downstream plans replace the stub with a `criterion_group! / criterion_main!` pair.

### .planning/perf-baselines.md

Created with:
- Header explaining hw-class keying rationale
- Shell commands to capture hw-class string on macOS and Linux
- Template `## hw-class: <FILL AT PHASE-END>` table (plan 06 populates with real numbers)

The literal `## hw-class:` string is what the harness smoke test asserts is present.

### CLAUDE.md §Performance Discipline

Added as a sibling subsection to `### TDD Discipline` under `## Conventions`:
- 10% slower → WARNING in phase VERIFICATION.md
- 25% slower → BLOCKER (phase fails)
- Phase 6+ mandatory bench rule + plan-checker contract
- References `.planning/perf-baselines.md` by path

### Harness smoke test (TDD)

`crates/beava-server/tests/phase5_5_harness_smoke.rs` — single `#[test] fn harness_scaffolding_present()` with 5 `assert!(contents.contains(...))` checks. Gated behind `required-features = ["testing"]` per project convention. Committed RED first (all 5 fail), then GREEN after implementation.
