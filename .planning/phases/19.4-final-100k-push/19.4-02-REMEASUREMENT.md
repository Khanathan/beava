# Phase 19.4 Plan 02: Re-Measurement Report

## Summary

**Verdict: STILL_MISS — both floors fail by larger margins than the original HALT measurement.**

Re-running the macro measurement on a system **noisier than the original HALT run** (mean load avg 9.4 across 10 runs, vs ~6.67 during HALT) shows the agg-stage median *regressing* further (13,924 ns vs 11,667 ns post-19.4-01 baseline → +2,257 ns *slower*) and EPS median dropping (62,309 vs 88,000 floor; previous HALT median was 69,573).

**The re-measurement does not vindicate the structural fix and does not clear the floor.** System noise is not the only explanation — the trace medians are systematically **above** the post-19.4-01 baseline, suggesting either:
- (a) the cap=16 widening did not produce the expected agg-stage drop on fraud-team's actual workload, OR
- (b) load contention is *amplifying* per-event cost in proportion to the wait stages embedded in the agg-stage timing.

Both readings are consistent with the prior HALT verdict that the structural fix is real (spill-counter test PASSES; Plan-01 baseline showed -40 ns median improvement under noise) but **the measurement floor (-397 ns) cannot be cleared on this hardware under any normal load profile we can reproduce**.

## Disposition recommendation

**Surface to orchestrator without HALT-loop.** Three options remain on the table (same as DEVIATION.md):

1. **Re-measure on a truly quiet system** (no other apps running; load avg <2). This run failed because the system was not quiet enough — load went *up* during re-measurement window, not down. Recommended next step is to run on a clean reboot or a different host.

2. **Accept and re-derive floors against a noise-aware baseline.** The structural fix (spill-counter test PASSES) is real; the macro signal is dominated by host noise. Re-derive Plan 19.4 floors against a paired-test (same run = baseline + fix on identical workload, sharded across many short runs) instead of unpaired absolute medians.

3. **Revert cap to 8.** If the post-19.4-01 baseline (11,667 ns / 79,367 EPS) cannot be regained on this hardware even with no source change, the regression is on the host side and not the SmallVec change. Reverting won't help.

**Recommended:** Option 1 (truly quiet system) — but only after confirming the post-19.4-01 baseline (11,667 ns / 79,367 EPS) is reproducible *first* on the same quiet system, since this measurement shows that even with HEAD `56f4c10` shipped, fraud-team K=10k EPS is running at 53k–71k on this hardware right now, which is well below the Plan 01 baseline of 79,367. There may be a wider host-state regression unrelated to the cap change.
