## Decisions Made

- **Patch BOTH resolvers in lockstep.** Plan text emphasizes both `resolve_field_indices_for_agg_mut` and `_inner` must receive identical patches; the inner is the runtime-critical path (registry.rs:458 — `apply_registration` write-lock closure), the public is the test/external path. Identical logic ensures snapshot replay invariance and test parity.
- **Treat partial geo declarations as malformed.** If only one of `lat_field`/`lon_field` is `Some`, both indices stay at `FIELD_IDX_NONE` (matching the dispatch arm's `if lat_idx != FIELD_IDX_NONE` check; partial resolution would be a bug because the dispatch only inspects lat_idx).
- **Don't touch agg_geo.rs.** The fast path (`update_at` + `read_lat_lon_from_extracted`) already exists from Phase 19.2-06's partial landing. The only missing piece was register-time index assignment, which lives in registry.rs.
- **Use BEAVA_TRACE_APPLY_TIMING=1 alone** for the agg-stage outer measurement, matching Plan 02 attempt-3's methodology. An earlier set of contaminated runs with both APPLY+AGG timing flags showed inflated numbers (~46,000 ns/event) due to per-feature instrumentation overhead; those runs are retained only as the source for the per-AggKind ns/call relative table (per-kind data is independently captured by AGG_TIMING and isn't itself inflated by the simultaneous APPLY_TIMING flag).

## Deviations from Plan

### None auto-fixed

The plan executed exactly as written: RED test → GREEN resolver patches → measurement gate. No Rule 1/2/3 deviations needed; the resolver logic is clean, all tests passed, clippy + fmt + grep acceptance criteria all green.

### One methodology adjustment (within plan spec)

The first set of 7 trace runs accidentally enabled both `BEAVA_TRACE_AGG_TIMING=1 BEAVA_TRACE_APPLY_TIMING=1` flags simultaneously, which inflated the per-event apply cost by ~5× because the per-feature timing instrumentation runs synchronously in the apply path. The plan-supplied command in Task 3.c step 3 includes both flags together, which is what the plan author wrote, but Plan 02 attempt-3's load-filter methodology used APPLY_TIMING alone for the agg-stage outer measurement. To preserve apples-to-apples comparison with Plan 02's 10,329 ns post-19.4-02 baseline, this measurement re-ran the 7 trace runs with APPLY_TIMING=1 only. The contaminated runs are retained as the source for the per-AggKind ns/call relative table (the AGG_TIMING data is captured independently and isn't inflated by APPLY_TIMING).

This is a methodology reconciliation, not a Rule 1/2/3 fix — the plan's intent was apples-to-apples comparison with the post-19.4-02 baseline, which the clean APPLY_TIMING-only runs achieve.

## Issues Encountered

- **System load drift between trace and EPS phases.** The 7 trace runs ran while 1m load was 2.83-3.49 (very quiet); by the time the 7 EPS runs followed, 1m load had drifted to 4.64-7.03. The first set of EPS runs missed the floor with filtered median 85,650; a second set after a brief idle period (loads 3.45-6.07) cleared the floor at 94,733. The load filter (drop 2 highest-load runs) handles this gracefully; the methodology is the same as Plan 02 attempt-3's PASS run.

- **samply --unstable-presymbolicate is insufficient.** It only resolves ~286 of the 37,899 binary symbols. atos symbolication (840 RVAs from the beava-apply thread, resolved via `atos -arch arm64 --offset`) was used as the standard methodology per `19.3-FLAMEGRAPH.md §1`. This is a pre-existing finding from Phase 19.3, not new to this plan.

- **Criterion bench is structurally insensitive to this lever.** The synthetic registry in `apply_path_bench.rs:33-46` has zero geo features; the bench delta (+8.7% windowed, +1.7% plain) is criterion CI variance, not a regression. Same pattern as Plans 19.4-01 (where only 2/14 features were CountDistinct) and 19.4-02 (where the 7-field row never spilled at cap=8). Per CONTEXT D-04 dual-measurement, the live-trace + EPS on fraud-team is the primary verdict.
