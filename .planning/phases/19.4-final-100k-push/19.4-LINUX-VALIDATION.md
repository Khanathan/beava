## Headline finding: EPYC ≠ M4 for fraud-team workload

Linux EPYC-Genoa is consistently **slower than Apple-M4** on this workload:

| Pipeline class | Linux:M4 ratio range |
|---|---|
| Pure-update (small, medium, large, large_phase9) | 0.96–1.08× — **parity** |
| Decay/velocity (medium_phase9) | 0.89× |
| Point/recency (phase8) | 0.83× |
| Geo (geo) | 0.82× |
| Sketches (medium-with-sketches, large-with-sketches) | 0.62–0.75× — **largest gap** |
| **fraud-team (full sketch + geo + decay mix)** | **0.66× (vs M4 loaded) / 0.50× (vs M4 clean)** |

### Why fraud-team falls hardest

1. **CPU-bound, not I/O-bound.** fraud-team pushes 110 features per event across 14 sources. The bottleneck is per-event CPU work (sketch updates, hash lookups, AggOp dispatch), not WAL fsync or TCP I/O. KVM-virtualized 2.0 GHz cores can't match Apple-M4's 4.4 GHz boost on this workload.

2. **Sketches are L1-cache-pressure-sensitive.** UDDSketch percentile and HLL count_distinct require ~1 KB working state per (entity, feature) pair. fraud-team's K=10k entities × 110 features → ~1.1 GB hot state, way past EPYC's per-vCPU L1d (32 KB) and even per-CCD L2 (1 MB). Apple-M4's much larger L2 (16 MB shared per cluster) is a dominant advantage on this shape.

3. **AVX-512 doesn't help the apply path.** Phase 19.4-A (CountDistinct identity hasher) and 19.4-D (ExtractedFields hoist) are scalar improvements. Beava's apply path doesn't currently SIMD-vectorize sketch updates, so EPYC's AVX-512 capability is unused.

4. **Hypervisor jitter on push-latency tail.** Linux p99 push latency on fraud-team is 427 ms vs Apple-M4's 320 ms. The KVM hypervisor adds steal time when the box is busy (we observed load 3.86 = ~25% utilization of 16 vCPU); this contributes to the wall-time deficit.

### Phase 19.4 wins still apply directionally

The 5-pipeline ladder (small/medium/large/large_phase9 + medium_phase9) is at 0.89–1.08× of M4 — these pipelines have lower sketch density and benefit more from the apply-stage hoist + identity hasher + lat/lon pre-extraction. The Phase 19.4 wins are real on Linux too; they just don't lift fraud-team enough on EPYC to clear 100k.

### Production deployment implication

Per the architectural memory `project_no_sharded_apply` (Redis-cluster pattern, multiple instances rather than sharded apply), the production target is:

> Customer runs N instances of beava (one per partition key prefix), each instance hits 100k+ EPS on its assigned shard.

EPYC at 51k EPS on a 110-feature pipeline meets a different bar than Apple-M4: customers using EPYC-Genoa for fraud-team-shaped workloads should plan for **2× the instance count** versus M4-based dev numbers, OR simplify the feature pack (drop sketches, use point/recency or decay instead). The 5-pipeline ladder shows Linux EPYC is **near-parity for non-sketch workloads** — i.e. customers doing decay/velocity-only or point/recency-only feature packs see no perf delta vs Apple-M4.

This is also the first Linux x86_64 reference in the throughput-baselines.md ledger; subsequent Phase 20+ Linux-class regression-gate runs should compare against these EPYC numbers, not the Apple-M4 numbers.

## Python SDK test suite (Linux x86_64)

### Tier 1 (pure unit tests, no server fixture)

**Selection:** `tests/test_agg.py tests/test_client.py tests/test_col.py tests/test_decay_velocity.py tests/test_embed.py tests/test_errors.py tests/test_eval_reference.py tests/test_events.py tests/test_fork_unit.py tests/test_operators.py tests/test_package_init.py tests/test_schema.py tests/test_suggest_config_cli.py tests/test_tables.py`

**Result: 266 PASSED in 2.03 s.** Zero failures, zero errors. Pure-Python SDK unit tests are fully clean on Linux Python 3.12.

### Full unit-test suite (incl. server-fixture-using tests)

**Command:** `pytest tests/ --ignore=tests/integration --ignore=tests/bench` (with `cargo` on PATH so `beava_binary` session fixture can build `beava` debug binary).

**Result: 657 PASSED, 40 failed, 22 errors in 6.05 s.**

The 40 failed + 22 errors are all in tests using the `beava_server` per-test fixture which spawns a fresh `beava --config /dev/null` subprocess on `BEAVA_LISTEN_ADDR=127.0.0.1:0`. Pattern: first test in a file passes, subsequent tests in the same file fail with `ConnectionRefusedError [Errno 111]`. The fixture's `proc.terminate()` + `proc.wait(timeout=5)` teardown does not appear to fully release the WAL/snapshot directory between fixture instances (the conftest spawns the server in pytest CWD `/root/tally/python/`, not in a per-test tmpdir, so each fixture inherits stale `beava-wal/` from the prior run).

**Cross-platform sanity check:** Running `pytest tests/test_app.py` on **macOS** with the same source tree produces an **identical** failure pattern (5 passed, 9 failed). This confirms the test-fixture issues are **NOT a Linux regression** — they are a pre-existing test-design issue on the v2/greenfield branch (HEAD `05a448d` is `wip: phase-19.4 paused at PASS`, with conftest changes still in flight per the v2 architecture refactor).

| Test suite | Linux x86_64 (this run) | macOS (sanity check) | Verdict |

| Pure unit (no server) | 266 / 266 PASS | (not measured today, historical: PASS) | LINUX OK |

### Tier 2 (integration tests)

**Selection:** `tests/integration/test_fork_extract_history.py tests/integration/test_fork_python_api.py`

**Result: 4 SKIPPED.**

Both files look for `target/release/tally` or `target/debug/tally` — the **legacy `tally` binary name** from before the rename to `beava`. Neither binary exists in v2. The skip is conditional via `pytest.skip("tally binary not built; run cargo build to enable")`. **This is stale test code, not a Linux issue.**

Additionally, both integration files use `@pytest.mark.timeout` which is not registered in `pyproject.toml`'s `markers` list, causing collection errors under `--strict-markers`. Worked around by passing `-o markers='timeout' --override-ini='addopts=-ra'` (still skipped due to the `tally` binary check above).

**Skipped per task scope.** The `tests/bench/` suite is for pytest-benchmark perf baselines (Phase 5.5); not part of this validation.

## Action items for follow-up

1. **Phase 20+ regression gate:** Use this Linux EPYC table as the per-pipeline regression baseline for x86_64 production-class hardware. Apple-M4 numbers remain the dev-time gate.

2. **Sketch path is the optimization frontier on x86_64.** The Linux:M4 ratio shows sketch-heavy pipelines (large-with-sketches 0.62×, fraud-team 0.66×) lose more on EPYC than non-sketch pipelines. Phase 20.x sketch optimizations targeting AVX-512 or cache-friendly layouts would benefit Linux x86_64 deployments most.

3. **fraud-team 100k gate is M-series-specific.** The PASS verdict in Phase 19.4-05 (102,800 EPS on M4) does not generalize to EPYC-Genoa at 51k EPS. The v0 ship gate should either: (a) be re-stated as "≥50k EPS on production-class Linux" or (b) include a multi-instance footnote per `project_no_sharded_apply` (single-instance EPYC = 51k, customer runs 2 instances = 100k+).

4. **Python SDK conftest cleanup (out-of-scope deferred):** the `beava_server` fixture should `tmp_path_factory.mktemp()` per test invocation and set `BEAVA_DATA_DIR` env var to that path so concurrent tests don't share WAL/snapshot state. This is a pre-existing v2 issue, identical pattern on macOS — file as a separate task post Phase 14.
