The type alias to update is at `agg_op.rs:232` (real source — note the lifetime parameter `'a` and the fully-qualified `crate::row::Value`):
```rust
pub type ExtractedFields<'a> = smallvec::SmallVec<[Option<&'a crate::row::Value>; 8]>;
```

becomes:
```rust
pub type ExtractedFields<'a> = smallvec::SmallVec<[Option<&'a crate::row::Value>; 16]>;
```

The lifetime `'a` is the borrow lifetime of the row whose values the slice references. It is NOT `'static`; consumers like `agg_geo.rs:42` (`read_lat_lon_from_extracted(extracted: &ExtractedFields<'_>, ...)`) use the elided lifetime form.

`ExtractedFields` consumers in `agg_apply.rs`, `agg_op.rs`, `agg_geo.rs`, `agg_state.rs`, and `registry.rs` are all variance-compatible: SmallVec exposes the same `len()`, `push`, `clear`, `iter`, indexing API regardless of inline cap. The only observable difference is `SmallVec::spilled()` — which is exactly what the RED test asserts on.

`SmallVec::spilled()` returns `true` iff the SmallVec has overflowed its inline buffer onto the heap. For a SmallVec with inline cap `N`, `spilled()` is `true` iff `len() > N`.

Cargo dependency `smallvec` is already a workspace dep (see `crates/beava-core/Cargo.toml:35-37`); no Cargo.toml edits needed.
