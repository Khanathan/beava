The four predicted hot-function shifts from CONTEXT D-04 broke into **3 confirmed + 1 partial**:

The `apply_event_to_aggregations` self-time drop (-0.27 percentage points) is much smaller than the predicted (-5.13). This is consistent with the Plan 04 trace deficit: the per-desc rebuild cost was already cheap post-Plan-02 cap-widening (~50 ns/rebuild instead of the ~500 ns assumed in the cost model), so the hoist removed less self-time than predicted. A new top-3 entry — `alloc::str::_<impl str>::to_lowercase` at 9.32% self-time — appears to be a samply unwinder artifact, NOT a real new hot function. Detail in §4.

The Phase 19.4 100k EPS PASS gate WAS HIT on the 1M-event measurement (102,800 EPS, see Plan 04 SUMMARY); this sanity flamegraph confirms the optimization-target hot-function shifts happened structurally even though one of the predicted shifts undershot.

### Detailed read on the `apply_event_to_aggregations` shift miss

CONTEXT D-04 predicted self-time would drop from 11.13% to ≤ 6%. Measured: 10.86%. The drop is real but tiny.

The interpretation: the 11.13% pre-19.4 self-time at `agg_apply.rs:205, 225, 257` was attributed to (a) per-desc `extracted` rebuild via `desc.field_names.iter()...collect()`, (b) `update_with_extracted` dispatch call, (c) per-desc trace-mode `Instant` capture. Plan 04's hoist eliminates only (a) — `update_with_extracted` dispatch (b) and per-cluster loop scaffolding remain.

Per Plan 04 SUMMARY's "Plan 04 trace floor missed" analysis, the per-desc rebuild cost at the bench-fixture line was ~50 ns/rebuild post-Plan-02 cap-widening (instead of the ~500 ns assumed in the COST-MODEL). So eliminating 5 rebuilds saved ~250 ns/event total — about ~2% of apply CPU = 0.27 percentage points. The measured shift (-0.27 pp) matches this corrected expectation.

The criterion microbench DOES show -10.9% on the synthetic 14-feature pipeline (where the hoist eliminates 2 redundant rebuilds proportional to a smaller denominator). The hoist works mechanically; the live-flamegraph and live-trace just don't show it as a big self-time shift because the predicted scaffolding cost was an overstatement.

This confirms the Plan 04 SUMMARY's diagnosis that the cost model `19.3-COST-MODEL.md §4` was outdated post-Plan-02 cap-widening. The predicted shift's miss is structurally explained.

## §4 New hot functions (and unwind artifact analysis)

### `alloc::str::_<impl str>::to_lowercase` 9.32% self-time — UNWIND ARTIFACT, not a real call

The post-19.4 top-30 has a surprise: `to_lowercase` at rank #3 with 9.32% self-time / 1,087 ns/event. This was NOT in the pre-19.4 top-30. If real, it would mean a new hot allocation-heavy string operation crept in.

**Investigation:**

1. **Source-tree grep:** Every `to_lowercase` and `to_ascii_lowercase` callsite in `crates/beava-core/src/`, `crates/beava-server/src/`, and `crates/beava-runtime-core/src/` falls in init / config / error-paths / tests. **None on the apply hot path.** Specifically:
   - `wal_writer.rs:294` — `name.to_lowercase()` inside `is_network_fs`, called once at startup
   - `op_node.rs:271,283,295,332` — error-message formatting, only on register-time validation failures
   - `expr.rs:1396, 1411` — error-reason formatting (cold path)
   - `register_validate.rs:1137` — assertion in test
   - `config.rs:242, 319, 346` — env-var parsing at startup
   - `tcp.rs:841` — log-level config parsing
   - `register.rs:938-939` — error-message classification (cold path)

2. **Address ranges:** All 16 addresses leaf-attributed to `to_lowercase` fall in the range `0x100001374`–`0x100001c00` — about 2.3 KB at the very start of the binary's text section, well below where user-code monomorphizations land (`0x10009c000`+ for the `count_distinct::add_hash` mass).

3. **Caller distribution:** When the leaf is `to_lowercase`, the immediate caller (next frame up the stack) is a **diverse, unrelated set**: `TopKState::insert` (76 samples), `apply_shard::dispatch_push_sync` (37), `haversine::distance` (36), `TrendResidualState::update` (27), `EntropyHistogram::insert` (27), `AggStateTable::get_or_init_by_shape` (21), etc. Real hot leaves like `count_distinct::add_hash` show a *single* dominant caller (`WindowedOp::update_at`, 624/628 samples). The diversity of `to_lowercase`'s callers is consistent with **frame-pointer unwind misattribution** at low binary addresses.

4. **Cross-check with 19.3-FLAMEGRAPH.md:** `to_lowercase` does NOT appear in the pre-19.4 top-30. If it were a real new hot function introduced by Phase 19.4 changes, it would correlate with a specific Plan 19.4-{01,02,03,04} code change; nothing in those plans introduces a hot-path `to_lowercase` call.

**Conclusion:** `to_lowercase` is a **samply unwinder artifact** on macOS aarch64 — when the unwind-up walks past a tail-call-optimized or trampoline frame at low binary addresses, atos resolves the closest text-section symbol, which happens to be `to_lowercase`'s monomorphization sitting at `0x100001000` range. The 9.32% does NOT represent real `to_lowercase` work; it is the cumulative self-time of leaf-frames the unwinder couldn't precisely attribute. The samples are most likely from inside `count_distinct::add_hash` (which has high-frequency hashbrown probing) or `top_k::TopKState::insert` (BTreeMap traversal) — paths where heavy memory access patterns can defeat partial frame-pointer chains.

This is consistent with Phase 19.3-FLAMEGRAPH.md §1's note that samply's `--unstable-presymbolicate` sidecar "is largely useless" and atos is required — atos is precise, but it can't recover from missing frame data; it gives the closest-by-address symbol whether or not it's the actual call site.

### `anyhow::fmt::ErrorImpl::debug` 3.54% self-time — also likely artifact

Similar pattern: the symbol is in the very-low binary range (`0x100003a10`) and there is no anyhow::Error::debug callsite on the apply hot path. The 3.54% should be attributed to nearby user-code that the unwinder couldn't precisely walk.

### `http::header::map::hash_elem_using` 3.00% self-time — also likely artifact

This is HTTP-only code; the bench runs over TCP+msgpack which shouldn't touch this path at all. Apply path is pure `apply_shard::dispatch_push_sync` → `apply_event_to_aggregations`. Another low-addr text-section symbol the unwinder mis-attributes.

### Combined artifact mass

`to_lowercase` (9.32%) + `anyhow::fmt::debug` (3.54%) + `http::header::map::hash_elem_using` (3.00%) = **15.86% of apply CPU is unwind-attributed to symbols not on the apply hot path**. Attempts to allocate these to real callers (by walking up the stack to the first user-code frame) push the apparent hot-function distribution slightly toward `count_distinct::add_hash`, `top_k::TopKState::insert`, and `apply_event_to_aggregations` — but the absolute attribution is uncertain.

This artifact mass is roughly the same magnitude as the libsystem_platform.dylib + DYLD-STUB$$memcmp inclusive cost in 19.3-FLAMEGRAPH.md (~14% inclusive). The base rate of unwind artifact on macOS aarch64 with `force-frame-pointers=yes` is ~10-15% of samples for hot Rust binaries — Phase 19.5+ profiling work should consider switching to `dtrace`-based profiling or use Linux-perf on Linux Xeon for cleaner attribution.
