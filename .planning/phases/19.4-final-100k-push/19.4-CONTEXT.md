## Decisions

### D-01: 19.4-A CountDistinct storage = hashbrown + identity hasher

**Decision:** Replace `std::collections::HashSet<u64>` at `count_distinct.rs:25-27` with `hashbrown::HashSet<u64, BuildHasherDefault<NoOpHasher>>`. The `NoOpHasher` returns the input u64 unchanged (no SipHash; the input is already FxHasher's output per Phase 19.2 D-02b — running another cryptographic hash on it is pure waste).

**Why:**
- Flamegraph (`19.3-FLAMEGRAPH.md` §2 row #3): `hashbrown::map::HashMap::insert` self-time = 9.36% of apply CPU = **1,234 ns/event**. 99% of these calls are inside CountDistinct's HashSet mode probing.
- Hash-output collision rate: identity hasher has ZERO hash-output collisions for distinct u64 inputs (whereas SipHash has ~2^-64 by birthday paradox).
- DoS resistance: Phase 19.2 D-02b already accepted FxHasher's non-cryptographic profile. The chain becomes "FxHash bytes → identity slot indexing"; both layers are non-cryptographic but Beava workloads are operator-controlled per memory `project_v2_devex_first`. No regression.
- The HashSet stores u64 hashes (not raw field values) — already approximate by design at ~2^-64 false-positive rate. This is unchanged by the hasher swap; it's a property of the FxHasher upstream step.

**Implementation notes for planner:**
- Add `NoOpHasher` type implementing `std::hash::Hasher` where `write_u64(x)` stores `x` and `finish()` returns it. `write` (byte slice) is unreachable! since u64-only API path is enforced by the type.
- `CountDistinctState::HashSet { hashes: HashSetU64 }` with `type HashSetU64 = hashbrown::HashSet<u64, BuildHasherDefault<NoOpHasher>>`.
- Snapshot wire-format: serializes as `Vec<u64>` (existing serde derive). Replay code instantiates the new collection type from the same on-disk u64 list — no migration needed.
- Promotion thresholds (`EXACT_THRESHOLD = 16`, `HASH_THRESHOLD = 1024`) unchanged.
- `add_hash(hash: u64)` API signature unchanged.

**Acceptance criteria for this decision:**
- `cargo test -p beava-core` passes (existing 606 tests + new identity-hasher slot-collision unit tests).
- Criterion bench shows CountDistinct windowed cost ≤ 200 ns/call (was 457 ns post-19.3-A).
- `grep -c 'std::collections::HashSet' crates/beava-core/src/sketches/count_distinct.rs` = 0.
- `grep -c 'hashbrown::HashSet' crates/beava-core/src/sketches/count_distinct.rs` ≥ 1.
- `NoOpHasher` test: insert two u64 values that mod-collide on a small-cap table, verify both stored as distinct entries (probing works).

### D-02: 19.4-D field-union scope = union across ALL descriptors at register-time

**Decision:** One `ExtractedFields` per event covering all fields any agg on the event source needs. Each agg has a per-source `field_idx[]` array pointing into this single buffer. Single pass to populate at apply-event entry; zero per-desc rebuild.

**Why:**
- fraud-team has 1 dominant source (Txn) with ~12 distinct fields across all 88 features. Per-source-schema option is functionally equivalent for fraud-team but adds source-dispatch overhead.
- Per-cluster option (5 separate ExtractedFields) doesn't capture the full predicted savings — still rebuilds 5× per event.
- Wasted slots: ~30 unused entries per event vs the ~12 used. Indexed access is 1-2 ns; waste is negligible.
- Predicted lift: -1,500 to -2,000 ns/event per `19.3-COST-MODEL.md` §4.

**Implementation notes for planner:**
- `EventDescriptor.apply_field_names: Vec<String>` (already declared per RESEARCH §6 amendment, currently `vec![]` at 15+ construction sites in `registry.rs`). Populate at register-time as the union of fields any agg on this source consumes.
- New `field_idx_into_event_extracted: Vec<u8>` per agg: maps the agg's local field positions to the event-level union positions.
- Apply-loop hoist site: `agg_apply.rs:152` (alongside `shape_cache`, per RESEARCH Q1). One `Vec<Option<&Value>>` local variable, `clear() + resize_with(union_size, || None)` per event.
- SmallVec inline cap widening (19.4-B from 8 to 16) covers the union size for fraud-team (~12 fields). Cap = max(union_size across all sources) + headroom. Static at 16 for v0.

**Acceptance criteria for this decision:**
- `EXTRACTED_BUILD_COUNT == event_count` instrumentation (was `5 × event_count` per descriptor).
- `peak_alloc` shows zero per-event ExtractedFields heap allocations on warm path.
- `grep -c 'apply_field_names: vec!\[\]' crates/beava-core/src/registry.rs` = 0 (was 15+).
- Bit-identical state cross-check: random-input event sequence on a 14-feature pipeline produces identical state via old per-desc rebuild vs new event-level hoist (use `f64::to_bits()` for float fields).

**Decision:** Land each sub-goal in isolation. Run live trace + EPS measurement after each. If measured lift < 75% of predicted, HALT — write DEVIATION.md and re-evaluate before proceeding to next sub-goal.

- Slowest path but most defensible. User explicitly chose this over compressed/parallel landing.
- The 5 plans land in ~5-7 wall-clock days assuming each plan + measurement is ~1 day; that's a single sprint.
