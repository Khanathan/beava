## Areas discussed

### Area 1: 19.4-A CountDistinct storage shape

**Question:** Storage shape for the < 1024-entry Exact mode.

**Options presented:**
1. hashbrown::HashSet<u64, BuildHasherDefault<NoOpHasher>> (Recommended)
2. Flat sorted Vec<u64> with binary_search
3. Three-tier: Vec ≤512, hashbrown 512-1024, HLL >1024

**User clarifications (3 rounds):**

Round 1 — User asked: "how do we even deal with collision". Claude explained the two collision concepts (slot-bucket vs hash-output) and DoS resistance trade-off. Open addressing + key equality identical in both options.

Round 2 — User asked: "but how do we get the u64 number". Claude traced the full data flow: field bytes → FxHasher (per Phase 19.2 D-02b) → u64 → currently SipHash → slot index. Explained that the u64 going into HashSet IS already a hash output of FxHasher applied to field bytes; SipHash is wasted second hashing.

Round 3 — User asked: "but if you put it like this it hashmap wont be able to differentiate different key with same u64 value right". Claude verified by reading the actual code (`count_distinct.rs:21-30`) — confirmed CountDistinctState stores u64 hashes (NOT raw values) in ALL three modes (ExactArray, HashSet, HLL). The "two different keys appearing identical" property is a pre-existing design choice (Phase 19.2 D-02b) at ~2^-64 collision rate, unchanged by hasher swap.

**User selection:** Option 1 (hashbrown + identity hasher).

**Why this matters:** Documented in CONTEXT D-01. The conversation revealed that CountDistinct has approximation built in at every mode (it stores hashes, not values). Identity hasher swap doesn't change correctness; it just skips redundant SipHash work. The acceptance criterion includes a unit test that pathological mod-collisions resolve correctly via probing.
