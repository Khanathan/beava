## Decisions Made

- **Vanilla fetch + setInterval (no htmx):** Plan 02 intentionally left the shell without any `hx-*` attributes. Plan 03's app.js owns all polling with a single shared `state.paused` gate â€” simpler to reason about than htmx's polling trigger lifecycle, and aligns with RESEARCH Pitfall 2.
- **Render once, patch in place:** dagre-d3's `render(inner, g)` is called exactly once in `loadTopology` and stored references (`state.dagInner`, `state.dagG`) let the 1 Hz tick do pure d3 `.text()` mutations on existing `<tspan>` nodes. This avoids layout flash and keeps the graph stable across ticks (RESEARCH Pattern 6).
- **Five sections for streams, four for views:** Per UI-SPEC, views omit Memory and Throughput because they have neither a bucket store nor a throughput counter; they substitute a Features section (lookup/derive list) for the State section.
- **Space-to-pause gated on non-input focus:** `if (document.activeElement.tagName === 'INPUT' || ...)` skip â€” so typing a space in the entity-lookup input doesn't freeze polling.
- **el/svgEl as XSS chokepoint:** Rather than relying on per-call audit for XSS safety, the `el()` and `svgEl()` helpers accept only `(tag, attrs, children)` where children are strings or DOM nodes. There is no path through these helpers to inject raw HTML, which makes the forbidden-substring regression test (`app_js_has_no_innerhtml_or_eval_sinks`) a tight fence rather than a best-effort audit.
- **Stream-scoped entity lookup:** The click-to-select context is "inspect ONE stream", so the lookup filters `/debug/key/{key}` response to the `{SelectedStream}.` prefix and strips it before rendering a feature grid. This is different from Phase 10's flat-tab lookup which showed ALL features for the key.
