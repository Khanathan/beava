# Phase 10.1: Interactive Debug UI Redesign - Research

**Researched:** 2026-04-10
**Domain:** Frontend rewrite of the Phase 10 Debug UI (5 files under `src/server/ui/`) + one additive backend field on `/debug/topology`. All work happens on top of the locked Phase 10 stack (htmx 1.9.12, d3 7.8.5, dagre-d3 0.6.4, vanilla JS) and under the same XSS / accessibility contract documented in Phase 10's UI-SPEC §13.1 and REVIEW WR-03.
**Confidence:** HIGH

## Summary

Phase 10.1 replaces Phase 10's four-tab flat layout with a split-view design where the dagre-d3 topology DAG is the primary navigation surface and a right-side drill-in panel renders per-stream details (memory, operator state, throughput, stream-scoped entity lookup). Edges carry live 5s-EWMA throughput labels refreshed at 1 Hz. The entire Phase 10 tab bar is deleted, which also resolves the broken Entity form wiring (REVIEW WR-02) as a side effect.

The one and only backend change is additive: `/debug/topology` node payloads gain an `operators` field alongside the existing `features` array. The cleanest source of truth for that field is the **already-persisted** `PipelineEngine::raw_register_jsons` map — the user's original register JSON shape already contains `{name, type, window, where, field, expr, ...}` per feature, with the original **string** `where` clause rather than the parsed `Expr` AST. Walking the `FeatureDef` enum variants would require a new AST → string serializer that does not exist and is strictly harder than passing through the raw JSON. [VERIFIED: `src/engine/pipeline.rs:858-866`, `src/server/protocol.rs:244-285`]

Three hard performance and correctness questions drive the frontend design:

1. **Don't re-render the DAG on every tick.** dagre-d3 0.6.4's `render(svg, g)` re-runs the full layout pass AND rebuilds every edge label sub-element via `root.select(".label").remove(); addLabel(...)` [VERIFIED: dagre-d3 0.6.4 `createEdgeLabels` function, inspected in `src/server/ui/vendor/dagre-d3.min.js`]. Calling it once per second flashes the graph. The canonical fix is to render once on initial load and then update edge label text in-place via `d3.select('g.edgeLabel[...] g.label text tspan').text(...)` without calling `render()` again. dagre-d3 uses the D3 update pattern and keys `g.edgeLabel` groups by `"v:w:name"`, so the elements persist across ticks as long as we don't re-invoke `render`.

2. **The global Pause button must halt BOTH htmx polls AND the vanilla-JS edge-label updater.** Phase 10 paused htmx by toggling `data-hx-disable` on every `[hx-trigger*="every"]` container. Phase 10.1's edge-label updater will NOT be an htmx swap (because dagre-d3 owns the SVG and htmx should never touch it) — it'll be a vanilla `fetch('/debug/throughput')` called from a `setInterval`. The pause logic must flip a shared `state.paused` flag that BOTH the htmx containers AND the `setInterval` callback respect before doing any work.

3. **Click-to-select node state stays dead simple.** With four tabs gone there is no router to worry about. A single `state.selectedStream` variable + a `setSelected(name)` function that (a) updates `.selected` CSS classes on `g.node` elements, (b) calls a panel renderer with the new selection, and (c) updates `aria-pressed` for a11y. No event bus, no custom events, no observer pattern. ~25 LOC total.

**Primary recommendation:** Render the DAG exactly once when topology data arrives, attach click handlers per node via `inner.selectAll('g.node').each(function(v){ d3.select(this).on('click', () => setSelected(v)); })`, drive edge-label updates via a vanilla `fetch` + `setInterval` loop that computes `const t = d3.select('#dag g.edgeLabel')...` selections and calls `.text()` directly, and gate both the htmx containers and the setInterval callback on the same `state.paused` flag. Pass the raw register JSON `features` array through `/debug/topology` as the new `operators` field — zero FeatureDef conversion needed.

### Pattern 1: dagre-d3 node click handlers

**Question:** How to attach click handlers to dagre-d3 rendered nodes in d3 v7?

**Answer:** dagre-d3 0.6.4 renders each node as an SVG group element `g.node` keyed by the name passed to `setNode()`. The node has no id attribute by default. The canonical d3 v7 pattern is to select all node groups after `render()` has run and attach handlers via `.on('click', ...)`.

**Verified from source:** `src/server/ui/vendor/dagre-d3.min.js` — the `createNodes` function:
```js
function createNodes(selection, g, shapes) {
  var simpleNodes = g.nodes().filter(...);
  var svgNodes = selection.selectAll("g.node").data(simpleNodes, function(v){return v}).classed("update", true);
  svgNodes.enter().append("g").attr("class", "node")...;
  // ...
  svgNodes.each(function(v) {
    var node = g.node(v);
    var thisGroup = d3.select(this);
    util.applyClass(thisGroup, node["class"], ...);
    // ...
  });
}
```

Each `g.node` is bound to its node key `v` (the name string). d3's `.each()` iteration passes that key as the first argument, so a click handler can close over it without a separate lookup.

**Canonical snippet:**
```javascript
// After: render(inner, g);
inner.selectAll('g.node').each(function(v) {
  const nodeData = g.node(v);      // {label, rx, ry, class, ...}
  const sel = d3.select(this);
  sel.style('cursor', 'pointer');
  sel.on('click', (event) => {
    event.stopPropagation();        // don't deselect via background click
    setSelected(v);                  // v is the node name, e.g. "Transactions"
  });
  // Keyboard a11y: make the node focusable and handle Enter/Space
  sel.attr('tabindex', 0)
     .attr('role', 'button')
     .attr('aria-label', v)
     .on('keydown', (event) => {
       if (event.key === 'Enter' || event.key === ' ') {
         event.preventDefault();
         setSelected(v);
       }
     });
});

// Deselect when the user clicks empty SVG background:
d3.select('#topology-svg').on('click', () => setSelected(null));
```

**Why this works:** dagre-d3's D3 update cycle preserves `g.node` elements across re-renders. BUT in Phase 10.1 we render the DAG only ONCE on topology load (see §6), so the click handlers attached here stay bound for the lifetime of the page. No rebind is needed on edge-label ticks because ticks don't call `render()`.

**Confidence: HIGH.** Source verified in vendored `dagre-d3.min.js`. d3 v7 `.on('click', handler)` and `.each(function(d){...})` are the documented idiomatic patterns [CITED: d3-selection docs, https://d3js.org/d3-selection/events].

### Pattern 2: dagre-d3 edge labels (one-time setup)

**Question:** How to add text labels to dagre-d3 edges?

**Answer:** Set a `label` property when calling `g.setEdge(from, to, {...})`. dagre-d3's default `labelType` is `"text"`, which renders the label as an SVG `<text>` element (XSS-safe, matches Phase 10's contract). **Do NOT set `labelType: "html"`** — Phase 10 REVIEW WR-03 bans that string in app.js via grep regression test.

**Verified from source:** `src/server/ui/vendor/dagre-d3.min.js` — the `addLabel` dispatcher:
```js
if (edge.labelType === "svg") { addSVGLabel(labelSvg, node); }
else if (typeof label !== "string" || node.labelType === "html") { addHtmlLabel(labelSvg, node); }
else { addTextLabel(labelSvg, node); }
```

When `label` is a string (the only case we use) and `labelType` is undefined (the default), it calls `addTextLabel`:
```js
function addTextLabel(root, node) {
  var domNode = root.append("text");
  var lines = processEscapeSequences(node.label).split("\n");
  for (var i = 0; i < lines.length; i++) {
    domNode.append("tspan").attr("xml:space", "preserve").attr("dy", "1em").attr("x", "1").text(lines[i]);
  }
  return domNode;
}
```

So every edge label is rendered as `<text><tspan>...</tspan></text>` inside the `g.edgeLabel` group. Multi-line labels use multiple `tspan` children. For compact `1.2k/s` style labels, there's always exactly one line → exactly one `tspan`.

**Canonical snippet (initial DAG build, called ONCE on topology load):**
```javascript
const g = new dagreD3.graphlib.Graph()
  .setGraph({ rankdir: 'LR', marginx: 24, marginy: 24, nodesep: 24, ranksep: 48 })
  .setDefaultEdgeLabel(() => ({}));

for (const node of topologyData.nodes) {
  g.setNode(node.name, {
    label: node.name,              // dagre-d3 default labelType: 'text' — XSS-safe
    class: 'node ' + (node.kind === 'view' ? 'view' : 'stream'),
    rx: 12, ry: 12,
    paddingX: 16, paddingY: 12,
  });
}

for (const edge of (topologyData.edges || [])) {
  g.setEdge(edge.from, edge.to, {
    label: '—/s',                  // placeholder, will be overwritten by the 1 Hz updater
    labelpos: 'c',                 // label at edge midpoint
    arrowhead: 'vee',
    curve: d3.curveBasis,
  });
}

const render = new dagreD3.render();
const inner = d3.select('#topology-svg').append('g').attr('id', 'dag-root');
render(inner, g);                   // <-- called exactly once per topology load
```

**Pitfall:** `labelType: 'html'` is explicitly banned file-wide by the `app_js_has_no_innerhtml_or_eval_sinks` regression test. [VERIFIED: `tests/test_debug_ui.rs:778-779`]. Even in a comment the string would fail — keep it out entirely.

### Pattern 3: Throughput label compact format

**Question:** How to format a number like 1234.56 as `'1.2k/s'` or 450 as `'450/s'` in vanilla JS without external libraries?

**Answer:** A tiny helper that handles four cases: null/NaN, 0, small (< 10), medium (< 1000), and thousands (≥ 1000). Phase 10's `fmtNumber` already does the `k/M` split but doesn't append `/s` and prints `0.0` for zero. Phase 10.1 needs a tighter format that matches the inline label aesthetic on DAG edges.

**Canonical snippet:**
```javascript
/**
 * Format an events/sec number as a compact label: '—/s', '0/s', '12.3/s',
 * '450/s', '1.2k/s', '12k/s', '1.2M/s'. Designed for dagre-d3 edge labels
 * where horizontal space is tight.
 *
 * Rules:
 *  - null / undefined / NaN         → '—/s'
 *  - exactly 0                      → '0/s'
 *  - (0, 10)                        → one decimal place, e.g. '12.3/s' stays '12.3/s',
 *                                      but 0.4 → '0.4/s'
 *  - [10, 1000)                     → integer, e.g. '450/s'
 *  - [1000, 1_000_000)              → '1.2k/s' (one decimal), '12k/s' (no decimal when ≥ 10)
 *  - [1_000_000, ∞)                 → '1.2M/s' (one decimal), '12M/s' (no decimal when ≥ 10)
 */
function formatFps(n) {
  if (n == null || Number.isNaN(n)) return '—/s';
  if (n === 0) return '0/s';
  const abs = Math.abs(n);
  if (abs < 10)      return n.toFixed(1) + '/s';          // '0.4/s', '4.2/s', '9.9/s'
  if (abs < 1000)    return Math.round(n) + '/s';         // '12/s', '450/s', '999/s'
  if (abs < 1e6) {
    const k = n / 1000;
    return (k < 10 ? k.toFixed(1) : Math.round(k)) + 'k/s';  // '1.2k/s', '12k/s'
  }
  const m = n / 1e6;
  return (m < 10 ? m.toFixed(1) : Math.round(m)) + 'M/s';
}
```

**Edge cases and why they matter:**
- **Null/NaN:** happens when `/debug/throughput` has never received a poll yet for a new edge, or when the stream doesn't exist in the throughput snapshot (which is common during the first 100 ms after a fresh register). Render `—/s`, never `undefined/s` or `NaN/s`.
- **Exactly 0:** happens for idle streams after EWMA decay. Render `0/s` (not `0.0/s` — the decimal is noise).
- **Fraction < 10:** happens for slow streams and for EWMA tails. One decimal place.
- **Decimal collapse above 10:** above 10 events/sec, the 10ths digit is noise — drop it for visual calm.

**Claude's Discretion call from CONTEXT:** pick ONE format and apply consistently. This helper is that choice.

**Confidence: HIGH.** Pure JS math, no library dependency, trivial to unit test.

### Pattern 4: Split-view layout with CSS

**Question:** Best practice for a sticky-header + split-view main + sticky-footer layout in dark-mode CSS?

**Answer:** CSS Grid on the body for the outer frame (header / main / footer rows), CSS Grid on `main` for the inner split (left canvas / right panel columns). Grid wins over flexbox here for two reasons:
1. Named template rows on the body give you explicit heights for header and footer without flex tricks.
2. The `minmax(0, 1fr)` idiom is the only robust way to make a Grid track shrink below its content — critical for the DAG SVG which has no intrinsic size.

**Scrolling:** each pane owns its own `overflow-y: auto`. The body itself never scrolls — only the main content panes do.

**Canonical skeleton (uses Phase 10 UI-SPEC §13.1 tokens verbatim):**

**Critical gotchas:**
1. **`min-width: 0` and `min-height: 0` on grid items** — without them, CSS Grid children cannot shrink below their intrinsic content size. The SVG has no intrinsic min-size but dagre-d3's rendered content can be wider than the viewport, which causes overflow and forces the grid track to grow. Setting `min-width: 0` on `.dag-canvas` allows it to shrink to any width; the SVG then clamps to 100% and dagre-d3's transform-scale-to-fit computation takes over.
2. **`overflow: hidden` on `main.split-view`** — establishes a containing block for the drill-in panel's `overflow-y: auto`. Without it, scroll cascades up to the body and the sticky footer breaks.
3. **`grid-template-columns: minmax(0, 1fr) 360px`** (NOT `1fr 360px`) — same `min-content` issue. `1fr` has an implicit `auto` minimum that refuses to shrink below content. `minmax(0, 1fr)` fixes it.
4. **Drill-in panel width:** 360px is a reasonable fixed default. Could be 320px (matches Phase 10's `.topology-detail` aside in `app.css:368`) or 400px (more breathing room for the five-section stack). Recommend **360px** — halfway, rounds to a grid-friendly number, matches Datadog's ServiceMap side panel width [CITED: Datadog public docs screenshots].

**Confidence: HIGH.** The `minmax(0, 1fr)` pattern is the canonical CSS Grid escape hatch documented widely [CITED: https://css-tricks.com/preventing-a-grid-blowout/]. The split-view pattern itself is Grafana Explore's canonical layout [CITED: https://grafana.com/docs/grafana/latest/explore/].

### Pattern 5: Click-to-select state management in vanilla JS

**Question:** Without a framework, what's the cleanest pattern for tracking "currently selected node" state and propagating it to a side panel renderer?

**Answer:** One module-scoped `state` object + one `setSelected(name)` function that imperatively updates the DOM and triggers panel renders. No pub/sub, no custom events, no observer pattern. The entire system fits in ~25 LOC.

**Canonical snippet:**
```javascript
// ---------- Module-scoped state ----------

const state = {
  paused: false,
  selectedStream: null,        // currently-selected node name, or null
  topology: null,              // last /debug/topology response (cached for panel renders)
  throughput: null,            // last /debug/throughput response (cached for panel renders)
  memory: null,                // last /debug/memory response (cached for panel renders)
  lastUpdate: null,
};

// ---------- Selection controller ----------

function setSelected(name) {
  // name may be a stream/view name or null (deselect)
  state.selectedStream = name;

  // 1. Update DOM selection classes on every rendered node.
  //    dagre-d3 keys g.node by the name we passed to setNode, so we match
  //    via d3 data binding — not by text content or attribute lookup.
  d3.select('#topology-svg').selectAll('g.node')
    .classed('selected', function(v) { return v === name; })
    .attr('aria-pressed', function(v) { return String(v === name); });

  // 2. Render the drill-in panel.
  renderDrillInPanel(name);
}

function renderDrillInPanel(name) {
  const panel = document.getElementById('drill-in-panel');
  if (!panel) return;
  clear(panel);

  if (name == null) {
    panel.setAttribute('data-empty', 'true');
    panel.appendChild(el('p', { class: 'drill-in-placeholder',
      text: 'Select a stream to see details' }));
    return;
  }
  panel.removeAttribute('data-empty');

  // Find the node in the cached topology data.
  const node = (state.topology && state.topology.nodes || [])
    .find((n) => n.name === name);
  if (!node) return;

  // Render the five sections from cached state. Each renderer is a pure
  // function that takes (panel, node, throughput, memory) and appends
  // DOM via textContent-only helpers.
  renderStreamHeader(panel, node);
  renderMemorySection(panel, node, state.memory);
  renderStateSection(panel, node);                    // reads node.operators
  renderThroughputSection(panel, node, state.throughput);
  renderEntityLookupSection(panel, node);
}
```

**Why this shape:**
- `setSelected` is the single mutation point for `selectedStream`. Every code path that changes selection calls it. This makes debugging trivial: set a breakpoint in `setSelected` and you see every change.
- The DOM update is idempotent — calling `setSelected('X')` twice is a no-op. Critical for click-to-toggle behavior.
- The `d3.select(...).classed(..., function(v){...})` pattern uses the bound node key (`v`), NOT the DOM text. This means node labels can be any valid string (including those with spaces, dashes, or other CSS-selector-hostile characters) without breaking selection.
- Panel re-renders from cached state so selection updates don't require a network round trip. The 1 Hz throughput poll will trigger its own re-render via a targeted updater (see §6).

**Selection → click handler closing the loop:**
```javascript
// Toggle selection on repeated click (CONTEXT §Specific Ideas):
// "Click a node → right panel populates. Click the same node again
//  OR click empty canvas → panel clears to placeholder."
inner.selectAll('g.node').each(function(v) {
  d3.select(this).on('click', (event) => {
    event.stopPropagation();
    setSelected(state.selectedStream === v ? null : v);
  });
});

// Click empty SVG background → deselect
d3.select('#topology-svg').on('click', () => setSelected(null));

// Esc key → deselect (only when no input focused)
document.addEventListener('keydown', (event) => {
  if (event.key !== 'Escape') return;
  const ae = document.activeElement;
  if (ae && (ae.tagName === 'INPUT' || ae.tagName === 'TEXTAREA')) return;
  setSelected(null);
});
```

**Confidence: HIGH.** This is the MDN-canonical "keep state in a module variable, propagate to DOM in one place" pattern [CITED: https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Using_closures]. Proven across countless small vanilla-JS UIs.

### Pattern 6: 1 Hz edge label updates without re-layout

**Critical performance question.** This is the single most important finding in this research.

**Question:** dagre-d3's `render()` re-runs the full layout pass which is expensive and visually jarring. Can we update edge labels in-place without re-invoking `render()`?

**Answer:** YES. dagre-d3 0.6.4 renders edge labels as `g.edgeLabel > g.label > text > tspan` where the `tspan` elements contain the actual label text. The `g.edgeLabel` groups are keyed by `"v:w:name"` and persist in the DOM across all subsequent interactions (they are only removed if you call `render()` again with a graph that no longer contains the edge). We can select these `tspan` elements directly via d3 and update their `.text()` without touching dagre-d3 at all.

**Verified from source:** `src/server/ui/vendor/dagre-d3.min.js` — the `createEdgeLabels` function:
```js
function createEdgeLabels(selection, g) {
  var svgEdgeLabels = selection.selectAll("g.edgeLabel")
    .data(g.edges(), function(e) { return util.edgeToId(e); })   // key = "v:w:name"
    .classed("update", true);
  svgEdgeLabels.exit().remove();
  svgEdgeLabels.enter().append("g").classed("edgeLabel", true).style("opacity", 0);
  svgEdgeLabels = selection.selectAll("g.edgeLabel");
  svgEdgeLabels.each(function(e) {
    var root = d3.select(this);
    root.select(".label").remove();                              // <-- rebuilds on render!
    var edge = g.edge(e);
    var label = addLabel(root, g.edge(e), 0, 0).classed("label", true);
    // ...
  });
  // ...
}
```

And `edgeToId`:
```js
function edgeToId(e) { return escapeId(e.v) + ":" + escapeId(e.w) + ":" + escapeId(e.name); }
```

**Important discovery:** `render()` DOES rebuild edge labels — `root.select(".label").remove()` then `addLabel(...)`. So if we called `render()` every second, it would both re-layout AND rebuild labels. We want neither.

**Instead:** never call `render()` after the initial render. Update the `tspan` text directly.

**Canonical snippet:**
```javascript
// Called on every tick of the 1 Hz throughput poll.
// Updates existing edge label text in place. NEVER calls render().
function updateEdgeLabels(throughputData) {
  if (!throughputData || !state.topology) return;

  // Build a lookup: stream name → ewma_5s rate
  const rateByName = Object.create(null);
  for (const s of (throughputData.streams || [])) {
    rateByName[s.name] = s.ewma_5s;
  }

  // For each edge in the topology, find the g.edgeLabel group and update
  // its tspan text. We select directly by walking the g.edgeLabel children
  // (dagre-d3 doesn't emit a data-* attribute we can query).
  //
  // dagre-d3 edge labels are bound to the edge object {v, w, name} via the
  // `edgeToId` key. We use d3.select(this).datum() inside .each() to read
  // the bound datum and match it against our edges[].
  d3.select('#topology-svg').selectAll('g.edgeLabel').each(function() {
    const edgeDatum = d3.select(this).datum();   // {v: "Transactions", w: "Aggregates", name: undefined}
    if (!edgeDatum || !edgeDatum.w) return;
    const rate = rateByName[edgeDatum.w];         // downstream's rate
    const text = formatFps(rate);
    // The label is rendered as: g.edgeLabel > g.label > text > tspan
    // Update the tspan content in-place — textContent-safe via d3 .text()
    d3.select(this).select('g.label').select('text').select('tspan').text(text);
  });
}
```

**Alternative selector if `.datum()` returns something unexpected:** build a lookup from the topology data and iterate the topology edges in parallel with the DOM nodes. But the `.datum()` approach is cleaner because it keeps the selection and the data colocated.

**Why this doesn't break dagre-d3:** we are updating text content, not geometry. dagre-d3's label `bbox` is computed once at render time and stored on the edge data — it's only used for layout positioning. Updating the tspan text does NOT re-trigger layout, so the edge label may end up slightly off-center if the new text is much wider or narrower than the placeholder. For compact labels like `1.2k/s` (never more than 6 characters), the width variance is < 5px and visually unnoticeable.

**If width variance becomes an issue:** right-pad the placeholder text with non-breaking spaces to reserve a consistent width at layout time, e.g. `'   —/s   '`. Cheap trick, no layout recompute needed.

**Pitfall verification:** Phase 10's REVIEW acknowledgment #1 confirms "Zero innerHTML, outerHTML, insertAdjacentHTML, document.write, eval(, Function(, or d3 .html() calls" — d3's `.text()` is the approved sink. Using `.text()` on the tspan does NOT trigger the WR-03 regression grep.

**Confidence: HIGH.** Verified by source inspection of `createEdgeLabels` in the vendored `dagre-d3.min.js` bundle. The D3 update pattern is load-bearing for dagre-d3's own architecture, so the `g.edgeLabel` persistence property is guaranteed.

### Pattern 6b: 1 Hz polling loop with pause gate

The edge-label updater needs its own polling loop because htmx can't touch the DAG SVG container without destroying it. The canonical pattern:

```javascript
// Start the edge-label updater when the DAG is first rendered.
let edgeUpdateHandle = null;

function startEdgeLabelUpdater() {
  if (edgeUpdateHandle != null) return;
  const tick = async () => {
    if (state.paused) return;                     // pause gate — same flag as htmx containers
    try {
      const res = await fetch('/debug/throughput');
      if (!res.ok) return;                         // silently skip; next tick retries
      const data = await res.json();
      state.throughput = data;                     // cache for panel re-renders
      updateEdgeLabels(data);
      updateLastUpdate();
      // If the drill-in panel is showing a throughput section, re-render that
      // section only (not the whole panel — that would blow away the entity
      // lookup input's current text and selection).
      if (state.selectedStream) renderThroughputSection(
        document.getElementById('drill-in-panel'),
        findNode(state.selectedStream),
        data
      );
    } catch (_) {
      // Swallow network errors — the next tick retries. For user-visible error
      // UX, surface via the footer "Lost connection" banner (§8.7).
    }
  };
  tick();                                           // fire immediately
  edgeUpdateHandle = setInterval(tick, 1000);
}

function stopEdgeLabelUpdater() {
  if (edgeUpdateHandle != null) {
    clearInterval(edgeUpdateHandle);
    edgeUpdateHandle = null;
  }
}

// Pause button handler (shared with htmx containers via data-hx-disable):
function setPaused(paused) {
  state.paused = paused;
  // htmx polling: disable every [hx-trigger*="every"] container
  document.querySelectorAll('[hx-trigger*="every"]').forEach((node) => {
    if (paused) node.setAttribute('data-hx-disable', 'true');
    else node.removeAttribute('data-hx-disable');
  });
  // Vanilla fetch polling: the `state.paused` check at the top of tick()
  // short-circuits the body. The setInterval timer itself keeps firing, but
  // each tick is a no-op until resume. This is cheaper than stop/start
  // and avoids the "did I miss a pause?" race when rapid-toggling.
  // UI update:
  const ctrl = document.querySelector('.poll-control');
  if (ctrl) ctrl.dataset.paused = String(paused);
  const btn = document.getElementById('pause-btn');
  if (btn) btn.textContent = paused ? 'Resume' : 'Pause';
  const label = document.querySelector('.poll-label');
  if (label) label.textContent = paused
    ? 'Paused · last update ' + (state.lastUpdate ? fmtTime(state.lastUpdate) : '—')
    : 'Live · 1 Hz';
  const live = document.getElementById('poll-status');
  if (live) live.textContent = paused ? 'Polling paused' : 'Polling resumed';
}
```

**Why the setInterval keeps running while paused:** stopping and restarting the timer on every pause creates race conditions if the user rapidly toggles. Keeping the interval alive with a flag check is O(1) cheaper per toggle and simpler to reason about. The `state.paused` guard runs in < 1 µs per tick, so the idle cost while paused is negligible.

**Confidence: HIGH.**

### Pattern 7: Entity lookup result filtering (client-side)

**Question:** Given a `/debug/key/{key}` response like `{key, computed_features: {"Transactions.tx_count_1h": 42, "Logins.login_count_1h": 3}}`, how to filter to just the selected stream?

**Answer:** The existing endpoint returns `computed_features` as a flat map with feature keys that may OR may NOT be prefixed by stream name. [VERIFIED: `src/server/http.rs:280-293` and `src/server/ui/app.js:327`] Phase 10's `app.js:327-376` already has the "if feature name contains `.` split on it, use the prefix as the source" convention. Phase 10.1 uses the same logic but additionally filters out features whose source doesn't match the selected stream.

**Canonical snippet:**
```javascript
async function lookupEntityForSelectedStream(rawKey) {
  const panel = document.getElementById('entity-result');
  if (!panel) return;
  clear(panel);

  const key = (rawKey || '').trim();
  const selectedStream = state.selectedStream;
  if (!key) {
    panel.appendChild(el('p', {
      class: 'entity-idle',
      text: 'Enter a key above to inspect its features.'
    }));
    return;
  }
  if (!selectedStream) return;  // should not happen — panel is only visible with selection

  // Call the existing backend endpoint. Always URL-encode the key because it
  // is user-supplied and may contain characters that break path routing.
  const url = '/debug/key/' + encodeURIComponent(key);
  let res;
  try {
    res = await fetch(url);
  } catch (e) {
    // Network error (offline, server down)
    panel.appendChild(el('div', { class: 'error-state' }, [
      el('h3', { text: 'Lookup failed' }),
      el('p', { text: 'Network error: ' + String(e.message || e) }),
    ]));
    return;
  }

  // 404: key not found at all
  if (res.status === 404) {
    panel.appendChild(el('div', { class: 'empty-state' }, [
      el('h3', { text: 'No features for "' + key + '"' }),
      el('p', { text: 'This key has not received any events recently, or has been evicted by TTL.' }),
    ]));
    return;
  }
  // Other 4xx/5xx
  if (!res.ok) {
    panel.appendChild(el('div', { class: 'error-state' }, [
      el('h3', { text: 'Lookup failed' }),
      el('p', { text: 'Server returned HTTP ' + res.status }),
    ]));
    return;
  }

  let data;
  try { data = await res.json(); } catch (e) {
    panel.appendChild(el('div', { class: 'error-state' }, [
      el('h3', { text: 'Lookup failed' }),
      el('p', { text: 'Invalid JSON response' }),
    ]));
    return;
  }

  // Filter computed_features to just the selected stream.
  // Phase 10 app.js §renderEntity uses the convention "{source}.{label}" →
  // the feature belongs to the source stream. Features without a dot are
  // considered orphans and are NOT shown (they couldn't be attributed to
  // any stream).
  const computed = data.computed_features || {};
  const filtered = {};
  for (const fullName of Object.keys(computed)) {
    const dot = fullName.indexOf('.');
    if (dot <= 0) continue;                        // orphan: skip
    const source = fullName.substring(0, dot);
    if (source !== selectedStream) continue;       // other stream: skip
    const label = fullName.substring(dot + 1);
    filtered[label] = computed[fullName];
  }

  // "stream has no features for this key" (empty object)
  if (Object.keys(filtered).length === 0) {
    panel.appendChild(el('div', { class: 'empty-state' }, [
      el('h3', { text: 'No features for "' + key + '"' }),
      el('p', { text: 'Stream "' + selectedStream + '" has not computed features for this key.' }),
    ]));
    return;
  }

  // Render the filtered features grid.
  const grid = el('div', { class: 'entity-feature-grid' });
  for (const name of Object.keys(filtered)) {
    const value = filtered[name];
    const cell = el('div', { class: 'entity-feature-cell' }, [
      el('div', { class: 'label', text: name }),                              // textContent
      el('div', { class: 'value', text: formatFeatureValue(value) }),          // textContent
    ]);
    grid.appendChild(cell);
  }
  panel.appendChild(grid);
}
```

**Edge cases explicitly handled:**
1. **Empty input** — placeholder "Enter a key above to inspect its features."
2. **No selection** — shouldn't happen (panel only visible when a stream is selected); early return is defensive.
3. **Network error (fetch throws)** — "Network error: {message}" in an error-state block.
4. **404** — "No features for {key}" empty state.
5. **Other 4xx/5xx** — "Server returned HTTP {status}" error state.
6. **Non-JSON body** — "Invalid JSON response" error state.
7. **Key exists globally but no features for selected stream** — "Stream X has not computed features for this key" empty state (the distinguishing error message CONTEXT §Out-of-scope flagged via "stream-scoped only" discipline).

**XSS safety:** every dynamic string (the typed key, the feature names, the feature values) goes through `textContent` via the `el({text: ...})` helper or `.textContent` directly. Never use string concatenation with `.innerHTML`. The `'"' + key + '"'` concatenation is SAFE because it flows into `textContent`, not `innerHTML`.

**Confidence: HIGH.** Matches Phase 10's existing `renderEntity` pattern with an added filter step.

### Pattern 8: FeatureDef enum → JSON serialization (the `operators` field)

**Question:** The backend change adds `operators: [{name, op, window, where, filter}]` to `/debug/topology` nodes. How to serialize this from the engine's `FeatureDef` enum?

**Critical finding:** **Don't convert the `FeatureDef` enum.** The engine already stores the user's original register JSON in `PipelineEngine::raw_register_jsons` and exposes it via `get_raw_register_json(name)`. That JSON contains the exact shape Phase 10.1 needs — `{name, type, field, window, where, expr, optional, ...}` — with the original **string** `where` clause (not the parsed `Expr` AST) and the original user-facing type names ("count", "sum", "derive", etc.).

**Verified sources:**
- `src/engine/pipeline.rs:205` — `raw_register_jsons: AHashMap<String, serde_json::Value>`
- `src/engine/pipeline.rs:858-866` — `store_raw_register_json` (write) and `get_raw_register_json` (read)
- `src/main.rs:119`, `src/server/tcp.rs:378,389`, `src/server/http.rs:160` — every register path populates `raw_register_jsons`
- `src/server/protocol.rs:262-285` — `FeatureDefRequest` DTO shape: `{name, type, field?, window?, bucket?, expr?, optional?, where?, on?, target?, backfill?}`

**Canonical snippet (Rust, inside `debug_topology` handler in `src/server/http.rs`):**
```rust
// EXTEND existing debug_topology handler (http.rs:307).
// Current shape (Phase 10):
//   nodes[] = [{name, kind, key_field, features: [<name>, ...], depends_on: [<dep>, ...]}]
// Phase 10.1 adds:
//   nodes[i].operators: [{name, op, window?, where?, field?, expr?, bucket?, optional?, backfill?}]

async fn debug_topology(State(state): State<SharedState>) -> Json<serde_json::Value> {
    let app = state.lock().unwrap_or_else(|e| e.into_inner());

    let mut nodes: Vec<serde_json::Value> = Vec::new();
    let mut edges: Vec<serde_json::Value> = Vec::new();

    for s in app.engine.list_streams() {
        let feature_names: Vec<&str> = s.features.iter().map(|(n, _)| n.as_str()).collect();

        // NEW: look up the raw register JSON for this stream and pull out the
        // features array. This gives us the original user-supplied strings for
        // window, where, field, and expr — no AST → string conversion needed.
        let operators: Vec<serde_json::Value> = app
            .engine
            .get_raw_register_json(&s.name)
            .and_then(|raw| raw.get("features"))
            .and_then(|f| f.as_array())
            .map(|arr| {
                arr.iter()
                    .map(|feat| {
                        // Project to the canonical Phase 10.1 shape.
                        // `type` becomes `op` in the output for readability.
                        // Every field is optional — we emit only what's present.
                        let mut out = serde_json::Map::new();
                        if let Some(n) = feat.get("name") { out.insert("name".into(), n.clone()); }
                        if let Some(t) = feat.get("type") { out.insert("op".into(),   t.clone()); }
                        if let Some(w) = feat.get("window") { out.insert("window".into(), w.clone()); }
                        if let Some(b) = feat.get("bucket") { out.insert("bucket".into(), b.clone()); }
                        if let Some(f) = feat.get("field")  { out.insert("field".into(),  f.clone()); }
                        if let Some(wh) = feat.get("where") { out.insert("where".into(),  wh.clone()); }
                        if let Some(e) = feat.get("expr")   { out.insert("expr".into(),   e.clone()); }
                        if let Some(o) = feat.get("optional") { out.insert("optional".into(), o.clone()); }
                        if let Some(bf) = feat.get("backfill") { out.insert("backfill".into(), bf.clone()); }
                        // Lookup-only: on, target
                        if let Some(on) = feat.get("on") { out.insert("on".into(), on.clone()); }
                        if let Some(tg) = feat.get("target") { out.insert("target".into(), tg.clone()); }
                        serde_json::Value::Object(out)
                    })
                    .collect()
            })
            .unwrap_or_default();  // empty array if no raw JSON (e.g., snapshot-restored state pre-raw_register)

        // Stream-level filter: also pull from raw JSON
        let filter = app
            .engine
            .get_raw_register_json(&s.name)
            .and_then(|raw| raw.get("filter"))
            .cloned()
            .unwrap_or(serde_json::Value::Null);

        nodes.push(serde_json::json!({
            "name": s.name,
            "kind": "stream",
            "key_field": s.key_field,
            "features": feature_names,                     // UNCHANGED — Phase 10 compat
            "operators": operators,                         // NEW — Phase 10.1 additive
            "filter": filter,                               // NEW — optional, stream-level
            "depends_on": s.depends_on.clone().unwrap_or_default(),
        }));
        for dep in s.depends_on.clone().unwrap_or_default() {
            edges.push(serde_json::json!({
                "from": dep,
                "to": s.name,
                "kind": "cascade",
            }));
        }
    }

    // Views: same treatment
    for v in app.engine.list_views() {
        let feature_names: Vec<&str> = v.features.iter().map(|(n, _)| n.as_str()).collect();

        let operators: Vec<serde_json::Value> = app
            .engine
            .get_raw_register_json(&v.name)
            .and_then(|raw| raw.get("features"))
            .and_then(|f| f.as_array())
            .map(|arr| arr.iter().map(|feat| {
                // Views have derive or lookup features; same projection.
                let mut out = serde_json::Map::new();
                if let Some(n)  = feat.get("name")   { out.insert("name".into(),  n.clone()); }
                if let Some(t)  = feat.get("type")   { out.insert("op".into(),    t.clone()); }
                if let Some(on) = feat.get("on")     { out.insert("on".into(),    on.clone()); }
                if let Some(tg) = feat.get("target") { out.insert("target".into(), tg.clone()); }
                if let Some(e)  = feat.get("expr")   { out.insert("expr".into(),  e.clone()); }
                serde_json::Value::Object(out)
            }).collect())
            .unwrap_or_default();

        nodes.push(serde_json::json!({
            "name": v.name,
            "kind": "view",
            "key_field": v.key_field,
            "features": feature_names,     // UNCHANGED
            "operators": operators,          // NEW
            "depends_on": Vec::<String>::new(),
        }));
        // lookup edges (unchanged)
        for (_fname, fdef) in &v.features {
            if let crate::engine::pipeline::ViewFeatureDef::Lookup { target_stream, .. } = fdef {
                edges.push(serde_json::json!({
                    "from": target_stream,
                    "to": v.name,
                    "kind": "lookup",
                }));
            }
        }
    }

    let topo_order: Vec<String> = app.engine.get_topo_order().to_vec();
    Json(serde_json::json!({
        "nodes": nodes,
        "edges": edges,
        "topo_order": topo_order,
    }))
}
```

**Frontend rendering logic for the State section** of the drill-in panel:
```javascript
function renderStateSection(panel, node) {
  const section = el('section', { class: 'drill-in-section drill-in-state' });
  section.appendChild(el('h2', { class: 'section-title', text: 'State' }));

  const operators = node.operators || [];
  if (operators.length === 0) {
    section.appendChild(el('p', { class: 'muted', text: 'No operators registered.' }));
    panel.appendChild(section);
    return;
  }

  const list = el('ul', { class: 'operator-list' });
  for (const op of operators) {
    // Compose a human-readable summary: e.g. "count · 30m · where status == 'failed'"
    const parts = [];
    if (op.op) parts.push(op.op);
    if (op.window) parts.push(op.window);
    if (op.field) parts.push('field: ' + op.field);
    if (op.where) parts.push('where: ' + op.where);
    if (op.expr) parts.push('expr: ' + op.expr);
    const summary = parts.join(' · ');

    const item = el('li', { class: 'operator-item' }, [
      el('span', { class: 'operator-name', text: op.name || '' }),
      el('span', { class: 'operator-summary', text: summary }),
    ]);
    list.appendChild(item);
  }
  section.appendChild(list);
  panel.appendChild(section);
}
```

**Why pass-through is better than enum walking:**
1. **Zero AST serialization required** — `Expr` has no `Display` or `Serialize` impl and building one is a new piece of code with its own bug surface.
2. **Original user strings preserved** — the raw JSON has the exact `"status == 'failed'"` text the user wrote, not a reconstructed normalization.
3. **Snapshot-compatibility is a known engine concern** — `raw_register_jsons` is already persisted to snapshots via `store_raw_register_json`. Phase 10.1 inherits whatever correctness guarantees the engine already provides.
4. **Zero coupling to FeatureDef internals** — if a future phase adds a new FeatureDef variant, the pass-through automatically emits its fields from the raw JSON. No `match` statement to update.

**Snapshot edge case:** If a snapshot-restored stream predates the `raw_register_jsons` store (i.e., the snapshot was taken before the engine started recording them), `get_raw_register_json` returns `None` and the handler emits `operators: []`. This is acceptable — the user sees "No operators registered" in the State section, which is unambiguous and not visually broken. The planner should surface this in a VERIFICATION task if there's concern about legacy snapshots.

**Confidence: HIGH.** Source-verified.

### Anti-Patterns to Avoid

- **DO NOT call `render(inner, g)` on every tick.** Full layout re-pass. Flashing. Stalls. Call it exactly ONCE on initial topology load.
- **DO NOT swap the DAG SVG container via htmx.** Any `hx-swap` that touches `#topology-svg` or its parent will destroy the rendered graph. Keep htmx OUT of the DAG canvas. Edge label updates go through vanilla `fetch` + d3 `.text()`.
- **DO NOT use `.innerHTML`, `.outerHTML`, `insertAdjacentHTML`, `document.write`, `eval(`, `new Function(`, `labelType: 'html'`.** Phase 10 WR-03 regression test fails on any match.
- **DO NOT set `labelType: 'html'` on dagre-d3** — even in a comment. The regression grep matches substrings anywhere.
- **DO NOT hand-walk the `FeatureDef` enum** to build the `operators` field. Use `raw_register_jsons` — the shape is already right.
- **DO NOT forget `encodeURIComponent` on the entity key.** User keys can contain `/`, `?`, `#`, or UTF-8 bytes that break the URL.
- **DO NOT use fixed-width columns (`1fr`) on the DAG grid track** — always `minmax(0, 1fr)` to allow content-shrink.
- **DO NOT `display: none` the DAG canvas on empty state** — the Phase 10 UI-REVIEW flagged this at 21/24 as a focal-point bug. Keep the canvas visible and overlay the empty-state centered content.
- **DO NOT add a background timer that fires even while paused.** The `state.paused` gate at the top of `tick()` is the clean approach — the `setInterval` keeps running but the body is a no-op.
- **DO NOT hold the `AppState` mutex across `.await` in the extended `debug_topology` handler.** Phase 10 RESEARCH §Pitfall 3 — currently the handler uses `let app = state.lock()...; /* sync read */ Json(...)`. The `operators` field addition is a sync `HashMap` lookup; no `.await` should be introduced.
- **DO NOT touch the snapshot format, the operator state, the state store, or any file under `src/engine/` or `src/state/`.** Phase 10.1 is frontend + one http.rs handler edit. Everything else is out of scope.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| DAG layout + SVG rendering | Manual node positioning, path routing, arrow drawing | `dagre-d3` (already vendored) | Graph layout is the hard part. Already solved and vendored. |
| SVG text rendering | Custom `<text>` elements with manual positioning | dagre-d3's default `addTextLabel` | Already handles baseline, bbox, multi-line, escaping. |
| JSON polling loop | Hand-rolled EventSource / WebSocket | htmx `hx-trigger="every 1s"` for non-DAG panels + vanilla `fetch + setInterval` for the DAG edge updater | htmx handles the pause gate via `data-hx-disable`. The edge updater is a 15-line `setInterval`. No library needed. |
| Tab switching | Any tab router | **There is no tab bar** | Phase 10.1 deletes it entirely. |
| Framework state management | Redux, Zustand, Signals | Single `state` object + `setSelected(name)` | ~25 LOC. Fits entirely in app.js. No dependency. |
| HTML escaping | Hand-rolled escape function | `.textContent` or d3 `.text()` | Browser-native, unbypassable by any weird input. |
| Number formatting | A library | `formatFps(n)` helper (§3) | 10-line vanilla JS function. `Intl.NumberFormat` is overkill. |
| Event bus / observer pattern | EventTarget / BroadcastChannel | Direct call from `setSelected` to renderers | Four renderers, one caller. Zero complexity budget. |
| Compact throughput label positioning | Hand-compute edge midpoint, angle, label transform | dagre-d3's native edge label (`labelpos: 'c'`) | dagre-d3 already computes midpoint + path-following label transforms. Free. |
| FeatureDef → JSON conversion | Pattern match on every enum variant | Pass through `raw_register_jsons` | Shape is already right. Zero risk. |

**Key insight:** Phase 10.1 has an unusually thin complexity budget — the backend change is ~15 LOC of Rust, and the frontend rewrite's load-bearing logic (selection, edge updates, layout) adds up to ~150 LOC of vanilla JS. The whole phase succeeds or fails on **discipline**, not on library choices. Every "should I build this?" question answers "no — use what's already there."

## Common Pitfalls

### Pitfall 1: Calling `render()` on every throughput tick

**What goes wrong:** The DAG flashes visibly every second. Nodes snap to new positions if label widths change. Click handlers need to be re-bound on every render. User reports "the graph jumps" or "nothing is clickable after a second."

**Why it happens:** dagre-d3's `render()` re-runs layout (`dagre.layout(g)`) AND rebuilds every label sub-element. The layout phase computes new node positions and edge routes; the label rebuild phase calls `addLabel` on every edge which detaches and reappends `<text>` children.

**How to avoid:** Call `render()` exactly ONCE after the initial `/debug/topology` load. For all subsequent edge label updates, select the existing `g.edgeLabel > g.label > text > tspan` elements directly and call `.text(newValue)` without invoking dagre-d3.

**Warning signs:** The DAG "flashes" at 1 Hz. New clicks stop working. Node positions drift after several ticks.

---

### Pitfall 2: htmx `hx-get` swaps destroying the DAG SVG

**What goes wrong:** Adding `hx-get="/debug/throughput"` with `hx-swap="innerHTML"` to the container wrapping the DAG SVG makes htmx nuke the SVG and replace it with the fetch response body, which is JSON — not HTML. The DAG disappears after one tick and the page shows `{"streams":[...]}` as text.

**Why it happens:** htmx's default `hx-swap` is `innerHTML`. Any element with an `hx-get` attribute and a trigger will have its innerHTML replaced with the response body.

**How to avoid:** htmx must NEVER touch the DAG canvas. Phase 10.1's `/debug/throughput` poll goes through a vanilla `fetch + setInterval` loop, NOT htmx. If any htmx container is added for any other reason (e.g., an htmx-polled entity lookup form), scope it to an element that is outside the DAG's subtree.

**Warning signs:** DAG disappears after first tick. Page shows raw JSON. Entity lookup works but DAG broken.

---

### Pitfall 3: Pause coverage incomplete — htmx paused but fetch loop still runs

**What goes wrong:** User clicks Pause. htmx polling halts (Phase 10's pause mechanism works). But the vanilla-JS edge-label updater keeps firing every second because it has its own `setInterval` that isn't gated by `data-hx-disable`.

**Why it happens:** Phase 10's pause logic was `document.querySelectorAll('[hx-trigger*="every"]').forEach(...)` which only affects htmx-polled elements. Phase 10.1 introduces a new non-htmx poller for edge labels that must be gated separately.

**How to avoid:** Introduce a single shared `state.paused` flag. The `setPaused(paused)` function both toggles `data-hx-disable` on htmx containers AND flips the flag. The `setInterval` callback checks `state.paused` at the top of every tick and returns early if paused. See §6b canonical snippet.

**Warning signs:** After clicking Pause, the footer timestamp still updates every second. Edge labels still change. htmx-polled panels (if any survive) freeze correctly.

---

### Pitfall 4: Forgetting to `stopPropagation` on node click

**What goes wrong:** Clicking a node both selects it AND deselects it (because the click bubbles to the SVG background handler which calls `setSelected(null)`). User perceives the panel flashing on and then off immediately.

**Why it happens:** DOM events bubble by default. The click handler on `g.node` fires first, calls `setSelected(v)`, then the event bubbles to `#topology-svg` which has its own click handler that calls `setSelected(null)`.

**How to avoid:** Call `event.stopPropagation()` inside the node click handler. The SVG background click handler should only fire for clicks on empty SVG space.

```javascript
sel.on('click', (event) => {
  event.stopPropagation();               // <-- critical
  setSelected(state.selectedStream === v ? null : v);
});
```

**Warning signs:** Clicking a node selects and then immediately deselects it. The drill-in panel briefly flashes.

---

### Pitfall 5: `display: none` on DAG canvas when empty

**What goes wrong:** On an empty state (no pipelines registered), setting `canvas.style.display = 'none'` collapses the 640px focal area to zero height. The split-view grid then redistributes space and the right panel grows to full width. Visual weight of the page disappears. Flagged at 21/24 in Phase 10's UI-REVIEW.

**Why it happens:** Phase 10's empty-state handling in `app.js:212` uses `style.display = 'none'`. Phase 10.1 must explicitly NOT do this.

**How to avoid:** Keep `.dag-canvas` visible and render the empty-state block as an absolutely-positioned overlay centered inside the canvas. Example:

```css
.dag-canvas {
  position: relative;   /* containing block for absolute children */
  /* ... */
}
.dag-canvas .empty-state {
  position: absolute;
  inset: 0;
  display: flex;
  flex-direction: column;
  align-items: center;
  justify-content: center;
  gap: var(--space-4);
  padding: var(--space-5);
  text-align: center;
  color: var(--text-tertiary);
}
```

And in JS:
```javascript

  const canvas = document.querySelector('.dag-canvas');
  canvas.appendChild(el('div', { class: 'empty-state' }, [
```

### Pitfall 6: Panel re-render clobbering entity input text

**What goes wrong:** User types a key into the entity lookup input inside the drill-in panel. One second later the 1 Hz throughput poll triggers a panel re-render. The `clear(panel)` call destroys the input element (including its text) and rebuilds the panel from scratch. User's typed key vanishes mid-typing.

**Why it happens:** If the panel renderer naively rebuilds the entire panel on every data update, inputs lose their state.

**How to avoid:** Two strategies:
1. **Granular re-renders** — when throughput or memory updates arrive, only re-render the specific section that depends on them (Throughput, Memory). The Entity Lookup section is rendered once on selection and never touched by data ticks.
2. **Panel re-render only on selection change** — selection change is a deliberate user action that justifies rebuilding the entity section. Data ticks only update cached `state.throughput`, `state.memory`, and targeted section re-renders.

**Recommendation:** option 2 with an exception — introduce a `renderThroughputSection(panel, node, throughput)` function that can be called in isolation by the edge-label updater. See §6b canonical snippet.

**Warning signs:** Typing in the entity input deletes characters between keystrokes. Focus leaves the input unexpectedly.

### Pitfall 7: `raw_register_jsons` returns `None` for some streams

**What goes wrong:** A snapshot-restored stream (from a snapshot taken before the engine started recording raw register JSONs) has no entry in `raw_register_jsons`. The `get_raw_register_json(name)` call returns `None`, and the new `operators` field is emitted as an empty array. The drill-in panel's State section shows "No operators registered" even though the stream is functional.

**Why it happens:** `raw_register_jsons` is populated on every register path (main.rs:119, tcp.rs:378/389, http.rs:160), but it might not be replayed on snapshot restore. (Planner should verify this in a Wave 0 task by reading the snapshot restore code in `src/state/snapshot.rs`.)

**How to avoid:** Two acceptable strategies:
1. **Tolerate gracefully** — emit `operators: []` and let the drill-in panel show "No operators registered" with a small footnote "(snapshot-restored, details unavailable)". Cheap, honest, matches user expectations.
2. **Fallback to FeatureDef walking** — if `raw_register_jsons` is empty for a stream, fall back to walking `StreamDefinition.features` and emitting a minimal shape: `{name, op: "(unknown)", note: "snapshot restored"}`.

**Recommendation:** Strategy 1 for Phase 10.1. If it becomes a real user complaint, upgrade to Strategy 2 in a follow-up. The backend contract allows `operators: []` — nothing asserts a specific length.

**Warning signs:** Drill-in State section empty for existing streams but non-empty for streams registered in the current session.

### Pitfall 8: Tab bar ghost markup left in index.html

**What goes wrong:** The rewrite keeps some leftover tab-bar CSS classes, IDs, or ARIA attributes (`role="tablist"`, `role="tab"`, `.tab-bar`, `#tab-topology`) because they were copy-pasted from Phase 10's `index.html` and the executor didn't delete every reference. A regression test (new for Phase 10.1) that asserts absence of these identifiers fails.

**Why it happens:** Wholesale rewrites often retain unused CSS classes for safety. The new regression test catches them.

**How to avoid:** Add a Phase 10.1 regression test that greps the served `/` response for the tab-bar substrings listed in CONTEXT §Decisions and fails if any are present. Examples:
```rust
let forbidden_tab_markup = &[
    ".tab-bar", "#tab-topology", "#tab-streams", "#tab-entity", "#tab-memory",
    "role=\"tablist\"", "role=\"tab\"",
];
for s in forbidden_tab_markup {
    assert!(!body.contains(s), "legacy tab-bar markup {:?} found in index.html", s);
}
```

**Warning signs:** Dead CSS rules targeting `.tab`. Unused `<a role="tab">` markup. Screen reader announces "tab list" when navigating.

### Pitfall 9: Compact throughput label width variance causing layout jitter

**What goes wrong:** The edge label text goes from `0/s` → `1.2k/s` → `450/s` → `12/s`. Each value has a different rendered width. dagre-d3's edge label positioning was computed at render time for the placeholder text; as text changes, the label appears slightly off-center, and may visually drift.

**Why it happens:** dagre-d3 computes the label bbox once, at the end of `createEdgeLabels`, and positions the label group using that bbox. Subsequent `.text()` updates do NOT re-center the label.

**How to avoid:** Right-pad the placeholder to a consistent width so the initial bbox is generous enough to accommodate any reasonable value. Example placeholder: `'      —/s      '` (14 chars). Since all realistic values (`0/s` through `999M/s`) are shorter, the label group is always wide enough and off-center drift is bounded to < 3 px.

Alternative: accept the minor drift. For a single-line, 6-character-max label, the drift is < 5 px and invisible to most users.

**Recommendation:** right-pad to 10 chars. Acceptable tradeoff — slightly wider edge labels (a few extra px of padding around the midpoint) in exchange for no drift.

**Warning signs:** Edge labels appear to wobble left-right as numbers change magnitude.

### Example 1: Full DAG render flow (one-time setup on topology load)

```javascript
async function loadTopology() {
  const res = await fetch('/debug/topology');
  if (!res.ok) {
    showTopologyError('HTTP ' + res.status);
    return;
  }
  const data = await res.json();
  state.topology = data;

  const svg = d3.select('#topology-svg');
  svg.selectAll('*').remove();                  // clean slate

  if (!data.nodes || data.nodes.length === 0) {
    drawEmptyState();
    return;
  }

  const g = new dagreD3.graphlib.Graph()
    .setGraph({ rankdir: 'LR', marginx: 24, marginy: 24, nodesep: 24, ranksep: 48 })
    .setDefaultEdgeLabel(() => ({}));

  for (const node of data.nodes) {
    g.setNode(node.name, {
      label: node.name,
      class: 'node ' + (node.kind === 'view' ? 'view' : 'stream'),
      rx: 12, ry: 12, paddingX: 16, paddingY: 12,
    });
  }
  for (const edge of (data.edges || [])) {
    g.setEdge(edge.from, edge.to, {
      label: '   —/s    ',              // right-padded placeholder
      labelpos: 'c',
      arrowhead: 'vee',
      curve: d3.curveBasis,
    });
  }

  const inner = svg.append('g').attr('id', 'dag-root');
  const render = new dagreD3.render();
  render(inner, g);                              // <-- ONCE

  // Fit-to-container transform (same as Phase 10)
  const bbox = inner.node().getBBox();
  const svgEl = document.getElementById('topology-svg');
  const svgW = svgEl.clientWidth || 800;
  const svgH = svgEl.clientHeight || 480;
  const scale = Math.min(
    (svgW - 24) / Math.max(bbox.width, 1),
    (svgH - 24) / Math.max(bbox.height, 1),
    1
  );
  const tx = (svgW - bbox.width * scale) / 2 - bbox.x * scale;
  const ty = (svgH - bbox.height * scale) / 2 - bbox.y * scale;
  inner.attr('transform', 'translate(' + tx + ',' + ty + ') scale(' + scale + ')');

  // Attach click handlers. This runs once per topology load; if topology is
  // re-fetched (e.g., on REGISTER), the whole function re-runs and rebinds.
  inner.selectAll('g.node').each(function(v) {
    const sel = d3.select(this);
    sel.style('cursor', 'pointer')
       .attr('tabindex', 0)
       .attr('role', 'button')
       .attr('aria-label', v)
       .on('click', (event) => {
         event.stopPropagation();
         setSelected(state.selectedStream === v ? null : v);
       })
       .on('keydown', (event) => {
         if (event.key === 'Enter' || event.key === ' ') {
           event.preventDefault();
           setSelected(v);
         }
       });
  });

  // Background deselect
  svg.on('click', () => setSelected(null));

  // Start the edge label updater now that the DAG is rendered
  startEdgeLabelUpdater();
}
```

## Open Questions

1. **Does `raw_register_jsons` survive snapshot restore?**
   - What we know: `raw_register_jsons` is populated on every live register path. Not verified whether it's part of the v6 snapshot format.
   - What's unclear: what `debug_topology` sees after a snapshot restore if raw JSONs weren't persisted.
   - Recommendation: planner adds a Wave 0 task to grep `src/state/snapshot.rs` for `raw_register_jsons`. If absent, tolerate gracefully (Pitfall 7 Strategy 1) and note in VERIFICATION.md that snapshot-restored streams show an empty State section. If present, everything works out of the box.

2. **Is the `filter` field worth including in the node payload?**
   - What we know: CONTEXT shows `"filter": null` in the sample JSON but CONTEXT §Decisions mentions it only in passing.
   - What's unclear: whether the drill-in panel's State section should render the stream-level filter alongside the per-operator where clauses.
   - Recommendation: include it as an additive field (cost: ~5 LOC), render it in the State section as "Filter: {expr}" above the operator list. If the planner prefers to defer, drop from the backend payload and re-add in a future phase. Cheap either way.

3. **Should the drill-in panel animate in (slide from right) or be always visible?**
   - What we know: CONTEXT calls it out as Claude's Discretion and recommends always visible to avoid layout shift.
   - What's unclear: whether users perceive the static panel as "there's something missing" when no node is selected vs a slide-in that feels responsive.
   - Recommendation: always visible with a placeholder ("Select a stream to see details"). Avoids layout shift, avoids animation complexity, avoids interaction with `prefers-reduced-motion`. Aligns with CONTEXT's recommendation.

4. **What happens when the selected stream is removed from the topology?**
   - Scenario: user selects "Transactions", then `/debug/topology` re-fetches (e.g., on register) and "Transactions" is no longer present.
   - What we know: selection state is held in JS only; the backend never knows about it.
   - Recommendation: after `loadTopology`, check `state.selectedStream` against the new `data.nodes`; if it's no longer present, call `setSelected(null)`. Trivial guard, ~3 LOC.

5. **Should the entity lookup input persist its value across selection changes?**
   - Scenario: user types `u_demo`, clicks Transactions, sees the filtered result. Then clicks Aggregates. Should the input still say `u_demo` and re-query for the new selection, or clear?
   - Recommendation: clear on selection change. The entity lookup is scoped to the selected stream — carrying the value across selections implies cross-stream comparison which CONTEXT explicitly excludes ("no cross-stream quick-jump chips"). Clean slate matches the mental model.

6. **What's the tabindex strategy for DAG nodes?**
   - What we know: Phase 10's nodes were not keyboard-focusable; Phase 10.1 should make them focusable (§Pattern 1 sets `tabindex=0`).
   - What's unclear: tab ordering — dagre-d3 appends nodes in graph iteration order, which matches registration order; this is a reasonable default.
   - Recommendation: `tabindex=0` on every node, let the DOM order drive tab sequence. No manual tabindex counter.
