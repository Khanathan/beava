# Phase 10.1: Code Review Report

**Reviewed:** 2026-04-10
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 10.1 delivers the interactive Debug UI redesign across three plans: an
additive `operators` field on `/debug/topology` node payloads, a rewritten
split-view HTML/CSS shell, and a ground-up vanilla-JS behavior module with
render-once dagre-d3, 1 Hz edge label updates, and a stream-scoped entity
lookup.

All Phase 10.1 critical invariants from the phase context hold up under
review:

- **Lock discipline (http.rs):** No `.await` while the `AppState` mutex is
  held in `debug_topology`, `debug_throughput`, `debug_memory`, or `debug_key`.
  `trigger_snapshot` correctly drops the guard (via scoped block) before the
  `tokio::task::spawn_blocking` await.
- **Additive topology contract:** `features: Vec<&str>` is still emitted
  alongside the new `operators` array on both stream and view nodes (http.rs
  lines 357, 405). Phase 10 tests that read `features` remain compatible.
- **Forbidden XSS sinks in app.js:** Grep against the file confirms zero hits
  for `.innerHTML`, `.outerHTML`, `insertAdjacentHTML`, `document.write`,
  `eval(`, `new Function(`, `labelType: 'html'`, or `labelType:"html"`. Every
  server- or user-supplied string flows through the `el({text:...})` helper
  (textContent) or d3 `.text()`. The regression test at
  `tests/test_debug_ui.rs:1195` codifies this.
- **dagre-d3 render-once:** `render(inner, g)` is called exactly once per
  topology load (app.js:528). The 1 Hz and 2 Hz loops only mutate existing
  `<tspan>` nodes via d3 `.text()` or re-render specific drill-in sections,
  never the SVG graph.
- **Pause gate consistency:** Both `tickEdgeUpdater` (app.js:652, 666) and
  `tickMemory` (app.js:697, 711) short-circuit on `state.paused` both before
  fetch and after the await, guarding against late pauses.
- **Entity lookup stream-scoped filter:** Client-side filter is present
  (app.js:834-842) — **but it is broken, see CR-01 below.**
- **CSS tokens inherited verbatim:** Design tokens in `:root` match the Phase
  10 UI-SPEC §13.1 layout (6 spacing slots with no 12px, 4 type sizes with no
  16px body, documented dark-mode palette, etc.). No token values appear to be
  altered.

The single critical finding is a logic bug in the entity lookup filter that
makes the feature non-functional in its shipped state — every lookup will
report "No features for {key}" even for valid entities with populated
features. There are also two warnings around granular re-render policy and
dead-fallback-code, and four info notes on minor issues.

## Critical Issues

### CR-01: Entity lookup filter never matches — stream filter assumes dotted feature names

**File:** `src/server/ui/app.js:834-842`

**Issue:** The entity lookup filter parses feature names from
`computed_features` as `Stream.feature_name` (dotted notation), but the backend
`/debug/key/{key}` endpoint returns **flat** feature names. The authoritative
implementation is `StateStore::get_all_features` at `src/state/store.rs:150`,
which iterates every stream's operators and inserts each into the result map
using only the operator name (`features.insert(name.clone(), op.read(now))`),
with no `StreamName.` prefix. So for an entity with two streams like
`Transactions(tx_count_1h)` and `Logins(login_count_1h)`, the server response
is:

```json
{
  "computed_features": {
    "tx_count_1h": 5,
    "login_count_1h": 2
  }
}
```

The filter loop then does:

```js
for (const fullName of Object.keys(computed)) {
  const dot = fullName.indexOf('.');
  if (dot <= 0) continue;   // tx_count_1h  ->  indexOf('.') === -1  ->  -1 <= 0 -> skip
  const source = fullName.substring(0, dot);
  if (source !== selectedStream) continue;
  ...
}
```

Because none of the flat feature names contain a `.`, every iteration hits
the `continue` at `dot <= 0` and **every feature is filtered out**. The user
always sees the "No features for '{key}'" empty state, regardless of whether
the entity has real data. This renders one of the five documented drill-in
panel sections non-functional.

The existing test `entity_lookup_reuses_existing_endpoint`
(`tests/test_debug_ui.rs:745`) only asserts that the endpoint returns a
non-empty `computed_features` object — it does not exercise the app.js
client-side filter, which is why this regression slipped through.

**Fix:** The server does not currently disambiguate which stream a feature
belongs to in `computed_features`, so the filter cannot rely on dotted names.
Two options:

**Option A (frontend-only, smaller diff):** Use the topology node's
`features` array (which *is* per-stream, see `http.rs:317`) to build an
allow-list of feature names for the selected stream, then match against the
flat keys:

```js
// Prefer computed_features, fall back to features (Phase 10 compat).
const computed = (data && data.computed_features) || {};
const node = findNode(selectedStream);
const allowed = new Set((node && node.features) || []);
const filtered = {};
for (const name of Object.keys(computed)) {
  if (!allowed.has(name)) continue;
  filtered[name] = computed[name];
}
```

Caveat: if two streams ever share an operator name, this collapses them —
acceptable in v1 since the engine already dedupes operator names per
registration.

**Option B (backend change):** Extend `StateStore::get_all_features` (or add
a new endpoint variant) to return `{stream_name, feature_name, value}`
tuples, then reconstruct the prefix client-side. Larger diff, touches a
widely-used hot path — defer unless A proves insufficient.

Either way, add a regression test that registers a two-stream pipeline,
pushes an event, calls the entity lookup path from JS (via the
`window.app.lookupEntityForSelectedStream` debug surface exposed at line 935),
and asserts the drill-in panel renders non-empty feature cells for the
selected stream.

## Warnings

### WR-01: Memory tick full-panel re-render wipes user-visible entity lookup results

**File:** `src/server/ui/app.js:717-727`

**Issue:** The 2 Hz memory loop `tickMemory` re-renders the **entire**
drill-in panel whenever a stream is selected and no element inside the panel
has focus:

```js
if (state.selectedStream) {
  const panel = document.getElementById('drill-in-panel');
  const node = findNode(state.selectedStream);
  if (panel && node && node.kind !== 'view') {
    const active = document.activeElement;
    const focusInPanel = active && panel.contains(active);
    if (!focusInPanel) {
      renderDrillInPanel(state.selectedStream);   // <-- full rebuild
    }
  }
}
```

`renderDrillInPanel` calls `clear(panel)` and then reconstructs every
section from scratch, including `renderEntityLookupSection`, which
hard-resets the input `value` to `''` (line 416) and the result `<div>` to
the "Enter a key above" idle copy (line 440). That means: every ~2 seconds,
as long as focus is outside the panel, any entity lookup results the user is
*reading* disappear and are replaced with the idle state.

Concrete regression scenario (once CR-01 is fixed):

1. User selects a stream, types `u_demo`, clicks "Look up".
2. Results render and the user moves focus away (button blurs).
3. ~2 s later, `tickMemory` re-renders the whole panel → results are wiped.

Contrast with the 1 Hz throughput tick (`tickEdgeUpdater`, line 672-678),
which correctly uses the granular `renderThroughputSection` that only
replaces the `[data-section="throughput"]` element. The comment at line 671
even calls out Pitfall 6 for this exact reason — the memory tick is
inconsistent with its own stated policy.

**Fix:** Refactor to a granular `renderMemorySection` replacement mirroring
the throughput pattern, and have `tickMemory` call only that:

```js
function renderMemorySectionInPlace(panel, node, memory) {
  const existing = panel.querySelector(
    '.drill-in-section[data-section="memory"]'
  );
  if (!existing) return;   // not yet rendered; initial render will insert it
  const section = el('section', { class: 'drill-in-section', 'data-section': 'memory' });
  // ... same body as renderMemorySection ...
  existing.replaceWith(section);
}

// in tickMemory:
if (state.selectedStream) {
  const panel = document.getElementById('drill-in-panel');
  const node = findNode(state.selectedStream);
  if (panel && node && node.kind !== 'view') {
    renderMemorySectionInPlace(panel, node, data);   // never clobbers entity section
  }
}
```

Drop the `focusInPanel` guard entirely — once the re-render is scoped to
only the Memory section, it can't clobber the entity input or result DOM
regardless of focus.

### WR-02: `renderDrillInPanel` guards `findNode(null)` correctly, but `renderDrillInPanel` is also reachable from `tickMemory` where `findNode` may return `null` if the topology reloads mid-tick

**File:** `src/server/ui/app.js:719-724`

**Issue:** `tickMemory` fetches the node via `findNode(state.selectedStream)`
and only re-renders if the node exists:

```js
const node = findNode(state.selectedStream);
if (panel && node && node.kind !== 'view') {
  ...
  renderDrillInPanel(state.selectedStream);
}
```

But `renderDrillInPanel` itself then re-runs `findNode(name)` internally
(line 205). In the narrow window between the tick's check and the internal
check, `state.topology` cannot change today (topology is loaded exactly
once), so this is currently safe. However, Phase 10.2 or a future "refresh"
affordance would introduce topology reloads, and then `state.topology` could
become a new object whose `nodes` array does not contain the previously
selected stream (e.g. the stream was DELETEd). In that case the second
`findNode` returns `null` and `renderDrillInPanel` would silently reset the
panel to its empty state without telling the user why.

This is not a live bug today, but it is a foot-gun for anyone extending the
UI with refresh capability. Worth noting because CR-01's fix will likely add
a regression test that intentionally swaps topologies.

**Fix:** Have `tickMemory` (and `tickEdgeUpdater`) detect the
selected-stream-vanished case explicitly and call `setSelected(null)` so the
user sees the deliberate "Select a stream" placeholder instead of a
mysteriously-blanked panel:

```js
const node = findNode(state.selectedStream);
if (state.selectedStream && !node) {
  setSelected(null);   // selected stream no longer exists -> explicit reset
  return;
}
```

Apply the same pattern in `tickEdgeUpdater`. Alternatively (and more
defensively), assert in `renderDrillInPanel` that the caller passed a name
that exists in the current topology, and early-return with no-op on mismatch
instead of flipping to the empty state.

## Info

### IN-01: Dead fallback to `data.features` in entity lookup response parsing

**File:** `src/server/ui/app.js:833`

**Issue:** The line
`const computed = (data && (data.computed_features || data.features)) || {};`
falls back to `data.features` if `data.computed_features` is missing, but
the current `/debug/key/{key}` handler (http.rs:283-294) emits **only**
`computed_features`, never `features`. There's no historical code path that
returned `features` either (confirmed via grep in src/). The fallback is
dead.

**Fix:** Drop the fallback for clarity, or — if it was intended as
forward-compat for a future renamed field — leave it and add an inline
comment naming the intended future contract. As-is the uninformed reader
cannot tell if it's defensive code or a missed cleanup.

### IN-02: `/health` endpoint has no `version` field, so `loadHealth` footer version update is effectively dead

**File:** `src/server/http.rs:19-21`, `src/server/ui/app.js:874-876`

**Issue:** The `/health` handler returns `{"status": "ok"}` with no `version`
field. `loadHealth` in app.js checks
`if (v && data && data.version) v.textContent = 'v' + data.version;`, which
is guarded by `data.version`, so it short-circuits and never writes. The
footer span keeps its initial `v—` placeholder from index.html forever. No
bug (the guard is correct), but the feature is dead and the intent is
ambiguous.

**Fix:** Either (a) wire a `version` field into the `/health` response so the
footer reflects the real build version, or (b) remove the version-display
branch in `loadHealth`. The frontend comment at line 879 calls the footer
"decorative", suggesting (b) is fine.

### IN-03: `handle_sync_command`'s `Metrics` import in tests is unused but the test file as a whole is fine

**File:** `tests/test_debug_ui.rs` (general)

**Issue:** None blocking — just noting that the test file exercises shell
structure via substring grepping (`split_view_shell_has_no_tab_bar`,
`split_view_shell_has_dag_canvas_and_drill_in_panel`) and XSS sink
enforcement (`app_js_has_no_innerhtml_or_eval_sinks`). These are
appropriately scoped regression guards and do not suffer from the known
"missing assertion" anti-pattern. Per the review rules I am *not* flagging
test files for style, but it's worth recording that the test coverage around
entity lookup (`entity_lookup_reuses_existing_endpoint`) is thin — it checks
the endpoint shape but not the client-side filter, which is how CR-01
slipped through.

**Fix:** When CR-01 is addressed, add a test that at minimum spins up the
server, registers two streams, pushes events for one key on both, and calls
the entity lookup endpoint with an assertion that the expected flat feature
names are present. That is the minimum contract the frontend filter must
satisfy. A DOM-level test is out of scope pending Phase 10.2's headless
browser harness.

### IN-04: `debug_memory` holds the AppState lock while iterating every entity and every stream name (potential contention, not a correctness bug)

**File:** `src/server/http.rs:477-529`

**Issue:** `debug_memory` holds the state mutex continuously while it
iterates `app.store.entity_keys()` and then re-enters `get_entity(key)` for
each key to count per-stream membership (lines 486-492). For a 1M-key state
store this is an O(N·M) walk under the mutex, blocking TCP hot-path PUSH/GET
while it runs. Performance issues are **out of scope for v1** per the review
policy, so I am flagging this as Info only: on the happy path with
≤100K keys it is fine, and the handler is on the management HTTP port, not
on the TCP hot path — so contention is the concern, not the hot path itself.

The original Phase 10 rollup version used only `app.store.entity_count()`
which is O(1); the new per-stream breakdown inherently needs the walk.
RESEARCH §Pitfall 8 discusses this; the trade-off was made deliberately.
Worth noting now so a future Phase 10.2 latency debugger can revisit if
measurements show it matters.

**Fix:** If contention shows up in practice, maintain a per-stream key
counter incrementally in the state store (bump on insert, decrement on
eviction) and read it in O(1) here. No action required in Phase 10.1.
