# Phase 10.1: Interactive Debug UI Redesign - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace Phase 10's flat four-tab Debug UI with a split-view interactive layout where the topology DAG is the primary navigation surface. Clicking a node opens a per-stream drill-in panel showing memory, operator state, throughput, and a stream-scoped entity lookup. DAG edges carry live throughput numbers (the downstream stream's 5s EWMA) refreshed at the existing 1 Hz polling cadence.

In scope:
- New split-view `index.html` shell (DAG canvas ~60-70% left, drill-in panel ~30-40% right)
- Removal of the Topology / Streams / Entity / Memory tab bar — the DAG is the only view
- Rewritten `app.js` with click-to-select node handlers, drill-in panel renderer, edge label updates, and the global pause toggle
- Additive backend change: add `operators` field to `/debug/topology` node payload with `{name, op, window, where_clause, filter}` per feature (keeps existing `features` array for backward compat)
- Updated `app.css` that reuses Phase 10 tokens verbatim but adds split-view layout, panel components, edge label styling
- Integration tests in `tests/test_debug_ui.rs` covering the new backend fields and split-view asset changes
- Empty state for when no pipelines are registered — centered hero on the DAG canvas with tally-mark icon and register-a-pipeline instructions

Out of scope:
- Any backend changes to `/debug/throughput` or `/debug/memory` (unchanged from Phase 10)
- New endpoints — everything uses existing `/debug/topology`, `/debug/throughput`, `/debug/memory`, `/debug/key/{key}` with the single additive field on topology
- Latency debugger (deferred to Phase 10.2, which runs after 10.1)
- Authentication / authorization on the debug port (still assumed bound to localhost / trusted network)
- Historical time-series storage, alerting, multi-user sessions
- Mobile or touch-optimized layout — desktop-only, same as Phase 10

</domain>

<decisions>
## Implementation Decisions

### Inherited from Phase 10 UI-SPEC §13.1 (unchanged, do not re-open)
- Tech stack: htmx 1.9.x + vanilla JS + dagre-d3 + d3 (all vendored via rust-embed from Phase 10)
- Spacing: exactly 6 tokens (4, 8, 16, 24, 32, 48), no 12px
- Typography: exactly 4 type sizes (12, 14, 20, 24), no 16px, 2 weights (400, 600)
- Color palette: dark-mode-only, restricted accent palette (`--accent-blue` for streams, `--accent-purple` for views, `--accent-warm` for warm orange / paused state)
- No build step, no npm, no external CDN fetches, single-binary via rust-embed
- XSS defense: every user-supplied string rendered via `.textContent` or d3 `.text()` — zero `.innerHTML` for user data

### Layout Paradigm
- **Split view** — DAG canvas on the left (~60-70% width), drill-in panel on the right (~30-40% width). Both visible simultaneously. Classic observability pattern (Grafana Explore, Datadog Service Map).
- No tab bar. The DAG is the only top-level view.
- Page shell structure:
  ```
  <header>tally' wordmark · live dot · Pause button</header>
  <main class="split-view">
    <section class="dag-canvas">...dagre-d3 DAG...</section>
    <aside class="drill-in-panel">...selected node details...</aside>
  </main>
  <footer>version · connection · last update</footer>
  ```
- When no node is selected, the drill-in panel shows a placeholder ("Select a stream to see details") or is hidden (planner's choice).

### Drill-in Panel Content (Essentials)
Five sections, in this top-to-bottom order:

1. **Stream header** — name, kind (stream / view), key_field, depends_on list
2. **Memory** — total bytes + active key count (from `/debug/memory` per_stream entry filtered by name)
3. **State** — operator list: each feature's name, operator type (count / sum / avg / min / max / distinct_count / last), window setting, where clause (if any). Sourced from the new `operators` field on `/debug/topology` nodes.
4. **Throughput** — ewma_5s / ewma_1m / ewma_5m for the selected stream (from `/debug/throughput`)
5. **Entity lookup** — input + button. On submit, calls `/debug/key/{key}` and renders the response filtered to just the selected stream's features. textContent-only, no innerHTML.

### Edge Throughput Semantics
- **Interpretation:** Edge `A→B` shows B's throughput (how many events per second are landing in B). Zero backend changes needed — uses the existing `/debug/throughput` response.
- **Display:** Inline label on the edge midpoint using dagre-d3 native edge labels. Refreshed at 1 Hz with the same polling cadence as Phase 10. Format is compact: `1.2k/s`, `450/s`, `12.3/s`.
- **Window source:** ewma_5s — matches the 1 Hz polling cadence, tracks "now" closely.
- **No thickness or color encoding** — just the text label. Keep visual complexity low.

### Old Tabs Disposition
- **Delete all four tabs.** Topology, Streams, Entity, Memory tabs are removed entirely from the UI.
- `WR-02` from Phase 10 REVIEW.md (broken Entity form wiring) is resolved as a side effect — the broken form is deleted.
- Deleted DOM elements: `.tab-bar`, `#tab-topology`, `#tab-streams`, `#tab-entity`, `#tab-memory`, their content panels, and the tab activation JS in `app.js`.
- The Pause button and the wordmark stay in the header. The "Live · 1 Hz" status dot stays.

### Entity Query Scope
- **Stream-scoped only.** The entity lookup input inside a drill-in panel calls `/debug/key/{key}` (unchanged backend), then the frontend filters the response to show ONLY the features belonging to the currently selected stream.
- Other streams' features for the same key are NOT shown.
- No "see all streams" toggle. No cross-stream quick-jump chips. Keep the scope discipline tight — matches the mental model "I'm looking at Stream A, show me A's features for this key."

### Empty State (No Pipelines Registered)
- Centered hero on the DAG canvas: tally-mark icon, heading "No pipelines registered", body "Use the Python SDK or REGISTER command to push a pipeline, then reload this page."
- Right panel shows a welcome placeholder or is hidden.
- Same copy the Phase 10 Topology tab used — directly inherit.

### Backend Contract Change (Single Additive Field)
- `/debug/topology` node payload gains an `operators` field alongside the existing `features` field:
  ```json
  {
    "name": "Transactions",
    "kind": "stream",
    "key_field": "user_id",
    "features": ["tx_count_30m", "tx_sum_1h"],
    "operators": [
      {"name": "tx_count_30m", "op": "count", "window": "30m", "where": null, "filter": null},
      {"name": "tx_sum_1h",    "op": "sum",   "window": "1h",  "field": "amount", "where": null}
    ],
    "depends_on": []
  }
  ```
- Purely additive: `features` stays as the string-list shape Phase 10 tests assert on. Phase 10 integration tests pass unchanged.
- Shape of each operator entry is derived from the engine's `FeatureDef` enum — planner will map the enum variants to JSON. Include enough fields to render "count over 30m where status='failed'" in the panel.
- View features (lookups) need their own shape — planner will decide whether to put them under the same `operators` field or a sibling `lookups` field. Claude's Discretion on that sub-shape.

### Pause Toggle Semantics (Inherited from Phase 10)
- Global Pause button in the header toggles all 1 Hz polling.
- When paused: edge labels freeze at their last value, drill-in panel's throughput/memory stop updating, the status dot flips to warm orange, the footer reads "Paused · last update {HH:MM:SS}".
- When resumed: everything resumes from the new tick.
- Space key toggles pause when no input is focused.

### Tab-Bar Arrow Keys and Entity Esc-to-Clear
- These were flagged by Phase 10 UI-REVIEW as at-risk-of-orphaning. With the tab bar deleted, tab-bar arrow keys no longer apply.
- Entity input Esc-to-clear: include in Phase 10.1 (the entity input moves into the drill-in panel, but Esc to clear the input and close the result section is still a free accessibility win).

### Accessibility Must-Haves (Inherited)
- Keyboard navigation: Tab cycles through interactive elements (Pause button, entity input, submit button, lookup results)
- Focus rings: 2px accent-blue outline with 2px offset on all focusable elements
- Reduced motion: `prefers-reduced-motion: reduce` disables node selection transitions and panel slide animations
- aria-live region for the pause state transition ("Polling paused", "Polling resumed")

### Claude's Discretion
- Exact width ratio between DAG canvas and drill-in panel (recommend 65/35 but planner can tune based on content density)
- Animation duration for node-selection transitions (recommend 120ms, matches Phase 10's existing transition-fast token)
- Whether the drill-in panel slides in from right or is always visible (recommend always visible with placeholder when no selection — avoids layout shift)
- Exact format of the compact throughput number (1.2k/s vs 1,234/s vs 1.2K/s) — pick one and apply consistently
- View node drill-in content — views have lookup features, not windowed operators; panel sections may differ. Discretion to adapt the Essentials layout for views specifically.
- Whether the deleted Phase 10 static assets (`index.html`, `app.css`, `app.js`) are rewritten in-place or replaced wholesale — planner's call, both are fine given this phase owns all five files.

</decisions>

## Existing Code Insights

### Reusable Assets from Phase 10
- `src/server/ui/vendor/htmx.min.js`, `d3.min.js`, `dagre-d3.min.js` — all three vendored with SHA256 manifest, drift-tested
- `src/server/ui.rs` — UiAssets rust-embed struct and handlers for `GET /` and `GET /static/{*file}` (axum 0.8 brace syntax). No changes needed.
- `src/server/http.rs` — `/debug/topology`, `/debug/throughput`, `/debug/memory`, `/debug/key/{key}` handlers. Only `/debug/topology` gets the additive `operators` field.
- `src/server/throughput.rs` — `ThroughputTracker` with corrected alpha-mixing EWMA (WR-01 fix landed). Unchanged.
- `src/server/ui/icons.svg` — tally-mark, alert-triangle, search, chevron-right, close, pause, play, copy, link symbols. Reusable.
- `src/server/ui/favicon.svg` — warm-orange tally-mark. Reusable.

### Established Patterns to Follow
- **textContent-only DOM writes** for any user-supplied string (entity keys, feature values, stream names, error messages). Never `.innerHTML` for user data.
- **Lock-once-then-build-JSON** pattern in http.rs handlers — no `.await` across the AppState mutex (RESEARCH §Pitfall 3 from Phase 10).
- **1 Hz htmx polling** via `hx-trigger="every 1s"` for the throughput updates. Pause via `data-hx-disable` on every `[hx-trigger*="every"]` container.
- **d3 `.text()` for all node and edge labels** in dagre-d3 — never `.html()`. dagre-d3 default labelType is 'text' which is XSS-safe.
- **Grep-based XSS sink regression test** (`app_js_has_no_innerhtml_or_eval_sinks` in `tests/test_debug_ui.rs`) from Phase 10 WR-03 fix. Phase 10.1 must keep this test passing.

### Integration Points
- `/debug/topology` — gains `operators` field. Phase 10 tests assert on existing `features` field; planner must ensure they stay green.
- `src/server/ui/index.html`, `app.css`, `app.js` — wholesale rewrite. Five files change: these three plus the two SVGs (icons and favicon are kept but may add new symbols for expand/collapse arrows, etc.).
- `tests/test_debug_ui.rs` — existing 15 tests stay. Add new tests for the `operators` field shape, the split-view asset shell (grep index.html for the new structure), and a regression test that the old tab bar is GONE.
- `src/engine/pipeline.rs` — `FeatureDef` enum — planner reads this to map variants to the new `operators` JSON shape. No changes to the enum itself unless a variant needs an accessor.

### Phase 10 Known Issues Resolved by This Phase
- **WR-02 (broken Entity form):** Deleted along with the Entity tab. No fix needed.
- **UI-REVIEW `§9.1` skeleton shimmer dead code:** Phase 10.1 is a rewrite — the planner should either ship the skeleton (it already exists in CSS) or delete it. Discretion.
- **UI-REVIEW DAG empty state collapses canvas:** Phase 10.1's empty state is centered on the DAG canvas by design — planner should not use `display: none` on the canvas.

<specifics>
## Specific Ideas

### User's Original Framing (verbatim)
"I'm thinking a more interactive UI where user see the whole pipeline and when they click one node they can go in and see memory profile (state) for each state held by this, also can do query based on entity there. Also the link between them should have edge and update number with latest throughput."

### Reference UX Patterns
- **Grafana Explore node graph panel** — canonical example of split-view DAG + side panel for drill-in. Click-to-select, right-panel-updates pattern.
- **Datadog Service Map** — similar split view, but with service-level metrics in the side panel. Our drill-in is simpler (just one stream's state).

### Concrete UX Decisions
- Click a node → right panel populates. Click the same node again OR click empty canvas → panel clears to placeholder.
- Keyboard shortcut: `Esc` clears the current selection.
- Entity lookup input lives inside the drill-in panel (not in the header). Only appears when a node is selected.
- The header Pause button is the only global control. Everything else is node-scoped.

</specifics>
