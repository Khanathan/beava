## Decisions Made

- **Pass-through from raw_register_jsons, not FeatureDef enum walk**: The engine already stores the user's original register JSON string-for-string via `store_raw_register_json`. Walking the FeatureDef enum would require reconstructing `where` strings from the parsed `Expr` AST (no `Display` impl exists). The raw JSON is the exact shape Phase 10.1 needs.
- **Field rename `type` -> `op` at projection boundary**: `type` is awkward in JS as `.type` and clashes with Rust reserved-word context. `op` is 2 chars, matches domain language ("count is an operator"), unambiguous in frontend template strings. CONTEXT explicitly specifies `op` in the example JSON.
- **Reduced view projection**: Views only have derive and lookup features — no windowed operators, no field, no where, no bucket, no optional, no backfill. Keeping the view branch short reduces noise and matches the plan's explicit guidance.
- **No top-level node `filter` field**: RESEARCH Pattern 8's canonical snippet includes a `filter` passthrough on stream nodes, but the plan explicitly de-scopes this. Keeping the diff minimal de-risks Plans 02 and 03 which depend on the operators contract.
- **Empty-array fallback (not null, not missing)**: The frontend will iterate with `(node.operators || []).forEach(...)`, and emitting a literal `[]` is the simplest invariant. This is safe for snapshot-restored streams that predate the `raw_register_jsons` store.

## Deviations from Plan

None — plan executed exactly as written. The test helper API differences the plan warned about (`where_expr` vs `where_clause`, `on_field` vs `on`, tuple-unpacked `http_get` response vs struct) were handled by mirroring the existing `register_test_pipeline` helper in the same file.

## Issues Encountered

- **Initial field-name mismatch in plan examples**: The plan's example test code used `where_clause: None` and `on: "merchant_id"`, but the actual `FeatureDef::Count` uses `where_expr: None` and `ViewFeatureDef::Lookup` uses `on_field: String`. The plan's own text told me to mirror the existing `register_test_pipeline` helper if field names differed — I followed that guidance and used the correct names. The plan also offered a fallback of using `where_expr: None` with the where-clause only in the raw JSON store, which is the path I chose for `topology_operators_pass_through_where_clause` (safer — the backend only reads the raw JSON, not the parsed AST).
- **Response shape in existing helpers**: The existing `http_get` helper returns a tuple `(u16, HashMap<String, String>, Vec<u8>)`, not a struct with `.status`/`.body` fields as the plan's example showed. I destructured into `(status, _headers, body)` consistently with existing tests.
