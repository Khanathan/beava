## Fixed Issues

### CR-01: Entity lookup filter never matches — stream filter assumes dotted feature names

**Files modified:** `src/server/ui/app.js`
**Commit:** `e959ef4`
**Applied fix:** Replaced the broken dotted-name parser
(`fullName.indexOf('.')` branch) in
`lookupEntityForSelectedStream` with the frontend-only Option A fix from
the review: build an allow-list `Set` from the selected topology node's
`features` array and intersect it with the flat keys from
`data.computed_features`. Also dropped the dead `data.features` fallback
(IN-01 was implicitly addressed as a side effect, since removing the
dotted-name path made the fallback trivially dead).

Key excerpt of the new filter:

```js
const computed = (data && data.computed_features) || {};
const node = findNode(selectedStream);
const allowed = new Set((node && node.features) || []);
const filtered = {};
for (const name of Object.keys(computed)) {
  if (!allowed.has(name)) continue;
  filtered[name] = computed[name];
}
```

Also added two regression tests in `tests/test_debug_ui.rs` (committed
separately as `3d985d4`, see below) pinning the contract:

1. `entity_lookup_filter_uses_flat_feature_names` — registers a
   two-stream pipeline (`Transactions` + `Logins`), pushes events for the
   same key on both, asserts `/debug/key/{key}` emits flat feature names
   (no `.` prefix), asserts `/debug/topology` node payloads carry
   matching per-stream `features` arrays, and simulates the frontend
   allow-list filter to prove it returns the expected non-empty subset
   and correctly excludes the other stream's features.
2. `app_js_entity_lookup_uses_allow_list_not_dotted_names` — grep-level
   source guard on the served `app.js` bytes. Asserts the old dotted-name
   parsing tokens (`fullName.indexOf('.')`, `fullName.substring(0, dot)`,
   `fullName.substring(dot + 1)`) are absent and the new allow-list
   markers (`data.computed_features`, `(node && node.features)`,
   `allowed.has(name)`) are present. Analogous to
   `app_js_has_no_innerhtml_or_eval_sinks`.

### WR-01: Memory tick full-panel re-render wipes user-visible entity lookup results

**Files modified:** `src/server/ui/app.js`
**Commit:** `7780606`
**Applied fix:** Refactored `renderMemorySection` to mirror the
throughput pattern — it now checks for an existing
`.drill-in-section[data-section="memory"]` and `replaceWith`s it in
place if present, or `appendChild`s it on first render. Then updated
`tickMemory` to call `renderMemorySection(panel, node, data)` directly
instead of `renderDrillInPanel(state.selectedStream)`, and dropped the
`focusInPanel` guard entirely as the review recommended — once the
re-render is scoped to only the Memory section, it physically cannot
touch the entity input DOM regardless of focus state.

This also fixes a subtle pre-existing smell: the old `focusInPanel`
guard caused the memory tick to stop updating entirely whenever the
entity input had focus, which was a silent regression of the 2 Hz
memory refresh. The granular pattern refreshes memory stats even while
the user is typing into the lookup field.

### WR-02: `tickMemory` / `tickEdgeUpdater` don't handle selected-stream-vanished case

**Files modified:** `src/server/ui/app.js`
**Commit:** `811de9b` (tickEdgeUpdater) — the `tickMemory` vanished-node
guard was folded into commit `7780606` (WR-01) because the granular
`renderMemorySection` call would null-deref on a vanished node, so the
guard was a prerequisite of WR-01's refactor.
**Applied fix:** Both tick functions now detect the case where
`findNode(state.selectedStream)` returns `null` (e.g. a future refresh
affordance reloaded the topology and the previously-selected stream
was DELETEd or a test intentionally swapped topologies) and call
`setSelected(null)` to reset the panel to its explicit "Select a stream
to see details" placeholder instead of silently blanking it.

```js
if (!node) {
  setSelected(null);
} else if (panel && node.kind !== 'view') {
  renderThroughputSection(panel, node, data);   // or renderMemorySection
}
```

Not a live bug in Phase 10.1 (topology is loaded exactly once) but
removes the foot-gun before Phase 10.2 adds a refresh affordance, and
before CR-01's new regression tests (which exercise multiple topology
states) have a chance to trip it.
