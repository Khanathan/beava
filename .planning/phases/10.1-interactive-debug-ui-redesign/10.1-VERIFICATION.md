### Deferred Gaps (discovered during manual smoke test)

Three minor gaps were surfaced during the live smoke test. None block shipping Phase 10.1 — they are scoped as follow-up work to Phase 10.2 or a later polish phase.

**Gap 1: Keyless streams show irrelevant drill-in sections**

`renderDrillInPanel` in `src/server/ui/app.js` does not branch on `node.key_field === null`, so clicking a keyless stream (like `RawEvents`) surfaces Memory, Throughput, and Entity Lookup sections. All three are semantically meaningless for keyless streams because keyless streams do not store per-entity state — they are pure event routers that fan out to downstream keyed aggregators.

- Severity: visual clutter, no correctness failure
- Fix scope: ~20 LOC frontend tweak + 1 regression test — add a keyless variant that shows only header + State section (operator list remains meaningful if the keyless stream has derive features)
- Defer to: Phase 10.2 or a polish phase

**Gap 2: View entity lookup always returns empty**

View nodes show the Entity Lookup section but queries always return "No features for '{key}'" because views compute their derived features on read and are not materialized in per-entity state. The `/debug/key/{key}` endpoint only returns features that exist in the per-entity `computed_features` map, and view features are never inserted there. The frontend stream-scoped filter therefore finds zero matches.

- Severity: view variant's entity lookup is non-functional
- Fix scope: either (a) backend evaluates view derive expressions on debug lookup and includes them in the response, or (b) frontend hides the Entity Lookup section for view nodes entirely
- Defer to: Phase 10.2 Latency Debugger (which already touches the debug endpoints) or a polish phase
- Recommended direction: option (a) — backend evaluation gives users an actual debugging surface for view features, which is the whole point of the drill-in panel

**Gap 3: No per-operator / per-feature byte breakdown in Memory section**

The `/debug/memory` endpoint tracks bytes at the stream level only (`{name, kind, key_count, estimated_bytes}`). The drill-in Memory section therefore shows only the whole-stream total, not a breakdown by operator state (e.g. how many bytes `tx_count_1h` buffers use vs `last_country` single-value vs `avg_amount_1h` sum+count buckets).

- Severity: granularity limited; users cannot identify which operator is dominating a stream's memory footprint
- Fix scope: backend walks each operator's state and computes its serialized size; frontend renders inline in the State section (bytes column per operator row) or as a new Memory-by-operator subsection
- Defer to: follow-up polish phase or Phase 10.2
