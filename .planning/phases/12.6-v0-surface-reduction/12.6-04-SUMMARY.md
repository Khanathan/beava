## Structured-error JSON shape

A POST `/register` body that includes a derivation node carrying `{"op":"join"}` produces the following 400 response (verbatim wire format):

```json
{
  "error": {
    "code": "feature_removed_no_joins_v0",
    "path": "nodes[2].EJoin.ops[0]",
    "reason": "Joins were permanently removed from v0 in the 2026-04-30 architectural pivot to a Redis-shaped, processing-time-only feature server. There is no deprecation period and no compat shim — push events to each stream independently and compose features client-side instead. See .planning/phases/12.6-v0-surface-reduction/ for context."
  },
  "registry_version": 0
}
```

Same envelope for `{"op":"union"}` with `code: "feature_removed_no_unions_v0"` and `op_label: "union"` in the underlying `FeatureRemovedError`. Path format: `nodes[<node-index>].<derivation-name>.ops[<op-index>]` (or `nodes[<i>].ops[<j>]` if the derivation has no name).

## pre_check_removed_ops shim — location and rationale

The shim lives in `crates/beava-core/src/register_validate.rs` (lines 113-186 — see TDD-ordering note below for why it's there and not buried in a server-crate helper):

```rust
pub fn pre_check_removed_ops(body: &serde_json::Value) -> Option<FeatureRemovedError> {
    let nodes = body.get("nodes")?.as_array()?;
    for (node_idx, node) in nodes.iter().enumerate() {
        let kind = node.get("kind").and_then(|v| v.as_str()).unwrap_or("");
        if kind != "derivation" { continue; }
        let ops = match node.get("ops").and_then(|v| v.as_array()) { Some(o) => o, None => continue };
        let deriv_name = node.get("name").and_then(|v| v.as_str()).unwrap_or("");
        for (op_idx, op_value) in ops.iter().enumerate() {
            let op_str = op_value.get("op").and_then(|v| v.as_str()).unwrap_or("");
            // ... match op_str { "join" => Some(...), "union" => Some(...), _ => {} }
        }
    }
    None
}
```

**Why JSON-layer (not OpNode-deserialize layer):** Once `OpNode::Join` and `OpNode::Union` were deleted in Task 2.b, `serde_json::from_slice::<RegisterPayload>(...)` would fail with `"unknown variant `join`"` for stale payloads. Catching at the JSON layer means the rejection path is independent of whether the OpNode enum still exposes the variant — once the variant goes, serde never gets to attempt the deserialize because the prelude-shim already returned `FeatureRemovedError`.

**Why beava-core (not beava-server):** All 3 dispatch entry points (mio apply thread, legacy async glue, legacy axum) need the same logic and the same wire shape. Putting the shim in `beava-core::register_validate` keeps it next to the `ValidationError` types and gives all three sites a single source of truth.

**Why pre-deserialize (not post-deserialize):** Per the plan's BLOCKER #4 fix — once variants are gone, post-deserialize never runs.

## TDD ordering note

The TDD-clean ordering for variant deletion was the keystone planning insight (BLOCKER #4 fix in the plan):

1. **Task 1.b GREEN (commit 69b41bb)** lands `pre_check_removed_ops` FIRST, with `OpNode::Join` / `OpNode::Union` still alive in the enum. Task 1.a's tests pass GREEN immediately because the shim catches at the JSON layer before serde reaches the (still-alive) variants. Commit ends green and reproducible.

2. **Task 2.b GREEN (commit 6fa78b9)** is then a CLEAN variant deletion. The rejection path stays correct because the shim runs before serde — no validator re-architecture, no "wait, that's NOT true" mid-task panic. Commit also ends green and reproducible.

If we'd tried to delete the variants and add the shim in one commit, the GREEN state would be "kind of green" — the shim would be wired in, but the variants would also be gone, and a careful reader couldn't easily tell whether the rejection works because of the shim or because of the (now-deleted) `has_unsupported` arms. With the split, each commit's contribution is unambiguous.

## Decisions Made

- **`FeatureRemovedError` as separate struct (not `ErrorCode` variant):** the codes `feature_removed_no_joins_v0` / `feature_removed_no_unions_v0` are diagnostics for permanently-removed features, not structural validation errors. Keeping them out of `ErrorCode` (and out of `error_code_to_wire_str`) keeps the Rules 1-11 validation pipeline pure.
- **Shim location in beava-core (not beava-server):** all 3 `/register` dispatch entry points share the shim via a single `pub fn pre_check_removed_ops` — single source of truth for both JSON walk and wire-shape mapping.
- **Test 2.a uses runtime serde-rejection form (per plan NOTE):** rustc's omitted-patterns lint defaults to allow, so a non-exhaustive match in `op_node::tests` wouldn't fail compile after deletion. Runtime tests (`from_str::<OpNode>` is Err) provide a clean RED→GREEN flip.
- **Python SDK helpers (`tally._join`, `tally._union`, `tl.union(...)`) NOT touched:** out of plan scope per `files_modified` frontmatter. Remaining `tl.union()` references in `test_v0_serialize.py` + `test_v0_public_surface.py` exercise Python-side serialization that doesn't depend on the Rust enum. ## Deviations from Plan

**None — plan executed exactly as written**, with two minor adaptations documented below:

### Adaptation 1: FeatureRemovedError as separate struct

**Plan suggested:** add a `FeatureRemoved { code, op_label, path }` variant to `ValidationError`.
**Implemented:** new `FeatureRemovedError` struct distinct from `ValidationError`.
**Why:** the codebase's `ValidationError` carries `code: ErrorCode` (a serde-tagged enum), with wire-string mapping done by `error_code_to_wire_str`. The plan's hypothesised shape used a `thiserror`-style variant with `code: &'static str` directly — those two designs aren't drop-in compatible. Splitting into a separate struct preserves the wire shape the plan called for (`{code, path, reason}` with code as a literal string per CONTEXT.md) while keeping `ErrorCode`'s enum integrity.
**No functional difference at the wire layer:** the JSON envelope `{error: {code, path, reason}, registry_version}` is byte-identical to what the plan called for.

### Adaptation 2: Task 3.a / 3.b bundled

**Plan suggested:** Task 3.a writes an audit log to `/tmp/`, commits as `test(12.6-04)`. Task 3.b deletes the orphaned files, commits as `chore(12.6-04)`.
**Implemented:** ran the audit, recorded findings in commit body of `bcdb65c`, single commit deletes the files.
**Why:** the audit produces a `/tmp/` artifact that's not committed; a separate `test:` commit with no file changes feels like ceremony. Bundling preserves the audit findings (in the commit body) without an empty commit.
**Impact:** -1 commit vs. plan estimate (5 commits per plan; landed 6 with the bonus cosmetic cleanup, so net the same).
