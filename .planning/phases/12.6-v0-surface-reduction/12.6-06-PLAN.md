<objective>
Hard rip event_time everywhere per D-03: zero compat at any layer.

Six surfaces:

1. **Push wire schema** — HTTP JSON + framed-TCP push payload bodies no longer accept `event_time_ms` (or any unknown field). The server-side parser refuses unknown fields and returns a structured 400 with `unknown_field_event_time_v0` error code.

2. **Register wire schema** — POST /register payload struct uses `#[serde(deny_unknown_fields)]` (or an equivalent strict-deny shape via a custom Visitor / pre-parse intercept). Legacy `event_time_field` / `tolerate_delay_ms` JSON keys are REJECTED at parse time with structured codes `unknown_field_event_time_v0` / `unknown_field_tolerate_delay_v0`. **D-03 verbatim:** "Hard rip everywhere — zero `event_time_ms` compat at any layer. … No deprecation window, no parse-and-strip, no warn-then-error." Silent-ignore is parse-and-strip — explicitly forbidden.

3. **EventDescriptor (Rust)** — `event_time_field: Option<String>` and `tolerate_delay_ms: Option<u64>` deleted from the registry descriptor. `DEFAULT_TOLERATE_DELAY_MS` const deleted.

4. **AppState global atomic** — `DevAggState.max_event_time_ms` (an `Arc<AtomicU64>`) deleted along with all readers and writers. (Plan 05 already removed the writer; Plan 06 deletes the field — Plan 06 depends on Plan 05.)

5. **WAL record schema bump** — `FORMAT_VERSION: u32 = 1` becomes `FORMAT_VERSION: u32 = 2`. The v2 record body has no event_time field. Recovery rejects v1 records with `SchemaVersionMismatch` (already exists in `error.rs`) — no migration shim per D-03.

6. **Snapshot body schema bump** — `SNAPSHOT_BODY_FORMAT_VERSION: u16 = 1` becomes `SNAPSHOT_BODY_FORMAT_VERSION: u16 = 2`. The v2 body has no `max_event_time_ms` field. v1 bodies fail with `UnsupportedVersion(1)` (already exists in `SnapshotBodyError`) — no migration.

7. **Fixture sweep** — Bucket 3 of CONTEXT.md says fixtures get migrated; the cost of D-03 is the sweep. Roughly ~50 test JSON fixtures still pass `"event_time_field": "..."` / `"tolerate_delay_ms": ...` to /register. They MUST be rewritten (those JSON keys deleted) so they round-trip the strict-deny register parser.

D-03 is explicit: "Hard rip everywhere — zero `event_time_ms` compat at any layer. v0 isn't released; no users to break." Pre-pivot WALs/snapshots are dev-only artifacts that get discarded; users running into v1-on-disk on a fresh-cut binary delete the wal and snapshot dirs and start over. Test fixtures with legacy JSON keys get rewritten — they are not "users" and we control them.

Wave 2 placement: depends on Plan 01 (TestServer rewrite) AND Plan 05 (Path X — must remove the `dev_agg.max_event_time_ms.fetch_max(...)` write before this plan deletes the field). Plan 06 touches the wire layer + persistence schema + register strict-deny + fixture sweep — files that Plan 04 doesn't touch.
