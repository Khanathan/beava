<objective>
Hard rip event_time everywhere per D-03: zero compat at any layer.

Five surfaces:

1. **Push wire schema** — HTTP JSON + framed-TCP push payload bodies no longer accept `event_time_ms` (or any unknown field). The server-side parser refuses unknown fields and returns a structured 400 with `unknown_field_event_time_v0` error code.

2. **EventDescriptor (Rust)** — `event_time_field: Option<String>` and `tolerate_delay_ms: Option<u64>` deleted from the registry descriptor. `DEFAULT_TOLERATE_DELAY_MS` const deleted.

3. **AppState global atomic** — `DevAggState.max_event_time_ms` (an `Arc<AtomicU64>`) deleted along with all readers and writers. (Plan 05 already removed the writer; Plan 06 deletes the field.)

4. **WAL record schema bump** — `FORMAT_VERSION: u32 = 1` becomes `FORMAT_VERSION: u32 = 2`. The v2 record body has no event_time field. Recovery rejects v1 records with `SchemaVersionMismatch` (already exists in `error.rs`) — no migration shim per D-03.

5. **Snapshot body schema bump** — `SNAPSHOT_BODY_FORMAT_VERSION: u16 = 1` becomes `SNAPSHOT_BODY_FORMAT_VERSION: u16 = 2`. The v2 body has no `max_event_time_ms` field. v1 bodies fail with `UnsupportedVersion(1)` (already exists in `SnapshotBodyError`) — no migration.

D-03 is explicit: "Hard rip everywhere — zero `event_time_ms` compat at any layer. v0 isn't released; no users to break." Pre-pivot WALs/snapshots are dev-only artifacts that get discarded; users running into v1-on-disk on a fresh-cut binary delete the wal and snapshot dirs and start over.

Wave 2 placement: parallel with Plan 04 (joins/unions removal) and Plan 05 (Path X). Plan 06 touches the wire layer + persistence schema — files that Plan 04 and Plan 05 don't touch. The cross-coordination point is `apply_shard.rs::dispatch_push_sync`: Plan 05 stops reading `event_time_field` from the row at apply time, and Plan 06 deletes the field declaration on EventDescriptor entirely. Plan 05 lands first per its task ordering; Plan 06 then tears down the field.

Output:
- Push payload schemas reject `event_time_ms` with structured 400 + error code `unknown_field_event_time_v0`
- `EventDescriptor` and `DevAggState` lose their event-time fields
- `FORMAT_VERSION` bumped to 2 in `crates/beava-persistence/src/record.rs` and `SNAPSHOT_BODY_FORMAT_VERSION` to 2
- Recovery rejects pre-pivot WAL records and snapshot bodies cleanly (no migration code)
- `crates/beava-server/tests/phase12_6_event_time_hard_rip.rs` proves all of the above end-to-end
</objective>
