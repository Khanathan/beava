<objective>
Surgically delete the Join + Union DAG nodes from `beava-core`, then add a JSON-prelude rejection shim that emits structured error codes when older Python SDK fixtures or hand-rolled JSON DAGs include `{"op": "join"}` or `{"op": "union"}` nodes. After this lands, the runtime cannot represent a join/union; the wire schema cannot deserialize one (because the OpNode variants are gone); and the register-time validator gives a clean v0 message instead of a confusing serde "unknown variant" error.

**TDD-clean task ordering (BLOCKER #4 fix):**

The naive ordering ("delete variants → catch at validate-time") breaks at the commit boundary: once OpNode::Join/Union are gone, `serde_json::from_value::<OpNode>(...)` fails with "unknown variant `join`" BEFORE the validator runs, so a validate-time rejection arm catches nothing. The TDD-clean ordering is:

1. **Task 1.b (GREEN, FIRST):** Add the `pre_check_removed_ops` JSON-prelude shim FROM THE START. The shim intercepts `{"op":"join"}` / `{"op":"union"}` BEFORE strict OpNode deserialize and emits the structured error code. Task 1.a tests pass GREEN immediately because the variants are still alive (the shim catches BEFORE serde reaches them). This commit ends in a fully green, reproducible state.

2. **Task 2.b (GREEN, SECOND):** A pure variant-deletion. Delete OpNode::Join, OpNode::Union, JoinType. Because the JSON-layer shim already runs FIRST, the rejection path still works post-deletion (the shim catches the JSON `op` field; serde never gets to attempt unknown-variant deserialize). No re-architecture mid-task. This commit also ends in a fully green, reproducible state.

Each task's GREEN commit is atomic + green + reproducible. No "wait, that's NOT true" mid-task panic. No prelude-and-delete-in-one-commit bundle. The shim shape is decided up front and lands first; the deletion lands second.

Why this matters in 12.6: deleting the variants from the enum is the surgical operation. But naively deleting them also deletes the deserialize path, which means a stale fixture's `{"op": "join"}` payload would surface as an opaque serde "unknown variant" error to users. The structured-error-code path makes the failure mode obvious — and putting the shim at the JSON layer (not the OpNode layer) means the rejection works whether or not the variant exists.
