# Phase 12.6 Plan 10: Mio-only architectural test — Summary

The CI tripwire that makes the locked Phase 12.6 architectural commitment
(`project_phase18_no_dual_runtime` — mio is the SOLE data-plane runtime)
mechanically self-enforcing. Plan 07 deleted the legacy axum data-plane
files; Plan 10 adds the test that catches any future re-introduction of a
second hot-path entry or a non-admin axum user. Plan 10 also surfaces and
deletes the last ~315 LOC of orphan axum dead-code from `registry_debug.rs`
that survived Plan 07's deletion sweep — those orphans WERE the only
third caller of `apply_event_to_aggregations` and would have made the
architectural test RED at HEAD if left in place.

## What landed

### Task 1.a (RED) — architectural test (commit `3c366e3`)

Created `crates/beava-server/tests/phase12_6_mio_only_dataplane.rs` with
three test functions:

1. **`only_apply_shard_and_recovery_call_apply_event_to_aggregations`** —
   walks `crates/beava-server/src/` recursively, reads each `.rs` file,
   strips `//`/`///`/`//!` comment lines, and asserts that
   `apply_event_to_aggregations(` appears ONLY in `apply_shard.rs` (mio
   hot path) and `recovery.rs` (cold-path WAL replay). Any third caller
   is reported as `"<path>: contains a call to apply_event_to_aggregations(...)"`.

2. **`axum_imports_only_in_admin_sidecar`** — same walk, comment-stripped,
   asserts that the patterns `axum::Router`, `axum::Json`,
   `axum::Extension`, `axum::extract::`, `axum::routing::`,
   `axum::http::`, `axum::body::`, `axum::response::`, `axum::middleware`
   appear ONLY in `http_admin.rs` (the canonical tokio admin sidecar
   bound by `ServerV18` at `cfg.admin_addr`).

3. **`sanity_allowlisted_callers_actually_contain_patterns`** — reads
   the three allowlisted files (`apply_shard.rs`, `recovery.rs`,
   `http_admin.rs`) and asserts each contains the pattern it is
   supposed to be the home for. Guards against the test silently
   becoming a no-op if a future refactor moves a canonical site.

The RED state at HEAD: `registry_debug.rs` violated BOTH invariants
(it called `apply_event_to_aggregations` at line 421 inside the orphan
`post_dev_apply_events` async fn, AND it imported `axum::Router` plus
`axum::routing::{get, post}` for orphan router builders). The test file
was committed in RED state with this real architectural regression as
the natural failing case.

The Cargo.toml was updated to register the new `[[test]]` entry under
the existing Plan-12.6-07 axum-killed test entry.

### Task 2 (docs) — CLAUDE.md update (commit `dfa6b6d`)

Added a new subsection at the end of CLAUDE.md `§Conventions`,
immediately after `Performance Discipline`:

```markdown
### mio-only Hot-Path Invariant (locked Phase 12.6)

**Rule:** The mio event loop is the only data-plane runtime. All
push/get/upsert/delete/retract dispatches go through
`crates/beava-server/src/apply_shard.rs::dispatch_*_sync` (specifically
`dispatch_one` → `dispatch_push_sync`). The only legitimate callers of
`apply_event_to_aggregations` are `apply_shard.rs` (mio data plane) and
`recovery.rs` (cold-path WAL replay on boot). axum is restricted to the
ServerV18 admin sidecar in
`crates/beava-server/src/http_admin.rs::BoundAdminServer`, bound on a
separate port (`cfg.admin_addr`) for `/health`, `/ready`, `/metrics`,
`/registry` only.

**Why:** Locked architectural commitments — `project_phase18_no_dual_runtime`
(single hot-path entry; admin endpoints stay on tokio) +
`project_redis_shaped_no_event_time_ever` (Redis-shaped, processing-time
only, no dual stack). New code MUST NOT introduce a parallel data-plane
runtime, a third caller of `apply_event_to_aggregations`, or `axum::*`
symbols (Router/Json/Extension/extract/routing/http/body/response/middleware)
outside `http_admin.rs`.

**How enforced:**
`crates/beava-server/tests/phase12_6_mio_only_dataplane.rs` walks the
workspace at test runtime and fails if either invariant is violated. CI
runs the test on every PR via `cargo test --workspace`. Companion:
`crates/beava-server/tests/phase12_6_legacy_axum_killed.rs` (Plan 12.6-07)
asserts the legacy axum files and symbols stay deleted.

**Reviving a second data-plane runtime requires:** explicit user override +
new ADR overturning `project_phase18_no_dual_runtime`.
```

### False positives during probe tuning

One minor false positive surfaced during probe 1 development: my first
probe form `let _probe = beava_core::agg_apply::apply_event_to_aggregations;`
(without `(` parens) did NOT trigger the test, because the grep needle
is `apply_event_to_aggregations(` with the open paren. This is the
INTENDED behaviour — the test catches function-CALL sites, not bare
function-pointer references. Strengthening the probe to use a real
(unreachable) call resolved the asymmetry. The test's grep needle is
correct: a bare reference is harmless dead-code; a call is hot-path
violation.

The allowlist tuning was also iterative:

- Initial draft used `server.rs` as the axum allowlist (per the plan's
  pseudo-code).
- After grep-ing `server.rs` post-Plan-07, confirmed it has ZERO
  `axum::` symbols — only doc comments mentioning "tokio/axum on the
  admin plane" — which the comment-stripper filters out.
- Switched the axum allowlist to `http_admin.rs` (the actual canonical
  tokio admin sidecar per Plan 07 Deviation 1), which matches today's
  reality.

  reflects the plan-author's mental model of the admin sidecar living

**3. [Rule 2 — Defense in depth] Added `sanity_allowlisted_callers_actually_contain_patterns`
test to guard against the test silently becoming a no-op.**

- **Found during:** Task 1.a — reviewing the test's failure modes.
- **Issue:** If a future refactor moves `apply_event_to_aggregations`
  out of `apply_shard.rs` (or moves the axum admin sidecar out of
  `http_admin.rs`) without updating the allowlist, the test would pass
  vacuously — it would walk a tree where no file contains the pattern,
  and report no violations. Silently becoming a no-op is the worst
  failure mode for a tripwire test.
- **Fix:** Added a third sanity test that reads each allowlisted file
  and asserts it actually contains the pattern it is supposed to be
  the home for. If `apply_shard.rs` ever stops containing
  `apply_event_to_aggregations(`, this sanity test fails — alerting
  the maintainer to update the allowlist.
- **Files affected:** `crates/beava-server/tests/phase12_6_mio_only_dataplane.rs`.
- **Commit:** 3c366e3 (the RED gate).
