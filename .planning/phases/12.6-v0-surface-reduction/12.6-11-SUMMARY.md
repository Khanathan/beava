# Phase 12.6 Plan 11: Post-axum-kill apply microbench — Summary

The criterion microbench that satisfies CLAUDE.md §Performance Discipline
for Phase 12.6. Drives `ApplyShard::dispatch_wire_request_with_row`
end-to-end through three operator shapes (simple counter, count_distinct
with HLL, windowed 60s sum) and appends the captured medians to
`.planning/perf-baselines.md` as Phase 12.6's regression-tripwire baseline.
Plan 13 will compare against these numbers; >25% regression on any cell
blocks Phase 12.6 verification.

#### `crates/beava-server/benches/common.rs` (181 LOC, NEW)

Shared bench-bootstrap helper module exposing:

- **`BenchHarness`** struct — bundle of `ApplyShard`, `Arc<AppState>`,
  `Arc<Registry>`, the `event_name` extracted from the register payload,
  and three keep-alive fields (`_wal_dir: TempDir`, `_wal_writer_handle:
  Option<JoinHandle<()>>`, `_rt: Runtime`).
- **`build_apply_shard_with_pipeline(register_payload: serde_json::Value) ->
  BenchHarness`** — single entry point. Pass a JSON register payload, get
  a ready-to-push harness. The implementation:
  1. Spawns a multi-threaded tokio runtime (1 worker, all features) so the
     Register code path's internal `current_thread::block_on` works.
  2. Spawns the legacy `WalSink` (Phase 7-7.5 cold-path WAL for register).
  3. Constructs Registry + DevAggState + IdemCache + AppState.
  4. **Spawns a `WalWriter`** with tick_ms=5 alongside the `WalBufferRing`
     (REQUIRED — see decision Rule-3 above for the bring-up hang debug).
  5. Builds the `ApplyShard`.
  6. Sends Register through `dispatch_wire_request_with_row` to populate
     the registry and grow the state tables.
  7. Extracts the first `kind: "event"` node's `name` from the payload
     so cells don't repeat the literal in two places.

The shape mirrors `crates/beava-server/benches/phase12_07_read_path.rs`'s
`setup_warm_app_state` (the canonical server-side bench bootstrap pattern,
itself derived from `crates/beava-server/tests/phase18_02_inline_wal_test.rs`
lines 150-174 — `dispatch_push_periodic_returns_committed_lsn`).

#### `crates/beava-server/benches/phase12_6_post_axum_kill_apply.rs` (320 LOC, NEW)

Three criterion bench groups, each with a CONCRETE body (no
`unimplemented!` stubs anywhere; verified via `grep -c unimplemented!` = 0):

| Cell | Pipeline | Pre-warm | Bench loop |
|---|---|---|---|
| `phase12_6/simple_counter/100_events` | `Tx.group_by(user_id).count()` | 1000 events × 100 entities | 100 events/iter via `iter_batched`, varied `user_id` |
| `phase12_6/sketch_heavy/100_events` | `Tx.group_by(user_id).count_distinct(session_id, window="1h")` | 1000 events with unique `session_id` (promotes CountDistinct past EXACT_THRESHOLD into HashSet mode) | 100 events/iter, fresh `session_id` per iter to keep HashSet mode active |
| `phase12_6/windowed_60s_sum/100_events` | `Tx.group_by(user_id).sum(amount, window="60s")` | 1000 events × 100 entities (WindowedOp buckets populated) | 100 events/iter, varied `amount` |

Each cell follows the same shape:
1. Build the register payload as a `serde_json::json!` literal.
2. Call `common::build_apply_shard_with_pipeline(payload)`.
3. Pre-warm: dispatch 1000 push events to amortize per-entity init.
4. `c.bench_function(...)` with `iter_batched(setup, routine, BatchSize::SmallInput)`:
   - `setup` builds a `Vec<Bytes>` of 100 JSON-serialized event bodies.
   - `routine` dispatches each via `WireRequest::HttpPush` → `ApplyShard::dispatch_wire_request_with_row`.

| `phase12_6/sketch_heavy/100_events` | 88.400 µs | **884.0 ns/event** | 83.054–96.841 µs | 9% (4 low mild, 1 high mild, 4 high severe) | **PASS** (first measurement) |

### Rationale per cell

**simple_counter (810.3 ns/event)** — End-to-end dispatch_push_sync with
a single Count feature. The closest analogue is Phase 19.4-E
`apply_path/warm_key/14_aggs` = 318.68 ns (agg-stage only, 14 features).
A rough cost model:

- ~100-200 ns: `sonic_rs::from_slice::<Row>` JSON parse for `{user_id, amount}` body
- ~50-100 ns: descriptor lookup + schema validate + strict-deny field iteration (Plan 12.6-06)
- ~50-100 ns: WAL record build + `WalBufferRing::append` (lock-free atomic memcpy)
- ~150-300 ns: agg-stage (1 Count feature; a fraction of Phase 19.4's 14-feature 316.95 ns)
- ~50-100 ns: bookkeeping counters + event_id_index insert

Sum ≈ 400-800 ns, in the same envelope as the measured 810 ns. The fixture is sane.

**sketch_heavy (884.0 ns/event)** — CountDistinct in HashSet mode after
pre-warm (1000 unique session_ids → past EXACT_THRESHOLD=16). This
exercises the same hot path Phase 19.4-A optimized (identity-hasher win).
The +73 ns/event vs simple_counter reflects the HashSet add + windowed
bucket maintenance cost. No regression detected; the Phase 19.4-A
optimization carries forward.

**windowed_60s_sum (882.9 ns/event)** — WindowedOp.update with
`SystemTime::now()` syscall as the time source (Plan 12.6-05 Path X swap;
the dispatch_push_sync calls `SystemTime::now().duration_since(UNIX_EPOCH)`
once per event). The bench measures *post-Path-X* steady state. The +73
ns/event vs simple_counter encompasses both the syscall (~10-30 ns
expected on macOS) and the WindowedOp bucket fold work (~40-60 ns).
No regression detected.

## Comparison-baseline choice and "first measurement" framing

The plan's <action> Step 5 enumerated three possible Phase 19.4 analogue
cells:

- simple_counter ≈ "Plan 19.4-04 fraud-team K=10k zipfian sustained_eps cumulative" / 10000 events / 1 RT
- sketch_heavy ≈ "Plan 18-11 hot-path optimization agg stage" or the count_distinct identity-hasher fix bench
- windowed_60s_sum ≈ "Plan 19.3-02 WindowedOp::update_at"

After examining each, the executor chose to record this as a "first
measurement" rather than computing a delta against a Phase 19.4 cell.
The rationale (per decision §1 above): Phase 19.4's `apply_path/warm_key/*`
benches measure the agg-stage in isolation (`apply_event_to_aggregations`
called directly with a pre-built `Row` and pre-resolved registry).
Phase 12.6's bench measures end-to-end `dispatch_push_sync`: HTTP/TCP push
entry → JSON parse → descriptor lookup → strict-deny field check →
schema validate → dedupe lookup → WAL serialize + append → agg-stage →
bookkeeping counters. The numbers differ by ~500 ns/event of legitimate
non-agg work; computing a delta would imply they're comparable when
they're not.

Plan 13 verification compares against THIS Phase 12.6 row going forward —
not against Phase 19.4. The 10% / 25% threshold gates apply to delta vs
this baseline, in the same hw-class.

## Bench shape & methodology notes

- **Bench harness shared via common.rs.** Criterion's `[[bench]]` integration
  targets cannot import each other, so the WAL+Registry+AppState bootstrap
  was extracted to a path-mod (`mod common;`) so future Phase 12.6+ benches
  reuse the same setup. The `BenchHarness` struct exposes `shard`, `state`,
  `registry`, `event_name` for the bench cell to drive.
- **WalWriter spawned alongside the ring (Rule 3 deviation).** Initial
  bring-up hung indefinitely on the second criterion warmup phase. Root
  cause: without a writer thread, sealed buffers never return to FREE
  state, and `do_seal_and_swap` blocks on `free_condvar` once the 3 × 16
  MiB ring exhausts. Fix: spawn `WalWriter::new(dir, ring, lsn, tick_ms=5)`
  inside `build_apply_shard_with_pipeline`. Documented in the helper's
  module-level docstring + a long inline comment.
- **BATCH_SIZE = 100.** Amortizes criterion's per-iteration overhead to
  <1% of measured time. Reported numbers are per 100-event batch; divide
  by 100 for ns/event.
- **`iter_batched` setup builds the body bodies inside the bench closure**
  (not at harness construction time) so each iter has fresh JSON bytes.
  This adds ~10 µs of `serde_json::to_vec` work to each iter's setup,
  which criterion subtracts via the `iter_batched` semantics — `setup`
  is run before the timer starts; only `routine` is timed.
- **Outliers in the 9-12% range** on all three cells. Apple-M4 macOS
  development machines are noisy under typical desktop load (browser
  tabs, background apps); criterion's median is robust to this and is
  the canonical regression-detection number. Phase 13 should re-run on
  Hetzner Linux EPYC-Genoa in a quieter environment for the
  production-shipping baseline.

## Deviations from plan

### Rule 3 — auto-fix blocking issue: WalWriter required for bench

**Found during:** Task 1.b initial run — bench hung indefinitely on the
second warmup phase.

**Issue:** The plan's <action> Step 2 code-block instantiated
`WalBufferRing` + `WalLsn` but no `WalWriter`. Without a writer, sealed
buffers never return to FREE state; `WalBufferRing::append` blocks on
`free_condvar` once the 3 × 16 MiB ring exhausts.

**Fix:** Added `WalWriter::new(dir, ring, lsn, tick_ms=5).spawn()` inside
`build_apply_shard_with_pipeline`. Stored as `Option<JoinHandle<()>>` on
`BenchHarness` to keep the writer thread alive for the bench duration.

**Files modified:** `crates/beava-server/benches/common.rs`.

**Why this is Rule 3 and not architectural:** the writer is part of the
canonical mio data-plane WAL setup (production `ServerV18::serve_with_dirs`
also spawns one). The plan's <action> code-block was a sketch; the
executor adapted to the actual WAL contract.

### Rule 1 — choice of cell-3 op (sum instead of count)

**Found during:** Task 1.b initial bench design.

**Issue:** The plan's <objective> initially proposed cell 3 as
`Tx.group_by(user_id).agg(cnt=bv.count())` with `window="60s"` — a
windowed count. A windowed count doesn't exercise the field-extraction
or numeric fold paths.

**Fix:** Used `sum(amount, window="60s")` instead. Both are valid 60s
windowed ops; sum gives broader coverage of the WindowedOp dispatch path
(field extraction + f64 fold) while still being sensitive to the
`SystemTime::now()` syscall the cell is meant to test.

**Why this is Rule 1 and not architectural:** the plan's <action>
ALSO showed a sum-based windowed op (Step 3 cell 3 was `sum(amount,
window="60s")`). The plan was internally inconsistent on this point;
the executor picked the more thorough one. Both interpretations are
within the plan's stated intent.
