## Threshold rationale

- **Pre-12.6 baseline:** 1,065 symbols across 6 deletion buckets + 0 compiler warnings.
- **Post-12.6 target:** 133 symbols (~87% reduction) + ≤5 compiler warnings.
- **Per-bucket targets** are NOT uniformly aggressive — they reflect:
  - **Plans 04 + 05:** near-complete elimination (≤5 each) since the only legitimate residual is error-message strings + grandfathered tests.
  - **Plan 06:** 95% reduction (588 → 30) — D-03 hard rip removes the wire/schema entirely; residuals are negative-path tests asserting rejection of pre-pivot wire fields.
  - **Plan 07:** 85% reduction (53 → 8) — three whole-file deletes plus `phase6_crash_probe` retention of `BEAVA_DEV_ENDPOINTS` is the only non-deletion item.
  - **Plan 08:** 86% reduction (111 → 15) — Python SDK keeps a tight set of negative tests as `feature_removed_*` arms.
  - **Plan 10:** 0% reduction, +9% upper-bound slack — enforcement-only; the count is a *grep ceiling* not a deletion target.
- **Compiler-warning slack:** ≤5 transient warnings. Wave-end commits MUST clean them. If any plan ends a wave with >5 net new warnings, the plan re-opens.
- **Plan 13 verification:** re-runs the 6 grep commands above; asserts each count ≤ target. If any count exceeds: Plan 13 verification BLOCKS. If orthogonal warnings have grown organically (new code shipped during 12.6 that legitimately needs `#[allow(dead_code)]`), Plan 13 may amend a target with explicit justification — but no silent slack.
