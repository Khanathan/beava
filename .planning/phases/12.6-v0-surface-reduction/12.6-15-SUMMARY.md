## What landed

### Bucket B — Test-side migrations off /dev/apply_*

The legacy axum `/dev/apply_events` and `/dev/apply_ops` routes are HTTP-only test shims. Per `project_phase18_no_dual_runtime` they don't exist on mio (correctly so — dev endpoints stay test-only).

- **`phase4_smoke.rs`** (5 calls migrated): `/dev/apply_ops` → in-process `apply_ops_white_box` helper that runs `ts.registry().compiled_chain(name).apply(row)` and JSON-serialises the result with the same conversion table as `registry_debug::value_to_json`. 7/7 tests pass.
- **`phase5_smoke.rs`** (12 calls migrated): `/dev/apply_events` → `/push/<source>` (mio data-plane route). Both paths drive the same `apply_event_to_aggregations` function; descriptor.event_time_field reads `event_time` from the row. SC4 byte-replay determinism preserved. 10/10 tests pass.
- **`phase9_smoke.rs`** (1 call migrated): same pattern. 4/4 tests pass.
- **`phase11_smoke.rs`**: residual `/dev/apply_events` references are inside dead/comment text; live tests already use `/push`. No change needed; 3/3 pass.
- **`phase10_sketch_recovery.rs`**: already uses `/push/Tx`. No change needed; 2/2 pass.

### Bucket A — TCP error-frame surface (3 phase2_5_smoke tests)

Per Plan 14 SUMMARY's deferral note: "TCP-side architectural issues unrelated to the mio HTTP gap-closure scope". Plan 15 adds the surface inline because each fix is small and the alternative (deferral) would block Phase 12.6's GREEN gate.

- **`criterion_5_unimplemented_opcode_returns_error_connection_stays_open`** — `WireRequest::Unknown { op }` apply-shard arm now distinguishes `OP_PUSH_SYNC` (reserved for Phase 12) → `op_not_implemented` from generic unknown → `unknown_op`. Connection stays open per spec (the apply path doesn't mark client closed for these).
- **`bonus_msgpack_register_returns_unsupported_ct_connection_stays_open`** — TCP parser arm for OP_REGISTER now rejects non-CT_JSON content_type bytes via `WireRequest::ParseError` carrying a `unsupported content_type for register` prefix; apply-shard recognises the prefix and emits `TcpError { code: "unsupported_content_type" }`. Connection stays open.
- **`criterion_7_oversize_frame_closes_connection`** — `io_thread_worker.rs` parse-error branch detects `FrameError::TooLarge { declared_len, limit }` and surfaces a new `ParseErrorKind::TcpFrameTooLarge { declared, limit }` variant. Because the connection MUST close after the error frame, the encoder runs INLINE in the worker (`inline_encode_frame_too_large`) before marking the client closed; the cleanup loop now flushes write_buf before closing the backend connection. The apply-side ParseErrorKind dispatch was extended too (for completeness — both legacy IoPool and the new io_thread_worker paths emit the right shape), though only the inline path is on the live test path.

Also: `TestServerBuilder.tcp_max_frame_bytes(N)` now sets `BEAVA_TCP_MAX_FRAME_BYTES` env var so the mio `parse_wire_request` worker reads the same limit (the legacy `cfg.tcp.max_frame_bytes` field is preserved for source-compat but is unused on mio).

10/10 phase2_5_smoke tests pass.

### Bucket C — Per-test residuals (12 tests across 8 binaries)

Each fix surfaced a distinct mio gap or test-fixture issue:

- **phase6_smoke + phase6_push (4+10 tests)** — Two architectural fixes:
  - **Byte-identical idem-cache replay (Rule 2):** mio's `GlueResponse::PushReplay` synthesised a generic `{idempotent_replay: true, registry_version: V}` envelope. Phase 6 success criterion #2 + the test require the second push response to be BYTE-IDENTICAL to the first (i.e. include the original `ack_lsn`). Added `Option<Bytes> cached_body` to PushReplay; `IdemCache::get_with_ack_lsn` returns both ack_lsn and cached body; both encoders pass cached_body verbatim when present.
  - **`X-Beava-Idempotent-Replay: 1` HTTP response header (Rule 2):** legacy axum push handler emits this header on dedupe replays; mio HTTP encoder didn't. Added `extra_headers` slot in encode_glue_response_http and the PushReplay arm appends the header.
  - **`push_persisted_to_wal` test migration:** under the locked mio architecture, push events go to hand-rolled `*.wal` segments via WalBufferRing → WalWriter, NOT the legacy `*.log` segments via WalSink. Migrated the test to read `*.wal` segments directly.
- **phase8_tcp_push (4 tests)** — Three TCP encoder fixes for legacy axum wire-shape parity:
  - PushAck body: include `idempotent_replay: false` flag (was `{ack_lsn, registry_version}` only).
  - PushError body: `{"error": {"code": "..."}}` shape (was `{"code": "..."}`).
  - PushReplay body: include cached `ack_lsn` and `idempotent_replay: true` so the second push body has the replay-LSN identity AND the discriminator flag (TCP has no header equivalent of HTTP's X-Beava-Idempotent-Replay).
- **phase7_restart_cycle (3 tests)** — Two recovery fixes:
  - **Snapshot loading on mio path (Rule 2):** `build_runtime_state` (Plan 12.6-01 extraction) skipped `load_snapshot_if_any`. Inserted before *.log replay; *.log replay now gates on snapshot_lsn so post-snapshot records are properly applied.
  - **WalWriter final-drain on shutdown (Rule 1 - Bug):** the run_serve_loop slept 6ms then dropped the JoinHandle, detaching the writer thread mid-tick and losing any active-buffer contents. Captured `wal_writer.shutdown_flag()` before spawn() consumes self; on shutdown set the flag THEN join the writer thread so we wait for its final-seal+drain+fsync block.
- **phase18_04_6_integration_test (1 test)** — `SERVER_SERIALIZER` was a `std::sync::Mutex` whose guard dropped before the first `.await` (only sequenced test starts). Switched to `tokio::sync::Mutex` and held the guard for the full test duration.
- **phase10_sketch_recovery (2 tests)** — Auto-resolved by the snapshot-load fix above.
- **wal_env_var_tunables (1 test)** — Added in-test `ENV_LOCK` mutex to serialise process-global env-var mutations (the comment about `--test-threads=1` at the cargo invocation isn't enforced under `cargo test --workspace`).
- **cli_smoke (1 test)** — Added `CLI_SUBPROCESS_LOCK` so the two binary-spawning tests don't compete; `env_var_overrides_listen_addr` now polls-until-bind for up to 3s instead of fixed 300ms sleep (macOS launch jitter under parallel cargo workloads pushed first-bind beyond 300ms occasionally).
