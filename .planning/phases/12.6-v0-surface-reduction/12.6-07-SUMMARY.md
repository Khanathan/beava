# Phase 12.6 Plan 07: Legacy axum data-plane kill — Summary

The architectural moment where mio becomes the SOLE data-plane runtime
per `project_phase18_no_dual_runtime`. Plan 07 deletes ~7475 LOC of legacy
axum surface (the plan estimated ~3500 LOC; actual delete more than
doubled because the legacy `crate::tcp` module + in-source axum-router
test mods totaled an additional ~2900 LOC of dead code that cascaded out
once their primary dependency went). The tokio admin sidecar
(`http_admin.rs::BoundAdminServer`) is preserved on its own port
(`cfg.admin_addr`); the mio data plane is the only path for /push,
/push-sync, /upsert, /delete, /retract, /table-get, /get, /register,
/ready, /registry on the data-plane port.

## Deviations from Plan

### Auto-fixed Issues (Rule 1+2 + Rule 4 surfaced inline)

**1. [Rule 4 — Architectural decision surfaced inline] http_admin.rs MUST be kept.**

- **Found during:** Task 1.b — Read of http_admin.rs.
- **Issue:** The plan's frontmatter `files_modified` list said "FULL DELETE" for http_admin.rs. The plan's `<interfaces>` section justified this with: "ServerV18's tokio sidecar (line 488+, serve()) has its own admin routes... http_admin.rs is fully duplicate." This is wrong. ServerV18 at line 488 (per the original line numbering) does:

  ```rust
  let admin = crate::http_admin::BoundAdminServer::bind(admin_addr, snapshot)
  ```

  That is, `http_admin.rs::BoundAdminServer` IS the tokio admin sidecar — there is no second copy inside server.rs. Deleting http_admin.rs would break `ServerV18::bind` entirely, which would cascade to break all production binary boot, all TestServer-based tests, the bench harness, and the `phase18_01_bind_v18` test.
- **Fix:** Keep http_admin.rs as-is. Update the architectural test in `phase12_6_legacy_axum_killed.rs` to NOT assert http_admin.rs is gone. This honors the plan's explicit success_criteria 3 ("Tokio admin sidecar (separate port) is preserved per project_phase18_no_dual_runtime") which trumps the file-level delete instruction.
- **Files affected:** `crates/beava-server/src/http_admin.rs` (kept); `crates/beava-server/tests/phase12_6_legacy_axum_killed.rs` (Test 1 specifies only push.rs / http.rs / push_and_get.rs files deleted, not http_admin.rs).
- **Commit:** Inline in the GREEN commit (4555e99).

**2. [Rule 4 — Architectural decision surfaced inline] TestServer's `.dev_endpoints(bool)` builder method MUST be kept.**

- **Found during:** Task 2.b — Read of testing.rs Step 8 instructions.
- **Issue:** The plan's Step 8 said "Delete the `.dev_endpoints(...)` builder method" with the rationale that "Plan 01 left it as a no-op shim". This was true at the time the plan was AUTHORED, but Plan 14 (executed before Plan 07) upgraded the method into a REAL toggle on AppState's `Arc<AtomicBool> dev_endpoints` flag (per Plan 14 SUMMARY: "TestServer's `.dev_endpoints(true)` builder method now flips the flag on the shared `AppState` post-spawn — was a no-op pre-Plan 14"). 77 callers across the workspace use the method. The mio /registry shim's gating depends on this flag (apply_shard.rs:484 — `if !self.state.dev_endpoints_enabled() { return 404 }`). Deleting the method without an architectural replacement would force migration of all 77 sites + would break the ability of tests to observe registry contents over the data-plane HTTP listener.
- **Fix:** Keep the `.dev_endpoints(bool)` builder method on TestServerBuilder. Updated its doc-comment to cite Plan 12.6-07's posture (production data-plane /registry permanently 404; the builder enables a test-only path on the data-plane HTTP listener for white-box assertions).
- **Files affected:** `crates/beava-server/src/testing.rs` (method PRESERVED with updated doc-comment); `crates/beava-server/src/lib.rs` (AppState's `dev_endpoints: Arc<AtomicBool>` field PRESERVED with updated doc-comment).
- **Commit:** Inline in the GREEN commit (4555e99).

**3. [Rule 2 — Missing critical functionality] In-source `mod tests` blocks deleted; coverage replicated.**

  - **registry_debug unit tests preserved**: `dev_agg_state_post_d03_has_no_legacy_max_field` (Phase 12.6 D-03 hard-rip guard) and `event_id_entry_stream_takes_arc_str` (Arc<str> identity invariant) survive the deletion sweep.
- **Files affected:** `crates/beava-server/src/register.rs` (test mod deleted), `crates/beava-server/src/feature_query.rs` (test mod deleted), `crates/beava-server/src/registry_debug.rs` (axum-router tests deleted; 2 unit tests preserved).
- **Commit:** Inline in the GREEN commit (4555e99).

**4. [Rule 1 — Bug fix] `crate::push::PushAck` reference replaced with inline JSON.**

- **Commit:** Inline in the GREEN commit (4555e99).

**5. [Rule 2 — Scope expansion] tcp.rs full delete (not in plan's full-delete list).**

- **Found during:** cargo check cascade after deleting Server::serve.
- **Issue:** `crate::tcp::TcpListenerHandle` and `crate::tcp::accept_loop_with_app` were the legacy tokio TCP accept loop. The only callers were inside the now-deleted `Server::serve` body. After Server is gone, tcp.rs is orphaned (1241 LOC of dead code). The mio TCP listener inside ServerV18's `run_serve_loop` has its own accept + parse + dispatch path via apply_shard.rs.
- **Fix:** Full delete of tcp.rs + lib.rs `pub mod tcp;` declaration.
- **Files affected:** `crates/beava-server/src/tcp.rs` (deleted), `crates/beava-server/src/lib.rs` (declaration removed).
- **Commit:** Inline in the GREEN commit (4555e99).

**6. [Rule 1 — Test infrastructure] phase6_crash + phase6_1_crash + phase6_crash_probe binary deleted.**

- **Found during:** cargo check cascade — phase6_crash_probe binary references deleted Server::bind.
- **Issue:** The phase6_crash_probe binary was a subprocess wrapper around `Server::bind` for SIGKILL-based crash UAT. After Server is deleted, the binary cannot compile. phase6_crash.rs and phase6_1_crash.rs are tests that spawn the binary as a subprocess.
- **Fix:** Delete the binary + both subprocess tests + corresponding Cargo.toml `[[bin]]` and `[[test]]` entries. Crash UAT coverage is implicit in the WAL durability test suite (phase6_smoke, phase6_push, phase7_restart_cycle exercise the same fsync-then-restart contracts via TestServer with snapshot+WAL replay across instance lifecycles).
- **Files affected:** `crates/beava-server/src/bin/phase6_crash_probe.rs` (deleted), `crates/beava-server/tests/phase6_crash.rs` (deleted), `crates/beava-server/tests/phase6_1_crash.rs` (deleted), `crates/beava-server/Cargo.toml` (entries removed).
