<objective>
Switch the time source for all windowed / point-recency / decay / velocity / bounded-buffer / geo operators from event_time_ms (read from the event body) to now_ms (computed server-side at apply time as the wall-clock at dispatch). This is Path X from CONTEXT.md Bucket 4 and the architectural commitment in project_redis_shaped_no_event_time_ever.

The change is mechanical at the trait level (parameter rename across the AggOp / AggState / WindowedOp surface) but semantically meaningful at the apply site: instead of reading event_time_ms from the parsed Row, the apply path captures std::time::SystemTime::now().duration_since(UNIX_EPOCH).as_millis() (or equivalent) at the start of dispatch_push_sync and threads that value through.

After this lands, "rolling 60s sum" means "the last 60 seconds of arrival-time" — same operator catalogue, same semantics for the user, different time source under the hood. Catalogue stays at 55 ops.

Wave 2 placement: parallel with Plan 04 (joins/unions deletion) and Plan 06 (event-time wire schema strip). They touch different files. Plan 05 owns the agg_*.rs rename + apply call site; Plan 06 owns the wire-schema (push payload + WAL header) deletion of event_time_ms. They meet at apply_shard.rs::dispatch_push_sync where Plan 05 stops reading from the Row and starts computing from the server clock.

Output:
- All event_time_ms parameter names in agg_*.rs renamed to now_ms (and _event_time_ms to _now_ms for impls that don't read it)
- agg_windowed.rs::bucket_index(now_ms) / bucket_epoch(now_ms) / fold(now_ms) consistent
- agg_apply.rs::apply_event_to_aggregations takes now_ms: i64 instead of event_time_ms: i64
- apply_shard.rs::dispatch_push_sync computes now_ms server-side (not from Row.event_time field)
- All windowed-op tests pass (existing tests pass event_time_ms = 1000 * i; renamed to now_ms = 1000 * i works mechanically)
- New integration test phase12_6_path_x_windowed_arrival_time.rs proves bucketing is on arrival-time, not event-time
</objective>
