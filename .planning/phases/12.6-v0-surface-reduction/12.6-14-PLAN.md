<objective>
Add the missing mio data-plane HTTP surface that Plan 01's TestServer→ServerV18 swap surfaced as gaps:
1. **`POST /upsert/{table_name}`** — temporal table upsert (v0-scope per stateful-architecture memory)
2. **`POST /delete`** — temporal table tombstone delete
3. **`POST /retract`** — explicit retract for table rows (501 for stream tables per D-12)
4. **`GET /registry` dev_endpoints gating** — 404 when `BEAVA_DEV_ENDPOINTS != 1`
5. **POST content-type 415 rejection** — match legacy axum content-type validation

After this plan, the failing test binaries in scope:
- `phase11_5_temporal_smoke` (5 tests on /upsert /delete /retract)
- `phase18_07_upsert_delete_rename_test`
- `phase2_smoke` (dev_endpoints gating + content-type)
- `phase2_5_smoke` (likely content-type)

…should turn green WITHOUT test-side changes.

**Approach:** Extract handler-body logic from `crates/beava-server/src/temporal_http.rs::upsert_handler|delete_handler|retract_handler` into pub helper functions that take `Arc<AppState>` + a body parse, then have mio's runtime_core_glue dispatch invoke them when it receives `WireRequest::HttpUpsert` / `HttpDelete` / `HttpRetract`. Same pattern Plan 01 used for /ready and /registry shims. Pre-serialize response body so HTTP encoder emits bytes-identical responses to legacy axum.

**TDD discipline phase 3+ (mandatory):** Each new mio surface = red test (new file `phase12_6_14_mio_temporal.rs` or per-feature) → green impl.

**Architectural anti-patterns to reject:**
- Do NOT add a second hot-path entry — this is mio's data plane gaining surface, not introducing a parallel runtime
- Do NOT reintroduce event_time anywhere (Plan 06 will hard-rip; the table row body has no event_time field at the user wire level)
- Do NOT add tokio to the data plane — admin sidecar stays separate

**Out of scope:**
- Test-side migrations for /dev/apply_events / /dev/apply_ops (those are Plan 12.6-15)
- Per-test residual fixes (Plan 12.6-15)
</objective>
