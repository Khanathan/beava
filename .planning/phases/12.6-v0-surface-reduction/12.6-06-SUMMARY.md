# Phase 12.6 Plan 06: Event-time hard rip (D-03) — Summary

**`event_time_ms` and `event_time_field` and `tolerate_delay_ms` are GONE from the v0 wire schema, the WAL record format, the snapshot body format, the EventDescriptor struct, the DevAggState global, and ~50 test fixtures. Stale fixtures sending those keys raise structured 400 errors with codes `unknown_field_event_time_v0` / `unknown_field_tolerate_delay_v0` per CONTEXT D-03 verbatim — no parse-and-strip, no warn-then-error, no compat shim.**

This implements the wire + storage layer of `project_redis_shaped_no_event_time_ever` (locked 2026-04-30) for the Beava v0 server: the post-pivot architectural commitment that windowed-op bucketing, decay arithmetic, and recency tracking all key on **server-side wall-clock at dispatch** (the `now_ms` value computed in `apply_shard.rs::dispatch_push_sync`'s `SystemTime::now().duration_since(UNIX_EPOCH)`), not on a body-derived event timestamp.

## Six surfaces touched

### 1. Push wire schema — strict-deny on `event_time_ms`

`apply_shard.rs::dispatch_push_sync` walks the parsed `Row.iter()` field list against `descriptor.schema.fields`; any field absent from the schema raises a structured 400 with code `unknown_field_event_time_v0` (special-case for `event_time` / `event_time_ms`) or `unknown_field_v0`. The check runs AFTER `Row::deserialize` (Beava's Row is a generic key-value map, so deny_unknown_fields on Row would be wrong) and BEFORE the schema-shape validate so callers get a precise error.

Test (Task 1.a → 1.b RED→GREEN):

```rust
#[tokio::test]
async fn push_payload_with_event_time_ms_returns_400_unknown_field() {
    // Register clean Tx { user_id: str, amount: f64 } — no event_time slot.
    // POST /push/Tx with body {"user_id":"u1","amount":42.0,"event_time_ms":1234567890}
    // Expect: 400 + error.code == "unknown_field_event_time_v0"
}
```

### 2. Register wire schema — strict-deny on `event_time_field` / `tolerate_delay_ms`

New JSON-prelude shim `register_validate::pre_check_legacy_event_time_keys` mirrors Plan 04's `pre_check_removed_ops` pattern: walks `body.nodes[*]` BEFORE strict `RegisterPayload` deserialize, returning `Some(FeatureRemovedError)` on the first hit with a structured wire shape:

## Path X behavioural test relaxations

Plan 05 Path X means windowed/velocity ops bucket on **server arrival-time** rather than body event_time. Events pushed back-to-back land in the same wall-clock millisecond on the server. Two tests had to relax their time-dependent assertions:

### `phase11_smoke::all_eleven_ops_round_trip_through_http`

Pre-pivot the test pushed events with body `event_time` 1 hour apart and asserted `kmh` (geo_velocity) returned `~111 km/h` based on the synthetic dt. Post-Path-X the events arrive back-to-back so dt is near-zero and kmh is very large. Relaxed assertion:

```rust
assert!(v.is_finite() && v >= 0.0, "kmh must be finite positive, got {v}");
```

### `phase9_smoke::phase9_register_all_16_ops_and_push_events`

Pre-pivot expected all 16 features non-null after 30 events. Post-Path-X four time-derivative features (`trend_amt`, `trend_residual`, `rate_of_change`, `interarrival`) legitimately return null when wall-clock dt is zero. Moved to a separate "structural-success-only" assertion list:

## Architectural commitment posture

This plan implements the wire + storage layer of `project_redis_shaped_no_event_time_ever` (locked 2026-04-30). Key constraints honored:

1. **No parse-and-strip / no warn-then-error** — D-03 verbatim. The strict-deny shim raises 400 with structured codes, NOT silently drops the legacy keys. Tested directly.
2. **No `legacy_event_time` warnings that imply the feature might come back** — the error reason text is unambiguous: "permanently removed from v0 in the 2026-04-30 architectural pivot. There is no deprecation period and no compat shim."
3. **No compat mode toggle** — there is no `BEAVA_LEGACY_EVENT_TIME=1` escape hatch. Removed means removed.
4. **No silent WAL upgrade path** — v1 records and v1 snapshot bodies fail with the existing `SchemaVersionMismatch` / `UnsupportedVersion` paths. Operators clear `wal_dir` / `snapshot_dir` before booting the new binary; pre-pivot WALs are dev artifacts.

## Decisions Made

See frontmatter `decisions:` field for the full list. Key calls:

- **Field rename, not deletion** — `max_event_time_ms` → `query_time_ms` (plan choice (a)). Rationale: the GET-path readers depend on the field; renaming aligns name with semantics; choice (b) would have required rewriting `compute_query_time_ms` to call `SystemTime::now()` directly (same end-state, more churn).
- **JSON-prelude shim, not serde deny_unknown_fields** — same reasoning as Plan 04. `deny_unknown_fields` cascades poorly through nested types; the pre-walk gives precise control over the structured error code emission point.
- **Push-side check at descriptor-schema layer, not Row deserialize** — Beava's Row is a generic key-value map; deny_unknown_fields on Row would break the user-defined event schema design. The strict-deny must run at the EventDescriptor boundary AFTER Row::deserialize.
- **Pre-pivot enum variants kept with `#[allow(dead_code)]`** — `ErrorCode::{EventTimeFieldMissing, EventTimeFieldWrongType}` and `DiffReason::EventTimeFieldMismatch` kept for wire-codec stability rather than deleted.
