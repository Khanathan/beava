# PLAN 22-01: REGISTER JSON consumer + aggregation dispatch

## Goal

Replace the v2.0 REGISTER JSON parser with a v0-shaped parser that consumes the payload emitted by `python/tally/_serialize.py` verbatim, and wire it into the existing pipeline engine. After this plan the Rust engine can accept every descriptor kind (StreamSource, TableSource, StreamDerivation op-chain, TableDerivation op-chain, TableDerivation with aggregation, join/union stubs that return NotImplemented for now), extract group_by keys, instantiate operator objects for all 16 AggOp types, and persist them in the existing `StreamEntityState` structure. No new operator math ships in this plan — it is the spine that plans 22-02 and 22-03 hang operator implementations off of.

## Success Criteria (observable, testable)

1. `V0RegisterPayload::parse(bytes)` succeeds on every payload `_serialize.py` emits for the 16 AggOp types; failure cases return a `TallyError::InvalidRegisterPayload { reason: String }` with a pointer to the offending field name.
2. Legacy v2.0 payloads (top-level `features: []`, `kind` missing, or `aggregation` absent on a TableDerivation) are rejected with the named error — no silent acceptance.
3. `register_v0(payload)` dispatches to five branches (source / op-chain / aggregation / join-stub / union-stub) and records each registered stream/table in the existing `PipelineEngine` registry.
4. For the aggregation branch: each `features[i]` produces an `OperatorState` variant via `build_operator`; the existing `StreamEntityState.operators` vector receives them on the first event for a key.
5. On PUSH, the group_by keys are read off the event, concatenated into the entity key (`"v1|u123"` single-key, `"v1|u123|m456"` composite — reusing the existing key-encoding if present in `StateStore`, otherwise a small new helper lives in `register.rs`), and the matching stream's operators all get `.push(event, enrichment, now)`.
6. GET against a Table emits a row containing the `group_by` keys + one field per aggregation feature; values come from each operator's `.read(now)`.
7. `python/tests/test_v0_register_roundtrip.py` passes: full pipeline defined in Python, `App.register(...)` succeeds, PUSH returns features, GET returns a row.
8. `cargo test --test test_register_json_v0` passes (all 16 op types × valid payload + 4 rejection cases).
9. No code path remains in the engine that parses the pre-Phase-21 v2.0 REGISTER payload. `grep -rn 'legacy_register\|v2_register\|per_stream_features'` returns 0 matches in `src/`.

## Implementation Steps

1. **Enumerate the payload contract.** Read `_serialize.py` and `_agg_ops.py::to_json` carefully; write `V0RegisterPayload` as a serde-tagged enum with the five variants:
   - `{kind:"stream", ...}` source — fields dict, optional history_ttl
   - `{kind:"table", mode, key_field|key_fields, ...}` source — fields dict, optional entity_ttl
   - `{kind, ops:[...]}` op-chain derivation — fields, depends_on
   - `{kind:"table", aggregation:{source, keys:[...], features:[...]}, ...}` aggregation
   - `{kind, join:{...}}` or `{kind:"stream", union:{sources:[...]}}` stubs that record the descriptor but return `NotImplemented` on PUSH (joins in 23, union in 22-03 or 23)
2. **Define `AggregationFeature`.** Serde struct matching every field `AggOp.to_json()` can emit: `name, type, supports_retraction, field?, window?, where?, bucket?, n?, quantile?, half_life?, k?, exact_threshold?, hybrid_alpha?, hybrid_precision?, hybrid_width?, hybrid_depth?`. All hybrid params flattened as top-level keys per the SDK contract.
3. **Write `build_operator(feat: &AggregationFeature, default_bucket_for: fn(&str)->Duration) -> Result<OperatorState>`.** Large match on `feat.type`:
   - `"count"` → existing `CountOp`-backed variant
   - `"sum" | "avg" | "min" | "max"` → existing variants (reuse what's in `operators.rs`)
   - `"variance" | "stddev"` → new `Variance`/`Stddev` variant stubs (bodies in 22-02)
   - `"percentile" | "count_distinct" | "top_k"` → new stubs (bodies in 22-03); capture `exact_threshold` + hybrid params on the struct
   - `"first" | "last" | "first_n" | "last_n" | "ema" | "lag"` → new stubs (bodies in 22-02)
   - Unknown → `InvalidRegisterPayload`
4. **Extend `OperatorState` enum (interface-first).** Add the 11 new variants with minimal fields matching 22-CONTEXT.md's §"State store integration". Bodies left as `todo!()` behind `#[allow(clippy::todo)]` but return `FeatureValue::Null` from `read`. Snapshot `SerializableOperatorState` gets matching variants — serde-compatible and round-trip-safe (22-03 validates with a snapshot test).
5. **Default bucket granularity helper.** Per 22-CONTEXT.md: 1 minute for windows ≥ 1h, 1 second for windows < 1h. Honors optional explicit `bucket` from the payload. Centralized in `register.rs::default_bucket`.
6. **Key extraction.** For aggregation descriptors, group_by keys are read off the event JSON. Single-key: plain string. Composite: join with `|` using the existing `EntityKey` encoding (reuse whatever `StateStore` already does for composite keys; if nothing exists yet, add a `fn encode_group_by(keys: &[String], event: &Value) -> Result<EntityKey>` in `register.rs`).
7. **Wire into `PipelineEngine`.** Replace the existing `register()` method with `register_v0(payload: V0RegisterPayload)`. The aggregation branch constructs a `FeatureDef` per feature and stores them on the Table descriptor's entry. On first event for a (stream_name, entity_key) pair, operators are materialized from `FeatureDef → OperatorState`.
8. **Burn the v2.0 bridge.** Delete:
   - The old `FeatureDef`'s fields that were exclusive to v2.0 (e.g., `backfill: bool` survives if still referenced; review per-variant)
   - The v2.0 REGISTER parsing path in `src/server/tcp.rs`
   - Any helper named `legacy_*` or `v2_*` in the engine
   - grep-gate this in the `<verify>` block of the tests below.
9. **Write Rust tests** (`tests/test_register_json_v0.rs`):
   - One test per AggOp type: builds a payload matching what Python would emit, parses, confirms the OperatorState variant is correct.
   - Rejection tests: legacy top-level `features`, missing `aggregation` on `TableDerivation` with no `ops`, unknown `type` tag, malformed window string.
10. **Write Python round-trip test** (`python/tests/test_v0_register_roundtrip.py`):
    - Spin up the server (reuse the existing test harness pattern in `python/tests/`); define a pipeline with at least 3 AggOps; `App.register(...)`; `App.push(...)`; `App.get(...)`; assert the row has the expected shape. For unimplemented ops in this plan, assert `FeatureValue::Null` (22-02/03 tighten).

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| `OperatorState` enum bloat breaks snapshot format | Serde tag-per-variant already isolates variants. Add one snapshot round-trip test per new variant (even with stub state) in `tests/test_snapshot.rs`. |
| Group_by key encoding diverges from existing `EntityKey` convention | First read `src/state/store.rs` and `src/types.rs` to find existing encoding; reuse verbatim. Document the chosen encoding in `register.rs` doc-comment so 22-02 and 23 can match. |
| Burning v2.0 path breaks an existing test | Expected — the v2.0 tests in `python/tests/` not ported yet live under `test_v0_*`. Old tests belong to the 744-test suite to be migrated in Phase 26. If any non-v0 test currently runs and references the old payload, mark it with `@pytest.mark.skip(reason="v2.0 payload removed, Phase 26 migration")`. |
| `depends_on` topological ordering not enforced on the Rust side | Python's `_collect_registrations` already emits topologically ordered frames. Rust can assume it; validate order at parse-time only if `depends_on[i]` references an unregistered name. |
| Phase 21 serializer emits fields the parser doesn't know about | Use `serde(deny_unknown_fields)` on the payload structs — forces 22-01 to evolve lockstep with Phase 21 changes. Better to fail loudly than silently drop. |
