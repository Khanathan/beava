<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TCP SET/MSET → Table↔Table cascade | Upsert on input table triggers synchronous cascade to output table; attacker could craft high-fanout MSET to amplify work |
| Tombstone propagation | Delete on input deletes output — correctness-critical, must not leak deleted data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-08 | DoS | MSET amplification via Table↔Table cascade | mitigate | Table↔Table merge is O(1) per key. Phase 22's MSET chunked-yielding already bounds the amplification factor to the chunk size (1024). Confirmed with `join_small_1c` benchmark cell (Task 3) |
| T-23-09 | Info Disclosure | Stale Joined row after tombstone on either input | mitigate | Task 1 test `tt_inner_tombstone_right_deletes_output` and `tt_left_tombstone_right_nulls_right_fields` explicitly verify tombstone propagation; leak would be caught by these tests |
| T-23-10 | Tampering | Partial-key smuggling | mitigate | Translator + SDK both reject partial-key joins. Engine is defense-in-depth; partial-key is blocked at REGISTER with a clear error |
| T-23-11 | Elevation | Cycle in Table↔Table DAG causing infinite cascade | mitigate | Cycle guard in translator (Task 1 step 3) rejects at REGISTER time |
</threat_model>
