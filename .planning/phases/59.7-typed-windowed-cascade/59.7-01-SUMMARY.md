## Deviations from Plan

### Rule 3 — blocking scope fixes (corrected semantics from plan text)

**1. [Rule 3 - Blocking] `is_typed_cascade_compatible` tightening is a no-op this wave**

- **Found during:** Task 2 plan-step 6 review (`grep` for `FeatureDef::Count`).
- **Issue:** Plan Task 2 Action 6 specified tightening `is_typed_cascade_compatible` via `FeatureDef::Count { window, .. } if window.is_none() || ...`. But `FeatureDef::{Count,Sum,Avg,Min,Max}` carry `window: Duration` (not `Option<Duration>`) — see `src/engine/pipeline.rs:30-68`. There is no `window.is_none()` branch to tighten.
- **Root cause:** Plan-author assumption mismatch with the enum shape. Phase 59.6 existing tests (`tests/typed_aggregation_parity.rs`) rely on `big_window()` (1 year) to make windowed Value-path ops behave like unwindowed typed ops — the "is there a typed windowed impl" question is *always* "there is one" for these variants.
- **Fix:** No change to `is_typed_cascade_compatible` this wave. Behavior preserved (every Count/Sum/Avg/Min/Max/Last/First is "compatible"). W4 cascade walker is the first consumer that actually needs to distinguish windowed-vs-unwindowed ops at the op-construction step (not the cascade predicate).
- **Files modified:** none (intentional no-op documented here).

**2. [Rule 3 - Blocking] `run_typed_agg_step` drives-windowed wiring deferred to W4**

- **Found during:** Task 2 plan-step 5 review.
- **Issue:** Plan Task 2 Action 5 asked to add a second loop to `run_typed_agg_step` calling `op.update_windowed(shard, ...)` after `update_typed`. That would change the behavior of `run_typed_agg_step` — today it drives unwindowed typed ops (Count/Sum/Avg per `typed_aggregation_parity.rs`); adding an unconditional `update_windowed` call would be wasted work for those ops.
- **Root cause:** The simple ops' default `update_windowed` is a no-op (as planned), so the call is semantically safe — but it still incurs per-op virtual dispatch overhead on the hot path. Since the only consumer this wave is the parity harness (which calls the ops through Shard directly), wiring `run_typed_agg_step` adds overhead with no correctness benefit. W4's cascade walker is the first consumer that needs the combined walk; deferring there keeps the hot path clean for this wave.
- **Fix:** Left `run_typed_agg_step` unchanged. Documented in the windowed-op module docstring: "Wave 1 keeps these trait defaults unused by `run_typed_agg_step` today — the parity harness drives the windowed op/ring pair directly, mirroring how `typed_aggregation_parity.rs` drives unwindowed ops. Wave 4 wires the windowed path into the cascade walker."
- **Files modified:** none (intentional no-op, documented inline + here).

**3. [Rule 3 - Blocking] `create_typed_op` factory unchanged this wave**

- **Found during:** Task 2 plan-step 8 review.
- **Issue:** Plan Task 2 Action 8 asked to update `create_typed_op` to emit `CountOpTypedWindowed` when `window.is_some() && bucket.is_some()`. But (a) `window`/`bucket` are `Duration` not `Option<Duration>`; and (b) the typed-op factory route today is only consumed by `run_typed_agg_step` — which the parity tests don't drive through. Altering the factory would change how existing Phase 59.6 `typed_aggregation_parity.rs` constructs its ops and could regress the unwindowed parity tests.
- **Fix:** Factory untouched this wave. W4 owns the factory gate (swap to windowed variants when the cascade walker consumes them).
- **Files modified:** none.

**5. [Rule 2 - Correctness] Sum-parity tests use a companion count ring**

- **Found during:** Task 2 parity test body authoring.
- **Issue:** Value-path `SumOp::read` emits `Missing` when `event_count.sum_all() == 0` (no events in window). Typed path's `SumOpTypedWindowed{I64,F64}::read` always returns `Int(0)` / `Float(0.0)` since there's no separate count.
- **Fix:** Parity test drives a companion `CountOpTypedWindowed` keyed to a separate `op_idx` and reads it alongside the Sum op: `cnt == Missing → Missing; else → Int/Float(sum)`. This matches Value-path semantics exactly. The W4 cascade walker will pair SumOp + implicit CountOp at op-construction time so callers don't need to plumb the companion manually.
- **Files modified:** tests/typed_windowed_aggregation_parity.rs.
