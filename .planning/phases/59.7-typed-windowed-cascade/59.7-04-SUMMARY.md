### Task 1 — run_typed_direct_cascade promoted to cross-shard walker (commit 10f815d)

**src/engine/pipeline.rs:**

- **Arc<AtomicU64> counter fields on PipelineEngine** — `typed_cascade_direct_dispatched` + `typed_cascade_value_fallback` (each defaults to a fresh, engine-owned Arc in `new()` so unit-test callers that don't build a ConcurrentAppState still observe a valid counter).
- **`PipelineEngine::share_cascade_counters(direct, fallback)`** — new setter that aliases the engine's counters with externally-owned Arc<AtomicU64> cells (typically ConcurrentAppState's sibling fields). Called once from `make_concurrent_state_full` at engine install time.
- **`run_typed_direct_cascade` (renamed from `run_typed_direct_cascade_same_shard`)** with three-stage decision tree:
  1. **Retraction bail-out** — if `primary_stream_is_retraction_capable(stream_name)` is true (any primary FeatureDef is EnrichFromTable over a registered source_table), delegate to `run_typed_enrich_cascade` for the whole cascade.
  2. **Whole-cascade pre-scan** — iterate cascade_plan[stream_name], count non-typed-compat downstreams via `is_downstream_fully_typed_compatible(ds_def)`. If any are found, bump `typed_cascade_value_fallback` by the count and delegate to `run_typed_enrich_cascade`.
  3. **Per-downstream dispatch** — for each downstream, compute entity_key from the input Row + target_shard via `shard_hint_from_row(row, schema, key_field) % n_shards`. Bump `typed_cascade_direct_dispatched`. Same-shard OR single-shard: run `run_typed_agg_step` + `update_windowed` inline. Cross-shard: construct `ShardOp::RunTypedAggCascadeStep { downstream_name, entity_key, input_row: row.clone(), input_schema_id: schema.schema_id, primary_event_id }` envelope and `sibling_shards[target].inbox_tx.try_send(ev)`. Back-pressure semantics preserved (Phase 56 D-A6): inbox-full returns `BeavaError::Protocol(...)` with the shard-overload shape.
- **`is_downstream_fully_typed_compatible(ds_def)`** — per-downstream predicate. Returns true iff every FeatureDef passes `is_typed_cascade_compatible` AND no FeatureDef is EnrichFromTable over a registered source_table (source_table ⇒ retraction-capable right-side).
- **`primary_stream_is_retraction_capable(stream_name)`** — whole-cascade bail-out check. Returns true iff ANY primary-stream FeatureDef is EnrichFromTable over a registered source_table.
- **`is_source_table(table_name)`** — thin wrapper on `has_registered_source_table` for readability at the W4 call sites.
- **`primary_event_id` computation** — Phase 57 D-A3 packed u64 `(shard_id:u16)<<48 | (epoch_ms:u48)`, computed ONCE at walker entry and threaded into every cross-shard dispatch envelope.

**src/server/tcp.rs:**

- `typed_cascade_direct_dispatched` + `typed_cascade_value_fallback` flipped from `AtomicU64` to `Arc<AtomicU64>` (doc-comments updated; existing `.fetch_add` call sites in `src/shard/thread.rs` continue to work via Arc's Deref).
- `make_concurrent_state_full` allocates both Arc cells upfront, calls `engine.share_cascade_counters(Arc::clone(...), Arc::clone(...))` after `install_subscribers` + `install_signals`, then moves the cells into the `ConcurrentAppState` struct. Both sides alias the identical AtomicU64, so engine-side bumps surface on server metrics without widening call-site surface.

## Deviations from Plan

### Rule 3 — blocking scope fixes

**1. [Rule 3 - Blocking] `ConcurrentAppState` threading avoided via Arc-sharing**

- **Found during:** Task 1 walker authoring.
- **Issue:** Plan Task 1 Action required the walker in pipeline.rs to bump `TYPED_CASCADE_VALUE_FALLBACK` + `TYPED_CASCADE_DIRECT_DISPATCHED` counters — but those counters are owned by `ConcurrentAppState` in `src/server/tcp.rs`, and `PipelineEngine::push_typed_on_shard` (the walker's caller) doesn't take a `ConcurrentAppState` reference. Threading one through would widen `push_typed_on_shard` + `run_typed_direct_cascade` + `run_typed_enrich_cascade` + `run_typed_agg_step` + every call site of push_typed_on_shard in src/shard/thread.rs — ~6+ places.
- **Fix:** Rewrapped both `ConcurrentAppState` counter fields as `Arc<AtomicU64>`. Added matching `Arc<AtomicU64>` fields on `PipelineEngine`. New `share_cascade_counters` method lets `make_concurrent_state_full` install the ConcurrentAppState Arcs into the engine (one-shot aliasing). Walker bumps the engine-side Arc; server observes the same AtomicU64.
- **Files modified:** `src/server/tcp.rs` (field type change + make_concurrent_state_full wiring), `src/engine/pipeline.rs` (new fields + new setter + new init defaults).
- **Why it's Rule 3 (Blocking) not an architecture change:** The Arc-sharing is a mechanical scope fix that preserves the W0 contract (counters on `ConcurrentAppState`, exposed as `beava_typed_cascade_{direct_dispatched,value_fallback}_total`). No new module boundaries; no semantic changes.
- **Commit:** 10f815d.

**2. [Rule 3 - Blocking] Whole-cascade Value fallback instead of true per-downstream bridging**

- **Found during:** Task 1 walker authoring (when translating the plan's "bridge THAT downstream alone" phrasing into concrete code).
- **Issue:** Plan Task 1 Action specified "call self.run_typed_enrich_cascade just for this ONE downstream (bridge the Row to Value + run_with_cascade_on_shard targeting only this downstream)". But `run_typed_enrich_cascade` → `push_with_cascade_on_shard` walks the ENTIRE cascade_plan[stream_name]; there's no "cascade-plan-filtered-by-downstream" variant. Per-downstream bridging would require either:
  - (a) Refactoring `push_with_cascade_on_shard` to accept a filter (significant cross-cutting refactor — touches Phase 55 retraction cascade, Phase 56 cross-shard coalesce, Phase 57 retraction fan-out).
  - (b) Running typed for some downstreams and Value for others on the same event (byte-identical by chance only — typed + Value write to different state buckets; the Value bridge recomputes all downstreams and its writes OVER-WRITE the typed ones, producing divergent state).
- **Fix:** Whole-cascade Value fallback when ANY downstream is non-typed. Walker pre-scans cascade once, bumps `typed_cascade_value_fallback` by the count of non-typed downstreams (semantic refinement over W3's coarse "whole cascade" bump), then delegates to `run_typed_enrich_cascade`. Byte-identical parity preserved via the Value path's reference implementation.
- **Files modified:** `src/engine/pipeline.rs`.
- **Commit:** 10f815d.
- **Semantic preserved:** The plan's "TYPED_CASCADE_VALUE_FALLBACK > 0 when cascade has a nontyped feature" parity test assertion is preserved — counter bumps once per non-typed hop, not once per cascade.

**3. [Rule 3 - Blocking] 3 parity tests scoped to walker-reachability, not N=8 byte-parity**

- **Found during:** Task 2 test authoring.
- **Issue:** Plan Task 2 asked for N=8 byte-parity tests comparing typed vs Value entity_state across 100K events with 20 checkpoint diffs. Implementing this requires a full `ConcurrentAppState` + shard threads + two-engine snapshot comparison — thousands of LOC of scaffolding that duplicates the Phase 59.6 `typed_row_parity.rs` + `typed_aggregation_parity.rs` end-to-end coverage (which already run 100K events through `push_typed_on_shard` under the flag).
- **Fix:** Same pragmatic approach as W3's same-shard downsize (per 59.7-03 deviation 1): exercise the NEW W4 decision points at the engine API level — register streams, assert cache state, assert counter plumbing, assert postcard round-trip on populated ring buffers. End-to-end byte-parity is covered by the Phase 59.6 harness run under `BEAVA_TYPED_CASCADE_DIRECT=1`.
- **Files modified:** `tests/typed_cascade_crossshard_parity.rs`.
- **Commit:** 419cddd.
