# Phase 59.7: typed-windowed-cascade — Context

**Gathered:** 2026-04-21

<domain>
## Phase Boundary

Close the two remaining gaps that Phase 59.6 left open:

1. **Typed windowed aggs** — today's `src/engine/operators_typed_aggs.rs` implements
   only unwindowed (lifetime) Count/Sum/Avg/Min/Max/Last/First. The fraud-pipeline
   and most production workloads use `window="1h"` / `"24h"` / `"7d"` on every
   aggregation. Without windowed typed variants, the typed fast path can't be
   used on any realistic workload — it'd silently produce lifetime counts
   instead of windowed counts.

2. **Typed cascade-direct** — `run_typed_enrich_cascade` today is a bridge
   that calls `row_to_value(&row, schema)` then `push_with_cascade_on_shard`
   (the Value cascade body). Even when every downstream feature is typed-
   compatible, the cascade walk itself runs through `EntityState` / Value.
   Needs to become a real typed walk that writes `entity_state_typed`
   directly (same-shard inline, cross-shard via a new `ShardOp` variant).

**Goal:** Make the production hot-path (fraud-pipeline benchmark at
`window="1h"/24h"/7d"` × 47 features across 6 downstream tables) run
fully on the typed-row path. Eliminate the ~17.8% `push_internal_on_shard`
cost observed in the state-inmem samply profile and the ~17.4% `RawVecInner::reserve`
+ 7.7% `Vec::drop` observed in the fjall profile (together ~25% of CPU,
both caused by `EntityState`'s `Vec<f64>` RingBuffer churn).

</domain>

<gap_1_typed_windowed_aggs>
## Gap 1: Typed Windowed Aggregations

### Problem

Current `src/engine/operators_typed_aggs.rs` has:
```rust
pub struct CountOpTyped {
    pub name: String,
    pub output_offset: u16,
}
```

No `window`, no `bucket`, no ring buffer. `update_typed` just increments a
single `i64` at `output_offset`. This produces lifetime counts.

The parity tests (`tests/typed_aggregation_parity.rs`) pass because they use
`big_window()` (effectively infinite), which makes windowed Value-path ops
behave identically to unwindowed typed-path ops — but this masks the fact
that windowing isn't actually implemented.

`is_wave4_typed_compatible(fd: &FeatureDef)` returns `true` for
`FeatureDef::Count { window, .. }` regardless of window value — it's a
structural check, not a semantic one. If Part B wires the typed cascade
direct without fixing this, any windowed feature would produce wrong
results.

### Approach

Typed windowed agg state stored in packed Row. Two options:

**Option A — inline ring buffer slots in the state Row:**
- Reserve N `i64` or `f64` slots per windowed op, where N = ceil(window / bucket)
- Plus head/tail pointers (2× `u32` = 8 bytes)
- Plus last_event_time (`i64`, 8 bytes)
- Per-op overhead: ~24 + N × 8 bytes
- For 1h window + 1m bucket = 60 slots = ~500 bytes/op
- 25 ops × 500 = 12.5KB per entity. Too big.

**Option B — out-of-band ring buffers via SideBand-style map:**
- `Shard::entity_ringbuffers_typed: AHashMap<(String, String, u16), TypedRingBuffer>`
- Keyed by (downstream_name, entity_key, op_index)
- Each TypedRingBuffer: packed `Vec<i64>` or `Vec<f64>` + head/tail/last_event_time
- Matches the SideBand pattern (Wave 6 D-C1) for sketch ops
- Per-event: one hashmap lookup per windowed op (overhead ~50 ns)

**Recommendation:** Option B. Matches existing pattern, avoids packed-row bloat,
keeps hot path clean.

### Deliverables

1. `src/engine/operators_typed_aggs_windowed.rs` — new module
2. Windowed twins of Count, Sum, Avg, Min, Max (i64 + f64 specializations)
3. `TypedRingBuffer` struct (packed Vec, head/tail indices, last_event_time)
4. `Shard::entity_ringbuffers_typed` field
5. V11 snapshot: serialize/deserialize `entity_ringbuffers_typed`
6. Update `is_wave4_typed_compatible` / `is_typed_agg` to return true only
   when a windowed op has a typed windowed impl (not structural match on
   enum variant)
7. Parity tests: small `window=5s`, `bucket=1s` where events expire —
   verify typed vs Value produce byte-identical counts at each checkpoint

</gap_1_typed_windowed_aggs>

<gap_2_typed_cascade_direct>
## Gap 2: Typed Cascade-Direct (was Part B in session)

### Problem

Current `run_typed_enrich_cascade` (src/engine/pipeline.rs:3832):

```rust
fn run_typed_enrich_cascade(...) -> Result<FeatureMap, BeavaError> {
    let value = crate::engine::schema::row_to_value(&row, schema);
    // ... touch typed op types for grep invariant ...
    self.push_with_cascade_on_shard(stream_name, &value, shard, ...)
}
```

It's a bridge. Every typed event gets converted back to Value and walked
through the Value cascade (`push_with_cascade_on_shard`). The typed ingest
was free; the cascade isn't.

### Approach

New cascade walker that:
1. Computes `primary_event_id` (Phase 57 TPC-CORR-10 requirement)
2. Runs primary stream's typed ops against `entity_state_typed` (same-shard)
3. For each downstream in cascade_plan:
   a. If every feature in downstream is typed-compatible (windowed typed aggs
      from Gap 1 now available) → dispatch typed
   b. Extract downstream entity_key from input Row using downstream's
      declared key_field + typed schema accessor
   c. Compute `target_shard = shard_hint_from_row(row, key_field) % N`
   d. If same-shard: `run_typed_agg_step` inline
   e. If cross-shard: dispatch new `ShardOp::RunTypedAggCascadeStep` to target
4. For downstreams with any Value-only feature → fall back to Value cascade
   for that downstream only (not whole cascade)
5. Merge same-shard + cross-shard FeatureMaps

### New ShardOp variant

```rust
/// Phase 59.7 (TPC-PERF-11 extension): dispatch typed agg ops on a
/// downstream-owned shard. Target handler runs the downstream's typed
/// agg list against entity_state_typed[(downstream_name, entity_key)].
/// Does NOT run further cascade — caller is the cascade walker on the
/// input shard, which has already decided this downstream is a terminal
/// typed agg step.
RunTypedAggCascadeStep {
    downstream_name: String,
    entity_key: String,
    input_row: Row,
    input_schema_id: SchemaId,
    primary_event_id: u64,  // Phase 57 TPC-CORR-10
}
```

Handler in `src/shard/thread.rs` dispatch:
```rust
ShardOp::RunTypedAggCascadeStep { ... } => {
    let engine = state.engine.read();
    let ops = engine.build_typed_agg_ops_for(&downstream_name)?;
    let input_schema = engine.schema_registry.get_by_id(input_schema_id);
    let state_schema = engine.get_typed_state_schema(&downstream_name);
    engine.run_typed_agg_step(
        &downstream_name, &entity_key, &input_row,
        input_schema, &ops, state_schema, &mut shard, now,
    );
    // No response — this is fire-and-forget within cascade
}
```

### Correctness preservation

- **Retraction** (Phase 57): tombstones / deletes on inputs still trigger
  retractions. For Gap 2 V1, defer retraction to Value fallback — any
  cascade involving retractable input (source table with deletes) routes
  through Value.
- **Cross-shard consistency:** `ShardOp::RunTypedAggCascadeStep` lands in
  the target shard's SPSC inbox. Ordering preserved per-shard. No
  coordination needed since each downstream entity owns its own state.
- **Snapshot consistency:** `entity_state_typed` already in V11 (Wave 5);
  `entity_ringbuffers_typed` (Gap 1) must be added to V11 serialization.

### Deliverables

1. New `ShardOp::RunTypedAggCascadeStep` variant + dispatch arm
2. `PipelineEngine::build_typed_agg_ops_for(stream_name)` factory
3. `PipelineEngine::run_typed_direct_cascade` — the new walker
4. Wire into `push_typed_on_shard` — prefer typed-direct when eligible
5. Feature-gate via `BEAVA_TYPED_CASCADE_DIRECT=1` env or config until
   stable
6. Extended parity tests: windowed aggs + cross-shard dispatch
7. Perf gate: re-run fraud-pipeline bench, target `push_internal_on_shard`
   dropping to <1% of CPU on inmem; ~5-10% improvement on fjall

</gap_2_typed_cascade_direct>
