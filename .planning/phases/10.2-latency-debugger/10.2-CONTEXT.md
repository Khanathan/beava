# Phase 10.2 — Latency Debugger: CONTEXT.md

## Phase Goal

Users can observe and debug per-command latency through percentile histograms (p50/p95/p99) broken down by TCP command (PUSH/GET/SET/MSET) and stream, surfaced as a contextual latency view in the Debug UI drill-in panel and a `/debug/latency` JSON endpoint on the HTTP management port.

## Success Criteria

1. GET /debug/latency returns JSON with per-TCP-command p50/p95/p99 + per-stream breakdown
2. Live per-command latency histograms refreshing at 1 Hz in the Debug UI drill-in panel
3. Slow-query view listing 20 slowest observed requests per command + originating stream
4. Latency tracking adds NO measurable overhead to PUSH hot path (p99 stays < 100us per Phase 6 budget)
5. Estimator is memory-bounded per stream regardless of request rate

## Decisions

### Decision 1: Estimator — Bucketed Histogram

**Choice:** Fixed-bin bucketed histogram with ~30 log-spaced bins covering 1us-10ms range.

**Rationale:**
- Zero heap allocations on the hot path — single array index + increment per sample
- Zero external crate dependencies — pure Rust, ~50 LOC, preserves single-binary ethos
- Matches the ring-buffer pattern already used in `ThroughputTracker` (src/server/throughput.rs)
- ~100-400 bytes memory per stream — smallest of the three options
- Sufficient accuracy for p50/p95/p99 over the expected latency range
- Trivial JSON serialization (array of bin counts + bin edges)

**Rejected alternatives:**
- HDR histogram (`hdrhistogram` crate): higher accuracy but adds external dependency, ~2-16KB per stream
- t-digest (`tdigest` crate): excellent at extreme tails but heap allocations during merges violate hot-path budget, overkill for p50/p95/p99

### Decision 2: UI Surface — Hybrid Drill-in Panel

**Choice:** Contextual latency display in the existing drill-in panel:
- **No node selected:** drill-in panel placeholder becomes a global latency dashboard showing command-level histograms (PUSH/GET/SET/MSET) with p50/p95/p99 and slow-query list
- **Stream node selected:** new 6th section in drill-in panel shows per-stream PUSH latency histogram + per-stream slow queries

**Rationale:**
- Reuses existing panel real estate — no new UI paradigms, no modals, no tabs
- Matches Phase 10.1 interaction model: click to drill in, see everything about that stream
- Global stats are always accessible (just deselect the node)
- Two render paths but same component structure

**Rejected alternatives:**
- Drill-in only (option 1): loses command-level stats for non-stream-attributed commands
- Edge tooltips (option 2): only covers PUSH between streams, poor discoverability
- Header overlay (option 3): disconnected from drill-in model, adds a new UI paradigm

### Decision 3: Measurement Scope — Wire-to-Wire, All Commands

**Choice:**
- **Scope:** All TCP commands measured: PUSH, GET, SET, MSET. REGISTER excluded (too rare).
- **Measurement point:** Wire-to-wire — from frame-received-complete to response-flush-start. This captures parse + dispatch + engine execution + response serialization.
- **Attribution:**
  - PUSH: per-stream attribution (request carries stream name natively)
  - GET, SET, MSET: command-level bucket only, no per-stream breakdown (GET is key-scoped not stream-scoped; SET/MSET bypass pipeline)

**Rationale:**
- Wire-to-wire gives the most useful debugging signal — users see what the client sees
- PUSH per-stream is natural and most valuable (hot path debugging)
- GET/SET/MSET per-stream attribution would require complex fan-out accounting with little debugging value
- REGISTER is a one-time startup op, tracking it adds noise

### Decision 4: Slow-Query Capture + Retention Window

**Choice:**
- **Histograms:** Rolling 5-minute window, matching ThroughputTracker's ewma_5m cadence. Bins reset/decay on a 5-minute rolling basis.
- **Slow-query view:** Min-heap of top 20 slowest requests per command type. Bounded, O(log 20) ≈ O(1) insert cost.
- **Slow-query record fields:** timestamp, command type, stream name (PUSH only, null for others), latency_us, key_preview (first 32 chars of entity key, for debugging context)

**Rationale:**
- 5-minute window matches existing throughput cadence — consistent mental model for users
- Min-heap is the simplest bounded data structure for "top N" tracking — no reservoir sampling complexity
- 20 entries per command is enough to spot patterns without unbounded growth
- Key preview (truncated) aids debugging without storing full payloads

## Constraints Carried Forward

These are non-negotiable and inherited from Phase 10/10.1:

1. **No `.await` across AppState mutex lock.** Latency measurement hooks into `handle_sync_command` — must follow the lock-process-unlock pattern.
2. **p99 PUSH < 100us budget.** Must benchmark before/after to prove no regression.
3. **No XSS sinks.** All new frontend code uses `.textContent` / d3 `.text()`. `app_js_has_no_innerhtml_or_eval_sinks` test must stay green.
4. **No DAG re-render on tick.** Any 1 Hz latency updater updates text in place, never calls `render(inner, g)`.
5. **549+ tests passing.** New latency tests add to this count; nothing breaks.
6. **Keep `push_latency_seconds` scalar** until `/debug/latency` endpoint is working and tested.
7. **Do NOT fold Phase 10.1 deferred gaps** (keyless drill-in, view entity lookup, per-operator memory) into this phase.
8. **Design tokens verbatim** from Phase 10 UI-SPEC §13.1 — spacing (4/8/16/24/32/48), font sizes (12/14/20/24), weights (400/600), dark-mode palette.

## Implementation Hooks

**Backend insertion points:**
- `src/server/tcp.rs` — `handle_sync_command` dispatch: wrap each command arm with `Instant::now()` / `.elapsed()` and record into the latency tracker
- `src/server/http.rs` — add `/debug/latency` handler following the lock-once-then-build-JSON pattern from `/debug/topology`
- New `src/server/latency.rs` — `LatencyTracker` struct (per-command bucketed histograms + per-stream PUSH histograms + slow-query heaps). Lives on `AppState`, updated inside the mutex lock.

**Frontend insertion points:**
- `src/server/ui/app.js` — `renderDrillInPanel()`: add latency section renderer for stream nodes; modify placeholder state to render global latency dashboard
- New `/debug/latency` fetch in the 1 Hz poll loop (or separate cadence if needed)

**Reusable patterns:**
- `ThroughputTracker` (src/server/throughput.rs) — structural template for per-stream + global tracking
- `renderThroughputSection()` in app.js — template for the new latency section renderer
