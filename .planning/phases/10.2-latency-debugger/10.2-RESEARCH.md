# Phase 10.2: Latency Debugger - Research

**Researched:** 2026-04-10
**Domain:** Per-command latency histograms (bucketed), HTTP endpoint, Debug UI drill-in panel
**Confidence:** HIGH

## Summary

Phase 10.2 adds a latency debugging subsystem to Tally's Debug UI and management API. The backend component is a new `LatencyTracker` struct (mirroring `ThroughputTracker`) that maintains per-command bucketed histograms, per-stream PUSH histograms, and bounded slow-query heaps. The frontend adds a 6th drill-in section for stream nodes and repurposes the empty-panel placeholder as a global latency dashboard.

The implementation is straightforward: ~30 log-spaced histogram bins covering 1us-10ms, zero heap allocations on the hot path (single array index + increment per sample), and a new `/debug/latency` JSON endpoint following the existing `debug_throughput`/`debug_topology` handler pattern. The frontend follows the `renderThroughputSection` granular-update pattern -- section-level swap via `data-section` attribute, never re-renders the DAG.

**Primary recommendation:** Build `LatencyTracker` as a standalone `src/server/latency.rs` module with the same structural pattern as `throughput.rs`. Use a two-buffer swap strategy for 5-minute rolling windows. Fetch latency data in the existing 1 Hz throughput tick to avoid a second polling interval.

### Pattern 1: Bucketed Log-Spaced Histogram

**What:** Fixed-size array of u64 counters with log-spaced bin edges covering 1us-10ms.

**Bin edge formula:** Use `10^(i * step)` where step = `log10(10000) / (N-1)` and N = 30 bins. This spaces bins geometrically from 1us to 10000us (10ms).

```rust
// [ASSUMED] — formula is standard log-spaced histogram design
const NUM_BINS: usize = 30;

// Precomputed bin edges in microseconds (1us to 10ms, log-spaced)
// Edge i = 10^(i * 4.0 / 29) microseconds
// Bin 0:  [0, 1.36us)
// Bin 1:  [1.36us, 1.86us)
// ...
// Bin 14: [~56us, ~77us)    <-- p99 target zone (100us)
// Bin 15: [~77us, ~105us)
// ...
// Bin 29: [~7356us, +inf)
const BIN_EDGES: [f64; NUM_BINS + 1] = /* precomputed at compile time */;

struct Histogram {
    counts: [u64; NUM_BINS],
    total: u64,
}
```

**Memory per histogram:** 30 * 8 bytes (counts) + 8 bytes (total) = 248 bytes. [VERIFIED: matches CONTEXT.md "~100-400 bytes/stream" range]

**Recording a sample (hot path):**
```rust
// [ASSUMED] — standard binary search on sorted array
impl Histogram {
    #[inline]
    fn record(&mut self, micros: f64) {
        // Binary search for the right bin — O(log 30) = ~5 comparisons
        let bin = match BIN_EDGES.binary_search_by(|edge| {
            edge.partial_cmp(&micros).unwrap_or(std::cmp::Ordering::Less)
        }) {
            Ok(i) => i.min(NUM_BINS - 1),
            Err(i) => (i.saturating_sub(1)).min(NUM_BINS - 1),
        };
        self.counts[bin] += 1;
        self.total += 1;
    }
}
```

**Computing percentiles (read path, not hot path):**
```rust
// [ASSUMED] — standard percentile-from-histogram interpolation
impl Histogram {
    fn percentile(&self, p: f64) -> f64 {
        if self.total == 0 { return 0.0; }
        let target = (p / 100.0 * self.total as f64).ceil() as u64;
        let mut cumulative = 0u64;
        for i in 0..NUM_BINS {
            cumulative += self.counts[i];
            if cumulative >= target {
                // Linear interpolation within the bin
                let lower = BIN_EDGES[i];
                let upper = BIN_EDGES[i + 1];
                let bin_count = self.counts[i];
                if bin_count == 0 { return lower; }
                let overshoot = cumulative - target;
                let fraction = 1.0 - (overshoot as f64 / bin_count as f64);
                return lower + fraction * (upper - lower);
            }
        }
        BIN_EDGES[NUM_BINS] // overflow bin
    }
}
```

**When to use:** Every command recording in `handle_sync_command` and `handle_mset`.

### Pattern 2: Two-Buffer Swap for 5-Minute Rolling Windows

**What:** Instead of per-bin timestamps or sub-buckets with time-based expiry, use two histogram buffers (current + previous) and swap every 2.5 minutes. A query reads current + unexpired fraction of previous.

**Why this over alternatives:**
- Periodic full reset every 5 min: produces cliff edges -- latency drops to zero right after reset. [ASSUMED]
- Sub-buckets with time-based expiry (like window operators): complex, more memory, overkill for debug display. [ASSUMED]
- Two-buffer swap: smooth transitions, predictable memory (2x histogram), simple timer. [ASSUMED]

```rust
// [ASSUMED] — standard double-buffer EWMA/rolling pattern
struct RollingHistogram {
    current: Histogram,
    previous: Histogram,
    current_start: Instant,
    swap_interval: Duration, // 2.5 minutes
}

impl RollingHistogram {
    fn record(&mut self, micros: f64, now: Instant) {
        if now.duration_since(self.current_start) >= self.swap_interval {
            std::mem::swap(&mut self.current, &mut self.previous);
            self.current = Histogram::default();
            self.current_start = now;
        }
        self.current.record(micros);
    }

    fn snapshot(&self) -> Histogram {
        // Merge current + previous for smooth view
        let mut merged = Histogram::default();
        for i in 0..NUM_BINS {
            merged.counts[i] = self.current.counts[i] + self.previous.counts[i];
        }
        merged.total = self.current.total + self.previous.total;
        merged
    }
}
```

**Swap interval:** 2.5 minutes = 150 seconds. At any query time, the merged view covers 2.5 to 5 minutes of data, providing a smooth rolling window. [ASSUMED]

**SlowQueryHeap:**
```rust
// [ASSUMED] — standard BinaryHeap min-heap pattern with Reverse
use std::collections::BinaryHeap;
use std::cmp::Reverse;

struct SlowQueryRecord {
    timestamp: SystemTime,
    command: CommandKind,
    stream: Option<String>,  // Some for PUSH, None for GET/SET/MSET
    latency_us: f64,
    key_preview: String,     // first 32 chars of entity key
}

struct SlowQueryHeap {
    heap: BinaryHeap<Reverse<SlowQueryRecord>>,  // min-heap by latency
    capacity: usize,  // 20
}

impl SlowQueryHeap {
    fn insert(&mut self, record: SlowQueryRecord) {
        if self.heap.len() < self.capacity {
            self.heap.push(Reverse(record));
        } else if let Some(min) = self.heap.peek() {
            if record.latency_us > min.0.latency_us {
                self.heap.pop();
                self.heap.push(Reverse(record));
            }
        }
    }
}
```

**Key design note:** SlowQueryRecord contains a String for stream name and key_preview. These are heap-allocated. However, slow-query insertion only happens when a sample exceeds the current minimum of the top-20 heap. After warmup, most samples skip the insertion entirely (just a single comparison against the heap minimum). This is acceptable for the hot path. [ASSUMED -- verified by O(1) amortized argument: after 20 recordings, most new samples are faster than the 20th slowest and skip with a single f64 comparison]

### Anti-Patterns to Avoid

- **`.await` inside AppState mutex lock:** MSET timing must be recorded in a SEPARATE lock acquisition after `handle_mset` returns. Never hold the lock across the cooperative yield points. [VERIFIED: CONTEXT.md Constraint 1, tcp.rs architecture]

- **Heap allocation in histogram `record()`:** The histogram's hot-path method must be array index + increment only. No String allocation, no Vec push, no HashMap lookup. Slow-query recording can allocate (String for key_preview) but only when the sample exceeds the heap minimum -- amortized O(1). [VERIFIED: CONTEXT.md Decision 1]

- **innerHTML or XSS-unsafe sinks:** All new DOM writes MUST use `el()` helper (which uses `.textContent`) or d3 `.text()`. The `app_js_has_no_innerhtml_or_eval_sinks` test checks 8 banned substrings. [VERIFIED: app.js lines 17-22, CONTEXT.md Constraint 3]

- **DAG re-render on tick:** Never call `render(inner, g)` from any tick function. Latency updates modify panel sections only, never SVG topology. [VERIFIED: app.js line 543 comment, CONTEXT.md Constraint 4]

- **Full panel re-render on tick:** Must use the `data-section` attribute + `replaceWith` pattern from `renderThroughputSection`/`renderMemorySection`. Never call `renderDrillInPanel` from a tick -- it would clobber the entity lookup input. [VERIFIED: app.js lines 686-701, 737-753 -- "Pitfall 6"]

## Common Pitfalls

### Pitfall 1: Lock Contention from Double Lock in MSET Path
**What goes wrong:** Recording MSET latency requires locking AppState AFTER `handle_mset` returns. If the MSET was large and another PUSH arrives between the MSET completion and the latency recording lock, the latency recording could be delayed.
**Why it happens:** MSET releases locks between chunks (cooperative yielding). The final latency recording is a separate lock acquisition.
**How to avoid:** The latency recording lock is trivially fast (single array increment + heap comparison). The delay is negligible. No special handling needed.
**Warning signs:** MSET latency values that seem inflated -- but this is unlikely since the measurement wraps the entire MSET, not just the lock acquisition.

### Pitfall 2: Histogram Bin Edge Off-by-One
**What goes wrong:** Binary search on bin edges returns wrong bin for edge-case values (exactly on a bin boundary, 0us, negative values, NaN).
**Why it happens:** `binary_search_by` with float comparison and edge cases.
**How to avoid:** Clamp input to `[0, +inf)`. Use explicit bin 0 for values below first edge, last bin for overflow. Write unit tests for boundary values: 0.0, 1.0 (first edge), 10000.0 (last edge), 100000.0 (overflow), f64::NAN.
**Warning signs:** Percentiles that don't match expected values in tests.

### Pitfall 3: Two-Buffer Swap Timing Race
**What goes wrong:** If no samples arrive for > 5 minutes, both buffers are stale. Query returns stale data.
**Why it happens:** The swap only happens on `record()`, not on `snapshot()`.
**How to avoid:** In `snapshot()`, check if `current_start` is older than 2 * swap_interval. If so, return an empty histogram. This provides a natural decay to zero for idle systems.
**Warning signs:** Latency stats that never go to zero after traffic stops.

### Pitfall 4: Global Dashboard Flicker on Selection Toggle
**What goes wrong:** Clicking a node → clicking background → clicking same node causes the global dashboard to flash briefly before the stream drill-in renders.
**Why it happens:** `setSelected(null)` triggers `renderDrillInPanel(null)` which renders the global dashboard. Then immediately `renderDrillInPanel(name)` is called for the re-selection.
**How to avoid:** This is actually fine -- the toggle goes through setSelected(null) which renders global, then the next click goes through setSelected(name) which renders stream view. The "flash" only occurs if the user rapidly clicks the same node, and the deselect-reselect is the intended behavior (click to toggle). No fix needed.

### Pitfall 5: key_preview String Allocation on Hot Path
**What goes wrong:** Creating a 32-char key preview string on every PUSH/GET/SET for slow-query evaluation allocates on the heap even when the sample doesn't qualify for the top-20.
**Why it happens:** Eager string allocation before checking whether the sample exceeds the heap minimum.
**How to avoid:** Check the heap minimum FIRST (just an f64 comparison). Only allocate the key_preview string if the sample qualifies for insertion. This makes the common case (sample doesn't qualify) zero-allocation.
**Warning signs:** Increased p99 PUSH latency after adding slow-query tracking.

### Pitfall 6: Empty Latency Data on First Load
**What goes wrong:** First page load fetches `/debug/latency` but no commands have been processed yet. All histograms are empty, percentiles are 0.
**How to avoid:** Frontend must handle `count: 0` gracefully -- show dashes or "No data" instead of 0us. Already handled by the `formatLatency` function returning '--' for null/NaN.

const NUM_BINS: usize = 30;

/// Precomputed bin edges in microseconds.
/// Edge[i] = 10^(i * 4.0 / 29.0) for i in 0..=30
/// This covers 1us (10^0) to 10000us (10^4) in 30 log-spaced bins.
fn compute_bin_edges() -> [f64; NUM_BINS + 1] {
    let mut edges = [0.0f64; NUM_BINS + 1];
    for i in 0..=NUM_BINS {
        edges[i] = 10.0f64.powf(i as f64 * 4.0 / NUM_BINS as f64);
    }
    edges
}

## Open Questions

1. **Bin edge precomputation: const fn vs lazy_static vs runtime init?**
   - What we know: Rust const fn cannot call `f64::powf` in stable Rust (as of 1.78). [ASSUMED]
   - What's unclear: Whether `lazy_static` or `std::sync::OnceLock` or a const array literal is the cleanest approach.
   - Recommendation: Use a `const` array literal with precomputed values (hardcode the 31 f64 values). Simplest, no runtime cost, no dependency.

2. **Should per-stream slow queries be separate from per-command slow queries?**
   - What we know: CONTEXT.md says "min-heap top 20 per command". PUSH is the only stream-attributed command.
   - What's unclear: Whether users want to filter slow queries by stream in the per-stream drill-in.
   - Recommendation: Keep one heap per command (4 heaps total). The stream drill-in can filter the PUSH heap's entries by stream name at display time. No need for per-stream heaps.

3. **Should MSET latency measure the entire chunked operation or individual chunks?**
   - What we know: CONTEXT.md says "wire-to-wire" measurement. MSET includes cooperative yielding between chunks.
   - What's unclear: Whether a 100K-key MSET showing 500ms is useful or misleading.
   - Recommendation: Measure the entire MSET operation (wire-to-wire is the contract). The slow-query view will naturally surface large MSETs. Individual chunk timing is internal and not user-facing.
