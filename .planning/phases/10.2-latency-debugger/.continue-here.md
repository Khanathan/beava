## Gray areas identified (4 total, 0 discussed before pause)

### Gray area 1/4: Estimator choice **(CRITICAL — flagged as blocking anti-pattern in prior session)**

What data structure tracks the p50/p95/p99 percentiles. This determines memory per stream, accuracy at the tail, and CPU cost per sample.

| Option | Memory per stream | p99 accuracy | CPU per sample | Pros | Cons |
|---|---|---|---|---|---|
| **Bucketed histogram (fixed bins)** | Smallest — ~100-400 bytes (array of u64 counters over log-spaced bins) | Good for typical ranges; saturates if traffic exceeds max bin | Lowest — single array lookup + increment | Dead simple; zero heap allocation per sample; easy to serialize; matches Phase 6 ring-buffer style | Must pick bin layout at startup; tail accuracy degrades if max bin sized wrong |
| **HDR histogram** (hdrhistogram crate) | ~2-16 KB — configurable precision, O(log range) bins | High — typically 1-3% relative error across full range | Low — bit-twiddling to find bin | Mature, well-tested, handles 1µs-1h range out of the box; accurate at any tail | Larger memory; external crate dependency (adds to binary size) |
| **t-digest** (tdigest crate) | ~1-4 KB — compressed cluster list | Excellent at extreme tails (p99.9+) | Highest — requires cluster merging logic | Best for extreme tails; mergeable across streams | Most complex; heap allocations during merges; overkill for p50/p95/p99 |

**Orchestrator recommendation: Bucketed histogram.** Reasons: matches the project's zero-dependency single-binary ethos, matches the Phase 6 ring-buffer style already used in `ThroughputTracker`, no external crate, no heap allocations on the hot path, trivial JSON serialization. For p50/p95/p99 over a 1µs-10ms range, a log-spaced array of ~30 bins gives plenty of resolution. The p99 < 100µs budget leaves no room for cluster merges or bit-twiddling that HDR/t-digest require.

**This was the area being presented when the session paused.** Resume here and wait for user decision.

### Gray area 2/4: UI surface location

Where does latency data live in the Phase 10.1 split-view layout? Phase 10.1 deleted all tabs so bringing them back is a regression. Success criterion 2 explicitly says "as a tab, node drill-in panel, or edge tooltips — decided in discuss after Phase 10.1 lands".

Options to propose:
1. **New section added to the per-stream drill-in panel** (sits alongside Memory, State, Throughput, Entity Lookup). Scoped to selected stream. Matches existing UX paradigm. Requires latency to be attributed per-stream.
2. **Edge tooltips** — hover or click an edge to see latency for the cascade target. Natural fit for edges but only covers PUSH latency between streams, not GET/SET/MSET.
3. **Header-level global overlay** — toggle button in the header opens a modal/drawer showing command-level latency histograms (PUSH/GET/SET/MSET) without per-stream drill-in.
4. **Hybrid: global in drill-in when nothing selected, per-stream when selected** — drill-in panel's placeholder becomes a global latency view when no node is selected, switches to per-stream latency when a node is selected.

Orchestrator leans toward **option 1** (new section in drill-in panel) with a **option 3 fallback** for command-level stats not attributable to a stream (GET/SET/MSET). This preserves Phase 10.1's interaction model: click a node to drill in, get everything about that stream including latency.

### Gray area 3/4: Measurement scope & attribution

Which commands get measured and how to attribute them to streams?

**Scope questions:**
- All TCP commands (PUSH/GET/SET/MSET/REGISTER)?
- Only hot path (PUSH/GET)?
- Include parse time + execute time? Just execute time? Wire-to-wire?

**Attribution questions (this is the tricky one):**
- **PUSH**: natural per-stream attribution — the request carries a stream name.
- **GET**: key-scoped, not stream-scoped. Options: (a) attribute to all streams whose features were returned (fan-out accounting), (b) bucket as "GET all" without per-stream breakdown, (c) attribute to the selected stream context.
- **SET / MSET**: direct writes bypass the pipeline. Options: (a) "direct" pseudo-stream bucket, (b) per-stream attribution when the features being written map to a known stream, (c) no per-stream breakdown for direct writes.
- **REGISTER**: zero-frequency ops, probably skip.

**Orchestrator recommendation**: measure all commands at the wire-to-wire level (receive frame end → flush response start). PUSH attributed per-stream natively. GET/SET/MSET go into a command-level "direct" bucket without per-stream attribution. REGISTER skipped (too rare to matter).

### Gray area 4/4: Slow-query view + retention window

How to capture and display the N slowest requests; how much history to keep.

**Slow-query capture options:**
- Ring buffer of N slowest per command (bounded, replaces oldest when new slowest arrives)
- Reservoir sampling with time bias (prefer recent) — more complex
- Fixed-size min-heap priority queue — O(log N) insert

**Retention window options:**
- Rolling 5-minute window (matches ThroughputTracker's ewma_5m scale)
- Rolling 1-hour window
- Cumulative all-time (never decays)
- Multiple scales simultaneously: 1m / 5m / 1h like throughput

**Orchestrator recommendation**: rolling 5-minute window for the histograms (matches throughput cadence), min-heap of top 20 slowest per command for slow-query view (simple, bounded, O(log 20) ~= O(1) insert). Keep slowest records with: timestamp + command + (optional stream) + latency + (optional key preview for debugging).

## Phase 10.1 deferred gaps (3 total — logged in 10.1-VERIFICATION.md)

These were discovered during the Phase 10.1 browser smoke test and deferred to 10.2 or a polish phase. Phase 10.2 discuss may fold some of them into the latency work if they intersect.

### Gap 1: Keyless streams show irrelevant drill-in sections

`renderDrillInPanel` in `src/server/ui/app.js` does not branch on `node.key_field === null`. Clicking a keyless stream (like `RawEvents`) surfaces Memory, Throughput, and Entity Lookup sections — all meaningless for keyless streams because they don't store per-entity state (they're pure event routers that fan out to downstream keyed aggregators).

**Fix scope**: ~20 LOC frontend tweak + 1 regression test. Add a keyless variant that shows only header + State section (operator list remains meaningful if the keyless stream has derive features).

**Intersection with Phase 10.2**: low — latency is still measurable for pushes to keyless streams, so this gap is orthogonal. Can be fixed standalone.

### Gap 2: View entity lookup always returns empty

View nodes show the Entity Lookup section but queries always return "No features for '{key}'" because views compute derived features on read and are NOT materialized in per-entity state. `/debug/key/{key}` only returns `computed_features` (flat map from per-entity storage). View features never land there, so the frontend stream-scoped filter finds zero matches.

**Fix scope options**:
- (a) Backend evaluates view derive expressions on debug lookup and includes them in the response
- (b) Frontend hides the Entity Lookup section for view nodes entirely

Recommended direction: **(a)** — backend evaluation gives users an actual debugging surface for view features, which is the whole point of the drill-in panel.

**Intersection with Phase 10.2**: medium — Phase 10.2 touches `/debug/*` endpoints, so fixing (a) could be folded in if scope allows.

### Gap 3: No per-operator / per-feature byte breakdown in Memory section

`/debug/memory` tracks bytes at the stream level only (`{name, kind, key_count, estimated_bytes}`). The drill-in Memory section shows whole-stream totals. Users can't identify which operator (tx_count_1h buffers vs last_country single-value vs avg_amount_1h sum+count buckets) dominates a stream's memory.

**Fix scope**: backend walks each operator's state and computes serialized size; frontend renders inline in the State section (bytes column per operator row) or as a new Memory-by-operator subsection.

**Intersection with Phase 10.2**: low — distinct subsystem (memory tracking vs latency tracking). Can be a separate polish phase.
