# Phase 10.2: Code Review Report

**Reviewed:** 2026-04-10T12:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Phase 10.2 adds per-command and per-stream latency tracking with p50/p95/p99 percentile histograms, a `/debug/latency` HTTP endpoint, and latency visualization in the Debug UI. The implementation is solid overall: the histogram and rolling-window design are correct and well-tested (14 unit tests), the lock discipline is clean (no `.await` across mutex), all DOM rendering uses `textContent` via the `el()` helper (no XSS sinks), and the existing `push_latency_seconds` scalar metric is preserved in the metrics endpoint. The slow-query heap is properly bounded and the `would_accept` gate on PUSH avoids hot-path allocations per Pitfall 5.

Two warnings relate to inconsistent slow-query allocation avoidance on GET/SET paths and an unnecessary string allocation on every MSET recording. Two info items cover a minor CSS token deviation and a comment-vs-code discrepancy in BIN_EDGES documentation.

## Warnings

### WR-01: GET/SET slow-query recording skips the `would_accept` guard

**File:** `src/server/tcp.rs:386-389` (GET), `src/server/tcp.rs:409-413` (SET)
**Issue:** The PUSH path (line 355) checks `slow_queries_would_accept()` before allocating the `key_preview` string and calling `maybe_record_slow()`. The GET and SET paths unconditionally clone and truncate the key into `kp` and then call `maybe_record_slow()`, which checks `would_accept` internally but after the allocation. This defeats the Pitfall 5 optimization for GET and SET: every GET and SET allocates a `String` for the slow-query key preview even when the heap is full and the sample would be rejected. Under high GET/SET throughput this creates unnecessary allocation pressure on the hot path.
**Fix:**
```rust
// GET path (around line 385):
let get_us = get_start.elapsed().as_secs_f64() * 1_000_000.0;
app.latency.record_command(crate::server::latency::CommandKind::Get, get_us, std::time::Instant::now());
if app.latency.slow_queries_would_accept(crate::server::latency::CommandKind::Get, get_us) {
    let mut kp = key.clone();
    kp.truncate(32);
    app.latency.maybe_record_slow(crate::server::latency::CommandKind::Get, None, get_us, kp);
}
```
Apply the same pattern to the SET path at line 409-413.

### WR-02: MSET slow-query always allocates an empty String

**File:** `src/server/tcp.rs:182`
**Issue:** The MSET latency path calls `maybe_record_slow(..., String::new())` on every MSET completion. While `String::new()` is a zero-capacity allocation (no heap alloc), the `maybe_record_slow` method still runs the `would_accept` check, constructs a `SlowQueryRecord` with `SystemTime::now()`, `stream.map(...)`, etc. when the sample qualifies. More importantly, there is no `would_accept` guard before calling `maybe_record_slow`, so on every MSET the method body runs even when the heap would reject. This is inconsistent with the PUSH pattern.
**Fix:**
```rust
if is_mset {
    let mset_us = cmd_start.elapsed().as_secs_f64() * 1_000_000.0;
    let mut app = state.lock().unwrap_or_else(|e| e.into_inner());
    app.latency.record_command(crate::server::latency::CommandKind::Mset, mset_us, std::time::Instant::now());
    if app.latency.slow_queries_would_accept(crate::server::latency::CommandKind::Mset, mset_us) {
        app.latency.maybe_record_slow(crate::server::latency::CommandKind::Mset, None, mset_us, String::new());
    }
}
```

## Info

### IN-01: CSS uses literal `8px` and `12px` instead of design tokens

**File:** `src/server/ui/app.css:585,599`
**Issue:** The `.histogram-bars` block uses `padding: 8px 0` (line 585) and `.latency-table` uses `font-size: 12px` and `margin: 8px 0` (lines 599-600). The Phase 10 design system defines `--space-2: 8px`, `--text-xs: 12px` as tokens. Using literal values instead of `var(--space-2)` and `var(--text-xs)` is a minor style inconsistency with the rest of the stylesheet, which exclusively uses design tokens.
**Fix:** Replace literal values with token references: `padding: var(--space-2) 0`, `font-size: var(--text-xs)`, `margin: var(--space-2) 0`. Same for line 602-603 `padding: 4px 8px` (use `var(--space-1) var(--space-2)`) and line 619-620 `gap: 8px`, `padding: 4px 0`.

### IN-02: BIN_EDGES comment exponents diverge from actual values

**File:** `src/server/latency.rs:24-56`
**Issue:** The inline comments claim the formula is `Edge[i] = 10^(i * 4.0 / 30.0)`, but the actual float values do not match that formula at all positions. For example, index 5 is documented as `10^0.667 = 4.217` but `10^0.667 = 4.642`. The actual constant 4.217 equals `10^0.625`. The histogram works correctly regardless because the precomputed floats are used directly, but the documentation formula is misleading for future maintainers.
**Fix:** Either regenerate the constants from the stated formula, or update the comments to reflect the actual formula used to generate them (appears to be `10^(i * log10(10000) / NUM_BINS)` with a different base or step).
