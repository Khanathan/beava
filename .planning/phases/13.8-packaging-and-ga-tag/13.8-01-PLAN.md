## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → cargo registry | crates downloaded during cibuildwheel before-build; supply-chain attack surface |
| GitHub Actions runner → PyPI | wheels uploaded with PYPI_API_TOKEN; if leaked, attacker can publish malicious beava wheels |
| End user `pip install beava` → bundled binary | binary executes on user box with user privileges |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13.8-01-01 | T (Tampering) | cibuildwheel before-build hook | mitigate | Pin all GitHub Actions to commit SHAs (not floating tags) per OSS security best practice; `Cargo.lock` checked in; `--locked` flag on cargo build |
| T-13.8-01-02 | I (Information disclosure) | PYPI_API_TOKEN | mitigate | Token lives only in GitHub repo secrets (never in repo files); plan-checker greps for `PYPI_API_TOKEN` in workflow files and confirms only `${{ secrets.PYPI_API_TOKEN }}` references; Plan 13.8-05 owns the actual `pypa/gh-action-pypi-publish` step |
| T-13.8-01-03 | T | bundled binary | mitigate | The `beava` Rust binary is built from the same commit as the wheel via `cargo build --release` in CI; checksum recorded in workflow log; Plan 13.8-05 attaches sha256 alongside |
| T-13.8-01-04 | E (Elevation) | `[project.scripts] beava = "_embed:_console_main"` | accept | console_script shim re-execs the bundled binary; runs at user privilege; no setuid; same trust boundary as `cargo install beava` |
