### D-02 (8-B) — Release strategy = soft-launch, announce 7-14d later (USER-LOCKED)

**Day 0 (Phase 13.8 closure):**
- Tag `v0.0.0` on `main`.
- Push to PyPI / npm / Docker Hub / ghcr.io / GitHub Releases.
- Verify install paths from clean machines:
  - `pip install beava` on macOS ARM64 + macOS Intel + Linux x86_64 + Linux ARM64
  - `npm install @beava/sdk` on Node 18+
  - `go get github.com/beava-dev/beava/sdk/go` on Go 1.21+
  - `docker pull beava/beava:v0.0.0` + `docker pull ghcr.io/beava-dev/beava:v0.0.0`
- Post NOTHING. Repo is publicly visible; no announcement.

**Day 1-14 (organic-finder soak):**
- Monitor GitHub issues / installs.
- If any channel install fails on a clean machine → tag `v0.0.1` with the fix; re-publish; restart soak.
- Track install-success metrics passively (e.g., PyPI download counts on the new package).

**Day ~14 (announcement):**
- Publish marketing assets (D-06).
- Post to Hacker News (Show HN), Twitter, dev.to.

**Why soft-launch:** lower-stress launch week. Catches packaging bugs (PyPI install fails on clean Python; npm tarball missing a file; Docker FROM mismatch; etc.) BEFORE the HN post lands. The tradeoff is a slower "launch announcement" rhythm — fine for an OSS infra tool with no funding-round pressure.

**No `v0.0.0-rc.N` pre-release tags.** Skipping the rc soak in favor of a clean v0.0.0 tag day 0.

### D-03 (8-C) — Wheel build matrix = 4 wheels (USER-LOCKED)

Build PyPI wheels for:
1. `manylinux2014_x86_64` (Linux x86_64) — primary deploy target for fraud / adtech servers
2. `manylinux2014_aarch64` (Linux ARM64) — Graviton / Raspberry Pi / cloud ARM
3. `macosx_11_0_arm64` (macOS ARM64) — primary dev workstation for many target users (Priya / fraud-team CTOs on M-series Macs)
4. `macosx_10_9_x86_64` (macOS Intel) — older MacBook Pros / Hackintoshes / CI runners

Each wheel bundles a pre-compiled `beava` binary appropriate to the platform. Built via `cibuildwheel` or maturin (planner picks).

**Deferred to v0.1+:**
- Windows x86_64 (`win_amd64`) — significant CI matrix complexity; can ship v0.1 if demand surfaces.
- Linux musl (Alpine) — manylinux glibc covers most users; musl wheel is v0.1+ if requested.

### D-09 (USER-DRIVEN scope amendment 2026-05-05) — `curl | sh` installer served from GitHub Releases

Added per user direction during Phase 13.5.4 acceptance-gate window. Lowest-friction install path for Linux/macOS users without Docker/Python preference. Matches Bun, Rust (rustup), Deno, fly.io, supabase CLI patterns — but uses GitHub Releases asset hosting (no project-domain infra dependency).

**Initial proposal (2026-05-05 morning)**: host at `https://beava.dev/install.sh` via Cloudflare Pages mirror.
**REVISED (2026-05-05 afternoon, USER-DRIVEN)**: drop Cloudflare Pages dependency — beava.dev is not on Pages. Serve `install.sh` as a GitHub Release asset alongside binary tarballs.

**Locked decisions:**
- **Hosting**: GitHub Release asset. User-facing URL: `https://github.com/beava-dev/beava/releases/latest/download/install.sh` (GitHub's `latest/download/` redirect always resolves to the most recent release's assets).
- **Plan 13.8-05 prerequisite**: release.yml's `github-release` job MUST upload `install.sh` (from repo root) as a release asset on every tag-push. One-line addition: `gh release upload <tag> install.sh`.
- **Default install dir**: `$HOME/.local/bin/beava` (no sudo); overridable via `BEAVA_INSTALL_DIR`.
- **Default version**: latest GitHub Release; overridable via `BEAVA_VERSION=v0.0.0`.
- **Verification**: SHA256 from sibling `<binary>.sha256` GH Release asset; install.sh refuses to install on mismatch.
- **PATH guidance**: print, never auto-modify shell profiles.
- **Smoke**: tag-push CI step runs `curl ... | sh` against the live release in a clean Ubuntu container; waits up to 5min for parallel release.yml jobs to upload install.sh asset.
- **Vanity URL** (`beava.dev/install.sh` → GH release asset): NOT scoped for v0. Post-v0 add via DNS redirect or tiny Cloudflare Worker if/when desired.

Plan: `13.8-04a` (added wave 1, no dep on 13.8-04 Docker work; depends on Plan 13.8-05 for asset upload).

### D-10 (USER-DRIVEN scope amendment 2026-05-05) — Homebrew tap = `beava-dev/homebrew-beava`

Added per user direction during Phase 13.5.4 acceptance-gate window. Canonical macOS install path for the developer demographic the v0 launch targets. README install matrix becomes 4-way: curl|sh / brew / Docker / pip.

**Locked decisions:**
- **Tap repo**: separate repo at `github.com/beava-dev/homebrew-beava` (Homebrew tap convention requires `homebrew-<name>`).
- **User call**: `brew tap beava-dev/beava && brew install beava` — Homebrew strips the `homebrew-` prefix automatically.
- **Platform matrix**: same 4 platforms as D-03 (linux-x86_64 / linux-arm64 / macos-arm64 / macos-x86_64); formula uses `on_macos { on_arm | on_intel }` + `on_linux { on_arm | on_intel }` blocks.
- **Bump workflow**: tag-push fires `.github/workflows/homebrew-bump.yml` in main repo; computes SHA256 for each tarball, sed-replaces version + 4 sha256 stanzas in `Formula/beava.rb`, commits + pushes to tap repo via SSH deploy key.
- **Bottling**: SKIPPED for v0; users get the GH Release tarball + Homebrew handles install. v0.1+ may add bottling.
- **One-time setup**: user creates the empty `homebrew-beava` repo, commits the seed Formula from `HOMEBREW-TAP-SEED.md`, generates SSH deploy key, adds private key as `HOMEBREW_TAP_DEPLOY_KEY` secret in main repo. Done once at user-claim time before the first `git tag v0.0.0`.

Plan: `13.8-04b` (added wave 1, no dep on 13.8-04 Docker work).
