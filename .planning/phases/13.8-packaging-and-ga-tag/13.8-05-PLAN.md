<objective>
Rewrite `.github/workflows/release.yml` from a single-job linux-binary-only release into a multi-channel orchestrator that fires PyPI / npm / Docker Hub / ghcr.io / Go-verify publish workflows in parallel on tag-push, plus attaches 3 platform binaries to a GitHub Release.

Purpose: Single tag (`v0.0.0`) triggers all v0 publish channels. Plans 13.8-01..04 produced reusable building blocks; this plan wires them into one tag-driven orchestrator. Per CONTEXT D-08, GitHub Actions is the CI provider; existing release.yml is the natural extension point.

Output: New `.github/workflows/release.yml` (~150-250 LOC) with these jobs:
1. `prepare` — verifies tag format, computes version metadata
2. `build-binaries` — produces linux-x86_64, linux-arm64, macos-arm64 native binaries (in parallel matrix, NOT in cibuildwheel — these are standalone binaries for the GitHub Release page)
3. `pypi-publish` — depends on wheels.yml artifact (via workflow_run or composite); calls `pypa/gh-action-pypi-publish`
4. `npm-publish` — calls (or reuses) npm.yml logic; runs `npm publish --provenance`
5. `docker-publish` — calls (or reuses) docker.yml logic; pushes to both registries
6. `go-verify` — calls (or reuses) go-verify.yml logic; runs post-tag resolution check
7. `github-release` — attaches binaries from job 2; depends on jobs 3-6 to ensure all-or-fail visibility
8. `release-summary` — posts step summary
</objective>

Current release.yml (84 LOC, single job):
- Trigger: `push: tags: ['v*']`
- Job `build-and-release`: cargo test + cargo build --release + create GitHub Release with linux-x86_64 binary attached
- This job is REPLACED by the multi-job orchestrator described in <objective>

Three patterns for invoking sibling workflows (planner picks at execute time):

**Pattern A — workflow_call (reusable workflows):** restructure wheels.yml / npm.yml / docker.yml / go-verify.yml each with a top-level `on: workflow_call:` trigger so release.yml can call them as job uses. CLEANEST. Adds ~5 LOC per called workflow.

**Pattern B — gh workflow run inline step:** release.yml runs `gh workflow run wheels.yml --ref ${{ github.ref }}` and polls the status. Asynchronous; harder to gate downstream jobs.

**Pattern C — duplicate the publish logic inline in release.yml:** the easiest to debug; the most LOC; risk of drift between release.yml and the per-channel workflows.

**Recommendation:** Pattern A (workflow_call). Plans 13.8-01..04 should be updated as part of this plan to add `workflow_call` triggers (one-line additions). This keeps each per-channel workflow callable BOTH standalone (for workflow_dispatch validation) AND from release.yml (for tag-push publish).

    on:
      push:
        tags:
          - 'v*'           # main release tag (v0.0.0)

      contents: write       # GitHub Release creation

          version: ${{ steps.parse.outputs.version }}
          is-go-tag: ${{ steps.parse.outputs.is-go-tag }}
        steps:
          - id: parse
            run: |
              TAG=${GITHUB_REF#refs/tags/}
              if [[ "$TAG" == sdk/go/v* ]]; then
                echo "version=${TAG#sdk/go/}" >> $GITHUB_OUTPUT
                echo "is-go-tag=true" >> $GITHUB_OUTPUT
              else
                echo "version=$TAG" >> $GITHUB_OUTPUT
                echo "is-go-tag=false" >> $GITHUB_OUTPUT
              fi

      pypi-publish:
        needs: prepare
        if: needs.prepare.outputs.is-go-tag == 'false'
        uses: ./.github/workflows/wheels.yml
        secrets:
          PYPI_API_TOKEN: ${{ secrets.PYPI_API_TOKEN }}

      npm-publish:
        needs: prepare
        if: needs.prepare.outputs.is-go-tag == 'false'
        uses: ./.github/workflows/npm.yml
        secrets:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}

      docker-publish:
        needs: prepare
        if: needs.prepare.outputs.is-go-tag == 'false'
        uses: ./.github/workflows/docker.yml
        secrets:
          DOCKERHUB_USERNAME: ${{ secrets.DOCKERHUB_USERNAME }}
          DOCKERHUB_TOKEN: ${{ secrets.DOCKERHUB_TOKEN }}

      go-verify:
        needs: prepare
        # runs for BOTH tag forms — validates the Go module is resolvable post-tag for both:
        #   - on `v*` tag: verifies the existing sdk/go/v<same> tag also resolves (cross-check)
        #   - on `sdk/go/v*` tag: primary go-get verification path
        uses: ./.github/workflows/go-verify.yml

      build-binaries:
        needs: prepare
        if: needs.prepare.outputs.is-go-tag == 'false'
        strategy:
          matrix:
            include:
              - { os: ubuntu-latest, target: x86_64-unknown-linux-gnu, name: beava-linux-x86_64 }
              - { os: ubuntu-24.04-arm, target: aarch64-unknown-linux-gnu, name: beava-linux-aarch64 }
              - { os: macos-14, target: aarch64-apple-darwin, name: beava-macos-aarch64 }
        runs-on: ${{ matrix.os }}
        steps:
          - uses: actions/checkout@<SHA>
          - uses: dtolnay/rust-toolchain@<SHA>
            with: { targets: ${{ matrix.target }} }
          - run: cargo build --release --bin beava --target ${{ matrix.target }} --locked
          - run: |
              mkdir -p release
              cp target/${{ matrix.target }}/release/beava release/${{ matrix.name }}
              chmod +x release/${{ matrix.name }}
              shasum -a 256 release/${{ matrix.name }} > release/${{ matrix.name }}.sha256
          - uses: actions/upload-artifact@<SHA>
            with: { name: ${{ matrix.name }}, path: release/* }

      github-release:
        needs: [prepare, pypi-publish, npm-publish, docker-publish, build-binaries]
        if: needs.prepare.outputs.is-go-tag == 'false'
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@<SHA>
          - uses: actions/download-artifact@<SHA>
