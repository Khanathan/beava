# Phase 6.1 — Throughput row (per-phase artifact for orchestrator merge)

**Captured:** 2026-04-24
**Branch:** phase-6.1-async-dur
**Commit:** `adbc3d1` (post 6.1-04 crash test commit; sync_mode default = Periodic)
**hw-class:** `Darwin-24.3.0 / 10 cores` (Apple-M4)

## Headline

Switching the default `/push` durability from per-event fsync to
periodic-fsync (Plan 6.1-01..03) lifted sustained EPS by **11.9× – 16.9×**
across all three pipeline shapes vs the Phase 7.5 baseline. The headline
small-pipeline number went from 990 EPS → 15,672 EPS (15.8× lift).

## Reproduce

```bash
cargo run -p beava-bench --release -- \
    --pipeline {small|medium|large} \
    --transport http \
    --duration-secs 20 \
    --parallel 64
```

`--parallel 64` is what saturates the (single-thread) WAL worker on
Apple-M4 once fsync no longer gates per-push latency. The Phase 7.5
baseline used `--parallel 8` because higher parallelism couldn't
extract more EPS while every push waited for a 7.4 ms F_FULLSYNC
roundtrip. Both numbers are recorded under their respective parallelism
because what we're measuring is the system ceiling, not a fixed-load
latency.

## Regression-gate calls

Per CLAUDE.md §Performance Discipline, the gate is "≥0.75× of prior
baseline in the same hw-class on simple-fraud (small) shape". 990 EPS
× 0.75 = 743 EPS minimum. We deliver 15,672 EPS. **PASS** with
significant headroom.

## SC7 verification

## Notes on the bottleneck shift

Phase 7.5: every push waited ~7.4 ms for `F_FULLSYNC` → throughput
fundamentally capped near `1 / 7.4 ms = 135 EPS / inflight-request`.
Adding clients past ~8 didn't help because the WAL writer was serializing
fsyncs.

Phase 6.1: Periodic mode resolves the ACK as soon as the in-memory
BufWriter write completes; fsync runs on a 2 ms timer regardless of
push activity. The bottleneck moves to:

1. HTTP roundtrip overhead (~1 ms for in-process loopback).
2. Single-writer aggregation lock (the apply step still serializes).
3. JSON parse + schema validation (per-push CPU cost).

Phase 13 will benchmark Linux `fdatasync` (~10× faster than macOS
F_FULLSYNC) and remove the in-process HTTP overhead via direct TCP
push. The 3 M EPS/core target is plausible there but out of scope for
this phase.
