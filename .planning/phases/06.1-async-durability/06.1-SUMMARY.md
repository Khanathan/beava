# Phase 6.1: Async Durability Default + push-sync Endpoint — Phase Summary

**Status:** complete
**Shipped:** 2026-04-24
**Branch:** phase-6.1-async-dur

## Headline

Flipped default `/push` durability from "ACK after fsync" to "ACK after
in-memory append; fsync runs on a 2 ms timer". Strict callers retain
fsync-per-event semantics via the new `POST /push-sync/{event_name}`
endpoint. Matches Kafka's acks=1 (default) vs acks=all (strict) mental
model.

End-to-end throughput on the small-fraud pipeline went from
**990 EPS → 15,672 EPS (15.8× lift)** on Apple-M4/Darwin. macOS
F_FULLSYNC stops gating the per-push critical path; bottleneck shifts
to HTTP overhead + the single-writer aggregation lock.

## What shipped

### Plan 6.1-01: SyncMode enum + WalSink dispatch
- `pub enum SyncMode { Periodic, PerEvent }` in `beava-persistence`.
- `WalSinkConfig.sync_mode: SyncMode` field, defaults to `Periodic`.
- `BEAVA_WAL_SYNC_MODE=periodic|per-event` env var.
- `WalSink::append_event_with_mode(payload, mode)` and
  `append_record_with_mode(record_type, payload, mode)`.
- `WalSink::append_event` now uses the sink's `default_mode` (set
  from `WalSinkConfig.sync_mode` at spawn).
- `worker_loop` refactored: `flush_batch` split into `stage_request`
  (recv-time, ACKs Periodic waiters) + `fsync_batch` (timer/threshold,
  ACKs PerEvent waiters).
- Background timer fires unconditionally on `fsync_interval_ms`; the
  flush is a no-op when nothing is staged.

### Plan 6.1-02: POST /push-sync/{event_name}
- New `push_sync_handler` mounted at `POST /push-sync/:event_name`
  in `push_router`.
- Internally calls `push_inner(state, event_name, body, SyncMode::PerEvent)`.
- Response shape unchanged: `{ack_lsn, idempotent_replay,
  registry_version}`. Idempotency cache works the same.

### Plan 6.1-03: Mode-dispatched apply order
- `push_handler` and `push_sync_handler` both delegate to `push_inner`
  with their respective `SyncMode`.
- `push_inner` calls `wal_sink.append_event_with_mode(payload, mode)`
  then runs the apply step. Because `append_event_with_mode` resolves
  at stage-time (Periodic) or post-fsync (PerEvent), the apply step is
  automatically mode-dependent without explicit branching.

### Plan 6.1-04: Phase 6.1 crash + push tests

  404 paths and aggregation-apply on both endpoints.

- Phase 6's existing `tests/phase6_push.rs`, `tests/phase6_crash.rs`,
  `tests/phase6_smoke.rs` all unchanged + still passing (existing
  WalSinkConfig struct-literal usages pinned to PerEvent).

## Deviations from the phase prompt (under Claude's discretion)

1. **Plans 6.1-02 + 6.1-03 collapsed into one feat commit.** The
   plan prompt structured them as separate tasks but the apply-order
   change is a one-line consequence of where `append_event_with_mode`
   resolves (stage-time vs post-fsync). Splitting them would have
   required either adding a stub /push-sync handler in 02 then
   refactoring it in 03, or introducing a temporary "apply order
   knob" — neither was net-positive. Both criteria (SC2, SC3) are
   verified.

2. **Probe binary unchanged.** The phase prompt left open whether to
   add a `phase6_1_crash_probe`. The existing `phase6_crash_probe`
   already supports per-request mode selection via the endpoint URL
   (`/push` vs `/push-sync`), so no probe changes were needed.

3. **`WalSink::append_record` (registry-bump path) is pinned to
   PerEvent.** Registry version transitions must be durable before
   future events reference them — Periodic mode would re-introduce
   the same race we eliminated in Phase 7. Documented in the source.

4. **TestServerBuilder gained no new explicit `sync_mode` setter.**
   The default `WalSinkConfig::default()` cascades through, and
   tests that need to override it set `BEAVA_WAL_SYNC_MODE` via the
   env path. Adding a builder setter is trivial follow-up if needed.
