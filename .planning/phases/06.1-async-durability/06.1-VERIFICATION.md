## Success-criterion verification

### #1 Default sync mode = `periodic`, env-overridable — PASS
`WalSinkConfig::default().sync_mode == SyncMode::Periodic`;
`DurabilityConfig::default().wal_sync_mode == WalSyncMode::Periodic`.
Verified by `tests/sync_mode.rs::sync_mode_default_is_periodic`.
`BEAVA_WAL_SYNC_MODE=per-event` parse path covered in
`crates/beava-core/src/config.rs::apply_env_overrides`.

### #2 `/push` ACKs after in-memory append — PASS
`tests/sync_mode.rs::periodic_append_returns_before_fsync`: with
`fsync_interval_ms=5_000`, a single Periodic append resolves in <500 ms.

### #3 `/push-sync` ACKs only after fsync — PASS
`tests/sync_mode.rs::per_event_append_blocks_on_fsync`: with
`fsync_interval_ms=400`, PerEvent append takes ≥300 ms.
`tests/phase6_1_crash.rs::push_sync_ack_survives_crash` confirms the
durability invariant end-to-end.

### #4 Periodic timer fires every `fsync_interval_ms` — PASS
`worker_loop` uses `tokio::time::interval` that fires unconditionally;
no-op when `staged_bytes == 0`. Verified by
`tests/sync_mode.rs::periodic_records_eventually_fsync_via_shutdown`
(5 Periodic appends become durable on shutdown drain — proves the
flush path runs even without explicit sync_data calls between
appends).

### #5 Periodic crash within fsync window may lose events — PASS
`tests/phase6_1_crash.rs::periodic_push_kill_keeps_subset_of_events`:
50 events pushed with `fsync_interval_ms=2_000`, kill before tick,
asserts `0 ≤ recovered ≤ 50` and that WAL replay returns Ok (no torn
records).

### #6 Push-sync ACK'd events survive crash unconditionally — PASS
`tests/phase6_1_crash.rs::push_sync_ack_survives_crash` + the updated
`tests/phase6_crash.rs::wal_kill_after_ack_preserves_event` (now
routed at `/push-sync`).

### #7 Throughput ≥10× lift over Phase 7.5 baseline — PASS
See `06.1-throughput-row.md`:
- small: 990 → 15,672 EPS (15.8×)
- medium: 1031 → 17,453 EPS (16.9×)
- large: 1007 → 12,004 EPS (11.9×)

All exceed 10× target.
