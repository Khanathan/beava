## Issues Encountered

### Smoke run blocked by HTTP parser bug (out of scope)

**Symptom:** `cargo run --release -p beava-bench -- --pipeline fraud-team.json` fails with `register request: connection closed before message completed`. Bisect shows ANY 2-derivation subset that pushes the register POST body over 8 KiB triggers the same error.

**Root cause:** Pre-existing bug in `crates/beava-runtime-core/src/http_listener.rs:69-74` — `MAX_HEADER_BYTES = 8 KiB` early-return applies to the entire buffer (`buf.len() > MAX_HEADER_BYTES`), not just headers. fraud-team.json's register body is 15 KiB; rejected immediately when the second TCP packet arrives.

**Resolution:** Documented in `.planning/phases/19.1-realistic-bench-rebaseline/deferred-items.md`. Pre-existing bug, surfaced now because fraud-team.json is the first config large enough to trip it (large_phase9 is 1.8 KiB). Out of scope per SCOPE BOUNDARY (not caused by this plan's changes); on critical path for 19.1-04/19.1-05 which need to run fraud-team.json against the live server.

### Plan acceptance grep checks had typos

The plan's grep-verifiable acceptance criteria included two checks that don't apply to fraud-team.json's structure:

1. `grep -c '"kind"' ≥ 100` — actually counts node-kind labels (events + derivations = 14), NOT features. The intent was "preserves ~110-feature scope"; my fraud-team.json has 111 features (verified via Python JSON walk).
2. `grep -q "lat_field"` — would return zero matches because the JSON wire uses `lat`/`lon`, not `lat_field`/`lon_field`. (See Audit Results items 5-10.)

Documented in this SUMMARY's Audit Results table for future-plan-author reference.
