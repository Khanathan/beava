| Option | Description | Selected |
|--------|-------------|----------|
| 4×32MiB tick=20ms | Middle ground. ~128MB resident WAL (vs original 48MB). Predicted to eliminate most of the bimodal wal_append > 1ms tail without 8×64MiB's 512MB RSS cost. Ship as default; expose env-var tunables for power users. | |

## Stretch scope (complex-pipeline optimization)

| Option | Description | Selected |
|--------|-------------|----------|
| Lazy buckets only | Replace WindowedOp's [Option<Box<AggOp>>; 64] with SmallVec<[(i64, Box<AggOp>); 4]> + lazy bucket allocation. Single concrete win: ~1500ns of 2576ns cold entity_row_init (60%); +50% EPS for cold-key complex shapes. ~1 week phase. | ✓ |
| Lazy buckets + same-key sketch batching | Lazy buckets + batch sketch updates for hot-key sketch ops. Best fraud-team coverage. ~2 weeks. Higher complexity (apply dispatch peek-ahead pattern). | |
| Lazy buckets + OP_PUSH_MANY | Lazy buckets + adopt OP_PUSH_MANY in bench. Lifts wire-stack ceiling for simple shapes (>1M EPS) but doesn't help fraud-team (apply-bound). | |
| All three | Lazy buckets + same-key batch + OP_PUSH_MANY. Full coverage. ~3-4 weeks. Risk: phase stalls or scope creeps. | |

**User's choice (round 1):** Asked clarifying question — "what is same-key batch update?"
**Claude's clarification:** Same-key batching detects runs of consecutive `(entity_key, agg_id)` writes in apply, calls sketch state once with the values vector instead of per-event update calls. Skips per-event borrow churn, lets HLL SIMD hashes, lets UDDSketch sort-and-merge, lets SpaceSaving skip repeated heap rebalances. ~30-50% lift for zipfian/hot-key complex shapes. Cost: requires apply-dispatch peek-ahead pattern.

**User's choice (round 2):** Lazy buckets only
**Notes:** User picked the simpler concrete win after understanding what same-key batching entails. Captured as D-16 in CONTEXT.md; same-key batch deferred to Phase 19.2 or later (D-17). OP_PUSH_MANY also deferred (D-18).

## Histogram windowed semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Defer to future phase | Out of 19.1 scope. Phase 19.1 already has 5 sub-goals; adding a new op family bloats the phase. Document the gap in Phase 20 (operator + streaming-semantics audit) backlog. | |
| Document UDDSketch as the path | No new code. Add a one-page doc + register-time warning that windowed distributions go through bv.percentile (UDDSketch) and bv.histogram is lifetime-only. | |
| Add windowed_histogram op family | New code, fills a real catalogue gap. Significant scope expansion. Probably its own phase. | |
| Decide during validation | If fraud-team.json validation surfaces real cases that need windowed_histogram, fold it in; otherwise defer. | |

**User's choice (round 1):** Asked clarifying question — "is this a real use case isn't it usually pxx only?"
**Claude's clarification:** User's instinct is correct. For real-time fraud serving, PXX (percentiles via UDDSketch) is the standard query — users want "p99 transaction amount over last 1h", not "bucket counts of transaction amounts." `bv.histogram` is mostly a diagnostic/distributional-shape tool used in batch analytics, rarely on the live-scoring path. The fraud-team.json `amount_histogram_24h` is almost certainly better expressed as `bv.percentile` over a 24h window.

**User's choice (round 2):** Defer to Phase 20 backlog
**Notes:** Captured as D-15 in CONTEXT.md. fraud-team.json `amount_histogram_24h` rewritten to `bv.percentile` per D-10.1 during Path B validation. Phase 20 documents the catalogue gap; Phase 19.1 adds no code for windowed histograms.
