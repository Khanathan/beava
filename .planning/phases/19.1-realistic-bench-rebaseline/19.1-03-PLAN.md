<objective>
Bump the default WAL config from 3×16 MiB tick=2ms to 4×32 MiB tick=20ms (~128 MiB resident) per CONTEXT D-01, and expose three env-var tunables (`BEAVA_WAL_BUFFERS`, `BEAVA_WAL_BUFFER_SIZE_MB`, `BEAVA_WAL_TICK_MS`) per D-02 so power users can dial up to 8×64 MiB tick=100ms or down to 3×16 MiB tick=2ms.

The existing config in `crates/beava-server/src/server.rs` (around lines 575-588) hardcodes:
```rust
let buf_bytes = 16 * 1024 * 1024;
let wal_ring = Arc::new(WalBufferRing::new(3, buf_bytes, Arc::clone(&wal_lsn)));
let wal_writer = WalWriter::new(&wal_writer_dir, Arc::clone(&wal_ring), Arc::clone(&wal_lsn), 2)
```

After this plan, those hardcoded values come from a `WalConfig::resolve_from_env()` helper with safe clamp ranges and a startup log line.

Per CONTEXT D-04, ALL Phase 18 WAL architecture invariants stay UNCHANGED:
- Lock-free apply path (single writer to active buffer)
- 3+ buffer state machine (active / sealed / flushing / free) — now defaults to 4
- Single writer+fsync thread
- Four-watermark discipline (committed_lsn / written_lsn / synced_lsn / acked_lsn)
- `O_APPEND` on the WAL file
- Refuse-on-network-FS startup check

Phase 19.1-03 only changes COUNT, SIZE, and TICK INTERVAL of buffers — nothing about the algorithm or buffer state machine.

Purpose: Phase 19's debug session uncovered a bimodal `wal_append > 1ms` tail under sustained 500k EPS — 1% of events stalled 1-42ms each, summing to 17.4 seconds of apply-thread stall time. Experimental fix (8×64 MiB tick=100ms) eliminated the tail and lifted EPS +33% but at 512 MB resident WAL. The 4×32 MiB tick=20ms middle-ground default should capture most of the lift at 128 MB RSS. Env-var tunables let power users dial higher (memory `project_phase19_wal_experiment`).

Output: `WalConfig::resolve_from_env()` helper, server.rs wired to use it, regression tests covering env-var parsing + clamp ranges. Two atomic commits per CLAUDE.md TDD discipline.

**Depends on Plan 19.1-01** because the bench wall_clock fix is the pre-condition for honest measurement. Without 19.1-01, we can't tell whether the WAL bump actually lifted EPS — the bench would still under-report wall_clock for any run shorter than 1s.
</objective>

<!-- The exact env-var contract per D-02 + D-03. -->

| Env var | Type | Default | Clamp range | Rationale |
|---------|------|---------|-------------|-----------|
| `BEAVA_WAL_BUFFERS` | usize | 4 | [2, 32] | Min 2 = degenerate ring (current sealed + next active); max 32 = sanity cap (32 × 256 MB = 8 GB upper limit per typo guard) |
| `BEAVA_WAL_BUFFER_SIZE_MB` | usize | 32 | [4, 256] | Min 4 MB = enough for 1 burst-tick at 500k EPS × 300B; max 256 MB = upper guard against operator typo (32 × 256 = 8 GB total possible) |
| `BEAVA_WAL_TICK_MS` | u64 | 20 | [1, 1000] | Min 1 ms = aggressive fsync (matches old default); max 1000 ms = 1 s lag (matches Redis default; deliberately doesn't go higher to keep the "we sub-second-fsync" identity) |

**Default config:** 4 × 32 MB tick=20ms → 128 MB resident WAL, 50ms tick frequency.
**Power-user max:** 8 × 64 MB tick=100ms → 512 MB resident, low fsync calls.
**Embedded min:** 3 × 16 MB tick=2ms → 48 MB resident (matches current), aggressive sync.

**Out-of-range behavior:** clamp silently to the nearest endpoint, log a warning at INFO level naming the original value AND the clamped value. Do NOT refuse to start (operators may inherit env from shell scripts they don't control; clamping is more forgiving).

**Parse-failure behavior:** if env var is set but unparseable (e.g., `BEAVA_WAL_BUFFERS=xyz`), log a warning, fall back to the default. Do NOT refuse to start (same rationale).

**Refuse-on-network-FS:** This check lives elsewhere in the WAL stack (Phase 18 invariant). It must continue to apply REGARDLESS of buffer count/size — confirm by reading the existing check site (likely in `WalWriter::new` or a `setup` fn in the persistence crate); the env-var path doesn't touch this.

    //! WAL config with env-var tunables — Phase 19.1-03 (D-01, D-02, D-03).
    //!
    //! - `BEAVA_WAL_BUFFERS` (default 4, range [2, 32])
    //! - `BEAVA_WAL_BUFFER_SIZE_MB` (default 32, range [4, 256])
    //! - `BEAVA_WAL_TICK_MS` (default 20, range [1, 1000])
    //!
    //! Phase 18 WAL architecture invariants (lock-free apply, multi-buffer state
    //! machine, single writer+fsync thread, four-watermark discipline, O_APPEND,
    //! refuse-on-network-FS) are PRESERVED. This module only changes COUNT, SIZE,
    //! and TICK INTERVAL of buffers.
    
    #[derive(Debug, Clone, Copy)]
    pub struct WalConfig {
        pub buffers: usize,
        pub buffer_size_mb: usize,
        pub tick_ms: u64,
    }
    
    impl WalConfig {
        pub const DEFAULT_BUFFERS: usize = 4;
        pub const DEFAULT_BUFFER_SIZE_MB: usize = 32;
        pub const DEFAULT_TICK_MS: u64 = 20;
    
        pub const BUFFERS_MIN: usize = 2;
        pub const BUFFERS_MAX: usize = 32;
        pub const BUFFER_SIZE_MB_MIN: usize = 4;
        pub const BUFFER_SIZE_MB_MAX: usize = 256;
        pub const TICK_MS_MIN: u64 = 1;
        pub const TICK_MS_MAX: u64 = 1000;
    
        pub fn resolve_from_env() -> Self {
            let buffers = parse_clamp_usize(
                "BEAVA_WAL_BUFFERS",
                Self::DEFAULT_BUFFERS,
                Self::BUFFERS_MIN,
                Self::BUFFERS_MAX,
            );
            let buffer_size_mb = parse_clamp_usize(
                "BEAVA_WAL_BUFFER_SIZE_MB",
                Self::DEFAULT_BUFFER_SIZE_MB,
                Self::BUFFER_SIZE_MB_MIN,
                Self::BUFFER_SIZE_MB_MAX,
            );
            let tick_ms = parse_clamp_u64(
                "BEAVA_WAL_TICK_MS",
                Self::DEFAULT_TICK_MS,
                Self::TICK_MS_MIN,
                Self::TICK_MS_MAX,
            );
            WalConfig { buffers, buffer_size_mb, tick_ms }
        }
    }
    
    fn parse_clamp_usize(name: &str, default: usize, lo: usize, hi: usize) -> usize {
        match std::env::var(name) {
            Ok(s) => match s.parse::<usize>() {
                Ok(v) => {
                    let clamped = v.clamp(lo, hi);
                    if clamped != v {
                        tracing::warn!(
                            target: "beava.wal",
                            kind = "wal.config.clamp",
                            env_var = %name,
                            requested = v,
                            clamped = clamped,
                            range_lo = lo,
                            range_hi = hi,
                            "WAL env var clamped to safe range"
                        );
                    }
                    clamped
                }
                Err(e) => {
                    tracing::warn!(
                        target: "beava.wal",
                        kind = "wal.config.parse_error",
                        env_var = %name,
                        value = %s,
                        error = %e,
                        default = default,
                        "WAL env var parse failed; falling back to default"
                    );
                    default
                }
            },
            Err(_) => default,
        }
    }
    
    fn parse_clamp_u64(name: &str, default: u64, lo: u64, hi: u64) -> u64 {
        // Same shape as parse_clamp_usize but for u64. Inline for clarity.
        match std::env::var(name) {
            Ok(s) => match s.parse::<u64>() {
                Ok(v) => {
                    let clamped = v.clamp(lo, hi);

                            env_var = %name,
                            requested = v,
                            clamped = clamped,
                            range_lo = lo,
                            range_hi = hi,
                            "WAL env var clamped to safe range"
                        );
                    }

                        error = %e,
