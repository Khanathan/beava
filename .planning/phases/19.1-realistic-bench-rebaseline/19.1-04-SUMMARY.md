## Decisions Made

1. **SmallVec inline cap = 4** (per CONTEXT D-20) — covers ~99% of typical fraud workloads where only 1-2 buckets are active per entity at any moment. Heap promotion on the 5th distinct-epoch event remains graceful (regular `Vec` underneath); test `test_windowed_op_smallvec_inline_cap_4_then_spills_to_heap` validates the boundary.
2. **`bucket_index()` kept as a pure mathematical function** rather than removed. Pre-fix it returned a slot index `0..64` used to address physical storage; post-fix the SmallVec is keyed by epoch_ms (`bucket_epoch()`), so `bucket_index` no longer addresses storage — but the existing test `windowed_count_bucket_index_is_pure_function_of_event_time` still validates the modular-arithmetic identity. This preserves API backward compat for any external code reasoning at the 64-slot abstraction level.
3. **Eviction policy = `swap_remove` of oldest-epoch entry** (linear scan for min, n ≤ 64). Could be improved to a min-heap or sorted Vec if profiling ever shows eviction cost dominates, but at the typical 1-2 active and worst-case 64-active scale, the constant factor of `swap_remove` wins. Test `test_windowed_op_evicts_oldest_at_max_buckets_cap` validates that pushing 65 distinct epochs leaves exactly 64 entries with the oldest (epoch=0) gone and newest (epoch=64) present.
4. **Snapshot format BREAKS clean** (per CONTEXT target_layout Option A). The OLD parallel-arrays serde representation is incompatible with the NEW SmallVec representation. Phase 7 recovery falls back to WAL replay on deserialization failure; operators with existing snapshots experience a slightly slower restart but no data loss. **Migration recipe** documented inline in module rustdoc and below in this SUMMARY.

## Snapshot Format Compatibility (D-04 / Phase 7 Architectural Invariant)

**Layout change:** the on-disk serde representation of `WindowedOp` changes from:

```
WindowedOp {
  inner_kind, bucket_ms, window_ms,
  buckets: tuple-encoded [Option<Box<AggOp>>; 64],
  bucket_epoch_start_ms: tuple-encoded [i64; 64],
  sketch_params: SketchParams,
}
```

to:

```
WindowedOp {
  inner_kind, bucket_ms, window_ms,
  buckets: SmallVec<[(i64, Box<AggOp>); 4]>,  // serializes as Vec<(i64, Box<AggOp>)>
  max_buckets: usize,
  sketch_params: SketchParams,
}
```

**Existing snapshots WILL fail to deserialize** against the new layout (the bincode/serde tuple wire format changed: 64 fixed slots → length-prefixed Vec-of-tuples + new `max_buckets: usize` field).

**Recovery path:** Phase 7's snapshot recovery falls back to WAL replay when snapshot deserialization fails. Operators with running Beava instances and committed snapshots transitioning across this commit:

```bash
# Operator migration recipe (1 command + restart):
rm -f /path/to/beava/data/*.snapshot
# (the *.wal file stays — restart re-applies all events from WAL on boot)
systemctl restart beava   # or your equivalent
```

This is consistent with Phase 7's snapshot-evolution policy and with v0 OSS launch reality (no shipped snapshots to migrate). The break is documented in:
- `crates/beava-core/src/agg_windowed.rs` module-level rustdoc
- The GREEN commit (`4d553f0`) body
- Plan 19.1 CONTEXT.md target_layout (Option A "clean break")

The `aggop_windowed_sum_30s_roundtrip` test in `crates/beava-core/tests/snapshot_body_roundtrip.rs` validates that NEW-format snapshots round-trip correctly via query-result equality.

## AGG-CORE-09 Cap Preservation

The 64-bucket cap from AGG-CORE-09 / SRV-APPLY-03 was previously enforced implicitly by the fixed array length. With the SmallVec layout it's enforced explicitly:

```rust
pub max_buckets: usize,  // = 64 by default; set via default_max_buckets serde fn
```

Eviction in `update` / `update_with_row`:

```rust
if self.buckets.len() >= self.max_buckets {
    self.evict_oldest_bucket();  // swap_remove of min-epoch entry
}
```

Test `test_windowed_op_evicts_oldest_at_max_buckets_cap`:
1. Pushes 65 events into 65 distinct epochs (window=64, bucket_ms=1).
2. Asserts `op.buckets.len() == 64`.
3. Asserts `epoch=0` is absent (evicted) and `epoch=64` is present (newest).

This is **stricter** than the old behavior — old code used `idx = (epoch / bucket_ms) % 64` with silent overwrite of older epochs at the same slot index. The new policy is deterministic + explicit (oldest epoch evicted, regardless of slot collision).
