### B. Bench wall_clock measurement fix

- **D-06:** Move `let elapsed = start.elapsed();` from after the `for w in workers { ... }; let _ = get_task.await; let _ = rss_task.await;` block to BEFORE that block. Captures honest elapsed time without contamination from 1s `get_task` sleep + 500ms `rss_task` sleep.

## Specific Ideas

### Why "lazy buckets only" for Phase 19.1's complex-pipeline optimization

Phase 19's debug session decomposed the cold-key entity init cost (`entity_row_init` = 2,576 ns mean):
- EntityKey clone + raw_entry insert: ~250 ns
- `Vec<AggOp>::with_capacity(20)`: ~150 ns
- 4 × `Box<WindowedOp>` heap alloc: ~300 ns
- **4 × `[Option<Box<AggOp>>; 64]` zero-init (memset 512B each): ~200 ns**
- **4 × `[i64; 64]` set to `i64::MIN`: ~200 ns**
- BloomMember bit array: ~200 ns
- 15 × core `AggOp::new`: ~450 ns
- Misc: ~300 ns

The `[Option<Box<AggOp>>; 64]` + `[i64; 64]` zero-init combination is **~400 ns × 4 instances = ~1,500 ns** — about 60% of the cold-key init cost. Replacing with `SmallVec<[(i64, Box<AggOp>); 4]>` (lazy alloc; most entities only see 1-2 active buckets) collapses this to ~50 ns when no buckets are active yet, freeing the ~1,500 ns for the productive work.

Same-key batch sketch updates (deferred per D-17) is a different lever — it helps HOT-key complex pipelines (zipfian where 1% of keys get 50% of events). For fraud-team's zipfian shape this would also help; but it requires apply-dispatch peek-ahead/accumulate-then-flush, which is a meaningful complexity bump. Phase 19.1 picks the simpler concrete win first; revisit batching when 19.1 lands and we re-measure.

OP_PUSH_MANY adoption (deferred per D-18) is a wire-stack lift — helps the cache-hot fixed shape that already hits 665k EPS. fraud-team is apply-bound; OP_PUSH_MANY doesn't help the primary tuning bench. Deferred.

### Why "env-tunable WAL with 4×32 MiB default"

The user picked option 4 of the WAL question — middle-ground default + tunables. Rationale:

- **4×32 MiB (~128 MB resident)** is large enough to absorb burst write rate at sustained 500k EPS without the bimodal `wal_append > 1ms` tail (1% of events stalling 1-42ms). Experimental data (8×64 MiB tick=100ms) eliminated the tail entirely and lifted EPS +33% on cache-hot fixed shape; 4×32 MiB is ~4× the original capacity and should capture most of that lift.
- **128 MB RSS** is a reasonable default — less than 2× original 48 MB, well below 8×64 MiB's aggressive 512 MB. Doesn't break cheap-deployment Beava (Raspberry Pi-class isn't a v0 target anyway).
- **tick=20ms** is a 10× longer flush window than original 2ms. Lets the writer batch more events per fsync; reduces fsync calls 10×. Trade-off: up to 20ms of unflushed in-memory data on crash for `acks=1` mode (default per Phase 6.1); `acks=all` mode (`/push-sync`) blocks until fsync regardless.
- **Env tunables** match the user's stated workflow ("we will use [fraud-team] as our main benchmark to tune stuff"). Power users tuning their own deployment can dial `BEAVA_WAL_BUFFERS=8 BEAVA_WAL_BUFFER_SIZE_MB=64 BEAVA_WAL_TICK_MS=100` for 512 MB-RSS max throughput; embedded users can dial `BEAVA_WAL_BUFFERS=3 BEAVA_WAL_BUFFER_SIZE_MB=16` for min RSS. Both modes are validated paths.

### Why fraud-team blast shapes are zipfian-only (for now)
