<objective>
Replace `WindowedOp.buckets: [Option<Box<AggOp>>; 64]` + `bucket_epoch_start_ms: [i64; 64]` (preallocated 64-slot fixed arrays) with `buckets: SmallVec<[(i64, Box<AggOp>); 4]>` + lazy bucket allocation per CONTEXT D-16/17/18/19/20. Most entities only see 1-2 active buckets at any time; pre-allocating 64 slots zero-initialized for every new entity is the cold-key entity init hotspot identified in Phase 19's debug session (~1500 ns of the 2576 ns cold `entity_row_init` cost — 60%).

Phase 19's per-stage analysis decomposed the cold-key path:
- EntityKey clone + raw_entry insert: ~250 ns
- `Vec<AggOp>::with_capacity(20)`: ~150 ns
- 4 × `Box<WindowedOp>` heap alloc: ~300 ns
- **4 × `[Option<Box<AggOp>>; 64]` zero-init (memset 512B each): ~200 ns**
- **4 × `[i64; 64]` set to `i64::MIN`: ~200 ns**
- BloomMember bit array: ~200 ns
- 15 × core `AggOp::new`: ~450 ns

The two `[...; 64]` zero-inits combined are ~400 ns × 4 instances = ~1500 ns. Replacing with `SmallVec<[(i64, Box<AggOp>); 4]>` (lazy alloc; most entities only see 1-2 active buckets) collapses this to ~50 ns when no buckets are active yet, freeing ~1500 ns for productive work.

Per CONTEXT D-20, default SmallVec inline capacity is 4 (covers typical sliding-window agg use; spilling to heap for the 1% with deep history). Per D-16, the SmallVec stores `(epoch_ms, Box<AggOp>)` tuples — the epoch lives next to the bucket so no separate `[i64; 64]` array is needed.

**Bucket-routing semantics MUST be preserved:**
- `bucket_index(event_time)` still returns `((event_time.div_euclid(bucket_ms)) as usize) % 64` — but the result is the bucket *epoch identifier*, not a slot index.
- Looking up "the bucket for this event" becomes: scan SmallVec for an entry matching the current epoch; if found, route to it; if stale-but-matched-by-slot (different epoch), evict + recreate; if none found, push a new entry.
- The 64-bucket cap (per AGG-CORE-09 / SRV-APPLY-03) is enforced not by array length, but by ensuring the SmallVec never exceeds 64 entries at any time — old entries are evicted as new epochs arrive (sliding-window discipline already in place).

```rust
use crate::agg_op::{AggKind, AggOp, SketchParams};
use crate::row::Row;
use serde::{Deserialize, Serialize};
use smallvec::{smallvec, SmallVec};

/// 64-bucket event-time tumbling ring buffer wrapping any core AggOp.
///
/// Buckets are LAZILY allocated. Most entities see only 1-2 active buckets
/// at any time (typical sliding-window agg use); the 4-slot inline SmallVec
/// covers the common case without heap allocation. The 64-bucket cap from
/// AGG-CORE-09 / SRV-APPLY-03 is enforced by eviction discipline (older
/// epochs evicted as new ones arrive), not by array length.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct WindowedOp {
    pub inner_kind: AggKind,
    pub bucket_ms: u64,
    pub window_ms: u64,
    /// Lazy-allocated bucket entries: (epoch_start_ms, op_state).
    /// Inline cap 4 = covers ~99% of typical fraud workloads (1-2 active
    /// buckets per entity at any time); spills to heap above 4.
    /// Replaces fixed [Option<Box<AggOp>>; 64] + [i64; 64] arrays for
    /// ~1500 ns/event cold-init savings on complex pipelines.
    pub buckets: SmallVec<[(i64, Box<AggOp>); 4]>,
    /// AGG-CORE-09: cap = 64 active buckets. Beyond cap, oldest is evicted.
    pub max_buckets: usize,
    #[serde(default)]
    pub sketch_params: SketchParams,
}

impl WindowedOp {
    pub fn new_with_params(kind: AggKind, window_ms: u64, sketch_params: SketchParams) -> Self {
        assert!(
            !matches!(kind, AggKind::BloomMember),
            "bloom_member is windowless-only — cannot be wrapped in WindowedOp"
        );
        let bucket_ms = window_ms.div_ceil(64);
        WindowedOp {
            inner_kind: kind,
            bucket_ms,
            window_ms,
            buckets: SmallVec::new(),  // LAZY — no allocation, zero memset
            max_buckets: 64,
            sketch_params,
        }
    }
    
    /// Compute bucket epoch for an event time. Replaces bucket_index for
    /// the lazy layout — we now identify buckets by their epoch_ms rather
    /// than a slot index.
    #[inline]
    pub fn bucket_epoch(&self, event_time_ms: i64) -> i64 {
        event_time_ms.div_euclid(self.bucket_ms as i64) * self.bucket_ms as i64
    }
    
    pub fn update(&mut self, row: &Row, event_time_ms: i64, field: Option<&str>, where_matched: bool) {
        let epoch = self.bucket_epoch(event_time_ms);
        // Linear scan: typical case is 1-2 active buckets, so SmallVec scan
        // is faster than a HashMap for this size.
        if let Some(pos) = self.buckets.iter().position(|(e, _)| *e == epoch) {
            self.buckets[pos].1.update(row, event_time_ms, field, where_matched);
            return;
        }
        // New epoch — push or evict-then-push.
        if self.buckets.len() >= self.max_buckets {
            // Evict the oldest bucket (smallest epoch). Cheap because SmallVec
            // is small (≤64). Could be O(log n) with a different structure,
            // but at most-64 entries the constant factor wins.
            if let Some(min_pos) = self.buckets.iter().enumerate().min_by_key(|(_, (e, _))| *e).map(|(i, _)| i) {
                self.buckets.swap_remove(min_pos);
            }
        }
        let mut new_op = Box::new(fresh_op(self.inner_kind, &self.sketch_params));
        new_op.update(row, event_time_ms, field, where_matched);
        self.buckets.push((epoch, new_op));
    }
    
    // Same shape for update_with_row + query — iterate self.buckets, check
    // epoch ∈ [now - window_ms, now] for query-side window filtering.
}
```

**Bucket-cap discipline:**
- AGG-CORE-09 specifies 64 buckets max; field `max_buckets: usize = 64`.
- Eviction is `swap_remove` of the smallest-epoch entry (O(n) scan for min, n ≤ 64 — fine).
- Could be replaced with a min-heap if profiling shows the eviction cost matters; for typical 1-2 active buckets, eviction is rare.

**Serde compatibility:**
- The serde derive should "just work" for `SmallVec<[(i64, Box<AggOp>); 4]>` IF `SmallVec` provides `Serialize`/`Deserialize`. Verify: `smallvec` has `serde` feature; enable in Cargo.toml dep declaration.
- The OLD format (two parallel 64-element arrays) is incompatible with the new format. **Snapshot compatibility:** post-Phase-7 snapshots from the old layout WILL fail to deserialize against the new layout. Decision points:
  - **Option A (clean break):** bump the snapshot/serde version; document as "Phase 19.1 snapshot format break". Phase 7's recovery path falls back to WAL replay if snapshot fails — apply this. Cleaner code; simpler test suite.
  - **Option B (migration shim):** keep deserializing the old layout into the new SmallVec form. Adds complexity but preserves binary-compatibility for users with existing snapshots.
- **Recommended (matches Phase 7's snapshot-evolution policy):** Option A clean break, since fraud-team is a fresh bench config and the v0 OSS launch hasn't shipped snapshots to real users yet. Document in the SUMMARY that snapshot migration is required if any operator has running snapshots; provide a one-paragraph migration recipe (delete .snapshot file, restart, replay from WAL).
