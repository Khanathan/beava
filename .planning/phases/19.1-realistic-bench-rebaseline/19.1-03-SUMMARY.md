## Default → Power-User Tuning Table

| Mode | BEAVA_WAL_BUFFERS | BEAVA_WAL_BUFFER_SIZE_MB | BEAVA_WAL_TICK_MS | Resident WAL | Use-case |
|------|--------------------|---------------------------|---------------------|--------------|----------|
| Embedded (min) | `3` | `16` | `2` | 48 MiB | Memory-constrained boxes; aggressive fsync (matches Phase 18 default exactly) |
| **Default (this plan)** | `4` | `32` | `20` | **128 MiB** | Typical fraud-server box; absorbs sustained 500k EPS without bimodal tail |
| Power-user (max practical) | `8` | `64` | `100` | 512 MiB | Big boxes; max throughput; trades 100ms crash-window for batched fsync. Verified to eliminate bimodal `wal_append > 1ms` tail in Phase 19 experimental run |
| Threat-model ceiling | `32` | `256` | `1000` | 8 GiB | OOM-typo guard; clamp does NOT allow above this |

## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Operator sets `BEAVA_WAL_BUFFER_SIZE_MB=99999` → instant 100 GiB allocation at startup | medium | Clamped to [4, 256]. Worst-case = 32 x 256 = 8 GiB ring footprint. Logs `kind=wal.config.clamp` WARN with requested + clamped values |
| Operator sets `BEAVA_WAL_BUFFERS=xyz` (parse-error) → server crashes at startup | medium | Falls back to default with `kind=wal.config.parse_error` WARN. Operator-script-friendly |
| Operator sets `BEAVA_WAL_BUFFERS=0` thinking "disable WAL" → server runs with 0 buffers, loses durability | high if not clamped | Clamped to BUFFERS_MIN=2 (degenerate ring requires at least 2 — current sealed + next active) |
| Operator sets `BEAVA_WAL_TICK_MS=0` hoping for per-event fsync → busy-loop in writer | medium if not clamped | Clamped to TICK_MS_MIN=1 ms. For per-event fsync, users use `/push-sync` (acks=all) — separate code path |
| Phase 18 WAL invariants silently broken | high | Plan body documents D-04 invariants UNCHANGED; code edit only changes constructor arguments. All 119 beava-server tests pass post-change (no regression) |
| Bigger buffers + longer tick = bigger crash window for `acks=1` mode (up to 20ms unflushed data on crash, vs old 2ms) | medium | Documented behavior of `SyncMode::Periodic` (Phase 6.1 design); for `acks=all` durability use `/push-sync`. tick=20ms is 10x the old default; power-user mode dialing tick=100ms accepts 100ms crash window. Documented in commit body |
| New attack surface from env vars | n/a | Env vars are operator-controlled at process start, not user-controlled. No HTTP/TCP endpoint exposes WAL config. ASVS L1 gate: not applicable |

## Decisions Made

- **Default = 4x32 MiB tick=20ms (D-01).** Middle ground between embedded (3x16) and experimental upper-bound (8x64); ~128 MiB resident is reasonable for typical fraud-server boxes; tick=20ms is 10x the old 2ms (batches more events per fsync) but still sub-second-fsync (preserves Beava's "we sub-second-fsync" identity).
- **Clamp ranges (D-02 + threat model).** buffers [2, 32], buffer_size_mb [4, 256], tick_ms [1, 1000]. Min values are minimum-viable (2 buffers = degenerate ring; 4 MB = enough for 1 burst-tick at 500k EPS x 300B; 1ms = aggressive fsync). Max values are OOM-typo guards (32 x 256 = 8 GiB ceiling; 1000ms = 1s lag matches Redis default).
- **Out-of-range = clamp + WARN, not refuse-to-start.** Operators inherit env vars from shell scripts they don't fully control; clamping is more forgiving than crashing.
- **Parse-error = fallback-to-default + WARN, not refuse-to-start.** Same rationale.
- **Wire site = `serve_with_dirs` only.** This is the canonical bench-side and ServerV18 path. The legacy `Server::serve` (axum-only HTTP, used by the current `beava` binary) does not allocate WalBufferRing at all and is irrelevant to the bench-throughput rebaseline this plan supports.
- **Smoke-test verification site = `beava-bench-v18`, not the `beava` binary.** Plan said "smoke run the server with default config to confirm log line appears"; the beava binary uses the legacy axum path which doesn't hit my code. The bench binary uses the canonical ServerV18 path and DID emit the expected `kind=wal.config.resolved` log line at startup with default + override + clamp + parse-error verifications.
