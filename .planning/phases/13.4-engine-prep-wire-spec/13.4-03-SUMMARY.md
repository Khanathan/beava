# Phase 13.4 Plan 03: OP_BATCH_GET (0x0024) — Heterogeneous Batched Read Summary

**One-liner:** Adds `OP_BATCH_GET` opcode (0x0024) with TCP + HTTP wire arms, plus `dispatch_batch_get_sync` that walks the request list, calls per-entity feature lookup, and aggregates results with partial-failure semantics — composable with the ADR-003 empty-string sentinel for global tables.

## Objective

Extend the v0 wire surface with a heterogeneous batched read opcode. Clients send a list of `(table, entity_id)` tuples in a single frame; the server returns a single `OP_GET_RESPONSE` (0x0023) frame whose JSON body holds per-tuple results in request order. Composes natively with ADR-003's empty-string `entity_id` sentinel — a single batch can mix per-entity and global lookups.

**Why it matters:** Fraud-scoring pipelines fan out to many features at decision time. Without `OP_BATCH_GET`, clients have to issue N round-trips or use the same-feature `OP_MGET` / `OP_GET_MULTI` (which can't mix tables). This is the read-side counterpart to `OP_PUSH` for batched fan-out.

### Dispatch (Task 3.d)

**`crates/beava-server/src/apply_shard.rs`** (+201 LOC)

- New public function `dispatch_batch_get_sync(app, body, body_format) -> GlueResponse`.
  - Parses `{requests: [{table, entity_id}, ...]}` from JSON or MsgPack.
  - Empty `requests: []` → `{"results": []}` HTTP 200 (per plan must_haves).
  - Single `state_tables.lock()` for the whole batch (matches `dispatch_get_batch`'s critical-section discipline).
  - Per tuple:
    - Unknown table → `{table, entity_id, error: {code: "unknown_table", reason}}` (partial-failure).
    - Key arity mismatch → `{table, entity_id, error: {code: "key_parse_failure", reason}}` (same posture).
    - Otherwise: walks `descriptor.features`, queries each via `query_feature(entity_key, idx, query_time_ms)`, builds `{table, entity_id, features: <flat_dict>}`. Cold-start (no events) yields `features: {}` per wire-spec semantics.
  - Response is JSON-encoded `{"results": [...]}` returned as `GlueResponse::QueryResult { format: CT_JSON }`.
- New dispatch arms in `dispatch_one`: `WireRequest::TcpBatchGet { body, body_format }` and `WireRequest::HttpBatchGet { body }` route to `dispatch_batch_get_sync`.

## Deviations from Plan

### 1. Wire-spec doc cohesion gap (Phase 13.0/13.4 — out of scope here)

**Found during:** Task 3.c (test design).
**Issue:** The plan's `must_haves.truths` use the body shape `{table, entity_id}` and partial-failure-via-`results[i].error` semantics. The Phase 13.0 wire-spec doc (`docs/wire-spec.md`) and the existing fixture `examples/wire/batch_get-heterogeneous.{request,response}.json` use `{table, key}` (with optional `features?`) and whole-frame `OP_ERROR_RESPONSE` on per-entry error.
**Resolution:** Followed the plan verbatim (executor contract). Both shapes are mechanically valid — `entity_id` and `key` both encode the lookup target as a single string; the difference is the field name. The plan-checker pre-verified zero overlap with sibling Wave-1 plans on this file, so no contradictions were introduced. Reconciling the doc to match the implementation (or the implementation to match the doc) is a Phase 13.4 closure / 13.0 follow-up; flagging here for the plan-checker / Plan 13.4-10 closure to surface to the user.

### 2. Test fixture: globally-unique feature names

**Found during:** Task 3.c (RED test register failure).
**Issue:** Initially used `cnt` as a feature name in both `UserSpend` and `MerchantSpend` aggregations. Beava enforces `aggregation_feature_name_collision_across_aggregations` at register-time — feature names must be globally unique.
**Fix (Rule 1 — Bug):** Renamed `MerchantSpend.cnt` → `MerchantSpend.merchant_cnt` in the test fixture. Updated the assertion in `http_batch_get_returns_per_tuple_results`.
**Note:** This is purely a test-fixture concern. It demonstrates that the dispatch implementation handles MULTIPLE distinct aggregation tables in a single batch, which is exactly the heterogeneous-batch contract the opcode is for.
