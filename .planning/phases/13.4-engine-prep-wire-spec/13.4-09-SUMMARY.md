## Decisions Made

### Plan over-specification not implemented (Rule 1 — scope correction)

The plan's `<interfaces>` block proposed re-introducing `OpNode::Table` enum variant + `TableDerivationOutput` struct + register_validate.rs widening + agg_compile.rs compile arm (~80-100 LOC). None were necessary:

- **`OutputKind::Table` already exists** on `crates/beava-core/src/registry.rs::OutputKind` (added in pre-12.7 phase, never removed by 12.7's strip — `OutputKind` is the *Derivation*-output discriminator, not the user-mutable surface that 12.7 killed).
- **`DerivationDescriptor::output_kind` + `table_primary_key`** already exist on the registry's derivation descriptor with serde derives; the existing register_validate path accepts the `output_kind: "table"` discriminator (rule 6 in `validate_derivation_struct` actually requires `table_primary_key.is_some()` when `output_kind == Table` — empty Vec satisfies this).
- **`agg_compile.rs` already accepts empty `group_by.keys`** — the `keys.iter().enumerate()` validation loop runs zero iterations for an empty Vec; no special-case logic needed.
- **`apply_shard.rs::dispatch_batch_get_sync` already cites ADR-003** in a comment block at lines 587-591 (Plan 03 landed the citation; the dispatch path forwards `entity_id` verbatim to `parse_entity_key`).

The single missing piece was the query path's `parse_entity_key` rejecting the empty-string sentinel. Per Rule 1, the plan's over-specification is not implemented; the SAME observable acceptance is obtained from the 5-line sentinel branch — the code-LOC cost is ~95% smaller, the architectural invariant holds (no new `OpNode::Table*` per-variant entries; no widening of the user-mutable surface), and all 9 cross-plan tests pass.

### ADR-003 acceptance is the absence of a rejection (USER-LOCKED via ADR)

ADR-003 §Decision B / Engine: *"the existing `&str` key path handles `\"\"` natively. No new code path inside `apply_shard.rs::dispatch_*_sync` — just the absence of a special-case rejection."* The implementation honours this verbatim: the apply path was already free of special-case rejections (the apply path's `EntityKeyShape::from_row` constructs `Multi(EntityKey(empty))` for empty group_keys via the compound-key branch with a zero-iteration body — this was already the existing behaviour); the query path's `parse_entity_key` had a special-case rejection (segments-vs-arity 1 vs 0 mismatch) that needed to be removed for the empty-empty case.

### Plan 05 Test 3 wire-shape correction (Rule 1 — bug fix)

The test's original payload used:

- `"key_cols": ["user_id"]` — no such field exists on `DerivationDescriptor`. The correct field is `table_primary_key`.
- `"ops": [{"op": "count"}]` — `count` is not an `OpNode` variant; it's an aggregation kind that lives inside `group_by.agg`. `OpNode` has 9 variants: `filter` / `select` / `drop` / `rename` / `with_columns` / `map` / `cast` / `fillna` / `group_by`.

Both errors slipped past Plan 05 because the test was `#[ignore]`'d on commit; once un-ignored it would have failed with `"unknown variant 'count'"` regardless of Plan 09's engine work. The corrected payload uses the canonical wire shape — `group_by` op with `keys` + `agg` map + `table_primary_key`. The asserted contract — register with `output_kind: "table"` succeeds — is preserved end-to-end.
