## Architectural Behavior

### Boot-time test_mode resolution

```
ServerV18::bind_with_config(http_addr, tcp_addr, admin_addr,
    ServerV18Config { persistence, test_mode })
  → effective_test_mode = cfg.test_mode
                           || env::var("BEAVA_TEST_MODE")
                                .map(|v| v == "1")
                                .unwrap_or(false)
  → build_runtime_state_with_persistence(persistence, .., effective_test_mode)
      → AppState::new(dev_agg, wal_sink, idem_cache)
      → app_state_inner.effective_test_mode = effective_test_mode
      → tracing::warn! (if true) — server.test_mode_enabled
```

Boot-time-only — read once, cached on the Arc<AppState>, never re-read. Runtime escalation is impossible per T-13.4-08-03.

Legacy `bind()` and `bind_with_state()` callers do NOT compute the env-var resolution — they default to `false`. The env-var path is reachable only through `bind_with_config` (the constructor `main.rs` and TestServer-style fixtures use). This is intentional: the env var is the deployment-knob path paired with `bind_with_config`; no caller can accidentally pick it up via the legacy constructors.

### Reset dispatch

```
WireRequest::TcpReset { body, body_format } | WireRequest::HttpReset { body }
  → apply_shard::dispatch_reset_sync(state)
      → if !state.effective_test_mode:
          return GlueResponse::ResetForbidden
      → state_tables.lock().clear()
      → state.dev_agg.registry.clear()       // bumps registry_version by 1
      → state.dev_agg.next_event_id.store(0, Release)
      → state.dev_agg.query_time_ms.store(0, Release)
      → tracing::info! — server.reset_completed
      → return GlueResponse::ResetOk { registry_version }
```

### Encoder mapping

| GlueResponse | HTTP status | HTTP body | TCP opcode | TCP body |
| --- | --- | --- | --- | --- |
| `ResetOk { registry_version: N }` | 200 OK | `{"reset": true, "registry_version": N}` | `OP_GET_RESPONSE (0x0023)` | same JSON body |
| `ResetForbidden` | 403 Forbidden | `reset_forbidden_body()` (see below) | `OP_ERROR_RESPONSE (0xFFFF)` | same JSON body |

Forbidden body (single canonical helper used by both encoders):

```json
{
  "error": {
    "code": "reset_disabled_in_production",
    "reason": "OP_RESET requires server test_mode (set BEAVA_TEST_MODE=1 or pass Config { test_mode: true } at server construction). See docs/error-codes.md."
  }
}
```

The reason text mentions BOTH opt-in paths so users see actionable error text — pinned by `phase13_4_reset_default_rejected::default_config_no_env_var_post_reset_returns_403_structured`.

## Deviations from Plan

### Auto-decided

**1. `OP_RESET_RESPONSE` was NOT created — reuse `OP_GET_RESPONSE (0x0023)` for the success frame**
- **Found during:** Task 8.d wiring of the TCP encoder.
- **Issue:** Plan's `<behavior>` for Task 8.a Test 2 said "pick `OP_GET_RESPONSE` (0x0023) since it's the existing generic-JSON-response frame, or define a new `OP_RESET_RESPONSE`. Auto-decision: reuse `OP_GET_RESPONSE` for symmetry."
- **Fix:** Picked the planner-default reuse path. The TCP encoder emits `OP_GET_RESPONSE` for `ResetOk`. Test 2 in `phase13_4_reset_gated_env_var.rs` asserts on this opcode.
- **Files modified:** `crates/beava-server/src/server.rs` (encoder), `crates/beava-server/tests/phase13_4_reset_gated_env_var.rs` (assertion).

**2. `reset_disabled_in_production` REPLACES (rather than ALIASES) the pre-D-03 sketch `reset_disabled` entry in docs/error-codes.md**
- **Found during:** Task 8.d docs edit.
- **Issue:** `docs/error-codes.md` already had a `reset_disabled` entry (HTTP 403, `enable_reset_op=false` config flag) — a pre-D-03 sketch from an earlier design pass. Plan calls for `reset_disabled_in_production` per D-03. The two could coexist as aliases or one could replace the other.
- **Fix:** Replaced the sketch entry in-place. The pre-D-03 entry described a single-flag gate (`enable_reset_op=false`) that does not exist in the codebase; keeping it as an alias would document a non-existent surface. Updated the HTTP status mapping table accordingly.
- **Files modified:** `docs/error-codes.md`.

**3. `Registry::clear()` was added on `beava-core::Registry` rather than inline in dispatch_reset_sync**
- **Found during:** Task 8.d implementation.
- **Issue:** `RegistryInner` is private; `inner.write()` is also private. dispatch_reset_sync needed to drop ALL descriptors + compiled chains + reverse indices. Two options: (1) expose `inner.write()` publicly so the server can mutate; (2) add a single targeted `clear()` method on `Registry`.
- **Fix:** Added `Registry::clear()` (option 2). Cleaner — narrows the public surface to a single contract method. Bumps `version` by 1 to preserve monotonicity.
- **Files modified:** `crates/beava-core/src/registry.rs`.

**4. `next_event_id` and `query_time_ms` atomics ALSO reset to 0 (beyond plan spec)**
- **Found during:** Task 8.d dispatch_reset_sync drafting.
- **Issue:** Plan's `<action>` describes Step 7 as "1. Clear all entity state. 2. Deregister all descriptors. 3. Build success response." It does not explicitly mention `next_event_id` or `query_time_ms`. After reset, these counters would still hold stale values from before the reset, which would confuse subsequent push/get operations (event_ids continuing monotonically into a fresh registry, query_time_ms locked at the pre-reset wall clock).
- **Fix:** dispatch_reset_sync also resets both atomics to 0 per Rule 2 (auto-add missing critical functionality — correctness requirement). The "clear" semantic in D-03 is "fresh empty state"; counters are part of that state.
- **Files modified:** `crates/beava-server/src/apply_shard.rs`.
