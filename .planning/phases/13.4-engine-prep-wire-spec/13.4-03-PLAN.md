<objective>
Add a new wire opcode `OP_BATCH_GET = 0x0024` plus its parser, dispatch, and response shape per Phase 13.0 wire-spec. The opcode is the read-side counterpart to `OP_PUSH` for batched fan-out reads — clients send heterogeneous (table, entity_id) tuples in a single frame and receive a single batched response. This composes natively with ADR-003's empty-string `entity_id` sentinel (global tables): a single batch can mix per-entity and global lookups.

**Why this matters:** Fraud-scoring pipelines fan out to many features at decision time (per `project_beava_product`: "batch lookup for sub-millisecond fraud/feature-serving decisions"). Without `OP_BATCH_GET`, clients have to issue N round-trips or use the heterogeneous-key-but-same-feature-set `OP_MGET` / `OP_GET_MULTI` (which can't mix tables).

Frame format (Phase 2.5 standard):
```
[u32 length-be][u16 op-be][u8 content_type][payload bytes...]
op = 0x0024 (OP_BATCH_GET)
content_type = 0x01 (CT_JSON) for now; 0x02 (CT_MSGPACK) reserved for v0.1
```

Request payload:
```json
{"requests": [
  {"table": "UserSpend", "entity_id": "alice"},
  {"table": "UserSpend", "entity_id": "bob"},
  {"table": "GlobalCounter", "entity_id": ""}    // global per ADR-003
]}
```

Response (frame is OP_GET_RESPONSE = 0x0023, payload below):
```json
{"results": [
  {"table": "UserSpend", "entity_id": "alice", "features": {"cnt": 12, "total": 145.50}},
  {"table": "UserSpend", "entity_id": "bob",   "features": {"cnt": 3,  "total": 22.00}},
  {"table": "GlobalCounter", "entity_id": "",  "features": {"events_total": 12345}}
]}
```

Partial failure (table "X" doesn't exist):
```json
{"results": [
  {"table": "UserSpend", "entity_id": "alice", "features": {...}},
  {"table": "X", "entity_id": "bob", "error": {"code": "unknown_table", "reason": "..."}}
]}
```
