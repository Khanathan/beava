<decisions>
## Implementation Decisions

### D-01 — force=True diff matrix: STRICT classification + categorized-lists payload (USER-LOCKED)

- **Destructive (requires force=True):** rename, type-change, op removal, agg removal, window-change, key-cols change
- **Additive (allowed without force):** new descriptor, new agg in existing block, new field on event source
- **Diff payload format:** `{"additive": [...], "destructive": [...]}` — categorized JSON lists (NOT JSON-Patch)
- Default register hits 409 with `force_required` error + diff payload
- `force=True` applies destructively + bumps `registry_version` + WAL records the change
- **Why strict:** silent recompute on window-change confuses users; force is the explicit opt-in for any mutation that could discard or recompute existing per-entity state.

### D-02 — In-memory persistence = pure in-RAM, NO WAL (USER-LOCKED)

- `Persistence::Memory` variant: state lives in RAM only; no replay log; no recovery
- On process restart: state is gone (clean slate)
- Snapshot is no-op (no disk writes at all in memory mode)
- **Why:** embed mode (`bv.App()` no-URL) is single-process anyway; recovery has no value if user restarts the binary. Cheapest implementation; zero complexity. Tests + notebook usage are the use case; production stays on disk.
- Acceptance: `tests/phase13_4_memory_persistence.rs` boots in memory mode, pushes events, asserts state present; restarts in memory mode, asserts cold-start (NO replay).

### D-03 — OP_RESET = full clear (state + registry); GATED on test_mode (USER-LOCKED)

- Wire payload: empty `{}` body
- Effect when in test_mode: deletes ALL entity state + deregisters ALL descriptors; user must re-register before next push; `registry_version` bumps to 0 (or resets per planner discretion)
- Effect when NOT in test_mode: returns structured error `reset_disabled_in_production` (HTTP 403 / wire opcode 0xFFFF)

**Server-side test_mode gate is set THREE ways (any one enables):**
1. **Shell env var:** `BEAVA_TEST_MODE=1` at server boot (ops/CI use case)
2. **Programmatic at server startup (Rust):** `Server::new(Config { test_mode: true, .. })` — for integration tests that spawn a server in-process
3. **Programmatic from SDK in embed mode:** `bv.App(test_mode=True)` → SDK passes `BEAVA_TEST_MODE=1` to the spawned embed binary; for `beava.test.fixture()` defaulting to `test_mode=True`. Network mode (`bv.App(url=..., test_mode=True)`) ignores the kwarg with a warning (the server controls; client can't override).

**Why this matters (USER directive):** test code shouldn't have to depend on shell env vars to enable reset. Test fixtures can do `app = bv.App(test_mode=True)` or `with beava.test.fixture() as app:` and have reset work — entirely defined in test code. Production-by-default semantics preserved (default `bv.App()` produces production-mode embed; explicit opt-in for test mode).

- New error code: `reset_disabled_in_production` added to `docs/error-codes.md`
- New tests:
  - `tests/phase13_4_reset_gated_env_var.rs` — env var path
  - `tests/phase13_4_reset_gated_programmatic.rs` — Rust `Server::new(Config { test_mode: true })` path
  - `tests/phase13_4_reset_default_rejected.rs` — default config (no test_mode anywhere) → reset rejected
- **Why this gate:** prevents prod servers being wiped accidentally (or maliciously) from a misconfigured client. Server controls; SDK can request but cannot override.

### D-04 — Architectural-test allowlist: SURGICAL PERMIT (USER-LOCKED)

- Edit `crates/beava-server/tests/phase12_7_no_table_surface.rs::FORBIDDEN_PATTERNS` in place: remove `OpNode::Table` from forbidden list
- Add new test `crates/beava-server/tests/phase13_4_table_derivation_allowed.rs` asserting registration of `output_kind: "table"` derivations succeeds

- Existing `phase12_7_legacy_table_handlers_killed.rs` UNCHANGED (file deletions still enforced)
- **Why surgical:** simplest change; minimal new infrastructure; matches ADR-001's narrow partial-overturn intent.
