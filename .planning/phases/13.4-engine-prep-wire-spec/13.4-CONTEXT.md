<decisions>
## Implementation Decisions

### D-01 — Op renames: hard-rename, no aliases (ADR-002)

**Decision.** Rename the 5 ops in Rust source + tests + bench configs in a single pass. NO aliases for backward compat (`avg` → `mean` is a hard cut; old name produces `unknown_op` error from JSON-prelude shim).

**Why:** v0 isn't released. Backward compat doesn't matter. Aliases would carry maintenance burden forever for zero user value. ADR-002 already locked the rename direction.

**How to apply:**
- `crates/beava-core/src/agg_op.rs` — rename `AggKind::Avg` → `AggKind::Mean`, `Variance` → `Var`, `Stddev` → `Std`, `CountDistinct` → `NUnique`, `Percentile` → `Quantile`. Update all match arms.
- `crates/beava-core/src/register_validate.rs` — rename in op-name-to-AggKind dispatch (the `op_str` match)
- `crates/beava-core/src/op_lifetime_bounds.rs` — rename in classification table
- `crates/beava-bench/configs/{fraud-team,small,medium,large}.json` — rename op-name strings
- All `tests/*.rs` files — rename op references; update assertions
- `python/beava/_agg.py` — Phase 13.5's job (this phase doesn't touch Python)

### D-02 — GET response: flat dict (no `row` envelope)

**Decision.** Per Phase 13.0-15 closure deviation, the actual wire shape uses flat dict (`{"feature_a": value, "feature_b": value}`), NOT `{"row": {...}}` envelope. Cold-start returns `{}` (empty dict).

**Why:** Phase 13.0-15 closure landed `examples/wire/get-global.response.json` and similar fixtures using flat-dict. 13.4 conforms to the existing locked fixtures.

**How to apply:**
- `crates/beava-runtime-core/src/wire_response.rs` (or equivalent) — GET response = flat dict serialized via existing `Row` type
- `crates/beava-server/src/apply_shard.rs::dispatch_get_*_sync` — return flat dict, not wrapped
- New test: `tests/phase13_4_get_row_shape.rs` — fetch single feature, assert flat dict; cold-start asserts `{}`

### D-03 — OP_BATCH_GET (0x0024): list-of-dicts response

**Decision.** Wire payload: `{"requests": [{"table": "T1", "entity_id": "alice"}, {"table": "T2", "entity_id": "bob"}, ...]}` → response: `{"results": [{...}, {...}, ...]}` (one flat dict per request, in same order). Cold-start per-entity is `{}`. Errors per-item via top-level `error` key on the failed entry.

**Why:** Heterogeneous batch (different tables per item) is the locked surface from CONTEXT.md. Order preservation is essential for SDK consumers to map back to their requests. Per-item error reporting handles partial-failure cases without aborting the whole batch.

**How to apply:**
- `crates/beava-core/src/wire.rs` — add `OP_BATCH_GET = 0x0024` constant
- `crates/beava-runtime-core/src/wire_request.rs` — `WireRequest::HttpBatchGet { requests: Vec<(String, String)> }` and TCP variant
- `crates/beava-runtime-core/src/router.rs` — `Route::BatchGet`
- `crates/beava-runtime-core/src/http_listener.rs` — POST /batch_get handler
- `crates/beava-server/src/apply_shard.rs::dispatch_batch_get_sync` — iterate requests; per-item lookup; collect into Vec<Row>
- New test: `tests/phase13_4_batch_get_heterogeneous.rs` — 3 different tables in one batch; assert order; assert cold-start = {}

### D-04 — HTTP routes: strict verb-style POST + JSON body

**Decision.** All 6 v0 endpoints are POST with JSON body — `/register`, `/push`, `/get`, `/batch_get`, `/reset`, `/ping`. NO path parameters (e.g., NOT `POST /push/{stream}` — stream name in JSON body instead).

**Why:** Per Phase 13.0-03 docs (http-api.md). Consistent style; CDN/WAF friendly; SDK code is uniform across endpoints. Pre-13.4 routes had mixed shapes (path params + body) which the wire-spec retired.

**How to apply:**
- `crates/beava-runtime-core/src/router.rs` — replace any `Route::Push { stream }` (path param) with `Route::Push` (parsed from JSON body)
- `crates/beava-runtime-core/src/http_listener.rs` — match on POST method + path; deserialize body
- Existing 4 admin GET routes (`/health`, `/ready`, `/metrics`, `/registry`) STAY GET on the admin sidecar (different port; per `project_phase18_no_dual_runtime`)
- Test: `tests/phase13_4_verb_style_routes.rs` — assert all 6 data-plane routes are POST; assert legacy path-param routes return 404

### D-05 — force=True register: additive vs destructive matrix

**Decision.** Two-bucket classification:
- **Additive (no force needed):** new descriptor, new agg in existing block, new event field
- **Destructive (requires force=True):** rename, type-change, op removal, agg removal, window change, key column change

Behavior:
- Default register: destructive change → 409 with structured `force_required` error + diff payload
- `force=True`: applies destructively + bumps `registry_version` + WAL records the registry change
- The diff payload (also returned by `dry_run=True`) is JSON: `{"additive": [...], "destructive": [...]}`

**Why:** Matches the locked semantics from `docs/schema-evolution.md` (Plan 13.0-12). `force=True` is the explicit user opt-in for destructive change; absent it, server protects.

**How to apply:**
- `crates/beava-core/src/register_validate.rs` — add `compute_diff(old: &Registry, new: &Registry) -> Diff` function (~150 LOC)
- Wire payload: `{"force": true, "descriptors": [...]}` flag
- Error code: `force_required` (already in `docs/error-codes.md`)
- Tests: `tests/phase13_4_force_register.rs` — additive succeeds without force; destructive fails 409; with force=True succeeds

### D-06 — dry_run=True register: returns diff, doesn't apply

**Decision.** Returns the diff payload (same shape as the `force_required` 409 body) with HTTP 200; `registry_version` unchanged; WAL untouched. Reuses `compute_diff` from D-05.

**Why:** Lets users preview a register operation before committing. Useful for CI checks, schema-evolution review tools.

**How to apply:**
- `crates/beava-core/src/register_validate.rs` — `compute_diff` reused; new `register_dry_run(diff) -> DryRunResult` wrapper (~30 LOC)
- Wire payload: `{"dry_run": true, "descriptors": [...]}` flag
- Tests: `tests/phase13_4_dry_run_register.rs` — assert no state mutation; assert diff returned

### D-07 — In-memory persistence backend: bounded-ring WAL

**Decision.** New variant `Persistence::Memory` with bounded ring buffer (default 16 MiB; configurable via env var `BEAVA_MEMORY_WAL_CAP_BYTES`). Old entries evicted FIFO when buffer fills. Snapshot operation is no-op in memory mode (state still in RAM; no disk writes).

**Why:** Embed mode (`bv.App()` no-URL) needs a fast no-disk option for notebooks + tests. Bounded-ring gives crash-tolerance within session (replay last N MiB of events on restart). NOT for production — that's `Persistence::Disk` (existing default).

**How to apply:**
- `crates/beava-persistence/src/lib.rs` — add `Persistence::Memory { ring: BoundedRing }` enum variant (~150 LOC)
- `crates/beava-persistence/src/memory.rs` — new module with `BoundedRing` impl (~80 LOC)
- `crates/beava-server/src/main.rs` — accept `--persistence memory` CLI flag
- Tests: `tests/phase13_4_memory_persistence.rs` — boot in memory mode; push events; restart in memory mode (NO replay since memory is gone); assert clean state

### D-08 — OP_RESET (0x0040): clears state, preserves registry

**Decision.** Wire payload: empty `{}` body. Effect: deletes all entity state for all tables; `registry_version` unchanged; descriptors stay; subsequent `app.get()` returns `{}` (cold-start) for all entities.

**Why:** Reset is for test/dev workflows ("clear state between test cases" / "reset demo data"). Preserving registry means users don't re-register after reset. Keeps the simple model: register once, push events, query, reset cycle.

**How to apply:**
- `crates/beava-core/src/wire.rs` — `OP_RESET = 0x0040` constant
- `crates/beava-runtime-core/src/wire_request.rs` — `WireRequest::HttpReset` variant
- `crates/beava-runtime-core/src/router.rs` — `Route::Reset`
- `crates/beava-runtime-core/src/http_listener.rs` — POST /reset handler
- `crates/beava-server/src/apply_shard.rs::dispatch_reset_sync` — `state_table.iter_mut().for_each(|t| t.entities.clear())` + WAL-record the reset (so replay reproduces)
- Tests: `tests/phase13_4_reset.rs` — push events; assert state; reset; assert cold-start; assert registry_version unchanged

### D-09 — Architectural-test allowlist: additive permit for table derivation

**Decision.** Update `crates/beava-server/tests/phase12_7_no_table_surface.rs` allowlist to permit `OpNode::Table` AS A REGISTERABLE OUTPUT KIND (i.e., `output_kind: "table"` in derivation register payloads). KEEP the existing rejection of `RecordType::TableUpsert/TableDelete/Retract`, `TemporalStore`, `MvccVersion`, `app.upsert/delete/retract` symbols.

**Why:** ADR-001 partial-overturn permitted `@bv.table` aggregation-output decorator. The architectural test must allow that surface while still rejecting user-mutable table verbs. Surgical allowlist edit, not a wholesale rewrite.

**How to apply:**
- Edit `phase12_7_no_table_surface.rs::FORBIDDEN_PATTERNS` — remove `OpNode::Table` from forbidden list (or add explicit allowlist entry)
- Add new test `phase13_4_table_derivation_allowed.rs` — register a derivation with `output_kind: "table"`; assert success
- Existing test `phase12_7_legacy_table_handlers_killed.rs` UNCHANGED (file deletions still enforced)

### D-10 — Global-table sentinel routing (ADR-003)

**Decision.** Accept `key: []` (empty array) at register-time as the wire-level signal for global table. Route via sentinel `entity_id = ""` (empty string) through the existing per-entity hashmap machinery — NO new dispatch path; the existing `&str` hashmap handles `""` natively.

**Why:** ADR-003 locked this design. Cheapest implementation: leverage existing per-entity machinery; the "global" entity is just an entity with a fixed empty-string key. Memory: one row per global table (negligible overhead).

**How to apply:**
- `crates/beava-core/src/register_validate.rs` — accept `key: []` in TableDescriptor; mark table as `is_global: true` internally
- `crates/beava-server/src/apply_shard.rs::dispatch_push_sync` — when table is global, key = `""` regardless of event payload (don't extract from event); state lookup uses `""` as hashmap key
- `crates/beava-server/src/apply_shard.rs::dispatch_get_sync` — when entity_id == `""` AND table is global, lookup `""` (returns flat dict or `{}`)
- Test: `tests/phase13_4_global_table.rs` — register global table; push events for multiple entities; assert single global state; `app.get(table, "")` returns aggregated; `app.get(table, "alice")` raises wrong-arity error

## Specific Ideas

- **D-01 hard-rename** matches the v0-not-released stance (no aliases; clean cut). ADR-002 is authoritative.
- **D-02 flat-dict** matches Phase 13.0-15's actual wire fixtures (`get-global.response.json`). Engine conforms to spec, not vice versa.
- **D-03 OP_BATCH_GET** order preservation is non-negotiable — SDK porters need to map response items back to request items.
- **D-04 verb-style** is the public contract. Pre-13.4 mixed-shape routes go away. SDK porters in 13.5/13.6 ONLY see verb-style.
- **D-05/D-06** force/dry_run are paired — same diff logic, different effect (apply vs preview).
- **D-07 in-memory** is for embed mode + tests only; production stays on disk-WAL. Bounded-ring is the cheapest crash-tolerance.
- **D-08 reset preserves registry** — keeps the user model simple (register once, then events+queries+resets).
- **D-09 surgical allowlist edit** — existing CI tripwires stay green; new tripwire `phase13_4_table_derivation_allowed.rs` locks the new permission.
- **D-10 sentinel `""`** — the cheapest possible implementation; no new code paths, just remove a would-be rejection.
- **Estimated 8-10 plans across 3 waves** matching ROADMAP §13.4 estimate (the new D-10 plan adds ~30 LOC to Wave 3).
