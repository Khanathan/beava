## Decisions Made

### D-04 (USER-LOCKED) — Surgical permit per ADR-001 partial overturn

ADR-001 narrowly revives `@bv.table` for **aggregation-output only** (no upsert/delete/retract; no MVCC). The over-broad `"OpNode::Table"` entry in FORBIDDEN_PATTERNS substring-matched both the bare `OpNode::Table` token and every variant, blocking Phase 13.4-09 (sentinel routing) and Phase 13.5 (Python `@bv.table` decorator) from re-introducing the variant on the derivation-output side. The surgical permit removes only that bare entry; the per-variant entries (`OpNode::TableUpsert`, `OpNode::TableDelete`) — plus the per-variant `RecordType::TableUpsert/TableDelete/Retract`, `WireRequest::HttpUpsert/HttpDelete/HttpRetract/HttpTableGet`, and `Route::Upsert/Delete/Retract/TableGet` entries — continue to enforce the user-mutable surface deny.

Six lockdown layers still cover the user-mutable write surface:

| Layer | Mechanism |
| --- | --- |
| OpNode write variants | `OpNode::TableUpsert`, `OpNode::TableDelete` (per-variant FORBIDDEN_PATTERNS) |
| WAL records | `RecordType::TableUpsert`, `RecordType::TableDelete`, `RecordType::Retract` (FORBIDDEN_PATTERNS) |
| Wire request variants | `WireRequest::HttpUpsert/HttpDelete/HttpRetract/HttpTableGet` (FORBIDDEN_PATTERNS) |
| Route enum | `Route::Upsert/Delete/Retract/TableGet` (FORBIDDEN_PATTERNS) |
| Misc symbols | `TemporalStore`, `MvccVersion`, `temporal_http`, `push_table`, `delete_table`, `fn retract(` (FORBIDDEN_PATTERNS) |
| Top-level register-time JSON shim | `pre_check_unsupported_node_kind` from Plan 12.7-01 — rejects `{nodes:[{kind:"table",...}]}` at HTTP 400 with `unsupported_node_kind`. Pinned by `phase13_4_table_derivation_allowed::top_level_table_register_still_rejected`. |

### Plan-prose deviation: 2 vs 3 per-variant entries

The plan's `<interfaces>` block (lines 85-91 of `13.4-05-PLAN.md`) and acceptance criterion `≥ 3` suggested three per-variant entries (`TableUpsert`, `TableDelete`, `TableRetract`). The live FORBIDDEN_PATTERNS list at start of plan only contained **two** OpNode per-variant entries — `TableUpsert` and `TableDelete`. `OpNode::TableRetract` was never present (a discrepancy in the plan's interface stub vs. the actual file).

**Resolution:** kept both per-variant entries (`TableUpsert`, `TableDelete`); the retract surface enforcement was never the responsibility of an `OpNode::TableRetract` entry — it is enforced via `RecordType::Retract`, `WireRequest::HttpRetract`, `Route::Retract`, and `fn retract(`, all UNCHANGED. The user-mutable retract surface stays fully enforced. This is a documentation-only deviation; the architectural invariant is preserved exactly as ADR-001 specifies.

This was tracked as a Rule-1 deviation (correctness over plan-prose consistency) and called out in the Task 5.b commit body.
