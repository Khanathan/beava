## Architectural Behavior

### Memory-mode boot path (`Persistence::Memory`)

```
bind_with_config(http, tcp, admin, cfg { persistence: Memory, test_mode: false })
  → bind(http, tcp, admin)                        // listeners + admin (unchanged)
  → build_runtime_state_with_persistence(Memory, ..)
      → init AppState { dev_agg, idem_cache }
      → SKIP recovery::load_snapshot_if_any(..)        ← D-02
      → SKIP recovery::replay_wal_from_lsn(..)         ← D-02
      → SKIP recovery::replay_handrolled_wal_dir(..)   ← D-02
      → wal_sink = WalSink::spawn_no_op()              ← D-02 (drops events on the floor)
      → wal_ring = WalBufferRing::new(..)              ← apply hot-path still uses this
      → wal_writer = spawn_no_op_wal_writer(ring, lsn) ← drains sealed → free, NO disk I/O
      → snapshot_task = None                            ← D-02 (scheduler not spawned)
      → ServerV18State { snapshot_task: None, .. }
```

### Disk-mode boot path (`Persistence::Disk { .. }`)

Identical to the pre-Plan-13.4-07 path. The refactor extracted the existing body into the `Persistence::Disk { .. }` arm without functional changes; the `bind()` and `bind_with_state()` constructors continue to land here through `Default::default()` returning `Persistence::Disk`.

### Why no-op WAL writer (and not skip the ring entirely)?

The mio data-plane apply path writes events to `WalBufferRing` via `wal_ring.append(..)`. The ring's backpressure guarantee (block on `free_condvar` when all buffers are sealed and no writer thread is freeing them) means we MUST have a thread that pops sealed buffers and returns them to the free pool — otherwise the apply thread blocks once `buffers × buffer_size_mb` worth of events accumulate (default 4 × 32 MiB = 128 MiB per process, then deadlock).

The no-op writer thread (`spawn_no_op_wal_writer`) mirrors `run_writer_loop`'s structure (sleep → seal_active → drain sealed buffers → check shutdown) but replaces `file.write_all` + `sync_file` with direct `return_to_free`, advancing the `mark_written` + `mark_synced` watermarks so any waiters that depend on the durable LSN unblock. Net memory mode behavior: events flow through the apply path → ring → free pool with zero file I/O and zero apply-thread stalls.
