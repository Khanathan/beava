# Phase 13.4 Plan 02: GET response → flat-dict per Phase 13.0-15 wire-spec — Summary

**Dropped the `{"result": ...}` envelope from `OP_GET_MULTI` / `OP_MGET` / HTTP `POST /get` so both transports emit the flat per-entity dict `{entity_id: {feature: value}}` directly per Phase 13.0-15 wire-spec; single-feature `GET /get/:feature/:key` and `OP_GET` paths unchanged at `{"value": <val>}`.**

- Single-line behavior change in `runtime_core_glue::dispatch_get_batch` swaps `serde_json::json!({"result": result})` for `serde_json::Value::Object(<result re-collected>)`. Both HTTP and TCP transports get the new shape automatically — same encoder backs both.

## Files Created/Modified

See frontmatter `key-files`. The behavior-bearing edit is a 14-line diff in `runtime_core_glue.rs`:

```rust
//   BEFORE
let body_json = serde_json::json!({"result": result});

//   AFTER (Phase 13.0-15)
let body_json = serde_json::Value::Object(
    result
        .into_iter()
        .map(|(k, v)| (k, serde_json::Value::Object(v.into_iter().collect())))
        .collect(),
);
```

The `result` value's type is `BTreeMap<String, BTreeMap<String, serde_json::Value>>`; the
re-collect pattern is required to flatten the inner `BTreeMap` into a `serde_json::Map`
without going through the `json!` macro (which would re-introduce the wrapper).

## Decisions Made

- **Envelope construction site is `runtime_core_glue::dispatch_get_batch`, not `feature_query.rs`** — per Plan 12.6-07's cleanup, the legacy axum router was deleted and `feature_query.rs` became a helpers-only module. The plan text named the historic location; the active code is in `runtime_core_glue.rs`.
- **Cold-start returns flat `{}`, not a feature-default row** — wire-spec at `docs/wire-spec.md:307` is explicit ("Cold-start returns `{}`"). Synthesizing default values for absent entities would be a behavior change unrelated to the envelope drop and outside Plan 02's scope.
- **Unknown-feature test accepts the existing `internal_error/feature_not_found` semantic** — the new `unknown_table` 404 belongs to Plan 13.4-04 (verb-style routes change the request shape from `{keys, features}` to `{table, key, features}`). Tightening the contract here would be a scope leak.

See `EXECUTOR-DEVIATION-02.md` for the long-form rationale on each.
