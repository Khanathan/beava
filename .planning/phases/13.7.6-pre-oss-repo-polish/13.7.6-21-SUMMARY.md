# Phase 13.7.6 Plan 21: GitHub issue + PR templates Summary

Refreshed all four `.github/` templates for OSS launch under `beava-dev/beava`:
stripped stale `petrpan26/` URLs, added Beava-specific fields (version + transport
+ minimal reproducer for bugs; v0 op-catalog link + deferred-surface framing for
feature requests), routed security disclosures to private GitHub Security
Advisories, and aligned the PR test-plan checklist verbatim with CONTRIBUTING.md.

## What changed

### `.github/ISSUE_TEMPLATE/bug_report.md` (rewrite)

Before: 38 LOC generic bug-report template; "Description / Steps to reproduce /
Expected / Actual / Environment / Logs" with ad-hoc bullets for Beava version, OS,
Python version, install method.

After: full standard frontmatter (`name`, `about`, `title: "[bug] "`,
`labels: ["bug"]`); structured sections in this order — Summary, Steps to
reproduce, Expected behavior, Actual behavior, Environment, Minimal reproducer,
Logs, Additional context. Environment block now asks for:

- Beava version (from `pip show beava`, `beava --version`, or commit SHA)
- Transport (`http` | `tcp` | `embed`) — new field; addresses must-have
- OS + arch (`Linux x86_64`, `Linux ARM64`, `macOS ARM64`)
- Python version (if SDK)
- How installed (`cargo install` | release binary | source | `pip install beava`)

The Steps to reproduce section explicitly asks for the pipeline definition and
the push/get sequence and mentions PII redaction.

### `.github/ISSUE_TEMPLATE/feature_request.md` (rewrite)

Before: 23 LOC; Use case / Proposed solution / Alternatives / Additional context.

After: full standard frontmatter (`name`, `about`, `title: "[feature] "`,
`labels: ["enhancement"]`); sections — Summary, Use case, Proposed API (with a
fenced Python example slot), Alternatives considered, **Already in the catalog or
deferred?**, Additional context.

The "Already in the catalog or deferred?" section links to the v0 op catalog
(`https://github.com/beava-dev/beava/tree/main/docs/operators`) and lists the
deferred surfaces (tables-with-mutation, joins, session windows, event-time /
watermarks, `bv.fork`) per `project_v0_events_only_scope`. A three-way checkbox
asks whether the feature is in v0 territory, v0.1+ territory, or unclear.

### `.github/ISSUE_TEMPLATE/config.yml` (rewrite)

Before: 5 LOC, single contact link to `https://github.com/petrpan26/beava/discussions`.

After: `blank_issues_enabled: false` plus two contact_links —

1. **GitHub Discussions** → `https://github.com/beava-dev/beava/discussions`
2. **Security disclosure** → `https://github.com/beava-dev/beava/security/advisories/new`,
   with explicit "Do NOT open a public issue for security concerns" framing
   (mitigates T-13.7.6-21-01).

Discord/Slack links intentionally omitted — see Decisions above.

### `.github/PULL_REQUEST_TEMPLATE.md` (rewrite)

Before: 28 LOC; Summary / Type of change / Testing / Breaking changes / Related
issues. The Testing checklist used `cargo test -- --test-threads=1`,
`cargo clippy --all-targets -- -D warnings`, `cargo fmt --check`,
`pytest tests/ -q`.

After: Summary, Linked issues, **Test plan** (verbatim CONTRIBUTING.md command
parity preserved), **Author checklist**, Breaking changes, Notes for reviewer.

Test plan checklist:

- `cargo test -- --test-threads=1`
- `cargo clippy --all-targets -- -D warnings`
- `cargo fmt --check`
- `cd python && python -m pytest tests/ -q` (if Python touched)
- Curl session / screenshot / log if user-visible behavior changed

Author checklist:

- Tests added or updated
- No stale repository URLs (canonical: `beava-dev/beava`)
- CHANGELOG.md updated under `[Unreleased]` (conditional — file may not exist yet)
- Public API / wire surface documented

No internal references (`Phase NN`, `.planning/`, `CLAUDE.md`, AI tooling).

## Deviations from Plan

None of the deviation rules (Rule 1–4) triggered. Three planning-level call-outs
worth recording:

1. **CONTRIBUTING.md command parity preserved verbatim.** The plan's example
   PR-template snippet showed `cargo test --workspace --features testing`, but
   the plan's must-have truth is "checklist parity with CONTRIBUTING.md". The
   canonical CONTRIBUTING.md uses `cargo test -- --test-threads=1`. Followed
   CONTRIBUTING.md (the canonical source) over the example block to avoid
   command divergence between the two contributor-facing files.
2. **CHANGELOG.md item kept conditional.** CHANGELOG.md does not yet exist in
   the repo. The PR template still asks contributors to update it "if
   user-visible change" — honest now, and survives the file landing later
   without further edits.
3. **Author-checklist URL-hygiene line softened.** Initial draft contained the
   literal string `petrpan26/` as anti-pattern guidance, which would surface
   in any future stale-URL grep audit. Replaced with positive guidance
   ("canonical: `beava-dev/beava`") so the file is clean under
   `grep -r 'petrpan26' .github/`.
