### Why zero HIGH findings

The four CONTEXT-flagged surfaces all have structural defenses:

1. **Predicate-string injection** — predicates parse to a typed AST in
   `crates/beava-core/src/expr.rs`, never `eval()`'d as code. Only three builtins
   (`cast`, `isnull`, `quadkey`) are reachable, none with IO/network/exec. Schema validation at
   `agg_compile.rs:834-844` rejects unknown field references at register time.
   `MAX_EVAL_DEPTH=512` guards stack at `eval.rs:47`.
2. **`force=true` destructive register** — defaults to `false`; destructive diff without force
   returns 409 + `force_required` (`apply_shard.rs:247-259`); `dry_run=true` wins over
   `force=true` so force can't escalate a what-if call to a real mutation
   (`apply_shard.rs:232-245`).
3. **`OP_RESET` test_mode gate** — boot-time-stable `effective_test_mode` flag stamped on
   `AppState` (`server.rs:411`), checked unconditionally at `apply_shard.rs:1408`. Returns
   HTTP 403 + `reset_disabled_in_production` when false. Runtime escalation impossible.
4. **Listener input handling** — TCP `max_frame_bytes` defaults to 4 MiB and is enforced by
   `decode_frame` (`wire.rs:233-269`); HTTP enforces `MAX_HEADER_BYTES=8 KiB` and
   `MAX_BODY_BYTES=4 MiB` for Content-Length bodies; HTTP `/register` strictly validates
   `application/json`; TCP `OP_REGISTER` strictly validates `CT_JSON`; structured 415 for
   wrong media-type.

## MEDIUM findings deferred to v0.0.x

1. **A09 — log-line interpolation of user-supplied identifiers.** `tracing::warn!` at
   `apply_shard.rs:1234` interpolates `entry.table` (user-supplied). Recovery WARN/ERROR lines
   surface descriptor names. Operators owning their network can decide log-shipping policy;
   v0 has no audit-log requirement (T-13.7.6-06-05 accepted in plan threat register).
2. **LLM04 / Deep-dive #4 — chunked-body bypasses MAX_BODY_BYTES.** The chunked-body parser
   at `http_listener.rs:330-377` accumulates `Vec<u8>` without enforcing `MAX_BODY_BYTES`;
   `client.read_buf` itself is uncapped during accumulation. Memory-bomb DoS for clients with
   open connections. Mitigated by SECURITY.md "deploy-behind-trust-boundary" scope. Tracked
   for v0.0.1 fix: cap chunked-body accumulation + cap `client.read_buf` capacity.
3. **Deep-dive #4 — expression-parser stack depth.** Recursive-descent parser at
   `crates/beava-core/src/expr.rs` has no explicit depth cap; pathological nested-paren input
   could stack-overflow the apply thread. Mitigated by 4 MiB body cap + cold-path reach.
   Tracked for v0.0.1 fix: add `MAX_PARSE_DEPTH=256` symmetric to `MAX_EVAL_DEPTH=512`.

All three are pure DoS, no RCE; all three are mitigated by the documented deployment scope
(private network / single-tenant / reverse-proxy TLS). None block OSS launch.
