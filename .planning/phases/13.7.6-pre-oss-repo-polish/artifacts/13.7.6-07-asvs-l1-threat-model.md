# Beava v0 — ASVS Level-1 Threat Model

**Status:** v0 launch baseline. Internal artifact (per Phase 13.7.6 D-06: public-facing
architectural-invariant docs are skipped; SECURITY.md cross-references this file
without duplicating content).
**ASVS version reference:** OWASP ASVS 4.0 / 5.0 chapter list (V1–V14).
**ASVS Level:** L1 (opportunistic attacker; single-tenant; low-risk; suitable for
applications without significant assurance requirements; verifiable without source-code
access).
**Last reviewed:** 2026-05-05.

## Trust boundary

- **Single tenant per process.** One Beava server instance serves one trust domain.
  Multi-tenant isolation is the operator's responsibility (run separate processes /
  separate VPCs / separate accounts).
- **Operator owns network isolation.** v0 ships **no authentication** on the data
  plane (HTTP `:8080` and TCP `:8081`) and no authorization. Deployments must bind to
  loopback or to a private network the operator controls. See `SECURITY.md` §Scope:
  *"Beava is designed to run behind a trusted boundary."*
- **Admin sidecar is a separate port.** The HTTP admin plane (`/health`, `/ready`,
  `/metrics`, `/registry`) binds on `cfg.admin_addr` (`crates/beava-server/src/main.rs:48-51`,
  default `127.0.0.1:8090`) — operators can firewall it independently from the data
  plane. The admin plane is read-only (`crates/beava-server/src/http_admin.rs:55-64`).
- **No encryption in transit or at rest in v0.** TLS is terminated by an upstream
  reverse proxy (nginx, Caddy, ALB, etc.). Disk artifacts (WAL + snapshots) live under
  `.beava/wal/` and `.beava/snapshots/` with default filesystem permissions.
- **Hot path is single-runtime.** The mio-only invariant
  (`crates/beava-server/tests/phase12_6_mio_only_dataplane.rs`) prevents accidental
  introduction of a second data-plane runtime that would broaden the attack surface.

## ASVS L1 controls coverage (V1 – V14)

The sections below walk the ASVS L1 chapter list one at a time, stating v0's posture
and citing concrete code or architectural-test evidence.

## V1 Architecture, design, threat modeling

This document is the v0 threat model. The trust boundary is single-tenant, single-process,
operator-network-owned; documented in `SECURITY.md` §Scope and reproduced above. Three
locked architectural invariants narrow the data-plane attack surface and are CI-enforced:
mio-only data plane (Phase 12.6 — `phase12_6_mio_only_dataplane.rs`), events-only surface
(Phase 12.7 — `phase12_7_no_table_surface.rs`), and memory governance (Phase 12.8 —
`phase12_8_lifetime_ops_have_bounds.rs` + `phase12_8_cold_entity_eviction.rs`). Reviving
any of them requires an explicit ADR overturn (per `CLAUDE.md` §Architecture).

## V2 Authentication

**DEFERRED — v0 ships no authentication on the data plane.** Both transports (HTTP
data plane on `cfg.listen_addr`; framed TCP on `cfg.tcp.host:port`) accept any
connecting client. This is documented in `SECURITY.md` lines 31–39 and is a deliberate
v0 scope decision (single-tenant, operator-owned network). v0.1+ will introduce
authentication if/when demand justifies it. **Mitigation today:** bind to localhost
or a private VPC; require clients to traverse an authenticated upstream proxy. The
admin sidecar inherits the same posture and is similarly firewall-gated.

## V3 Session management

**Not applicable.** Beava's wire protocol is connection-stateless aside from
Redis-style strict-FIFO request ordering on a single TCP connection
(`docs/wire-spec.md` §Connection model). There is no session token, no cookie, no
server-side session table. Each connection is an independent stream; closing it
clears all per-connection state.

## V4 Access control

**Accept — single-tenant model.** Every connecting client has identical privileges:
register / push / get / mget on the registry shared by the process. There is no
per-resource ACL. Destructive operations are gated by content (`force=True`) or
boot-time flag (`effective_test_mode` for `OP_RESET`), not by identity. Multi-tenant
isolation is the operator's responsibility (run separate processes / separate VPCs).

## V5 Validation, sanitization, encoding

JSON payloads are validated against `docs/wire-spec.md` JSON Schema 2020-12 contracts
at the prelude shim layer (`crates/beava-runtime-core/src/wire_request.rs:728-790`).
Register payloads with unknown `kind` values are rejected with structured error code
`unsupported_node_kind` (Phase 12.7 events-only enforcement,
`crates/beava-server/tests/phase12_7_unsupported_node_kind.rs`). TCP frames exceeding
`tcp_max_frame_bytes` (default 4 MiB; `crates/beava-server/src/server.rs:72`) are
rejected with `frame_too_large` and the connection is closed
(`crates/beava-runtime-core/src/tcp_listener.rs:728-735`). Predicate-string injection
on `/register` is the primary tampering vector and is covered by the diff /
force-required gate plus the cross-referenced OWASP review (artifact 13.7.6-06).

## V6 Cryptography

**Not applicable in v0.** Beava ships no built-in TLS, no encryption at rest, no
secrets in source. TLS termination is delegated to an upstream reverse proxy as
documented in `SECURITY.md` line 35. WAL and snapshot files on disk are protected
only by filesystem permissions; operators handling regulated data should encrypt the
underlying volume (LUKS, eBS encryption, etc.). v0.1+ may revisit on-the-wire TLS if
direct internet exposure becomes a supported deployment mode.

## V7 Error handling, logging

Error responses use structured error codes (`force_required`, `frame_too_large`,
`unsupported_node_kind`, `reset_disabled_in_production`) carried in the HTTP body or
TCP `OP_ERROR_RESPONSE` frame. Internal logs use `tracing` with named fields rather
than format strings, reducing log-injection risk
(`crates/beava-server/src/apply_shard.rs:1232,1436`). Phase 13.7.6-06 Workstream-C
performs a PII grep sweep against `tracing::warn!` / `tracing::error!` sites to
confirm user-supplied IDs are not interpolated into unstructured log lines. No stack
traces are returned to clients; on internal panic the process exits and the supervisor
restarts.

## V8 Data protection

In-memory entity state and on-disk WAL/snapshot artifacts are tenant-scoped and
co-located with the process. The operator is the data controller; Beava does not
classify, redact, or expire data automatically (except via opt-in cold-entity-TTL,
V0-MEM-GOV-01). Cross-process access requires filesystem access to `.beava/wal/` and
`.beava/snapshots/`. There is no encryption at rest in v0; operators handling
regulated data should encrypt the underlying volume.

## V9 Communication

HTTP/1.1 + JSON on `cfg.listen_addr`; framed TCP `[u32 length][u16 op][u8 content_type][payload]`
on `cfg.tcp.host:port`. **No TLS in v0** — terminate at an upstream reverse proxy
(`SECURITY.md` line 35). The admin sidecar is HTTP/1.1 only; same termination
guidance. Loopback binds (default `127.0.0.1`) provide local-machine isolation; for
multi-host deployments operators must run TLS at the proxy AND ACL the data-plane
ports to the proxy's egress IP.

## V10 Malicious code

Dependency hygiene is enforced by `cargo audit` and `cargo deny` (Workstream-C-2,
artifact `13.7.6-02-cargo-audit.txt`; configuration in `deny.toml`). License
allow-list, advisory ignores, banned-crate sources, and source-registry restrictions
are all PASS at v0 launch. Two unmaintained advisories (RUSTSEC-2025-0141 bincode 1.x;
RUSTSEC-2025-0057 fxhash) are accepted with documented v0.0.x migration plans. Python
SDK lints via `ruff check` + `mypy --strict` (Workstream-C-3); TS via `tsc --noEmit`
(C-4); Go via `go vet ./...` (C-5).

## V11 Business logic

Two destructive operations are gated:
- **`force=True` register** — required to apply any destructive registry diff
  (descriptor removal, schema-shape change). Default behavior is 409 +
  `force_required`. Implementation: `crates/beava-core/src/register_validate.rs:1155-1170`
  + `crates/beava-server/src/apply_shard.rs:212-260`.
- **`OP_RESET` / `POST /reset`** — wipes all per-entity state and registered
  descriptors. Gated by boot-time `effective_test_mode` flag (set via
  `BEAVA_TEST_MODE=1` or `Config { test_mode: true }`); production boots return
  `ResetForbidden` (HTTP 403 / TCP `OP_ERROR_RESPONSE`). Implementation:
  `crates/beava-server/src/apply_shard.rs:1407-1443`. Tripwire: dispatch checks the
  flag stamped at `AppState` build time and never re-reads `BEAVA_TEST_MODE`,
  preventing runtime escalation (`apply_shard.rs:1404-1406` rationale comment).

## V12 Files & resources

WAL and snapshot paths are configured via `cfg.durability.wal_dir` /
`cfg.durability.snapshot_dir` and default to `.beava/wal/` / `.beava/snapshots/`
(`crates/beava-server/src/main.rs:64-69`; testing helpers
`crates/beava-server/src/testing.rs:208-219`). User input never reaches a filesystem
path; there is no file upload or path-derived endpoint. Disk usage is bounded by WAL
rollover policy (Phase 19 baseline: 4 × 32 MiB segments by default) plus snapshot
retention.

## V13 API

The wire surface is normatively specified by `docs/wire-spec.md` (JSON Schema 2020-12).
HTTP routes: `POST /register`, `POST /push/:event`, `GET /get/:feature/:entity`,
`POST /mget`, `POST /reset`. TCP opcodes: `OP_PING`, `OP_REGISTER`, `OP_PUSH`,
`OP_GET`, `OP_MGET`, `OP_RESET`. Both transports share JSON body shapes. Unknown HTTP
routes return 404; unknown TCP opcodes return `OP_ERROR_RESPONSE` and the connection
is closed. Admin routes (`/health`, `/ready`, `/metrics`, `/registry`) are read-only
and bound on a separate port (`http_admin.rs:55-64`).

## V14 Configuration

Configuration is loaded from a TOML file (`cli.config`) plus six `BEAVA_*` env vars
read once at boot (`crates/beava-server/src/server.rs:92-144`):
`BEAVA_WAL_BUFFERS`, `BEAVA_WAL_BUFFER_SIZE_MB`, `BEAVA_WAL_TICK_MS`,
`BEAVA_IO_THREADS`, `BEAVA_TEST_MODE`, `BEAVA_MEMORY_GOV_ENFORCE`. The hot path
never re-reads env, preventing runtime escalation of e.g. `BEAVA_TEST_MODE`. Defaults
are safe: production boot has `effective_test_mode = false`, memory governance
enforced, 4 MiB max frame, mio-only data plane. Dependency policy lives in
`deny.toml` (Workstream-C-2). CI workflow audit is captured in artifact
`13.7.6-22-workflows-audit.md`.

## Posture statement for OSS users

**Beava v0 is a single-tenant in-memory feature server. It assumes you deploy it
behind a network boundary you control.**

What v0 protects against:
- Oversized TCP frames (4 MiB default cap; reject + close connection).
- Malformed register payloads (JSON Schema 2020-12 validation; structured error codes).
- Accidental destructive operations (`force=True` gate on register diffs;
  `effective_test_mode` gate on `OP_RESET`).
- Unbounded memory growth from stateful operators (memory-governance invariants;
  CI-enforced tripwires).
- Known-CVE dependencies (`cargo audit` + `cargo deny` in CI).

What v0 explicitly does NOT do:
- **No authentication.** Both HTTP and TCP data planes accept any connecting client.
  Deploy on loopback or a private VPC; put an authenticated reverse proxy in front
  if you need it.
- **No authorization.** Every client gets full register / push / get privileges on
  the same registry. For multi-tenant isolation, run separate processes.
- **No encryption.** TLS is the upstream proxy's job; WAL and snapshot files are
  filesystem-permission-protected only — encrypt the volume yourself if regulated
  data is involved.
- **No audit log of individual pushes** (only register / reset events are logged).
- **No rate limiting.** Use your upstream proxy.

If your deployment model is "Beava bound to localhost behind an authenticated
reverse proxy with TLS termination, fronting trusted internal services," v0 is in
scope for security reports. If you are exposing Beava directly to the internet,
that is out of scope (per `SECURITY.md` §Scope) and you should reconsider your
architecture.
