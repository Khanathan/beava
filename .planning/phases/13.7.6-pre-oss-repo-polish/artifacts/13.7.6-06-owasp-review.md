# OWASP Top-10 (2021) + LLM Top-10 review — Beava v0 attack surface

## Executive summary

Reviewed Beava v0's full attack surface against OWASP Top-10 (2021) and OWASP LLM Top-10 (2025).
**HIGH-severity findings: 0.** Three MEDIUM findings deferred-with-reason to v0.0.x post-launch
hardening (chunked-body unbounded growth, expression-parser stack depth, log-line interpolation
of user-supplied identifiers). All four CONTEXT-flagged surfaces (predicate-string injection,
`force=True` destructive register path, `OP_RESET` test_mode gate, listener input handling) are
in-scope and verified mitigated. Consistent with the explicit deployment scope declared in
`SECURITY.md` (private network / reverse-proxy TLS termination / single-tenant / no auth in v0),
v0 ships with a posture acceptable for OSS launch behind a trusted boundary.

## Scope

- v0 is events-only (locked Phase 12.7); `@bv.table` / upsert / delete / retract are removed
  surface; the events-only invariant gate runs at register time
  (`crates/beava-core/src/register_validate.rs::pre_check_unsupported_node_kind`).
- Data plane: hand-rolled mio event loop in `crates/beava-runtime-core/`; the only legitimate
  caller of `apply_event_to_aggregations` is `apply_shard.rs::dispatch_*_sync`. axum is
  restricted to the admin sidecar (`http_admin.rs::BoundAdminServer`).
- Public surface: HTTP/1.1 + JSON (curl/LB/WAF reach) and custom-framed TCP
  `[u32 length][u16 op][u8 content_type][payload]` for low-latency.
- No auth, no TLS in v0 — operators are expected to terminate TLS at a reverse proxy and run on
  a private network. Documented in `SECURITY.md §Scope`.

## Per-category notes

### A01 Broken Access Control
NOT-APPLICABLE-BY-DESIGN. v0 ships single-tenant with no per-request authz. `SECURITY.md §Scope`
documents the deploy-behind-trust-boundary contract. `/reset` is gated separately by
`effective_test_mode` (Deep-dive #3); `force=true` register requires explicit kwarg
(Deep-dive #2). No multi-tenant auth surface exists to break.

### A02 Cryptographic Failures
NOT-APPLICABLE-BY-DESIGN. v0 has no in-flight TLS and no at-rest encryption. `SECURITY.md §Scope`
requires reverse-proxy TLS termination. WAL + snapshot bytes are local files; protection is
via filesystem permissions owned by the operator.

### A03 Injection
APPLICABLE-MITIGATED. Predicates parse to a typed AST (`crates/beava-core/src/expr.rs`) — never
`eval()`'d. Only three builtins (`cast`, `isnull`, `quadkey`) are reachable, none with
IO/network/exec. Schema propagation rejects unknown field references at register time. JSON
register payloads run through the JSON-prelude shim chain before strict serde so
removed/legacy/unsupported constructs surface as typed 400s. See Deep-dive #1.

### A04 Insecure Design
APPLICABLE-MITIGATED. The two architectural invariants — mio-only data plane (locked Phase 12.6)
and events-only surface (locked Phase 12.7) — are enforced by CI tripwire tests
(`phase12_6_mio_only_dataplane.rs`, `phase12_7_no_table_surface.rs`) that walk the workspace at
test runtime and fail if a forbidden symbol resurfaces. Both run on every CI PR via
`cargo test --workspace`.

### A05 Security Misconfiguration
APPLICABLE-MITIGATED. The three risky operations are all gated:
- `force=true` defaults to `false`; destructive register without force returns HTTP 409 +
  `force_required` (Deep-dive #2).
- `OP_RESET` returns HTTP 403 + `reset_disabled_in_production` unless boot-time
  `effective_test_mode=true` (Deep-dive #3).
- Memory governance defaults ON; the explicit-off escape hatch is `BEAVA_MEMORY_GOV_ENFORCE=0`
  resolved at boot.

### A06 Vulnerable & Outdated Components
APPLICABLE-MITIGATED. Plan 13.7.6-02 cargo-audit shows 0 vulnerabilities; two unmaintained-only
warnings (`bincode 1.3.3`, `fxhash 0.2.1`) tracked separately. Cross-ref
`artifacts/13.7.6-02-cargo-audit.txt`. cargo-deny gates landed in Plan 13.7.6-02.
Python/TS/Go SDK lints landed in Plans 13.7.6-03/04/05.

### A07 Identification & Authentication Failures
NOT-APPLICABLE-BY-DESIGN. v0 has no auth surface; `SECURITY.md §Scope` declares auth
out-of-scope. Not a regression because there was never an auth contract.

### A08 Software & Data Integrity Failures
APPLICABLE-MITIGATED. Wire format pinned at v1 (`FORMAT_VERSION = 1`,
`SNAPSHOT_FORMAT_VERSION = 1`, `SNAPSHOT_BODY_FORMAT_VERSION = 1`); pre-12.7 dev WALs/snapshots
fail the version-byte check on boot. Events-only invariant tripwire
(`phase12_7_no_table_surface.rs`) rejects re-introduction of forbidden symbols.

### A09 Security Logging & Monitoring Failures
APPLICABLE-OPEN (informational). Log lines interpolate user-supplied `table` field at
`apply_shard.rs:1234` and descriptor names in recovery WARN/ERROR lines. v0 has no audit-log
requirement (T-13.7.6-06-05 accepted in plan threat register). Operators owning their network
decide log-shipping policy. **MEDIUM finding deferred to v0.0.x** — see carry-forward list.

### A10 SSRF
NOT-APPLICABLE. Beava is a server, not a client. The runtime makes no outbound HTTP.
The expression evaluator has no network primitives (verified via grep over `eval.rs`/`expr.rs`).

## CONTEXT-flagged surface deep-dives

### Deep-dive #1 — Predicate-string injection at register time (T-03-02-01)

**Surface:** register payload predicates feeding `crates/beava-core/src/expr.rs::parse` and
`crates/beava-core/src/agg_where.rs::evaluate_where_predicate`.

**Code path:** `WireRequest::Register { payload }` → `apply_shard.rs:86` →
JSON-prelude shim chain (lines 93–185) → strict serde at line 189 → diff classification at
line 227 → `register_check_force_required` at line 247 → `execute_register_with_wal` at
line 313 → schema propagation calls `expr::parse(where_str)` at
`crates/beava-core/src/agg_compile.rs:816`.

**Findings:**

1. **Predicates are parsed to a typed AST, never `eval()`'d as code.** The grammar in
   `expr.rs` (locked Phase 3 D-08; Phase 4 D-01) is purely arithmetic / boolean / comparison /
   field-ref / literal / call. `Call` arguments resolve only against `BUILTINS` table at
   `expr_builtins.rs:54-69` which has exactly three entries: `cast`, `isnull`, `quadkey`. None
   reach IO, network, filesystem, env, or process spawn (verified via grep
   `Command|Process|exec|shell|system|fs::|File::|TcpStream|UdpSocket|net::|env::|reqwest`
   over `eval.rs` and `expr.rs` — zero matches).
2. **Field references are validated against upstream schema** at `agg_compile.rs:834-844`;
   unknown fields surface as `AggregationInvalidWhere` with code; the request is rejected at
   register time. No way to leak unrelated field names by referencing them in a predicate.
3. **Evaluator has stack-depth guard.** `MAX_EVAL_DEPTH=512` at `eval.rs:47`; expressions
   exceeding the limit return `Value::Null` (`eval.rs:61`) instead of stack-overflowing.
4. **JSON-prelude shim chain** runs BEFORE strict serde and emits structured error codes for
   removed / legacy / unsupported / unbounded constructs (apply_shard.rs:93-184). Joins,
   `event_time_field`, `kind: table`, and `Unbounded` lifetime ops are all rejected with
   typed errors before the predicate parser ever runs.

**Risk:** LOW. The "injection" surface here is functionally equivalent to a typed
expression-language interpreter (think: a sandboxed `eval` whose ops are enumerated at
compile-time). The only known stress vector is parser stack depth on extremely-nested
parens — see Deep-dive #4 / chunked-body finding for the related body-size discussion.

**Same-phase fix:** none required.

### Deep-dive #2 — `force=true` destructive register path

**Surface:** `WireRequest::Register { payload }` carrying `{"force": true, ...}`.

**Code path:** `apply_shard.rs:219-225` parses `force` and `dry_run` from the payload;
`apply_shard.rs:247-259` calls `register_check_force_required(&diff, force)` — when
`diff.destructive` is non-empty AND `force=false`, returns HTTP 409 + structured
`force_required` error. When `force=true` AND `diff.destructive` non-empty, `apply_shard.rs:266-302`
collects the to-remove descriptors and calls `force_remove_descriptors`. `dry_run=true` wins
over `force=true` (apply_shard.rs:232-245) — `force` cannot escalate a "what would this do?"
call to a real mutation.

**Findings:**

1. **Default-deny:** `force` defaults to `false` if absent (`apply_shard.rs:221`). The
   destructive path is only reached when the client explicitly opts in.
2. **Idempotent shape:** `force=true` on a non-destructive payload is a no-op for the
   destructive branch (`apply_shard.rs:266` `if force && !diff.destructive.is_empty()`); only
   actual destructive diffs trigger removal.
3. **Boolean-coerce safety:** `serde_json::Value::as_bool().unwrap_or(false)` — non-boolean
   `force` values default to false, not true. No string-truthy weirdness.
4. **Audit trail:** the `force=true` removal goes through `Registry::force_remove_descriptors`
   followed by `execute_register_with_wal` which emits a `RegistryBump` to the WAL.

**Risk:** LOW. Destructive behavior is opt-in, default-deny, audit-trailed.

**Same-phase fix:** none required.

### Deep-dive #3 — `OP_RESET` opcode test_mode gate

**Surface:** `WireRequest::TcpReset { body, body_format }` (TCP) and `WireRequest::HttpReset { body }`
(HTTP `POST /reset`).

**Code path:** both transports route to
`apply_shard.rs::dispatch_reset_sync` (line 1407). The first statement is the gate:

```rust
pub fn dispatch_reset_sync(state: &std::sync::Arc<crate::AppState>) -> GlueResponse {
    if !state.effective_test_mode {
        return GlueResponse::ResetForbidden;
    }
    // ...
}
```

`GlueResponse::ResetForbidden` maps to HTTP 403 + `reset_disabled_in_production` body / TCP
`OP_ERROR_RESPONSE`.

**Findings:**

1. **Boot-time-only flag.** `effective_test_mode` is stamped on `AppState` at
   `server.rs:411` (`let effective_test_mode = cfg.test_mode;`). It is never re-read at
   runtime — runtime escalation is impossible (cf. T-13.4-08-03 in plan threat register).
2. **`cfg.test_mode` resolution.** `Config::from_env()` at `server.rs:124` resolves
   `BEAVA_TEST_MODE=1` → `test_mode=true`; default is `false` (`server.rs:71`). The
   `TestServer` builder's `.test_mode(true)` sets `cfg.test_mode = true` at
   `testing.rs:346` for in-process tests without env var coupling.
3. **State-clear scope.** When the gate passes, `dispatch_reset_sync` clears
   per-entity state, registry descriptors, and cold-start counters. Per-event WAL ring
   buffers are NOT touched (see comment at `apply_shard.rs:1396-1400`); this is intentional
   so a reset+re-register starts from a clean slate but the durability log keeps its prior
   bytes for forensic recovery.
4. **Tracing.** Successful resets emit a `tracing::info!` at `apply_shard.rs:1436` with
   `kind = "server.reset_completed"` for monitoring.

**Risk:** LOW. The defense is the early-return; it's unconditional and the flag is
boot-time-stable.

**Same-phase fix:** none required.

### Deep-dive #4 — HTTP + TCP listener input handling

**Surfaces:**
- `crates/beava-runtime-core/src/tcp_listener.rs::parse_wire_request` (TCP frame decoder)
- `crates/beava-runtime-core/src/http_listener.rs::parse_http_request` (HTTP/1.1 request parser)
- `crates/beava-core/src/wire.rs::decode_frame` (TCP frame codec)

**Frame size limits:**

1. **TCP** — `decode_frame(buf, max_frame_bytes)` at `wire.rs:233` returns
   `FrameError::TooLarge` when declared length > `max_frame_bytes + 3` (HEADER_AFTER_LEN_BYTES).
   Caller (`server.rs:2112`) passes `4 * 1024 * 1024` (4 MiB) sourced from
   `Config::tcp_max_frame_bytes` (default 4 MiB at `server.rs:139`). Oversized frames close the
   connection (`ParseErrorKind::TooLarge` → `RingItem::ParseError` at `server.rs:2129`).
2. **HTTP** — three caps:
   - `MAX_HEADER_BYTES = 8 * 1024` (`http_listener.rs:46`); rejected with `ParseError::TooLarge`
     when more bytes accumulate without finding `\r\n\r\n` (`http_listener.rs:77-79`).
   - `MAX_HEADERS = 64` slots passed to `httparse` (`http_listener.rs:48`).
   - `MAX_BODY_BYTES = 4 * 1024 * 1024` (`http_listener.rs:50`); enforced for
     Content-Length bodies at `http_listener.rs:157-159`.

**Content-Type validation:**

1. **HTTP /register POST** — strict `application/json` validation (`http_listener.rs:184-200`);
   rejected with `WireRequest::HttpUnsupportedMediaType` → 415 `unsupported_media_type`.
2. **HTTP non-/register POST** — does NOT validate Content-Type (`http_listener.rs:181-184`);
   downstream JSON parse failure becomes a 400. Documented in the comment.
3. **TCP** — content-type byte (`u8`) on each frame; `OP_REGISTER` requires `CT_JSON` (0x01)
   and rejects others with structured `unsupported_content_type` ParseError
   (`tcp_listener.rs:742-748`). `OP_PUSH`, `OP_GET`, etc. accept either `CT_JSON` (0x01) or
   `CT_MSGPACK` (0x02); other byte values surface as `unsupported_content_type` (e.g.
   `tcp_listener.rs:797-799`).

**JSON parser hardening:**

1. **Strict serde** — register payloads deserialize with `serde_json::from_slice::<RegisterPayload>`
   (`apply_shard.rs:189`). Unknown fields / wrong types surface as `invalid_registration` 400 with
   path + reason from `register::format_serde_error_public`.
2. **JSON-prelude shim chain** — `pre_check_removed_ops` / `pre_check_legacy_event_time_keys` /
   `pre_check_unsupported_node_kind` / `pre_check_unbounded_op_in_lifetime_mode` run BEFORE
   strict deserialize so structured "feature removed" errors land instead of opaque serde
   "unknown variant" errors.
3. **Recursion.** `serde_json` (and the high-perf `sonic_rs` parser used on the hot path
   for push bodies at `server.rs:2103`) has a default recursion limit of 128 — adequate against
   pathological-nesting stack-overflow.

**Findings:**

1. **MEDIUM — Chunked transfer-encoding bypasses MAX_BODY_BYTES.** The chunked-body parser
   at `http_listener.rs:330-377` accumulates all chunk data into `body: Vec<u8>` without
   enforcing `MAX_BODY_BYTES`. Concretely, `body.extend_from_slice(&data[pos..data_end])` at
   line 374 is unbounded; an attacker sending many small chunks summing to >>4 MiB will
   accumulate all of them before the `0\r\n\r\n` terminator triggers
   `ChunkedResult::Complete`. Combined with `client.read_buf` itself growing unboundedly while
   waiting for the terminator (`server.rs:2067`'s `extend_from_slice` is uncapped), this is a
   memory-bomb vector for clients owning their own connections.
   - **Severity: MEDIUM.** Attacker needs to be able to open a connection (mitigated in
     prod by the SECURITY.md "deploy-behind-trust-boundary" scope); no remote-code-execution;
     pure DoS via memory growth.
   - **Disposition: defer-with-reason.** Cap chunked-body accumulation at `MAX_BODY_BYTES`
     (return `ParseError::TooLarge` once `body.len() + chunk_size > MAX_BODY_BYTES`); cap
     `client.read_buf` capacity in `read_and_parse_client_to_channel` so a slow-loris client
     can't grow it past `MAX_HEADER_BYTES + MAX_BODY_BYTES + slack`. Tracked for v0.0.1.

2. **MEDIUM — Expression-parser stack depth on pathological nested-paren predicates.** The
   recursive-descent parser at `crates/beava-core/src/expr.rs` has no explicit depth cap; each
   `(` opens a Rust stack frame in `parse_atom → parse_expr → parse_or → ... → parse_atom`. A
   4 MiB predicate string of `(((...))` (≈2 MiB of opens) consumed at register time would
   recurse ~2M frames and stack-overflow the apply thread. The 4 MiB body cap bounds the
   input size, and the apply thread's stack is the default Rust main-thread stack (~8 MiB),
   but very deeply-nested predicates could still reach the limit and abort the process.
   - **Severity: MEDIUM.** Register is a cold path and operators control who can register
     (per SECURITY.md scope — single-tenant + deploy-behind-trust-boundary); aborting a
     server is a DoS not RCE.
   - **Disposition: defer-with-reason.** Add `const MAX_PARSE_DEPTH: usize = 256;` to the
     `Parser` and increment/decrement around each parenthesised entry; surface
     `ParseError { col, reason: "expression too deeply nested" }` past the cap. Symmetric to
     `MAX_EVAL_DEPTH=512` already in `eval.rs`. Tracked for v0.0.1.

3. **LOW — TCP frame parser depth.** `parse_msgpack_envelope` and `skip_msgpack_value` are
   recursive over msgpack tags. The 4 MiB TCP frame cap bounds input; msgpack tag-depth would
   bottom out from frame size before stack pressure becomes lethal. Tracked as informational.
   - **Severity: LOW.** Mitigated structurally by frame cap.
   - **Disposition: defer (v0.0.x optional hardening).**

**Same-phase fix:** none required for v0 launch — the `SECURITY.md` deployment scope
(private network / single-tenant) makes both MEDIUM findings out-of-scope-for-direct-internet
exposure. Both findings are tracked for v0.0.1.

## Overall posture statement

Beava v0 is a single-tenant, no-auth, processing-time, events-only feature server designed to
run behind a trusted boundary. Within that scope, every CONTEXT-flagged surface is verified
mitigated with structural defenses (parsed-AST predicates, default-deny `force=true`,
boot-time `effective_test_mode` gate, frame/body size caps + content-type validation). Three
MEDIUM findings (chunked-body unbounded growth, expression-parser stack depth, log-line
identifier interpolation) are deferred-with-reason to v0.0.x hardening — none reach
remote-code-execution or unauthenticated state corruption; all are pure DoS or operational
hygiene gaps mitigated by the documented deployment scope. Cargo-audit shows zero CVEs (only
two unmaintained crate warnings tracked separately). The mio-only and events-only invariants
are enforced by CI tripwire tests that run on every PR. **Posture: acceptable for OSS launch.**
