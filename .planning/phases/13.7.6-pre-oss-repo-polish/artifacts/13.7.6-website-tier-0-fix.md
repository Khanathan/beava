## Gap 2 ŌĆö `e.agg(...)` without `group_by(K)` in keyed `@bv.table`

**The shipped reality:** `_app.py::_to_register_json` serialises `@bv.table(key="user_id") def T(e): return e.agg(...)` as `ops:[{op:"group_by", keys:[], agg:{...}}]` with `table_primary_key:["user_id"]`. The empty `keys` collides with the declared key and the engine rejects subsequent GETs with `key_parse_failure` ("does not match table 'T' group_keys arity (0)"). Every test in `python/tests/v0/test_transport_equivalence.py` uses the explicit `txn.group_by("user_id").agg(...)` form (e.g. lines 99ŌĆō101, 121ŌĆō123, 145ŌĆō147).

**Fix direction:** When the `@bv.table` decorator carries `key=`, the body MUST chain `e.group_by("<same_key>").agg(...)`. When the decorator carries no `key=` (global table per ADR-003), bare `e.agg(...)` IS the correct shape ŌĆö it routes to a singleton aggregation with no per-entity sharding.

**Pages modified:**

| Page | Change |
| ---- | ------ |
| `/index.html` (homepage) | The 18-line `PipelineShowcase` snippet ŌĆö `Session(key="session_id")` body now chains `e.group_by("session_id").agg(...)`. The `Global(key="__global__")` form was simultaneously corrected to `@bv.table` (no `key=`) ŌĆö the canonical ADR-003 global form documented in `docs/quickstart.md` ŌĆö so its bare `e.agg(...)` is now shipped-code-correct. Both the JS template literal (clipboard payload) and the syntax-highlighted JSX (visible code block) were updated in lockstep. |
| `/guide/chapter-1/` | The two tokenized pipeline snippets (`pipelineBasic` for `UserStats(key="user_id")`, `pipelineDashboard` for `Dashboard(key="user_id")`) ŌĆö both bodies now chain `e.group_by("user_id").agg(...)`. |
| `/guide/recipes/fraud/` | Both `IpSignals(key="ip")` and `UserSignals(key="user_id")` bodies in the `code` literal AND the visible JSX `<pre>` chain `.group_by(K)` before `.agg(...)`. |
| `/docs/concepts/streams/` | Prose in "What's next" bullet: `e.agg(...)` ŌåÆ `e.group_by(...).agg(...)`. |
| `/docs/get-started/quickstart/` | Prose in "Where to go next" bullet (mirror of the above). |

**Before (homepage hero):**
```python
@bv.table(key="session_id")
def Session(e: FeedClick):
    return e.agg(
        my_feeds  = bv.count(),
        my_streak = bv.consecutive(),
    )

@bv.table(key="__global__")
def Global(e: FeedClick):
    return e.agg(...)
```

**After:**
```python
@bv.table(key="session_id")
def Session(e: FeedClick):
    return e.group_by("session_id").agg(
        my_feeds  = bv.count(),
        my_streak = bv.consecutive(),
    )

@bv.table   # no key= -> global table
def Global(e: FeedClick):
    return e.agg(...)
```

The single residual is the global-table block on the homepage, where bare `e.agg(...)` IS the shipped-code-correct form (mirrors `docs/quickstart.md`'s `TotalImpressions` example). Pattern-as-bug count = 0.

## Gap 3 ŌĆö `bv.App("0.0.0.0:6400")` bare host:port

**The shipped reality:** `python/beava/_app.py` line 470 calls `urlparse(url).scheme` and rejects with `ValueError: unsupported URL scheme: ''` if no scheme is present. `bv.App()` (no URL ŌåÆ embed), `bv.App(url="http://...")`, and `bv.App(url="tcp://...")` are the three accepted shapes. Per `beava.example.yaml` line 5, the canonical default port is `8080` (NOT 6400, NOT 7380 ŌĆö the three "default" ports in 8 different docs pages was Gap 18, of which this is the homepage subset).

**Fix direction:** `bv.App("0.0.0.0:6400")` ŌåÆ `bv.App(url="http://0.0.0.0:8080")` everywhere. (HTTP is the default-shape transport for the docs-narrative; users who want TCP fast-path can switch the scheme.)

**Pages modified:**

| Page | Change |
| ---- | ------ |
| `/index.html` (homepage) | Both the `code` literal (clipboard) and the visible `<pre>` JSX |
| `/guide/recipes/fraud/` | Both the `code` literal and the visible `<pre>` JSX |
| `/guide/chapter-1/` | Both `pipelineBasic` and `pipelineDashboard` token arrays + the listening-line in the RegisterCommand terminal mock |
| `/docs/get-started/quickstart/` | The `bv.App("127.0.0.1:8080")` example + the `beava server --host ... --port ...` CLI command (which doesn't exist; replaced with `beava -c beava.yaml` per the actual binary CLI) |

## Gap 4 ŌĆö Chained `.serve()` server-bind

**The shipped reality:** `bv.App` exposes `batch_get, close, get, ping, push, register, reset` only. There is no `serve` method anywhere in `python/beava/`. The SDK's network mode is a *client* of an already-running `beava` server binary; the binary is spawned separately (`beava -c beava.yaml`, OR via embed mode where `bv.App()` no-URL spawns it as a subprocess). There is no Python-side `bind()` operation.

**Fix direction:** Drop `.serve()` from the chain. The narrative becomes: spawn the `beava` binary in another shell, then `bv.App(url=...).register(...)` connects to it.

**Pages modified:**

| Page | Change |
| ---- | ------ |
| `/index.html` (homepage) | Removed `.serve()` from the chain; reformatted into `app = bv.App(url=...) ; app.register(...)` two-line shape with a `# Spawn the beava binary in another shell, then connect:` lead-in comment |
| `/guide/recipes/fraud/` | Same two-line shape |
| `/guide/chapter-1/` | Same two-line shape, plus updated the JSDoc above the `RegisterCommand` component (no longer claims "calls `bv.App(...).register(...).serve()`") + updated the listening-line ("listening on 0.0.0.0:6400" ŌåÆ "connected to beava server at 0.0.0.0:8080") |

**Before:**
```python
bv.App("0.0.0.0:6400").register(FeedClick, Session, Global).serve()
```

**After:**
```python
# Spawn the beava binary in another shell, then connect:
app = bv.App(url="http://0.0.0.0:8080")
app.register(FeedClick, Session, Global)
```
