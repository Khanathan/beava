# Phase 13.7.6 Plan 27: bench-v18 sustained-mode hang fix + burst-vs-sustained label discipline

Fixes two related launch-blocker bugs in `beava-bench-v18`: (1) the deadline-only sustained path deadlocked at shutdown when the sender was parked on `Semaphore::acquire_owned()` waiting for permits the receiver would never return; (2) the human-report `sustained_eps:` label was emitted unconditionally, mislabelling 1.5-second burst rates from `--total-events`-capped runs as 60-second sustained rates. Most committed numbers in `.planning/throughput-baselines.md` through Phase 13 are therefore burst rates — Plan 13.7.6-28 will re-baseline.

## Root cause — Bug 1: sustained-mode deadlock

`run_tcp_continuous_push_worker` (in `crates/beava-bench/src/bin/beava-bench-v18.rs`) splits the worker into a sender task and a receiver loop. The sender uses `Semaphore::acquire_owned().await` to gate `pipeline_depth` in-flight pushes; permits are returned by the receiver via `add_permits(1)` on each ack.

The receiver had **only one** path that closed the semaphore — the cap-cross path inside the 'recv loop:

```rust
if let Some(cap) = total_cap {
    if new_count >= cap {
        stop.store(true, Ordering::Release);
        sem.close();         // <-- only here
        break 'recv;
    }
}
```

Every **other** exit path (deadline + `stop` flipped by the main task at line 890; socket EOF; `decode_frame` error; the 50 ms tick noticing `stop`) would `break 'recv` without closing the semaphore. Then:

```rust
flush_latencies(&push_hist, &mut latency_batch).await;
let _ = sender_handle.await;   // <-- blocks forever
```

If at the moment `stop` flipped the sender was parked on `acquire_owned().await` with all permits forgotten and no acks coming back, the sender could not wake — `acquire_owned` only wakes on (a) a permit becoming available or (b) the semaphore being closed. The receiver in turn waited on `sender_handle.await`. **Full deadlock; observed 2026-05-05 hanging at ~0.1 % CPU for 14+ minutes before being killed.**

`run_tcp_burst_push_worker` and the HTTP path are unaffected: burst mode has no semaphore (it uses a `read_n_frames` batch round-trip), and HTTP is request-response per event.

## Fix

`crates/beava-bench/src/bin/beava-bench-v18.rs:1557` — added `sem.close()` immediately before `sender_handle.await`. Closing an already-closed semaphore is a no-op, so the cap-cross path that closes inside 'recv is unaffected. Now every exit from 'recv funnels through one close site:

```rust
flush_latencies(&push_hist, &mut latency_batch).await;
sem.close();                  // wakes sender if parked on acquire_owned()
let _ = sender_handle.await;  // sender exits via the Err branch
```

## Root cause — Bug 2: methodology mislabel

`format_report` emitted the `sustained_eps:` label unconditionally. With `--total-events 1_000_000 --duration-secs 60` at our ~600K EPS, the run finishes in ~1.5 s — an 1.5-second burst rate, but reported as `sustained_eps:` against the 60-second deadline framing. Every committed throughput number in `.planning/throughput-baselines.md` that was captured with `--total-events` is therefore mislabelled.

## Fix

`crates/beava-bench/src/bin/beava-bench-v18.rs:1604` — conditional label:

```rust
let elapsed_secs = r.elapsed.as_secs_f64();
let duration_secs = cli.duration_secs as f64;
let is_sustained_run = elapsed_secs >= duration_secs * 0.95;
let eps_label = if is_sustained_run {
    "sustained_eps:"
} else {
    "burst_eps:    "
};
```

5% slack tolerates the 50 ms deadline-poll quantum, tokio teardown overhead, and pool-build variance. The label change is purely additive in the human report — a reader skimming the 17-line summary now sees instantly which mode produced the number.

The struct field name `WorkloadResult.sustained_eps` is unchanged for API stability (kept since Phase 18; mathematically `push_count / elapsed.as_secs_f64()` is what it always was). A doc comment now flags that it can render either label.

### Out of scope — heavy-params hang

The plan's example verification command uses `--parallel 32 --pipeline-depth 1024`. With the bench-side fix applied, this configuration **still hangs** — but the cause is **server-side**, not in the bench. A `sample` capture taken during the hang shows the bench's tokio workers parked on `cv_wait`, the server's `beava-apply` thread parked on `semaphore_wait_trap`, and both `beava-io-worker-0/1` parked on `semaphore_wait_trap` — i.e. server starvation at extreme fan-out (32 connections × 1024 in-flight = up to 32 K simultaneous in-flight pushes against a 2-IoPool worker default on macOS / 10 cores). Moderate parallelism (parallel=16, pdepth=256, ~3.5 M events in 5 s) works correctly. Filing a separate ticket / future plan for the server-side starvation is the right move; it is not in scope of this plan and not blocking the launch (the documented Reproduce block uses moderate values).

## Implications for `.planning/throughput-baselines.md`

Most committed throughput numbers through Phase 13 were captured with `--total-events N` and so are **burst rates**, not sustained rates. With the bench now self-labelling, **Plan 13.7.6-28 will re-baseline** by re-running the rows under `--duration-secs 60` (no event cap) and confirming the output reports `sustained_eps:` rather than `burst_eps:`.

The README throughput claim (WEBSITE-GAPS.md §43) is correlated: it cites a number that is likely the burst rate, not the 60-second sustained rate. Plan 13.7.6-28 will produce the corrected sustained baselines; a downstream README-text fix can reference those.
