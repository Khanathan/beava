# Phase 13.7.6 Plan 25: Throughput-baselines Reproduce-block fix Summary

Fix `.planning/throughput-baselines.md` Reproduce block (it was an unrunnable copy-paste — `cargo run -p beava-bench` had no `default-run`, and the flags `--pipeline / --transport / --duration-secs / --parallel 8` didn't match any actual binary's CLI surface) and add `default-run = "beava-bench-v18"` to `crates/beava-bench/Cargo.toml` so plain `cargo run -p beava-bench --release` resolves to the production harness.

## What was done

### Part A — `.planning/throughput-baselines.md` Reproduce block rewrite

Replaced the broken 5-line Reproduce block with a full working invocation. New block:

- Opens with a paragraph distinguishing the production harness `beava-bench-v18` (multi-worker, real parallelism, used for ledger rows) from the polished smoke surface `beava-bench throughput` (single-threaded; `--parallel` flag is a no-op in the throughput harness today).
- Shows the actual working invocation including the required `cd crates/beava-bench` step (because v18 reads `./configs/{pipeline}.json` relative to CWD), `--wire-format msgpack`, `--blast-shape zipfian --cardinality 10000`, `--pipeline-depth 1024`, and `--parallel 32` — flags that match the production-shaped configurations used by recent ledger rows.
- Notes the `cargo build --release -p beava-bench --bin beava-bench-v18` build prerequisite, and that after Plan 13.7.6-25's `default-run` change plain `cargo run -p beava-bench --release -- …` also works.
- Adds a **sustained-rate vs burst-rate** caveat: use `--duration-secs 60` WITHOUT `--total-events`. Setting `--total-events N` causes the run to finish in ~1.5 s at our throughput (a burst rate, not the sustained-rate number Phase regression gates target per CLAUDE.md §Performance Discipline).

### Part B — `crates/beava-bench/Cargo.toml` `default-run`

Added `default-run = "beava-bench-v18"` to `[package]` with a comment pointing at Plan 13.7.6-25 and the original symptom (plain `cargo run -p beava-bench` failing with "could not determine which binary to run" because the crate ships four `[[bin]]` targets — `beava-bench`, `-legacy`, `-v18`, `-v2`).

Verified by:

1. `cargo metadata --no-deps` reports `default_run: beava-bench-v18` for the package — Cargo accepts the field cleanly with all four `[[bin]]` targets defined.
2. `cargo run -p beava-bench --release -- --help` now prints `Running 'target/release/beava-bench-v18 --help'` followed by the v18 standalone harness help text (previously this command failed with "could not determine which binary to run").
