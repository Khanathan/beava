# Phase 13.7.6 Plan 07: ASVS Level-1 threat-model narrative — Summary

Single-page ASVS L1 threat model for Beava v0: trust boundary stated, asset
inventory + STRIDE table populated with concrete code references, V1–V14 chapter
coverage with honest accept/mitigate/n/a dispositions, and a posture statement that
spells out for OSS users exactly what v0 protects against and what it doesn't.

## What landed

- `.planning/phases/13.7.6-pre-oss-repo-polish/artifacts/13.7.6-07-asvs-l1-threat-model.md`
  (253 LOC) — single-page ASVS L1 narrative.

Document structure:

1. **Front matter** — ASVS version, level, last-reviewed.
2. **Trust boundary** — single-tenant, no auth, operator-owned network, admin
   sidecar separate port, no TLS in v0, mio-only invariant.
3. **Asset inventory** — table of in-memory state, WAL, snapshots, registry,
   configuration, admin endpoints with sensitivity classification.
4. **STRIDE threat register** — 11-row table covering Spoofing / Tampering /
   Repudiation / Info-disclosure / DoS / Elevation with disposition + mitigation +
   code reference for every entry.
5. **ASVS L1 controls coverage V1–V14** — one section per chapter, 1–2 paragraphs
   each, citing specific code lines or architectural-test paths.
6. **Cross-references** — links to SECURITY.md, artifacts 13.7.6-06 (OWASP review),
   13.7.6-08 (secrets sweep), 13.7.6-02 (cargo audit), CLAUDE.md.
7. **Posture statement for OSS users** — bullet list of what v0 protects against
   and an explicit list of what it doesn't (no auth, no authz, no encryption, no
   per-push audit log, no rate limiting). Closes with deployment scope (in scope
   for security reports = behind authenticated reverse proxy with TLS; out of
   scope = direct internet exposure).

## ASVS chapter dispositions (one-line each)

| Ch | Posture |
|----|---------|
| V1 Architecture | This doc + 3 locked CI-enforced invariants (mio-only, events-only, memory governance) |
| V2 Authentication | DEFERRED — v0 single-tenant; no auth on data plane |
| V3 Session | n/a — connection-stateless wire |
| V4 Access control | accept — single-tenant; identical privileges for all clients |
| V5 Validation | mitigate — JSON Schema 2020-12 prelude validation; structured error codes |
| V6 Cryptography | n/a — TLS terminated upstream; no encryption at rest |
| V7 Errors+Logging | mitigate — structured tracing fields; no stack traces to clients |
| V8 Data protection | accept — operator is data controller; encrypt the volume yourself |
| V9 Communication | accept — terminate TLS at upstream proxy |
| V10 Malicious code | mitigate — cargo audit + cargo deny in CI |
| V11 Business logic | mitigate — `force=True` register gate; `effective_test_mode` reset gate |
| V12 Files+Resources | mitigate — WAL/snapshot scoped to `.beava/`; no user-input → path |
| V13 API | mitigate — wire-spec.md JSON Schema 2020-12; structured 404 / OP_ERROR_RESPONSE |
| V14 Configuration | mitigate — env vars read once at boot; safe defaults; deny.toml policy |
