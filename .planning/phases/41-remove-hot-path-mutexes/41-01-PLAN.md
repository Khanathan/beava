# Plan 41-01 — Remove hot-path mutex contention

## Goal

Turn strace's 88%-futex into <60%. Remove or defang four per-PUSH mutexes (`recent_events`, `metrics.events_total`, `throughput`, `latency`). Multi-connection aggregate throughput should rise ~30-50% (from ~540k to ~700k+ eps in the 8-process 8-stream case). Single-stream single-client unchanged.

## Task T1 — Feature-gate `recent_events`

**Files:** `Cargo.toml`, `src/server/tcp.rs` (AppState field + push site), `src/server/http.rs` (endpoint + route).

**Subtasks:**
- Add `demo = ["server"]` to `[features]` in `Cargo.toml`.
- Gate `pub recent_events: ...` field on `AppState` with `#[cfg(feature = "demo")]`.
- Gate construction in `tcp.rs:307` with `#[cfg(feature = "demo")]`.
- Gate push call at `tcp.rs:1554`: `#[cfg(feature = "demo")] state.recent_events.lock().push(ev);`.
- Gate `public_recent_events` handler + `.route("/public/recent-events", ...)` in `http.rs`.
- Update `scripts/check-feature-builds.sh` to also check `cargo build --features demo` compiles.
- Test: default build + `cargo test` green; `cargo build --features demo` green; both have no regressions.

**Done when:** default build has `state.recent_events` entirely removed from the binary; demo-feature build still has it working.

## Task T2 — Atomic `events_total`

**Files:** `src/server/tcp.rs` (AppState + push sites), `src/server/http.rs` (read sites).

**Subtasks:**
- Add `pub events_total: AtomicU64` directly on `ConcurrentAppState` (next to the Metrics mutex, not inside it).
- In the two per-push sites (`tcp.rs:1123` single-path and `tcp.rs:1501` batch-path): replace `state.metrics.lock(); metrics.events_total += 1;` with `state.events_total.fetch_add(1, Relaxed);`.
- Any read sites (`/metrics` handler, `/public/stats`): read via `state.events_total.load(Relaxed)` instead of via the Metrics lock.
- Leave other Metrics fields alone — they're rare writes, no need to atomicize.
- Default test suite stays green.

**Done when:** single-path + batch-path pushes no longer lock `state.metrics` just to bump events_total.

## Task T3 — Atomic-ring `throughput`

**Files:** `src/server/tcp.rs` (AppState + push call), `src/server/http.rs` (read), possibly new `src/server/throughput.rs`.

**Subtasks:**
- Inspect current `ThroughputTracker` struct. Identify what `/metrics` actually exposes from it (EPS rolling? unique-keys-in-window?).
- Replace with `pub struct AtomicThroughput { buckets: [AtomicU64; 60] }` — 60 one-second buckets. `bump()` uses `SystemTime::now().as_secs() % 60` to index, `fetch_add(1, Relaxed)`.
- Replace `state.throughput.lock().bump_unique(touched, now_inst)` at `tcp.rs:1118` with `state.throughput.bump(count)` (lock-free).
- `eps_5s()` reader sums the 5 buckets ending at `now-1s` (avoids the currently-being-written bucket for consistency).
- **If** unique-key-per-window was exposed via `/metrics` and is load-bearing: keep the existing mutex-based tracker BUT decouple from the per-push hot path. E.g., sample uniqueness on a separate timer. Document the trade-off.
- **If not** load-bearing: drop uniqueness counter; note in SUMMARY.

**Done when:** per-push throughput bump is lock-free. `/metrics` EPS rolling still accurate within one-second granularity.

## Task T4 — Sampled `latency`

**Files:** `src/server/tcp.rs` (latency lock sites).

**Subtasks:**
- Add `pub latency_sample_counter: AtomicU64` on AppState.
- At each `state.latency.lock()` call (lines 610, 781, 1135, 1620, 1698, 2250 — audit all):
  - If it's per-push (called on every event): replace with
    ```rust
    let n = state.latency_sample_counter.fetch_add(1, Relaxed);
    if n % LATENCY_SAMPLE_STRIDE == 0 {
        let mut latency = state.latency.lock();
        latency.record_push(micros);
    }
    ```
  - If it's rare (e.g., on REGISTER or admin ops): leave unchanged.
- `const LATENCY_SAMPLE_STRIDE: u64 = 16;` in `src/server/tcp.rs` or similar.
- `/metrics` latency histogram reads — no change. Histogram sampled 1-in-16 still shows representative p50/p99 within a small relative error.
- Document: "histograms sampled at 1-in-16; use the --no-sample flag at build time if exactness needed".

**Done when:** per-push latency lock acquired ~94% less often.

## Stop-and-report triggers

- `ThroughputTracker` unique-key counter turns out load-bearing for a metric we care about → flag; ship the atomic-ring without the unique-keys part + note.
- `latency` histogram API doesn't tolerate decimation — flag; keep fully-sampled but document the hot-path cost.
- `#[cfg(feature = "demo")]` ripples farther than CONTEXT expects (other demo-only code shares the gate) → just gate them together; note.
- Single-stream regression >5% after the atomic counters → rollback events_total change and flag.
- 8-proc 8-stream aggregate regresses (shouldn't — we're removing locks) → rollback and profile.
