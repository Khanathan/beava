## Key findings (commit-worthy data, not yet documented)

### 1. Wall-clock measurement bug

| N (events) | reported wall (1s get-interval) | reported EPS | reported wall (10ms get-interval) | actual EPS |
|------------|--------------------------------:|-------------:|----------------------------------:|-----------:|
| 1k | 1000ms | 1k | — | ~5-10k |
| 10k | 1002ms | 10k | 514ms | 19k |
| 100k | 1000ms | 100k | 504ms | 198k |
| 200k | 1007ms | 199k | 504ms | 397k |
| 500k | 1010ms | 495k | 1008ms | **496k** ✓ |

For N ≥ ~500k, wall_clock matches reality. Below that, it's dominated by background-task sleep cycles.

### 2. WAL bimodal-tail (eliminated by config bump)

Original 3 buffers × 16 MiB, tick_ms=2: trace at N=500k showed **4,900 events (1%) with wal_append > 1ms** (range 1-42ms each), summing to 17.4 SECONDS of apply-thread stall time.

Experimental fix: 8 buffers × 64 MiB, tick_ms=20-100: bimodal tail entirely gone (0 events with wal_append > 1ms). Throughput lift: 500k → **660k EPS** for cache-hot fixed shape.

Trade-off: 8×64MiB = 512MB resident WAL (vs 48MB original). Heavy. Probably 4×32MiB = 128MB is the right default if landed.

NOT committed — needs proper Phase 19.0.1 plan with TDD.

### 3. Pipeline-complexity bottleneck crossover

| Pipeline | fixed (cache-hot) | zipfian (realistic) | uniform (cold) |
|----------|------------------:|--------------------:|---------------:|
| small (1 op) | 663k | **536k** | 370k |
| medium | 664k | 535k | 340k |
| large_phase9 (15 features) | 664k | 587k | 265k |
| **large-with-sketches** (5 sketches) | **644k** | **213k** | **61k** |
| geo | 668k | 386k | 386k |

**Two regimes confirmed:**
- Simple pipelines: wire-stack-bound at ~665k EPS (fixed shape — TCP/IoPool/Tokio roundtrip)
- Complex pipelines: apply-bound (`large-with-sketches` zipfian = 213k, theoretical = 250k)

Per-op trace breakdown for `large-with-sketches`:

| Op | ns/call | % features |
|----|--------:|-----------:|
| Percentile (UDDSketch) | 257 | 22% |
| Entropy | 224 | 19% |
| TopK (SpaceSaving) | 221 | 19% |
| CountDistinct (HLL) | 138 | 12% |
| BloomMember | 44 | 4% |
| Core (Sum/Avg/Min/Max) | ~20 | 5% each |

**5 sketches consume 76% of features-time despite being 25% of features.**

### 4. Allocation cost analysis (entity_row_init = 2576 ns cold)

Decomposed via first-principles analysis: NOT map lookups (only ONE raw_entry probe per call). The cost is **`WindowedOp::buckets: [Option<Box<AggOp>>; 64]`** preallocation — every new entity gets a 64-slot bucket array zero-initialized for EACH windowed op (4 of them in `large-with-sketches`).

**Cold-key path budget (predicted):**
- EntityKey clone + raw_entry insert: ~250 ns
- `Vec<AggOp>::with_capacity(20)`: ~150 ns
- 4 × Box<WindowedOp> heap alloc: ~300 ns
- 4 × `[Option<Box<AggOp>>; 64]` zero-init (memset 512B each): ~200 ns
- 4 × `[i64; 64]` set to i64::MIN: ~200 ns
- BloomMember bit array: ~200 ns
- 15 × core AggOp::new: ~450 ns
- Misc: ~300 ns
- **Predicted ~2050 ns vs observed 2576 ns** ✓ within 25%

Optimization win #1 (lazy bucket allocation in WindowedOp): replace `[Option<Box<AggOp>>; 64]` with `SmallVec<[(i64, Box<AggOp>); 4]>`. Saves ~1500 ns/event for cold keys (60% of init cost). Most entities only see 1-2 active buckets.
