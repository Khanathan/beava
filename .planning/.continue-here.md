## Key empirical findings from this session

**Apply-thread cost decomposition** (Hetzner Linux EPYC KVM, perf record):

| Workload | apply CPU | Real work | Channel/mio | Kernel I/O | Allocator |
|---|---:|---:|---:|---:|---:|
| Saturated push | 3.6% | 47% (1.7% total) | 42% | 5% | 6% |
| Heavy read | 11% | 9% (1% total) | 10% | 16% | 10% |

**Headline:** apply thread is 3.6% CPU during "saturated" push at 51k EPS — server is fundamentally underutilized. Bench-side allocator (`osq_lock` → `mprotect` → `malloc` → `BytesMut::reserve_inner`) hits 19% of total CPU. Multi-process bench (8 separate processes via new `--remote-addr`) confirms server is the actual bottleneck (1-client 51k EPS = 8-client 51k EPS aggregate).

**Push apply per-event work:** 12 µs/event on fraud-team (mostly aggregation; AggOp::update + CountDistinct + WindowedOp + TopKState). Real-work-dominant: agg work is 47% of apply CPU.

**Read apply per-event work:** 542 ns p50, 1.21 µs mean (single-cell). Overhead-dominant: orchestration is 91% of apply CPU.
