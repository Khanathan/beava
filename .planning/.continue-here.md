## Two Follow-Ups On My Mind

### 1. Valkey-style IO architecture rework (post-v0 / v0.1+)

**Problem.** Beava's IO worker design has each worker own its own `mio::Poll` and independently poll its assigned client subset, sending parsed RingItems to apply via crossbeam channel. Valkey is fundamentally different: ONE `epoll_wait` on main thread, IO threads are pure SPSC/SPMC consumers that never call `epoll_wait`.

**Verified Valkey source** (`valkey-io/valkey/src/io_threads.c::IOThreadMain`):

```c
while (1) {
    pthread_testcancel();
    // PRIORITY 1: drain private SPSC queue
    while ((batch_count = spscDequeueBatch(&io_private_inbox[id], ...)) > 0) { ... }
    // PRIORITY 2: drain shared SPMC queue
    void *tagged_job = spmcDequeue(&io_shared_inbox);
    if (tagged_job) { ... }
    // sleep on mutex if both queues empty
    if (processed == 0) {
        pthread_mutex_lock(&io_threads_mutex[id]);
        pthread_mutex_unlock(&io_threads_mutex[id]);
    }
}
```

**Beava current** (`crates/beava-runtime-core/src/io_thread_worker.rs`):
- Each worker owns mio::Poll + Waker + clients HashMap
- Workers continuously poll their own clients
- Send RingItems via crossbeam to apply thread (which spin-loops on `try_recv` then falls to 50µs `recv_timeout`)

**Cost overhead beava pays that Valkey doesn't:**
- N+1 `epoll_wait` syscalls per tick (vs 1 for Valkey)
- Crossbeam channel hop per push event
- Apply thread busy-spin / recv_timeout
- For few-hot-connections-at-high-pd workloads (our bench shape), this overhead is wasted

**Plan doc:** `.planning/ideas/valkey-io-architecture-rework.md` — 4-phase migration:
- A: measure actual cost (1 day) — Phase A is the GATE. If channel overhead is < 5% of total push CPU, abandon rework.
- B: consolidate poll on apply thread (3-5 days)
- C: add maxclients + slow-client backpressure (1-2 days)
- D: validate at scale (2 days)

**Open question:** does the trace overhead actually justify the rework? Until Phase A measurements are done, this is speculative. Apply-CPU is 88% of total push time per the per-stage trace; channel overhead is at most 12%.

### 2. Per-entity memory overhead investigation (Phase 13 ship-blocker)

**Problem.** Empirical per-entity RSS overhead from the r8g max-cardinality test (this session, 120 GiB pods):

| Pipeline | Features | Per-entity | Notes |
|---|---|---|---|
| **small** | 1 (count) | **~1 KB** | matches expectations |
| **medium** | 5 (count/sum/avg/min/max) | **~5.6 KB** | within budget |
| **large_phase9** | 15 (incl. Phase 9 decay/velocity) | **~18 KB** | over budget |
| **fraud-team** | ~90 features × 9 aggs | **~22 KB** | **3× over budget** |

**CLAUDE.md says:** *"~7KB per entity for a rich 30-feature pack → ~700GB for 100M entities."* The 7 KB budget holds for medium-shape workloads. fraud-team's heavier sketch ops blow past it.

**Likely culprits to investigate:**

1. **HLL `count_distinct`** state — sparse + dense representations, register array, etc.
2. **UDDSketch `percentile`** — bucket array growing with seen value range
3. **TopK SpaceSaving** — counter array sized by `k` parameter
4. **Histogram bounded-buffer ops** — fixed buffer per entity even when underutilized
5. **`bv.entropy`** — categorical buckets capped by `max_categories`
6. **Phase 9 decay/velocity ops** — running mean/variance in EWMA, ring buffers in `inter_arrival_stats`, `outlier_count`

**Investigation plan (rough):**

1. Add a `bench-mem` binary that registers each operator type individually, pushes N events to one entity per op, and prints `std::mem::size_of_val` of the AggOp state.
2. Sum up per-feature sizes for fraud-team's actual aggregation declarations and compare to measured 22 KB.
3. Identify the top 3 contributors. Decide:
   - (a) Cut overhead 3× by repacking sketch internals (compact-rep HLL? variable-bucket UDDSketch?)
   - (b) Reframe v0 pitch — "1M entities per simple feature pack" instead of generic claim
   - (c) Recommend r8g.8xlarge (256 GiB) for fraud-team-shape workloads
   - (d) Some combination

**Output:** add to Phase 13 CONTEXT or write a separate `.planning/ideas/per-entity-memory-budget.md` similar to the Valkey doc.

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention |
|---|---|---|---|
| `pkill -f /work/beava` self-kill | `pkill -f` matches the bash script's cmdline (which contains the path string), causing pkill to kill its own parent before completing. Cost ~30 min of bench-debugging this session. | advisory | Use PID-file pattern: `& echo $! > /tmp/beava.pid; disown` then `kill -9 $(cat /tmp/beava.pid)`. Implemented in `/tmp/beava-bench-aws/restart-helper.sh`. |
| Single-process bench saturation | At 16+ workers/target in one bench-v2 process, in-process `AsyncMutex` contention degrades both EPS and latency below the per-shard server ceiling. | advisory | Sweep config showed 1 worker × pd=1024 wins on latency; 4 workers × pd=1024 wins on EPS for light pipelines. |
| Trusting "Valkey 8 model" code comments | Beava's IO worker has comments claiming Valkey 8 model. Verifying against actual Valkey source revealed beava diverges (per-worker mio::Poll vs Valkey's main-only epoll). | advisory | Always verify architectural-equivalence claims against upstream source before treating as design constraint. |
| Conflating cluster-autoscaler desired with intent | EKS cluster autoscaler reverts node-group desiredSize to 0 when no pending pods demand the nodes. Manual `aws eks update-nodegroup-config --desiredSize=4` won't stick if no pods are scheduled. | advisory | Apply pod manifests FIRST, autoscaler scales up to meet demand. Or temporarily raise `minSize` to keep nodes alive between bench runs. |
