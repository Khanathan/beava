### Q1 — Python dataframe API similarity check

Users should be able to use Beava's API for dataframe-style work. The SDK chain (`filter` / `select` / `with_columns` / `cast` / `fillna`) is already polars-shaped, but there are gaps to close so a data-scientist familiar with pandas/polars can use Beava without learning a foreign API.

**Open aspects:**
- Gap analysis — actual SDK shape vs pandas/polars idioms (what feels natural, what doesn't)
- Aggregation chain shape — `events.group_by(...).agg(...)` (current) vs simpler `events.agg(group_by=, ...)` for common cases
- Eager-vs-deferred mental model — pandas/polars are eager (`df.assign` reads immediately); Beava is register-then-push (deferred). How do we frame this difference in docs without confusing dataframe-native users?
- DataFrame interop — does `app.get(...)` return something that can be `pd.DataFrame.from_records(...)`-friendly? `FeatureResult` shape question.
- Recipe parity — every pandas recipe (group_by/rolling/agg) should have a Beava equivalent in docs

**Phase target:** Phase 13 docs/SDK polish, possibly with API touchpoints in Phase 12.6 if gaps surface during the SDK sweep.

**Existing memory connection:** `project_pandas_mapping_revisit` already flagged that beava.dev homepage uses simplified `events.agg(key=, window=, ...)` form that doesn't match real decorator API.

### Q2 — Schema evolution UX (additive + warn/block on destructive)

Three flows users will hit on day one:
- **Adding a new field** to an existing event: should work seamlessly (currently does — additive registration with version bump → 200 OK)
- **Removing a field** (destructive): server should **warn AND block** with a clear, human-readable explanation. Currently returns 409 with structured JSON `{error: {code: "destructive_change", diff: {...}}}`. The polish gap is making the error human-readable + obvious recovery path.
- **Adding a new pipeline / derivation**: should work additively (currently does, but validate end-to-end UX)

**Open aspects:**
- Error message format — 409 body shape: structured JSON only, or also a `human_message` field?
- Preview mode — `app.validate(*descriptors, against_server=True)` that returns the diff WITHOUT registering? Or HTTP `POST /register/dry-run`?
- Force-override — does v0 allow `force=True` to bypass destructive-change block (with explicit acknowledgment) or is the only path "rename + re-register"?
- Adding-new-pipeline UX — friction-free; validate end-to-end with a real dataframe-style add-derivation flow.
- Warning vs error distinction — is there a class of changes that should warn-not-block (e.g. type widening int→f64)? Out-of-scope for v0?

**Phase target:** Phase 12.6 (server-side error message work since wire schema is bumping anyway) + Phase 13 SDK polish (preview mode + docs).

### Q3 — Recovery benchmark — easy-to-run reproducible harness

Users (and CI) should be able to validate the ≤ 30s RTO at 10GB state on NVMe target with one command. Currently the target is documented but there's no reproducible harness. Also: the same harness should validate post-Phase-12.6 wire-schema-bump compatibility (old-schema-WAL → new-schema-replay).

**Open aspects:**
- Harness shape — new `beava-bench-recovery` binary, or a flag on `beava-bench-v18`?
- Workload knobs — entity count, feature count per entity, WAL size, on-disk vs tmpfs, fsync mode (`Periodic` vs `PerEvent`)
- Output format — same row format as `.planning/throughput-baselines.md` for consistency, or separate `recovery-baselines.md`?
- CI integration — every PR, or only Phase-13 ship-gate runs?
- Schema-version handoff testing — recovery harness should also test old-schema-WAL → new-schema-replay path during the Phase 12.6 wire bump

**Phase target:** Phase 13 perf-gate work (alongside throughput rebaseline + Hetzner sweep).
