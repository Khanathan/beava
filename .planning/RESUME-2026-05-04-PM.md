## CRITICAL — Pre-OSS-release blockers

### Blocker 1: Sibling event-derivation auto-routing (architectural)

**The test_lit.py failures expose this:** when a downstream `@bv.event def UserClicks(tagged: Click)` annotates a raw event source, but an upstream `Tagged.with_columns(...)` derivation has added columns to that source, the runtime doesn't auto-route through the Tagged derivation. The downstream consumer sees the raw schema, not the augmented one.

**Where this breaks (5 test_lit failures):**
- `test_lit_constant_column` — `bv.lit("web") AS source` column added by Tagged derivation; downstream UserClicks doesn't see it
- `test_lit_explicit_filter_literal` — `where bv.lit(True)` predicate; literal column not visible
- `test_lit_force_float_division` — `bv.lit(1.0)` cast column not visible
- `test_lit_value_types` — multi-type literal columns not visible
- `test_lit_immutability` — register fails with constraint violation

**Why this matters for OSS release:**
The whole `bv.lit(...)` API is documented in `docs/sdk-api/python.md` and ADR-003 as the canonical way to add constant columns. If the homepage demo or any example uses `bv.lit`, it's broken. The 5 failures aren't edge cases — they're the "constant column" use case that ANY user adding a category column to an event will hit.

**Estimated scope:** 1-2 days. Touches:
- Server-side: derivation-graph router needs to detect sibling Tagged derivations on the same event source and prefer them when downstream annotates the raw source
- OR: registry validation should reject the ambiguity at register-time and force users to explicitly annotate the Tagged derivation
- Either way: needs an architectural decision (D-XX) before plan-phase

**Recommended action:** start a Phase 13.5.2 (or 13.5.1-08) with `/gsd-discuss-phase` to lock the routing semantics, then plan + execute. Don't try to inline-fix.
