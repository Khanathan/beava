## SDK deferrals (from 2026-05-03 design session)

| Item | Why deferred | Workaround in v0 |
|---|---|---|
| **Java SDK** | Not in top-3 popularity for beava's startup/modern-stack target audience; large maintenance lift (Maven/Gradle/JVM compat); slower adoption velocity | Wire spec doc enables community port |
| **Async / `AsyncApp`** | Sync covers v0; async is parallel API, easy additive change | `asyncio.to_thread(app.push, ...)` |
| **`app.deregister(name)`** | No server-side opcode yet; force-re-register handles drop case | `app.register(*reduced, force=True)` |
| **`app.migrate(...)`** | Non-trivial schema evolution (re-key, fold, reshape) — needs design work | Additive-only or `force=True` drop+rebuild |
| **Lifecycle hooks** (`before_push`, `after_register`, retry callbacks) | Niche middleware feature; not blocking launch | Wrap `app.push()` in user code |
| **Structured `ErrorCode` enum** (Python) | String-match on `exc.code` works | `if exc.code == "unknown_table": ...` |
| **Schema introspection** (`app.list_descriptors()`, `app.get_schema(name)`, `app.diff_pending(*new)`) | Useful for tooling but not v0 launch-critical | `app.ping()` returns `registry_version`; `dry_run=True` returns the diff |
| **Embed binary version-introspection** beyond version-pin (`app.embed_info()`) | Niche debugging feature | Check `beava --version` separately |
| **`OP_PUSH_BATCH`** (multi-event push frame) | Single connection means low overhead; loop-push is fine | `for event in batch: app.push(...)` |
| **`OP_PUSH_SYNC`** (acks=all durable push) | Phase 6.1 designed but not wired through; v0 push is acks=1 (Kafka-style) | Use disk-mode embed for tests; production accepts acks=1 |
| **Multi-conn pool** inside one `App` | Single-conn + auto-reconnect handles common cases; horizontal concurrency via multiple processes | Spawn multiple `App` instances |
| **HTTP/2** | HTTP/1.1 + keep-alive sufficient for v0 | n/a |
| **msgpack-on-HTTP** (currently TCP-only) | TCP is the perf path; HTTP is the curl/WAF path | Use TCP for production; HTTP for setup |
| **Brew formula** (`brew install beava`) | Optional polish; pip + docker cover most install paths | `pip install beava` or `docker run beava/beava` |

## v0.0.x — alias removals & deprecations

Items scheduled for removal in the v0.0.x cleanup pass after v0 GA. These are
one-release deprecation paths added during v0 hardening; their alias / shim
code lives in production but emits WARN-level deprecation logs.

| Item | Added | Reason | Removal trigger |
|---|---|---|---|
| `BatchGetReqEntry` `serde(alias = "entity_id")` (Phase 13.4.1 D-04) | 2026-05-04 | One-release back-compat for the `entity_id` → `key` field rename on `POST /batch_get` + `OP_BATCH_GET`. Custom `Deserialize` impl tracks `from_alias: bool`; `dispatch_batch_get_sync` emits a `tracing::warn!` per entry that used the alias ("BatchGetReqEntry: deprecated 'entity_id' field name; rename to 'key'; alias removed in v0.0.x"). | First v0.0.x patch cycle: drop the `from_alias` field on `BatchGetReqEntry` + the custom `Deserialize` arm for `entity_id` + the WARN-emission block in `apply_shard.rs::dispatch_batch_get_sync`. Plan 03's `phase13_4_1_entity_id_alias_warn.rs` precedent test flips: `phase13_4_1_alias_entity_id_deserializes` should turn RED (request 400/500 when `entity_id` is supplied), `phase13_4_1_alias_entity_id_emits_warn_log` becomes obsolete (no log to assert). |

### Lock HTTP `/push` body shape (Phase 13.5.1 Plan 07e Deviation 4, 2026-05-04)

**Status:** SDK `HttpTransport.send_push` sends `{event, data}` (matches the live server parser at `crates/beava-runtime-core/src/http_listener.rs:284,301,313`). Documentation drift introduced in Phase 13.4 verb-style migration — `docs/http-api.md:176` and PATTERNS.md still describe `{event_name, fields}`.

**Removal action (v0.0.x cleanup):** pick one (`{event, data}` lock OR rename to `{event_name, fields}` everywhere) and align `docs/http-api.md` + the parser in `http_listener.rs` + `python/beava/_transport.py::HttpTransport.send_push` + the corresponding TS / Go SDK send-push helpers. Wire-shape is private (no curl-hand-construction in production), so a coordinated rename in v0.0.x is acceptable.

### Standardize TCP success-response opcodes (Phase 13.5.1 Plan 07e Deviation 5, 2026-05-04)

**Status:** SDK `TcpTransport.send_reset` checks `OP_GET_RESPONSE = 0x0023` for success (matches live server at `crates/beava-server/src/server.rs:2338-2344` returning `OP_GET_RESPONSE` with body `{"reset": true, "registry_version": N}` per `apply_shard.rs:1538-1539`). PATTERNS.md A5 documents `OP_RESET` as returning its own opcode `0x0040` — but server returns the GET-response opcode.

**Removal action (v0.0.x cleanup):** standardize on either generic `OP_JSON_RESPONSE` for any JSON-bodied success response, or per-op `OP_RESET_RESPONSE` / `OP_PING_RESPONSE`. Document the chosen scheme and align SDK + server. Wire-shape is private (no curl-hand-construction; framed TCP is for high-perf clients), so a coordinated rename in v0.0.x is acceptable.

### Remove `_tcp_get_single` / `_http_get_single` private helpers (Phase 13.5.1 D-04, 2026-05-04)

**Status:** Renamed `tcp_get_single` / `http_get_single` → `_tcp_get_single` / `_http_get_single` in Phase 13.5.1 to mark them private (USER-LOCKED D-04). Public surface is `send_get` / `send_batch_get` only. Schedule full removal for the v0.0.x cleanup.

**Sites:**
- `python/beava/_transport.py::HttpTransport._http_get_single` (line ~173)
- `python/beava/_transport.py::TcpTransport._tcp_get_single` (line ~639)
- `python/beava/_transport.py::EmbedTransport._tcp_get_single` (line ~809; one-line delegate)
- Callers: `python/benches/read_bench_tcp.py`, `python/tests/integration/test_phase12_09_msgpack_get.py`

**Removal action:** delete the three method definitions; migrate the bench + integration test to `send_get` (which now supports the same `feature` / `key` shape via `app.get(table, key)`); delete or rewrite `python/tests/integration/test_phase12_09_msgpack_get.py` if it solely exercises the legacy CT_MSGPACK fast-path.

## How to add an item to this list

When deferring something in a session, append to the appropriate section above with:
- Item name
- Why deferred (1-2 sentence rationale)
- Workaround in v0 (so users have a path today)
- Estimated effort (LOC) for v0.1 implementation

For larger directions (e.g. historical engine, session windows), create a dedicated doc in `.planning/ideas/` and link it from § Big architectural directions.
