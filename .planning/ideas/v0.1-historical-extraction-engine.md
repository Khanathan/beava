# v0.1+ — Historical extraction engine (SpeeDB local + SlateDB S3)

**Status:** Captured 2026-05-03 from the v0-launch design session. NOT in v0 critical path.
**Tech choices locked:** SpeeDB for local single-node deployments; SlateDB for cloud / S3-backed deployments.
**Decision needed before promoting to a phase:** which v0.1 minor (v0.1.0 / v0.1.1 / v0.2.0 etc.) ships this, and whether to gate on adoption signals (do users actually want point-in-time queries enough to justify the engineering lift?).

---

## TL;DR

Beava v0 is **events-only, in-memory, processing-time only**. State lives in RAM; the WAL exists for crash durability, not as a queryable history. v0.1+ adds an **optional historical extraction engine** that:

- Persists every event to a queryable historical store (NOT just the WAL-for-durability)
- Allows backfill: spin up a new `@bv.table` and have it compute from historical events instead of only forward
- Enables point-in-time queries (in arrival-time semantics, NOT event-time — preserves `project_redis_shaped_no_event_time_ever`)
- Supports cross-session forensics: re-run the same pipeline against historical events for incident investigation
- Cold-starts new entities with warm features by replaying their historical events

**Two backends:**
- **Local** (single-node, on-disk) → [SpeeDB](https://github.com/speedb-io/speedb) (faster RocksDB-compatible fork)
- **Cloud** (multi-instance, object-storage-native) → [SlateDB](https://github.com/slatedb/slatedb) (LSM on S3/R2/GCS, designed for serverless)

**Deployment story:**
- Local `beava` + SpeeDB → single-node, fast hot path + on-disk historical
- Cloud `beava` + SlateDB → multi-instance via shared S3 history; **read-only siblings** can attach to the same bucket for offline analysis (e.g., a developer pointing a local `beava` at production's S3 bucket for debugging)

**Architectural placement:** historical writes happen on a **separate I/O thread**, off the apply hot path. The single-threaded apply loop remains the bottleneck; historical persistence is async via crossbeam channel.

## Why v0.1, not v0

### v0 limits

Per `project_redis_shaped_no_event_time_ever` (locked 2026-04-30), v0 commits to:
- No event-time anywhere
- No point-in-time queries
- No watermarks / late-event handling
- Processing-time only

These commitments serve simplicity + correctness for the bid-time-decisioning use case. Users today get current materialized features and that's it.

### What's missing for non-bid-time use cases

Users with historical/batch use cases (some prominent ones in fraud + ad-tech) need:
- **Backfill** — adding a new `top_k_merchants_30d` feature shouldn't require waiting 30 days for warm signal; should compute from history immediately
- **Forensic queries** — "what features did this user have at 14:32:00 UTC when the suspicious transaction landed" requires historical event replay
- **Model retraining** — pulling 90 days of event-aligned features for offline ML training
- **Cold-start enrichment** — a returning user who hasn't transacted in 60 days should have features computed from their full history, not from cold

v0 explicitly defers all of these. v0.1 adds them.

---

## Tension with locked commitments

`project_redis_shaped_no_event_time_ever` says "no event-time, no PIT". The historical engine seemingly contradicts this.

**Resolution: arrival-time-shaped historical queries only.**

- Historical store indexes events by `(event_name, arrival_lsn, key)`, NOT by user-supplied event-time
- "Point-in-time" queries become "events that ARRIVED before LSN N" or "events that arrived in time-range T1..T2"
- No watermarks, no late-event handling, no event-time bucketing
- **Backfill** uses the same `now_ms()` time source the live apply uses; replays events in arrival order

This preserves the spirit of the locked commitment (no event-time anywhere in the model) while enabling the use cases that motivate the historical engine.

If users genuinely need event-time PIT (e.g., "what was X's score AT the moment they did Y, not when we received Y"), that's a v0.2+ discussion that requires a new ADR overturning the lock. Punt for now.

## SpeeDB local backend

[SpeeDB](https://github.com/speedb-io/speedb) is a faster RocksDB-compatible fork by the Speedb team. API-compatible with RocksDB so we can swap in/out, but with measurable improvements:
- Faster compactions (no bloom filters in the hot path)
- Lower P99 tail
- Better write amplification

**Why SpeeDB over RocksDB:** if we're paying the LSM tax, get the faster one. SpeeDB is drop-in.
**Why SpeeDB over native fjall:** beava previously had fjall (Phase 12.6 era — removed). SpeeDB is more proven at scale; fjall is newer. SpeeDB also gives us a clear S3-counterpart story via SlateDB (both are LSM-shaped).

### Local deployment shape

```
beava (single binary)
  ├── apply thread          ← hot path (mio, single-threaded)
  ├── WAL writer            ← Phase 18 hand-rolled, 4×32MiB ring
  ├── snapshot writer       ← periodic full-state snapshot
  └── historical writer     ← NEW: receives events from apply via channel,
                              writes to SpeeDB on a separate I/O thread

SpeeDB layout:
  /var/lib/beava/historical/
    ├── events/              ← column family per event_name; key = (arrival_lsn, entity_key)
    ├── manifests/           ← LSM metadata
    └── checkpoints/         ← occasional consistent snapshots for fast recovery
```

### v0.1 wire spec for historical queries

```http
POST /historical/get
{
  "table": "UserImpressionFeatures",
  "key": "u4812|c101",
  "as_of_lsn": 1234567,
  "as_of_arrival_time": null   // alternative: arrival timestamp instead of LSN
}

→ {
  "row": {"imps_1h": 5, "imps_24h": 50, "last_seen_ts": 1714780800.0},
  "computed_at_lsn": 1234567,
  "events_replayed": 47
}
```

```http
POST /historical/replay
{
  "table": "NewFeatureTable",        // newly-registered table that needs backfill
  "from_arrival_lsn": 0,
  "to_arrival_lsn": "current"
}

→ {
  "status": "running",
  "job_id": "rep-abc123",
  "estimated_events": 5_000_000,
  "estimated_duration_sec": 1800
}
```

(Async job — `GET /historical/replay/job/{job_id}` polls progress.)

## SlateDB S3 backend

[SlateDB](https://github.com/slatedb/slatedb) is a 2024-era LSM key-value store designed natively for cloud object storage. Built for serverless workloads where you don't want to manage local disk.

**Why SlateDB:**
- Native S3 / R2 / GCS / Azure Blob support
- Pay-as-you-go storage cost (matches cloud pricing model)
- Read-shareable across processes (multiple `beava` instances can read the same bucket)
- Designed for cloud-first deployments (no need to provision EBS or local NVMe)

**Why not RocksDB-on-S3 (e.g., RocksDB Cloud):** SlateDB is purpose-built; less impedance mismatch. RocksDB-Cloud was discontinued by Rockset.

### Cloud deployment shape

```
beava cluster (multiple writers + multiple read-only siblings)

  Production writers (e.g. 4 instances, sharded by entity-key):
    beava-1   ─┐
    beava-2   ─┼── all writing to s3://my-beava-prod/historical/
    beava-3   ─┤
    beava-4   ─┘

  Read-only attached siblings (anywhere with S3 access):
    beava-readonly-laptop   ── attaches to s3://my-beava-prod/historical/
                                (can serve historical queries; cannot push events)

    beava-multi-region-replica  ── read-only sibling in another AWS region
```

### Multi-process readable

Key property: **SlateDB's design supports multiple readers on the same bucket.** This unlocks:

1. **Local dev against prod data:** A developer points their local `beava` at `s3://prod-bucket/`, gets read-only access to live production history. Useful for debugging "why did this user get scored as fraud?" without needing prod credentials beyond S3 read.

2. **Multi-region reads:** A `beava-readonly` instance in `us-east-1` can serve historical queries from a `us-west-2` writer's S3 bucket. Latency-bounded by S3 cross-region read but works.

3. **Forensic isolation:** Spin up an ephemeral `beava-readonly` for incident investigation; tear it down after. No risk to production state.

4. **Backup-as-read-replica:** S3 lifecycle policies + bucket replication give you cross-region disaster recovery for free.

## I/O thread architecture

The single-threaded apply loop is the bottleneck and the architectural lock (`project_no_sharded_apply`). Historical persistence MUST NOT slow it down.

```
┌─────────────────────────────────────────────────────┐
│ Apply thread (single)                              │
│ ─────────────────                                  │
│  - Receives push events from mio                  │
│  - Updates in-memory state (current materialized) │
│  - Writes WAL (durability)                         │
│  - Sends event copy to historical channel ────────┼──┐
└─────────────────────────────────────────────────────┘  │
                                                          ▼
┌─────────────────────────────────────────────────────┐
│ Historical I/O thread (single, dedicated)          │
│ ─────────────────────                               │
│  - Receives events from crossbeam channel         │
│  - Batches by event_name + key prefix             │
│  - Writes to SpeeDB (local) or SlateDB (S3)       │
│  - Tracks watermark: "all events through LSN N    │
│    are durable in historical store"               │
│  - Backpressures apply if channel fills (rare —   │
│    historical writes should be much faster than   │
│    real-time event arrival rate, except S3 spikes)│
└─────────────────────────────────────────────────────┘
```

**Apply-thread overhead added:** crossbeam `try_send` (non-blocking) — typically <50 ns per event when the channel has slack. If the historical thread can't keep up, the channel fills, `try_send` returns `Full`, apply either:
- (a) Drops the event (configurable behavior; log it; historical accuracy degrades)
- (b) Blocks apply until space frees (preserves accuracy; degrades real-time throughput)

Default: **(b)** with a configurable channel-depth (e.g. 100K events of slack) + `historical_full_strategy = "block" | "drop"` config flag.

For SlateDB (S3 latency tail can spike): channel depth needs to absorb minute-scale S3 hiccups without blocking apply. Probably 1M+ event slack with disk-backed overflow if memory is constrained.

## Backfill semantics

When a user adds a new `@bv.table`, two paths:

**Cold start (current v0 behavior, kept for v0.1):** new table starts empty; populates as new events arrive.

**Backfill (v0.1+ NEW):** explicit opt-in via `app.register(*new_descriptors, backfill=True)`. Server:
1. Validates the new descriptors as additive (or `force=True` for destructive)
2. Replays all historical events that match the new table's upstream events (filtered by event_name)
3. Populates the new table's state in arrival order
4. Bumps `registry_version`
5. Returns the registration response with `backfill_job_id` for progress polling

```python
# v0.1 SDK shape
result = app.register(
    NewFeatureTable,
    backfill=True,
    backfill_lookback="30d",   # or "all" / "1h" / specific arrival LSN
)
# result.backfill_job_id → "bf-xyz"

# Poll progress
while True:
    status = app.backfill_status("bf-xyz")
    if status.done: break
    print(f"backfill: {status.events_replayed:,} / {status.total_events:,}")
    time.sleep(5)
```

**Backfill duration estimate:** for a 30-day lookback at 100K EPS sustained → 30 × 86,400 × 100,000 = ~260 billion events. At 1M EPS replay throughput (no I/O bottleneck on local SpeeDB), that's 260,000 seconds ≈ 3 days. SlateDB will be slower (S3 read latency); probably 10-50× depending on bucket region + LRU cache hit rate.

For most use cases, backfill is a one-time operation per new table. Run it overnight; users see warm features in the morning.

## Open questions for v0.1 phase planning

These need answers before promoting this idea to a phase:

1. **Wire spec versioning.** Does adding `/historical/*` routes warrant a v1 prefix on the URL? Or stay at `/` (current v0 contract)? Probably worth `/v1/` because historical is a clearly different surface.

2. **Snapshot vs WAL relationship.** v0 uses snapshot+WAL for durability; the historical engine is a third store. Does the snapshot become redundant (historical store has all events; can recover state by replay)? Probably YES — historical engine subsumes snapshot for v0.1+, simplifying the durability story.

3. **Configuration surface.** How does a user pick SpeeDB vs SlateDB? Likely an env var or config file:
   ```
   BEAVA_HISTORICAL_BACKEND=speedb   # default for embed mode (local disk)
   BEAVA_HISTORICAL_BACKEND=slatedb  # for cloud deployments
   BEAVA_HISTORICAL_BACKEND=none     # opt-out (preserves v0 in-memory behavior)
   BEAVA_HISTORICAL_PATH=/var/lib/beava/historical    # SpeeDB path
   BEAVA_HISTORICAL_S3_BUCKET=s3://my-bucket/         # SlateDB target
   ```

4. **Read-only sibling discovery.** How does a `beava-readonly` instance know which bucket to attach to? Same env vars as the writer, with `BEAVA_READONLY=true` flag? Need to think about credential rotation, region selection, etc.

5. **Backfill-to-new-table during live writes.** If new events arrive while backfill is replaying, can the new table be partially queryable? Probably yes — backfill writes alongside live writes, both keyed by arrival LSN; queries see whichever data is materialized.

6. **Storage tiering.** For long-term history (e.g. 7+ years for compliance), can SlateDB integrate with S3 Glacier? Probably yes via lifecycle policies but worth designing the read story.

7. **Cost model.** Per-EPS storage cost on S3: 100K EPS × 86,400s × 1KB avg event = 8.6 TB/day raw. At S3 standard pricing ~$0.023/GB/month = ~$200/day = ~$6K/month for a single high-volume tenant. Compaction reduces this 5-10× but operators will care about this number. Doc the math.

## Decision blockers (v0.1 entry gate)

Before promoting this from idea-doc to active phase, need:

1. **At least 1 user actively asking for it** (no premature optimization)
2. **Cost model validated** with a real cloud-scale tenant (the $6K/month-per-tenant napkin math must align with what users will pay)
3. **Wire spec for `/historical/*` reviewed by SDK contributors** (TS / Go SDKs need to expose it; design once across all 3 languages)
4. **Bench numbers** for replay throughput on both SpeeDB and SlateDB at realistic load (500K+ events/sec replay target)

Until those land, this stays as an idea-doc.
