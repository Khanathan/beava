# v0.1+ idea: symbolic Python frontend (chalk-style)

> **Status:** post-v0 idea sketch. Not on the v0 ROADMAP. Captured 2026-04-23 during autonomous-mode discussion.

## TL;DR

Replace the chained-DSL feature-authoring API (`bv.col(...).gt(100) & bv.col(...).eq("X")`) with idiomatic Python that gets symbolically traced at decoration time and compiled to the same typed expression IR our Rust evaluator already runs.

Reference: chalk.ai's symbolic Python interpreter — <https://chalk.ai/blog/symbolic-python-interpreter>. Beava version uses our existing Phase 4 evaluator as the backend instead of Velox.

## How it works (two layers)

1. **Symbolic values + operator overloading** — at decoration time, the SDK calls the user's function with `SymbolicCol("name")` placeholders. Python's `__add__`, `__gt__`, etc. dispatch on those symbols and build expression-tree nodes.
2. **AST rewriting for control flow** — `if/else`, ternary, `and/or` short-circuit, comparison chains, `match` are *eager* in Python and don't reify via operator overloading. So at decoration time we `inspect.getsource(fn) → ast.parse → walk → rewrite into symbolic-IR builder calls (if_else, when, in_, ...)`. Then call the rewritten function with symbolic placeholders → get the typed IR tree.

Output is JSON expression IR — same wire format we already use for `/register`. Server-side: extend the Phase 4 evaluator with `IfElse`, `LetBinding`, `FuncCall`, `Compare`, `In` nodes + a small dispatch table for known pure functions (`abs`, `min`, `max`, `len`, `math.sqrt`, `math.radians`, etc.).

## What it unlocks

| Capability | Today (chained DSL) | After |
|------------|---------------------|-------|
| Local variables / let-bindings | No | Yes |
| `if/else`, ternary | `bv.when(...)` only | Native Python |
| Function reuse across features | No (each feature self-contained) | Yes (define helpers, compose) |
| Composable named predicates for `where=` | Copy-paste chains | First-class functions |
| mypy / IDE autocomplete / refactor | No | Yes |
| Imports from pure libs (math, your own helpers) | No | Yes (resolve at decoration) |

## What it does NOT unlock

- Side effects (HTTP calls, IO, model-server calls)
- Heavy ML inference (scipy/numpy/torch can't be symbolically traced)
- Loops with dynamic bounds, recursion, `eval`, mutation
- Custom aggregation operators (separate idea below)

Anything outside the symbolic-traceable subset gets rejected at decoration time with line/column-pointing errors. No silent fallback.

## Why not Velox

Chalk's backend is Velox (Meta's vectorized C++ query engine). Wrong tool for Beava because:

1. **Workload shape mismatch** — Velox wins on batch/columnar SIMD. Beava is per-event keyed state. There's no batch dimension to vectorize.
2. **Single-binary Rust** — Velox is C++ + ~50 transitive deps (Folly, fmt, glog, Boost, BISON, FLEX, …). Static linking blows the 200MB binary cap (PKG-05) by 3–5×.
3. **Operating model** — Velox assumes async query-execution with operator pipelines. Beava is `tokio current_thread` with single-thread apply loop.

Steal the *IR design ideas* from Velox (nullability tracking, type inference, function dispatch table, vector signatures) — re-implement in pure Rust. Don't link the code.

DataFusion / Polars / Arrow Rust kernels have the same workload-shape mismatch. Right tech, wrong problem for Beava.

## Cost to ship

- **Python**: ~500–1000 LoC AST rewriter using stdlib `ast` + `inspect` + `dis`. Reference impls: numba.jit frontend, jax.jit tracer.
- **Rust**: ~500 LoC extending the Phase 4 evaluator with new node types + pure-function dispatch table.
- **Wire**: additive JSON node types in REGISTER payload — backwards compatible with v0 servers.
- **Binary**: 0 added size. No new dependencies.

Estimate 2 phases of work in v0.1.

## Bail-out behavior

- **Strict mode (recommended)**: reject unsupported AST nodes at decoration time with clear error pointing at the source line. "We don't yet support `for` loops with dynamic bounds; rewrite as a comprehension or use `bv.fold(...)`."
- **Sidecar mode (chalk's path)**: fall back to Python subprocess at query time. Slow, but functional. Add only if real users demand it.

## Composability

Symbolic functions compose cleanly with aggregations:

```python
@bv.expr
def is_suspicious(amount, hour, country):
    return amount > 10_000 and (hour < 4 or hour > 23) and country in {"XX", "YY"}

@bv.feature
def suspicious_count_1h():
    return bv.count(
        where=is_suspicious(bv.col("amount"), bv.col("hour"), bv.col("country")),
        window="1h",
    )
```

Stateless transforms = symbolic interpreter. Stateful aggregations = declarative `bv.<op>` operators. The two compose at the `where=` / `value=` boundary.

# v0.2+ extension: custom stateful aggregations via the same symbolic trick

> Builds on the v0.1 symbolic frontend. Captured here so the design is coherent.

## The KV-store + reducer framing

User intuition: a custom aggregation is just "a per-entity state slot + a function that updates it on each event." That's correct for the simplest cases — `count`, `sum`, `min`, `max`, `last`, `streak`. But streaming aggregations have 4–5 things batch reducers don't:

1. **Windowing** — N bucket states (typically 64), framework owns bucketing + age-out
2. **Retraction** — when a bucket expires, "subtract" its contribution. Some ops have a clean inverse (count, sum, avg via `(count, sum)` pair); others don't (max needs a heap, percentile needs a decrement-supporting sketch like UDDSketch).
3. **Snapshot serde** — Phase 7 requires bincode round-trip for every state struct
4. **Memory budget** — sketches have configurable error → memory; framework wants to estimate per-entity cost
5. **Read-time computation** — `state` ≠ `value`; `avg` stores `(count, sum)` and divides at read time

## Minimal honest trait

```rust
trait CustomAgg {
    type State: Default + Serialize + DeserializeOwned;
    type Output: Into<Value>;

    fn update(&self, state: &mut Self::State, event: &Row);
    fn read(&self, state: &Self::State) -> Self::Output;

    // Required for windowed operators (most users):
    fn subtract(&self, state: &mut Self::State, expiring: &Self::State) {}
    fn merge(&self, a: &Self::State, b: &Self::State) -> Self::State { ... }

    // Required for memory-bounded sketches:
    fn memory_estimate(&self, _state: &Self::State) -> usize { size_of::<Self::State>() }
}
```

## The symbolic-Python version (the cool one)

Same chalk-style trick as v0.1, applied to mutable state:

```python
@bv.custom_agg(key="user_id")
class MovingDeviation:
    state: dict = {"count": 0, "mean": 0.0, "m2": 0.0}  # Welford's online stddev

    def update(self, state, event):
        state.count += 1
        delta = event.amount - state.mean
        state.mean += delta / state.count
        state.m2 += delta * (event.amount - state.mean)

    def read(self, state):
        return (state.m2 / state.count) ** 0.5 if state.count > 1 else 0.0
```

The symbolic interpreter traces `update` and `read` at decoration time, produces typed IR with mutable-state-slot accesses, compiles to a state machine in the Rust evaluator. Framework owns: per-entity HashMap, windowing, snapshot serde of the typed state dict, memory budgeting. User owns: the math.

This is the **same compiler path as v0.1**, just with mutable state slots added to the IR.

## Three tiers of "UDF" for clarity

| Tier | Example | Status |
|------|---------|--------|
| Pure stateless transforms | `amount_usd = amount * rate` with null handling | ✅ v0.1 (`@bv.expr`) |
| Custom stateful reducers | Welford's online stddev, custom decay schedules, domain scoring | ✅ v0.2+ (`@bv.custom_agg`) |
| Side-effecting / inference | call model server, return predicted score | ❌ separate sidecar; v1.x |

## Why this matters

It changes Beava's positioning from "stateful streaming engine with a chained DSL" to **"streaming feature platform where user features ARE Python code, including their own stateful reducers."** That's the difference between Snowpark and Snowflake-with-UDFs, or PyTorch and Caffe-with-Lua-config.

The chained-DSL becomes invisible. Users write Python; we trace, type-check, and compile to native server-side execution. UDFs in the meaningful sense (~95% of feature-authoring requests) collapse into "just Python."
