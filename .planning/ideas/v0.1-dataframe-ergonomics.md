# v0.1+ idea: polars-grade dataframe ergonomics

> **Status:** post-v0 idea sketch. Not on v0 ROADMAP. Captured 2026-04-23
> from SDK-syntax-audit (`.planning/audits/sdk-syntax-vs-dataframes.md`)
> + scalar-output discussion in autonomous session.

## TL;DR

Add a thin SDK sugar layer that makes Beava feel ~99% identical to polars
for stateless feature authoring, while preserving every streaming-native
asymmetry (decorators, `where=`, `as_of=`, retraction, `window=`).

Composition matters: this sketch + `v0.1-symbolic-python-frontend.md`
together turn Beava from "a streaming engine with a Python DSL" into
"polars, but features run server-side and update on every event." That's
the marketing line.

Audit headline (from `sdk-syntax-vs-dataframes.md`): Beava is **~71%
strict-green to polars** after these fixes (~41% as currently shipped).
24 of 34 audited operations become syntactically identical to polars.

---

## The 8 sugar items

Each item is additive — it does NOT break existing `bv.col(...)` /
`bv.count(...)` / dict-keyed-agg syntax. Old and new both compile to
the same JSON IR.

### 1. Class-attribute typed columns

**Today:**
```python
@bv.event
class Transaction:
    user_id: str
    amount: float

bv.col("amount") > 100   # works but stringly-typed
```

**v0.1:**
```python
Transaction.amount > 100   # → bv.col("amount") > 100, typed
Transaction.user_id == "alice"
```

The decorator already has the type info. Make `Transaction.amount`
return a typed `Col[float]` so mypy + IDE autocomplete + rename refactors
just work. Polars users feel home immediately.

**Cost:** ~30 LoC in the decorator, `__getattr__` returning a `Col`
proxy. **High leverage.**

---

### 2. Method-on-column aggregations

**Today:**
```python
bv.sum("amount", window="5m")
bv.count(window="5m")
```

**v0.1 (additive — both shapes coexist):**
```python
Transaction.amount.sum(window="5m").alias("total")
Transaction.amount.count(window="5m")
Transaction.amount.percentile(0.99, window="5m")
```

Matches polars `pl.col("amount").sum().alias("total")` exactly.

**Cost:** ~50 LoC — `Col.sum()`, `Col.count()`, etc. each return an
`AggExpr` with `.alias("name")`. **Pairs with item 8 (alias form).**

---

### 3. Accept `timedelta` / `pd.Timedelta` for `window=`

**Today:**
```python
bv.count(window="5m")  # only string accepted
```

**v0.1:**
```python
from datetime import timedelta
import pandas as pd

bv.count(window="5m")
bv.count(window=timedelta(minutes=5))
bv.count(window=pd.Timedelta("5min"))   # pandas user types this naturally
```

All three resolve to the same internal duration. Pandas user gets a
Beava feature on muscle memory.

**Cost:** ~10 LoC in window parser. **Cheapest possible v0 fix.**

---

### 4. Optional `.filter(...).agg(...)` chain (alongside `where=` kwarg)

**Today:**
```python
bv.count(where=Transaction.status == "decline", window="5m")
```

**v0.1:**
```python
# Both shapes work:
bv.count(where=Transaction.status == "decline", window="5m")
Transaction.where(Transaction.status == "decline").count(window="5m")
```

The `where=` kwarg is actually closer to SQL `WHERE` (which is the
streaming lingua franca — Materialize, RisingWave, Flink-SQL all use
predicate-as-kwarg). But polars users expect the chain. Ship both;
both compile to the same IR.

**Cost:** ~30 LoC — `Event.where(predicate)` returns a `FilteredEvent`
proxy whose aggregator methods inject `where=` into the underlying spec.

---

### 5. `asof()` table method (alongside `as_of=` kwarg)

**Today:**
```python
prices_table.lookup(symbol="AAPL", as_of=event.ts)
```

**v0.1:**
```python
# Polars-shaped:
prices_table.asof(event.ts, by="symbol")

# Natural English:
prices_table.at(time=event.ts, symbol="AAPL")

# Original (still works):
prices_table.lookup(symbol="AAPL", as_of=event.ts)
```

Polars 0.18+ has `df.join_asof(prices, on="ts", by="symbol")`. The
`asof()` method matches that idiom. `at()` is the natural English
phrasing for users who didn't grow up on polars.

**Cost:** ~20 LoC, three method aliases over one underlying call.

---

### 6. `.correct()` / `.delete()` table operations

**Today:**
```python
fraud_labels.upsert(transaction_id="tx-123", label="fraud")
app.retract(event_id=...)   # low-level primitive
```

**v0.1:**
```python
# Adding:
fraud_labels.upsert(transaction_id="tx-123", label="fraud")

# Correcting (retract original + upsert new value):
fraud_labels.correct(transaction_id="tx-123", label="ok")

# Pure rollback (drop the row):
fraud_labels.delete(transaction_id="tx-123")
```

Pandas user thinks `df.drop(idx)` and `df.loc[idx] = new_value` — they
get matching verbs. `app.retract()` stays as the lower-level primitive
for users who need event-id precision.

**Cost:** ~40 LoC — both methods are 2-call wrappers over retract +
upsert primitives.

### 7. **`output_kind: "scalar"` for ungrouped aggregations** ★

**This is the most important item — it's a real type system change,
not just sugar.**

**Today (broken):**
```python
@bv.derived(source="Transaction")
def total_revenue_5m():
    return bv.sum("amount", window="5m")    # ← does NOT compile;
                                            # group_by is mandatory
```

**Today's workaround (gross):**
```python
@bv.derived(source="Transaction")
def total_revenue_5m():
    return Transaction.group_by(bv.const("global")).agg(
        v=bv.sum("amount", window="5m")
    )

# Query: GET /get/v/global   ← invented "global" key
```

**v0.1 (proper):**
```python
@bv.derived(source="Transaction")
def total_revenue_5m():
    return bv.sum("amount", window="5m")  # ← no group_by → scalar

@bv.derived(source="Transaction")
def per_user_revenue_5m():
    return bv.sum("amount", group_by="user_id", window="5m")
                                          # ← group_by present → table
```

Wire shape:
```json
{
  "kind": "derivation",
  "name": "total_revenue_5m",
  "output_kind": "scalar",         ← NEW: alongside "table" and "stream"
  "ops": [{"op": "agg", "spec": {"op": "sum", "params": {"field": "amount", "window": "5m"}}}],
  "schema": {"fields": {"value": "f64"}, "optional_fields": []}
}
```

Query:
```bash
GET /get/total_revenue_5m              → {"value": 12345.67}
POST /get {"features": ["total_revenue_5m"]}
                                       → {"result": {"total_revenue_5m": 12345.67}}
```

No PK in URL. Returns the value directly, not nested under a `_global` key.

**Why this is right (vs the implicit-`_global`-key SDK trick):**
- Type honesty: scalar features show as `output_kind: scalar` in
  `GET /registry`. Future you reads the registry and knows immediately
  there's one value, not a "table with one row keyed on `_global`."
- Server fast path: no `HashMap<EntityKey, Slot>` lookup, no PK parse.
  `state.scalar_features[feature_id].read()` — direct read.
- Snapshot/recovery: trivial — add `scalar_state: HashMap<FeatureName, AggState>`
  beside existing `state_tables: HashMap<NodeName, StateTable>`.
- Composability: opens door to scalar observability features
  (`registry_version`, `events_per_sec_ewma`, `uptime_seconds`).
- Polars parity: `df["amount"].sum()` returns a scalar, not a 1-row
  DataFrame. Beava should match.

**Cost:** ~1 plan of server work — new `OutputKind::Scalar` enum
variant, `ScalarState` container, `/get` resolver dispatches by kind.
Snapshot serde extends with one new section. SDK desugars
"no-group_by aggregation → output_kind=scalar." Probably 200–400 LoC
across SDK + server. **Single biggest leverage item in this sketch.**

**Open design question for v0.1 plan-time:** does the singleton agg
still go through the WAL+apply loop? Probably yes (apply-after-fsync /
apply-after-append per Phase 6.1; same durability contract). The
scalar output is purely a *read shape* + *layout* optimization, not
a write-path one.

### 8. Operator-name aliases + dataframe-canonical names

From the syntax audit's open questions — pick canonical names, keep
the other as aliases for SDK ergonomics:

```python
# Aliases the audit recommended:
bv.distinct       == bv.count_distinct      (drop the noun, polars-style)
bv.p99(field)     == bv.percentile(field, 0.99)   (sugar — common case)
bv.consecutive    == bv.streak              (more readable)
bv.is_null        == bv.isnull              (snake_case, polars)

# Cast accepts Python builtins:
bv.col("x").cast(float)    # equivalent to .cast("f64")
bv.col("y").cast(int)      # equivalent to .cast("i64")
bv.col("z").cast(str)      # equivalent to .cast("str")

# Join kwarg rename (audit's #2 v0 fix):
left.join(right, on="user_id", how="left")   # was: type="left"
```

**Cost:** ~50 LoC of aliases. Pure additive, no breaking change.

## Composition with the symbolic frontend (v0.1-symbolic-python-frontend.md)

The two sketches stack:

```python
# Symbolic frontend: write Python, get typed IR
@bv.expr
def is_high_value_decline(amount, status, fees):
    if status != "decline":
        return False
    return (amount + fees) > 1000

# Ergonomics layer (this sketch): polars-shaped composition
fraud_count_5m = (
    Transaction
        .where(is_high_value_decline(
            Transaction.amount,
            Transaction.status,
            Transaction.fees,
        ))
        .count(window="5m")
        .alias("fraud_count_5m")
)
```

Result: native Python flow control + polars-shaped chains + streaming
state. Closest competitor in syntactic feel is Materialize SQL, which
makes you write SQL. Beava's pitch becomes "feels like polars; runs
like a streaming engine."

## Roadmap fit

Both v0.1 sketches together = roughly one milestone of work
(2–3 phases). Order:

1. **v0.1 Phase A**: scalar `output_kind` (item 7) — server change,
   foundational. Without this, "ungrouped aggregations" stays broken.
2. **v0.1 Phase B**: ergonomics layer (items 1, 2, 3, 4, 5, 6, 8) —
   pure SDK, ~500 LoC, no server change. Ship after A.
3. **v0.1 Phase C**: symbolic Python frontend (separate sketch) — the
   biggest single SDK feature, paired research direction.

Items 1–6, 8 are SDK-only. Item 7 is SDK + server. Item 7 is also the
prerequisite for several common feature shapes ("global counter for
this stream") that should work day-1 in the playground.

## What this is NOT

- Not a v0 commitment. v0 ROADMAP is locked at 18 phases and ships
  with the chained-DSL surface.
- Not a `pl.DataFrame` clone — Beava remains declarative + server-resident
  + state-mutating. The visual similarity is in the *expression-building
  syntax*, not the execution model.
- Not a substitute for the symbolic frontend. The ergonomics layer
  makes the existing API polars-shaped; the symbolic frontend makes
  feature *bodies* native Python. Both wins, neither replaces the other.
- Not free — item 7 (scalar output) is genuine server work. The other
  7 are SDK sugar.

## Open questions for v0.1 plan-time

1. **Which scalar features ship with v0.1 Phase A?** Just user-defined,
   or also a small built-in catalog (`registry_version`, `events_per_sec`,
   `uptime`, `pending_pushes`)? The built-ins are useful observability
   primitives.
2. **Does scalar share the WAL apply-after-* contract?** Yes by default;
   confirm at plan time.
3. **`bv.expr` decorator vs class-attribute typed columns** — there's
   conceptual overlap. Class-attribute (item 1) is a typed `Col` proxy;
   `bv.expr` is full-Python AST. Neither replaces the other; they
   compose. Document the boundary clearly so users know which to pick.
4. **Marketing site usage** (from audit open question): the marketing
   site uses `e.agg(my_count=bv.count(), my_streak=bv.consecutive())`
   syntax that doesn't compile today. Item 7 (scalar) + item 2
   (method-on-col) + item 8 (aliases) together close most of that gap.
   Treat this sketch as the v0.1 commitment that makes the marketing
   copy true.
