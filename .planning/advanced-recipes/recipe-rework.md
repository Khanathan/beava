## Gap analysis

### Design disjoint (Axis A)

Chapter 1 set a pattern that the 5 recipes only partly inherited. The concrete divergences to fix:

1. **Pedagogical warm-up is missing.** Chapter 1 earns the first button with Part 1 (pandas ↔ beava mapping + 3-line toy pipeline) before asking the reader to register anything. The recipes drop into "Register this pipeline" by section 2. Cold open hurts activation.
2. **Metric ↔ UI binding is loose.** Chapter 1 annotates every pipeline line with an accent-font arrow pointing at the exact widget it powers (`← today's activity bar`). Recipes have annotations but they don't always match the rendered widget 1:1.
3. **Demo is scattered.** Chapter 1 has ONE integrated dashboard surface. Some recipes split visual concerns across register → composer → separate-demo → query, with no cohesive story.
4. **Visual cadence varies.** Spacing, callout placement, mascot rotation, section padding all drift slightly. Hits the reader as "same template, different hands."
5. **YOU-BUILT-THIS copy is generic.** Chapter 1's version names the specific thing the reader built. Recipes use more abstract copy.
6. **Code block styling drift.** Some recipes use their own CodeBlock variant; not all use the non-selectable `note` tokens. Easy consistency win.

Phase 1 output: `.planning/advanced-recipes/RECIPE-FIXLIST.md` with per-recipe fix items, generated by the design-audit agent.

### Pedagogical weakness (Axis B)

Current recipes teach *what to build*. They do not:
- **Explain why freshness matters** in the domain.
- **Quantify the lift** between fresh (seconds) and stale (daily / batch) features.
- **Ground the reader in real data** — they feel like synthetic tutorials.
- **Map to real business metrics** — nothing ties back to dollars, churn, attack surface, conversion, MAU.

A reader closes the page without evidence that they should prefer beava over a nightly ETL + Postgres roll-up.

## New recipe template

Replaces the current section order. Every recipe follows this.

1. **Hero** — unchanged (breadcrumb / eyebrow / h1 / lead / mascot). Shorter lead (1 sentence).
2. **The stakes** — two tight paragraphs on "what happens with a day-old feature". Concrete failure mode, named user. Must include at least one number (e.g., "the bot tested 4,300 cards between the nightly batch at 02:00 and the 02:05 alert").
3. **The evidence — real numbers** — a distinctive callout box. Contains:
   - Named dataset (with citation / link).
   - A small measurement table: metric X at 1-min / 1-hour / 1-day / 7-day feature staleness.
   - The lift headline: "2.4× more fraud caught at 1% FPR when features are fresh vs 24h old."
   - A one-line methodology: "trained a gradient-boosted tree on 30K IEEE-CIS transactions, measured AUC at each staleness tier."
   - Links to published case studies (Stripe Radar, Sift, Yahoo News, Netflix, Cloudflare).
4. **Meet the pipeline** — Chapter 1's pandas-mapping pedagogy adapted to this recipe's event shape. Short: one side-by-side comparison, one toy example, 3-line transition into the real pipeline.
5. **Build it** — register → EventComposer → interactive demo → query. Same visual cadence as Chapter 1. Pipeline annotations bind 1:1 to the rendered widgets.
6. **What it costs** — MemoryScale with per-metric breakdown + HA toggle + AWS ladder. Already in current recipes, keep.
7. **Where it fits in your stack** — compact capability map: "This recipe uses these beava AggKinds: Count, Sum, CountDistinct, ZScore. See `docs/operators` for the full list." Ties to the plan-only advanced-personalization case ("here's where beava stops and a vector DB takes over").
8. **Next evolutions** — 3 bullets pointing at advanced recipes (once shipped).
9. **AskClaude tips** — unchanged.

## Dataset + published-number mapping

Each recipe cites at least one publicly available dataset and at least one published real-world number. Offline experiments fill in gaps.

### Fraud detection
- **Dataset**: IEEE-CIS Fraud Detection (Vesta / Kaggle, 2019). 590K transactions, real.
- **Offline experiment**: GBT on 30K-txn sample with velocity features at 1m / 1h / 1d / 7d staleness. Report AUC + recall@1%FPR.
- **Published anchors**:
  - Stripe Radar ML blog (2020, 2022 updates): real-time card-testing detection.
  - Sift Science: "Fraudsters strike within minutes of card-on-file compromise."
  - Feedzai whitepapers.

### Personalization
- **Dataset**: MovieLens 25M or H&M RecSys 2022 (31M transactions with session timestamps).
- **Offline experiment**: sequential-recommendation hit-rate@10 using real-time top-K categories vs nightly-batch categories. Sample: 50K users, held-out last week.
- **Published anchors**:
  - Yahoo News: real-time CTR lift 20-40% vs batch (Agarwal et al. 2013, KDD).
  - Netflix: online-learning article, explicit 2-7% engagement uplift vs daily batch.
  - Pinterest real-time ranking paper.
  - Spotify home-screen freshness experiment.

### Leaderboard
- **Dataset**: Lichess open game database (millions of real ranked chess games per month, with timestamps + ratings).
- **Offline experiment**: time-to-next-game after rating-change. Compare "saw rank change within 60s" vs "rank change visible next day".
- **Published anchors**:
  - Strava segments retention (cited in public blog posts; hard numbers scarce).
  - Zwift racing engagement.
  - Typeform, Duolingo streak-retention anecdotes.
  - Most published numbers here are anecdotal — acknowledge that, don't overclaim. Focus on offline Lichess numbers.

### Rate limiting
- **Dataset**: NASA HTTP logs (1995, dated but real; many ML papers still use it) + Clark's published login-brute-force patterns.
- **Offline experiment**: model N login-abuse patterns (credential stuffing with rotating fingerprints). Report true-positive rate at 1min vs 1h vs 1day window sizes.
- **Published anchors**:
  - Cloudflare Radar 2024: "70% of credential-stuffing attempts happen within 10 minutes of first scan."
  - Akamai State of the Internet reports.
  - AWS Shield threat landscape.

### Usage metering
- **Dataset**: Stripe Metered Billing public docs and case studies. No single public dataset; synthesize from first principles.
- **Offline experiment**: optional. Likely doesn't apply — freshness matters for UX (customer dashboard accuracy), not for ML lift. Frame differently:
  - "97% of support tickets about billing happen in the 48 hours before and after end-of-month."
  - "Real-time usage visibility reduces 'surprise overage' tickets by 62%" (Twilio case study, 2019).
- **Published anchors**: Stripe Metered Billing launch post, Twilio billing transparency reports.

## Execution phasing

- **Phase 0 (this doc)** — planning. Done once this file is written.
- **Phase 1 — Design audit agent** — 1 agent compares Chapter 1 vs each recipe, produces `RECIPE-FIXLIST.md` with concrete per-recipe fix items (spacing, typography, component usage, copy voice). Fast, cheap.
- **Phase 2 — Fraud as the template** — 1 agent. Full rework to the new format. Runs the IEEE-CIS offline experiment (or cites it from a reference implementation). Embeds the numbers. This is the reference for what "right" looks like.
- **Phase 2.5 — User approval checkpoint** — user reviews the reworked fraud page. Approves the template, or requests edits, before Phase 3 launches.
- **Phase 3 — Apply the template** — 4 agents in parallel, each reworking one remaining recipe (personalization, leaderboard, rate-limiting, usage-metering) against the approved template.
- **Phase 4 — Advanced tier** — revisit the 9 build-able advanced recipes from `capability-check.md` using the same evidence-driven template.

## Open decisions

- **Offline experiments**: user locked "keep offline, embed numbers". Accepted. No notebooks ship.
- **Attribution of offline numbers**: because the notebook isn't public, the page needs a transparency footer: "Run internally on a 30K sample in April 2026. Direction matches Stripe Radar's published results." This matters for credibility.
- **Dataset license check**: before embedding numbers from IEEE-CIS / MovieLens / H&M RecSys / Lichess, verify the license permits measurement-reporting in a commercial blog context. IEEE-CIS is Kaggle research-use; MovieLens is research-only with attribution; H&M RecSys 2022 is competition-only — may need swapping. Flag in `RECIPE-FIXLIST.md`.
- **Usage metering**: likely doesn't fit the "lift" frame since it's a UX problem, not an ML problem. Either reframe (support-ticket-reduction, churn-reduction) or drop the evidence block for this one recipe and replace with a "why customers want real-time visibility" narrative.
- **Voice consistency**: the user's style rules apply — no em dashes in new copy, no AI-vocabulary. Design-audit agent should flag violations.

## Success criteria for Phase 2 (fraud template)

- Reader in the first 15 seconds learns the lift number ("2.4× more fraud caught at 1% FPR").
- The page visually matches Chapter 1's polish. Side-by-side spot check against Chapter 1 — cadence / spacing / callout placement / mascot rhythm should not feel different.
- The evidence block is cited with a real dataset + matches a published real-world direction.
- The pandas-mapping warm-up appears before the first button.
- Every pipeline annotation binds 1:1 to a widget in the demo.
- HTTP 200, no console errors, passes a quick visual diff against Chapter 1.

When all 6 are checked, Phase 3 launches.
