So what is bv.fork()?

It's basically a with block that spawns a local replica of live prod state, scoped to whatever keys you care about. You iterate features against REAL production bytes — not stale staging data — then close the context and prod doesn't care.

That's it. That's the primitive.

Most feature stores I've used make you pick between stale staging (your test says 47.3, prod says 50.1, you burn two days finding the difference) or poke at prod and pray (no isolation, hope nothing explodes). Fork is the third option I always wanted — it's the whole reason I built this thing.

Does it solve all skew? No. It closes the staging-data skew axis. Feature-logic drift between replay and live push is still on you — if you forgot to handle late events, fork won't save you. But the "my test data is lying to me" axis, the one that actually burns two days of debugging, is gone.

Fork semantics FAQ

What does "scoped replica" mean technically?
When you enter bv.fork(host, scope=filter), Beava opens a TCP connection to the prod server and requests a snapshot of all state matching the scope filter. That snapshot lives in a local in-memory replica. You compute against it; prod never sees your reads.
What about writes during the context?
You choose. Default is snapshot isolation — replica is frozen at entry, prod writes don't propagate. Pass tail=True for streaming replication — replica catches CDC updates from prod as they land. Snapshot for reproducibility; tail for realism.
My DAG has transitive keys. What does scope pull?
Beava walks the feature DAG and pulls every upstream dependency of the scoped keys. If OnboardingSignals(user_id=u123) reads from a stream filtered by a different column, the replica pulls what the DAG needs. If transitive scope exceeds a configurable budget, Beava errors rather than silently under-fetching (I got burned by this once — loud is better than quiet).
PII / data governance?
Scope filter is enforced server-side before data leaves prod. The prod server can require a scoped access token; the token's claims gate which keys a client can request. Audit log at /debug/audit. Full enterprise RBAC ships in Cloud.
Can a fork write back to prod?
No. Forks are read-only replicas; closing the context destroys local state. To ship a feature change, you commit the pipeline definition and beava deploy through your normal path.

Why Beava exists

The feature loop. You write a feature in Python on your laptop. You backfill against 30 days of history. You fork a scoped replica of prod, verify values match. You ship. Same file, laptop to production. Four primitives — @bv.stream, @bv.table, bv.replay(), bv.fork() — one lifecycle, one person owning the whole thing.

One binary, any box. Beava is a single Rust binary. In-memory state. Append-only event log (group-commit fsync before ack). Binary TCP protocol or curl — your choice. Runs on a MacBook, a VPS, a Hetzner box, a k8s pod. Same binary, same config, same mental model.

Scoped for the long tail. Teams under 1M events/sec with working sets that fit in RAM (modern instances reach 1.5 TB+). That's the long tail of real-time ML workloads — the shops where the ops cost of distributed streaming exceeds the business value of real-time. If you're past 1M eps with a platform team that already runs Flink well — Beava probably isn't a good fit, and Flink is what you want. Beava is for everyone else.

How Beava fits the landscape

Most feature platforms today assume a Kafka/Flink substrate underneath them — fine if you have a team running it. Materialize and RisingWave are incremental SQL engines with different ergonomics. Beava is single-node first, Python-authored, scope-honest for teams where the ops cost of distributed streaming exceeds the business value. Different tools for different slopes of the scale curve — and a team that already runs Flink should keep running Flink.

💾 Durability, recovery, semantics

  • Event log — fsync'd before ack, group-commit batched. Worst-case data loss: ~1s.
  • Snapshots — every 5 min (tunable). Boot = load snapshot + replay log (~30s / 10M events).
  • Replica — CDC from primary; manual promotion on primary death. Auto-failover is Cloud.
  • OOM safety — fails loud at BEAVA_MEMORY_LIMIT_MB. Resize the box, restart. No silent corruption.
  • Delivery — at-least-once with client retry. Exactly-once counters via event_id dedup.

Full semantics doc: SEMANTICS.md.
