left column wins; the right column is downstream and must be regenerated.

## Architectural commitments

These are locked invariants; edits to source must preserve them, and
documentation must reflect them.

- **v0 events-only at the top level** â€” `kind: "table"` register payloads return `unsupported_node_kind`; tables are produced via `kind: "derivation", output_kind: "table"`.
- **Processing-time only in v0** â€” `event_time` / `event_time_ms` field names are reserved and rejected with `unknown_field_event_time_v0`.
- **mio-only data plane** â€” push/get/upsert/delete/retract dispatch goes through `crates/beava-server/src/apply_shard.rs::dispatch_*_sync`. axum is restricted to the admin sidecar in `http_admin.rs`. CI test: `crates/beava-server/tests/phase12_6_mio_only_dataplane.rs`.
- **No tables top-level**; **no joins, sessions, `bv.fork`, playground**.
- **Wire format locked** â€” opcodes never change shape; new wire features get new opcode numbers.
- **Pagefind cannot index docs/\*** â€” those pages are React+Babel JSX (`className`), not static HTML. Tracked TODO in `beava-website/scripts/build-search.mjs`.

## SDK reference pages â€” markdown source consideration

The 12 SDK pages under `beava-website/project/sdk/` are hand-written HTML
(~6,500 LOC total). They mount shared chrome via React+Babel (SiteHeader /
SiteFooter / SdkSidebar / SdkTOC / SdkPager) and load `sdk-shell.css` for
prose styling. Authoring is 80% prose + 20% chrome boilerplate.

A markdown-based pipeline was considered for this PR. **Findings:**

- No existing markdown-to-HTML build script. `beava-website/scripts/`
  contains only `build-search.mjs`, `build-llms-txt.mjs`, `check-links.mjs`,
  `scrub-slop.mjs` â€” none render markdown.
- The docs/\*\*/\*.html pages use a parallel pattern (React+JSX inside
  Babel templates), not markdown. Pagefind can't index them either.
- A migration would need: (1) a build script that renders MD â†’ static HTML
  with the SDK chrome injected at build time; (2) an MD source file per
  current HTML page (re-authored or programmatically converted); (3) a
  templating layer that handles the per-page mount data (sidebar `active`,
  pager prev/next, TOC anchors); (4) a regen step in the deploy workflow.
- Estimated ~2-3 days of work; produces lighter source files
  (~150-300 LOC each in markdown vs 400-1100 LOC in HTML).
- Side benefit: the same pipeline could fix the docs/\* Pagefind regression
  by emitting real HTML with `class=` attributes from MD source.

**Decision: deferred.** Worth doing in a dedicated PR alongside fixing
the docs/\* search regression â€” both want the same builder. Tracked here.
