Install. Push. Query.

Install an SDK, point it at a beava server, and make your first push and query — in five minutes, in any language.

Pick a path

Most readers want the install + curl path below; if you'd rather poke at a hosted instance, the playground takes you there in one click.

Install & run locally

One brew tap or one docker run. The whole thing fits in your terminal history.

Five minutes →
Try the playground

A hosted beava in your browser. Push synthetic events, read counters, no install.

Open playground →

Heads up. The playground is a shared instance — use it to learn, not to load-test. For anything past 100 RPS, run beava locally; you'll see the same behavior at 50× the throughput.

01 Install beava

Pick whichever you have closest to hand. All three give you the same single binary on your $PATH.

terminal 
# tap once, install once
brew tap beava/beava
brew install beava

# now you have `beava` on your PATH
beava --version
# → beava 0.14.2 (darwin/arm64)
terminal 
# persistent volume keeps your aggregates across restarts
docker run -d --name beava \
  -p 7421:7421 \
  -v beava-data:/var/lib/beava \
  ghcr.io/beava/beava:0.14
terminal 
# one-line install for linux + macOS
curl -fsSL https://get.beava.dev | sh
terminal 
go install github.com/beava/beava/cmd/beava@latest

Tip. beava ships static. There's no JVM to size, no broker to configure, no schema registry to register against. If beava --version prints, you're done.

02 Define your first counter

Pipelines in beava are declared in TOML and registered over HTTP. Here's a 24-hour rolling counter of page_view events, keyed by user.

pipelines/page_views.toml 12 lines 
[pipeline]
name      = "user_page_views"
key       = "user_id"
window    = "24h"     # rolling, not calendar
operator  = "counter"

[pipeline.match]
event     = "page_view"
where     = "path != '/healthz'"

[pipeline.retention]
tick      = "60s"
memory    = "adaptive"

Register it once. beava picks it up live — no restart.

terminal 
beava apply pipelines/page_views.toml
# ✓ pipeline 'user_page_views' registered (rev 1)

03 Push events, read the counter

beava speaks HTTP — every operation is a single request. Open two terminals, or just one if you're feeling brave.

request · push event 
curl -X POST \
  http://localhost:7421/push \
  -H "content-type: application/json" \
  -d '{
    "event": "page_view",
    "user_id": "u_001",
    "path": "/pricing",
    "ts": "2026-05-07T14:30:01Z"
  }'
response · 6 ms 200 OK 
{
  "ok": true,
  "accepted": 1,
  "pipelines": [
    "user_page_views"
  ],
  "latency_ms": 5.8
}

Now ask beava for the running aggregate:

request · read counter 
curl http://localhost:7421\
  /q/user_page_views\
  ?key=u_001
response · 0.9 ms 200 OK 
{
  "key":    "u_001",
  "value":  14,
  "window": "24h",
  "asof":   "2026-05-07T14:30:02Z"
}

Pipeline parameters

Every operator shares the same shape. Here are the fields you'll touch in your first week.

name required string

Stable identifier for the pipeline. Used in the query path: /q/<name>. Lowercase, underscores, no slashes.

key required string

Field on the event used as the rollup key. Common values: user_id, tenant_id, ip. Composite keys use a dotted path.

window duration

Length of the rolling time window. Accepts Go duration strings — 30s, 15m, 1h, 24h, 7d. Maximum 30 days.

default 1h

operator enum

One of counter, velocity, last_n, topk, budget. See the operator reference.

default counter

tick duration

Bucket resolution. Smaller ticks give smoother aggregates and cost more memory. The default is sane for most workloads.

default 60s

Common questions

Where does beava store its state?

Aggregates live in memory and snapshot to disk on a tick. Default location is /var/lib/beava. You can change it with --data-dir or by setting BEAVA_DATA in your env.

beava does not persist your raw events — it persists the aggregates over them. If you need replay, keep your events somewhere else (S3, your warehouse).

What happens if beava restarts mid-window?

It re-reads the most recent snapshot and resumes. You'll lose at most one tick of un-flushed events; for a 60-second tick that's a maximum of one minute of in-flight data. Tune the tick down if your application can't tolerate that.

Can I run beava in HA?

Yes — run two replicas behind a sticky-by-key load balancer. There's a longer write-up in Operations · High availability. For most workloads a single beava on a t3.small handles tens of thousands of events per second.

How does beava compare to Redis?

Redis is a key-value store you can build rolling counters on top of (with Lua + ZSETs + a lot of care). beava is a server that already is rolling counters — and velocities, leaderboards, last-N-seen — without you assembling the primitives.
