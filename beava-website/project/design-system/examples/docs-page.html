Introduction

beava is a single-binary feature server for real-time features — rolling counters, velocities, last-N-seen, leaderboards, and rate limits. HTTP in, HTTP out. No Kafka to babysit, no JVM to tune.

What beava is

beava sits in the gap between Redis and a full streaming stack. You push events at it over HTTP and ask running aggregates over rolling time windows, also over HTTP. The whole thing is one Go binary, configured with a small TOML file.

Concretely, that means you can replace a typical Kafka → Flink → Redis pipeline — three systems to operate, three failure modes to understand — with a single process you can drop on a t3.small and forget about.

Note. beava is not a database. It does not persist your raw events; it persists the aggregates over them. If you need to replay history, keep your events somewhere else (S3, your warehouse, Kafka if you must).

Vision

Real-time features should not require a streaming priesthood. Most teams that ship rolling counters end up writing surprising amounts of code to do something a database almost does, but doesn't. We think the right shape is small, opinionated, and HTTP-first:

  • One binary you can run anywhere a Linux process runs.
  • Operators that match the questions teams actually ask: “how many in the last N seconds?”, “who are the top K right now?”.
  • An API that fits in a curl command, on purpose.

We're inspired by what DuckDB did for analytics and what SQLite did for embedded storage. beava aims for the same shape, applied to streaming features.

Features

Rolling counters

Sliding time windows over any event stream. Define a window, push events, query the current count in sub-millisecond time. Read the rolling-counter docs →

Leaderboards

Top-N by any metric, always sorted, never stale. Updated incrementally as events arrive.

Rate limits

The same primitive that powers counters powers rate limits. Set a budget per key, ask for a decision in a single HTTP call.

When to use beava

Use beava when you need running aggregates over events with a freshness budget measured in milliseconds and an operations budget measured in “not very many”. Don't use beava as a primary store, and don't use it when the same question can be answered with a SQL query against your warehouse.
