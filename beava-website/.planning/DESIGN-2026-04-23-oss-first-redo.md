## Problem Statement

The website's job is to empower a specific developer to go from "landed on beava.dev" → "installed beava, registered a feature, querying it in production" in the smallest possible window of effort and time. Today's site is partway through a panel-audit-driven rewrite focused on surviving HN skepticism. That plan is ~70% defensive (durability section, Redis preemption, perf transparency) and ~30% offensive ("Or just ask Claude," new Chapter 1). The new framing re-weights toward empowerment, which means acceleration beats defense.

## Demand Evidence

- Today's 18:50 handoff documents v1 + v2 panel audits with ~4/10 v2 personas independently flagging specific concerns (curl install rejection, durability hole, Redis comparison)
- Beava is pre-launch OSS with a pre-scheduled HN + Reddit + Twitter reach plan
- Target-user session (this doc): Priya, YC fintech CTO, currently building velocity checks in Redis+Lua, one backend engineer is spending ~40% of his time on fraud plumbing — beava's exact wedge
- External research: 43.5% of early OSS adopters discover via tech social platforms (HN, Reddit, dev.to); 20.3% via word of mouth — confirming the landing page needs to CONVERT reach, not CREATE it

## Status Quo (what Priya lives with today)

- redis + custom Lua scripts for velocity rules
- each new feature = one more Lua script + custom DSL + maintenance
- one backend engineer spending ~40% on fraud plumbing
- sprint deadline for fraud rules in prod
- has CASH ($500/mo is fine) but not ENGINEER-TIME
- has autonomy — she can pick the tool without a 6-month review cycle

## Target User & Narrowest Wedge

**Priya — 10-person YC fintech, CTO.**

The narrowest wedge beava.dev can ship:

- Priya lands on beava.dev (via HN, Twitter, or YC founder referral)
- Within 90 seconds: understands what beava does + decides it could work for her
- Within 10 minutes: has installed beava locally, registered a velocity feature, queried it with curl
- Within a weekend: has fraud velocity rules live in staging with her own events flowing through
- Within two weeks: in production, replacing redis+Lua for velocity

The website's job is the 90-second decision + the 10-minute install-and-query. Everything downstream is Priya's work, not ours.

## Constraints

- **Tech stack**: Static HTML + React-via-Babel (existing pattern). Don't rewrite the rendering stack. Use what's shipped.
- **Local dev**: Python `http.server :8765` (existing). No build pipeline required to iterate. Pagefind adds a build step for `/docs` but stays static.
- **Memory planning**: Site runs on static hosting. Zero server cost except an `/api/waitlist` endpoint for the cloud form.
- **Voice**: OSS-developer. Not SaaS-product. Matches DuckDB / bun / Linear restraint, not Tinybird / Materialize sprawl.
- **Timing**: This design is a 3x scope increase over the 18:50 handoff. Realistic: 3 weeks of focused work, or 1 long week with CC on rails. If shipping under HN-launch time pressure, sequence matters — do the home first.

## Premises

All agreed during this session:

1. **The panel audit was partly noise for Priya.** Durability section = still binding (reframed for recovery time on a single box, not completeness). Why-not-Redis = demoted (Priya already left Redis). Perf transparency = reframed to "runs on your box" not "3M eps/core."
2. **The queue was missing Priya's vertical.** No fraud-specific recipe page existed in the 18:50 queue. Recipes are now a first-class surface, not a future afterthought.
3. **Home page should answer "what + how" only.** Trust moves to `/docs`. Application moves to `/guide/recipes`. Pedagogy moves to `/guide/chapter-1`.
4. **OSS-first voice over SaaS-product voice.** No "Book a demo" / "Contact sales." No Pricing nav item. No customer-logo band. Commercial offering surfaces only as a dismissable banner.
5. **Cloud offering stays minimal.** Banner + waitlist form page at `/cloud`. Never a top-nav tab. Phase when cloud is real.
6. **Community is hosting, not reach.** /community hosts ongoing engagement (GH Discussions primary, Discord later). Reach is a separate launch-plan concern (HN, Reddit, Twitter, dev.to, LinkedIn).

## Approaches Considered

### APPROACH A: Execute the 18:50 queue as-is

Ship items 1-10 from the earlier handoff in order. All panel-audit-derived. No re-planning cost.

**Pros:** momentum, evidence-backed, actionable today
**Cons:** 70% defensive mix persists; Priya's vertical has no dedicated surface; durability/perf framed for the Raj panel audience, not Priya; Why-not-Redis spends words on a battle Priya already won
**Effort:** ~5-6 days human / ~3-4 hrs CC
**Rejected:** the re-framing ("empower Priya") doesn't match the queue's shape. Executing it ships a version of the page that's optimized for a different user.

### APPROACH B: SaaS-product-style landing (the Tinybird / Chalk pattern)

Full marketing site: hero + customer logos + features matrix + pricing + demos + testimonials + CTAs throughout. 12-14 scrolls.

**Pros:** conventional, proven, supports a cloud-first business model
**Cons:** user explicitly rejected this direction ("less like product, very close to developers"); no customer logos yet; no cloud product yet; tone would clash with the playful beaver voice that's already shipped
**Effort:** ~3-4 weeks
**Rejected:** wrong shape for an OSS project with pre-launch users.

### APPROACH C: Minimal OSS purity (the SQLite pattern)

Single-page site. Hero + install + one code example + README-style feature list + footer. No /guide, no /docs, no /community, just the repo.

**Pros:** maximum OSS credibility, shipped in a weekend, nothing to maintain
**Cons:** no pedagogy surface → Chapter 1 ("Event = row, Stream = dataframe") has nowhere to live; no recipe grid → Priya can't see "fraud velocity checks" as a dedicated page; no cmd-K search → power users bounce looking for the rate-limit operator docs
**Effort:** ~1 week
**Rejected:** too bare for beava's 40+ operator complexity. SQLite is 1 feature with deep docs; beava is 40+ features that need browsable recipes.

### APPROACH D: OSS-first with layered depth (RECOMMENDED)

Home pitches "what + how" in 7 tight sections. `/guide` holds pedagogy + recipes. `/docs` is Chalk-style reference with cmd-K. `/community` aggregates engagement. Cloud surfaces as a banner, not a nav tab. DuckDB-pattern, not ClickHouse-pattern.

**Pros:**
- Respects OSS voice (no phone-home, no pricing-first, no customer logos)
- Multiple surfaces each do one job (home = pitch, guide = learn, docs = reference)
- Priya's vertical gets a dedicated recipe page
- Scales to more recipes + more docs without reshaping the home page
- Home stays under 7vh (Linear-tier restraint)
- pagefind cmd-K gives the Chalk-feeling without phone-home

**Cons:**
- 3x scope vs 18:50 queue; requires sequenced execution
- `/docs` and `/community` add surfaces to maintain

**Effort:** 3 weeks human / ~6-8 hrs CC over 3 focused sessions

## Recipe page template

Every recipe page at `/guide/recipes/<slug>` follows this structure:

```
1. HERO                 ← mascot (per-card pose) + recipe name + 1-line problem
2. THE PROBLEM          ← one paragraph, named-user style
                          (e.g., "Priya at a 10-person fintech is wiring Redis+Lua velocity
                          rules by hand. Every new rule = one more Lua script.")
3. THE PIPELINE         ← ~15 lines of Python, copy-paste-runnable
4. REGISTER IT          ← curl command to POST /pipelines/<name>
5. QUERY IT             ← curl POST /batch/<table> with 3 user IDs
6. WHAT YOU GET         ← real latency + memory on typical hardware
                          (e.g., "5ms P99 for a 1000-user batch query, ~2GB RAM for 1M entities")
7. NEXT STEPS           ← 2-3 related recipes + /docs deep-links
                          + "Or just ask Claude → `/beava add rate-limit for login by IP`"
```

Visual rhythm: different mascot pose per recipe; different pipeline shape per recipe (counting / velocity / top-k / threshold+ban / sum+window).

## Chapter 1 widget inventory

`/guide/chapter-1` reuses the pedagogy plan from the 18:50 handoff item #10, with renewed commitment to shipping interactive widgets:

- `<EventsTable/>` — rows append as reader clicks
- `<AggregationTable/>` — derived view updating live off the events table
- `<ClickButton/>` — user-driven event source (primary + secondary click types)
- `<WindowDemo/>` — 5-minute rolling window with visual decay
- `<CodeCompare left right/>` — beava Python side-by-side with pandas equivalent
- `<PipelineDiagram/>` — animated one-event-two-views-from-one-write

All client-side JavaScript. Zero backend calls. The chapter teaches the abstraction by making the reader operate it in-page.

**Constraint:** beava v0 does not support Form A (derived expressions). Every code block in Chapter 1 uses single-operator aggregations only. No `avg = total / sessions`. This matches what Priya will actually write in production.
